# nextjs是什么

Nest **是带完整构建规范后端应用的 Node.js 框架**。使创建可重用的逻辑部分成为可能，开发人员可以在多种不同类型的应用程序中利用这些逻辑部分。 从技术上讲，一旦创建了适配器，Nest 便可以使用任何 Node HTTP 框架。
目前支持两个 HTTP 平台：express 和 fastify。 您可以根据您的需求选择最适合平台。

- platform-express： Express 是一个著名的、极简的、专为 node 开发的 web 框架。它久经考验、适用于生产环境的软件库，并且拥有大量的社区资源。默认情况下使用 @nestjs/platform-express 软件包。许多用户对 Express 都很满意，并且无需采取任何操作即可启用它。
- platform-fastify： Fastify 是一个高性能且低开销的框架，高度专注于提供最高的效率和速度。

# Controllers

- 请求入口（路由处理层）:HTTP Request → Controller → Service → Response
- Controller 不负责业务逻辑，处理应用程序的特定请求。只负责“转发 + 组织输入输出”，一个控制器可有多个路由，每个路由可以执行不同的操作。
- **Controller 是“声明路由规则”的地方，而不是执行路由匹配的引擎，真正干活的是底层的：Express（默认）或 Fastify**
  > Controller 在“启动阶段”把路由注册到 Router 里

## Redirection

服务器告诉客户端“去另一个地址”

2. 本质是返回一个 3xx 状态码 + Location header
   HTTP/1.1 302 Found
   Location: https://example.com

```js

  @Get('go')

  @Redirect('https://www.baidu.com/', 302)
  redirect() {
    // 会覆盖掉@Redirect装饰器
    // 动态重定向
    return {
      url: 'https://www.bilibili.com/',
      statusCode: 302,
    };
  }
```

2. 使用场景：OAuth 登录、URL 迁移、传统 SSR、前端后分离的单页面项目建议返回数据让前端跳转

```js

// 后端：
返回 302 + Location
```

3. 重定向理解

```js
301  永久  ❌可能改方法（POST→GET）
302  临时  ❌可能改方法
307  临时  ✅不改方法
308  永久  ✅不改方法

一、临时重定向 vs 永久重定向
1️⃣ 临时重定向（302 / 307）
👉 含义：这个跳转只是“暂时的”，原 URL 以后还会继续用
👉 特点：浏览器不会强缓存，搜索引擎不会更新索引
👉 场景：
登录跳转
AB 测试
临时维护页

2️⃣ 永久重定向（301 / 308）
👉 含义：资源地址“彻底变了”，以后都用新地址
👉 特点：浏览器可能缓存，搜索引擎会更新索引（SEO）
👉 场景：
域名迁移
API 路径升级（接口）
URL 规范化

二、会不会改变请求方法
301 / 302 的问题：
POST /login
→ 302
→ 浏览器可能变成
GET /new-url
👉 问题：请求体丢失、语义变了

307 / 308 的改进
POST /login
→ 307
→ 仍然是
POST /new-url
保证：方法不变、body 不丢
```

4. 案列

```js

  // 🔴 302：可能改变方法
  @Post('test-302')
  @Redirect('/target', 302)
  test302(@Body() body) {
    console.log('👉 进入 test-302，方法是 POST，body:', body);
  }

  // 🟢 307：不会改变方法
  @Post('test-307')
  @Redirect('/target', 307)
  test307(@Body() body) {
    console.log('👉 进入 test-307，方法是 POST，body:', body);
  }

  // 🎯 最终接收请求的接口
  @Get('target')
  getTarget(@Req() req) {
    console.log('🎯 最终到达 target（GET）');
    return {
      method: 'GET',
      message: '你被重定向成 GET 了',
    };
  }

  @Post('target')
  postTarget(@Req() req, @Body() body) {
    console.log('🎯 最终到达 target（POST），body:', body);
    return {
      method: 'POST',
      body,
    };
  }
```

## State sharing

1.  前端与后端本质区别：
    | 前端 | 后端|  
     |-------|-------|
    | 单用户 | 多用户并发 |  
     | state 可全局共享 | state 必须隔离 |  
     | 内存安全 | 容易数据串 |

> state 越集中越好（Redux）；state 默认不共享（无状态）；

2. 默认所有 Controller / Service 是单例（Singleton） → 多个请求会共享同一个实例里的状态

> 可以共享 ≠ 应该共享

```js
@Injectable()
export class CounterService {
  count = 0;

  add() {
    this.count++;
    return this.count;
  }
}

用户A → count = 1
用户B → count = 2
这就是“状态被共享了”
```

3. 哪些“状态”可以共享
   > 无特点: 用户信息 / 无请求上下文
   > 只要数据“不跟某个具体用户/请求绑定” → 可以共享
   > 大家读同一份数据 → 没问题
   > 判断:“这个数据，两个不同用户同时访问，会不会冲突？ 会 → ❌ 不能共享

- 配置（config）
- 缓存（cache）：为了减少重复计算 / 查询，把结果临时存起来
  > 数据库查询缓存：例如关于用户等同一个 id → 所有人查到的是同一份数据
  > 配置 / 字典缓存：例如项目中有些静态数据的配置（管理员类型等）所有人都一样 → 可以共享
  > 第三方接口缓存：汇率、天气、配置接口等，同一时间返回一样 → 可以共享
- 数据库连接
- 工具函数
- 常量数据：运行过程中不会变化的数据
  > 枚举 / 状态
  > 固定配置
  > 静态映射表

```js
enum UserRole {
  ADMIN = 'admin',
  USER = 'user'
}
const MAX_RETRY = 3;
const statusMap = {
  0: 'pending',
  1: 'success',
  2: 'failed'
};
```

4. 不能共享
   > 只要数据“跟用户或请求有关” → 不能共享

- 当前用户信息
- 请求数据
- 登录状态
- 临时变量（和请求相关）

5. 三种处理方式

```js
1️⃣ Singleton（默认）：全局共享->用于 缓存 / 配置
@Injectable()
export class CounterService {
  count = 0;

  add() {
    this.count++;
    return this.count;
  }
}
用户A → add() → 1
用户B → add() → 2
用户C → add() → 3

2️⃣ Request Scope： 每个请求一个实例（隔离）
@Injectable({ scope: Scope.REQUEST })
export class CounterService {
  count = 0;

  add() {
    this.count++;
    return this.count;
  }
}
用户A → add() → 1
用户B → add() → 1
用户C → add() → 1

3️⃣ 参数传递（最推荐）：最安全、最清晰、无副作用
// 客户端传过来
service.method(userId)
```

## Asynchronicity

2. 什么是 Asynchronicity: 异步(不阻塞线程的执行方式)；

> 底层是 Node.js：单线程 + 事件循环。
> node的世家循环：https://www.yuque.com/zhuanjiao-lml37/ec17ie/so3wp2eti8tdbszd#yg75H
> 服务直接卡死: while(true) {} 2. 总结：NestJS 支持同步、Promise 和 Observable 三种返回方式，**并会自动处理异步结果**。由于 Node.js 是单线程事件循环模型，**所有 IO 操作必须使用异步方式，以避免阻塞服务。**

```js
1️⃣ Promise（最常用）
@Get()
async findAll() {
  return await this.userService.findAll();
}
👉 本质：返回 Promise --> Nest 自动 await

2️⃣ 普通值（同步）
@Get()
findAll() {
  return [1, 2, 3];
}

👉 Nest 会自动包装成 Promise

3️⃣ Observable（进阶）
@Get()
findAll(): Observable<any> {
  return of([1, 2, 3]);
}

👉 用于：流式数据、WebSocket、RxJS 场景（用“数据流（stream：可以持续返回多个结果（流））”来处理异步）

@Get()
async findAll() {
  return data;
}
Nest 内部其实做了👇
1. 检测返回值类型
2. 如果是 Promise → await
3. 如果是 Observable → subscribe
4. 最终统一返回响应
```

3. 常见坑（非常重要）

```js
坑1：忘记 await
@Get()
getUsers() {
  return this.userService.findAll(); // Promise
  // return await this.userService.findAll(); //推荐
}
虽然 Nest 能处理，但：错误捕获可能出问题、代码不清晰

坑2：写阻塞代码
@Get()
heavyTask() {
  for (let i = 0; i < 1e9; i++) {}
  return 'done';
}

坑3：串行 await
await a();
await b();
// await Promise.all([a(), b()]); //推荐

坑4：错误处理不当

try {
  await xxx();
} catch(e) {
  // 忘记 throw： 错误被吞掉 ❌
}

```

## Request payloads

Request payload = 客户端发送到服务器的数据（通常是 body）。常见来源：

- POST / PUT / PATCH 请求的 body
- JSON / form-data / x-www-form-urlencoded

1. 使用DTO
   DTO 是一个对象，它指定了数据在网络上的传输方式（**用来约束数据结构**）。我们可以使用TypeScript接口或简单的类来定义 DTO 模式。
   > 不会校验，必须加 ValidationPipe,后面学

- 但是，**我们建议在这里使用类**。为什么呢？类是 JavaScript ES6 标准的一部分，因此它们在编译后的 JavaScript 中作为实际实体保持不变。相比之下，**TypeScript 接口在转译过程中会被移除，这意味着 Nest 无法在运行时引用它们**。这一点很重要，因为像管道这样的功能依赖于在运行时访问变量的元类型，而这只有通过类才能实现。
  > TypeScript只存在于编译阶段（类型检查），运行时完全消失
  > next 在编译后还需要ValidationPipe（校验）、Transformation（自动转换）、Reflect Metadata（元数据反射），都需要实际的代码，这点只有Clas才能做到

```js
export class CreateUserDto {
  name: string;
  age: number;
}
@Post('adduser')
  createUser(@Body() dto: CreateUserDto) {
    console.log('dto->', dto);
    return {
      ...dto,
    };
  }
```

2. 理解大概的生命周期

```text

   ↓
body-parser（解析 JSON、参数处理）
   ↓
Middleware（可选）：日志、CORS、token 预处理
   ↓

   ↓

   ↓

   ↓

   ↓

   ↓
HTTP Response

它不会：
```

## Library-specific approach

1. 返回响应
   在 NestJS 中，库特定方式（Library-specific Approach）是指根据不同的 HTTP 库（如 Express 或 Fastify）来调整 NestJS 控制器的使用方式。NestJS 允许你选择使用不同的 HTTP 框架（如 Express 或 Fastify）来处理 HTTP 请求，尽管这些框架的底层实现不同，**NestJS 的控制器和依赖注入系统保持一致。**

- Fastify：一个高性能的 HTTP 框架，主要用于需要低延迟和高吞吐量的应用。
- Express：默认的 HTTP 框架，广泛用于 Node.js 应用开发。
  > NestJS 允许你选择底层的 HTTP 框架（默认是 Express），控制器方法用来处理请求并返回响应。

2. 直接使用底层 HTTP 库的响应对象：可以通过 @Res() 装饰器来注入该响应对象。这样你就可以手动控制响应的各个方面，比如设置响应头、状态码等

```js
@Get('cats')
  findAll(@Res() response: Response) {
    // 你可以使用响应对象来自定义响应
    return response
      .status(401)
      .json({ message: 'This is the custom response handling with @Res()' });
  }
```

3. 缺点：

- @Res() 会绕过拦截器：使用 @Res() 时，NestJS 不会再通过拦截器格式化响应。
- 解决方法：
  - 手动在每个使用 @Res() 的方法中保持统一响应格式。
  - 在拦截器中检查是否使用了 @Res()，如果使用了则跳过格式化。
  - 增加测试的难度

```js
import { Injectable, ExecutionContext } from '@nestjs/common';
import { CallHandler, NestInterceptor } from '@nestjs/common';
import { Observable } from 'rxjs';
import { map } from 'rxjs/operators';
import { Response } from 'express';

@Injectable()
export class ResponseFormatInterceptor<T> implements NestInterceptor<T> {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const request = context.switchToHttp().getRequest();
    const response = context.switchToHttp().getResponse<Response>();

    // 检查是否使用了 @Res()（即，直接使用了底层响应对象）
    if (response) {
      // 如果使用了 @Res()，直接返回，不做拦截
      return next.handle();
    }

    return next.handle().pipe(
      map(data => ({
        status: 'success',
        data,
        message: 'Request was successful',
      })),
    );
  }
}
```

# Provider

- Provider 是一个用于封装依赖的类、值、工厂方法或实例。它是 NestJS 提供的一种机制，**用于管理和注入服务、模块中的其他类等依赖项**。such as services, repositories, factories, and helpers。

1. Provider 是依赖注入（DI）系统的基础：

- 服务（Service）：通常我们使用 Provider 来定义服务类（例如：CatsService）。
- 值：除了服务类，Provider 还可以是常量或对象。
- 工厂：Provider 还可以是工厂函数，这样我们可以按需创建对象。
- 类：任何带有 @Injectable() 装饰器的类都可以成为一个 Provider。

## Service

最常见的 Provider 类型就是 服务类（Service）。

1. 创建一个 Service 类：通过 @Injectable() 装饰器，NestJS 会自动将 Service 类 注册为一个 Provider。
   > 支持es6的类语法

```js
import { Injectable } from "@nestjs/common";

@Injectable()
export class CatsService {
  getCats() {
    return ["Milo", "Bella", "Charlie"];
  }
}
```

2. 注册 Service 到 Module 中：

```js
import { Module } from "@nestjs/common";
import { CatsService } from "./cats.service";

@Module({
  providers: [CatsService],
})
export class CatsModule {}
```

3. 注入 Provider 到 Controller：constructor中使用 private 关键字在构造函数中声明和初始化 catsService 属性
   > 在类内部可访问，但外部无法直接修改它

```js
import { Controller, Get } from '@nestjs/common';
import { CatsService } from './cats.service';

@Controller('cats')
export class CatsController {
  constructor(private readonly catsService: CatsService) {}

  @Get()
  findAll() {
    return this.catsService.getCats();
  }
}
```

## Dependency injection

1. 什么是依赖注入 (DI)?
   依赖注入是一种设计模式，IoC的一种实现方式，用来处理类之间的依赖关系。在 NestJS 中，**依赖注入允许框架自动将所需的服务或组件注入到类中，而无需手动管理它们的实例**。
   > 允许你将对象的创建过程交给外部容器（如 NestJS 的 IoC 容器）来管理，而不是让对象自己创建它的依赖
2. NestJS 中的依赖注入：
   在 NestJS 中，依赖注入是通过 providers（提供者）机制实现的。一个提供者可以是一个服务、工厂、或者值（例如数据库连接）。在@Module()注册，通过在类的构造函数中使用 @Injectable() 装饰器，NestJS 会自动为你处理依赖注入。

## Scopes

> 先理解常见的作用域，后面再看https://docs.nestjs.com/fundamentals/injection-scopes
> 作用域 决定了对象实例的生命周期，决定了服务是作为单例使用，还是每次请求都创建一个新的实例。NestJS 提供了几种不同的 作用域 (Scope) 选项，你可以通过设置作用域来控制对象实例的生命周期。

1. 默认作用域：单例 (Singleton)

- 默认情况下，所有的服务（provider）都是单例的
- 应用程序启动后，每个提供者只会实例化一次，并且在整个应用生命周期内共享同一个实例。
- 这种模式非常**适合状态无关的服务**，例如数据库连接池、日志记录服务等。

```js
@Injectable()
export class CatsService {
  // 这个服务会在整个应用中只有一个实例
}
```

2. 请求作用域：REQUEST

- 对于每个传入的请求，都会创建一个新的提供程序实例。请求处理完成后，该实例会被垃圾回收。
- 在 请求作用域 中，NestJS 会确保每个 HTTP 请求都有自己的 CatsService 实例。
- 这种作用域适用于需要针对每个请求独立处理的场景、依赖请求数据的服务，例如处理请求级别的缓存或者会话信息、 每个请求的独立缓存等。
  > 为每一个 HTTP 请求创建一个新的实例。这意味着每次处理一个请求时，都会创建一个新的 Service 实例，而这个实例只会在当前请求期间存活。

```js
@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  // 这个服务会为每一个 HTTP 请求创建一个新实例
}
```

3. TRANSIENT ：每个事务创建一个实例
   你可能需要为每一个数据库事务创建一个新的实例。NestJS 支持 事务作用域，但这通常需要与事务管理工具（如 TypeORM、Sequelize 等）结合使用。

- 务作用域 每次注入时都会创建一个新的实例。它适用于一些要求每次都要有独立实例的场景，例如每个数据库事务都有独立的操作。
- 适用于数据库事务、复杂的业务逻辑需要每个执行单元有独立状态等场景。
  > 每次注入时都会创建一个新的实例，它的生命周期非常短暂，仅限于当前的注入上下文。

```js
@Injectable({ scope: Scope.TRANSIENT })
export class CatsService {
  // 这个服务每次注入时都会创建一个新实例
}
```

## Custom providers

1. 类:常见的 @Injectable() 类

- 用于业务逻辑

```js
@Injectable()
export class CatsService {
  getCats() {
    return ["Milo", "Bella"];
  }
}
```

> 学到这里 https://docs.nestjs.com/fundamentals/custom-providers#standard-providers

2. 值 Provider
   值 Provider 是通过提供一个常量或对象实例来注册的：

- 在这种情况下，'CATS' 就是一个 token，NestJS 依赖注入系统将会注入 cats 数组。
- 用于将常量或配置项作为依赖注入时

```js
import { Module } from "@nestjs/common";
const cats = ["Milo", "Bella"];
@Module({
  providers: [
    {
      provide: "CATS",
      useValue: cats, // 提供一个值
    },
  ],
})
export class CatsModule {}
```

3. 工厂 Provider

- 当你的 Provider 实例化过程复杂，或者需要动态生成时，使用工厂函数。

```js
@Module({
  providers: [
    {
      provide: "CATS",
      useFactory: () => {
        return ["Milo", "Bella"]; // 动态生成值
      },
    },
  ],
})
export class CatsModule {}
```

4. 异步 Provider:
   如果你需要在启动时执行异步操作（如从数据库加载配置），你可以使用异步 Provider：
   useFactory 返回一个 Promise，它会在启动时等待异步操作完成，然后将值注入到依赖注入系统中。

- 当需要从数据库或外部 API 获取数据时，使用异步 Provider。

```js
@Module({
  providers: [
    {
      provide: "ASYNC_CONFIG",
      useFactory: async () => {
        const config = await getConfigFromDatabase();
        return config;
      },
    },
  ],
})
export class CatsModule {}
```

5. 自定义 Provider Token
   有时候，你可能需要根据不同的场景来创建多个 Provider。NestJS 允许你使用自定义的 token 来区分不同的 Provider。

```js
const CatsService = Symbol("CatsService");

@Module({
  providers: [
    {
      provide: CatsService,
      useClass: CatsService, // 使用类
    },
  ],
})
export class CatsModule {}
```

6. Scope Provider（作用域 Provider）
   NestJS 支持 作用域（Scope） Provider，这意味着你可以控制 Provider 的实例化生命周期。例如，某些 Provider 可能在每个请求中都需要一个新的实例（Request Scoped）。
   CatsService 的实例会在每个请求中创建一个新的实例。

```js
@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  getCats() {
    return ["Milo", "Bella"];
  }
}
```

# 什么时候选择哪种框架：Express与Fastify

1. Express：

- Express 是默认的 HTTP 框架，适合大多数 Web 应用。
- 对性能要求不高的应用非常适合，且社区资源丰富，支持也更为广泛。

2. Fastify：

- 如果你的应用需要处理大量请求，或者对响应时间有严格要求，Fastify 会是一个不错的选择。
- Fastify 适合微服务架构、高并发请求的应用，尤其是在性能至上的场景下。
  无论你选择 Express 还是 Fastify，NestJS 的控制器使用方式保持一致，只是底层平台的不同会影响应用的 性能特性 和 配置方式。

3. 坑点：常见问题
   平台特定配置问题：从 Express 切换到 Fastify 时，确保应用配置正确，否则可能会出现启动错误或者行为不一致。

- 功能差异：Fastify 提供了很多性能优化的功能，这些功能可能和 Express 不完全一致。虽然 NestJS 的大部分框架在两者之间相同，但某些特性（如日志记录、验证）在平台之间可能表现不同。
- 第三方库的兼容性：某些第三方库或者中间件可能不完全兼容 Fastify，因为 Fastify 是一个与 - Express 不同的 HTTP 框架。如果你决定从 Express 切换到 Fastify，可能需要检查并确保所有的第三方库与 Fastify 兼容。

- 基于默认 Express 适配器）

```text

HTTP Request
   ↓
body-parser（解析 JSON、参数解析）
   ↓
Middleware（可选）：日志、CORS、token 预处理
   ↓
Nest Router（匹配 Controller + 方法）
   ↓
Guard（权限判断）：决定“这个请求能不能进 Controller”、拦截请求
   ↓
Interceptor（前置）：包装请求/响应、统一日志、性能监控
   ↓
Pipe（🔥就在这里处理 @Body/@Param）
   ↓
Controller Handler（方法执行）：真正执行业务逻辑的地方
   ↓
Interceptor（后置）：统一返回格式、响应加密、日志记录
   ↓
Exception Filter（异常处理）
   ↓
HTTP Response
```
