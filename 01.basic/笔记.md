# nextjs是什么

Nest **是带完整构建规范后端应用的 Node.js 框架**。使创建可重用的逻辑部分成为可能，开发人员可以在多种不同类型的应用程序中利用这些逻辑部分。 从技术上讲，一旦创建了适配器，Nest 便可以使用任何 Node HTTP 框架。
目前支持两个 HTTP 平台：express 和 fastify。 您可以根据您的需求选择最适合平台。

- platform-express： Express 是一个著名的、极简的、专为 node 开发的 web 框架。它久经考验、适用于生产环境的软件库，并且拥有大量的社区资源。默认情况下使用 @nestjs/platform-express 软件包。许多用户对 Express 都很满意，并且无需采取任何操作即可启用它。
- platform-fastify： Fastify 是一个高性能且低开销的框架，高度专注于提供最高的效率和速度。

# Controllers

- 请求入口（路由处理层）:HTTP Request → Controller → Service → Response
- Controller 不负责业务逻辑，处理应用程序的特定请求。只负责“转发 + 组织输入输出”，一个控制器可有多个路由，每个路由可以执行不同的操作

## 路由路径、动态路由

路径拼接：由控制器中声明的（可选）前缀与请求方法装饰器中指定的路径组合而成

## Route wildcards 路由通配符

**推荐 :param而不是**

- \*拿不到具体路径
- :param\* 更可控
- 兼容性更好（尤其是 Fastify）： Fastify 对\*支持更严格、官方更推荐 :param\*

```js
@Get('*')
handleAll() {}
// 匹配当前 Controller 下所有路径


 @Get('files/*path')
  getFile(@Param('path') path: string[]) {
    return path;
  }
// /files/path/1/2123/123/1  ->
// [
//     "path",
//     "1",
//     "2123",
//     "123",
//     "1"
// ]
```

## Redirection

服务器告诉客户端“去另一个地址”

2. 本质是返回一个 3xx 状态码 + Location header
   HTTP/1.1 302 Found
   Location: https://example.com

```js

  @Get('go')
  // 默认是 302
  @Redirect('https://www.baidu.com/', 302)
  redirect() {
    // 会覆盖掉@Redirect装饰器
    // 动态重定向
    return {
      url: 'https://www.bilibili.com/',
      statusCode: 302,
    };
  }
```

2. 使用场景：OAuth 登录、URL 迁移、传统 SSR、前端后分离的单页面项目建议返回数据让前端跳转

```js
// 前端：一个get请求
window.location.href = '/home'
// 后端：
返回 302 + Location
```

3. 重定向理解

```js
301  永久  ❌可能改方法（POST→GET）
302  临时  ❌可能改方法
307  临时  ✅不改方法
308  永久  ✅不改方法

一、临时重定向 vs 永久重定向
1️⃣ 临时重定向（302 / 307）
👉 含义：这个跳转只是“暂时的”，原 URL 以后还会继续用
👉 特点：浏览器不会强缓存，搜索引擎不会更新索引
👉 场景：
登录跳转
AB 测试
临时维护页

2️⃣ 永久重定向（301 / 308）
👉 含义：资源地址“彻底变了”，以后都用新地址
👉 特点：浏览器可能缓存，搜索引擎会更新索引（SEO）
👉 场景：
域名迁移
API 路径升级（接口）
URL 规范化

二、会不会改变请求方法
301 / 302 的问题：
POST /login
→ 302
→ 浏览器可能变成
GET /new-url
👉 问题：请求体丢失、语义变了

307 / 308 的改进
POST /login
→ 307
→ 仍然是
POST /new-url
保证：方法不变、body 不丢
```

4. 案列

```js

  // 🔴 302：可能改变方法
  @Post('test-302')
  @Redirect('/target', 302)
  test302(@Body() body) {
    console.log('👉 进入 test-302，方法是 POST，body:', body);
  }

  // 🟢 307：不会改变方法
  @Post('test-307')
  @Redirect('/target', 307)
  test307(@Body() body) {
    console.log('👉 进入 test-307，方法是 POST，body:', body);
  }

  // 🎯 最终接收请求的接口
  @Get('target')
  getTarget(@Req() req) {
    console.log('🎯 最终到达 target（GET）');
    return {
      method: 'GET',
      message: '你被重定向成 GET 了',
    };
  }

  @Post('target')
  postTarget(@Req() req, @Body() body) {
    console.log('🎯 最终到达 target（POST），body:', body);
    return {
      method: 'POST',
      body,
    };
  }
```

## Subdomain Routing

> Express 才有，Fastify 没有

1. 根据不同子域名，进入不同的后端逻辑。不同业务 进入不同入口

- 不同子域 = 不同业务入口
- 适合大系统拆分
- 常配合 Nginx 使用

```js

@Controller({ host: ':account.example.com' })
export class AccountController {
  @Get()
  getInfo(@HostParam('account') account: string) {
    return account;
  }
}

如果没有子域路由，你会这样做：
example.com/api
example.com/admin
example.com/user
问题：
路由混在一起
权限边界不清晰
系统耦合高

服务器根据 Host 分流：
api.example.com   → API模块
admin.example.com → 管理模块
```

2. 使用场景：

```js
微服务拆分
SaaS 系统（最典型：每个子域 = 一个租户）
后台系统隔离
多业务系统（不同业务完全独立）
shop.example.com
blog.example.com

国际化 / 地区系统
us.example.com
cn.example.com
```

## State sharing

1.  前端与后端本质区别：
    | 前端 | 后端|  
     |-------|-------|
    | 单用户 | 多用户并发 |  
     | state 可全局共享 | state 必须隔离 |  
     | 内存安全 | 容易数据串 |

> state 越集中越好（Redux）；state 默认不共享（无状态）；

2. 默认所有 Controller / Service 是单例（Singleton） → 多个请求会共享同一个实例里的状态

> 可以共享 ≠ 应该共享

```js
@Injectable()
export class CounterService {
  count = 0;

  add() {
    this.count++;
    return this.count;
  }
}

用户A → count = 1
用户B → count = 2
这就是“状态被共享了”
```

3. 哪些“状态”可以共享
   > 无特点: 用户信息 / 无请求上下文
   > 只要数据“不跟某个具体用户/请求绑定” → 可以共享
   > 大家读同一份数据 → 没问题
   > 判断:“这个数据，两个不同用户同时访问，会不会冲突？ 会 → ❌ 不能共享

- 配置（config）
- 缓存（cache）：为了减少重复计算 / 查询，把结果临时存起来
  > 数据库查询缓存：例如关于用户等同一个 id → 所有人查到的是同一份数据
  > 配置 / 字典缓存：例如项目中有些静态数据的配置（管理员类型等）所有人都一样 → 可以共享
  > 第三方接口缓存：汇率、天气、配置接口等，同一时间返回一样 → 可以共享
- 数据库连接
- 工具函数
- 常量数据：运行过程中不会变化的数据
  > 枚举 / 状态
  > 固定配置
  > 静态映射表

```js
enum UserRole {
  ADMIN = 'admin',
  USER = 'user'
}
const MAX_RETRY = 3;
const statusMap = {
  0: 'pending',
  1: 'success',
  2: 'failed'
};
```

4. 不能共享
   > 只要数据“跟用户或请求有关” → 不能共享

- 当前用户信息
- 请求数据
- 登录状态
- 临时变量（和请求相关）

5. 三种处理方式

```js
1️⃣ Singleton（默认）：全局共享->用于 缓存 / 配置
@Injectable()
export class CounterService {
  count = 0;

  add() {
    this.count++;
    return this.count;
  }
}
用户A → add() → 1
用户B → add() → 2
用户C → add() → 3

2️⃣ Request Scope： 每个请求一个实例（隔离）
@Injectable({ scope: Scope.REQUEST })
export class CounterService {
  count = 0;

  add() {
    this.count++;
    return this.count;
  }
}
用户A → add() → 1
用户B → add() → 1
用户C → add() → 1

3️⃣ 参数传递（最推荐）：最安全、最清晰、无副作用
// 客户端传过来
service.method(userId)
```

## Asynchronicity

2. 什么是 Asynchronicity: 异步(不阻塞线程的执行方式)；

> 底层是 Node.js：单线程 + 事件循环。
> node的世家循环：https://www.yuque.com/zhuanjiao-lml37/ec17ie/so3wp2eti8tdbszd#yg75H
> 服务直接卡死: while(true) {} 2. 总结：NestJS 支持同步、Promise 和 Observable 三种返回方式，**并会自动处理异步结果**。由于 Node.js 是单线程事件循环模型，**所有 IO 操作必须使用异步方式，以避免阻塞服务。**

```js
1️⃣ Promise（最常用）
@Get()
async findAll() {
  return await this.userService.findAll();
}
👉 本质：返回 Promise --> Nest 自动 await

2️⃣ 普通值（同步）
@Get()
findAll() {
  return [1, 2, 3];
}

👉 Nest 会自动包装成 Promise

3️⃣ Observable（进阶）
@Get()
findAll(): Observable<any> {
  return of([1, 2, 3]);
}

👉 用于：流式数据、WebSocket、RxJS 场景（用“数据流（stream：可以持续返回多个结果（流））”来处理异步）

@Get()
async findAll() {
  return data;
}
Nest 内部其实做了👇
1. 检测返回值类型
2. 如果是 Promise → await
3. 如果是 Observable → subscribe
4. 最终统一返回响应
```

3. 常见坑（非常重要）

```js
坑1：忘记 await
@Get()
getUsers() {
  return this.userService.findAll(); // Promise
  // return await this.userService.findAll(); //推荐
}
虽然 Nest 能处理，但：错误捕获可能出问题、代码不清晰

坑2：写阻塞代码
@Get()
heavyTask() {
  for (let i = 0; i < 1e9; i++) {}
  return 'done';
}

坑3：串行 await
await a();
await b();
// await Promise.all([a(), b()]); //推荐

坑4：错误处理不当

try {
  await xxx();
} catch(e) {
  // 忘记 throw： 错误被吞掉 ❌
}

```
