# 逆变、协变、双向协变和不变

在 TypeScript 中，类型系统支持逆变、协变、双向协变和不变等概念。这些概念是理解 TypeScript 类型系统的基础，尤其在函数参数类型和返回值类型的兼容性问题上起着重要作用。

## 类型变换概述

类型变换描述了当类型 A 和类型 B 存在继承关系时，是否可以在使用 A 的地方使用 B。

![20251229120725](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251229120725.png)

常见的类型变换包括：

1. 协变（Covariance）：如果 A 是 B 的子类型，那么 `F<A>` 也是 `F<B>` 的子类型。`F<A>` 可以赋值给 `F<B>`。协变通常用于函数的返回值类型。

2. 逆变（Contravariance）：如果 A 是 B 的子类型，那么 `F<B>` 是 `F<A>` 的子类型。`F<B>` 可以赋值给 `F<A>`。逆变通常用于函数的参数类型。

3. 双向协变（Bivariance）：允许函数的参数类型同时表现出协变和逆变的特性。虽然理论上不安全，但它为保持代码兼容性提供了方便。

4. 不变（Invariance）：`F<A>` 既不是 `F<B>` 的子类型，也不是其超类型。`F<A>` 和 `F<B>` 是完全不兼容的类型。

## 协变

协变允许子类型被视为其父类型。如果类型 A 是类型 B 的子类型，那么在使用泛型、函数返回值、数组等结构时，`T<A>` 也可以被视为 `T<B>` 的子类型。

![20251229133415](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251229133415.png)

图中展示了协变的核心概念：当 `Dog` 是 `Animal` 的子类型时，`Box<Dog>`、`Dog[]` 和 `() => Dog` 都可以安全地赋值给对应的父类型。这是因为协变保证了"更具体"的类型可以替代"更一般"的类型，符合里氏替换原则。

```ts
class Animal {}
class Dog extends Animal {}

// 泛型协变
interface Box<T> {
  value: T;
}
let animalBox: Box<Animal>;
let dogBox: Box<Dog> = { value: new Dog() };
animalBox = dogBox; // 合法：Box<Dog> 可以赋值给 Box<Animal>

// 函数返回值协变
type AnimalFactory = () => Animal;
type DogFactory = () => Dog;
let createAnimal: AnimalFactory;
let createDog: DogFactory = () => new Dog();
createAnimal = createDog; // 合法：返回 Dog 的函数可以赋值给返回 Animal 的函数

// 数组协变
let dogs: Dog[] = [new Dog()];
let animals: Animal[];
animals = dogs; // 合法：Dog[] 可以赋值给 Animal[]
```

### 协变的潜在问题

虽然协变提供了类型灵活性，但在可变数据结构（如数组）中，它可能导致运行时类型不一致的问题。由于数组是协变的，`Dog[]` 可以赋值给 `Animal[]`，但这样会允许将其他 `Animal` 的子类型（如 `Cat`）推入数组中，破坏了原本只包含 `Dog` 的假设。

```ts
class Animal {}
class Dog extends Animal {}
class Cat extends Animal {}

let animals: Animal[] = [new Dog()];
animals.push(new Cat()); // 合法，但可能导致运行时类型不一致
```

如果后续代码假设数组中只有 `Dog` 实例并调用 `Dog` 特有的方法，就会导致运行时错误。这就是为什么只读容器可以安全地使用协变，而可读写容器应该使用不变的原因。

## 逆变

逆变是 TypeScript 类型系统中与协变相对的概念。在逆变关系中，如果类型 A 是类型 B 的子类型，那么包含 B 的复合类型反而是包含 A 的复合类型的子类型。这种关系主要体现在函数参数类型上。

![20251229144742](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251229144742.png)

图中展示了逆变的特性：虽然在继承关系中 `Dog` 是 `Animal` 的子类型，但在函数参数类型中，关系是相反的。接受 `Animal` 参数的函数类型 `Handler[Animal]` 可以赋值给接受 `Dog` 参数的函数类型 `Handler[Dog]`，这就是逆变。

这种"反向"关系是类型安全的。能够处理所有 `Animal` 的函数，自然也能处理 `Dog`（因为 `Dog` 是 `Animal` 的一种）。但反过来则不安全，因为处理 `Dog` 的函数可能会调用 `Dog` 特有的方法（如 `bark()`），而这些方法在 `Animal` 上不存在。

```ts
class Animal {
  feed(): void {
    console.log("Feeding an animal");
  }
}

class Dog extends Animal {
  bark(): void {
    console.log("Woof!");
  }
}

type AnimalHandler = (animal: Animal) => void;
type DogHandler = (dog: Dog) => void;

let handleAnimal: AnimalHandler = (animal: Animal) => {
  animal.feed();
};

// 逆变允许将处理父类型的函数赋值给处理子类型的函数
let dogHandler: DogHandler;
dogHandler = handleAnimal; // 合法，这是逆变

const myDog = new Dog();
dogHandler(myDog); // 安全
```

`handleAnimal` 只会使用 `Animal` 类型上存在的方法（如 `feed()`），而这些方法在 `Dog` 类型上也一定存在，因此这种赋值是类型安全的。

### 事件处理中的逆变

事件处理是逆变的典型应用场景：

```ts
type GenericEventHandler = (event: Event) => void;
type MouseEventHandler = (event: MouseEvent) => void;

const handleAnyEvent: GenericEventHandler = (event) => {
  console.log("Event occurred:", event.type);
};

const handleMouseEvent: MouseEventHandler = handleAnyEvent; // 合法
```

在 React 和 DOM 事件处理中，逆变允许我们将通用的 `Event` 处理器赋值给更具体的 `MouseEvent` 处理器，这在实际开发中非常有用。

## 双向协变

双向协变是 TypeScript 类型系统中的一个特殊行为，只有在禁用 `strictFunctionTypes` 选项时才会出现。在这种模式下，函数参数类型允许同时具有协变和逆变的特性，即两个函数类型之间可以相互赋值，无论其参数类型的继承关系如何。

![20251229144813](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251229144813.png)

图中展示了双向协变的特性：`Handler[Dog]` 和 `Handler[Animal]` 可以相互赋值，这种双向的类型兼容性就是双向协变。

TypeScript 早期为了兼容 JavaScript 的灵活性，允许函数参数类型的双向协变。但这会牺牲类型安全性，因此现代 TypeScript 项目通常会启用 `strict` 模式（包含 `strictFunctionTypes: true`），禁用双向协变。

双向协变的行为受到 `strictFunctionTypes` 编译选项的影响：

- `strictFunctionTypes: true`（默认，在 `strict` 模式下）：强制函数参数类型为逆变，不允许双向协变
- `strictFunctionTypes: false`：允许函数参数类型的双向协变

```ts
class Animal {
  feed(): void {}
}

class Dog extends Animal {
  bark(): void {}
}

type DogHandler = (dog: Dog) => void;
type AnimalHandler = (animal: Animal) => void;

let handleDog: DogHandler = (dog: Dog) => {
  dog.feed();
  dog.bark();
};

let handleAnimal: AnimalHandler = (animal: Animal) => {
  animal.feed();
};

// 只有在 strictFunctionTypes: false 时，以下两种赋值才都合法
handleAnimal = handleDog; // 协变方向（危险！）
handleDog = handleAnimal; // 逆变方向（安全）
```

协变方向的赋值（`handleAnimal = handleDog`）是危险的，可能导致运行时错误：

```ts
class Cat extends Animal {
  meow(): void {}
}

let dogOnlyHandler: DogHandler = (dog: Dog) => {
  dog.bark(); // 使用了 Dog 特有的方法
};

// 协变方向的赋值：危险！
let generalHandler: AnimalHandler = dogOnlyHandler; // 在 strictFunctionTypes: false 时合法

const cat = new Cat();
generalHandler(cat); // 运行时错误：cat 没有 bark 方法
```

问题发生的原因：`dogOnlyHandler` 期望接收 `Dog` 并调用 `bark` 方法，但被赋值给 `generalHandler` 后，可以接受任何 `Animal`（包括 `Cat`），导致运行时调用不存在的方法。

因此，强烈建议始终保持 `strictFunctionTypes: true`（或启用 `strict` 模式）：

```json
{
  "compilerOptions": {
    "strict": true
  }
}
```

## 不变

不变是类型系统中最严格的型变规则，它要求类型必须精确匹配，不允许任何形式的子类型或父类型替换。

![20251229151607](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251229151607.png)

图中展示了不变的特性：虽然 `Dog` 是 `Animal` 的子类型，但 `Container[Dog]` 和 `Container[Animal]` 之间不能相互赋值，它们是完全不兼容的类型。

```ts
class Animal {
  name: string;
  constructor(name: string) {
    this.name = name;
  }
}

class Dog extends Animal {
  breed: string;
  constructor(name: string, breed: string) {
    super(name);
    this.breed = breed;
  }
}

// 可读写的容器应该是不变的
class Container<T> {
  private _value: T;

  constructor(value: T) {
    this._value = value;
  }

  getValue(): T {
    return this._value;
  }

  setValue(value: T): void {
    this._value = value;
  }
}

const animalContainer = new Container<Animal>(new Animal("Animal"));
const dogContainer = new Container<Dog>(new Dog("Rex", "Shepherd"));

// 以下赋值都是不合法的
// animalContainer = dogContainer; // 错误
// dogContainer = animalContainer; // 错误
```

不变的必要性在于保证可读写容器的类型安全。如果可读写容器是协变的，会导致严重的类型安全问题：

```ts
class MutableBox<T> {
  content: T;

  setContent(value: T): void {
    this.content = value;
  }

  getContent(): T {
    return this.content;
  }
}

const dogBox = new MutableBox<Dog>();
// 如果允许协变
const animalBox: MutableBox<Animal> = dogBox; // 假设合法

animalBox.setContent(new Animal()); // 危险：将 Animal 放入期望 Dog 的容器
const myDog: Dog = dogBox.getContent(); // 运行时错误：myDog.breed 不存在
```

如果 `MutableBox<T>` 是协变的，我们可以通过 `animalBox` 将普通的 Animal 对象放入容器，然后通过 `dogBox` 取出并尝试访问 Dog 特有的属性，这将导致运行时错误。

不变通常与可变性（mutability）密切相关。根据泛型类型的操作特性，应选择不同的型变规则：

- 只读操作：泛型类型只允许读取操作（如 `Readonly<T>`），可以安全地设计为协变
- 只写操作：泛型类型只允许写入操作，可以安全地设计为逆变
- 读写操作：泛型类型同时允许读取和写入操作，应该设计为不变，以确保类型安全

## 总结

理解协变、逆变、双向协变和不变是掌握 TypeScript 类型系统的关键。下图展示了四种型变规则的关系和应用场景：

![20251229152719](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251229152719.png)

四种型变规则对比：

| 型变类型 | 定义                                                              | 应用场景                       | 优势                         | 潜在问题                         |
| :------- | :---------------------------------------------------------------- | :----------------------------- | :--------------------------- | :------------------------------- |
| 协变     | 子类型可赋值给父类型，`T<Dog>` 可赋值给 `T<Animal>`               | 函数返回值、数组、只读容器     | 提高灵活性，允许更具体的实现 | 在可变容器中可能导致运行时错误   |
| 逆变     | 父类型可赋值给子类型，`(Animal) => void` 可赋值给 `(Dog) => void` | 函数参数、只写容器             | 允许更通用的处理函数         | 在非严格模式下可能被双向协变覆盖 |
| 双向协变 | 子类型和父类型可相互赋值                                          | 函数参数（非严格模式下）       | 提供最大的灵活性和兼容性     | 可能导致类型安全问题             |
| 不变     | 类型必须精确匹配                                                  | 可读写容器、严格类型匹配的场景 | 提供最高级别的类型安全保证   | 限制了类型的灵活性               |

在实际开发中，应遵循以下原则：

1. 协变 适用于函数返回值和只读容器，保证"更具体"的类型可以替代"更一般"的类型
2. 逆变 适用于函数参数，保证"更通用"的处理函数可以用于特定场景
3. 双向协变 应避免使用，建议启用 `strictFunctionTypes` 来禁用它
4. 不变 适用于可读写容器，确保类型精确匹配以避免运行时错误

根据数据结构的特性（只读、只写或可读写）选择合适的型变规则，在类型安全性和灵活性之间取得平衡。
