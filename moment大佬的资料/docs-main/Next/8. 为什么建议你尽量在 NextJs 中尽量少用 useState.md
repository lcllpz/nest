在构建 Web 应用时，我们常常会用 `useState` 来管理用户交互产生的状态，比如搜索关键词、筛选条件、分页页码等。

但在 Next.js 中，特别是采用 App Router 架构的项目里，其实有一种更简洁、更优雅的做法 —— 用 URL 查询参数 search params 来代替 `useState`。

这样不仅能让页面状态与 URL 同步，带来更好的用户体验，还能方便书签保存、页面分享，并天然支持服务端渲染（SSR）。换句话说，它让应用的状态“可见、可分享、可还原”，从而让交互逻辑更符合 Web 的本质。

## 为什么要用 URL 查询参数替代 useState

相比使用 `useState` 管理组件内状态，在 Next.js（尤其是基于 App Router 架构）中，将 UI 状态绑定到 URL 查询参数（search params）是一种更推荐、更优雅的做法，原因如下：

1. 状态可分享、可还原：将搜索条件、筛选项、分页信息等写入 URL 后，用户只需复制链接即可分享或保存为书签，重新打开即可还原完整状态；而 `useState` 中的状态仅存在内存中，刷新或跳转后即丢失。

2. 天然支持 SSR：URL 查询参数可在服务端直接访问，使得在 `generateMetadata`、`getServerSideProps` 或 React Server Components 中处理状态更为便捷；相比之下，`useState` 需要额外逻辑同步初始状态，处理复杂。

3. 状态持久性更强：URL 中的状态具备天然持久性，即使刷新页面或通过浏览器历史返回，查询参数仍会保留，状态得以一致；而基于 `useState` 的状态在刷新后会重置。

4. 更有利于 SEO：搜索引擎只能识别 URL，而无法访问 JavaScript 内存中的状态。将状态体现在 URL 上，有助于搜索引擎索引页面的不同状态，提升可见度。

5. 浏览器行为更自然：使用 URL 查询参数时，浏览器的前进和后退操作会自动同步状态，无需手动维护历史逻辑，交互体验更符合用户预期。

综上，将 UI 状态写入 URL 查询参数，不仅能让应用状态“可见、可分享、可持久”，还简化了 SSR、SEO 与状态同步的复杂度，是 Next.js 应用中更优雅、更可靠的方案。

  { id: 12, name: "蓝牙耳机", category: "electronics" },
];

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);

  const q = searchParams.get("q")?.toLowerCase() || "";
  const category = searchParams.get("category") || "all";
  const page = parseInt(searchParams.get("page") || "1", 10);
  const pageSize = 5;

  // 过滤数据
  const filtered = allProducts.filter((p) => {
    const matchKeyword = p.name.toLowerCase().includes(q);
    const matchCategory = category === "all" || p.category === category;
    return matchKeyword && matchCategory;
  });

  const total = filtered.length;

  // 分页处理
  const start = (page - 1) * pageSize;
  const pagedItems = filtered.slice(start, start + pageSize);

  return NextResponse.json({ items: pagedItems, total });
}

"use client";

import { useEffect, useState } from "react";

export default function ProductsPage() {
  const [input, setInput] = useState("");

  const [category, setCategory] = useState("all");

  const [total, setTotal] = useState(0);
  const [loading, setLoading] = useState(false);

  const pageSize = 5;

      setLoading(false);

        <input

          }}
        >
          搜索

      </div>

            key={cat}
            style={{
              marginLeft: 8,

            onClick={() => {

              setPage(1);
            }}
          >
            {cat}
          </button>
        ))}
      </div>

      <div style={{ marginBottom: 10 }}>
        <button
          disabled={page <= 1}
          onClick={() => setPage((p) => Math.max(1, p - 1))}

        <span style={{ margin: "0 8px" }}>

当前实现存在以下问题：

1. 刷新后状态丢失：每次刷新页面时，`useState` 中的状态会被重置，导致用户的搜索条件、分页信息等丢失，必须重新设置。

2. 无法精确分享页面状态：如果用户想分享某一页的商品链接，当前的 URL 不包含搜索词、分类或分页信息。因此，分享的链接无法指向具体内容，接收者打开链接后会默认跳转到第一页。

3. SEO 问题：当页面状态（如搜索关键词、分类、分页等）保存在客户端时，搜索引擎无法抓取这些状态，无法为不同查询或页面提供独立的索引和排名。这会影响页面的曝光度和排名，降低站点的 SEO 效果。

## URL 查询参数方案

为了解决这些问题，可以通过将查询条件（如搜索关键词、分类、页码等）绑定到 URL 查询参数。我们现在要修改我们的前端代码了：

```tsx
import { Suspense } from "react";
import ClientPagination, { CategoryButtons } from "@/components/index";
import ClientSearch from "@/components/search";

type Product = {
  id: number;
  name: string;
};

type Props = {
  searchParams: {
    q?: string;
    category?: string;
    page?: string;
  };
};

// 获取商品数据的函数
async function fetchProducts(q: string, category: string, page: number) {
  const params = new URLSearchParams({
    q,
    category,
    page: String(page),
  });

  const res = await fetch(
    `http://localhost:3000/api/products?${params.toString()}`
  );
  const data = await res.json();

  return {
    items: data.items,
    total: data.total,
  };
}

export default async function ProductsPage({ searchParams }: Props) {
  const resolvedSearchParams = await searchParams;
  const { q = "", category = "all", page = "1" } = resolvedSearchParams;

  const pageNum = parseInt(page, 10);

  const data = await fetchProducts(q, category, pageNum);
  const products = data.items;
  const total = data.total;
  const totalPages = Math.ceil(total / 5);

  return (
    <div style={{ padding: 20 }}>
      <h1>📦 商品列表（SSR + 查询参数）</h1>

      <Suspense fallback={<div>Loading...</div>}>
        <div>
          <ClientSearch q={q} category={category} />

          <CategoryButtons q={q} category={category} />

          <ul>
            {products.map((item: Product) => (
              <li key={item.id}>{item.name}</li>
            ))}
          </ul>
        </div>
      </Suspense>

      <ClientPagination
        pageNum={pageNum}
        totalPages={totalPages}
        q={q}
        category={category}
      />
    </div>
  );
}
```

"use client";

type PaginationProps = {
  pageNum: number;
  totalPages: number;
  q: string;
  category: string;

export default function ClientPagination({
  pageNum,
  totalPages,
  q,
  category,
}: PaginationProps) {
  return (
    <div style={{ marginBottom: 10 }}>
      <button
        disabled={pageNum <= 1}

          (window.location.href = `/products?q=${q}&category=${category}&page=${

          }`)

        disabled={pageNum >= totalPages}

            pageNum + 1
          }`)

        下一页

通过这种修改，能够带来以下优化效果：

1. 页面刷新后状态持久化：查询参数会随着 URL 保留下来，即使页面刷新，URL 中的参数依然存在，页面可以根据这些参数恢复之前的状态，无需重新设置。

2. 便于分享和跳转：用户可以分享包含查询参数的链接，接收者打开链接时，可以准确看到分享者所看到的页面内容，而不是仅仅跳转到默认的第一页。

3. 搜索引擎可抓取：URL 查询参数（如 `?q=javascript&page=2`）能够被搜索引擎识别和抓取，从而为不同的页面状态生成独立的索引。例如，针对不同的搜索关键词或分页内容，搜索引擎会为每个具体的 URL 提供独立的排名和索引。

4. 提升页面曝光：每个包含不同查询参数的 URL 都可以被搜索引擎单独索引和展示，这样能大大增加页面在搜索结果中的覆盖面，吸引更多用户点击访问。

通过这种方式，不仅能确保页面状态的持久性，还能提高用户体验和分享的便捷性。

这种优化可以通过将 URL 查询参数作为状态存储方式来实现。在 Next.js 中，我们可以使用 `useSearchParams` 来轻松实现这一功能，并且与服务端渲染（SSR）兼容。

## 总结

通过将应用状态绑定到 URL 查询参数，我们能够实现更好的状态持久性、页面分享和 SEO 优化。相比于使用 `useState` 这种仅在内存中保存状态的方式，URL 查询参数可以让页面在刷新后依然保持用户操作的结果，同时支持链接分享和搜索引擎索引。这样不仅提升了用户体验，也让状态管理更加简洁，并天然兼容服务端渲染（SSR）。

如果希望在实际开发中更方便地实现这一机制，可以使用开源库 [nuqs](https://www.npmjs.com/package/nuqs)。它提供了诸如 `useQueryState` 和 `createSearchParamsCache` 等实用工具，让你能像使用 `useState` 一样管理查询参数，同时在客户端和服务端之间无缝同步，是在 Next.js 中处理 URL 状态的高效解决方案。
