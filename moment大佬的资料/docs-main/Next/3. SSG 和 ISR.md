## SSG 基本概念

SSG（Static Site Generation） 是指在构建时（build time）将页面预渲染为静态 HTML 文件，部署后用户访问直接读取这些 HTML，无需实时请求服务器渲染内容。

传统 SSG 的特点：

- 每个页面都是完整的 `.html` 文件

- 构建后是纯静态文件，无需 JS 也能访问

- 适用于博客、文档、产品页等静态内容

这让 SSG 变得更灵活，但也更复杂。

![20250708235123](https://raw.githubusercontent.com/xun082/md/main/blogs.images20250708235123.png)

在 App Router + React Server Components（RSC） 架构中，SSG 不再是“直接输出完整 HTML”，而是：

> 输出一个 HTML 外壳（`index.html`）+ 一个组件内容结构描述（`index.txt`），

> 然后由客户端 React 加载 `.txt` 来还原组件页面。

我们来看看 NextJS 中的 SSG 生成的格式是怎么样的：

![20250708182346](https://raw.githubusercontent.com/xun082/md/main/blogs.images20250708182346.png)

其中上面有两个文件：

1. `index.html`：页面结构骨架：head、布局、空容器（React 挂载点）

2. `index.txt`：RSC 编码格式，描述组件树、内容数据、元信息（metadata）

用户访问页面时，浏览器首先加载 `index.html`，这是一个包含基本结构的空壳页面。随后会下载必要的 JS 脚本和 `.txt` 文件，React 客户端解析 `.txt` 中的组件结构（如 `$`、`I[]`、`children` 等），还原并激活完整页面内容。

页面内容并不是一开始就完整呈现，因为实际内容保存在 `.txt` 文件中，HTML 只是一个托管壳。React 客户端加载 `.txt` 后还原组件结构，实现懒加载、组件级拆分等现代特性，但这也意味着在 JS 脚本加载和 Hydration 完成之前，页面是“未激活”的。

即使在现代架构中，SSG 依然具备高性能、低成本和高安全性等优势，因其生成的静态资源可部署到 CDN 并长期缓存。相比传统静态站点，结合 App Router 和 RSC 后，SSG 还能保留异步组件和状态管理能力，实现“静态外壳 + 动态体验”的平衡。

SSG 页面初始只加载一个空壳 HTML，真实内容需等 JS 和 `.txt` 文件加载后才能呈现。若滥用懒加载或 Suspense，可能导致页面结构渲染延迟，影响首屏速度与 SEO。内容更新也较繁琐，通常需整页重建，除非引入 ISR 优化。

## SSG 的优点

### 更快的首屏加载速度

在使用 SSR（服务端渲染）的网站中，当用户在浏览器中请求某个页面时，服务器会即时将该页面的 HTML 内容渲染好并发送给浏览器。这意味着浏览器在接收到响应后，立即就可以显示出完整的页面结构和内容，而不需要等待 JavaScript 的下载、解析和执行过程。

这种方式大大提升了页面的首屏加载速度，让用户能够更快地看到网站的实际内容，从而改善了所谓的“感知性能”（Perceived Performance）——即使后台数据仍在加载，用户也会觉得“页面加载很快”。

由于页面内容在初始加载时就已经被呈现，这种渲染方式特别适合以内容展示为主的网站，比如新闻门户、博客文章、产品详情页等。这类网站的用户通常更关注阅读体验和信息呈现速度，因此 SSR 是一个非常合适的选择。

![20250712212608](https://raw.githubusercontent.com/xun082/md/main/blogs.images20250712212608.png)

### 更好的 SEO 友好性

相比于 CSR（客户端渲染），SSR（服务端渲染）具有更好的 SEO 友好性，因为服务器返回的是已经包含完整内容的 HTML 页面，搜索引擎爬虫可以直接读取和抓取。而 CSR 则需要额外配置预渲染或动态渲染机制，否则其内容很难被爬虫识别和收录。

## SSG 的缺点

虽然 SSG 有着极高的性能、安全性和 SEO 优势，但它并不适用于所有场景。下面是它的主要缺点和成因解释：

### 构建时间长，随内容增长而线性/指数增加

当我们的网站内容很多（比如成千上万个产品页或文章页）时，每次构建都要重新生成所有静态页面，构建时间会越来越长。

这个问题的主要原因是 SSG 是构建时生成，意味着每次更新都要把整个站点重新编译一次，尤其是使用 Markdown 或 CMS 拉取数据的大型站点，很容易导致构建时间从几秒增加到几分钟、甚至几十分钟。

### 适合频繁更新的数据或实时内容

静态网站是“预渲染”的，无法动态响应内容变化，比如库存数量、订单状态、用户评论等。

这些信息如果需要实时变化，在 SSG 中只能通过客户端 JS 动态拉取（称为“客户端动态注水”），这就丧失了 SSG 本来的静态优势。

### 缺乏动态内容个性化支持

SSG 的页面是“一次构建、所有人共用”的，无法针对不同用户显示不同内容。

如果我们有个用户中心、登录信息、权限展示等，这类个性化内容只能通过 JS 加载或配合其他服务（如 Edge Middleware），大大增加了开发复杂度。

由于 SSG 在构建时就生成了静态的 HTML 页面，因此所有用户访问到的内容都是统一的，无法根据不同用户的身份或状态进行个性化渲染。

比如，当用户登录后，页面无法自动显示“欢迎你，小明”这样的定制信息；又比如，在后台管理系统中，不同权限的用户需要看到不同的操作按钮，这种基于身份控制的界面也无法通过 SSG 实现，必须依赖客户端的 JavaScript 动态渲染来完成。这样不仅增加了开发复杂度，也削弱了静态站点的性能优势。

## SSG 总结

SSG 是一种在构建阶段将页面预渲染为静态 HTML 文件的技术，用户访问时无需服务器处理即可直接加载页面，性能极佳。它非常适合内容稳定、更新频率低的站点，如博客、文档、产品介绍页，并具备出色的 SEO 和安全性。相比动态渲染，SSG 构建速度可能会随着内容增长而变慢，且不适合频繁更新或需要个性化展示的场景。如果网站内容主要是公开的、面向所有用户一致的，SSG 是极高效的选择。

## ISR 基本概念

ISR（增量静态生成） 是由 Next.js 提出的渲染模式，它结合了 SSG 的高速与 SSR 的动态，通过“按需更新静态页面”的方式，让网站既能快速响应，又能动态更新数据。

通俗地说就是：

> 页面初始是静态的，访问后如果内容过期，系统会在后台重新生成一份新的静态 HTML 并替换老的页面，但不影响当前访问者体验。

ISR 的原理是在构建阶段或首次访问时生成静态页面，并将其缓存到 CDN。用户访问页面时，如果缓存未过期，直接返回静态 HTML，速度极快；如果已过期，系统会在后台异步地重新生成最新页面。

这个更新过程不会影响当前用户体验，下一位用户访问时即可看到更新后的页面。这样就实现了“静态性能 + 动态更新”的平衡，避免了每次构建全站重生成的问题。

## 如何在 NextJs 中使用

接下来我们借助官方文档提供的 Demo，在这个代码示例中我们展示了如何在 Next.js 15 中通过 app/ 目录配合 `revalidate` 和 `generateStaticParams` 来实现 增量静态生成（ISR）。该页面用于展示博客文章内容，数据从远程 API 获取。

```ts
interface Post {
  id: string;
  title: string;
  content: string;
}

export const revalidate = 60;

export const dynamicParams = true;

export async function generateStaticParams() {
  const posts: Post[] = await fetch("https://api.vercel.app/blog").then((res) =>
    res.json()
  );
  return posts.map((post) => ({
    id: String(post.id),
  }));
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  );
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  );
}
```

这段代码展示了如何使用 generateStaticParams 在构建时生成部分文章页面，同时通过 export const revalidate = 60 设置增量再生周期，实现页面静态缓存每 60 秒自动更新一次。

用户访问时能享受到 SSG 的速度，而内容也能随着时间自动刷新，无需手动重构发布。

## ISR 的优点

虽然 ISR 提供了静态性能和内容实时性的平衡，但首次访问未生成的页面时，仍会经历类似 SSR 的加载延迟。同时，后台静默更新虽然不会中断用户体验，但也意味着用户短时间内可能看到旧数据。

对于强实时性要求的应用（如秒级更新或个性化内容），ISR 的定时再生机制可能不够及时。此外，虽然配置简单，但在自建部署环境中仍需手动处理缓存、预热等细节，略微增加了运维复杂度。

## SSG 总结

ISR（Incremental Static Regeneration） 是 Next.js 提供的一种混合渲染方案，它结合了 SSG 的高性能与 SSR 的动态更新能力。页面初次访问时生成静态 HTML，并在设定的时间间隔后自动在后台重新生成，实现“按需增量构建”。

用户总能快速访问页面，而内容也能定期保持最新，适用于内容经常更新但无需实时展示的场景。通过简单配置 `revalidate` 参数，开发者就能轻松实现高性能与内容更新的平衡。

## SSG 和 ISR 总结

SSG（静态网站生成） 在构建时一次性将所有页面渲染为静态 HTML，访问速度极快、部署简单，但内容更新依赖重新构建全站，不适合频繁变动的数据。

ISR（增量静态生成） 则在保留静态性能的同时，引入了按需后台更新机制，只需配置 `revalidate` 时间即可定期刷新页面内容。

相比之下，SSG 更适合内容稳定的网站，而 ISR 更适合需要定期更新但不要求实时响应的场景。
两者都具备出色的 SEO 和加载性能，区别主要在于构建策略与内容更新机制的灵活性。
