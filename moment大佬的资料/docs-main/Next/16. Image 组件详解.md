## 前言

图片往往占据网页大小的 40% 以上，是影响页面性能的关键因素。Next.js 的 Image 组件基于原生 img 标签做了深度优化，自动实现懒加载、响应式适配、格式转换等功能。相比原生 img 标签需要手动实现这些优化，Image 组件让图片优化变得简单高效。

下图对比了原生 img 标签与 Next.js Image 组件的差异：

![20260205222719](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260205222719.png)

从图中可以看到，原生 `img` 标签存在立即加载所有图片、尺寸固定、格式未优化等问题，而 Next.js Image 组件通过懒加载、自动响应式、现代图片格式转换、防止布局偏移等优化，大幅提升了图片加载性能和用户体验。

## 1. 图片与 LCP

根据统计，图片大小占网页总大小的 40% 以上（HTML、CSS、JS 的总和才占 30%）。在 70% 的移动端页面上，最大的内容元素都是图片。

LCP（Largest Contentful Paint，最大内容绘制）是 Google 核心性能指标之一，衡量首屏最大元素的渲染时间。浏览器会在每一帧渲染时标记最大元素，当用户交互（点击、滚动）时停止记录，最后一次记录的时间就是 LCP 值。良好的用户体验要求 LCP 在 2.5 秒以内。

LCP 的计算流程如下：

![20260205222804](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260205222804.png)

从流程图可以看出，浏览器会持续跟踪页面中的最大内容元素，每当有更大的元素出现时就更新记录。一旦用户开始与页面交互，浏览器就停止跟踪，此时记录的最后一个元素的渲染时间就是最终的 LCP 值。这就是为什么优化首屏最大图片的加载速度如此重要。

## 2. Image 组件

Next.js 的 Image 组件在原生 `img` 基础上实现了自动尺寸优化（WebP/AVIF）、防止布局偏移、懒加载、模糊占位符等功能。基础使用很简单：

```jsx
// app/page.js
import Image from "next/image";

export default function Page() {
  return (
    <Image
      src="/profile.png"
      width={500}
      height={500}
      alt="Picture of the author"
    />
  );
}
```

### 组件属性

Image 组件的属性分为几类，基础属性（`src`、`width`、`height`、`alt`）、优化属性（`priority`、`placeholder`、`quality`）、布局属性（`fill`、`sizes`、`style`）、事件回调（`onLoad`、`onError`）。

### 2.1. 基础属性

`src` 是必传属性，支持两种方式，静态导入本地图片或传入 URL 字符串。使用本地图片时，通过 `import` 导入 `.jpg`、`.png`、`.webp` 等格式的文件

```jsx
// app/page.tsx
import Image from "next/image";
import profilePic from "./me.png";

export default function Page() {
  return (
    <Image
      src={profilePic}
      alt="Picture of the author"
      // width={500} automatically provided
      // height={500} automatically provided
      // blurDataURL="data:..." automatically provided
      // placeholder="blur" // Optional blur-up while loading
    />
  );
}
```

静态导入的好处是 Next.js 会自动读取图片尺寸，所以不需要手动传 `width` 和 `height`。注意 `import` 必须是静态的，动态的 `await import()` 或 `require()` 不支持。

![20260205223735](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260205223735.png)

他现在完全是根据原图片的宽高来自动读取的，因为我们本地读取的图片，是完全是可以经过设计好得来的。

使用远程图片时，`src` 传入 URL 字符串，此时必须手动提供 `width` 和 `height`。这两个属性主要用于推断图片的宽高比，避免加载时发生布局偏移，但不决定图片最终的渲染尺寸（还可以通过 CSS 控制）

```jsx
// app/page.tsx
import Image from "next/image";

export default function Page() {
  return (
    <Image
      src="https://nextjs.org/_next/image?url=https%3A%2F%2Fh8DxKfmAPhn8O0p3.public.blob.vercel-storage.com%2Fstatic%2Fblog%2Fturbopack-incremental-computation%2Fturbopack--dark.png&w=3840&q=75"
      alt="Picture of the author"
      width={500}
      height={500}
    />
  );
}
```

使用远程图片时需要在 `next.config.js` 中配置白名单，防止恶意使用：

```jsx
// next.config.ts
/** @type {import('next').NextConfig} */
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "nextjs.org",
        pathname: "/**",
      },
    ],
  },
};

module.exports = nextConfig;
```

`width` 和 `height` 以像素为单位，除非是静态导入或使用 `fill` 属性，否则必须提供。

`alt` 属性用于无障碍访问和 SEO，当图片无法加载时作为降级文本。它应该准确描述图片内容，但不要重复周围已有的文字。如果图片纯粹是装饰性的，可以使用空字符串 `alt=""`。

### 2.2. 自定义加载器

如果需要使用第三方图片服务（如 Cloudinary、Imgix）或自定义 CDN，可以通过 `loader` 函数自定义图片 URL 的生成逻辑

```jsx
"use client";

const imageLoader = ({ src, width, quality }) => {
  return `https://cdn.example.com/${src}?w=${width}&q=${quality || 75}`;
};

<Image loader={imageLoader} src="me.png" alt="..." width={500} height={500} />;
```

`loader` 接收 `src`、`width`、`quality` 参数，返回完整的图片 URL。注意使用 `loader` 需要客户端组件。如果项目中大量图片都用同一个 CDN，可以在 `next.config.js` 中全局配置 `loaderFile`，避免每个组件都写一遍。

### 2.3. 布局控制

当图片尺寸未知时，可以用 `fill` 属性让图片填充父元素。使用时父元素需要设置定位（`relative`、`fixed` 或 `absolute`），图片会自动设为 `absolute` 定位

```jsx
// app/page.js
import Image from "next/image";
import profilePic from "./image.png";

export default function Page() {
  return (
    <div
      style={{
        width: "200px",
        height: "200px",
        backgroundColor: "#ccc",
        position: "relative",
      }}
    >
      <Image src={profilePic} alt="Picture of the author" />
    </div>
  );
}
```

正常显示如下：

![20260205224532](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260205224532.png)

如果添加 `fill` 属性后：

```jsx
<Image src={profilePic} alt="Picture of the author" fill={true} />
```

效果如下，图片会被拉伸以适应容器：

![20260205224624](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260205224624.png)

配合 `object-fit` 可以控制填充方式。`contain` 会完整显示并保持比例，`cover` 会裁剪填充并保持比例，默认则直接拉伸填充

```jsx
<Image src={pic} alt="..." fill style={{ objectFit: "cover" }} />
```

### 2.4. 响应式图片

HTML5 的 `srcset` 和 `sizes` 属性用于实现响应式图片，让不同设备加载不同尺寸的图片。`srcset` 有两种用法，一种是根据像素密度（1x、2x），一种是根据图片宽度（480w、800w）。

像素密度的用法

```jsx
<img
  srcset="elva-fairy-320w.jpg, elva-fairy-480w.jpg 1.5x, elva-fairy-640w.jpg 2x"
  src="elva-fairy-640w.jpg"
  alt="Elva dressed as a fairy"
/>
```

`srcset` 是由逗号分隔的一个或多个字符串组成，每段字符串由以下部分组成

1. 指向图像的 URL
2. 一个空格（可选）
3. 一个像素密度描述符（一个正浮点数，后面紧跟 `x` 符号）

如果我们给图片应用一个 CSS 样式

```css
img {
  width: 320px;
}
```

它的宽度在屏幕上就是 320 像素（CSS 像素）。浏览器会计算出正在显示的显示器的分辨率，然后显示 srcset 引用的最适合的图片。如果是普通的分辨率，一个设备像素表示一个 CSS 像素，那就会加载 `elva-fairy-320w.jpg`，它的大小是 39KB，如果设备是高像素，用两个或者更多的设备像素表示一个 CSS 像素，那就会加载 `elva-fairy-640w.jpg`，它的大小是 93KB。

再说第二种情况，相同的图片内容，但依据设备显示的更大或者更小。使用示例代码如下

```jsx
<img
  srcset="elva-fairy-small.jpg 480w, elva-fairy-large.jpg 800w"
  src="elva-fairy-large.jpg"
  alt="Elva dressed as a fairy"
/>
```

`srcset` 的语法与刚才略有不同，它定义了浏览器可选择的图片设置以及每个图片的大小。分析它的语法，依然是由逗号分隔的一个或多个字符串组成，每段字符串由以下部分组成：

1. 指向图像的 URL
2. 一个空格
3. 图片的固有宽度（以像素为单位）。注意这里使用宽度描述符 `w`，而非 `px`。但一个 `w` 对应 1 个像素。图片的固有宽度是指它的真实大小。

此时我们就告诉了浏览器，这张图片有两种备选图片可以显示，一张是 `elva-fairy-small.jpg`，这张图片的宽度是 480px，一张是 `elva-fairy-large.jpg`，它的宽度是 800px。

那浏览器怎么知道用哪张图片呢？比如当前设备视口宽度是 640px，是选择 480px 的图片还是选择 800px 的图片呢？

为了帮助浏览器判断，你就需要写 `sizes` 属性。`sizes` 属性就是一组媒体查询条件，告诉浏览器什么样的条件使用什么样的图片。一个使用示例如下

```jsx
<img
  srcset="elva-fairy-small.jpg 480w, elva-fairy-large.jpg 800w"
  sizes="(max-width: 600px) 480px,
         800px"
  src="elva-fairy-large.jpg"
  alt="Elva dressed as a fairy"
/>
```

`sizes` 也是由逗号分隔的一个或多个字符串组成，每段字符串由以下部分组成

1. 一个媒体条件，例子中为 `(max-width:600px)`，它表示当视口的宽度小于等于 600px 时
2. 一个空格
3. 当媒体条件为真时图片将填充的宽度，这个宽度可以是固定值（比如 `480px`），也可以是相对于视口的宽度（如 `50vw`），但不能是百分比。如果没有媒体条件，表示是默认生效。当浏览器成功匹配第一个媒体条件时，剩下所有的条件都会被忽略，所以顺序很重要。

有了这些属性后，浏览器会执行以下步骤：

1. 检查设备宽度
2. 检查 `sizes` 列表中哪个媒体条件是第一个为真
3. 查看该媒体查询对应的槽大小
4. 加载 `srcset` 列表中引用的最接近所选槽大小的图像

比如在这个例子中，如果浏览器的视口是 480px，那么 `sizes` 中的第一个条件 `(max-width: 600px)` 就为真，所以选择 480px 大小，因为 480px 与固有宽度 `480w` 最接近，所以加载 `elva-fairy-small.jpg`。通过这种方式就可以实现移动端加载小图片，从而加快移动端的加载速度。

下图展示了浏览器如何根据 sizes 和 srcset 选择合适的图片：

![20260205225355](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260205225355.png)

从流程图可以看出，浏览器首先检查设备视口宽度，然后遍历 `sizes` 条件找到第一个匹配的媒体查询，获取对应的槽大小后，在 `srcset` 中选择最接近该大小的图片，最后结合设备像素比决定加载哪张图片。这个机制确保了不同设备能加载最合适的图片尺寸。

讲完 `img` 元素的 `srcset` 和 `sizes` 属性，回到 Image 组件上。

Next.js 的智能处理

- 你无需手动设置 `srcset`，Next.js 会自动生成
- 设置 `sizes` 属性会影响生成的 `srcset` 的值
- 未设置 `sizes` 时使用像素密度描述符（1x、2x）
- 已设置 `sizes` 时使用固有宽度描述符（640w、750w）

这种自动化处理大大简化了响应式图片的使用。

`quality` 控制图片质量，取值 1-100，默认 75。一般情况下默认值就够用了，除非对图片质量有特殊要求。

### 2.5. 优先加载

`priority` 设为 `true` 时会禁用懒加载并预加载图片，适合首屏可见的关键图片（Hero 图、Banner 等）。如果 LCP 元素是图片但没设置 `priority`，开发环境会有警告提示。注意不要滥用，只给真正重要的首屏图片加 `priority`

```jsx
<Image src={heroImage} alt="Hero" priority placeholder="blur" />
```

### 2.6. 加载占位

`placeholder` 有三个值，`empty`（默认无占位）、`blur`（模糊占位）、`data:image/...`（自定义占位图）。静态导入的图片用 `blur` 会自动生成模糊占位，远程图片需要手动提供 `blurDataURL`

```jsx
// 静态导入 - 自动生成
<Image src={heroImg} alt="Hero" placeholder="blur" />

// 远程图片 - 需要提供 blurDataURL
<Image src="..." width={500} height={300} alt="..."
  placeholder="blur"
  blurDataURL="data:image/jpeg;base64,..."
/>
```

生成 `blurDataURL` 可以用在线工具（如 png-pixel.com）或 Plaiceholder、Sharp 等库。

### 2.7. 其他常用属性

`style` 属性用于设置 CSS 样式，修改宽度时记得设置 `height: 'auto'`，否则图片会变形

```jsx
<Image src={img} alt="..." style={{ width: "100%", height: "auto" }} />
```

`onLoad`、`onLoadingComplete`、`onError` 是事件回调，需要使用客户端组件

```jsx
<Image onLoadingComplete={(img) => console.log(img.naturalWidth)} />
```

`loading` 属性控制加载行为，默认 `lazy`（懒加载），一般不建议改成 `eager`。

`unoptimized` 属性设为 `true` 时会跳过优化，直接使用原图。

## 3. 配置选项

除了通过 props，你也可以在 `next.config.js` 中配置图片组件。

### 3.1. remotePatterns

为保护应用远离恶意用户，当使用外部图片的时候需要配置 `remotePatterns`。这是一个安全机制，防止未授权的图片源消耗服务器资源。

基础配置示例

```jsx
// next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "nextjs.org",
        pathname: "/**",
      },
    ],
  },
};

module.exports = nextConfig;
```

此配置的含义

- 仅允许 `https://nextjs.org` 开头的图片
- 其他协议、主机名、端口或不匹配的路径都会返回 400 错误

`remotePatterns` 支持使用通配符来匹配多个域名或路径，下图说明了两种通配符的区别：

![20260206091916](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260206091916.png)

单星号（`*`）只能匹配单层子域或路径段，而双星号（`**`）可以匹配任意层级。注意双星号不能用在模式中间，只能用在开头或结尾。

通配符配置示例

```jsx
// next.config.js
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "**.nextjs.org", // 匹配所有子域
      },
    ],
  },
};
```

支持的 URL 示例

- ✓ `https://assets.nextjs.org/image.jpg`
- ✓ `https://cdn.asia.nextjs.org/image.jpg`
- ✓ `https://a.b.c.nextjs.org/image.jpg`
- ✗ `http://nextjs.org/image.jpg`（协议不匹配）
- ✗ `https://other.com/image.jpg`（域名不匹配）

实用配置模式

```jsx
// 多个图片源
module.exports = {
  images: {
    remotePatterns: [
      // Next.js 官方资源
      {
        protocol: "https",
        hostname: "nextjs.org",
        pathname: "/**",
      },
      {
        protocol: "https",
        hostname: "assets.nextjs.org",
        pathname: "/**",
      },
      // 用户上传内容
      {
        protocol: "https",
        hostname: "**.amazonaws.com",
        pathname: "/user-uploads/**",
      },
      // 第三方图片服务
      {
        protocol: "https",
        hostname: "images.unsplash.com",
      },
    ],
  },
};
```

### 3.2. loaderFile

如果需要为所有图片统一使用自定义 CDN 或图片服务，可以在 `next.config.js` 中配置 `loaderFile`。该文件必须导出一个默认函数：

```jsx
// next.config.js
module.exports = {
  images: {
    loader: "custom",
    loaderFile: "./my/image/loader.js",
  },
};

// my/image/loader.js
("use client");
export default function myImageLoader({ src, width, quality }) {
  return `https://example.com/${src}?w=${width}&q=${quality || 75}`;
}
```

这会应用到所有的 `next/image` 实例，如果要修改个别图片，使用 `loader` prop。

### 3.3. deviceSizes

如果你知道用户的设备宽度，那你可以使用 `next.config.js` 的 `deviceSizes` 来声明一系列的设备宽度断点。当 `next/image` 组件使用 `sizes` prop 的时候，这些宽度会被用来推断正确加载的图片。

如果没有配置，默认值是

```jsx
// next.config.js
module.exports = {
  images: {
    deviceSizes: [640, 750, 828, 1080, 1200, 1920, 2048, 3840],
  },
};
```

### 3.4. imageSizes

你可以使用 `next.config.js` 的 `imageSize` 属性声明一系列的图片宽度。

如果没有配置，默认值是

```jsx
// next.config.js
module.exports = {
  images: {
    imageSizes: [16, 32, 48, 64, 96, 128, 256, 384],
  },
};
```

`imageSize` 和 `deviceSizes` 会影响图片生成最终的 `srcset` 尺寸

那么问题来了，都是用来生成最终的 `srcset`，直接用一个数组不就好了吗？为什么非要用两个数组？

下图说明了 `deviceSizes` 与 `imageSizes` 的使用场景和区别：

![20260206092458](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260206092458.png)

从图中可以看到，当不使用 `sizes` prop 时，Next.js 只会使用 `deviceSizes` 生成像素密度描述符（1x、2x），适合全屏图片。当使用 `sizes` prop 时，会合并 `deviceSizes` 和 `imageSizes` 生成固有宽度描述符（640w、750w），适合响应式图片。

简单来说，`deviceSizes` 用于处理全屏或大图片且总是生效，`imageSizes` 用于处理小图片且仅在使用 `sizes` prop 时生效。`imageSizes` 的所有值应该小于 `deviceSizes` 的最小值。使用 `sizes` prop 通常表示图片宽度小于全屏宽度。

示例说明

```jsx
import Image from "next/image";
import profilePic from "./image.png";

export default function Page() {
  return (
    <Image
      src={profilePic}
      sizes="(max-width: 600px) 160px,
      320px"
      alt="Picture of the author"
    />
  );
}
```

对应生成的 `srcset` 就包含了 `deviceSizes` 和 `imageSizes` 的所有尺寸

当你使用了 `sizes` prop 的时候，说明图片的宽度是小于全屏宽度的。`imagesSizes` 的中的所有值应该都小于 `deviceSizes` 中的最小值。

### 3.5. formats

Next.js 默认的图片优化 API 会自动通过请求中的 `Accept` 请求头检测浏览器支持的图片格式。

下图展示了 Next.js 如何根据浏览器能力自动选择最优的图片格式：

![20260206092523](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260206092523.png)

Next.js 会优先尝试使用 AVIF 格式（文件最小），如果浏览器不支持则降级到 WebP，最后兜底使用原始格式。动画图片会直接使用原始格式，避免破坏动画效果。这种智能降级策略确保了在保证兼容性的同时获得最佳性能。

数组中的顺序决定优先级。浏览器支持多个格式时，使用数组中第一个匹配的。源图片为动图或无匹配时，回退到原始格式。

我们来看图片格式对比：

| 格式 | 文件大小    | 浏览器支持 | 优势             | 劣势               |
| ---- | ----------- | ---------- | ---------------- | ------------------ |
| JPEG | 基准        | 100%       | 兼容性最好       | 文件较大，无透明度 |
| PNG  | 较大        | 100%       | 支持透明度       | 文件大，不适合照片 |
| WebP | 减少 25-35% | 96%+       | 较小，支持透明度 | 少数旧浏览器不支持 |
| AVIF | 减少 40-50% | 85%+       | 最小，质量好     | 编码慢，兼容性较差 |

默认配置仅启用 WebP，推荐配置同时启用 AVIF 和 WebP：

```jsx
// 默认配置
module.exports = {
  images: {
    formats: ["image/webp"],
  },
};

// 推荐配置
module.exports = {
  images: {
    formats: ["image/avif", "image/webp"], // AVIF 优先，WebP 降级
  },
};
```

生产环境推荐同时启用 AVIF 和 WebP。AVIF 放在前面可获得更小的文件，WebP 作为降级方案保证兼容性，原始格式自动兜底确保所有浏览器都能显示。以同一张图片为例，原始 JPEG 100KB，WebP 格式 65KB（减少 35%），AVIF 格式 50KB（减少 50%）。

### 3.6. minimumCacheTTL

图片会根据请求动态优化并存储在 `<distDir>/cache/images` 目录。优化后的图像文件会被用于后续请求，直到缓存过期。当匹配到过期的文件时，过期图片会立刻失效，同时图片会在后台重新优化并使用新的失效日期储存在缓存中。

下图展示了 Next.js 图片缓存的完整流程：

![20260206092704](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260206092704.png)

从图中可以看出，Next.js 的缓存策略采用了"stale-while-revalidate"模式。当缓存过期时，会立即返回旧图片（保证响应速度），同时在后台重新优化图片（更新缓存）。这种机制在保证用户体验的同时，确保图片能够及时更新。

缓存状态说明（通过响应头 `x-nextjs-cache` 查看）

- `MISS` 表示路径不在缓存中，首次请求
- `HIT` 表示路径在缓存中，且未过期
- `STALE` 表示路径已缓存但超出重新验证时间，会在后台更新

过期时间可以通过两种方式配置，`minimumCacheTTL` 配置项用于全局设置，`Cache-Control` 响应头的 `max-age` 字段用于单个图片。如果 `s-maxage` 和 `max-age` 都存在，`s-maxage` 优先。两种方法同时设置时取较大值。

`minimumCacheTTL` 配置项用来设置缓存优化图片的过期时间，它以秒为单位。使用示例如下

```jsx
// next.config.js
module.exports = {
  images: {
    minimumCacheTTL: 60,
  },
};
```

如果你需要更改每张图片的缓存行为，你可以通过 `headers` 配置图片资源的 `Cache-Control` 标头。

大部分时候，建议使用静态图片导入，它会自动对文件内容进行哈希处理，并使用 `immutable` 的 `Cache-Control` 标头。

目前还没有重新验证缓存的机制，所以最好将 `minimumCacheTTL` 的值设低一点、否则你可能需要手动修改 `src` prop 或者删除 `<distDir>/cache/images` 以更新缓存。

### 3.7. 其他配置项

其他不太常用的配置项：

- `disableStaticImages`：禁用静态导入功能（当与其他插件冲突时使用）
- `dangerouslyAllowSVG`：允许优化 SVG 图片（需要额外安全配置，谨慎使用）
- `domains`：远程图片域名列表（已废弃，建议用 `remotePatterns`）

## 4. 动画图像

默认 `loader` 会自动跳过动画图片的优化并按原样展示，这是因为优化过程可能会破坏动画效果。

下图展示了 Next.js 如何处理动画图片：

![20260206092744](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260206092744.png)

Next.js 会自动检测图片是否为动画格式（GIF、APNG、WebP），如果是动画则跳过优化保持原始效果。如果设置了 `unoptimized` 属性，可以跳过检测步骤直接使用原图，节省处理时间。

Next.js 支持三种动画格式，它们各有特点：

- GIF 是最常见的动画格式，但文件较大
- APNG 是 PNG 动画，支持透明度，文件也较大
- WebP 动画文件较小，兼容性较好，是推荐的选择

使用动画图片时，Next.js 会自动检测并跳过优化。如果想提升性能，可以显式设置 `unoptimized` 属性：

```jsx
<Image src="/animation.gif" width={300} height={300} alt="..." unoptimized />
```

从性能角度考虑，建议优先使用 CSS 动画或 Lottie（文件小 90% 以上）替代 GIF。如果必须用动画图片，优先选择 WebP 格式，比 GIF 小 30-50%。

## 5. 响应式图片与布局偏移

实现响应式图片需要设置 `sizes` 和 `style`。根据图片来源选择合适的方式

```jsx
// 1. 本地静态图片
import me from "./me.jpg";
<Image src={me} alt="..." sizes="100vw" style={{width: '100%', height: 'auto'}} />

// 2. 远程图片（需要指定宽高比）
<Image
  src="https://..."
  width={500}
  height={300}
  alt="..."
  sizes="100vw"
  style={{width: '100%', height: 'auto'}}
/>

// 3. 不知道宽高比（用 fill）
<div style={{position: 'relative', width: '100%', height: '300px'}}>
  <Image src="..." alt="..." fill sizes="100vw" style={{objectFit: 'cover'}} />
</div>
```

Next.js 要求设置图片尺寸是为了防止布局偏移（CLS）。没有尺寸时，图片加载会导致内容跳动，影响用户体验。下图对比了有无尺寸设置的区别：

![20260206092819](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260206092819.png)

从对比中可以清楚地看到，没有尺寸设置时，图片加载会导致后续内容突然下移，造成糟糕的用户体验。而设置尺寸后，浏览器会预留空间，图片加载时不会影响布局。

Image 组件通过三种方式防止偏移：静态导入自动提取尺寸、手动指定 `width/height`、或使用 `fill` 让父元素提供尺寸。

## 6. 总结

Image 组件的核心在于三点，首屏关键图片用 `priority` 预加载，远程图片记得配置 `remotePatterns` 白名单，不知道尺寸的用 `fill` 配合父元素定位。掌握这些基本就能应对大部分场景了。更多细节可以查看下方的参考链接。
