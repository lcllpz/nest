你如果经常在 GitHub 上关注开源项目，对 Vercel 这家公司一定不会陌生。很多热门项目在有人提交分支时，都会看到 Vercel 的机器人自动部署预览环境。而点进 Vercel 官网，你也会注意到一个高频出现的关键词——Edge。

Edge Function 不仅由 Vercel 提供，像 Netlify、AWS 等厂商也有类似服务。那么，Edge Function 究竟是什么？

## 什么是 Edge Runtime？

Edge Runtime 是一种运行 JavaScript/TypeScript 的轻量级服务端执行环境，部署在全球各地的边缘节点（接近用户的 PoP/数据中心）。它通常基于 V8 Isolates（轻量沙箱），不是传统的容器或虚拟机，因此启动极快、并发伸缩灵活、延迟低。代表实现包括 Vercel Edge Runtime / Edge Functions 与 Cloudflare Workers 等。

核心特点：

- 更靠近用户：代码在距离用户最近的边缘节点执行，减少跨区域网络传输，从而降低 TTFB（Time To First Byte，首字节时间） ——也就是浏览器从发出请求到收到第一个字节响应所花费的时间。

- 轻量隔离模型：使用 V8 isolate，避免为每个请求启动容器/VM，冷启动几乎可忽略。

- Web 标准 API 优先：以 `fetch`、`Request/Response`、`URL`、`crypto`、`Streams` 等 Web 平台 API 为主，而不是完整的 Node.js 内置模块。

## 与 Node.js Runtime 的区别

| 对比维度   | Edge Runtime                                                                                                                                              | Node.js Runtime                                                      |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| API 能力   | 基于 Web 标准 API，如 `fetch`、`Request`、`Response`、`crypto.subtle` 等。不支持 `fs`、`net`、`child_process` 等 Node 内置模块，因此部分 npm 包无法使用。 | 提供完整的 Node.js 原生 API，几乎所有 npm 包均可运行。               |
| 启动与扩缩 | 基于 V8 Isolates，启动速度极快，可根据请求量自动进行细粒度弹性伸缩。                                                                                      | 通常运行在容器或进程中，冷启动时间更长，扩缩速度较慢。               |
| 部署位置   | 部署在全球多个边缘节点，让代码在离用户最近的地方执行，从而显著降低延迟。                                                                                  | 部署集中于少量数据中心或区域，用户距离较远时延迟更高。               |
| 适用场景   | 低延迟响应、实时个性化、A/B 测试、请求预处理（鉴权/重写/重定向）、流式渲染等。                                                                            | 大规模计算、长时任务、需要访问文件系统或依赖原生模块的复杂后端逻辑。 |

Next.js 提供两种服务端运行时。

1.  当你需要使用 Node 内置模块或执行较重逻辑时，应选择 Node.js Runtime；

2.  若追求极致性能与低延迟响应，则可使用 Edge Runtime。

## Edge Runtime 的性能与平台限制（以 Vercel 为例）

Edge Runtime 追求即刻响应、短时执行的设计理念，因此在性能表现和运行时长上都有明确约束：

- 起始响应时间：函数需在约 25 秒内开始返回数据，否则流式传输能力会受到限制。开始输出后，单次流式响应最长可持续 300 秒。
- 执行时长上限：从 2025 年 3 月 1 日起，Edge Runtime 的单次执行时间上限为 300 秒，超过将被强制终止。
- 计费方式与性能指标：Vercel 按 CPU 执行单元（Execution Unit） 计费，每个单元代表 50 ms 的实际计算时间，不包含 I/O 等待或空闲时间。
- 高可用与容错能力：Edge Runtime 在全球多区域节点自动冗余部署，并具备故障转移机制，确保在单点故障或区域中断时仍可稳定服务。

此外，Cloudflare Workers 等同类平台同样基于 V8 isolate 技术，强调毫秒级启动与高并发处理能力，但其限制（如内存上限、超时机制、OOM 行为）略有不同。

详情可移步[vercel 文档](https://vercel.com/changelog/new-execution-duration-limit-for-edge-functions)

## 优缺点与最佳实践建议

Edge Runtime 的优势在于它非常擅长处理轻量、低延迟的逻辑场景，例如请求预处理、权限鉴权、A/B 测试、个性化渲染或流式数据返回。

由于代码运行在离用户最近的边缘节点，能够显著降低 TTFB（Time To First Byte，首字节时间），从而带来更快的首屏响应体验。

此外，Edge Runtime 基于 Web 标准 API（如 `fetch`、`Response`、`crypto.subtle`），可以与各种 HTTP 驱动的 Serverless 服务（如 Upstash、PlanetScale、Vercel KV）天然配合，非常适合在边缘节点执行动态逻辑或轻计算。

但与此同时，它也有明确的限制。Edge Runtime 并不支持 Node.js 的原生模块（例如 `fs`、`net`、`child_process` 等），因此无法执行文件读写、进程操作或使用依赖原生绑定的 npm 库。

它也不适合执行重计算或长时间运行的任务，更不适合携带大型依赖包。总体来看，Edge Runtime 应被视作一个快速响应层——让你的业务逻辑尽可能贴近用户端运行，而非完全替代传统后端服务。

在使用时，可以遵循以下最佳实践来发挥 Edge Runtime 的最大价值：

- 优先使用 Web API：尽量选择不依赖 Node 内建模块的库，例如用 `fetch` 发送请求、用 `crypto.subtle` 做加密签名，或通过 `ReadableStream` 实现流式响应。

- 做好分层与解耦：将需要 Node 能力（如图像处理、PDF 生成、FFmpeg、数据库原生驱动等）的逻辑放在 Node.js Runtime 或后端服务；而将前置逻辑（鉴权、分流、个性化展示、代理）放在 Edge。

- 控制包体体积与依赖路径：保持 Edge 函数轻量化，避免引入过大的依赖，只在必要的路径上执行 Edge 逻辑。

- 优化流式输出：对于长时间响应（如大模型流式生成），应尽快返回 `Response(stream)`，在 25 秒内启动响应，并在 300 秒限制内完成任务。

- 关注平台参数：不同平台在执行时长、包体大小、节点覆盖等方面略有差异，应根据业务需求合理配置并关注更新。

## 总结

Edge Runtime 是一种轻量级、低延迟的运行时环境，让代码能够在距离用户最近的边缘节点执行，从而显著降低网络延迟并提升页面的首字节响应速度。它非常适合处理请求预处理、权限鉴权、动态渲染和流式返回等轻量逻辑，也能与 Serverless 架构配合完成快速、分布式的服务端计算。与传统的 CDN 不同，CDN 负责缓存和传输静态内容，而 Edge Function 则让代码在边缘运行、动态生成内容——一个是静态加速器，另一个是分布式计算平台。
