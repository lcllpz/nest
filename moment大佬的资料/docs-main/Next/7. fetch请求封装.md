> [源码地址](https://github.com/xun082/DocFlow/blob/main/src/services/request.ts)

在真实项目里，直接在页面里到处写：

```ts
const res = await fetch("", { method: "GET" });
const data = await res.json();
```

很快就会失控：

- 每个请求都要手动拼 URL、加 token、加 headers；

- 错误处理风格到处不同，有的 `try/catch`，有的直接不管；

- 业务 code（`code !== 0` 这类）要一遍遍写判断；

- token 过期、自动刷新、重试、SSE 这类高级逻辑更是无处安放。

在大部分中大型项目里，请求层都会再封一层抽象。就像我们现在这个 Request 类一样，统一提供给所有页面和业务代码使用。这样做还有一个额外的好处：当我们需要替换底层实现时，只要改这一层封装就够了，而不用去全项目地改调用代码。

比如哪天想把底层的 fetch 换成 axios，只需要在 Request 内部切换实现，对外暴露的 get/post 等方法完全不用动。

## 一、整体设计

我们先看最外层设计：

```ts
class Request {
  baseURL: string;
  defaultTimeout: number;
  defaultRetries: number;
  defaultRetryDelay: number;

  // 刷新 token 相关状态
  private maxRefreshAttempts = 2;
  private currentRefreshAttempts = 0;
  private isRefreshing = false;
  private refreshTokenPromise: Promise<string> | null = null;
  private failedQueue: QueuedRequest[] = [];

  constructor(
    baseURL: string,
    options?: { timeout?: number; retries?: number; retryDelay?: number }
  ) {
    this.baseURL = baseURL;
    this.defaultTimeout = options?.timeout || 10000;
    this.defaultRetries = options?.retries || 0;
    this.defaultRetryDelay = options?.retryDelay || 1000;
  }

  // ... 一堆方法 ...
}

const request = new Request(process.env.NEXT_PUBLIC_SERVER_URL || "", {
  timeout: 15000,
  retries: 1,
  retryDelay: 1000,
});
```

在这里的代码中我们做了几个重要的决策：

1. 用类来封装，而不是一堆工具函数，这样做的好处主要有以下几个方面：

   - 可以保存一些「跨请求的状态」：比如当前是否在刷新 token、失败队列等；

   - 可以有默认配置：`baseURL`、默认超时、默认重试次数；

   - 也可以根据需要创建多个实例，对接多个后端服务。

2. 我们把所有的全局请求策略都收敛在这一层封装中，包括超时控制、重试机制、token 过期后的自动刷新与请求排队、以及错误统一上报到 Sentry 等逻辑，从而让上层业务代码保持纯净、无感知。

之后，我们所有的 `get/post/put/delete/patch` 都只由这个类暴露出来，对外表现得就像一个小型的 HTTP SDK。

## 二、类型定义和统一数据结构

好的封装，第一步是把“数据结构”定清楚，只有先明确类型结构，后续的封装逻辑才能既严谨又可扩展——这也是 TypeScript 项目中最基础但最重要的一环。

### 2.1 HTTP 方法、请求参数类型

在封装请求工具时，不同接口的参数和配置虽然各不相同，但底层 `fetch` 都需要一个统一的结构。我们可以先定义出请求层的核心类型：

```ts
type Method = (typeof HTTP_METHODS)[keyof typeof HTTP_METHODS];

interface Params {
  cacheTime?: number; // Next.js 的缓存时间
  params?: Record<string, any>; // 业务参数
  timeout?: number; // 超时时间（毫秒）
  retries?: number; // 重试次数
  retryDelay?: number; // 重试间隔（毫秒）
  headers?: Record<string, string>; // 自定义请求头
  withCredentials?: boolean; // 是否携带 cookie
  signal?: AbortSignal; // 取消请求
  errorHandler?: ErrorHandler; // 自定义错误处理
}

interface Props extends Params {
  url: string;
  method: Method;
  mode?: RequestMode;
  token?: string; // 可覆盖默认的 auth_token
}
```

上面的代码设计思路可以这样理解：

- `Params`：抽象出所有请求的通用配置，比如超时、重试、缓存、请求头等，保证不同方法调用时参数一致、可复用。
- `Props`：在 `Params` 的基础上，补充 `url`、`method`、`token` 等必要字段，用于真正发起请求。
- `Method`：通过从 `HTTP_METHODS` 常量中提取类型，保证请求方法类型安全，避免拼写错误。

这一层类型定义相当于请求封装的“地基”，它让可变的部分（业务参数）与稳定的部分（请求结构）分离，既统一了接口格式，又确保后续在扩展功能或更换底层实现（如从 `fetch` 换成 `axios`）时，上层业务完全无感。

### 2.2 统一错误与统一响应格式

在请求封装中，统一错误与响应结构是让上层代码保持简洁和稳定的关键。我们不希望每个页面都去判断 `res.code` 或手动写 `try/catch`，所以这一层的目标是：让“错误处理”和“结果解析”在封装层就完成标准化。

```ts
export class RequestError extends Error {
  url: string;
  status?: number;
  statusText?: string;
  data?: any;
  ...
}

export interface ApiResponse<T> {
  code: number;
  message: string;
  data: T;
  timestamp: number;
}

export interface RequestResult<T> {
  data: ApiResponse<T> | null;
  error: string | null;
  status?: number;
}
```

设计思路解析

1. `RequestError`：用自定义错误类封装错误信息，替代裸的 Error('xxx')，同时携带 status、statusText、data 等上下文，便于统一上报（如 Sentry）和集中处理。

2. `ApiResponse<T>`：定义后端接口的标准返回结构（code、message、data、timestamp），让前端能以固定格式解析响应，避免各接口风格不一致。

3. `RequestResult<T>`：统一业务层接收结果的结构，成功时 `{ data: ApiResponse<T>, error: null }`，失败时 `{ data: null, error: '错误提示', status }`，让上层逻辑只需判断 error 是否为空即可。

通过这种设计，业务层只需简单判断：

```ts
if (result.error) {
  // 失败逻辑
} else {
  // 成功逻辑
}
```

甚至不需要做判断，直接渲染成功的结果，因为错误可以在具体的调用的时候再传入一个参数，这个参数可以根据不同的错误添加不同的页面交互提供给用户。

无需再到处写 `try/catch` 或判断 `res.code !== 0`，所有错误处理逻辑都被收敛在封装内部，使业务代码更加纯净、聚焦。

不过，要真正发挥这一层的作用，后端也必须保持接口返回结构一致，即所有接口都遵循统一的格式，如：

```json
{
  "code": 0,
  "message": "success",
  "data": { ... },
  "timestamp": 1731234567890
}
```

只有这样，前端才能依赖统一的结构去做自动化的错误解析、状态判断和上报，让整个前后端协作更稳健、可维护。

## 三、请求拦截

封装 fetch 的第一步，就是屏蔽所有底层请求细节，让业务层只关注 `发什么请求`，而无需考虑 `请求该怎么写`，比如：

- GET 请求是否要手动拼接 query？

- POST 请求 body 要不要自己 `JSON.stringify`？

这些都不应该让业务代码去操心，因此，我们在请求封装里定义了一个 `interceptorsRequest`，用于在请求真正发出前，自动处理各种公共逻辑：

```ts
interceptorsRequest({
  url,
  method,
  params,
  cacheTime,
  mode,
  token,
  headers: customHeaders,
  withCredentials,
}: Props) {
  let queryParams = '';
  let requestPayload: any = undefined;

  const headers: Record<string, string> = { ...customHeaders };

  // 1. 处理 token：参数优先，其次从 cookie 取
  const authToken = token || (typeof window !== 'undefined' ? getCookie('auth_token') : null);
  if (authToken) {
    headers['Authorization'] = `Bearer ${authToken}`;
  }

  // 2. 处理缓存策略（Next.js）
  const config: Config =
    cacheTime !== undefined
      ? cacheTime > 0
        ? { next: { revalidate: cacheTime } }
        : { cache: 'no-store' }
      : { cache: 'force-cache' };

  // 3. 区分 GET/DELETE 与其他方法
  if (method === HTTP_METHODS.GET || method === HTTP_METHODS.DELETE) {
    if (params) {
      queryParams = new URLSearchParams(
        Object.entries(params)
          .filter(([, v]) => v !== undefined)
          .map(([k, v]) => [k, String(v)]),
      ).toString();
      url = queryParams ? `${url}?${queryParams}` : url;
    }
  } else {
    if (params) {
      if (params instanceof FormData || params instanceof URLSearchParams) {
        requestPayload = params;
      } else {
        headers['Content-Type'] = headers['Content-Type'] || 'application/json';
        requestPayload = JSON.stringify(params);
      }
    }
  }

  return {
    url,
    options: {
      method,
      headers,
      mode,
      credentials: withCredentials ? HTTP_CREDENTIALS.INCLUDE : HTTP_CREDENTIALS.SAME_ORIGIN,
      body: requestPayload,
      ...config,
    },
  };
}
```

接下来我们来理解一下统一处理逻辑：

1. 请求会自动补上 `Authorization`，先用外部传入的 `token`，没有时再从 `cookie` 取，这样既能兼容 `SSR`，也能满足部分接口独立使用 `token` 的需求。

2. 统一缓存策略：根据 `cacheTime` 自动设置缓存模式，`> 0` 时使用 `{ next: { revalidate: cacheTime } }`，`= 0` 时禁用缓存（`no-store`），未设置则采用强制缓存（`force-cache`），在 Next.js 项目中能灵活控制服务端与客户端的缓存行为。

3. 根据请求方法自动处理参数，让业务层无需手动拼接或关心请求格式。

   - 对于 GET/DELETE：自动将 `params` 序列化并拼接到 URL。

   - 对于 POST/PUT/PATCH：将 `params` 处理为请求体：

     - 若为 `FormData` / `URLSearchParams`，直接透传；

     - 否则自动 `JSON.stringify`，并补齐 `Content-Type: application/json`。

经过这层拦截器，业务层的请求调用可以极简到：

```ts
request.get("/api/user", {
  params: { id: 123 },
});
```

我们后面就无需再关心诸如 `query` 是否需要拼接、`body` 是否要序列化、`header` 是否要补充或缓存如何配置等细节——所有这些通用逻辑都被集中封装在请求层中，业务代码只需专注于“要请求什么”，而非“这个请求该怎么写”。

## 四、响应后：统一处理 HTTP 状态与业务状态

在请求返回阶段，响应拦截器 `interceptorsResponse` 负责对服务端返回结果进行统一解析与判断。
它的核心目标是：无论错误来源于 HTTP 层还是业务逻辑层，都能以统一的结构被识别、封装并传递到上层，从而实现稳定、一致的请求结果处理。

设计思路如下流程图所示：

![20251110153809](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251110153809.png)

如下代码是一些实现示例：

```ts
async interceptorsResponse<T>(res: Response, url: string): Promise<T> {
  const status = res.status;
  const statusText = res.statusText;

  // 1. 处理 HTTP 状态码
  if (!res.ok) {
    let errorMessage = HTTP_STATUS_MESSAGES[status] || `HTTP 错误: ${status} ${statusText}`;
    let errorData = null;
    try {
      const contentType = res.headers.get('content-type');
      if (contentType?.includes('application/json')) {
        errorData = await res.json();
        if (errorData.message) errorMessage = errorData.message;
      } else {
        const textData = await res.text();
        if (textData) errorMessage = textData;
      }
    } catch {}
    throw new RequestError(errorMessage, url, status, statusText, errorData);
  }

  // 2. 处理非 JSON 响应
  const contentType = res.headers.get('content-type');
  if (contentType && !contentType.includes('application/json')) {
    return res as unknown as T;
  }

  // 3. 处理 JSON 响应
  try {
    const data = await res.json();
    if (
      data &&
      data.code !== undefined &&
      data.code !== 0 &&
      (data.code < 200 || data.code >= 300)
    ) {
      throw new RequestError(
        data.message || data.reason || '请求失败',
        url,
        status,
        statusText,
        data,
      );
    }
    return data as T;
  } catch (error) {
    if (error instanceof RequestError) throw error;
    throw new RequestError('解析响应数据失败', url, status, statusText);
  }
}
```

通过在响应层统一处理，所有异常都会被标准化为 `RequestError`，无论是网络超时、HTTP 状态错误还是业务逻辑失败，都能在同一套错误处理流程中被捕获、上报或提示。这样，业务代码只需判断是否存在 `error` 字段，而无需再区分问题来自网络层还是业务层，逻辑更清晰，维护也更轻松。

## 五、执行请求：超时控制 · 重试机制 · Sentry 追踪

`httpFactory` 承担了请求执行的核心职责，不仅封装了底层 `fetch` 的调用逻辑，还同时集成了超时控制、失败重试和 Sentry 埋点追踪，使整个请求链路更加可控、稳定且具备可观测性。

它的设计思路如下流程图所示：

![20251110160803](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251110160803.png)

如下实现逻辑示例：

```ts
async httpFactory<T>(props: Props): Promise<T> {
  const {
    url = '',
    params = {},
    method,
    mode,
    token,
    timeout = this.defaultTimeout,
    retries = this.defaultRetries,
    retryDelay = this.defaultRetryDelay,
    signal,
    ...rest
  } = props;

  const fullUrl = this.baseURL + url;

  // 1. Sentry breadcrumb：记录请求信息
  addSentryBreadcrumb({ ... });

  // 2. 预处理请求参数
  const req = this.interceptorsRequest({
    url: fullUrl,
    method,
    params: params.params,
    cacheTime: params.cacheTime,
    mode,
    token,
    headers: rest.headers,
    withCredentials: rest.withCredentials,
  });

  // 3. 构造实际请求函数
  const makeRequest = async (): Promise<T> => {
    const fetchPromise = fetch(req.url, { ...req.options, signal });
    let res: Response;

    // 3.1 超时控制
    if (timeout) {
      const timeoutPromise = this.createTimeoutPromise(timeout);
      res = await Promise.race([fetchPromise, timeoutPromise]);
    } else {
      res = await fetchPromise;
    }

    addSentryBreadcrumb({ ...status: res.status... });

    return this.interceptorsResponse<T>(res, fullUrl);
  };

  // 4. 包装为可重试的请求函数
  const requestFn = async () => {
    try {
      return await makeRequest();
    } catch (error) {
      addSentryBreadcrumb({ ...level: 'error', data: { error: String(error) } });
      throw error;
    }
  };

  // 5. 执行带重试与 token 刷新的请求
  return this.executeWithRetry(requestFn, retries, retryDelay);
}
```

通过在执行层统一封装超时控制、重试机制与日志追踪，请求的整个生命周期被彻底标准化——具备可控性（timeout）、可恢复性（retry）与可观测性（Sentry）。业务层无需再关心重试策略或日志收集的细节，只需调用 `request.get/post()`，即可完成一条稳定、可追踪且可维护的请求链路。

> 后续如果对大规模重试流量更敏感，还可以将当前的固定间隔重试演进为指数退避（Exponential Backoff）：每次重试的等待时间按 2 的倍数递增（例如 500 ms → 1 s → 2 s → 4 s）。在此基础上再叠加少量随机抖动（jitter），可以有效避免大量客户端在同一时间点集中重试，从而进一步减轻服务端压力。

## 六、核心：带 token 刷新的重试机制

`executeWithRetry` 是整个封装的大脑之一，用来在一次调用里自动兜底各种失败场景：

```ts
async executeWithRetry<T>(
  requestFn: () => Promise<T>,
  retries: number,
  retryDelay: number,
  isRetryAfterRefresh: boolean = false,
): Promise<T> {
  try {
    return await requestFn();
  } catch (error) {
    // 1. 处理 401：尝试刷新 token
    if (error instanceof RequestError && error.status === 401 && !isRetryAfterRefresh) {
      if (typeof window !== 'undefined') {
        if (!getCookie('refresh_token')) {
          throw error;
        }

        try {
          if (this.isRefreshing) {
            return this.addRequestToQueue(requestFn);
          }

          await this.handleTokenRefresh();

          return this.executeWithRetry(requestFn, 0, 0, true);
        } catch {
          throw error;
        }
      } else {
        throw error;
      }
    }

    // 2. 其他错误，根据 retries 重试
    if (retries > 0 && !(error instanceof RequestError && error.status === 401)) {
      await new Promise((resolve) => setTimeout(resolve, retryDelay));
      return this.executeWithRetry(requestFn, retries - 1, retryDelay, isRetryAfterRefresh);
    }

    throw error;
  }
}
```

整体流程其实非常简单：请求执行失败并返回 401 时，系统会优先尝试使用本地保存的 `refresh_token` 获取新的访问令牌（`access_token`）；若刷新成功，则重新发起原请求并返回结果；若 `refresh_token` 也已过期或无效，则直接抛出异常，引导用户重新登录。这一机制保证了在认证过期的情况下，用户请求能自动恢复，无需业务层手动干预。

对于非 401 的错误，若仍有剩余重试次数，系统会在设定的 `retryDelay` 后自动再次发起请求；当重试次数耗尽或错误类型不再适合重试时，则直接抛出异常。通过这种机制，短暂的网络抖动或服务波动都能被自动修复，业务层只需接收最终结果或标准化错误，无需关心中间的失败与重试过程。

## 七、token 刷新与请求队列：解决并发 401 的优雅方式

为了支撑重试逻辑，封装中实现了一套完善的“Token 刷新 + 请求队列”机制。

其中，`refreshAccessToken` 专职负责刷新流程：它绕过统一的 `httpFactory`，直接使用原生 `fetch` 调用刷新接口，携带 `refresh_token` 获取新的访问令牌。刷新成功后将新的 Token 写入 Cookie，失败则清除本地认证信息并引导用户重新登录。

整个刷新过程被完全隔离成一个独立的“认证行为”，与普通业务请求解耦，不会产生相互干扰。接下来由 `handleTokenRefresh` 统一调度刷新过程并驱动等待队列，确保多请求并发场景下的刷新行为安全、有序。

```ts
private async handleTokenRefresh(): Promise<string> {
  if (this.isRefreshing && this.refreshTokenPromise) {
    return this.refreshTokenPromise;
  }

  if (this.currentRefreshAttempts >= this.maxRefreshAttempts) {
    throw new RequestError('刷新令牌次数过多，请重新登录', '', 401, 'Unauthorized');
  }

  this.isRefreshing = true;
  this.currentRefreshAttempts += 1;

  this.refreshTokenPromise = this.refreshAccessToken()
    .then((newToken) => {
      this.isRefreshing = false;
      this.refreshTokenPromise = null;
      this.currentRefreshAttempts = 0;
      this.processQueue(null);
      return newToken;
    })
    .catch((error) => {
      this.isRefreshing = false;
      this.refreshTokenPromise = null;
      this.processQueue(error);
      throw error;
    });

  return this.refreshTokenPromise;
}
```

`handleTokenRefresh` 通过 `isRefreshing` 与 `refreshTokenPromise` 确保同一时刻只会触发一次 Token 刷新，其他同时返回 401 的请求会被挂起等待结果。同时利用 `currentRefreshAttempts` 与 `maxRefreshAttempts` 控制最大刷新次数，避免陷入无限重试。刷新成功后调用 `processQueue(null)` 重新执行所有挂起请求，失败则调用 `processQueue(error)` 统一失败回调。

而 `addRequestToQueue` 则负责将待重试的请求封装进队列：

```ts
private addRequestToQueue(requestFn: () => Promise<any>): Promise<any> {
  return new Promise((resolve, reject) => {
    this.failedQueue.push({ resolve, reject, requestFn });
  });
}
```

这意味着在并发 401 的场景下，只有第一个请求会真正触发刷新操作，其余请求都会静静排队等待结果，刷新成功后自动重放，失败则统一抛错。整个过程对业务层完全透明，实现了高并发下安全、优雅的 Token 刷新机制。

## 八、统一错误处理：全局兜底 + 自定义扩展点

当请求封装已经具备了超时、重试、Token 刷新等能力后，最后一步就是构建统一的错误处理机制。这一层既要能在全局范围内兜底各种异常，又要为业务侧提供可插拔的自定义扩展点。

在这里，所有的异常都会被集中处理并分类：

- 对于封装内部抛出的 `RequestError`，会判断是否需要上报到 Sentry（例如 401、404 这类可预期错误可以跳过），并根据状态码调用相应的处理器，如 `unauthorized`、`forbidden` 或 `serverError`；

- 对于网络错误（`Failed to fetch`），会统一上报到 Sentry，并返回提示文案“网络连接错误，请检查您的网络”；

- 如果是用户主动取消请求（`AbortError`），则被视为正常行为，不进行上报；

- 其余未知错误，则作为兜底情况统一上报，并返回默认错误提示，防止异常遗漏。

随后，通过 `handleRequest` 方法将这些错误转换为统一的数据结构 `{ data, error, status }`，从而让调用方始终以同样的方式处理请求结果：

```ts
async handleRequest<T>(
  requestFn: () => Promise<ApiResponse<T>>,
  errorHandler?: ErrorHandler,
): Promise<RequestResult<T>> {
  try {
    const data = await requestFn();
    return { data, error: null };
  } catch (error) {
    const handlers =
      typeof errorHandler === 'function' ? { default: errorHandler } : errorHandler;

    const errorMessage = this.handleRequestError(error, undefined, {
      ...(handlers as any),
      default: handlers?.default || handlers?.onError,
    });

    if (typeof errorHandler === 'function') {
      errorHandler(error);
    } else if (errorHandler?.onError) {
      errorHandler.onError(error);
    }

    const status = error instanceof RequestError ? error.status : undefined;

    return { data: null, error: errorMessage, status };
  }
}
```

成功时，直接返回 `{ data, error: null }`；失败时，无论是网络异常、业务错误还是手动抛出的 `RequestError`，都被格式化成 `{ data: null, error, status }`。同时，调用方可通过传入自定义的 `errorHandler`，在特定业务场景中覆写默认逻辑，实现差异化处理。

通过这一层的设计，所有请求的异常都能被`标准化`、`集中化`、`可观测`地处理。业务代码只需判断是否存在 `error`，即可完成错误分支逻辑，不再需要额外的 `try/catch` 或复杂的错误分流。
最终，整个请求链路实现了从发起、响应到异常处理的完整闭环，既统一又灵活。

## 九、对业务暴露的最终 API

走到这一步，请求层的封装已经完成所有底层逻辑——包括参数处理、响应解析、Token 刷新、错误兜底与日志上报。
而对业务层而言，只需要暴露几个最直观的 HTTP 方法即可：

```ts
get<T>(url: string, params?: Params, mode?: RequestMode, token?: string) {
  return this.handleRequest<T>(
    () => this.internalRequest<T>('GET', url, params, mode, token),
    params?.errorHandler,
  );
}

post<T>(url: string, params?: Params, mode?: RequestMode, token?: string) {
  return this.handleRequest<T>(
    () => this.internalRequest<T>('POST', url, params, mode, token),
    params?.errorHandler,
  );
}

// 其它 put/delete/patch 类似
```

其中，`internalRequest` 只是一层极薄的包装，用于衔接上层调用与底层的 `httpFactory`：

```ts
private async internalRequest<T>(
  method: Method,
  url: string,
  params?: Params,
  mode?: RequestMode,
  token?: string,
): Promise<ApiResponse<T>> {
  return this.httpFactory<ApiResponse<T>>({
    url,
    params,
    method,
    mode,
    token,
    timeout: params?.timeout,
    retries: params?.retries,
    retryDelay: params?.retryDelay,
    signal: params?.signal,
    headers: params?.headers,
    withCredentials: params?.withCredentials,
  });
}
```

通过这一层抽象，业务开发者不再需要关心任何底层实现，只需像写普通函数一样调用：

```ts
const res = await request.get<User>("/api/v1/user/info", {
  params: { params: { id: 1 } },
});

if (res.error) {
  // 显示错误信息
} else {
  // 真正的数据在 res.data!.data
}
```

这种封装的核心价值在于，让业务调用保持一致、简洁且可预测。无论使用 `GET` 还是 `POST`，都能获得相同的 `{ data, error, status }` 结构，而底层的超时控制、重试机制、Token 刷新与错误上报都被透明地封装在内部，从而大幅提升开发效率并降低维护成本。

## 总结

这套基于 `fetch` 的封装，将请求从构建到返回的每一个环节——包括参数整理、响应解析、超时与重试控制、Token 刷新以及统一错误处理——完整串联成一个高内聚的闭环体系。

通过类的方式集中管理状态与配置，它让请求行为在不同项目中都能保持一致性与可预测性。

无论面对网络异常、认证过期还是业务错误，系统都能自动兜底、重试并上报，业务层仅需处理结构化的 `{ data, error, status }` 结果。

更重要的是，这种封装几乎不依赖具体框架，能无缝迁移到任意项目中复用，仅需替换基础配置或底层请求实现即可。

它让请求调用的体验像使用 SDK 一样简洁，却具备企业级 HTTP 客户端的健壮性与扩展性。
