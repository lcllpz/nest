Partial Prerendering（部分预渲染）是一种渲染策略，它允许我们在同一个路由中结合静态内容和动态内容。这样既能提升页面的初始加载性能，又能支持个性化的动态数据。

## 基本使用

要在 NextJs 中开启 PPR 模式，首先你得确保你的 NextJs 版本带有 `canary` 关键字，例如 `15.1.1-canary.17`,否则会报这样的错误：

![20250711150844](https://raw.githubusercontent.com/xun082/md/main/blogs.images20250711150844.png)

确定版本没有问题了，我们需要在 `next.config.js` 文件中开启 PPR:

```ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  experimental: {
    ppr: "incremental", // 启用 PPR 模式
  },
};

export default nextConfig;
```

`incremental` 是目前唯一可用的值，表示启用“增量 PPR”，只对特定页面启用（不是全局默认启用）。

我们需要在对应的 layout 或者 page 组件中显式声明启用：

```tsx
export const experimental_ppr = true;

export default function Layout({ children }: { children: React.ReactNode }) {
  return <>{children}</>;
}
```

如果我们在父级启用了 PPR，但想再子路径中关闭它，可以这样子操作：

```tsx
export const experimental_ppr = false;
```

## PPR 工作原理

使用 `静态渲染（Static Rendering）` 时，HTML 会提前生成 —— 可以是在构建时生成，也可以通过再验证（revalidation）机制生成。生成的结果会被缓存，并在多个用户和请求之间共享。

在 部分预渲染（Partial Prerendering） 中，Next.js 会为某个路由预渲染一个静态的页面壳（shell）。这个壳可以包含布局（layout）以及其他不依赖于请求时数据的组件。

在使用 `动态渲染（Dynamic Rendering）` 时，HTML 是在 请求时生成的。这让我们可以基于请求时的数据提供个性化的内容。

一个组件如果使用了以下 API，就会变成动态组件：

1. `cookies`

2. `headers`

3. `connection`

4. `draftMode`

5. `searchParams 属性`

6. `unstable_noStore`

7. `fetch({ cache: 'no-store' })`

在 `Partial Prerendering` 中，若组件使用了上述这些 API，会触发一个特殊的 React 错误，告诉 `Next.js` 该组件不能被静态渲染，构建时会报错。

对于这种组件，我们可以使用一个 `Suspense` 包裹这个组件，让它在运行时再去渲染（即延迟渲染）。在 `Partial Prerendering` 中，`Suspense` 被用来标记动态边界（dynamic boundaries），即组件树中哪些部分是需要延迟渲染的。

## 从构建目录了解它的工作原理

首先我们编写一个简单的 Demo，如下代码所示：

```tsx
import { Suspense } from "react";

export const experimental_ppr = true;

async function SlowComponent(props: { delay: number }) {
  await new Promise((resolve) => setTimeout(resolve, props.delay));
  return <div>Component blocked for {props.delay}ms</div>;
}

async function Content() {
  return (
    <div>
      <SlowComponent delay={1000} />
      <SlowComponent delay={2000} />
      <SlowComponent delay={5000} />
    </div>
  );
}

export default function Home() {
  return (
    <div>
      <h1>Example Header</h1>
      <Suspense fallback={<div>loading...</div>}>
        <Content />
      </Suspense>
    </div>
  );
}
```

在上面的图片中我们可以看到完整的信息，接下来我们将大概讲解一下它们分别代表什么：

### `4.html`

`4.html` 是这个页面的 静态 HTML 壳，在构建阶段生成，内容包括如下部分：

1. 静态部分，例如布局、Header、静态文本；

   - `Suspense` 动态组件的位置会被替换成：

   - `<template>`：组件实际内容的位置（空占位）；

2. `fallback`：例如 loading... 占位内容；

页面初次请求时，服务器直接把这份 HTML 返回给浏览器，实现快速首屏渲染（First Paint）。

### 4.prefetch.rsc

`4.prefetch.rsc` 是页面中被 `Suspense` 包裹的动态组件所对应的 服务器组件（Server Component）渲染结果片段。它并不是传统意义上的 JSON 数据或完整 HTML，而是符合 React Server Components 协议的一种 RSC 流格式，通常以二进制或特殊结构传输。

如下图所示：

![20250711163456](https://raw.githubusercontent.com/xun082/md/main/blogs.images20250711163456.png)

当浏览器首次加载 `4.html` 时，会优先渲染其中构建阶段生成的静态内容（如布局和头部），而被 `Suspense` 包裹的动态部分此时只显示 fallback 占位符。

接着，浏览器根据页面中嵌入的指令发起对 `4.prefetch.rsc` 的请求，获取由服务端流式生成的动态组件数据。这个 `.rsc` 文件并不是普通的 JSON，而是 React Server Components 协议定义的一种结构化数据格式，包含组件树、props、静态资源引用等信息。浏览器在接收 `.rsc` 数据的过程中，会由 React 运行时逐步解析并替换掉页面中的 fallback，从而实现动态内容的流式注入和页面的渐进拼装。

可以说，`4.prefetch.rsc` 是连接静态 HTML 与运行时动态渲染之间的关键桥梁，是 Partial Prerendering 实现快速首屏与动态更新的核心机制之一。

### `4.meta`

`4.meta` 是描述 `/example/4` 页面构建产物状态的元数据文件，它包含了该页面的 HTTP 状态、资源预加载、缓存策略、构建依赖路径、以及延迟渲染（Postponed）的组件段落信息。

这个文件在运行时帮助 Next.js 精准控制页面的缓存行为、Streaming 边界恢复、以及按需更新机制，是 Partial Prerendering 能正常运行的重要调度信息载体。

![20250711163955](https://raw.githubusercontent.com/xun082/md/main/blogs.images20250711163955.png)

### `page.js.nft.json`

`page.js.nft.json` 是 Next.js 在构建阶段生成的模块依赖追踪文件（Nested File Trace），它列出了页面入口文件 `page.js` 在渲染过程中所依赖的所有模块、Node 包、chunk 文件以及运行时资源。这个文件不会被发往客户端，而是为服务端运行环境（如 Vercel 的 Edge Function 或 Node.js Server）提供打包、缓存和部署策略的依据。

其中的 `files` 列表详细列出所有依赖的物理路径，包括 `node_modules` 中的框架库（如 React、Next.js）、项目自身的配置文件（如 `package.json`）、以及构建生成的 `webpack chunk` 等。它确保在 Serverless 场景下，部署平台可以根据这个清单将渲染某个页面所需的最小文件集打包进函数运行环境，从而提升冷启动效率和缩小体积。

此外，这个文件还能帮助构建系统判断依赖变更，进而实现增量构建与缓存复用。

在使用 Partial Prerendering（PPR） 和 `React Server Components（RSC）` 的项目中，`page.js.nft.json` 作为构建系统的底层元数据文件，是实现按需部署、流式渲染支持和边缘计算兼容的关键支撑之一。它从底层确保页面运行时的完整性和效率，是 Next.js 构建产物中极为重要的一环。

![20250711165937](https://raw.githubusercontent.com/xun082/md/main/blogs.images20250711165937.png)

### `page_client-reference-manifest.js`

`page_client-reference-manifest.js` 是 Next.js 在构建阶段为 `React Server Components` 架构生成的客户端模块映射清单，它记录了某个页面中所有被标记为 Client Component 的模块信息。该文件会告诉浏览器：哪些组件必须在客户端执行、它们的模块 ID、导出名称、所属路径、是否启用了懒加载（`async: true`）、以及关联的 JavaScript chunk 文件。

这个文件的结构通常包含 `ssrModuleMapping` 和 `clientModules` 两个核心部分：前者用于从服务端渲染时识别哪些组件将被浏览器接管；后者则为客户端提供实际加载这些组件所需的元数据（比如路径、chunk 名称等）。在页面加载后，React runtime 会读取这个 manifest，并据此从 CDN 或本地静态资源目录中加载相应模块，完成 `hydrate` 或事件绑定。

特别是在 Partial Prerendering（PPR） 模式下，页面最初是由服务端返回一个静态壳（HTML shell），其中的动态组件被 `Suspense` 占位。随后，当这些组件需要被懒加载或激活时，浏览器就会查阅 `page_client-reference-manifest.js`，加载并挂载对应的 JS 逻辑，实现完整的交互行为。

因此，这个 manifest 文件是客户端和服务端协作的桥梁，它让 React 知道哪些模块需要从“静态标记”变为“可交互组件”，是 RSC 和 PPR 模式下支撑“渐进式拼装 + 懒加载运行”的关键组成部分之一。

### 一个流程图搞清楚执行原理

Partial Prerendering（PPR）在构建阶段会将页面中的静态部分渲染为 HTML shell，并将动态组件序列化为 `.rsc` 数据文件。

用户发起请求时，如果项目部署环境中有使用 CDN 或边缘节点（如 Vercel Edge），静态 HTML shell 可以被优先缓存并返回，提升首屏加载速度。浏览器在渲染 shell 的同时展示 `Suspense` 的 fallback 占位，而服务端则并行启动 RSC 渲染流程，通过流式（Streaming）方式返回动态内容片段。

浏览器逐步接收这些内容并替换占位区域，React 运行时完成组件拼装与 UI 更新。最终，客户端加载所需的 JS 模块，完成事件绑定与页面交互的 hydration。

## 总结

Partial Prerendering（PPR）是 Next.js 在 `React Server Components` 基础上引入的一种混合渲染模式，它将页面拆分为静态和动态两部分，在构建时预渲染静态内容，同时将动态组件延迟到请求时再生成。

相比传统的 SSG（静态生成），PPR 保留了构建时生成 HTML 的快速首屏优势，但不再受限于“全页面必须静态”的约束。与 SSR（服务端渲染）相比，PPR 减少了每次请求都要从头渲染整个页面的成本，显著提升了性能和缓存利用率。

相较于 ISR（增量静态生成），PPR 不依赖固定的 revalidate 时间，而是通过 `Suspense` 和 streaming 实现真正的“动态按需渲染”。它允许开发者在一个页面中自由组合构建时与运行时逻辑，从而兼顾了 SSG 的性能和 SSR 的灵活性。

最终，PPR 提供了更快的首屏加载、更细粒度的渲染控制以及更优秀的用户感知体验，是现代应用性能与动态性平衡的理想方案。

PPR 还保留了 SSG 的静态缓存优势，同时通过组件级动态加载避免 SSR 的全量重渲染，带来了更细粒度、更高命中率、更具弹性的缓存策略，是目前最“缓存友好”的混合渲染模式之一。
