在前端里，网络请求方式主要是 XHR 和 fetch。而在 Next.js 中，官方更推荐使用 `fetch`，因为它不仅是标准的 Web API，还与框架的 数据缓存、重验证、去重和 Edge Runtime 深度集成，能直接享受框架级的性能优化和一致的运行时支持。

## `fetch` 被 Next.js 扩展，直连框架缓存与重验证

Next.js 在原生 `fetch()` 的基础上进行了扩展，让你在服务端请求时就能定义 持久缓存和重验证策略（这里指的是 Next.js 的 Data Cache，不是浏览器缓存）。因此你会看到一些特有的选项，比如 `cache`、`next.revalidate` 和 `next.tags`。

详情可以参见 [NextJs 官方文档](https://nextjs.org/docs/app/api-reference/functions/fetch)

通过这些扩展，Next.js 可以让请求结果具备生命周期和标签化管理：

- `revalidate`：设置数据过期时间（单位秒）；

- `revalidateTag` / `revalidatePath`：根据标签或路径精确刷新缓存。

这些特性是 `axios.get(...)` 无法自动集成的。若要在 Axios 中实现类似效果，你得自行搭建缓存层和失效机制。而在 Next.js 中，`fetch` 已经与框架的缓存与重验证系统紧密耦合，开箱即用。

## 与 React/Next 渲染管线和 Edge Runtime 的原生协作

在 Server Components 中，你可以直接使用 `await fetch`，Next.js 会自动应用缓存、请求去重等优化机制。
即使在 `generateMetadata` 或 `generateStaticParams` 等预渲染函数中，`fetch` 也会被自动记忆化（memoize），避免多处重复请求，从而与 RSC 的流式渲染、静态生成和预渲染策略自然协作。

同时，`fetch` 也是 Edge Runtime 的通用数据获取方案。由于 Edge 基于 Web 标准 API 的子集（仅支持如 `fetch`、`Request`、`Response` 等接口，而非完整的 Node API），像 Axios 这类依赖 Node 能力的库无法在 Edge 环境运行。相比之下，`fetch` 在 Edge、Node 和浏览器三端均可无缝使用，因此成为官方推荐的统一选择。

## 总结

`fetch` 是浏览器与 Node.js 原生支持的 Web API，在 Next.js 中开箱即用，无需任何额外安装。
它被框架扩展后，可直接参与 Next.js 的数据缓存、重验证和去重机制，并与 Edge Runtime 完全兼容。
相比之下，Axios 需要额外安装依赖包，且无法自动接入这些框架级优化，因此官方更推荐使用 `fetch` 作为默认的数据请求方案。
