在 Next.js 16 的 App Router 中，文件路由系统通过特殊文件约定（如 `layout.js`、`page.js`）和路由段配置来控制页面的渲染行为、缓存策略和运行时环境。本文将详细介绍这些核心概念和使用方法。

## 1. 段（Segment）的概念

在 Next.js 官方文档中，经常会看到 `Segment`（段）这个概念。理解段的概念对于掌握 Next.js 的路由系统至关重要。

`URL Segment` 指的是由斜杠（`/`）分隔的 URL 路径的一部分，而 `URL Path` 是域名后面的完整 URL 部分。例如，对于 URL `https://example.com/dashboard/settings`，URL Path 是 `/dashboard/settings`，URL Segments 是 `dashboard` 和 `settings`。

在 Next.js 的 App Router 中，`Route Segment`（路由段）指的是 `app` 目录中的每个文件夹，每个文件夹都代表一个路由段，并映射到对应的 URL Segment。

例如，对于文件结构：

```
app/
  dashboard/
    settings/
      page.js
```

对应的路由为 `/dashboard/settings`，其中：

- `/`：根段（Root Segment）

- `dashboard`：路由段（Route Segment）

- `settings`：叶段（Leaf Segment，没有子节点的段）

每个路由段都可以包含特殊文件（如 `page.js`、`layout.js` 等）来定义该路由段的行为。

## 2. 路由组（Route Groups）

路由组是一种文件夹约定，允许我们按类别或团队组织路由。路由组不会影响 URL 路径，仅用于组织文件结构。

路由组主要用于按团队、关注点或功能组织路由，将不同团队负责的路由分开组织，按功能模块组织路由（如管理后台、营销页面等）。此外，路由组还可以用于定义多个根布局，为不同的路由组创建不同的根布局，例如管理后台和营销网站可以使用完全不同的布局结构。同时，路由组也支持选择性共享布局，让特定路由段共享一个布局，同时保持其他路由段独立。

### 2.1. 基本概念

路由组通过将文件夹名称用括号包裹来创建：`(folderName)`。这个约定表示该文件夹仅用于组织目的，**不会包含**在路由的 URL 路径中。

![20251209151709](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251209151709.png)

例如，对于以下文件结构：

```
app/
  (admin)/
    dashboard/
      page.js
  (marketing)/
    about/
      page.js
    blog/
      page.js
```

对应的路由为：

- `/dashboard`（来自 `app/(admin)/dashboard/page.js`）

- `/about`（来自 `app/(marketing)/about/page.js`）

- `/blog`（来自 `app/(marketing)/blog/page.js`）

注意：路由组文件夹 `(admin)` 和 `(marketing)` 不会出现在 URL 路径中。

### 2.2. 多个根布局示例

使用路由组可以创建多个根布局。每个路由组可以有自己的根布局：

```
app/
  (admin)/
    layout.js  // 管理后台的根布局
    dashboard/
      page.js
  (marketing)/
    layout.js  // 营销网站的根布局
    about/
      page.js
    blog/
      page.js
```

在这个例子中：

- `app/(admin)/layout.js` 是管理后台的根布局

- `app/(marketing)/layout.js` 是营销网站的根布局

- 每个根布局都需要定义 `<html>` 和 `<body>` 标签

## 3. 特殊文件约定

在 Next.js 的 App Router 中，有一些特殊的文件约定用于定义页面、布局和处理特定场景。本节我们将详细介绍 `layout.js`、`page.js` 和 `default.js` 这三个核心文件。

### 3.1. layout.js

`layout.js` 文件用于定义应用的布局结构。它允许我们创建嵌套布局，以便在不同的路由下复用相同的布局组件。

#### 基本用法

```tsx
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <section>
      <nav>导航栏</nav>
      <main>{children}</main>
    </section>
  );
}
```

#### 根布局（Root Layout）

根布局是 `app` 目录中最顶层的布局，用于定义 `<html>` 和 `<body>` 标签以及其他全局共享的 UI。

```tsx
// app/layout.tsx
export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="zh-CN">
      <body>{children}</body>
    </html>
  );
}
```

其中值得注意的是：

- 根布局必须定义 `<html>` 和 `<body>` 标签

- 不应该手动添加 `<head>` 标签（如 `<title>` 和 `<meta>`），而应该使用 [Metadata API](https://nextjs.org/docs/app/getting-started/metadata-and-og-images) 来自动处理

- 可以使用[路由组](https://nextjs.org/docs/app/api-reference/file-conventions/route-groups)创建 `多个根布局`（详见[路由组章节](#2-路由组route-groups)）

- 在多个根布局之间导航会导致 `完整页面加载`（而不是客户端导航）

#### Props

##### children（必需）

Layout 组件必须接受并使用 `children` prop。在渲染时，`children` 会被填充为布局所包裹的路由段组件，主要是子 [Layout](https://nextjs.org/docs/app/api-reference/file-conventions/layout) 或 [Page](https://nextjs.org/docs/app/api-reference/file-conventions/page) 组件，但也可能是其他特殊文件，如 [Loading](https://nextjs.org/docs/app/api-reference/file-conventions/loading) 或 [Error](https://nextjs.org/docs/app/getting-started/error-handling)（在适用时）。

##### params（可选）

`params` 是一个 Promise，解析后包含从根段到该布局的动态路由参数对象。

```tsx
// app/dashboard/[team]/layout.tsx
export default async function Layout({
  children,
  params,
}: {
  children: React.ReactNode;
  params: Promise<{ team: string }>;
}) {
  const { team } = await params;
  return (
    <section>
      <h1>团队: {team}</h1>
      {children}
    </section>
  );
}
```

| 路由示例                          | URL            | `params`                           |
| --------------------------------- | -------------- | ---------------------------------- |
| `app/dashboard/[team]/layout.js`  | `/dashboard/1` | `Promise<{ team: '1' }>`           |
| `app/shop/[tag]/[item]/layout.js` | `/shop/1/2`    | `Promise<{ tag: '1', item: '2' }>` |
| `app/blog/[...slug]/layout.js`    | `/blog/1/2`    | `Promise<{ slug: ['1', '2'] }>`    |

`注意`：在 Next.js 16 中，`params` 是一个 Promise，必须使用 `async/await` 或 React 的 [`use`](https://react.dev/reference/react/use) 函数来访问其值。

#### Layout Props Helper

我们可以使用 `LayoutProps` 辅助函数为布局添加强类型的 `params` 和从目录结构推断的命名 slot。类型在 `next dev`、`next build` 或 `next typegen` 期间自动生成，无需导入。

```tsx
// app/dashboard/layout.tsx
export default function Layout(props: LayoutProps<"/dashboard">) {
  return (
    <section>
      {props.children}
      {/* 如果有 app/dashboard/@analytics，会作为类型化的 slot 出现 */}
      {/* {props.analytics} */}
    </section>
  );
}
```

Layout 在导航期间会被客户端缓存和重用，因此有以下限制：

- `请求对象`：Layout 不能访问原始请求对象，需要使用 [`headers`](https://nextjs.org/docs/app/api-reference/functions/headers) 和 [`cookies`](https://nextjs.org/docs/app/api-reference/functions/cookies) API

- `查询参数`：无法访问搜索参数，应在 Page 中使用 [`searchParams`](https://nextjs.org/docs/app/api-reference/file-conventions/page#searchparams-optional) 或客户端使用 [`useSearchParams`](https://nextjs.org/docs/app/api-reference/functions/use-search-params)

- `路径名`：无法访问路径名，应在客户端使用 [`usePathname`](https://nextjs.org/docs/app/api-reference/functions/use-pathname)

- `子路由段`：无法访问下方路由段，应在客户端使用 [`useSelectedLayoutSegment`](https://nextjs.org/docs/app/api-reference/functions/use-selected-layout-segment) 或 [`useSelectedLayoutSegments`](https://nextjs.org/docs/app/api-reference/functions/use-selected-layout-segments)

### 3.2. page.js

`page.js` 文件用于定义路由的页面组件。每个路由段都应该有一个对应的 `page.js` 文件，用于使路由可公开访问。

#### 基本用法

```tsx
// app/page.tsx
export default function HomePage() {
  return <h1>欢迎来到首页</h1>;
}
```

#### Props

##### params（可选）

`params` 是一个 Promise，解析后包含从根段到该页面的动态路由参数对象。

```tsx
// app/blog/[slug]/page.tsx
export default async function Page({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  return <div>文章: {slug}</div>;
}
```

| 路由示例                        | URL                 | `params`                           |
| ------------------------------- | ------------------- | ---------------------------------- |
| `app/blog/[slug]/page.js`       | `/blog/hello-world` | `Promise<{ slug: 'hello-world' }>` |
| `app/shop/[tag]/[item]/page.js` | `/shop/1/2`         | `Promise<{ tag: '1', item: '2' }>` |
| `app/blog/[...slug]/page.js`    | `/blog/1/2`         | `Promise<{ slug: ['1', '2'] }>`    |

##### searchParams（可选）

`searchParams` 是一个 Promise，解析后包含当前 URL 的搜索参数对象。在客户端组件中应使用 [`useSearchParams`](https://nextjs.org/docs/app/api-reference/functions/use-search-params) hook。

```tsx
// app/search/page.tsx
export default async function SearchPage({
  searchParams,
}: {
  searchParams: Promise<{ [key: string]: string | string[] | undefined }>;
}) {
  const { q } = await searchParams;
  return <div>搜索: {q}</div>;
}
```

#### 在客户端组件中使用 params

要在客户端组件（不能是 `async`）中使用 `params`，我们可以使用 React 的 [`use`](https://react.dev/reference/react/use) 函数来读取 Promise：

```tsx
// app/blog/[slug]/page.tsx
"use client";

import { use } from "react";

export default function Page({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = use(params);
  return <div>文章: {slug}</div>;
}
```

#### Page Props Helper

我们可以使用 `PageProps` 辅助函数为页面添加强类型的 `params` 和 `searchParams`。类型会在构建时自动生成，无需导入。

```tsx
// app/blog/[slug]/page.tsx
export default async function Page(props: PageProps<"/blog/[slug]">) {
  const { slug } = await props.params;
  const { q } = await props.searchParams;
  return (
    <div>
      文章: {slug}, 搜索: {q}
    </div>
  );
}
```

### 3.3. default.js

`default.js` 文件用于在并行路由（Parallel Routes）中，当 Next.js 无法恢复某个 slot（插槽）的活跃状态时，渲染一个回退 UI。

#### 并行路由基础

并行路由允许我们同时渲染一个或多个页面，这些页面可以在同一个视图中独立导航。在并行路由中，使用 `@folder` 语法来定义 slot（插槽）。

#### default.js 的作用

在客户端导航（软导航）时，Next.js 会跟踪每个 slot 的活跃状态（子页面）。但在硬导航（完整页面加载，如刷新页面）时，Next.js 无法恢复这些活跃状态。在这种情况下，`default.js` 文件会被渲染，用于显示那些不匹配当前 URL 的子页面。

考虑以下文件结构：

![20251209150244](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251209150244.png)

```
app/
  @team/
    settings/
      page.js
  @analytics/
    default.js
    page.js
  default.js
  layout.js
  page.js
```

在这个结构中，`@team` slot 有一个 `settings` 页面，但 `@analytics` 没有。

当导航到 `/settings` 时：

- `@team` slot 会渲染 `settings` 页面

- `@analytics` slot 会保持当前活跃的页面

但在刷新页面时，Next.js 无法恢复 `@analytics` slot 的活跃状态，此时会渲染 `@analytics/default.js`。如果 `default.js` 不存在，对于命名 slot（如 `@team`、`@analytics` 等）会返回错误，要求我们必须定义 `default.js` 才能继续。

#### 返回 404 的默认行为

如果我们希望保持旧的行为（在这些情况下返回 404），可以创建一个 `default.js` 文件，内容如下：

```tsx
// app/@team/default.js
import { notFound } from "next/navigation";

export default function Default() {
  notFound();
}
```

#### children slot 的 default.js

由于 `children` 是一个隐式的 slot，我们还需要为 `children` slot 创建 `default.js` 文件，用于在 Next.js 无法恢复父页面活跃状态时渲染回退 UI。如果不为 `children` slot 创建 `default.js`，路由会返回 404 页面。

#### params 参数

`default.js` 可以接收 `params` 参数，它是一个 Promise，解析后包含从根段到 slot 子页面的动态路由参数。

```tsx
// app/[artist]/@sidebar/default.js
export default async function Default({
  params,
}: {
  params: Promise<{ artist: string }>;
}) {
  const { artist } = await params;
  // 使用 artist 参数
}
```

| 文件路径                                   | URL          | `params`                                     |
| ------------------------------------------ | ------------ | -------------------------------------------- |
| `app/[artist]/@sidebar/default.js`         | `/zack`      | `Promise<{ artist: 'zack' }>`                |
| `app/[artist]/[album]/@sidebar/default.js` | `/zack/next` | `Promise<{ artist: 'zack', album: 'next' }>` |

> 在 Next.js 16 中，所有并行路由槽位都需要显式的 `default.js` 文件，否则会导致构建失败。

关于并行路由的更多信息，可以参考 [Next.js 官方文档 - Parallel Routes](https://nextjs.org/docs/app/api-reference/file-conventions/parallel-routes)。

### 3.4. loading.js

`loading.js` 文件用于创建有意义的加载 UI，它基于 [React Suspense](https://react.dev/reference/react/Suspense) 实现。通过这个约定，我们可以在路由段内容流式传输时，从服务器显示即时加载状态。一旦内容加载完成，新内容会自动替换加载状态。

![20251209153018](https://raw.githubusercontent.com/xun082/md/main/blogs.images20251209153018.png)

在 `loading.js` 文件中，我们可以添加任何轻量级的加载 UI，如骨架屏、旋转器，或者未来屏幕的一小部分有意义的内容（如封面照片、标题等）。默认情况下，这个文件是一个 [Server Component](https://nextjs.org/docs/app/getting-started/server-and-client-components)，但也可以通过 `"use client"` 指令用作客户端组件。

```tsx
// app/dashboard/loading.tsx
export default function Loading() {
  return <LoadingSkeleton />;
}
```

在同一个文件夹中，`loading.js` 会被嵌套在 `layout.js` 内部。它会自动将 `page.js` 文件及其下方的所有子组件包裹在 `<Suspense>` 边界中。

除了使用 `loading.js`，我们还可以手动为 UI 组件创建 Suspense 边界，实现更细粒度的加载控制。这样可以实现流式服务器渲染（从服务器逐步渲染 HTML 到客户端）和选择性水合（React 根据用户交互优先处理哪些组件）。

```tsx
// app/dashboard/page.tsx
import { Suspense } from "react";
import { PostFeed, Weather } from "./Components";

export default function Posts() {
  return (
    <section>
      <Suspense fallback={<p>Loading feed...</p>}>
        <PostFeed />
      </Suspense>
      <Suspense fallback={<p>Loading weather...</p>}>
        <Weather />
      </Suspense>
    </section>
  );
}
```

> 流式传输不影响 SEO，Next.js 会在流式传输前处理元数据（对于静态爬虫）。某些浏览器会缓冲小于 1024 字节的流式响应。`loading.js` 不支持静态导出。

## 4. 路由段配置（Route Segment Config）

路由段配置是一组特殊的导出配置，可以在 `layout.js`、`page.js` 或 `route.js` 文件中使用，用于控制该路由段的渲染行为、缓存策略和运行时环境等。

### 4.1. 基本用法

路由段配置的使用非常简单，只需在文件中导出对应的配置变量即可：

```javascript
// layout.js | page.js | route.js
export const dynamic = "auto";
export const dynamicParams = true;
export const revalidate = false;
export const fetchCache = "auto";
export const runtime = "nodejs";
export const preferredRegion = "auto";
export const maxDuration = 5;

export default function MyComponent() {}
```

### 4.2. 配置选项总览

以下是 Next.js 16 中所有可用的路由段配置选项：

| `配置选项`                                                                                                         | `类型`                                                                                                                    | `默认值`     | `说明`                           |
| ------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------- | ------------ | -------------------------------- |
| [dynamic](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#dynamic)                 | `'auto' \| 'force-dynamic' \| 'error' \| 'force-static'`                                                                  | `'auto'`     | 控制路由的动态渲染行为           |
| [dynamicParams](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#dynamicparams)     | `boolean`                                                                                                                 | `true`       | 控制动态路由参数的处理方式       |
| [revalidate](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#revalidate)           | `false \| 0 \| number`                                                                                                    | `false`      | 设置路由的重新验证时间（秒）     |
| [fetchCache](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#fetchcache)           | `'auto' \| 'default-cache' \| 'only-cache' \| 'force-cache' \| 'force-no-store' \| 'default-no-store' \| 'only-no-store'` | `'auto'`     | 控制 fetch 请求的缓存行为        |
| [runtime](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#runtime)                 | `'nodejs' \| 'edge'`                                                                                                      | `'nodejs'`   | 设置运行时环境                   |
| [preferredRegion](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#preferredregion) | `'auto' \| 'global' \| 'home' \| string \| string[]`                                                                      | `'auto'`     | 设置 Edge Runtime 的首选执行区域 |
| [maxDuration](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#maxduration)         | `number`                                                                                                                  | 部署平台设置 | 设置函数的最大执行时间（秒）     |

> 配置选项的值必须是静态的，可以在构建时分析。例如 `revalidate = 600` 是有效的，但 `revalidate = 60 * 10` 是无效的。路由段配置会向下继承，子路由会继承父布局的配置，但子路由的配置可以覆盖父布局的配置。

接下来我们详细讲解每个配置选项的作用。

### 4.3. dynamic

`dynamic` 配置用于控制路由的渲染模式。Next.js 支持两种渲染模式：`静态渲染`（构建时渲染，结果缓存到 CDN）和`动态渲染`（请求时渲染，可获取用户信息）。使用 `cookies()`、`headers()`、`searchParams` 等动态函数会自动触发动态渲染。

```javascript
// layout.js | page.js | route.js
export const dynamic = "auto";
// 'auto' | 'force-dynamic' | 'error' | 'force-static'
```

选项说明：

- `'auto'`（默认值）：自动判断，检测到动态函数或不缓存的数据请求时使用动态渲染

- `'force-dynamic'`：强制动态渲染，禁用所有 fetch 缓存

- `'error'`：强制静态渲染，如果使用动态函数会在构建时报错

- `'force-static'`：强制静态渲染，动态函数返回空值

### 4.4. dynamicParams

`dynamicParams` 用于控制当访问未通过 `generateStaticParams` 预生成的动态路由参数时的行为。

```javascript
// layout.jsx | page.jsx
export const dynamicParams = true; // true | false
```

- `true`（默认值）：访问未预生成的动态路由参数时按需生成

- `false`：访问未预生成的动态路由参数时返回 404

> 此配置仅对动态路由段有效，设置 `dynamic = 'error'` 或 `'force-static'` 会将默认值改为 `false`。

### 4.5. revalidate

`revalidate` 用于设置路由的重新验证时间（ISR - Incremental Static Regeneration）。此配置仅支持 Node.js Runtime。

```javascript
// layout.jsx | page.jsx | route.js
export const revalidate = false;
// false | 0 | number（秒）
```

- `false`（默认值）：资源无限期缓存

- `0`：禁用缓存，总是动态渲染

- `number`：设置重新验证的时间间隔（秒），如 `60` 表示最多缓存 60 秒

> 路由会选择最低的 `revalidate` 值作为重新验证频率。

### 4.6. fetchCache

`fetchCache` 用于覆盖路由中所有 `fetch` 请求的默认缓存行为。默认情况下，动态函数之前的请求会被缓存，之后的不会被缓存。

```javascript
// layout.jsx | page.jsx | route.js
export const fetchCache = "auto";
// 'auto' | 'default-cache' | 'only-cache' | 'force-cache'
// 'force-no-store' | 'default-no-store' | 'only-no-store'
```

- `'auto'`（默认值）：根据动态函数位置自动决定

- `'default-cache'`：未设置时默认使用 `force-cache`

- `'only-cache'`：确保所有请求都被缓存，否则报错

- `'force-cache'`：强制所有请求被缓存

- `'default-no-store'`：未设置时默认使用 `no-store`

- `'only-no-store'`：确保所有请求都不被缓存，否则报错

- `'force-no-store'`：强制所有请求不被缓存

> `'force-*'` 选项优先级高于 `'only-*'`，不能同时使用 `'force-cache'` 和 `'force-no-store'`。

### 4.7. runtime

`runtime` 用于设置路由的运行时环境。

```javascript
// layout.jsx | page.jsx | route.js
export const runtime = "nodejs";
// 'nodejs' | 'edge'
```

- `'nodejs'`（默认值）：支持完整的 Node.js API，适合复杂服务器逻辑

- `'edge'`：基于 Web API，启动快、延迟低，但功能受限，不支持 `revalidate`

### 4.8. preferredRegion

`preferredRegion` 用于设置 Edge Runtime 的首选执行区域。需要配合 `runtime = 'edge'` 使用，仅在支持多区域的平台（如 Vercel）上生效。

```javascript
// layout.jsx | page.jsx | route.js
export const preferredRegion = "auto";
// 'auto' | 'global' | 'home' | string | string[]
```

- `'auto'`（默认值）：自动选择最接近请求的区域

- `'global'`：在所有可用区域中执行

- `'home'`：在项目主区域执行

- `string | string[]`：指定区域 ID，如 `'iad1'` 或 `['iad1', 'sfo1']`

### 4.9. maxDuration

`maxDuration` 用于设置 Serverless Function 的最大执行时间（秒）。主要在 Serverless 平台上使用。

```javascript
// layout.jsx | page.jsx | route.js
export const maxDuration = 5; // 秒
```

### 4.10. generateStaticParams

`generateStaticParams` 用于在构建时预生成动态路由的静态页面，与 `dynamicParams` 配合使用。

```javascript
// app/products/[id]/page.js
export async function generateStaticParams() {
  return [{ id: "1" }, { id: "2" }, { id: "3" }];
}
```

### 4.11. 配置继承和最佳实践

子路由会继承父布局的配置，但可以被覆盖。Next.js 会选择最低的 `revalidate` 值作为路由的重新验证频率。

```javascript
// app/layout.js - 根布局
export const revalidate = 3600;
export const runtime = "nodejs";

// app/dashboard/layout.js - 覆盖父布局
export const revalidate = 60;
export const runtime = "edge";
```

最佳实践：

1. 在根布局中使用默认配置（`'auto'`），让 Next.js 自动优化

2. 仅在特定路由中覆盖需要特殊控制的配置

3. 使用 `next build` 查看构建输出，了解路由的渲染模式

## 总结

Next.js 16 的文件路由系统基于文件系统构建，通过特殊文件约定和路由段配置，让我们能够以声明式的方式控制路由的渲染、缓存和运行时行为。理解段（Segment）的概念是掌握整个路由系统的关键，而路由组则帮助我们更好地组织大型应用的文件结构。在实际开发中，建议优先使用默认配置，让 Next.js 自动优化，只在确实需要特殊控制时才进行覆盖。
