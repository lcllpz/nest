前端工程师的职责早已不再局限于浏览器端，尤其在处理性能优化问题时，更需具备全栈视角，全面审视系统细节。

随着 Vue、React 等现代框架的发展，大型项目的开发效率显著提升，代码结构也更加清晰易维护。然而，这些框架在首屏渲染中普遍存在性能瓶颈。

## 页面渲染的发展

在早期没有 Vue、React 等前端框架时，网站开发主要依赖 JSP、PHP 等服务端技术，HTML 页面由后端动态生成并直接返回浏览器渲染。

但随着前端复杂度的提升，这种完全依赖后端渲染的方式在开发效率和交互性能上逐渐暴露出局限，因而被现代前端开发模式所取代。

现代框架以 MVVM 和组件化为核心，取代了传统的 MVC 模式，不仅提高了开发效率，也降低了维护成本。框架通过数据驱动页面更新，让前端工程师专注于业务逻辑的实现。

此外，虚拟 DOM、前端路由、组件化等特性极大提升了开发体验，但也带来了性能隐患：页面渲染需等待框架及业务代码加载并解析完成，导致首屏出现“白屏”的问题。框架越复杂，加载成本越高，用户体验越差。

这种现象类似于高级语言与底层语言的权衡：前者开发友好但运行性能不如后者。前端开发同样需要在性能与效率之间找到平衡。

我们无法回到 JSP/PHP 的开发模式，虽然它们首屏快，但在现代项目中开发效率低、维护成本高。因此，应从框架内部寻找优化首屏渲染的方案，在保障性能的同时保留现代开发优势。

## React 的一些优化方案

React 为了在框架层面做优化，经历了一系列的演进和改进。以下是 React 在框架层面做优化的主要里程碑，按照先后顺序进行讲解。

1.  React 16 引入了 Fiber 架构，它重新实现了 React 的核心算法，使得 React 能够更好地处理大型应用和复杂场景，提高了渲染性能和响应速度。

2.  Fiber 架构同时引入了任务调度器与优先级机制，允许 React 根据任务类型与紧急程度动态安排渲染顺序，从而优化用户交互体验。

3.  React 16.6 引入了懒加载和 Suspense 特性，实现组件级的懒加载，减小初始加载体积，加快首屏速度。

4.  React 提供了服务器端渲染（SSR）的支持，可以在服务器端渲染 React 组件并将渲染结果直接返回给客户端，以提高首屏加载速度、改善 SEO 和用户体验。

5.  React 18 引入了增量渲染技术，可以先渲染出可见区域的内容，然后在后台继续加载和渲染其他内容，提高了页面的响应速度和用户体验。

React 在框架层面做了许多优化，包括引入 Fiber 架构、Hooks、懒加载和 Suspense、服务器端渲染、增量渲染等，这些优化使得 React 具备了更好的性能、开发体验和扩展性，成为了前端开发中最流行的框架之一。

## 客户端渲染（CSR）

客户端渲染（CSR）模式下，初始加载的 HTML 页面几乎不包含实际内容。浏览器需先加载并执行 JavaScript 文件中的 React 代码，才能动态生成页面结构。同时，JavaScript 还负责绑定页面的交互事件，实现完整的界面渲染与用户操作响应。

## 传统 SSR

传统的 SSR 服务端渲染，是指服务端直接吐出具有数据的 HTML 页面，而不是在客户端拼接的 HTML。单纯实现 SSR 是很传统的技术，很简单，不分语言，随便用 php 、jsp、asp、node 等都可以实现。

而现今流行的 SPA 单页应用，则是 CSR 客户端渲染，指的是在客户端通过 Ajax 请求来拼装数据，所有页面都是在客户端拼接好的。

传统 SSR 模式虽然渲染快、对 SEO 友好，但缺乏良好的前端交互能力和开发体验，主要问题包括如下几个方面：

1. 缺乏前端交互：只能刷新整页，无法实现局部更新或无刷新切换

2. 用户体验差：每次跳转都刷新，页面闪烁、加载慢

3. 代码不复用：前后端重复写逻辑，维护成本高

4. 开发效率低：前端样式、逻辑混杂在服务端代码中，耦合严重

5. 前端主导性差：前端无法控制路由、状态管理、视图更新等

如果只实现 SSR 其实没意义，技术上没有任何发展和进步，否则 SPA 技术就不会出现。然而，SPA 也并非万能。其最突出的两个问题是：

1. 首屏渲染慢：所有内容需等待 JS 加载、执行完毕，用户看到内容慢，容易出现“白屏”

2. SEO 表现差：搜索引擎爬虫抓不到动态生成的内容

传统 SSR 只是在服务器拼 HTML，它解决不了复杂交互和现代开发的问题，而现代 SSR（如 Next.js）是为了结合 SSR 和 SPA 各自优点，实现快速首屏 + 良好体验的双赢模式。

## 前端同构应用（SSR + SPA）

最好的方案是：SSR + SPA 的前端同构应用。第一次访问页面是服务端渲染，基于第一次访问，后续的交互就是 SPA 的效果和体验，还不影响 SEO。简单说就是一个前端项目里的组件，部分服务端渲染后输出，部分由客户端异步渲染，既保障网页渲染速度，也有利于搜索引擎 SEO。

同构的意思是：服务端和前端使用同一套渲染层代码，在服务端执行虚拟 DOM(一般用 Node.js)，此时服务端使用和前端相同的虚拟 DOM 的原理来拼接 HTML 模板。

得益于 Node.js 既是后端又是用 JS 来写，使得使用一套代码就能在服务端和客户端都执行渲染，最大限度的重用代码（同构），减少开发维护成本。当前的前端同构应用，需要采用 react 或者 vue 等前端框架，与 node SSR 结合来实现。

### 同构应用需要解决的 3 个问题

所谓“同构”，指的是前后端共用一套代码，既可在服务端渲染页面，也可在客户端执行逻辑。这样可以最大程度地实现代码复用，避免维护两套渲染逻辑，提高开发效率与一致性。

#### 路由同构：双端路由如何维护？

在同构应用中，将路由配置抽离为独立模块（如 `routes-config.js`），是实现服务端与客户端共享路由逻辑的关键。这种做法既避免重复定义，也确保了页面结构的一致性。

在客户端，`BrowserRouter` 配合 `<Routes>` 加载配置中的组件；而在服务端，`StaticRouter` 根据 `req.url` 匹配相应的路由，并结合 React 的 `renderToString` 进行服务端渲染，最终生成 HTML 返回给浏览器。

通过这种结构，路由与组件的绑定逻辑可以完全复用，适配 React Router v6 的声明式 `<Route>` API，同时也支持嵌套路由的渲染，确保前后端页面输出保持一致。

#### 数据预取同构：获取数据的方法和逻辑写在哪里？

数据预取同构的核心在于：无论是在服务端还是客户端，组件都能通过统一的方式获取所需数据，并在渲染前完成注入，确保页面渲染的一致性和完整性。

在实现同构数据预取时，常见的方式包括以下几种（推荐前两种）：

##### 1. 使用高阶组件封装数据获取逻辑

通过高阶组件为页面组件绑定统一的数据获取方法（如 `withSSR(WrappedComponent, getInitialProps)`）。
这样，页面组件可以专注于通过 `props` 获取渲染所需的数据，避免在组件内部自行发起请求，提高可维护性与复用性。

##### 2. 在组件上定义静态数据获取方法

为组件定义静态方法（如 `Component.getInitialProps`），服务端渲染时可直接调用，客户端在必要时也可复用。例如：

```js
Index.getInitialProps = async (ctx) => { ... }
```

这种方式逻辑集中，清晰易维护，是传统 SSR 应用的主流方案。

##### 3. 将数据请求方法与路由绑定

在路由配置中为每条路由关联对应的 `getInitialProps` 方法。当服务端根据 URL 匹配到路由后，可判断该路由是否包含数据预取逻辑并执行。

```js
if (route.getInitialProps) {
  data = await route.getInitialProps(ctx);
}
```

适合在动态路由结构中集中处理数据加载逻辑。

#### 渲染同构：如何复用服务端 HTML？

即便路由匹配和数据预取都已完成，服务端也能成功输出 HTML，但仍可能出现一个常见问题就是当浏览器执行完 JS 并初始化 React 后，页面发生组件重复渲染，导致闪烁现象。

这是由于服务端和客户端的渲染内容不一致导致的水合失败（hydration mismatch）：

- 浏览器首次执行时，如果没有拿到与服务端一致的数据，React 会判断组件状态不同，强制重新渲染；

- 即使 DOM 内容完全相同，只要 `props` 不一致，React 也可能触发组件重新执行 render。

解决方案就是渲染同构，它的核心目标是让客户端在首次执行时拥有与服务端相同的数据状态，从而顺利完成 Hydration 而不触发二次渲染。

##### 实现方式一：服务端注入数据到 HTML 中

在服务端渲染完组件后，将获取到的 `initialProps` 数据序列化为 JSON 并注入到 HTML 模板中，使浏览器端 JS 可以直接读取使用。

```js
// server 模板注入
let html = template.replace(
  "%DATA%",
  `window.__INITIAL_DATA__ = ${JSON.stringify(initialProps)}`
);
```

##### 实现方式二：客户端读取并传入 App 初始化

```js
// client.js
const initialData = window.__INITIAL_DATA__ || null;
hydrateRoot(container, <App initialData={initialData} />);
```

在组件层，初始数据作为 props 一路透传给页面组件，确保首屏渲染与服务端一致：

```js
<Route path={item.path} element={<Component initialData={initialData} />} />
```

##### 实现方式三：高阶组件封装数据逻辑（如 `withSSR`）

使用高阶组件（如 withSSR(WrappedComponent, getInitialProps)）是一种常见且推荐的数据预取方式：

- 服务端：调用组件静态方法 `getInitialProps()` 获取数据

- 客户端：组件挂载后检测是否已有数据，如无则发起请求

- 两端共享同一套数据获取逻辑，实现代码复用

```js
SSRComponent.getInitialProps = getInitialProps;
```

这使得页面组件可以专注于 `props` 展示，完全解耦数据逻辑。

渲染同构的关键，在于服务端将预取数据注入 HTML，客户端在 hydrate 阶段读取并传入组件，确保 props 与服务端完全一致，从而避免闪烁与重复渲染问题。

### 其他问题

解决了上面 3 个问题后，可以说「同构」应用的基本实现原理就清楚了，但还有一些问题需要解决。

#### Node.js 环境限制

在 React/Vue 等前端框架实现服务端渲染（SSR）时，代码会在 Node.js 环境中执行。而 Node.js 是运行在服务器上的，不具备浏览器环境，因此有以下几点需要特别注意：

##### 1. 没有 `window`、`document` 等浏览器对象

在服务端执行时，以下对象均为 `undefined`：

- `window`

- `document`

- `navigator`

- `location`

如下错误示例所示：

```js
const width = window.innerWidth; // 服务端执行会直接抛错
```

我们可以将其编写为如下形式：

```js
if (typeof window !== "undefined") {
  const width = window.innerWidth;
}
```

##### 2. 无法使用 `localStorage` / `sessionStorage`

Web Storage API 属于浏览器环境，服务端同样无法访问。

如下错误示例所示：

```js
const token = localStorage.getItem("token"); // SSR 阶段会崩溃
```

可以通过判断是否处于客户端环境，再安全地访问浏览器特有的 API。对于无法在服务端访问的 `localStorage`，可以使用如下方式进行防护性处理，并在服务端通过 Cookie 或参数传递进行替代：

```js
if (typeof window !== "undefined") {
  const token = localStorage.getItem("token");
}
```

这样可以避免 SSR 阶段访问不存在的对象导致渲染报错，同时确保客户端正常读取本地存储数据。

##### 3. 不要在模块顶层直接执行副作用代码

SSR 中所有代码都是在请求阶段即时加载的，所以避免在模块顶层直接访问浏览器 API 或执行副作用逻辑。

推荐将副作用逻辑放在：

- `componentDidMount` / `useEffect` 中（只在客户端执行）

- 或使用 `typeof window !== 'undefined'` 做环境判断

#### css 过滤

##### 问题一：服务端不支持 CSSOM

由于 Node.js 环境下没有 DOM，自然也无法使用 CSSOM（如 `document.styleSheets`、`getComputedStyle` 等）：

- 无法访问浏览器原生样式对象

- 无法自动加载 `.css` 文件（不像浏览器通过 `<link>` 标签加载）

##### 问题二：服务端渲染可能缺少样式

如果未做样式处理，SSR 仅返回 HTML 结构，样式内容依赖客户端加载，结果可能是：

- 用户首屏看到“无样式页面”

- 样式延迟加载，页面闪烁

##### 常见 CSS 方案（根据项目需求选择）

###### 1. CSS Modules + Webpack 配置支持 SSR

适用于传统 React 项目

```jsx
import styles from "./index.module.css";

<div className={styles.title}>Hello</div>;
```

服务端使用 CSS Modules 时，需要在 Webpack 中启用 `css-loader` 的 `modules: true` 选项，并结合 `isomorphic-style-loader` 或 `style-loader/ssr` 将样式正确注入到服务端生成的 HTML 中，确保页面在首屏就具备完整样式。

###### 2. CSS-in-JS 方案（推荐）

如 styled-components、emotion、linaria 等 CSS-in-JS 方案原生支持 SSR。以 styled-components 为例，服务端通过 `ServerStyleSheet` 收集组件样式并生成 `<style>` 标签插入 HTML，客户端 Hydration 时则复用这些样式，避免重复渲染和样式闪烁。

```jsx
const sheet = new ServerStyleSheet();
const html = renderToString(sheet.collectStyles(<App />));
const styles = sheet.getStyleTags(); // 插入到 HTML 中
```

###### 3. 原子化类名方案（Tailwind CSS）

原子化 CSS 方案（如 Tailwind CSS）在构建时就生成完整样式，运行时无需动态注入，天然适配 SSR。结合 `PostCSS + PurgeCSS` 可裁剪未使用的类名，减小样式体积，最终输出的 HTML 已包含所有必要样式，无需额外处理。

###### 4. 框架内置方案（如 Next.js）

像 Next.js 这类框架开箱即用，内置支持 CSS Modules、SASS 和 styled-jsx，并能自动提取首屏关键样式（critical CSS），完成 SSR 场景下所需的样式收集与注入，无需开发者手动配置。

#### 动态路由 SSR

在 React 的 SPA 模式中，常会使用组件懒加载（即按需加载）进行代码拆分，以减少首屏 JS 体积、提升加载效率。但在 SSR 场景下，这种策略会引发一系列问题。

而 React SSR 默认不支持异步组件。因为服务端渲染要求同步返回完整组件树，而懒加载组件是异步的，服务端无法等待其加载完成，自然无法正确渲染该组件。

具体问题流程入如下所示：

1. 服务端渲染时，遇到异步组件无法正常渲染，只能输出结构占位，例如 `<span>正在加载...</span>`；

2. 客户端加载 JS 后才开始真正加载该组件；

3. 由于客户端拿到的真实组件与服务端输出的 HTML 不一致，导致 React Hydration 报错或触发“闪屏”；

4. 最终在客户端完成组件加载并重新渲染页面，用户才看到正确内容。

其中一个方案是使用 `@loadable/component` 实现 SSR 友好的懒加载

```bash
pnpm add @loadable/component @loadable/server
```

```js
// routes-config.js
import loadable from "@loadable/component";

const Detail = loadable(() => import("./Detail"));
```

它支持在服务端预加载异步组件，避免渲染时 HTML 内容缺失；同时提供 `ChunkExtractor` 工具，将所需的样式和 JS chunk 注入到服务端 HTML 中，全面兼容 React 18 的 SSR 渲染流程。是实现 React 同构应用中 SSR + 动态路由 + 代码拆分的主流方案。

#### SEO 支持（路由页动态生成 TDK）

在服务端渲染（SSR）中，除了要渲染页面的主体内容，还需要为每个页面动态生成 TDK（标题、描述、关键词），以便搜索引擎能正确理解并抓取页面内容，提升 SEO 效果。

什么是 TDK？

- `T`：Title，页面标题

- `D`：Description，页面摘要说明

- `K`：Keywords，页面关键词

TDK 是 SEO 的核心组成部分，通常应与页面内容紧密相关，且每个页面应有独立的 TDK 信息。

再 React+SSR 的场景中，[`react-helmet`](https://github.com/nfl/react-helmet) 是一个专为 React 应用设计的头部信息管理工具，支持：

- 组件内声明式设置 `<title>`、`<meta>` 等标签

- 服务端和客户端共享使用（支持同构）

- SSR 模式下可通过 `Helmet.renderStatic()` 提取并插入 HTML

无论是服务端渲染还是客户端切换，建议在数据预取阶段，将 TDK 信息嵌入接口返回的 `initialData` 中，例如：

```json
{
  "data": { ... },
  "tdk": {
    "title": "首页 - 我的站点",
    "description": "这是首页描述内容",
    "keywords": "首页,SSR,SEO"
  }
}
```

在客户端中我们可以使用 `<Helmet>` 注入头部标签：

```jsx
import { Helmet } from "react-helmet";

function App({ initialData }) {
  const { tdk = {} } = initialData || {};

  return (
    <div>
      <Helmet>
        <title>{tdk.title}</title>
        <meta name="description" content={tdk.description} />
        <meta name="keywords" content={tdk.keywords} />
      </Helmet>
      首页内容
    </div>
  );
}
```

页面组件无需手动操作 DOM，Helmet 会自动处理 `<head>` 标签内容更新。

服务端提取 TDK 信息并注入 HTML 模板：

```js
import { Helmet } from "react-helmet";
import { renderToString } from "react-dom/server";

// 1. 渲染组件
const html = renderToString(<App initialData={data} />);

// 2. 获取 TDK 信息
const helmet = Helmet.renderStatic();

// 3. 拼接到模板中
const fullHtml = `
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  ${helmet.title.toString()}
  ${helmet.meta.toString()}
</head>
<body>
  <div id="root">${html}</div>
</body>
</html>
`;
```

值得注意的是，必须在 `renderToString` 后调用 `Helmet.renderStatic()`，否则无法获取完整 TDK 信息。

## SSR 的缺点

### 1. 首屏快，但 TTI 慢

在 SSR（Server-side Rendering）中，浏览器拿到的第一屏确实是已经“填满内容”的 HTML：

```html
<h1>Welcome, Jane!</h1>
<button>Click me</button>
```

这些 HTML 标签看起来像能用，但它们其实还没有绑定任何事件。

这就形成了一个关键的体验差距，用户视觉感知页面已经出现了，看起来可以操作，但是实际组件还没有激活，组件未绑定，这就导致用户点击无反应、延迟、卡顿或者错误提示。

举个例子，你进入了一个新闻页面，文章文字已经出来了，但点评论、菜单、分享按钮都没有反应，甚至页面跳转按钮也点不了。

这时用户会感觉`卡`、`假`、`不顺畅`，而这正是 Hydration 的空档期（gap） 所造成的。

这段 `交互空白期` 长短取决于多个因素：如果 JS bundle 较大、网络较慢（如 3G）、页面结构复杂或用户设备性能较低，都会导致 Hydration 延迟，页面虽然快速可见，但交互响应却明显滞后，影响真实的可用体验。

### 2. 服务器压力大

所谓服务器压力大，指的是：

- CPU 占用高（需要运行大量计算）

- 内存使用大（保存上下文状态、数据、渲染缓存）

- 请求响应慢（并发请求堆积，处理变慢）

- 扩容代价高（服务节点要加机器，不是加 CDN）

这些都会导致服务稳定性下降、成本上升，甚至用户访问超时或 5xx 错误。

在 SSR 中，每个用户请求都会触发一次完整的渲染任务，包括运行 JavaScript、构建组件树、发起数据请求、拼接 HTML 并返回。不同于客户端渲染只需初始化一次，SSR 每次访问都重新执行一遍渲染流程，计算无法复用，服务器压力显著更大。

假设我们的网站有以下负载：

- 每秒 500 个用户请求页面

- 每个页面 SSR 渲染耗时 80ms

那么：

```
500 * 80ms = 每秒消耗 40,000ms = 40 CPU 核/秒
```

当大量请求同时涌入时，SSR 容易导致服务器 CPU 满载，TTFB 上升，Node.js 的事件循环被阻塞，响应延迟显著增加。若渲染中还涉及数据库或远程调用，压力会进一步放大，最终可能导致请求超时、用户体验变差，甚至服务崩溃。

### 3. 延迟受地理位置影响

在 SSR 模式下，页面必须由服务器实时渲染，因此所有请求都必须回到服务端执行逻辑、拼接 HTML 并返回。这意味着：

1. 如果用户离服务器较近（如同城或同区），响应较快；

2. 如果用户跨洲访问（如亚洲用户访问部署在欧美的服务器），则需要更长的网络往返时间（RTT）。

即使服务端性能再强，也无法绕过物理距离带来的延迟瓶颈，导致同一个页面在国内加载很快、海外却变慢，最终使 TTFB 不稳定，影响 SEO 和用户体验。

## 同构应用的额外缺点（SSR 的升级难题）

### 1. Hydration 不稳定，容易错乱

在 SSR 模式下，React 首先在服务器端渲染出一份 HTML 页面，然后客户端再通过 JavaScript 进行 Hydration（激活绑定事件）。

这一步要求客户端 React 重新渲染一次，并且结果必须和服务端返回的 HTML 一字不差，否则会产生“内容不匹配”的警告或错误。

```bash
Warning: Text content did not match.
Server: "Hello"
Client: "Hello 👋"
```

这代表服务器返回的 HTML 是 `Hello`，但客户端在 Hydration 阶段渲染出了 `Hello 👋`，两者不一致。

造成 mismatch 的常见原因：

- 使用了 `Date.now()`、`Math.random()`、`new Date()` 等非确定性值

- 初始数据未在服务端对齐（如请求未同步或异步顺序不同）

- 客户端和服务端渲染环境差异（如语言、时区等）

- 条件渲染逻辑依赖于浏览器特性（如 `window.innerWidth`）

最终结果可能是：

- 页面加载时闪一下（重新渲染）
- UI 部分失效，交互不能用
- 在严格模式或 React 18 下直接报错并阻止渲染

举个例子：

```jsx
function Greeting() {
  const now = new Date().toLocaleTimeString();
  return <p>现在时间是：{now}</p>;
}
```

在 SSR 时 now = 12:01:01，但客户端一加载是 12:01:04，就会 mismatch：

```bash
Warning: Text content did not match.
Server: "现在时间是：12:01:01"
Client: "现在时间是：12:01:04"
```

React 会先显示服务器渲染的内容，然后替换成客户端的版本 → 用户看到“闪一下”

### 2. Hydration 成本高

Hydration 本质上是 React 在客户端重新遍历已有的 DOM，完成事件绑定、状态恢复和副作用执行。页面结构越复杂，Hydration 过程就越耗时，导致 TTI（可交互时间）延迟，出现“页面看得见但点不了”的尴尬体验。

## 总结

SSR（Server-Side Rendering） 是指页面在服务端将组件渲染成 HTML，再返回给浏览器，能大幅提升首屏速度和 SEO 效果。它解决了 CSR 首屏“白屏”的问题，但也带来了 Hydration 空档期、服务器性能瓶颈、以及复杂度增加等挑战。

每个请求都需要服务端执行一次完整的渲染流程，意味着高并发场景下容易造成 CPU 压力、响应变慢甚至崩溃。同时，由于 HTML 是动态生成的，难以跨节点缓存，扩展性也不如 SSG 或纯 CSR。

在同构（Isomorphic）应用中，前后端共用一套代码，提升了复用性与开发效率，但也要求开发者严格区分运行环境，防止浏览器专属 API 在服务端报错。最终，SSR 是性能与复杂度之间的一场权衡，适合需要快速首屏和良好 SEO 的内容型网站或初始加载体验要求高的应用场景。

从性能指标上看，SSR 显著优化了 FCP（First Contentful Paint），因为服务端已经生成完整 HTML，浏览器拿到后可以立即开始渲染内容，无需等待 JS 加载与执行。与此同时，LCP（Largest Contentful Paint） 也因首屏主要内容（如标题、首图、Banner）预渲染完成而显著加速，使页面在用户眼中的“变快”更加明显，有助于整体用户体验与 SEO 排名。
