# 12. AOP 和横切关注点

> 所属部分为第三部分"设计思想与底层基础"

## OOP 的局限性

前面几篇我们已经把 `NestJS` 的很多基础骨架串起来了，包括模块、依赖注入、装饰器、执行上下文。接下来要看一个更偏设计思想的问题：

为什么很多公共逻辑明明不是核心业务，却总会反复出现在各个接口里。

很多人学过面向对象之后，会觉得只要把类拆好、职责分清楚，系统就应该足够清晰。但项目一做大就会发现，仅靠 `OOP` 还不够。

原因不在于 `OOP` 不好，而在于它更擅长解决"纵向职责划分"问题。比如：

- 控制器负责接请求
- 服务层负责业务处理
- 仓储层负责数据访问

这些都是沿着业务主线往下拆的，非常适合用对象和层次来组织。

可真实项目里还有另一类逻辑，它们不属于某一个单独业务模块，却会反复出现在很多地方。比如：

- 记录日志
- 做权限校验
- 统计耗时
- 包装统一响应
- 处理缓存
- 捕获异常

这些逻辑的问题在于，它们不是只属于某个控制器，也不是只属于某个服务。它们会横着穿过很多业务点。

这就是 `OOP` 的一个天然局限。它很擅长组织业务对象，却不天然擅长处理那些跨很多对象、跨很多模块同时存在的公共逻辑。

所以理解 `AOP` 的第一步，不是先记概念，而是先意识到：

不是所有问题都能靠继续拆类来解决。

## 什么是横切关注点

有了上面的铺垫，再看"横切关注点"这个词就不难了。

所谓横切关注点，指的是那些会同时影响很多业务流程、很多模块、很多接口，但又不属于某个单独业务核心的公共问题。

你可以把业务主线想象成一条条竖着的流程：

- 用户注册
- 用户登录
- 创建订单
- 查询订单
- 支付回调

而日志、鉴权、限流、缓存、监控这些能力，则像是一条条横着切过去的线。它们会在很多流程上都出现。

所以它们被叫做"横切关注点"，本质上是因为：

它们横向穿过了多条业务链路。

举个更贴近代码的例子。

假设你有很多接口都要做这几件事：

- 进入方法前打印日志
- 判断用户是否已登录
- 统计执行耗时
- 出错时记录异常信息

这些逻辑并不是"创建用户业务"本身，也不是"查询订单业务"本身。但它们会一遍又一遍附着在这些业务外面。

这类问题如果不单独处理，最终就会变成：

每个业务方法都掺着一层公共逻辑，业务本体反而越来越看不清。

所以横切关注点这个概念，真正想提醒你的不是术语，而是一个设计事实，有些逻辑天生就不适合散落在业务代码里。

## 为什么日志 鉴权 监控 缓存不该散落在业务代码里

这一节是 `AOP` 真正的工程动机。

很多团队最开始没有意识到横切问题时，代码会慢慢长成这样：

```typescript
import { Controller, Get, Req, UnauthorizedException } from '@nestjs/common';
import type { Request } from 'express';

@Controller('users')
export class UsersController {
  @Get('profile')
  async profile(@Req() req: Request): Promise<{ ok: boolean }> {
    console.log('start /users/profile');

    if (!req.headers.authorization) {
      console.error('unauthorized');
      throw new UnauthorizedException('please login first');
    }

    const start = Date.now();

    const result = { ok: true };

    console.log(`time cost: ${Date.now() - start}ms`);
    return result;
  }
}
```

这段代码的问题不是不能运行，而是混杂了太多不同层次的事情：

- 业务入口
- 日志记录
- 鉴权判断
- 耗时统计
- 异常输出

只要这种写法多复制几次，系统很快就会出现几个典型问题：

- 同一类逻辑到处重复，改规则时要改很多地方
- 业务代码被公共逻辑淹没，可读性越来越差
- 不同人写法不一致，系统行为慢慢失去统一性
- 日志、鉴权、缓存等能力很难全局升级和收口

比如今天你想把日志格式统一一下，明天想给所有接口补请求耗时统计，后天想给需要登录的接口统一加鉴权。如果这些逻辑散在每个方法里，维护成本会越来越高。

所以从工程角度看，日志、鉴权、监控、缓存这些东西不该散落在业务代码里，原因不是"代码不好看"，而是：

它们根本不是某个业务方法私有的事情。

它们应该被抽到更统一的位置去处理，这样系统才能保持一致性和可维护性。

## AOP 的核心思想

`AOP` 的全称是 `Aspect-Oriented Programming`，通常翻译成"面向切面编程"。

第一次看这个名字时，很容易觉得它特别抽象。其实如果把前面几节理解了，`AOP` 的核心思想可以压缩成一句非常直白的话：

把那些横向重复出现的公共逻辑，从业务主线里抽出来，在统一切入点上处理。

这里最关键的不是"切面"这个词，而是"统一切入"。

也就是说，业务代码还是只关心业务本身，而日志、鉴权、缓存、异常、响应包装这些公共逻辑，不再由每个业务方法自己手写，而是通过框架机制在合适时机插进去。

这张图表达的就是 `AOP` 的核心感觉：

业务逻辑仍然是中间那条主线。
公共能力在前后或周围切入。
最终让两类关注点彼此分离。

所以 `AOP` 并不是为了炫技，也不是为了让代码显得抽象，而是为了回答一个很现实的问题：

重复出现的公共逻辑，到底应该放在哪里，才能既统一又不污染业务代码。

从这个角度看，`AOP` 真正解决的是关注点分离问题。

## NestJS 为什么天然适合做 AOP

如果把前面十几篇内容一起看，你会发现 `NestJS` 天然很适合做 `AOP`，并不是偶然。

因为它已经具备了做横切抽象最关键的几个基础条件：

- 有装饰器和元数据，能描述结构和意图
- 有依赖注入容器，能统一组织横切能力对象
- 有 `ExecutionContext` 和 `ArgumentsHost`，能在运行时获取执行现场
- 有完整请求生命周期，能在不同阶段插入公共逻辑

也正因为这些基础都已经存在，所以 `NestJS` 不需要你自己从零搭一套切面机制。它直接把很多常见横切点做成了框架级能力：

- `Guard` 负责做放行前判断，适合登录校验、权限控制
- `Interceptor` 负责包裹执行前后，适合日志、耗时统计、缓存、统一响应
- `Pipe` 负责参数进入业务前的校验和转换
- 异常过滤器负责统一错误收口

你会发现，这些机制其实都不是零散堆出来的功能，而是 `AOP` 思想在 `NestJS` 里的具体落地方式。

比如：

- 想在业务执行前做权限校验，可以用 `Guard`
- 想在业务前后记录日志和耗时，可以用 `Interceptor`
- 想在参数进入方法前统一校验，可以用 `Pipe`
- 想在异常抛出后统一整理错误输出，可以用过滤器

如果用一个非常简化的视角看，`NestJS` 的请求链路本身就已经很像一条可插拔的切面链路。如下图所示。

这张图展示的是，`NestJS` 在请求进入、业务执行、响应返回的不同阶段，分别提供了哪些横切切入点。

框架已经天然提供了很多可切入的位置。你要做的，不是把公共逻辑硬塞进业务代码，而是判断它应该落在哪个横切机制里。

如果从职责上再压缩一下，可以先这样记：

- 需要在执行前决定"让不让过"，优先想到 `Guard`
- 需要在执行前后包裹流程，优先想到 `Interceptor`
- 需要在参数进入业务前处理数据，优先想到 `Pipe`
- 需要在失败后统一收口，优先想到异常过滤器

所以可以把这一章最后落成一句很关键的话：

`NestJS` 并不是额外"支持一下 AOP"，而是它的请求执行模型本身就天然适合承载 `AOP`。

## 小结

这一篇最重要的，不是把 `AOP` 当成一个单独术语记住，而是把它和你每天写的接口代码联系起来。

可以先把核心结论记成这样：

- `OOP` 擅长组织业务对象，但不天然擅长处理横向公共逻辑
- 横切关注点指的是会穿过很多业务流程的公共能力
- 日志、鉴权、监控、缓存、异常这类逻辑不该散落在业务方法里
- `AOP` 的核心是把这些公共逻辑抽出来，在统一切入点处理
- `NestJS` 的 `Guard`、`Pipe`、`Interceptor`、过滤器，本质上就是 `AOP` 思想的框架化落地

你也可以把这一章总结成一句更直接的话：

业务代码应该只回答"我要做什么"，横切机制负责统一处理"所有人都要顺手做的那些事"。

下一节，我们会继续往前看设计模式。到那里你会更清楚，除了 `AOP` 这种横切思想之外，`NestJS` 还借用了哪些经典软件设计模式来组织框架和业务代码。
