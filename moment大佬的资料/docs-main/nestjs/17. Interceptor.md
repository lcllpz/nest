`Interceptor` 实现 `NestInterceptor` 接口，核心方法是 `intercept(context, next)`。`context` 仍是 `ExecutionContext`，与守卫、过滤器共享同一套现场抽象。`next` 是 `CallHandler`，`next.handle()` 返回一个 `Observable`，会在后续管道与控制器方法执行完毕时发出业务返回值。

拦截器的特殊之处在于它同时罩住“往下游推进”和“从下游回来”两段。`intercept` 里在调用 `next.handle()` 之前写的逻辑，会发生在管道与控制器之前；对 `next.handle()` 返回的 `Observable` 做 `pipe` 时，算子既能观察控制器吐出的数据，也能在出错、完成时介入。

如果某些场景不想继续执行下游，你也可以不调用 `next.handle()`，直接返回一个新的 `Observable`，实现提前返回或兜底逻辑。拦截器因此非常适合做与返回值形状相关的横切能力，例如统一包装、脱敏、统计耗时、在响应路径上打日志。

拦截器可以注册在全局、控制器或路由上。出站一侧遵循官方文档里"后进先出"的解析习惯，与入站顺序对称，排查响应阶段问题时要把全局、控制器、路由三层叠加顺序一起算进去。

从链路位置看，中间件与守卫在拦截器之前执行。拦截器在管道之前进入，但业务上常把"真正改参数"的事交给管道，把"拿到最终返回值再加工"的事交给拦截器里的 `Observable` 管线。下面用一张简图表示拦截器与管道、控制器之间的包裹关系，箭头表示控制流与数据流的大致方向。

## 可运行 demo：统一返回外壳与耗时 header

为了让拦截器的入站与出站效果更直观，这里用全局拦截器做两个事情：把 `GET /interceptor/*` 的返回值包成统一结构，并在响应头加 `x-latency-ms`。

- `GET /hello` 会因为在方法上标记了 `@SkipEnvelope()` 而跳过外壳包装
- `GET /interceptor/demo` 返回普通对象，外壳会包进 `code/message/data`
- `GET /interceptor/delay` 返回 `Promise`，用来感受耗时统计覆盖 Promise 场景

main.ts（注册全局拦截器）

```typescript
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";
import { ApiEnvelopeInterceptor } from "./api-envelope.interceptor";
import { LatencyInterceptor } from "./latency.interceptor";

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  const apiEnvelope = app.get(ApiEnvelopeInterceptor);
  const latency = app.get(LatencyInterceptor);
  app.useGlobalInterceptors(apiEnvelope, latency);

  await app.listen(process.env.PORT ?? 3000);
}

bootstrap().catch((err) => {
  console.error(err);
  process.exitCode = 1;
});
```

```typescript
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  NestInterceptor,
} from "@nestjs/common";
import { Reflector } from "@nestjs/core";
import type { Observable } from "rxjs";
import { map } from "rxjs/operators";
import { SKIP_ENVELOPE_KEY } from "./skip-envelope.decorator";

interface ApiEnvelope<T> {
  code: number;
  message: string;
  data: T;
}

@Injectable()
export class ApiEnvelopeInterceptor implements NestInterceptor {
  constructor(private readonly reflector: Reflector) {}

  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const shouldSkip = this.reflector.getAllAndOverride<boolean>(
      SKIP_ENVELOPE_KEY,
      [context.getHandler(), context.getClass()],
    );

    if (shouldSkip) return next.handle();

    return next.handle().pipe(
      map((data: unknown) => {
        const body: ApiEnvelope<unknown> = {
          code: 0,
          message: "ok",
          data,
        };
        return body;
      }),
    );
  }
}
```

```typescript
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  NestInterceptor,
} from "@nestjs/common";
import type { Response } from "express";
import type { Observable } from "rxjs";
import { tap } from "rxjs/operators";

@Injectable()
export class LatencyInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const started = process.hrtime.bigint();
    const response = context.switchToHttp().getResponse<Response>();

    return next.handle().pipe(
      tap({
        finalize: () => {
          const ms = Number(process.hrtime.bigint() - started) / 1e6;
          response.setHeader("x-latency-ms", ms.toFixed(2));
          console.log(`[interceptor] latency ${ms.toFixed(2)}ms`);
        },
      }),
    );
  }
}
```

## 请求前后拦截

`intercept` 方法里，所谓"请求侧"一般指在 `return next.handle()` 之前同步执行的代码。这里可以读 `context.switchToHttp().getRequest()`，打请求日志、写入追踪 `id`、做轻量计时起点，但不要在这里假设控制器已经跑完。

所谓"响应侧"一般指对 `next.handle()` 返回的 `Observable` 使用 `rxjs` 的 `tap`、`map`、`catchError`、`finalize` 等算子。`tap` 适合副作用，例如记一条"业务方法已返回"的调试信息。`map` 适合改写向外抛出的数据。`catchError` 可以把下游异常映射成兜底值或重新抛出，用于降级或统一错误体。

下面是一个只演示结构的骨架，省略具体日志实现。入站与出站都通过 `tap` 打点，真实项目里可换成结构化日志与指标上报。

```typescript
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  NestInterceptor,
} from "@nestjs/common";
import type { Observable } from "rxjs";
import { tap } from "rxjs/operators";

@Injectable()
export class LifecycleInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const request = context.switchToHttp().getRequest<{ url?: string }>();
    const started = Date.now();
    console.log(`before handler ${request.url ?? ""}`);

    return next.handle().pipe(
      tap({
        next: () => {
          console.log(`after handler ${Date.now() - started}ms`);
        },
        error: (err: unknown) => {
          console.error("handler error", err);
        },
        finalize: () => {
          console.log("stream finalized");
        },
      }),
    );
  }
}
```

`return` 之前的代码在 `next.handle()` 调用前执行，对应管道与控制器尚未运行。`tap` 里 `next` 在下游发出正常值时触发，`finalize` 在流结束或取消时触发。若控制器返回的是 `Promise`，框架会转成 `Observable`，仍走同一条流。若需要取消或超时，可在管道上再接 `timeout` 等算子，注意与业务长任务的超时策略对齐。

## 响应统一包装

很多团队希望 `HTTP` 体始终是同一外壳，例如 `code`、`message`、`data`，而控制器只返回领域对象。这类需求放在拦截器的 `map` 里最直观，因为此时管道已经跑完，拿到的是控制器方法的返回值。

下面示例展示“用 `map` 把返回值包进固定结构”的写法。配合 `Reflector` 读路由元数据，还可以像 demo 那样用 `@SkipEnvelope()` 选择性跳过外壳。

```typescript
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  NestInterceptor,
} from "@nestjs/common";
import type { Observable } from "rxjs";
import { map } from "rxjs/operators";

interface ApiEnvelope<T> {
  code: number;
  message: string;
  data: T;
}

@Injectable()
export class TransformInterceptor implements NestInterceptor {
  intercept(
    _context: ExecutionContext,
    next: CallHandler,
  ): Observable<unknown> {
    return next.handle().pipe(
      map((data: unknown) => {
        const body: ApiEnvelope<unknown> = {
          code: 0,
          message: "ok",
          data,
        };
        return body;
      }),
    );
  }
}
```

若部分路由需要返回原始流，例如文件下载、服务端推送，应为这些路由跳过包装拦截器，或用元数据标记"裸响应"，在拦截器里用 `Reflector` 读元数据后原样透传。否则统一外壳会破坏二进制或流式响应。

分页、列表类接口有时已经在服务层拼好了 `{ list, total }`，包装层只加外层字段即可，避免重复嵌套 `data.data`。

## 小结

`Interceptor` 用 `intercept` 包裹下游管道与控制器：入站逻辑在 `next.handle()` 之前执行，出站与错误处理写在返回的 `Observable` 管线上。它适合做统一响应形状、耗时统计、日志与脱敏等横切能力，配合 `map`、`tap`、`finalize`、`catchError` 实现即可。

同时把缓存、降级、重试等能力按读写语义、幂等和错误类型区分清楚，别让拦截器承担本该由网关或消息队列完成的事情。
