路由、参数、`DTO`、配置，多半管的是请求怎么进门。这一篇换到出口：失败时怎样返回错误，成功时怎样让结构稳定、可预期。

新手常见的两种凑合写法：

- 到处 `throw new Error("xxx")`
- 在控制器里 `return { success: false, message: "xxx" }`

能跑，但很难形成团队级契约。成熟一点至少要问清楚：状态码怎么选、错误体长什么样、业务失败和机器故障要不要分家、日志和监控从哪接、前端能不能按同一种结构解析。

`NestJS` 里抛出的异常一般不会直接把进程打崩，而是走异常管道，最后变成 HTTP 错误响应。`HttpException` 及其子类还会带上约定好的状态码和响应体。

```typescript
import { Controller, Get, NotFoundException, Param } from "@nestjs/common";

@Controller("users")
export class UsersController {
  @Get(":id")
  findOne(@Param("id") id: string): { id: string } {
    if (id !== "1") {
      throw new NotFoundException("用户不存在");
    }

    return { id };
  }
}
```

访问不存在的用户时，得到的是框架处理过的 `404`，而不是控制器里随手拼的杂凑对象。关键认知是：异常在这里是走统一通道的，不是抛完就没人管。

## `HttpException` 与内置子类

`HttpException` 同时表达两件事：这是不是 HTTP 语义上的失败，以及用哪个状态码。

最基础的用法：

```typescript
import { Controller, Get, HttpException, HttpStatus } from "@nestjs/common";

@Controller("demo")
export class DemoController {
  @Get()
  test(): never {
    throw new HttpException("禁止访问", HttpStatus.FORBIDDEN);
  }
}
```

第二个参数还可以传对象，用来定制响应体字段，便于和前端约定 `code`、`errors` 等：

```typescript
throw new HttpException(
  {
    message: "禁止访问",
    bizCode: 1003,
  },
  HttpStatus.FORBIDDEN,
);
```

日常更推荐直接用语义化子类，读代码的人不用背数字：

- `BadRequestException`
- `UnauthorizedException`
- `ForbiddenException`
- `NotFoundException`
- `ConflictException`
- `InternalServerErrorException`

```typescript
import {
  BadRequestException,
  ConflictException,
  NotFoundException,
} from "@nestjs/common";

function validateUserInput(email: string, exists: boolean): void {
  if (!email) {
    throw new BadRequestException("邮箱必填");
  }

  if (exists) {
    throw new ConflictException("邮箱已被占用");
  }

  throw new NotFoundException("用户不存在");
}
```

习惯可以记成：能用语义类就别只丢裸的 `Error`，异常类型本身就是契约的一部分。

## 业务失败和系统故障

至少要在大脑里分两类：

- 业务失败：服务整体健康，只是这次操作不合规则，例如用户不存在、密码错、库存不足、订单状态不允许继续
- 系统故障：依赖连不上、超时、磁盘或未知缺陷，需要日志、告警、排查

前者适合用明确的 `4xx` 表达，让调用方改请求或提示用户。后者更适合落 `5xx` 或经过滤器统一打成 `500`，详细信息进日志，对外只给保守文案，避免把堆栈和内部路径暴露给公网。

```typescript
import { ConflictException, Injectable } from "@nestjs/common";

interface CreateUserDto {
  email: string;
  name: string;
}

@Injectable()
export class UsersService {
  create(body: CreateUserDto): { id: string; email: string; name: string } {
    const exists = body.email === "tom@example.com";

    if (exists) {
      throw new ConflictException("邮箱已被占用");
    }

    return {
      id: "1",
      email: body.email,
      name: body.name,
    };
  }
}
```

数据库宕机时不要伪装成"邮箱冲突"那类业务提示，否则运维和前端都会被误导。可预期的业务失败用具体异常，未捕获的意外交给过滤器兜底。

若团队还需要与 HTTP 状态正交的"业务错误码"，可以自定义异常类，在 `getResponse()` 里带上 `bizCode`，过滤器里原样透传或映射进统一外壳。

## 全局异常过滤器

默认处理器能工作，多人协作时往往会再收一层：固定错误 JSON 形状、补 `requestId`、打日志、对接监控。

过滤器是最后一道收口，控制器、服务、守卫、管道里抛出的异常最终都能汇聚到这里（未被更窄的 `@Catch()` 抢先吃掉的前提下）。

下面示例做了几件事：正确解析 `HttpException.getResponse()`（字符串、`message` 数组、对象都要考虑到，否则 `ValidationPipe` 报错时你会拿到奇怪结构）、未知异常记日志并返回保守文案、可选地把路径和时间戳写进响应。

```typescript
import {
  ArgumentsHost,
  Catch,
  ExceptionFilter,
  HttpException,
  HttpStatus,
  Logger,
} from "@nestjs/common";
import type { Request, Response } from "express";

@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly logger = new Logger(AllExceptionsFilter.name);

  catch(exception: unknown, host: ArgumentsHost): void {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();

    const status =
      exception instanceof HttpException
        ? exception.getStatus()
        : HttpStatus.INTERNAL_SERVER_ERROR;

    const message = this.resolveMessage(exception);

    if (!(exception instanceof HttpException)) {
      this.logger.error(
        exception instanceof Error ? exception.stack : String(exception),
      );
    }

    response.status(status).json({
      code: status,
      message,
      path: request.url,
      timestamp: new Date().toISOString(),
    });
  }

  private resolveMessage(exception: unknown): string | string[] {
    if (!(exception instanceof HttpException)) {
      return "服务暂时不可用";
    }

    const res = exception.getResponse();
    if (typeof res === "string") {
      return res;
    }
    if (typeof res === "object" && res !== null && "message" in res) {
      const msg = (res as { message: unknown }).message;
      if (typeof msg === "string" || Array.isArray(msg)) {
        return msg;
      }
    }
    return exception.message;
  }
}
```

在 `main.ts` 里挂全局：

```typescript
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";
import { AllExceptionsFilter } from "./common/filters/all-exceptions.filter";

async function bootstrap(): Promise<void> {
  const app = await NestFactory.create(AppModule);

  app.useGlobalFilters(new AllExceptionsFilter());

  await app.listen(3000);
}

void bootstrap();
```

也可以改成在 `AppModule` 里用 `APP_FILTER` 注册，便于和别的全局 `Provider` 一起管理依赖：

```typescript
import { Module } from "@nestjs/common";
import { APP_FILTER } from "@nestjs/core";
import { AllExceptionsFilter } from "./common/filters/all-exceptions.filter";

@Module({
  providers: [
    {
      provide: APP_FILTER,
      useClass: AllExceptionsFilter,
    },
  ],
})
export class AppModule {}
```

`@Catch()` 不带参数表示捕获所有异常；若只想改写 `HttpException` 的外观，可以写成 `@Catch(HttpException)`，其余异常仍交给默认层。

异常不要在每个接口里各自格式化，应在统一出口收敛，过滤器里顺便接日志和监控。

## 统一成功体与拦截器

只收错误不够，成功体往往也要约定：`data` 放载荷、`code` 表示业务成功、分页结构是否共用同一套壳子。各写各的很快会把前端拖进 if-else 地狱。

一种常见约定：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "id": "1",
    "name": "汤姆"
  }
}
```

失败时仍可保持同一外壳，例如：

```json
{
  "code": 409,
  "message": "邮箱已被占用",
  "data": null
}
```

原则只有两条：团队内部风格一致；业务字段 `code` 不能替代 HTTP 状态码的职责。该 `401` 还是 `401`，该 `409` 还是 `409`，`code: 0` 只表示"在 HTTP 成功前提下业务也成功"这一类语义，不要混成"全程 200、全靠 body 区分"，除非团队明文规定且网关、监控都认。

成功包装适合用拦截器，在控制器返回之后包一层：

```typescript
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  NestInterceptor,
} from "@nestjs/common";
import { Observable, map } from "rxjs";

interface ApiResponse<T> {
  code: number;
  message: string;
  data: T;
}

@Injectable()
export class TransformInterceptor<T> implements NestInterceptor<
  T,
  ApiResponse<T>
> {
  intercept(
    _context: ExecutionContext,
    next: CallHandler<T>,
  ): Observable<ApiResponse<T>> {
    return next.handle().pipe(
      map((data) => ({
        code: 0,
        message: "ok",
        data,
      })),
    );
  }
}
```

```typescript
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";
import { TransformInterceptor } from "./common/interceptors/transform.interceptor";

async function bootstrap(): Promise<void> {
  const app = await NestFactory.create(AppModule);

  app.useGlobalInterceptors(new TransformInterceptor());

  await app.listen(3000);
}

void bootstrap();
```

实际工程里还要注意几件事：

- 若个别接口返回文件流、`StreamableFile` 或 `SSE`，盲目 `map` 会破坏流式响应，要在拦截器里判断类型后原样放行
- 健康检查、指标接口有时希望保持裸 JSON，可通过路径前缀判断后跳过包装
- 控制器若已经手动包了 `{ code, data }`，要么约定禁止手写，要么用标记位（例如自定义装饰器或 `Symbol`）让拦截器识别后不再包第二层

执行顺序上，全局拦截器与过滤器由框架按文档约定调用，记住一句就够：正常链路走拦截器里的 `map`，抛错走过滤器，两条线不要互相抢活。

## 小结

接口输出是一套设计，不是几个零散技巧。`HttpException` 家族把 HTTP 语义说清楚，业务失败和机房故障分开对待，全局过滤器解析 `getResponse`、兜住未知异常并打日志，成功体用拦截器包成团队约定形状，同时保留真实状态码。流式响应和特殊路由要在拦截器里留白。做到这些，前后端和运维才能在同一套秩序下对话。
