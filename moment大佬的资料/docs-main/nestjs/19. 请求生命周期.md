
一次 `HTTP` 请求进入 `Nest` 后，框架会根据已注册的路由与模块把请求送到对应的 `handler`。把它理解成从外到内的通道，直觉会更稳。

路径匹配容易和直觉不一致。`global prefix`、版本前缀、反向代理剥掉的前缀都会影响中间件里看到的 `path`。排查时先对照中间件打印到的路径与控制器声明的路径是否在同一坐标系。

中间件是最早介入的语义层，主要做报文级处理。全局中间件先于模块中间件执行。因为这阶段还没确定命中的 `handler`，通常不做依赖路由元数据的判定。

中间件常做三件事：记录与追踪，补齐跨域与安全头等传输信息，快速拦截明显非法请求。

中间件是推进式。调用 `next()` 才能继续进入后续层。既不 `next` 也不提前返回，就可能让请求卡住。

若链路中出现未捕获异常，请求会跳过后续步骤，进入异常过滤器链，从更靠近路由的一侧向外扩展到控制器级与全局。

## 主链路上的组件与顺序

上面这一段顺序可以收成下面这张总表，便于和中间件、守卫、管道、拦截器、过滤器各篇对照。`Service` 不是生命周期增强器，单独列出这一步是为了说明控制器方法内部还会继续调用提供者完成业务。

1. 入站请求进入适配器与路由层。
2. `Middleware`：先全局绑定的中间件，再模块级中间件。
3. `Guards`：全局守卫、控制器级守卫、路由级守卫。
4. `Interceptors`（进入控制器方法之前）：全局、控制器级、路由级拦截器。
5. `Pipes`：全局管道、控制器级管道、路由级管道、路由参数级管道。
6. `Controller` 路由处理方法。
7. `Service` 或其它由控制器注入的提供者（若业务在此层继续调用）。
8. `Interceptors`（请求返回阶段）：路由级、控制器级、全局拦截器，顺序与入站相反。
9. `Exception filters`：仅在出现未捕获异常时介入，路由级、控制器级、全局过滤器。
10. 服务器向客户端发出响应。

入站阶段拦截器仍是全局先于控制器先于路由，出站阶段则是路由先于控制器先于全局，形成先进后出。过滤器与其它组件不同，不是全局优先，而是从最贴近当前路由的一层开始，再向外扩展到控制器级与全局。

这张图把成功路径的六站流程与未捕获异常时的过滤器链分支同时画清楚，并标注拦截器入站与出站的方向差异。

下面按各组件在框架中的职责，把常见需求与推荐落点收成对照，便于选型时一眼定位。

| 需求类型 | 优先落点 | 简要依据 |
| --- | --- | --- |
| 访问日志、`trace id`、与具体 `handler` 无关的入口处理 | 中间件 | 贴近 `req` 或 `res`，先于路由语义 |
| 跨域、安全头、请求头补齐等传输层准备 | 中间件 | 只依赖原始报文与 `next` |
| 认证、授权、是否允许进入某个 `handler` | 守卫 | 有 `ExecutionContext`，可读路由与元数据 |
| 路由级访问策略，例如 `@Roles()`、开关策略 | 守卫 | 守卫能把判定绑定到 `handler` |
| `DTO` 校验、标量解析、入参清洗与默认值 | 管道 | 在方法执行前处理参数绑定结果 |
| 字段级变换，例如把字符串转数字、把枚举收敛到约束内 | 管道 | 管道直接影响控制器参数形状 |
| 统一响应体、耗时、返回数据加工、脱敏 | 拦截器 | 包裹 `handle()` 与 `Observable` 出站 |
| 缓存包装、降级兜底、重试等返回语义增强 | 拦截器 | 可在出站侧统一处理返回与错误通道 |
| 未捕获异常的稳定响应体与监控 | 异常过滤器 | 异常出口集中映射 |

再以一则请求把各站串起来。假设 `PATCH /cats/123?published=true`，`body` 为更新数据。请求先经过全局与模块中间件，例如打访问日志、写入追踪号。接着守卫判断当前主体是否有权更新该资源。通过后进入拦截器入站侧，例如开始计时。随后管道处理各参数对象，控制器级与路由级管道对各参数对象的顺序为 `query`、`params`、`body`，参数级管道从最后一个参数执行到第一个。接着执行控制器方法，内部再调服务层做持久化。返回值沿拦截器出站侧传出，可能被统一包装。若任一步抛出未捕获异常，则跳过剩余生命周期，由最近的异常过滤器处理并返回。

## 谁先执行、谁后执行

背诵版顺序可以这样记，拿到线上现象先快速对齐执行链。

1. 入站成功路径依次是 中间件、守卫、拦截器前置、管道、控制器与服务、拦截器后置、返回响应。
2. 未捕获异常时会跳转到异常过滤器链，顺序依次是 路由级过滤器、控制器级过滤器、全局过滤器、再返回响应。
3. 拦截器入站走外到内，而拦截器出站走内到外。过滤器的顺序与拦截器出站同样是由更靠近路由的一侧开始处理，再向外扩展。

记住三条主规则，再加上两条细节，基本就能把顺序与排错一起拿下。

第一，`Guard` 在 `Pipe` 之前。未通过守卫时，通常不会继续执行后续参数绑定与校验，避免无资格请求仍消耗校验资源。

第二，拦截器是包裹式的。`intercept` 里必须调用 `CallHandler` 的 `handle()`，后面的管道与控制器才会运行。入站侧拦截器顺序为全局、控制器、路由，出站侧因 `Observable` 解析方式为先进后出，顺序变为路由、控制器、全局。于是日志里可能出现入站先打印全局再打印路由，出站先打印路由再打印全局，这不是错记，而是流式出栈机制带来的形状差异。

第三，异常过滤器的解析顺序与其它横切组件相反，从路由级到控制器级再到全局。若路由级过滤器已经处理并写出响应，外层不会再收到同一次异常。异常也不能像中间件那样在过滤器之间层层传递。需要链式效果时，需通过继承等方式实现。

两条细节如下。

第一，管道内部的顺序容易记反。同一方法上若对多个参数挂了管道，控制器级与路由级管道处理各参数对象时，顺序是 `query`、`params`、`body`。参数专属管道则按参数声明从最后一个到第一个执行，与从左到右的书写顺序不一致。排错时要按 `query`、`params`、`body` 与参数从后往前的规则核对，不要凭书写顺序猜测。

```typescript
@Patch(":id")
update(
  @Body() body: unknown,
  @Param("id") id: string,
  @Query("published") published: string,
) {
  return null;
}
```

如果 `@Body`、`@Param`、`@Query` 各自挂了参数级管道，参数级管道会先跑 `published` 再跑 `id`，最后才跑 `body`。控制器级与路由级管道对参数对象的处理顺序也同样遵循 `query`、`params`、`body` 这个方向。

第二，请求作用域会改变"谁在跑"而不只是"先跑谁后跑"。请求级作用域的控制器与提供者会在每个入站请求上新建实例，并沿依赖链向上传播。滥用请求作用域会带来额外实例化开销，和生命周期理解混在一起时，容易误以为"同一个服务在整段链路里始终是单例"。若只是为了读取当前请求对象，优先用请求上下文或显式传参，再考虑是否真的需要请求级提供者。

几组概念在文档里常被分章学习，合在一起时容易混。可以记下面三对边界。

中间件与守卫。中间件更早，只保证在 `route handler` 之前执行，不知道即将命中哪个处理器。守卫在中间件之后，在拦截器与管道之前，持有 `ExecutionContext`，可知 `handler` 与类，适合结合 `@Roles()` 等元数据。

守卫与管道。守卫回答能不能继续，管道回答参数是否合法、是否需要变形。顺序上守卫在前，无资格时通常不必再跑完整参数校验。

拦截器与过滤器。拦截器包裹成功与失败两条路径上的 `Observable`，可用 `catchError` 观察管道、控制器、服务抛出的错误。过滤器面向未捕获异常的最终出口，与业务内 `try` 捕获的错误无关。二者都可能接触异常，但触发条件与职责不同。

## 横切逻辑放错会怎样

把逻辑放到不合适的层，通常表现为症状不一致、覆盖不到、或排查成本突然变高。下面用几种最常见错位来对照你会遇到的现象。

1. 鉴权写进中间件，代码想读 `@Roles()`。中间件阶段没有方法级元数据，结果往往是规则永远不生效，或者要维护一份路径到角色的映射表。
2. `DTO` 校验写进守卫或控制器里。校验失败时会让守卫阶段和返回路径变得混乱，而且需要在多个接口重复写同一套错误组织方式。
3. 统一响应包装写进控制器。你会在每个接口复制相同结构，后续想改字段也要批量改代码。返回阶段横切本来更适合交给拦截器。
4. 异常格式化写进业务里的 `try/catch`。一旦某条链路漏了 `catch`，全局异常响应就会出现风格漂移。未捕获异常也无法统一落到同一个过滤器逻辑。
5. 返回值缓存写在中间件。中间件先于路由语义，只能看到原始请求，拿不到最终返回值。缓存命中封装通常应发生在拦截器的返回路径附近。
6. 重试或降级放在错误的层。比如把不幂等操作也重试，会放大下游故障的影响。也可能导致错误被 `Observable` 链捕获后进入了你没有预期的分支。

把这些错位和本章的顺序图对上号，你会很快知道下一步应该去哪个组件里修，而不是在业务方法里到处加日志。

## 为什么理解链路顺序非常重要

`Nest` 里大量代码不是业务核心，而是横切在请求通道上的增强逻辑，例如日志、鉴权、权限、参数转换、`DTO` 校验、统一响应、耗时统计、异常整形。这些能力分别落在中间件、守卫、管道、拦截器、过滤器上，插入节点不同，能拿到的上下文也不同。

中间件贴近原始 `request` 与 `response`，不知道即将执行的是哪个 `handler`。守卫与拦截器、过滤器使用 `ExecutionContext`（或 `ArgumentsHost`），能切换到 `HTTP`、`RPC`、`WebSocket` 等宿主，并拿到 `getHandler()`、`getClass()`，从而结合 `@Roles()`、`@SetMetadata()` 等元数据做细粒度控制。管道专注入参形状与合法性。拦截器包住 `handle()` 返回的 `Observable`，适合统一包装返回值、在返回路径上做加工。过滤器只处理未捕获异常，负责把异常映射成稳定的 `HTTP` 响应。

若顺序理解错误，会出现鉴权写在中间件里却要读路由元数据、在管道里做放行判断、在守卫里做 `DTO` 转换这类错层。把逻辑放在能拿到正确上下文的一层，才能减少控制器与业务方法里的重复样板。

理解顺序之后，你就能更快判断错放逻辑的表现。例如把鉴权写进中间件却又想读 `@Roles()`，你会在中间件阶段就拿不到 `handler` 元数据。把 `DTO` 校验写进守卫或控制器，你会得到更难测试的分支，并且失败时仍执行了大量后续处理。把统一响应写进控制器，你会在每个接口里复制相同代码，而不是把责任交给拦截器。

使用 `@Res()` 手动接管原生响应时，许多依赖 `Nest` 响应流的拦截器行为会失效或不适用。理解这一点能避免"明明挂了全局拦截器却包不住响应"的困惑。

## 链路问题的排查思路

排查时你不需要把每一层都写成 verbose 日志。只要在关键节点统一打点，你就能快速判断卡在了哪一步。

1. 入口打点：请求进来时的 path、方法、trace id。这个点对应中间件或入口层。
2. 守卫打点：进入 `canActivate` 时打印当前 `handler`、需要的角色或策略、以及拒绝原因。
3. 管道打点：在校验或转换失败处输出参数类型信息与错误字段路径。
4. 拦截器打点：在 `intercept` 的请求侧和响应侧分别打印开始与结束耗时，必要时打印是否走了 `catchError` 分支。
5. 过滤器打点：在过滤器的 `catch` 里打印异常类型、映射后的状态码与错误码。

先确认异常有没有被业务自己捕获。只有未捕获异常会触发异常过滤器，生命周期其余部分会被跳过。若在 `service` 里 `try` 吃掉错误并返回默认值，全局过滤器不会介入，这不是过滤器注册失败，而是异常没有冒泡。

再确认绑定层级。全局、控制器、路由三级叠加时，中间件与守卫、管道大致是由外到内推进，而拦截器出站阶段会以相反方向叠加。过滤器从局部到全局。排查时给每层打带前缀的日志，对照你期望的方向，重点核对拦截器出站是否呈反向、管道参数是否从最后一个参数开始。

路径与前缀是否一致。全局前缀、版本、`nginx` 剥路径都会导致中间件看到的路径与控制器声明不一致，表现为"中间件执行了、路由匹配不到"或相反。

再确认你写的组件是否真的能触发。比如需要返回值加工的拦截器，如果控制器使用了 `@Res()` 接管响应，可能拿不到 Nest 默认的响应流。需要参数级校验的管道，如果参数没有正确挂载，或装饰器与管道绑定位置不对，也可能表现为"看起来没跑"。

若现象与时间有关，检查是否在拦截器里对 `Observable` 使用了会改变顺序或重复订阅的算子，以及是否在过滤器与拦截器里双重写响应导致 `headers already sent`。

最后把排查过程收成清单，省时间也更稳定。

1. 异常是否未捕获。若被 `try/catch` 吃掉，异常过滤器不会触发。
2. 顺序是否符合预期。守卫在管道之前，拦截器出站方向与入站相反。
3. 绑定层级是否正确。全局、控制器、路由、参数级都可能影响执行。
4. 路径坐标系是否一致。中间件看到的路径要能和路由声明对上。
5. 响应流是否被接管。使用 `@Res()` 时要重新评估拦截器与包装策略是否仍适用。
6. `Observable` 是否重复订阅或改变时序。含 `retry`、`catchError`、`share` 等算子时要额外核对。

## 小结
把请求当作一条处理主线来记：中间件→守卫→拦截器入站→管道→控制器与业务→拦截器出站。若出现未捕获异常，就会进入异常过滤器并映射为稳定响应返回。

最容易混淆的只有三点：拦截器入站与出站方向相反、过滤器从路由级逐层扩展到全局、管道在参数级别上从后往前执行。

排查优先从“异常是否未捕获”开始，然后核对顺序、绑定层级、路径坐标系以及响应流是否被接管。
