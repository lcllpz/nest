经典设计模式的书里，每种模式都有清晰的意图。放到 `NestJS` 之后，你会发现类名未必像 `Factory` 或 `Strategy` 那样直白，但意图仍会落在几个可辨识的切口上：对象由谁创建、何时可被替换、以及横切逻辑插入请求链的哪一段。

下面这份笔记采用“对齐定位点”的方式。你读完一遍后，在项目代码里会更快抓住对应模式的影子：看到 `useFactory` 你就想到工厂，看到注入令牌你就想到策略，看到作用域你就想到生命周期，看到适配器和平台选择你就想到适配，而守卫、拦截器、管道、异常过滤器则把 AOP 的味道带进来。

把这些模式和 `NestJS` 的常见 API 写法对齐之后，读源码和写模块会少很多猜测，也更容易把重构落在正确边界。

## 工厂模式

工厂模式的核心在于，把“创建过程”从调用方里抽离出来。调用方不直接 `new` 具体类，而是让一段可替换的工厂按条件或配置产出实例。依赖方向通常落在抽象接口或令牌上，而不是具体实现类。

在 `NestJS` 里，最直接的对应是工厂型提供者（`useFactory`）以及返回 `DynamicModule` 的模块装配方法（例如 `forRoot`、`forRootAsync`、`register` 一类写法）。

工厂型提供者可以是同步函数，也可以返回 `Promise`。函数参数通过 `inject` 声明依赖，由容器在需要时解析依赖并执行工厂。它特别适合这些场景：连接串、第三方客户端、按环境拼出来的配置对象，以及需要根据运行时条件决定用哪套实现的情况。

下面是一段最小示例，展示如何用 `ConfigService` 拼出只含连接信息的 `DbConfig`，工厂本体在注释里按 `Nest` 的提供者写法注册。

```typescript
import { Injectable } from "@nestjs/common";

interface DbConfig {
  url: string;
}

@Injectable()
export class ConfigService {
  getDbUrl(): string {
    return process.env.DATABASE_URL ?? "postgres://localhost/app";
  }
}

// 在 Module 的 providers 中注册示意：
// {
//   provide: "DB_CONFIG",
//   useFactory: (config: ConfigService): DbConfig => ({
//     url: config.getDbUrl(),
//   }),
//   inject: [ConfigService],
// }
```

与 `useClass`、`useValue` 相比，`useFactory` 负责的是创建过程本身，而不是直接给出一个类或已有实例。你可以把它理解成，当对象创建依赖环境和运行时信息时，把“拼装逻辑”放到工厂里，让调用方只关心注入结果。

再进一步看模块级工厂。`forRoot`、`forRootAsync`、`register` 等返回 `DynamicModule` 的写法，本质是模块装配阶段按参数生成一份模块定义，把导入列表、提供者、导出一次性配好。它更像是“把一整组依赖按配置装起来的产品接口”。

```typescript
import { DynamicModule, Module } from "@nestjs/common";
import { ConfigModule, ConfigService } from "@nestjs/config";

interface DbConfig {
  url: string;
}

export const DB_CONFIG = "DB_CONFIG";

@Module({})
export class DbModule {
  static forRootAsync(): DynamicModule {
    return {
      module: DbModule,
      imports: [ConfigModule],
      providers: [
        {
          provide: DB_CONFIG,
          useFactory: (config: ConfigService): DbConfig => ({
            url:
              config.get<string>("DATABASE_URL") ?? "postgres://localhost/app",
          }),
          inject: [ConfigService],
        },
      ],
      exports: [DB_CONFIG],
    };
  }
}
```

手写 `new Service()` 时，创建细节往往散落在业务方法、启动脚本和测试夹具里。工厂把创建收进容器的注册表，测试里替换实现、换假对象会更直接，也更容易保证“替换点只有一个”。

## 策略模式

策略模式描述的是一族可互换的算法或行为。调用方只看到统一接口，真正选择哪一种由运行期配置决定，从而避免在调用方堆很长的分支。

在 `NestJS` 里，最常见的策略落点不是到处写 `if`，而是利用依赖注入把“选择点”集中到模块装配处。你可以把识别标准简化成一句话：被调用方依赖抽象接口或注入令牌，而被装配方用配置决定到底注入哪一个实现。

下面用认证类策略做一个更贴近容器装配的示例。

```typescript
import { Inject, Injectable, Module } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";

interface User {
  id: string;
}

export interface AuthStrategy {
  validate(token: string): Promise<User | null>;
}

@Injectable()
class JwtAuthStrategy implements AuthStrategy {
  async validate(token: string): Promise<User | null> {
    if (!token) return null;
    return { id: "user-from-jwt" };
  }
}

@Injectable()
class LocalAuthStrategy implements AuthStrategy {
  async validate(token: string): Promise<User | null> {
    if (!token) return null;
    return { id: "user-from-local" };
  }
}

export const AUTH_STRATEGY = Symbol("AUTH_STRATEGY");

@Injectable()
class AuthService {
  constructor(@Inject(AUTH_STRATEGY) private readonly strategy: AuthStrategy) {}

  async login(token: string): Promise<User | null> {
    return this.strategy.validate(token);
  }
}

@Module({
  providers: [
    JwtAuthStrategy,
    LocalAuthStrategy,
    {
      provide: AUTH_STRATEGY,
      useFactory: (
        config: ConfigService,
        jwt: JwtAuthStrategy,
        local: LocalAuthStrategy,
      ): AuthStrategy => {
        const mode = config.get<"jwt" | "local">("AUTH_MODE") ?? "jwt";
        return mode === "local" ? local : jwt;
      },
      inject: [ConfigService, JwtAuthStrategy, LocalAuthStrategy],
    },
    AuthService,
  ],
  exports: [AuthService],
})
export class AuthModule {}
```

此时 `AuthService` 既不关心 `JWT` 还是本地账号，也不需要写分支。策略选择只发生在模块注册时，这也是策略模式在 `NestJS` 里最容易维护的一种形态。

测试时同样如此。你可以在测试模块里直接覆盖 `AUTH_STRATEGY` 的提供者，用假的策略对象替换实现，从而保持被测类签名不变、只对边界负责。

## 单例模式与默认作用域

单例表示在约定范围内只有一个实例。`NestJS` 里 `@Injectable()` 默认对应 `Scope.DEFAULT`。在单个应用进程里通常只构造一次，后续注入拿到同一引用。更关键的是，语义上它是由容器管理生命周期，而不是业务代码里自己维护全局变量。

下面是一个适合共享内存缓存的示例，注意默认仍是单例，内部 `Map` 在多次注入间共享。

```typescript
import { Injectable, Scope } from "@nestjs/common";

@Injectable() // 默认 Scope.DEFAULT，容器内单例
export class CacheService {
  private readonly store = new Map<string, string>();

  get(key: string): string | undefined {
    return this.store.get(key);
  }

  set(key: string, value: string): void {
    this.store.set(key, value);
  }
}
```

若改成 `Scope.REQUEST`，每个进入容器的请求会拿到独立实例。这让你可以安全地把“请求级上下文”写进实例字段，而不用额外做线程隔离。`Scope.TRANSIENT` 则表示每次注入都可能新建，适合轻量、无共享状态、又希望隔离生命周期的对象。

```typescript
import { Inject, Injectable, Scope } from "@nestjs/common";
import { REQUEST } from "@nestjs/core";
import type { Request } from "express";

type RequestWithTrace = Request & { traceId?: string };

@Injectable({ scope: Scope.REQUEST })
export class RequestContextService {
  constructor(@Inject(REQUEST) private readonly req: RequestWithTrace) {}

  getTraceId(): string {
    return this.req.traceId ?? "unknown";
  }
}
```

理解默认单例有助于判断，为什么连接池、缓存客户端这类有状态的共享资源常放在服务里。也能避免在单例里不小心引入请求级可变字段，导致跨请求串数据。至于一致性与并发，仍然是异步并发下的管理问题，不需要把它理解成多线程写锁的同一套模型。

## 适配器模式

适配器解决的是两边接口对不上的问题。中间包一层，把一侧的调用翻译成另一侧能接受的 API。上层依赖的是适配器暴露的稳定形状，从而减少对底层实现的耦合。

`NestJS` 里最显眼的是 `HTTP` 平台适配。框架允许你接 `Express` 或 `Fastify`。业务层依然写控制器、管道、守卫这一套抽象，换底层引擎时不必重写路由结构。

你在启动入口看到的“平台选择”就是适配器的可执行形态，例如用 `ExpressAdapter`。

```typescript
import { NestFactory } from "@nestjs/core";
import { ExpressAdapter } from "@nestjs/platform-express";
import * as express from "express";

async function bootstrap() {
  const expressApp = express();
  const app = await NestFactory.create(
    AppModule,
    new ExpressAdapter(expressApp),
  );
  await app.listen(3000);
}
```

`WebSockets` 的 `WsAdapter`、微服务里的传输层适配，同样属于把外部协议栈收敛成框架认得的扩展点。文档里出现 `adapter` 时，可以按“换实现、保持上层用法不变”来理解。

## 装饰器语法与经典装饰器模式

设计模式里的装饰器模式，通常指运行时给对象再包一层，在不改原类定义的前提下叠加职责。本质是对象组合，让行为按需叠加。

而 `TypeScript` 与 `NestJS` 里的 `@Controller()`、`@Injectable()`、`@UseGuards()` 更接近“声明式元数据”。它们主要承担元数据与声明钩子，配合反射让容器在启动期组装管线。你不需要把它们想成在运行期对已有实例一层层包裹。

不过从体验上看，你仍然能把 `Nest` 装饰器理解成“扩展了调用链”。差别在于，扩展由框架调度完成，而不是你手写一个装饰器类树。

下面用一个常见的自定义元数据示例把思路对齐。`@Roles()` 只负责声明；`RolesGuard` 负责读取并把声明映射到执行结果。

```typescript
import {
  CanActivate,
  ExecutionContext,
  Injectable,
  SetMetadata,
} from "@nestjs/common";
import type { Request } from "express";

export const ROLES_KEY = "roles";
export const Roles = (...roles: string[]) => SetMetadata(ROLES_KEY, roles);

@Injectable()
export class RolesGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const roles = Reflect.getMetadata(ROLES_KEY, context.getHandler()) as
      | string[]
      | undefined;

    if (!roles || roles.length === 0) return true;

    const req = context
      .switchToHttp()
      .getRequest<Request & { user?: { roles?: string[] } }>();

    const userRoles = req.user?.roles ?? [];
    return roles.some((r) => userRoles.includes(r));
  }
}
```

你会发现这类代码的关键不在“装饰器对象怎么包”，而在“元数据如何被框架读取，以及读取结果如何影响执行分支”。

## 发布订阅模式

发布订阅让发送方不必知道谁在处理，订阅方也不必知道事件从哪发出。中间用事件总线或消息中间件解耦，把“业务动作”与“后续反应”结构性分离。

在 `NestJS` 里，最常用的落地方式是把领域事件当作副作用的表达。例如用户注册完成后，发送欢迎邮件和写审计日志可以各自独立订阅，彼此不需要耦合在同一个业务方法里。

```typescript
import { Injectable } from "@nestjs/common";
import { EventEmitter2, OnEvent } from "@nestjs/event-emitter";

export class UserRegistered {
  constructor(public readonly userId: string) {}
}

@Injectable()
export class UserService {
  constructor(private readonly events: EventEmitter2) {}

  async register(userId: string): Promise<void> {
    this.events.emit("user.registered", new UserRegistered(userId));
  }
}

@Injectable()
export class UserEventSubscriber {
  @OnEvent("user.registered")
  async onUserRegistered(event: UserRegistered): Promise<void> {
    void event.userId;
  }
}
```

也要分清同步方法调用与事件。事件适合通知、最终一致性、削峰填谷。如果调用方必须立刻拿到返回值，应使用普通服务方法，或微服务里的请求响应式调用，而不是指望事件回调。

## 模块化、依赖注入与横切关注点如何叠在一起

单独强调其中任何一个词都偏窄，合在一起才贴近 `Nest` 的整体设计。

- 模块化：用 `@Module` 划边界，用 `imports`、`exports` 控制可见性，避免全局单例到处长。动态模块让配置驱动装配、按环境切换整组依赖成为可能。
- 依赖注入：用容器拼出可测试、可替换的对象图。工厂与策略都依赖注入来选实现，业务里少写裸 `new`，边界更清晰。
- `AOP`：守卫、拦截器、管道、异常过滤器把鉴权、日志、校验、错误映射从业务方法里横切出去，减少复制粘贴。

三者的大致分工可以用下图概括。

![20260330162818](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260330162818.png)

模块回答哪些类参与应用以及如何分组。依赖注入回答这些类如何创建、如何连在一起。`AOP` 回答请求进入业务方法前后，横切逻辑插在管线的哪一段。

如果你在读某段代码时不确定它属于哪一层，可以先问自己它是在“定义对象图”，还是在“创建与替换对象”，最后才是“把横切逻辑挂在哪段执行链”。把定位点问清，理解会快很多。

## 小结

上文逐条对照了书名里常见的模式与 `Nest` 里的落点，这里再收成一份便于回看的总表，方便和源码、文档对照。

- 工厂：`useFactory`、返回 `DynamicModule` 的 `forRoot` 与 `register` 一类工厂方法，把实例创建和模块装配写进配置，而不是散落在业务里。
- 策略：同一抽象多套实现配合自定义注入令牌，把策略选择集中到模块装配处，减少控制器和业务方法里的长分支。
- 单例与作用域：默认 `Scope.DEFAULT` 接近进程内单例，适合连接池与缓存一类共享资源，`REQUEST` 与 `TRANSIENT` 则在需要按请求或按次隔离时打破这一约定。
- 适配器：`HttpAdapter`、`WsAdapter`、微服务传输层适配，把 `Express`、`Fastify`、各类协议栈接到统一扩展点上，上层写法保持稳定。
- 装饰器语法与经典装饰器模式：`Nest` 与 `TypeScript` 装饰器管的是元数据与声明，GoF 装饰器管的是运行期包一层，链式横切落在守卫、拦截器、管道一侧。
- 发布订阅：进程内可用 `@nestjs/event-emitter`，跨进程落到消息主题与队列，与必须同步返回值的普通方法调用、请求响应式微服务调用区分开。

这些模式不是彼此独立的技巧清单，它们都叠在同一条主线上，也就是前面一节里的模块化、依赖注入与 `AOP`。读官方文档时若觉得概念交叉，可以先回到本节总表定位自己关心的模式，再按专题翻到模块、`IoC`、装饰器、`ExecutionContext`、`AOP` 几篇把细节补全，整体图景会连贯得多。
