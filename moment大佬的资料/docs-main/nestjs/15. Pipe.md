`Pipe` 实现 `PipeTransform` 接口，在参数绑定进控制器方法之前，对入参做一次变换或校验。它拿到的通常是已经按位置解析出来的原始值或对象，而不是整份原始 `HTTP` 报文，因此职责边界比中间件更靠内、更贴近方法签名。

`transform` 方法接收当前参数值和 `ArgumentMetadata`，可以返回同步值，也可以返回 `Promise`。若认为入参不合法，应抛出异常，常见做法是抛出 `BadRequestException`，由框架映射成 `4xx` 响应。校验通过则把返回值交给后续绑定逻辑，控制器方法最终拿到的就是管道出口的值。

管道可以挂在全局、控制器、路由或单个参数上，解析顺序遵循官方文档里的说明，全局先于控制器，控制器先于路由，同一层级里按绑定顺序叠加。与中间件、守卫不同，管道直接参与"这个参数最终长什么样"，适合把字符串变成数字、把普通对象变成带校验的 `DTO` 实例、去掉多余字段等。

## 参数校验、转换、清洗

这三件事经常一起出现，但在代码里可以拆成多层管道或一个 `ValidationPipe` 配齐。

校验解决的是"值是否合法、是否满足业务约束"。配合 `class-validator` 与 `class-transformer` 时，常把 `body` 转成 `DTO` 类实例，再按装饰器规则检查属性类型、长度、枚举、嵌套对象。非法时直接失败，避免脏数据进入服务层。

转换解决的是"类型与形状是否符合方法期望"。例如路径上的 `id` 在传输层总是字符串，方法里要的是数字，用 `ParseIntPipe` 把 `"123"` 变成 `123`。查询串里的布尔、枚举也可以在这一步统一成 `TypeScript` 侧的类型。

清洗解决的是"多给的数据要不要丢掉、缺省要不要补上"。`ValidationPipe` 的 `whitelist` 会剥掉 `DTO` 上没有声明的字段，降低批量赋值风险。`transform` 与 `enableImplicitConversion` 一类选项可以把字符串形式的数字、布尔自动转成目标类型。默认值可以用 `DefaultValuePipe` 或在 `DTO` 里用装饰器表达。

下面是一个在方法参数上使用内置管道的示意，仅说明管道出现在参数装饰器里的写法，路径与业务名需按项目替换。

```typescript
import { Controller, Get, Param, ParseIntPipe } from "@nestjs/common";

@Controller("items")
export class ItemsController {
  @Get(":id")
  findOne(@Param("id", ParseIntPipe) id: number) {
    return { id };
  }
}
```

`ParseIntPipe` 会拦截非数字字符串并抛出 `BadRequestException`，控制器方法里拿到的 `id` 已是 `number`。若希望缺参时不报错，可以使用内置选项里的 `optional` 等开关，具体以当前版本文档为准。

## 内置 Pipe 的使用

`Nest` 自带一批常用管道，减少手写样板。它们大多在失败时抛出 `BadRequestException`，错误信息可配合 `ValidationPipe` 的 `exceptionFactory` 做统一格式。

典型内置管道包括下面这些。

- `ValidationPipe`：与 `class-validator`、`class-transformer` 配合，校验并转换类 `DTO`，支持 `whitelist`、`forbidNonWhitelisted`、`transform` 等选项，常在 `main` 里 `useGlobalPipes` 挂全局。
- `ParseIntPipe`、`ParseFloatPipe`、`ParseBoolPipe`：把字符串参数解析成对应标量类型，可配置可选参数。
- `ParseEnumPipe`：把字符串约束在枚举取值内。
- `ParseUUIDPipe`：校验 `UUID` 格式。
- `ParseArrayPipe`：把查询串解析成数组，并配合项级管道。
- `DefaultValuePipe`：在参数缺失时提供默认值，常与其它解析管道组合。

全局注册适合统一 `API` 的校验策略，例如禁止未声明字段、统一转换查询与 `body`。局部注册则适合某个路由需要更严或更松的规则，避免全局一条规则套死所有接口。

下面演示在入口挂全局 `ValidationPipe` 的常见写法。一般在 `NestFactory.create` 拿到应用实例之后立刻调用 `useGlobalPipes`。生产环境往往还会关闭过于详细的校验错误信息，或接入自己的日志与国际化。

```typescript
import { ValidationPipe } from "@nestjs/common";
import type { INestApplication } from "@nestjs/common";

function applyGlobalValidation(app: INestApplication): void {
  app.useGlobalPipes(
    new ValidationPipe({
      whitelist: true,
      forbidNonWhitelisted: true,
      transform: true,
    }),
  );
}
```

把 `applyGlobalValidation` 放进 `bootstrap` 里，或在多环境配置里按开关调整 `ValidationPipe` 选项，避免各入口复制粘贴同一坨注册代码。

## 自定义 Pipe

自定义管道同样是实现 `PipeTransform<TInput, TOutput>`，在类上标记 `@Injectable()`，以便注入配置服务、仓储或其它提供者。`ArgumentMetadata` 里带有 `type`、`metatype`、`data` 等字段，需要按参数下标或类型做分支时可以用。

下面是一个极简的自定义管道，把字符串转成非负整数，非法时抛 `BadRequestException`。真实项目里可以抽成可复用的工厂，或把上下界做成构造参数。

```typescript
import {
  Injectable,
  PipeTransform,
  ArgumentMetadata,
  BadRequestException,
} from "@nestjs/common";

@Injectable()
export class PositiveIntPipe implements PipeTransform<string, number> {
  transform(value: string, _metadata: ArgumentMetadata): number {
    if (!/^\d+$/.test(value)) {
      throw new BadRequestException("id must be a non-negative integer");
    }

    const n = Number.parseInt(value, 10);
    if (!Number.isSafeInteger(n) || n < 0) {
      throw new BadRequestException("id must be a non-negative integer");
    }

    return n;
  }
}
```

示例工程里有两个路由，你访问时就能对照看到内置管道和自定义管道的差别。

- `GET /items/123` 使用内置 `ParseIntPipe`，把字符串参数解析为 `number`
- `GET /items/positive/123` 使用自定义 `PositiveIntPipe`，只允许非负整数

比如传入 `-1` 或 `abc` 这种非法值，`PositiveIntPipe` 会直接抛出异常，框架返回 `400`。下面是对应的输出截图：

![20260330171601](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260330171601.png)

在参数上使用方式和内置管道一样，写成 `@Param("id", PositiveIntPipe)` 即可。如果管道需要做异步校验，把 `transform` 改成 `async`，返回 `Promise`。框架会等待管道解析完成，再把结果交给控制器方法。

另一种常见做法是封装第三方校验库，例如 `Joi`。在构造函数里注入 `schema`，在 `transform` 里调用校验并把错误映射成 `BadRequestException`。同一份输入通常只选一套校验策略就够了。

## Pipe 在请求链路中的位置

按请求生命周期来说，请求到达控制器方法之前，先经过守卫，再跑拦截器的前置逻辑，然后把参数交给管道完成绑定与校验，最后才进入控制器方法。

管道抛异常时，后面的异常处理链路会接住它。你可以把下面这张简图当作顺序索引，方便和守卫、拦截器对应起来看。

如下图所示。

![20260330173032](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260330173032.png)

流程顺序是守卫 -> 拦截器前置 -> 管道 -> 控制器方法。虚线表示管道抛出异常时进入异常处理链路。

管道内部还有细粒度顺序。官方文档说明，同一方法上若对多个参数挂了管道，参数级管道的执行顺序与参数声明顺序相反，即从最后一个参数到第一个参数，控制器级与路由级管道仍按文档所述的全局到局部叠加。排查"为什么先跑了这个管道"时，需要同时看方法签名顺序和类上的 `@UsePipes` 声明。

管道与守卫的分工可以简单记成，守卫回答"能不能进这个方法"，管道回答"参数绑成什么样才算合法"。与守卫、拦截器一样，管道可以 `inject` 依赖，因此复杂规则不必写在纯函数里，但仍应避免在管道里做重型业务查询，以免每个参数都重复打穿数据库，必要时缓存或下沉到服务层。

## 小结

- `Pipe` 在参数进入控制器方法之前运行，负责变换与校验，出口值即方法参数最终值。
- 校验、转换、清洗常与 `ValidationPipe`、`ParseIntPipe` 等内置管道配合，全局与局部注册可组合，按需收紧或放宽策略。
- 自定义管道实现 `PipeTransform`，可依赖注入，执行顺序需结合官方文档里全局、控制器、路由、参数级以及参数顺序的规则理解。
- 在整条链路中，管道位于守卫与拦截器前置阶段之后、控制器方法之前，与守卫、拦截器的职责边界不同。

把本篇与中间件、守卫、拦截器、`ExecutionContext` 几篇连起来读，可以更清楚每一层该放什么逻辑，避免在管道里做鉴权、在中间件里做 DTO 校验这类错层。
