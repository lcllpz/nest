注册与登录本质上在解决同一个问题：用户身份如何建立、如何被验证，以及后续请求如何被持续识别。

一条稳妥且容易落地的实践是，把链路拆成注册、登录、可选刷新三段。`access token` 保持短有效期，`refresh token` 负责续期并支持撤销、轮换，这样能兼顾体验与安全性。

## 密码加密

密码加密的目标很直接：数据库泄露后不能还原明文，同时提高暴力破解成本。实现上直接用成熟密码哈希库，不要自己手写散列逻辑。

要点如下。

1. 使用带盐的慢哈希算法，例如 `argon2` 或 `bcrypt`。
2. 保存的是哈希字符串，不要保存明文密码。
3. 验证时使用库提供的 `verify` 能力，避免自己拼接盐与参数。
4. 失败信息保持模糊，避免暴露“用户名存在但密码错误”的判别面。
5. 保存哈希的参数标记，便于后续算法升级或重新计算。
6. 登录失败要配合限流与告警，降低暴力破解的成功率。

下面是一个 `argon2` 的简化示例。

```typescript
import * as argon2 from "argon2";

export async function hashPassword(plain: string): Promise<string> {
  return argon2.hash(plain);
}

export async function verifyPassword(
  hashed: string,
  plain: string,
): Promise<boolean> {
  return argon2.verify(hashed, plain);
}
```

常见错误是用 `sha256`、`md5` 这类快速散列或可逆加密（例如 `AES`）来处理密码，这两种方式都不适合密码存储。另一个高风险问题是把密码写进日志，排查再方便也不能这么做。

## JWT 基础

`JWT` 本质是“可验签的身份凭证”，不是“加密保险箱”。客户端能读 payload，但不能在不破坏签名的前提下篡改内容，所以安全重点不在“藏字段”，而在“严格验签 + 严格验声明 + 合理过期策略”。

落地时抓住六件事：

1. 签发时选定算法与密钥策略（例如 `HS256`、`RS256`）。
2. 校验时至少检查签名与 `exp`，按需补 `iss`、`aud`。
3. payload 建议保留最小字段：`sub`、`iat`、`jti`。
4. 明确区分 `access token` 与 `refresh token` 的用途与入口。
5. `access token` 走短有效期，`refresh token` 走服务端可撤销机制。
6. 需要密钥轮换时，用 `kid` 或多版本密钥校验方案平滑过渡。

经验上，`access token` 宁可短、不要长，长期会话交给 `refresh token`；payload 只放必要标识，不放密码哈希、密钥片段等敏感信息。

下面给一个签发 access token 的示例（写法示意）。

```typescript
type JwtAccessPayload = {
  sub: string;
};

type JwtOptions = {
  expiresInSeconds: number;
};

export function createAccessToken(
  sign: (payload: JwtAccessPayload, options: JwtOptions) => string,
  userId: string,
  options: JwtOptions,
): string {
  const payload: JwtAccessPayload = { sub: userId };
  return sign(payload, options);
}
```

在 NestJS 里，JWT 签发与校验通常由 `JwtModule` 与自定义的 `JwtStrategy` 配合完成。

## Token 送达与存储

token 怎么发、怎么存，会直接影响风险面。一个常见组合是：`access token` 放内存并通过 `Authorization` 头发送，`refresh token` 放 `HttpOnly` cookie 仅用于换新。

这样做的核心收益是把风险拆开：`access token` 短期可控，泄露窗口小；`refresh token` 脚本不可读，且服务端可撤销。若 refresh 走 cookie，记得加 `CSRF` 防护（同站策略、双重 token 等），避免“自动带 cookie”被滥用。

只要 refresh 能被识别、轮换、撤销，你就能在用户体验和安全性之间取得更稳定的平衡。

## Passport 在 NestJS 中的角色

`Passport` 在 `NestJS` 里的价值，不是“替你做业务鉴权”，而是把认证流程标准化。你可以把它理解成一条固定管线：提取凭证、校验凭证、产出用户上下文，然后把业务判断留在自己的服务层。

工程里常见分工如下：

1. `AuthModule` 负责组织认证能力并对外导出依赖。
2. `Strategy` 负责凭证提取与有效性校验。
3. `AuthGuard` 负责触发策略并把校验结果挂到请求上下文。

典型组合通常是：`LocalStrategy` 处理登录入站校验，`JwtStrategy` 保护需要登录态的业务接口。

## 本地登录与 Token 登录

本地登录这条链路可以拆成四步：客户端提交账号密码、`LocalStrategy` 执行校验、`AuthService` 签发令牌、客户端在后续请求里携带 access token。注册、登录本身一般不挂 `JWT Guard`，真正需要身份的业务接口再挂 `JWT Guard`，这样职责更清晰。

下面是一个更接近 `NestJS` 实战的 `LocalStrategy` 写法示意。

```typescript
import { Injectable, UnauthorizedException } from "@nestjs/common";
import { PassportStrategy } from "@nestjs/passport";
import { Strategy } from "passport-local";

type LoginUser = {
  id: string;
  username: string;
};

type AuthServiceLike = {
  validateUser(username: string, password: string): Promise<LoginUser | null>;
};

@Injectable()
export class LocalAuthStrategy extends PassportStrategy(Strategy, "local") {
  constructor(private readonly authService: AuthServiceLike) {
    super({ usernameField: "username", passwordField: "password" });
  }

  async validate(username: string, password: string): Promise<LoginUser> {
    const user = await this.authService.validateUser(username, password);
    if (!user) {
      throw new UnauthorizedException("用户名或密码错误");
    }
    return user;
  }
}
```

Token 登录时，客户端通过 `Authorization: Bearer <token>` 携带 access token。`JwtStrategy` 负责校验签名与过期时间，校验通过后把用户标识挂到请求上下文，后续控制器就可以拿到当前登录用户。

在实战里，`JwtStrategy` 建议只做“token 可信性”校验与最小上下文恢复，不要把复杂权限判断塞在 `validate` 里。角色、资源级权限、租户隔离等规则，交给 `Guard` 或业务服务层处理，会更容易维护。

下面是一个 `JwtStrategy` 的写法示意。

```typescript
import { Injectable, UnauthorizedException } from "@nestjs/common";
import { PassportStrategy } from "@nestjs/passport";
import { Strategy, ExtractJwt } from "passport-jwt";

type JwtPayload = {
  sub: string;
  jti: string;
  exp: number;
};

type JwtUser = {
  userId: string;
  tokenId: string;
};

@Injectable()
export class JwtAuthStrategy extends PassportStrategy(Strategy, "jwt") {
  constructor() {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: process.env.JWT_ACCESS_SECRET ?? "dev_secret",
    });
  }

  async validate(payload: JwtPayload): Promise<JwtUser> {
    if (!payload?.sub) {
      throw new UnauthorizedException("无效 token");
    }

    return { userId: payload.sub, tokenId: payload.jti };
  }
}
```

## 刷新令牌与轮换（推荐做法）

如果你使用 refresh token，核心目标只有一个：让会话可续期，但令牌泄露后不能长期复用。落地上通常记录 refresh 的 `jti`，并在每次刷新时执行“旧 token 作废 + 新 token 发放”。

常见规则可以这样约定。

1. 登录或刷新成功时，发放新的 refresh token，并生成新的 `jti`。
2. 服务端把旧 `jti` 标记为已使用或已撤销，避免重放。
3. 如果检测到旧 `jti` 再次出现，可以选择吊销该用户全部 refresh，强制重新登录。
4. 退出登录时删除 refresh cookie，并在服务端撤销对应的 refresh 状态。

最后提醒两点：

1. 如果你在接口里使用 `@Res()` 手动接管响应，要确认拦截器与异常链是否仍按预期生效，避免认证失败返回形态不一致。
2. refresh 轮换依赖服务端状态，别把它做成“纯无状态”；否则 token 一旦泄露，撤销能力会明显下降。

## 小结

在 `NestJS` 里，`Passport` 提供的是认证流程骨架，`LocalStrategy` 与 `JwtStrategy` 分别解决登录校验与请求鉴权，业务规则仍应留在 `Guard` 或服务层。真正稳定的线上方案，关键不只在“能签发 token”，还在 access 的短周期策略、refresh 的轮换撤销策略，以及异常响应在整条链路上的一致性。
