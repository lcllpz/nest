`HTTP` 场景下，中间件是进入路由与 `Nest` 管线之前最先碰到的一层。它工作在底层适配器暴露的 `request`、`response` 和 `next` 这一套上，适合处理与具体控制器方法、参数绑定关系不大的横切逻辑。

在 `Nest` 里，中间件通常写成可注入的类，实现 `NestMiddleware` 接口，在实现 `NestModule` 的模块里通过 `configure` 挂到路径上。典型用途包括下面这些。

- 打访问日志、生成或透传 `request id`。
- 配置 `CORS`、安全响应头、压缩等与传输相关的策略。
- 对 `Cookie` 或 `Authorization` 做存在性、格式的粗校验，把明显非法的请求挡在外面。
- 包装与 `Express` 生态兼容的现成中间件，例如 `helmet`、`compression`，在外层包一层 `NestMiddleware` 再交给 `apply`。

中间件执行顺序由 `MiddlewareConsumer` 的注册顺序决定。同一前缀上可以多次 `apply`，链上多个中间件按注册先后依次执行，全部调用 `next()` 之后，才进入 `Nest` 的守卫、管道、拦截器与控制器。中间件拿不到 `ExecutionContext`，也拿不到方法级元数据，这是它和守卫、拦截器的重要分界线，后面几节会展开。

从整链视角看，一次 `HTTP` 请求大致经过下面这条路径。中间件始终贴在适配器一侧，离原始请求最近。

下面是一个最小骨架，展示类如何实现 `NestMiddleware` 并在模块里对路由生效。`forRoutes` 既可以写路径与方法枚举，也可以传入控制器类，让中间件覆盖该控制器声明的全部路由。

```typescript
import {
  Injectable,
  Module,
  NestMiddleware,
  NestModule,
  MiddlewareConsumer,
  RequestMethod,
} from "@nestjs/common";
import type { Request, Response, NextFunction } from "express";

@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction): void {
    console.log(`${req.method} ${req.url}`);
    next();
  }
}

@Module({})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer): void {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes({ path: "hello", method: RequestMethod.GET });
  }
}
```

`use` 方法可以改成 `async`，在内部 `await` 异步初始化或远程探测，只要最终要么调用 `next()`，要么直接结束响应。若中间件里抛出同步错误，或把错误传给 `next(err)`，行为与 `Express` 错误处理管线一致，仍由底层适配器承接。忘记调用 `next()` 又未向客户端写响应时，请求会一直挂起，排查这类问题时优先检查分支是否漏了 `next`。

若要对全局前缀下的所有路由生效，需要结合 `app.setGlobalPrefix` 一起理解路径匹配。`exclude` 常用于健康检查、静态资源路径，避免被日志或鉴权中间件重复打满。

```typescript
// 示意：同一 configure 内可多次 apply，整体顺序仍先于守卫
// consumer
//   .apply(LoggerMiddleware)
//   .exclude({ path: "health", method: RequestMethod.GET })
//   .forRoutes(CatsController);
// consumer
//   .apply(AnotherMiddleware)
//   .forRoutes({ path: "dogs", method: RequestMethod.ALL });
```

上面片段仅作结构示意，类名与路径需与项目一致。`exclude` 与 `forRoutes` 的组合能精细控制作用面，比手写路径字符串更不容易和控制器改名脱节。

## 日志、跨域、Token 粗校验

这三类需求在中间件里出现频率很高，但各自侧重点不同。

日志类中间件只依赖原始请求与响应对象，不关心业务参数怎么解析，放在中间件里很合适。可以记录方法、路径、状态码、耗时，也可以把 `trace id` 写进响应头，方便前后端联调。若需要统计"业务结果码"或慢查询，往往还要结合拦截器或服务层埋点，因为中间件阶段业务尚未执行完毕。

跨域与安全头往往直接复用 `Express` 中间件或适配层提供的等价能力。在 `Nest` 里除了自己写 `NestMiddleware`，也可以在 `main.ts` 里对 `INestApplication` 取底层实例后 `use`，效果上仍是请求进入业务逻辑之前执行。团队需要统一约定，避免同一类头在 `main`、功能模块、`nginx` 等多处重复设置，排查时找不到到底是谁覆盖谁。

`Token` 粗校验指的是只判断有没有、格式像不像，例如 `Bearer` 是否拆开、字符串是否为空。中间件里适合尽早拒绝明显不合法的请求，减少无效流量进入后续管线。若校验通过，常见做法是把解析结果挂到 `req.user` 或自定义属性上，需要在项目里用模块扩充为 `Request` 声明这些字段，并与守卫、控制器里的类型约定对齐。细粒度的角色、资源级权限仍应放在守卫或服务里，见最后一节。

## Express Middleware 与 Nest Middleware 的关系

默认 `HTTP` 适配器使用 `Express` 时，`NestMiddleware` 的 `use` 方法签名与 `Express` 中间件一致，参数仍是 `Request`、`Response`、`NextFunction`。从运行方式上看，`Nest` 只是把中间件纳入模块配置与依赖注入，执行时仍是适配器上的管道。

在模块里用 `MiddlewareConsumer.apply(SomeMiddleware)` 注册的是 `Nest` 管理的中间件类。若使用 `app.use` 在根实例上挂载，则是直接操作底层 `Express` 应用，不经过 `Nest` 的 `apply` 链，但执行顺序上仍属于进入框架路由前的层。全局 `app.use` 与模块 `configure` 混用时，要对照官方说明和实际打印顺序理解先后，不要只靠直觉。

为了让执行顺序更直观，下面用同一套代码给你做完整对照。示例里用 `NestMiddleware` 在 `GET /hello` 上做日志耗时统计。

main.ts（全局 Express 风格中间件）

```typescript
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  await app.listen(process.env.PORT ?? 3000);
}

bootstrap().catch((err) => {
  console.error(err);
  process.exitCode = 1;
});
```

logger.middleware.ts（NestMiddleware 类）  

```typescript
import { Injectable, NestMiddleware } from '@nestjs/common';
import type { NextFunction, Request, Response } from 'express';

@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction): void {
    const startedAt = Date.now();

    res.on('finish', () => {
      const durationMs = Date.now() - startedAt;
      console.log(
        `[middleware] ${req.method} ${req.originalUrl} - ${res.statusCode} - ${durationMs}ms`,
      );
    });

    next();
  }
}
```

app.module.ts（把 NestMiddleware 挂到某些路由）  

```typescript
import { MiddlewareConsumer, Module, NestModule, RequestMethod } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';
import { LoggerMiddleware } from './logger.middleware';

@Module({
  imports: [],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer): void {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes({ path: 'hello', method: RequestMethod.GET });
  }
}
```

![20260330170349](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260330170349.png)

同一个请求的直观理解是，请求命中 `GET /hello` 后会先执行 `LoggerMiddleware`，然后才进入由 Nest 管理的守卫、管道、拦截器与控制器流程。

第三方 `Express` 中间件多数是 `(req, res, next) => void` 形态，可以包在一个 `NestMiddleware` 类里，在 `use` 中调用该函数。这样既不破坏包本身的实现，又能放进 `MiddlewareConsumer` 与依赖注入体系。

使用 `Fastify` 时，底层不是 `Express`，部分纯 `Express` 中间件不能原样复用，需要换用 `Fastify` 插件或官方推荐的适配方式。`Nest` 文档会说明平台差异，选型时要读清当前 `HTTP` 适配器，避免假设所有中间件行为与 `Express` 完全一致。

依赖注入方面，实现 `NestMiddleware` 的类可以标记 `@Injectable()`，由容器解析构造函数依赖，例如注入配置服务、日志服务。这与裸函数形式的 `Express` 中间件不同，后者无法直接享受 `Nest` 的注入，除非在外面包一层工厂提供者或手写闭包。

## Middleware 的应用边界

中间件适合处理路由已经匹配到应用、但尚未进入具体控制器方法这一大段区间内的共性需求。一旦逻辑需要知道"是哪个处理器方法"、"方法上挂了哪些元数据"、"路由参数名叫什么"，就更适合交给守卫、管道或拦截器。

可以按下面几条判断是否该用中间件。

- 只依赖原始请求、响应、路径、头、查询串，不依赖 `Controller` 上的装饰器信息，中间件通常够用。
- 需要访问 `ExecutionContext`、反射 `handler` 与 `class` 上的元数据，应使用守卫或拦截器。
- 需要对已解析的 `body`、`query`、`param` 做校验与变换，管道更合适。
- 需要在返回客户端前后包装结果、统计耗时、统一异常外观，拦截器或过滤器更贴切。
- `WebSocket`、`Microservice` 等非 `HTTP` 入口不走同一套 `HTTP` 中间件，需要在该传输层自己的扩展点里处理。

下面这张表只做粗粒度对照，具体选型仍以业务为准。

| 能力                    | 中间件             | 守卫           | 管道             | 拦截器               |
| ----------------------- | ------------------ | -------------- | ---------------- | -------------------- |
| 接触原始 `req` 或 `res` | 直接               | 多经上下文封装 | 多针对已解析参数 | 可包进出的执行       |
| 方法级元数据            | 无                 | 有             | 有               | 有                   |
| 典型场景                | 日志、跨域、粗校验 | 鉴权、角色     | 校验与变换       | 日志、缓存、包装响应 |

全局中间件与模块级中间件可以并存，要注意顺序。`main.ts` 里越早 `use` 的通常越早执行。模块里 `configure` 注册的顺序也会影响链式调用先后。排查问题时可以打日志确认实际顺序，避免和文档心理模型不一致。

## 为什么复杂权限不适合写在 Middleware

复杂权限往往同时依赖身份、角色、资源标识、数据范围、业务规则，还需要和控制器上的元数据对齐，例如"这个路由只对管理员开放"或"只能操作自己的订单"。这类信息在 `Nest` 里通常通过自定义装饰器加在方法或类上，由守卫在运行时读取 `Reflector` 与 `ExecutionContext` 做决策。

中间件拿不到方法级元数据，也不知道最终会派发到哪个 `handler`。若在中间件里硬编码路径与权限的映射表，路径一改就要同步改中间件，和路由声明两套真相，维护成本高。中间件里也不适合塞大量异步业务查询，否则每个请求都会执行一遍，即便后面被守卫拒绝，已经浪费了数据库或下游调用。

更合理的分工是下面这样。

- 中间件做轻量、与业务弱耦合的检查，例如 `JWT` 是否存在、格式是否合法，可选地把解析后的主体挂在请求上。
- 守卫做与路由、角色、策略绑定的鉴权，能注入服务、读数据库、组合 `Passport` 策略。
- 服务层再做资源归属、细粒度数据权限，避免把全部规则堆在管线最外层。

这样权限逻辑集中在可测试的守卫与服务里，中间件保持薄而稳定，请求链路也更容易推理。

## 小结

- 中间件贴近底层 `HTTP` 适配器，处在守卫、管道、拦截器之前，适合日志、跨域、安全头、`Token` 粗校验等与具体处理方法无关的横切逻辑。
- `MiddlewareConsumer` 支持按路径、方法、控制器类注册，支持 `exclude`，可与全局前缀组合使用。`NestMiddleware` 与 `Express` 中间件在默认栈上形状相近，`Nest` 额外提供模块级注册与依赖注入，换用 `Fastify` 时要核对兼容性。
- 需要元数据、方法上下文或复杂鉴权时，应把主逻辑放在守卫与业务服务中，而不是在中间件里堆分支与映射表。需要包装响应或做方法级计时，优先考虑拦截器。

把本篇与前面 `AOP`、守卫与拦截器、`ExecutionContext` 几篇对照阅读，可以更清楚每一层该承担什么，避免整条链路上重复做同一件事。
