数据库访问方式是否合适，会长期牵动重构与换栈的成本。先说明原生 `SQL`、查询构建器与 `ORM` 各自适合什么场景，再落到 `NestJS` 里如何组织模块与完成连接初始化。涉及 `PrismaClient` 与 `CLI` 的示例按当前推荐写法编排，细节仍以官方文档为准。

## ORM、查询构建器与原生 SQL

三者要处理的是同一类问题，也就是把应用数据可靠地写入数据库并在需要时读回，只是抽象层次与可控性不同。

原生 `SQL` 最直接，表达能力强，复杂报表、批量维护、强依赖数据库特性的语句往往最顺手。代价是字符串拼接容易埋坑，跨库移植差，类型与结构要靠团队纪律补全。

查询构建器（例如 `Knex` 一类）在字符串与完全手写 `SQL` 之间加了一层链式 `API`，参数绑定更自然，也能做一定程度的复用。它通常不替你管表与实体的一一映射，领域模型仍要自己在服务层维护。

`ORM` 把表、关联、迁移、（部分产品还有）仓储模式绑进一套模型里，业务代码更多面向对象与关系而不是裸 `SQL`。上手快、协作成本低，遇到极复杂的查询或性能尖峰时，往往要退回原生或构建器写一小段兜底查询。

可以粗略记成一条轴：控制力与表达自由度从原生 `SQL` 到 `ORM` 递增，样板代码与约定从 `ORM` 到原生 `SQL` 递增。没有绝对更好的那一端，只有更贴合团队熟练度与业务复杂度的选择。

## TypeORM 与 Prisma 的定位

在 `Nest` 生态里，`TypeORM` 出现早、和装饰器、`Repository` 模式结合紧密，文档与示例多，很多教程默认就是它。它与 `TypeScript` 的装饰器和类型系统贴得很紧，迁移、实体、关系映射都在一套装饰器语法下完成，团队若熟悉类与元数据风格，读起来会很顺。

`Prisma` 把 schema 当成单一真相来源，客户端由 `schema` 生成，类型与查询 `API` 跟模型强绑定。迁移、`Prisma Client`、（可选的）`Prisma Studio` 同属一条工具链。当前线路里客户端形态更精简，连接方式与工程配置往往集中在 `Prisma Config` 与驱动适配器上，接线时按这一路线组织即可。

两者都能支撑中大型项目。更实用的比较维度往往是：团队更偏向围绕实体类组织模型，还是更偏向以 `schema` 文件为单一来源、是否需要大量手写优化级 `SQL`、以及你对迁移与工作流工具的偏好。

## 如何选择适合自己的方案

先问三个很具体的问题，比列功能表更有用。

第一，团队主力语言经验。若大部分人从 `Java`、`.NET` 背景过来，熟悉实体类与仓储，`TypeORM` 的心智负担通常更低。若更在意从数据库结构一键得到类型安全的客户端，`Prisma` 往往更讨喜。

第二，查询复杂度与性能诉求。报表、统计、批量任务占比高时，任何 `ORM` 都会出现写法很漂亮但性能或表达能力跟不上的情况。与其在 `ORM` `API` 上硬拧，不如接受在服务层单独留一块用原生 `SQL` 或查询构建器的区域。

第三，运维与协作成本。迁移谁跑、种子数据谁维护、本地与 `CI` 是否同构，这些流程比框架名字更能决定日常体验。`Prisma` 的 `CLI` 默认依赖根目录的 `prisma.config.ts`，且往往不会自动加载环境变量，需要在脚本里显式引入 `dotenv` 或等价方案，与早年只靠 `schema` 里写死连接串的习惯不同。

若项目使用 `MongoDB`，先在官方文档确认当前 `Prisma ORM` 对该数据源的支持范围，再决定选型。

## 建立数据库连接

以 `PostgreSQL` 与 `Prisma ORM` 为例，演示在 `NestJS` 里常见的接法。其他关系型数据库同理，换成对应官方适配器包即可。

### 环境与依赖

部署与开发前对照官方文档，确认 `Node` 与 `TypeScript` 满足当前工具链要求。`prisma` 与 `@prisma/client` 应使用相同主版本并同步升级，避免生成物与运行时脱节。

安装层面，除 `@prisma/client` 外，运行时需要通过驱动适配器接入数据库。以 `pg` 为例，需要 `@prisma/adapter-pg` 与 `pg` 本体。`CLI` 侧建议安装 `dotenv`，并在调用 `prisma` 命令前加载环境变量（`CLI` 通常不会默默读取 `.env`）。

### `schema` 与 `prisma.config.ts`

生成器宜采用 `prisma-client` 提供方，并且必须在 `generator` 里显式写 `output`，客户端不再默认落在 `node_modules`。`datasource` 块里的 `url`、`directUrl` 等宜迁到 `Prisma Config`，由根目录的 `prisma.config.ts` 统一给 `CLI` 提供连接信息。

`prisma/schema.prisma` 示例：

```prisma
datasource db {
  provider = "postgresql"
}

generator client {
  provider = "prisma-client"
  output   = "../src/generated/prisma"
}
```

项目根目录 `prisma.config.ts` 示例：

```typescript
import "dotenv/config";
import { defineConfig, env } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: {
    path: "prisma/migrations",
  },
  datasource: {
    url: env("DATABASE_URL"),
  },
});
```

初始化、改模型之后，记得单独执行 `npx prisma generate`。当前 `migrate dev` 与 `db push` 往往不会顺带执行 `generate`，需要把生成步骤写进脚本或流水线。种子数据通常也不随迁移自动执行，需要时使用 `npx prisma db seed` 显式触发。

### 模块格式说明

`Prisma` 以 `ESM` 为主发布。新项目若将 `package.json` 的 `type` 设为 `module`，并按官方建议调整 `tsconfig.json` 的 `module` 与 `moduleResolution`，与官方文档最一致。已有 `NestJS` 工程若仍是 `CommonJS` 构建，需要把构建链路与 `Prisma` 生成物的模块格式对齐，具体以当前 `Nest` 版本与打包工具为准，必要时查阅官方文档中与 `ESM` 相关的说明。

### 在 `NestJS` 中注入 `PrismaClient`

构造 `PrismaClient` 时必须传入适配器。一种常见的 `PrismaService` 写法是在应用启动时连接，在进程退出前断开，例如：

```typescript
import { Injectable, OnModuleDestroy, OnModuleInit } from "@nestjs/common";
import { PrismaPg } from "@prisma/adapter-pg";
import { PrismaClient } from "../generated/prisma/client";

@Injectable()
export class PrismaService
  extends PrismaClient
  implements OnModuleInit, OnModuleDestroy
{
  constructor() {
    const adapter = new PrismaPg({
      connectionString: process.env.DATABASE_URL,
    });
    super({ adapter });
  }

  async onModuleInit(): Promise<void> {
    await this.$connect();
  }

  async onModuleDestroy(): Promise<void> {
    await this.$disconnect();
  }
}
```

`import` 路径要和你 `generator output` 的实际目录一致，示例里假定生成结果在 `src/generated/prisma`。生产环境更推荐通过 `ConfigService` 读取连接串，而不是直接使用 `process.env`，为突出 `Prisma` 初始化本身而写短。

若使用 `Prisma Accelerate`，不要把加速层 `URL` 传给 `PrismaPg` 这类直连适配器，应按官方文档使用扩展（`extension`）与 `accelerateUrl` 的组合。直连与加速两条路径不要混在同一套构造参数里。

### 连接池与 `SSL`

通过适配器走底层驱动（如 `node-pg`）的连接池时，默认超时等行为与过去内置查询引擎时代的默认值可能不同。若出现连接挂起或超时，需要按驱动文档调整池参数，而不是只盯着连接字符串里的几项开关。

`SSL` 校验策略也更贴近 `Node` 侧默认。若遇到证书校验错误，要么按安全要求配置可信根证书，要么在明确接受风险的前提下在适配器选项中调整 `ssl`（仅作过渡手段）。

### `TypeORM` 在 `NestJS` 中的接法（对照）

若选型是 `TypeORM`，通常使用 `@nestjs/typeorm` 的 `TypeOrmModule.forRootAsync`，把 `host`、`port`、`username` 等放进 `DataSource` 配置，实体通过 `forFeature` 注入各模块。迁移可以用 `TypeORM` 自带命令或团队选定的流程。与 `Prisma` 不同，`TypeORM` 不遵循同一套 `Prisma Config` 与适配器约定，配置以所选版本的官方文档为准。

## 数据访问层的职责

无论用哪种技术栈，建议在 `NestJS` 里把所有数据库访问收敛到清晰的一层，而不是在控制器里散落查询。

服务层对外暴露用例级别的接口，例如 `createOrder`、`findUserByEmail`，内部再调用 `PrismaService` 或 `Repository`。控制器只负责 `HTTP` 形状、状态码与 `DTO` 映射。

这一层适合承担事务边界、幂等键、乐观锁版本号、软删除条件等与持久化强相关、又与 `HTTP` 无关的规则。跨表一致性尽量用显式事务表达，避免在多个独立提交里拼运气。

查询返回什么字段，最好在服务层做裁剪或映射，不要把 `ORM` 实体或 `Prisma` 返回的整坨对象直接当成对外契约。后面换字段名或拆表时，你会感谢这一层缓冲。

使用 `Prisma` 时还要留意：旧的 `$use` 中间件已移除，横切关注点请改用 `Client Extensions`。内置 `Metrics` 若已不再提供，指标应改用驱动或扩展自行暴露。

连接在独立模块里完成初始化，查询从服务层进入，控制器保持单薄，这样即使日后更换具体的数据库技术，模块边界仍然站得住脚。

## 小结

把数据库访问当成一条清晰的工程线来做：选好持久化方案，控制好原生 `SQL`、查询构建器与 `ORM` 的边界；在 `NestJS` 里把连接初始化放在独立模块，把查询与事务放进服务层；需要用 `Prisma` 时，按适配器与生成物的约定完成客户端初始化，同时把 `generate`、`seed` 这类步骤显式纳入流程即可。
