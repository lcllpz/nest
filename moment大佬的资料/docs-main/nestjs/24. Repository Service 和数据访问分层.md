# 24. Repository Service 和数据访问分层

> 所属部分：第六部分 数据库与数据建模

## Entity 是什么

在数据访问分层里，`Entity` 更偏向领域层概念，指一个业务对象以及它应当遵守的规则。它关注的是业务不变量，以及状态变化是否被允许。

它回答的是：

- 这个业务对象是什么，它有哪些必须成立的属性
- 对外如何表达它的状态变化，以及变化是否允许

`Entity` 不等同于数据库表或 `ORM` 模型。数据库表结构会随着迁移变化，而领域对象更希望保持稳定，避免每次改表就推翻一整套业务逻辑。

实践上，把字段映射与默认值等持久化细节放在 `Repository`，把业务不变量与行为放在 `Entity`。比如邮箱是否允许为空、状态流转是否越过中间态、一次更新是否需要同时调整多个字段，这类规则都更适合写成领域对象的方法。

一个简单经验是，一旦你发现业务规则散落在 `Controller` 或 `Service` 的一堆 `if` 里，通常意味着 `Entity` 的封装还不够清晰。

## Repository 负责什么

`Repository` 负责把领域对象落地到数据库，并把数据库返回的数据还原回领域对象。它让上层只关心业务用例，不必理解具体的表结构与查询方式。

更具体地说，它通常承担这些工作：

- 隐藏底层持久化细节，例如查询条件、分页参数、`Prisma` 或 `TypeORM` 的调用方式
- 完成从持久化模型到领域 `Entity` 的映射，以及反向映射
- 提供以业务用例为导向的数据接口，例如按 `id` 查、按唯一字段查、保存或更新

`Repository` 尽量不要承载业务决策。它的职责是把数据拿到、把数据存进去，而不是决定业务上能不能这样做。能不能的判断属于 `Service` 或领域对象本身。

方法签名建议尽量稳定。读接口通常返回领域对象或 null，写接口返回成功或让调用方拿到持久化后的结果。关于异常语义，建议在 `Service` 里把数据库层的错误转换为业务可理解的异常，例如冲突与资源不存在，避免把底层错误细节直接暴露给上层。

## Service 和 Repository 不要混写

把 `Service` 和 `Repository` 混在一起，最常见的后果是：

- Service 变成查询拼装器，后续加条件、改表、换 `ORM` 都会牵连大范围改动
- Repository 里开始出现大量业务 `if`，导致难以复用，也很难写单元测试
- 当需要跨多个持久化操作时，事务边界会越来越模糊

更稳的分工方式是：

- `Service` 写业务规则与编排逻辑。它决定调用哪些 `Repository` 方法、按什么顺序校验与变更状态，也负责事务边界
- `Repository` 专注数据读写与映射。它提供可用的数据，而不是可执行的业务动作

如果某个业务规则非常接近数据结构，例如唯一性约束或可选字段的基本校验，也建议让它落在领域层 `Entity` 里，或者把规则作为 Service 的前置条件，避免 Repository 变成规则仓库。

通常也不要让 `Controller` 直接调用 `Repository`。`Controller` 负责 HTTP 形状与 DTO 映射，具体用例如何编排与校验交给 `Service`，才能保持层次干净。

## 业务逻辑与数据访问解耦

解耦带来的收益很直接：

- 测试更容易。你可以把 `Repository` 用测试替身替换掉，只测试业务规则与编排流程，而不必依赖数据库
- 换库或调整查询成本更可控。Repository 内部可以重构查询与映射，Service 的业务用例接口尽量保持不变
- 更好维护事务与错误处理。事务边界放在 Service，错误由 Service 统一转成可理解的异常，再由上层处理

实践中建议把数据访问的变动限定在 Repository。表字段调整、索引优化、查询条件微调，尽量都在 Repository 内完成，让 Service 只关心业务语义。

当一个用例需要多个读写操作时，优先在 `Service` 里组织事务。实现层面通常有两种选择：让 Service 自己持有事务客户端并传给 Repository，或让 Repository 提供带事务参数的执行路径。无论选哪种，事务边界都应当是可控且对用例透明的。

## 一个标准的数据访问分层示例

下面用 `User` 作为示例，展示 `Entity`、`Repository`、`Service` 三层如何配合。示例里用 `Prisma` 作为持久化实现，你也可以把实现换成 `TypeORM` 或原生 `SQL`，上层接口不需要跟着大改。

先看领域对象 `User`。它只关心业务状态与允许的变更，不直接依赖数据库模型。

```typescript
export interface UserProps {
  id: string;
  email: string;
  name: string;
  createdAt: Date;
}

export class User {
  readonly id: string;
  private email: string;
  private name: string;
  readonly createdAt: Date;

  constructor(props: UserProps) {
    this.id = props.id;
    this.email = props.email;
    this.name = props.name;
    this.createdAt = props.createdAt;
  }

  getEmail(): string {
    return this.email;
  }

  getName(): string {
    return this.name;
  }

  changeEmail(nextEmail: string): void {
    this.email = nextEmail.trim();
  }

  changeName(nextName: string): void {
    this.name = nextName.trim();
  }

  toPersistenceCreateInput(): {
    id: string;
    email: string;
    name: string;
    createdAt: Date;
  } {
    return {
      id: this.id,
      email: this.email,
      name: this.name,
      createdAt: this.createdAt,
    };
  }
}
```

接下来定义 `Repository` 接口。`Service` 只依赖接口，不关心具体是 `Prisma` 还是其它实现。

```typescript
import { User } from "./user.entity";

export interface UserRepository {
  findById(id: string): Promise<User | null>;
  findByEmail(email: string): Promise<User | null>;
  save(user: User): Promise<void>;
}
```

然后提供一个 `Prisma` 实现。它把读写与映射集中在这里，业务判断仍由 `Service` 或领域对象承担。

```typescript
import { Injectable } from "@nestjs/common";
import { PrismaClient } from "../generated/prisma/client";
import { User } from "./user.entity";
import { UserRepository } from "./user.repository";

@Injectable()
export class PrismaUserRepository implements UserRepository {
  constructor(private readonly prisma: PrismaClient) {}

  async findById(id: string): Promise<User | null> {
    const record = await this.prisma.user.findUnique({
      where: { id },
    });
    if (!record) return null;

    return new User({
      id: record.id,
      email: record.email,
      name: record.name,
      createdAt: record.createdAt,
    });
  }

  async findByEmail(email: string): Promise<User | null> {
    const record = await this.prisma.user.findUnique({
      where: { email },
    });
    if (!record) return null;

    return new User({
      id: record.id,
      email: record.email,
      name: record.name,
      createdAt: record.createdAt,
    });
  }

  async save(user: User): Promise<void> {
    const input = user.toPersistenceCreateInput();
    await this.prisma.user.upsert({
      where: { id: user.id },
      create: input,
      update: {
        email: input.email,
        name: input.name,
      },
    });
  }
}
```

最后是 `UserService`。它负责编排用例流程，例如注册时要检查是否已存在，然后再创建与保存。事务也通常在这里组织：当涉及多个 `Repository` 调用时，把事务边界交给 `Service` 会更清晰。

```typescript
import { ConflictException, Injectable } from "@nestjs/common";
import { User } from "./user.entity";
import { UserRepository } from "./user.repository";

@Injectable()
export class UserService {
  constructor(private readonly users: UserRepository) {}

  async register(input: {
    id: string;
    email: string;
    name: string;
  }): Promise<User> {
    const existed = await this.users.findByEmail(input.email);
    if (existed) {
      throw new ConflictException("email already exists");
    }

    const user = new User({
      id: input.id,
      email: input.email,
      name: input.name,
      createdAt: new Date(),
    });

    await this.users.save(user);
    return user;
  }
}
```

如果后续扩展到更新邮箱、修改名称等用例，依然遵循同一条原则。`Service` 决定做什么，`Repository` 决定怎么读怎么写，领域 `Entity` 决定规则怎么表达。

## 小结

稳定的数据访问分层形成一条闭环：`Entity` 表达业务规则，`Repository` 只负责读写与映射，`Service` 负责用例编排与事务边界。把数据库细节收敛到 `Repository`，把业务决策放在 `Entity` 与 `Service`，后续改表、换 ORM、重构查询都能把影响面控制在局部。
