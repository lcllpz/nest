前一篇里，`IoC` 和容器解释了对象怎么被创建和注入。继续往下看，自然会问：框架凭什么知道某个类是控制器、某个方法对应哪条路由、构造函数的参数类型各是什么。

答案在装饰器和元数据里。

装饰器不是"魔法语法"，而是挂在类、方法、参数上的入口函数，用来附加结构信息，供启动阶段读取。入门可以只记一句：装饰器是在给程序结构打标签。

```typescript
import { Controller, Get } from "@nestjs/common";

@Controller("users")
export class UsersController {
  @Get()
  findAll(): string[] {
    return ["汤姆", "杰瑞"];
  }
}
```

`@Controller("users")` 标明类的角色和路径前缀，`@Get()` 标明 HTTP 方法与路由片段。表面是语法，背后是框架要消费的声明。

从 `TypeScript` 角度看，装饰器是作用于声明的特殊函数，运行时机和版本策略与编译选项有关，下面会单独提。

## 类装饰器、方法装饰器与参数装饰器

按出现位置分三类最稳。

### 类装饰器

描述类在系统里的身份，例如 `@Controller()`、`@Injectable()`、`@Module()`、`@Catch()`。

```typescript
import { Injectable } from "@nestjs/common";

@Injectable()
export class UsersService {
  findAll(): string[] {
    return ["汤姆", "杰瑞"];
  }
}
```

`@Injectable()` 不在执行业务，而是在声明"请把我纳入依赖注入图"。

### 方法装饰器

描述方法的对外语义，路由族最常见：`@Get()`、`@Post()`、`@Put()`、`@Delete()`，还有 `@UseGuards()`、`@UseInterceptors()` 等横切挂载。

```typescript
import { Controller, Get, Post } from "@nestjs/common";

@Controller("users")
export class UsersController {
  @Get()
  findAll(): string[] {
    return ["汤姆", "杰瑞"];
  }

  @Post()
  create(): { ok: boolean } {
    return { ok: true };
  }
}
```

### 参数装饰器

说明参数从请求的哪一块来：`@Param()`、`@Query()`、`@Body()`、`@Headers()`、`@Req()` 等。

```typescript
import { Body, Controller, Param, Post } from "@nestjs/common";

interface CreateUserDto {
  name: string;
}

@Controller("users")
export class UsersController {
  @Post(":id")
  create(
    @Param("id") id: string,
    @Body() body: CreateUserDto,
  ): { id: string; name: string } {
    return { id, name: body.name };
  }
}
```

三类分工可以记成：类说角色，方法说行为与横切挂载点，参数说数据从哪来。

`TypeScript` 里还有属性装饰器，`Nest` 在 HTTP 场景里用得相对少，知道存在即可。

## 元数据是什么

`metadata` 常译成"元数据"，这里指描述程序结构本身的信息，而不是业务返回值那种数据。

方法里跑的是业务逻辑，而"属于哪个控制器、HTTP 方法是什么、路径怎么拼、参数装饰器顺序如何、构造函数参数类型是什么"这类信息，就是元数据。

如果说装饰器是贴标签的动作，元数据就是标签被写进 `Reflect` 能力之后、框架真正能读到的内容。你看到的是 `@Get()`，运行时起作用的是键值形式的元数据记录。

## `reflect-metadata` 与编译开关

装饰器只负责触发注册，`Nest` 还要在启动时读回这些信息。`reflect-metadata` polyfill 了 `Reflect.defineMetadata`、`Reflect.getMetadata` 等 API，让元数据能挂在构造函数、原型和方法上。

`CLI` 生成的项目通常在入口最顶部有一句 `import "reflect-metadata"`，保证在任何使用元数据的代码之前执行。若手写启动文件，不要忘记这一点。

更关键的是 `tsconfig` 里与装饰器相关的选项。依赖注入要能根据类型自动解析构造函数参数，需要 `emitDecoratorMetadata: true`，这样编译器会发出参数类型的设计期元数据。再配合 `experimentalDecorators: true`（当前生态里 `Nest` 仍普遍走这一套传统装饰器管线）。

```json
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}
```

若关掉 `emitDecoratorMetadata`，仅写 `@Injectable()` 而不写 `@Inject`，部分场景下容器可能猜不出参数类型。日常开发保持开启最省心。

`TypeScript` 5 起引入了标准阶段装饰器，与上述"实验装饰器"不是同一条实现路径。跟 `Nest` 官方脚手架与文档对接时，以当前模板为准，不要混用两套配置。

## 自定义元数据与 `Reflector`

框架内置装饰器之外，可以用 `SetMetadata` 给类或方法挂上任意键值，供守卫、拦截器读取。典型用途是权限、公开接口标记、审计标记等。

```typescript
import { SetMetadata } from "@nestjs/common";

export const IS_PUBLIC_KEY = "isPublic";

/** 标记该路由跳过登录校验之类逻辑，具体消费方在 Guard 里实现 */
export const Public = () => SetMetadata(IS_PUBLIC_KEY, true);
```

```typescript
import { Controller, Get } from "@nestjs/common";
import { Public } from "./public.decorator";

@Controller("health")
export class HealthController {
  @Public()
  @Get()
  ping(): string {
    return "ok";
  }
}
```

在守卫或拦截器里注入 `Reflector`，用 `getAllAndOverride` 或 `get` 合并类级与方法级元数据，再决定放行与否。键名建议集中定义成常量，避免字符串散落。

## 自定义参数装饰器

`createParamDecorator` 用来封装"从 `ExecutionContext` 里算出某个参数"的重复逻辑，本质仍是参数装饰器，只是工厂帮你写好了取值过程。

```typescript
import { createParamDecorator, ExecutionContext } from "@nestjs/common";

/** 从已挂载的 request.user 取当前用户，具体谁在前面赋值由认证流程决定 */
export const CurrentUser = createParamDecorator(
  (_data: unknown, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest<{ user?: { id: string } }>();
    return request.user;
  },
);
```

```typescript
@Get("me")
getProfile(@CurrentUser() user: { id: string } | undefined) {
  return user ?? { id: "guest" };
}
```

`ExecutionContext` 在守卫、拦截器、过滤器里也会反复出现，用来在同一套 API 下切换 HTTP、RPC 等宿主。

## `NestJS` 如何用装饰器描述结构

把依赖注入和装饰器放在一起看，`Nest` 做了两件事：容器负责怎么造对象，装饰器与元数据负责这些对象在系统里扮演什么。

你很少手写一张巨大的路由表或依赖清单，而是把结构写回类与方法上：

```typescript
import { Controller, Get, Injectable, Module } from "@nestjs/common";

@Injectable()
export class UsersService {
  findAll(): string[] {
    return ["汤姆", "杰瑞"];
  }
}

@Controller("users")
export class UsersController {
  constructor(private readonly usersService: UsersService) {}

  @Get()
  findAll(): string[] {
    return this.usersService.findAll();
  }
}

@Module({
  controllers: [UsersController],
  providers: [UsersService],
})
export class UsersModule {}
```

对框架而言，这里已经包含：`UsersService` 可注入、`UsersController` 是前缀 `/users` 的控制器、`findAll` 对应 `GET /users`、控制器构造依赖 `UsersService`、模块把二者包在同一边界。业务代码与结构声明叠在同一份源码里，读起来像说明书，而不是两套东西来回对表。

## 声明式开发在说什么

声明式强调写清意图，而不是逐步手写注册、分发、解析的底层过程。`@Controller`、`@Get`、`@Injectable`、`@Module` 都是在向框架申报结构，复杂度从"自己编排流程"挪到了"理解约定与元数据管线"。

心态上可以换一句：装饰器不是死记硬背的关键字，而是你和 `Nest` 沟通用的语言。接受这一点后，路由自动注册、依赖自动解析、横切能力为什么能挂在方法上，都会连成一片。

## 小结

装饰器在类、方法、参数上打标签，元数据是标签落地后可被读取的结构信息，`reflect-metadata` 与 `tsconfig` 里的 `experimentalDecorators`、`emitDecoratorMetadata` 共同支撑这套机制，入口文件记得先引入 polyfill。`SetMetadata` 加 `Reflector` 承担自定义声明，`createParamDecorator` 把重复的参数解析收成可读名字。`Nest` 把路由、模块、注入关系写回源码，形成声明式风格，你写的是类与方法，同时也在写系统结构说明书。
