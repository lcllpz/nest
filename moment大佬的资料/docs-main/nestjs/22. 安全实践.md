安全不是单独一个中间件能包圆的。它从输入进入系统的第一行开始，经过路由、持久化、响应头，一直到配置与密钥如何存放。这一章按 `NestJS` 项目里最常见的面，把输入校验、典型 `Web` 攻击面、限流、响应头与配置安全串成一条可落地的检查线。

## 输入校验

不信任任何来自客户端的数据。查询参数、路径参数、请求体、请求头里的自定义字段，都应视为不可信输入，在进入业务逻辑之前完成类型与形状校验。

在 `NestJS` 里最常用的组合是全局 `ValidationPipe` 加上基于 `class-validator` 与 `class-transformer` 的 `DTO`。管道会在进入控制器方法之前把原始对象转成带类型的实例，并按装饰器规则校验。

```typescript
import { ValidationPipe } from "@nestjs/common";

async function bootstrap(): Promise<void> {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(
    new ValidationPipe({
      whitelist: true,
      forbidNonWhitelisted: true,
      transform: true,
      transformOptions: { enableImplicitConversion: true },
    }),
  );
  await app.listen(process.env.PORT ?? 3000);
}
```

`whitelist` 会剥掉 `DTO` 上未声明的字段，降低批量赋值与意外字段带来的风险。`forbidNonWhitelisted` 在出现多余字段时直接报错，便于及早发现客户端乱传字段。`transform` 与隐式类型转换配合，可以减少字符串数字混用导致的隐蔽逻辑错误。

`DTO` 示例：

```typescript
import { Type } from "class-transformer";
import {
  IsEmail,
  IsInt,
  IsOptional,
  IsString,
  Max,
  Min,
} from "class-validator";

export class CreateUserDto {
  @IsEmail()
  email!: string;

  @IsString()
  name!: string;

  @IsOptional()
  @Type(() => Number)
  @IsInt()
  @Min(1)
  @Max(120)
  age?: number;
}
```

复杂结构用嵌套 `DTO` 与 `@ValidateNested()`，数组用 `@IsArray()` 与 `@ArrayMinSize()` 等，避免在控制器里手写 `if` 链。文件上传、富文本等场景要在业务层再做白名单与长度限制，不要只靠前端提示。

校验失败时，`ValidationPipe` 会抛出 `400` 与结构化错误信息。生产环境建议通过异常过滤器统一脱敏，避免把内部字段名与堆栈细节直接暴露给终端用户。

## 对 SQL 注入、XSS 与 CSRF 的基础认知

这三类问题出现频率高，但落脚点不同。`SQL` 注入主要出在把不可信字符串拼进查询。`XSS` 主要出在把不可信内容当作 `HTML` 或脚本执行。`CSRF` 主要出在浏览器在已登录状态下自动带上身份凭证发起非预期请求。

### SQL 注入

使用 `TypeORM`、`Prisma` 等时，优先使用参数化查询或查询构建器占位符，不要把用户输入直接拼进原始 `SQL` 字符串。

```typescript
// 不推荐，存在注入风险
// await repo.query(`SELECT * FROM user WHERE email = '${email}'`);

// 推荐，使用参数
await repo
  .createQueryBuilder("u")
  .where("u.email = :email", { email })
  .getOne();
```

原生查询若不可避免，必须使用驱动提供的参数绑定，而不是字符串拼接。动态表名、列名若必须来自外部，应在服务端用白名单映射到固定标识，禁止把用户字符串直接当作标识符拼进 `SQL`。

### XSS

后端职责主要是：不要把未净化的用户内容当作可信 `HTML` 输出给浏览器，并在响应头侧配合内容安全策略。纯 `API` 返回 `JSON` 时，默认不会执行脚本，风险更多出在管理后台模板或将来接入的 `SSR`。若存在富文本存储，应在写入与输出策略上统一白名单标签，或由专门组件渲染。

配合安全响应头里的 `Content-Security-Policy` 可以显著降低脚本滥用面，具体见下文 `Helmet` 一节。

### CSRF

典型场景是用户已登录站点 `A`，在站点 `B` 上触发表单，浏览器自动带上 `A` 的 `Cookie` 导致非预期写操作。前后端分离且接口使用无状态 `Bearer Token`、并由前端在 `Authorization` 头显式携带时，传统 `CSRF` 面会小很多，因为第三方站点读不到本地存储里的 `token`。

若使用基于 `Cookie` 的会话，需要组合策略，例如 `SameSite` 属性、`CSRF Token`、关键写操作二次确认等。`NestJS` 侧要明确哪些路由依赖 `Cookie` 会话，并与前端约定凭证传递方式，避免混用导致防护缺口。

## 限流与防刷

公开接口、登录、验证码、重置密码等路径容易被脚本批量调用。限流不能代替业务风控，但能挡住粗粒度刷量，给数据库与下游服务减压。

常用做法是使用 `@nestjs/throttler`，按 `IP`、全局或自定义键做窗口与次数限制，并可与全局 `Guard` 组合。

```typescript
import { Module } from "@nestjs/common";
import { APP_GUARD } from "@nestjs/core";
import { ThrottlerGuard, ThrottlerModule } from "@nestjs/throttler";

@Module({
  imports: [
    ThrottlerModule.forRoot([
      {
        name: "short",
        ttl: 1000,
        limit: 3,
      },
      {
        name: "medium",
        ttl: 10000,
        limit: 20,
      },
    ]),
  ],
  providers: [
    {
      provide: APP_GUARD,
      useClass: ThrottlerGuard,
    },
  ],
})
export class AppModule {}
```

对登录失败次数、短信发送次数等，还应在业务层用账户维度或手机号维度做计数与锁定，限流只解决网络层吞吐，不解决撞库与定向攻击的全部问题。

## Helmet、CORS 与安全响应头

`Helmet` 是一组与安全相关的 `HTTP` 头中间件的集合，用来降低点击劫持、`XSS`、嗅探等常见浏览器侧风险。在 `NestJS` 里通常在 `main.ts` 对底层 `Express` 实例调用 `use`。

```typescript
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";
import helmet from "helmet";

async function bootstrap(): Promise<void> {
  const app = await NestFactory.create(AppModule);
  app.use(helmet());
  await app.listen(process.env.PORT ?? 3000);
}
```

默认配置已经包含多项合理默认值。若前端需要嵌入第三方页面或旧版客户端，某些头可能需要按需放宽，改动前应先读官方说明，避免为了省事直接关掉整块策略。

跨域方面，`enableCors` 要解决的是浏览器是否允许前端页面向你的 `API` 发起跨域请求，而不是替代认证。

```typescript
app.enableCors({
  origin: process.env.FRONTEND_ORIGIN ?? "http://localhost:5173",
  credentials: true,
  methods: ["GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"],
  allowedHeaders: ["Content-Type", "Authorization"],
});
```

生产环境应把 `origin` 配成明确的白名单，避免使用通配符同时又开启 `credentials`。与认证章节配合时，`Cookie` 场景还要关心 `SameSite` 与域名一致性。

## 敏感信息与配置安全

密钥、数据库连接串、第三方 `API Key` 不应写进仓库。`NestJS` 常用 `@nestjs/config` 从环境变量与 `.env` 文件加载配置，由部署环境注入真实值。

```typescript
import { Module } from "@nestjs/common";
import { ConfigModule } from "@nestjs/config";

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      envFilePath: [".env.local", ".env"],
    }),
  ],
})
export class AppModule {}
```

`.env` 只放本地与示例占位，`.env` 应加入 `.gitignore`。`CI` 与生产环境用平台提供的密钥管理或变量注入。日志里禁止打印完整连接串与 `token`，错误上报里要对敏感字段做脱敏。

`JWT` 密钥要足够随机且可轮换，过期时间与刷新策略要与业务风险匹配。调试接口、`Swagger`、`GraphQL Playground` 等在生产默认应关闭或加访问控制。

## 小结

安全在 `NestJS` 里靠多层叠加，而不是指望某一个中间件。输入侧用 `ValidationPipe` 与 `DTO` 收口形状，持久化侧用参数化查询降低 `SQL` 注入面，浏览器侧用 `Helmet` 与克制的 `CORS` 收紧响应头与跨域，网络上用限流挡粗刷，密钥与配置只进环境或密钥管理、不进仓库，`XSS` 与 `CSRF` 则随输出形态与会话形态分别处理。

发布前不妨对照检查全局校验是否生效、查询是否避免拼接、敏感路由是否限流、`origin` 与白名单是否按环境收紧、日志与错误是否脱敏、生产是否关掉调试面，把这几项固定进发布流程，比事后补洞成本低得多。
