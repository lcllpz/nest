权限控制是很多 NestJS 项目里最容易先跑通再返工的模块。难点往往不在权限模型本身，而在于它横跨认证、路由入口、业务规则、数据范围、缓存一致性、审计日志、前端按钮可见性，甚至还会牵动组织结构。

不少团队早期只判断登录态，或只判断某个接口谁能进，业务一复杂就会撞上这些现实问题：

- 同一接口，不同角色看到的数据范围不同，怎么收口
- 一个用户多个角色，冲突和优先级怎么定
- 角色权限变更后，在线用户多久生效
- 前端按钮藏住了，后端是否真拦住了
- 规则散落在控制器、服务、`SQL` 里，后续谁来维护

因此权限不能只理解成在路由或控制器上挂一个 `Guard`。它更像一条从身份确认到入口授权再到数据边界的完整链路。

## 认证与授权各管什么

权限设计里最常见的误区，是把认证和授权混成一件事。两者常串在同一条请求链上，职责却不同。

`Authentication` 只管身份是否可信，也就是请求是否来自某个已登录且可核对的用户。常见检查包括是否已登录、`token` 是否有效、是否过期、是否被篡改、当前请求对应哪个账号。通过后，一般会把身份信息挂到请求上下文，例如：

```typescript
req.user = {
  id: "u_1001",
  email: "test@example.com",
  roles: ["admin"],
};
```

这一阶段先弄清楚请求是谁发的，还不涉及能做哪些操作。

`Authorization` 在身份已明确的前提下，判断能否执行当前操作。例如是否允许访问 `POST /users`、能否删除某笔订单、能否导出报表、能否查看他部门数据、能否操作不属于自己的资源。

认证失败通常返回 `401 Unauthorized`。认证通过但权限不足，通常返回 `403 Forbidden`。

在 `NestJS` 里，一条典型请求大致按下面顺序走：

1. `JwtAuthGuard` 一类组件先完成认证，解析并校验 `token`
2. 认证通过后，把用户身份挂到请求上下文
3. `RbacGuard` 或其他授权 `Guard` 读取路由规则并做授权判断
4. 授权通过才进入控制器和业务层
5. 若还有资源级约束，再在服务层补数据范围校验

对应到职责上可以记成三层，下面一节会展开拆边界，这里先留个总印象：

- 认证对应能否进入系统
- 授权对应能否进入这个接口
- 资源权限对应能否动这条具体数据

三层缺一不可。

## 三层边界（认证、入口授权、资源校验）

实际项目里，权限出问题，很多时候不是没写规则，而是三层没划清。稳妥做法是让认证、入口授权、资源校验各管一段，不要混写。

身份认证只回答请求来自谁。入口授权只回答这个接口能不能进。资源校验只回答这条具体数据能不能碰。

以更新订单为例：

```http
PATCH /orders/:id
```

一般会依次经过三步。先验 `token` 确认身份，再检查是否具备 `order.update` 这类接口权限，最后按订单归属人、部门、状态判断是否允许改这条数据。

若只做前两层、不做第三层，常见后果包括：

- 用户有 `order.update`，却不该改这张具体订单
- 用户能进接口，却未必能操作他人创建的数据
- 审计可以查看订单，却不该编辑订单

接口级权限和资源级权限不是一回事。前者管入口是否可达，后者管数据边界。

## RBAC 分层模型

最常见、也最经典的是 `RBAC`，基于角色的访问控制。

核心思想是解耦。权限不直接绑在用户上，先绑到角色，再把角色分给用户。组织变化、人员变动时，优先改角色与绑定关系，而不是给每个用户逐条改权限。

一个最基础的 `RBAC` 模型通常包含四个概念：

1. `User`：系统用户
2. `Role`：角色，权限的集合载体
3. `Permission`：最小授权单元
4. `Session`：当前会话中激活的角色集合

`Session` 在很多业务里容易被忽略，在模型里却很重要。用户可以有多个角色，但每次会话未必需要激活全部。安全要求高的场景里，常会要求用户在本次会话里明确当前工作身份，例如审计、审批、运维等。

标准 `RBAC` 分层一般从下面四层讲起。

### RBAC0

`RBAC0` 是最基础的版本，只关心四个要素：用户、角色、权限、会话。

![20260331151458](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260331151458.png)

上图从左到右可以理解为，用户通过绑定获得角色，角色再承载多条权限，用户不直接挂权限，而是经由角色间接拿到授权。底下的会话表示"本次会话里激活了哪些角色"，真正参与鉴权的是当前会话里打开的那一批角色，而不是把账号下所有角色一次性全开。

落地时，`RBAC0` 实现成本最低，也是多数业务最先采用的形态，后期维护相对省心。

### RBAC1

`RBAC1` 在 `RBAC0` 基础上增加了角色继承，角色可以排成上下级，高层自动带上低层的权限，配置时不必为每个岗位重复勾选同一批能力。

![20260331160617](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260331160617.png)

上图仍保留用户、会话、权限这几块关系，中间多了一条纵向的继承链。用户照旧通过绑定拿到角色，会话照旧决定"本次激活哪几个角色"，权限照旧是跟着角色走。和 `RBAC0` 的差别在于，角色不再是完全扁平的一组名字，而是能表达"谁包含谁"。链上越靠上的节点，越会把下面几层的权限一并算进来。

例如：

- `super_admin` 继承 `admin`
- `admin` 继承 `operator`
- `operator` 继承 `viewer`

这样 `super_admin` 会顺着链条把下层角色的权限一路收齐，维护时主要改底层或中间层，顶层不必重复堆叠。组织层级清晰、岗位之间天然存在包含关系的企业系统，往往最先用到这一层。

### RBAC2

`RBAC2` 在 `RBAC0` 基础上增加了约束规则，用来卡在"分配角色"和"会话里启用角色"两条链路上，让授权不仅说得通，还能过审计。

![20260331162108](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260331162108.png)

上图仍是用户、角色、权限、会话的主干，右上角单独画出约束块，虚线指向它们各自生效的位置。典型例子是财务录入和财务复核这类职责分离，既可能写成账号级互斥，也可能写成同一次登录里不能两个身份一起开，对应下面两类约束的差别。

常见约束包括：

- 互斥角色：分配阶段就不允许同一用户同时持有两个冲突角色，从账号层面拆开职责
- 先决角色：未拿到前置角色时，不能授予后续更高风险或更靠后的角色
- 基数限制：同一角色在组织里名额有限，例如审批岗最多三个人
- 动态约束：账号上可以绑定多个角色，但本次会话只能激活其中一个，避免同一条流程里既录入又复核

这一层在金融、医疗、政务等强合规场景里尤其重要。

### RBAC3

`RBAC3` 同时包含 `RBAC1` 的角色继承和 `RBAC2` 的约束规则，也就是在 `RBAC0` 的用户、角色、权限、会话骨架上，把继承链和各类约束放进同一套模型里。

![20260331163201](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260331163201.png)

这张图把前面几节拆开讲的内容收成一张总览。左侧是用户获得角色，中间是带继承链的角色层，右侧是通过角色聚合的权限，左下角是本次会话激活的角色，右上角是约束块。图下方的演进条对应一种常见落地顺序，先跑通主流程，再补继承，再补约束，最后收敛到完整模型。

从理论上看，`RBAC3` 是标准模型里信息最全的一层。工程实践里却很少有项目一上来就做成完整 `RBAC3`，更现实的做法是：

- 先用 `RBAC0` 把主流程跑通
- 角色关系开始重复、明显需要上下级包含时，引入 `RBAC1`
- 合规和风控变严、互斥与先决、名额与会话级限制这类规则浮出水面时，引入 `RBAC2`
- 继承与约束都要长期共存、规则相对稳定时，再向 `RBAC3` 收敛

四层之间的依赖关系可以用下图理解：

![20260331163627](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260331163627.png)

如果你在项目里拿不准该停在哪一层，可以参考下面几条经验：

- 团队还在快速迭代、角色边界经常改，优先停在 `RBAC0`
- 岗位关系已经稳定、存在清晰的上下级包含，再补 `RBAC1`
- 出现互斥岗位、审批隔离、合规与名额这类硬规则，再补 `RBAC2`
- 继承与约束同时存在、且短期内不会大改，再进入 `RBAC3`

## 用装饰器声明路由规则

在 `NestJS` 里，授权规则最常见的写法是把规则写进路由的 `metadata`。控制器可以显式声明这个接口要哪些角色、哪些权限，读起来很直观。

先定义 `metadata` 的 `key` 和装饰器。

```typescript
import { SetMetadata } from "@nestjs/common";

export const ROLES_KEY = "roles";
export const PERMISSIONS_KEY = "permissions";

export function Roles(...roles: string[]): MethodDecorator & ClassDecorator {
  return SetMetadata(ROLES_KEY, roles);
}

export function Permissions(
  ...permissions: string[]
): MethodDecorator & ClassDecorator {
  return SetMetadata(PERMISSIONS_KEY, permissions);
}
```

然后在控制器中声明访问规则：

```typescript
import { Controller, Get, Post } from "@nestjs/common";

@Controller("users")
export class UsersController {
  @Get()
  @Roles("admin", "auditor")
  @Permissions("user.read")
  listUsers(): string {
    return "ok";
  }

  @Post()
  @Roles("admin")
  @Permissions("user.create")
  createUser(): string {
    return "created";
  }
}
```

优点是声明式、和路由贴得近，打开控制器就能看见这条接口的准入条件。

若不想在每个方法重复写装饰器，可以在类上先声明默认规则，再在方法上覆盖或细化。例如：

```typescript
@Controller("users")
@Roles("admin")
export class UsersController {
  @Get()
  @Permissions("user.read")
  listUsers(): string {
    return "ok";
  }

  @Post()
  @Permissions("user.create")
  createUser(): string {
    return "created";
  }
}
```

类级别适合定模块默认门槛，方法级别适合定具体动作差异。

## Guard 结合 metadata 做鉴权

`Guard` 在请求进入控制器之前执行，一般通过 `Reflector` 读取路由上的 `metadata`，再对照请求上下文里的用户身份做授权判断。

写 `RbacGuard` 之前，建议先把约定定死，至少包括下面几类：

- 类级与方法级 `metadata` 是否都要支持，方法级是覆盖类级还是与类级合并
- 角色与权限各自是按任一满足放行，还是必须全部满足才放行
- 未登录与权限不足是否用不同状态码区分
- 是否提供 `@Public()` 跳过鉴权、是否对超级管理员直接放行

下面给出一版可直接按需裁剪的示例。

```typescript
import {
  CanActivate,
  ExecutionContext,
  ForbiddenException,
  Injectable,
  UnauthorizedException,
} from "@nestjs/common";
import { Reflector } from "@nestjs/core";

export const ROLES_KEY = "roles";
export const PERMISSIONS_KEY = "permissions";
export const IS_PUBLIC_KEY = "is_public";

type CurrentUser = {
  id: string;
  roles: string[];
  permissions: string[];
};

@Injectable()
export class RbacGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);

    if (isPublic) {
      return true;
    }

    const requiredRoles =
      this.reflector.getAllAndOverride<string[]>(ROLES_KEY, [
        context.getHandler(),
        context.getClass(),
      ]) ?? [];

    const requiredPermissions =
      this.reflector.getAllAndOverride<string[]>(PERMISSIONS_KEY, [
        context.getHandler(),
        context.getClass(),
      ]) ?? [];

    const request = context.switchToHttp().getRequest<{ user?: CurrentUser }>();
    const user = request.user;

    if (!user) {
      throw new UnauthorizedException("未登录");
    }

    if (user.roles.includes("super_admin")) {
      return true;
    }

    const rolesOk =
      requiredRoles.length === 0 ||
      requiredRoles.some((role) => user.roles.includes(role));

    const permissionsOk =
      requiredPermissions.length === 0 ||
      requiredPermissions.every((permission) =>
        user.permissions.includes(permission),
      );

    if (!rolesOk || !permissionsOk) {
      throw new ForbiddenException("权限不足");
    }

    return true;
  }
}
```

实现上有几处值得单独说。

### getAllAndOverride 与覆盖顺序

`Reflector` 的 `getAllAndOverride` 会按参数顺序依次查找 `metadata`，返回第一个命中的值。示例里把 `context.getHandler()` 放在前、`context.getClass()` 放在后，因此先看方法、再看类，方法上有 `@Roles()` 等声明时优先生效，没有才回退到类级。

```typescript
this.reflector.getAllAndOverride<string[]>("roles", [
  context.getHandler(),
  context.getClass(),
]);
```

这正好对应类级默认、方法级覆盖的写法。若两处声明要合并进同一数组而不是互相覆盖，应改用 `getAllAndMerge`，并自行去重和定优先级。

### 角色与权限的匹配约定

示例里角色按任一匹配、权限按全部匹配，是常见默认，但不是唯一答案。角色偏岗位时往往满足其一即可，权限偏动作清单时往往要全部具备，也可以反过来约定，关键是团队内统一、长期稳定。

建议在内部文档里写死一份约定。下表是一版容易达成共识的参考：

| 项目         | 推荐规则     | 说明                       |
| ------------ | ------------ | -------------------------- |
| 角色匹配     | 任一匹配通过 | 偏岗位语义，满足其一即可   |
| 权限匹配     | 全部匹配通过 | 偏动作语义，通常要同时满足 |
| 元数据优先级 | 方法覆盖类   | 类默认、方法特例           |
| 未登录       | `401`        | 未认证，无有效用户上下文   |
| 权限不足     | `403`        | 已认证，授权不足           |

不少项目把两种失败都打成 `ForbiddenException`，语义含糊，排障也不利。未建立用户上下文时用 `401`，已登录但授权不够时用 `403`，前端和日志都更好区分。

## Public 与白名单路由

实际项目里总有一部分接口要绕开登录或绕开统一鉴权，例如：

- 登录
- 刷新 `token`
- 健康检查
- 接口文档
- 对外公开查询

全局挂上授权 `Guard` 时，这些路由需要白名单，否则会误拦。常见做法是定义 `@Public()`：

```typescript
import { SetMetadata } from "@nestjs/common";

export const IS_PUBLIC_KEY = "is_public";
export const Public = (): MethodDecorator & ClassDecorator =>
  SetMetadata(IS_PUBLIC_KEY, true);
```

在 `Guard` 开头优先读取：

```typescript
const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
  context.getHandler(),
  context.getClass(),
]);

if (isPublic) {
  return true;
}
```

这样授权 `Guard` 可以全局挂上，白名单路由也能照常放行。

## 用 APP_GUARD 全局挂载

权限 `Guard` 更适合用 `APP_GUARD` 全局注册，而不是在每个控制器上手动 `@UseGuards()`。默认所有请求都走同一套授权，不容易漏接口。`Guard` 作为全局 `provider` 可以注入缓存、配置、用户服务、权限服务等依赖。以后要接权限快照、灰度、审计打点，也不必逐控制器改代码。

示例：

```typescript
import { Module } from "@nestjs/common";
import { APP_GUARD } from "@nestjs/core";

@Module({
  providers: [
    {
      provide: APP_GUARD,
      useClass: RbacGuard,
    },
  ],
})
export class AuthzModule {}
```

若项目里同时有认证 `Guard` 和授权 `Guard`，职责可以这样记：前者把 `req.user` 建起来，后者基于 `req.user` 判断是否放行。谁先执行取决于注册顺序和模块组织，硬原则只有一条，授权不能早于认证。

## 前端按钮与后端强制授权

很多后台会做按钮级权限，例如缺少 `user.create` 时隐藏新增用户、缺少 `order.refund` 时隐藏退款、缺少 `report.export` 时隐藏导出报表。

这能减少误操作，也让页面更贴角色。但前端只能管展示，不能代替后端鉴权。页面和脚本都可被改动，用户也可以绕过浏览器直接请求接口，按钮看不见不等于接口不可达。

更稳妥的分工是，前端按权限做展示和按钮显隐，后端在接口入口做强制授权，资源层再补数据范围校验。体验在前端，安全在后端。

## 小结

权限控制的核心，是在认证之上把 `RBAC`、装饰器、`Guard`、资源与数据范围、缓存一致性与审计测试串成一条可演进的链路，而不是只加一个 `Guard`。落地时宜先跑通认证与 `RBAC0`，用装饰器声明规则、用全局 `Guard` 做接口级授权，再在服务与查询层补资源权限，业务变复杂时再按需引入 `RBAC1`、`RBAC2`、部门范围、临时授权、多租户与字段级权限等，避免第一天做满、后面推倒重来。上线前建议自检认证与 `req.user` 是否全项目一致、全局授权是否无漏拦、高风险数据是否在查询层带范围条件、权限变更是否在可接受时间内生效且缓存失效可验证、`401` 与 `403` 是否分清、核心接口是否覆盖权限矩阵以防越权。
