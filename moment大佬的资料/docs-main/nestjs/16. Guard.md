`Guard` 实现 `CanActivate` 接口，在请求已经通过中间件、即将进入拦截器前置阶段与管道之前，回答一个问题，即当前调用是否允许继续往下走。返回值是许可信号，不是业务数据。若返回假值或抛出特定异常，框架会中断后续管线，把拒绝结果交给异常层或默认拒绝响应。

与管道不同，守卫不负责把字符串变成数字、不负责把 `body` 洗成 `DTO`。与中间件不同，守卫能拿到 `ExecutionContext`，可以知道即将执行的是哪个类、哪个方法、路由参数长什么样，也能配合 `Reflector` 读取类与方法上的自定义元数据。因此鉴权、角色、资源级策略、特性开关这类既依赖请求又依赖路由声明的逻辑，放在守卫里比放在中间件里更自然。

守卫可以注册在全局、控制器或单个路由上，解析顺序与官方文档一致：全局先于控制器，控制器先于路由。同一装饰器上写多个守卫时，按从左到右依次执行，前一个通过才轮到下一个。

当策略需要复用时，可以把公共部分提成基类或服务。守卫本身保持薄，把复杂规则下沉到可注入的领域服务里。

`Guard` 同样适用于 `RPC` 与 `WebSocket`，通过 `context.getType()` 与 `switchToRpc`、`switchToWs` 取到对应宿主对象。本篇默认以 `HTTP` 为主，多协议写法与 "ExecutionContext 和 ArgumentsHost" 一篇一致。

全局守卫可以用 `APP_GUARD` 提供者注册，这样不用在每个控制器上重复 `@UseGuards`。代价是所有入口都会先经过该守卫，公开接口、健康检查、文档路由往往要配合白名单装饰器或专门判断，否则会出现"登录接口自己也要求已登录"这类循环。控制器级与路由级守卫适合局部加严，例如整个 `AdminController` 套一层 `RolesGuard`，某个导出接口再套一层速率或签名校验。

编写守卫时建议保持纯函数倾向。输入是 `ExecutionContext` 与已注入服务，输出是布尔或许可决策。避免在守卫里直接操作响应对象写回业务成功体。

响应的格式与包装交给拦截器或控制器，守卫只负责放行或拒绝。

## canActivate 的执行时机

按官方请求生命周期，中间件执行完毕之后，首先轮到守卫。全局守卫、控制器守卫、路由守卫按由外到内的顺序叠加，与 `Passport` 组合时常见的 `JwtAuthGuard` 也遵循同一套绑定规则。

`canActivate` 的签名接收 `ExecutionContext`，返回 `boolean`、`Promise<boolean>` 或 `Observable<boolean>`。异步场景包括查库确认会话、远程拉取权限列表、调用身份提供方做自省。

若返回 `Observable`，框架会订阅并在发出结果后决定是否放行。需要关注冷 Observable 的行为，以及错误是否会按预期传播到异常管线。

同步守卫适合纯内存判断，例如解析 `JWT` 载荷后比对角色列表、检查请求头是否存在某字段。一旦涉及 `I/O`，优先写成 `async` 方法或返回 `Promise`，避免在同步函数里用阻塞调用拖住事件循环。

返回 `true` 表示放行，后续拦截器前置阶段、管道、控制器方法会按顺序执行。返回 `false` 时，框架通常会抛出禁止访问类异常，客户端往往看到 `403`。

更常见的做法是在规则不满足时直接 `throw new UnauthorizedException()` 或 `throw new ForbiddenException()`。这样可以带上更明确的错误信息，也便于后续异常层统一整形。

守卫里抛出的其它未捕获异常同样会走异常管线，与业务代码里抛出没有本质区别。因此不要在守卫里吞掉错误又返回 `true`，除非刻意允许匿名访问。

## 登录校验、权限校验

这两件事经常被混在一个守卫里写，概念上最好拆开理解。

登录校验回答的是身份是否成立，也就是当前请求能否对应到一个已认证主体。常见做法包括校验会话 `Cookie`、校验 `Bearer` `JWT` 签名与有效期、校验 `OAuth` 访问令牌。未通过时通常返回 `401`，语义是"不知道你是谁"或"你的凭证无效"。

权限校验回答的是已认证主体能不能做这件事，涉及角色、细粒度策略、资源归属。例如管理员才能删库、用户只能改自己的资料。未通过时更常见是 `403`，语义是"知道你是谁，但你不能碰这个资源"。

工程里可以分层实现，例如下面两种。

- 认证守卫只负责把令牌解析成用户对象并挂到请求上，例如 `request.user`，失败则 `401`。
- 授权守卫或基于元数据的 `RolesGuard` 只读 `request.user` 与路由上的角色要求，失败则 `403`。

这样单元测试可以单独替换认证或授权，日志与监控也能按状态码区分"未登录"与"无权限"。若把两步揉在同一个类里，分支一多就难以维护。

角色类授权可以自定义装饰器存允许的角色列表，守卫里用 `Reflector.getAllAndOverride` 合并方法级与类级元数据，再与用户身上的角色数组求交。下面是一段常见骨架，角色名与类型需与业务用户模型一致。

```typescript
import { SetMetadata } from "@nestjs/common";

export const ROLES_KEY = "roles";
export const Roles = (...roles: string[]) => SetMetadata(ROLES_KEY, roles);
```

```typescript
import {
  CanActivate,
  ExecutionContext,
  Injectable,
  ForbiddenException,
} from "@nestjs/common";
import { Reflector } from "@nestjs/core";
import { ROLES_KEY } from "./roles.decorator";

interface RequestUser {
  roles?: string[];
}

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const required = this.reflector.getAllAndOverride<string[]>(ROLES_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!required || required.length === 0) {
      return true;
    }
    const request = context.switchToHttp().getRequest<{ user?: RequestUser }>();
    const user = request.user;
    const ok = user?.roles?.some((r) => required.includes(r));
    if (!ok) {
      throw new ForbiddenException();
    }
    return true;
  }
}
```

未标注角色的路由在示例里直接放行，若项目要求"默认拒绝"，应改成缺省元数据时返回 `false` 或抛异常，并在文档里写清约定。

`Passport` 策略里的本地登录、`JWT` 策略，多半落在认证一侧。基于 `Reflector` 读 `@Roles()` 再比对用户角色的逻辑，落在授权一侧。两者都用守卫承载，但类名与模块边界分开，后续接 `CASL`、`RBAC` 表驱动时也更顺。

## JWT Guard 示例

下面给出一套可直接改的拆分思路。核心是把 JWT 认证从控制器里剥离开：策略负责把令牌解包成用户信息，守卫负责放行或拒绝，模块负责把这些能力装进容器。

策略层使用 `@nestjs/passport` 的 `PassportStrategy`，指定 `jwt` 名称，与守卫里 `AuthGuard('jwt')` 的字符串一致。`validate` 的返回值会被挂到 `request.user`，供后续授权守卫或业务服务使用。

```typescript
import { Injectable } from "@nestjs/common";
import { PassportStrategy } from "@nestjs/passport";
import { ExtractJwt, Strategy } from "passport-jwt";

interface JwtPayload {
  sub: string;
  username: string;
}

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy, "jwt") {
  constructor() {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: process.env.JWT_SECRET,
    });
  }

  validate(payload: JwtPayload): { userId: string; username: string } {
    return { userId: payload.sub, username: payload.username };
  }
}
```

生产环境把密钥放到配置中心或环境变量里，`dev-only` 仅作占位。如果 `validate` 里要做黑名单或吊销校验，就注入对应服务即可，并确保策略类也注册在模块 `providers` 中。

守卫层继承 `AuthGuard('jwt')`，这样控制器里就可以直接写 `@UseGuards(JwtAuthGuard)`。如果大多数路由都需要保护，可以把守卫用 `APP_GUARD` 做全局注册，再用元数据把少数公开路由放行。

```typescript
import { ExecutionContext, Injectable } from "@nestjs/common";
import { AuthGuard } from "@nestjs/passport";
import { Reflector } from "@nestjs/core";
import { IS_PUBLIC_KEY } from "./public.metadata";

@Injectable()
export class JwtAuthGuard extends AuthGuard("jwt") {
  constructor(private readonly reflector: Reflector) {
    super();
  }

  canActivate(context: ExecutionContext) {
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (isPublic) {
      return true;
    }
    return super.canActivate(context);
  }
}
```

这里通过 `IS_PUBLIC_KEY` 与 `@Public()` 做白名单：标记为公开的路由直接返回 `true`，其余由父类继续完成 `JWT` 校验。这是常见的“全局 `JWT` + 白名单”做法。

模块里需要注册 `JwtModule`、`PassportModule`、`JwtStrategy`，并把 `JwtAuthGuard` 放进 `providers`。确保 `JwtModule.register` 或 `registerAsync` 的 `secret` 与策略里使用的密钥一致，否则就会出现签名和验证不匹配的常见问题。

签发令牌通常在登录服务里完成，跟守卫校验要用同一份 `secret` 与算法选项。过期时间、`issuer`、`audience` 若在签发侧配置，策略里的 `PassportStrategy` 参数也要对齐，否则合法令牌会被拒绝。

本地联调时可以先用固定载荷的 `JWT` 验证整体链路。写集成测试时更推荐模拟 `AuthGuard`，或在测试模块里替换 `JwtStrategy` 的 `validate`，避免每个用例都依赖真实签名流程。

## Guard 和 Middleware 的区别

两者都出现在控制器之前，但观察现场的方式与能做的事不同。下面用表格做粗粒度对照，细节仍以项目为准。

| 维度     | 中间件                               | 守卫                                         |
| -------- | ------------------------------------ | -------------------------------------------- |
| 运行位置 | 适配器上，贴近原始 `req` 或 `res`    | 框架管线内，`Nest` 已选好路由与处理器        |
| 上下文   | 通常只有 `req`、`res`、`next`        | `ExecutionContext`，可切 `HTTP`、`RPC`、`WS` |
| 元数据   | 读不到方法装饰器与控制器类上的元数据 | 可用 `Reflector` 读自定义装饰器              |
| 典型用途 | 日志、跨域、粗粒度 `Token` 存在性    | 认证、授权、角色、与路由绑定的策略           |
| 依赖注入 | `NestMiddleware` 类可注入            | `Guard` 类可注入                             |
| 返回值   | 调用 `next()` 推进                   | `canActivate` 布尔或可观察对象               |

实务上可以这么分工。中间件做与具体处理器无关的薄层，例如打访问日志、拒绝明显非法的 `Authorization` 格式。守卫做与路由、角色、策略相关的判断，需要读 `@Roles()`、`@Permissions()` 这类信息时只能用守卫或更靠后的横切层。

若把复杂权限写进中间件，往往要维护一份路径到规则的映射表，和路由声明两套真相。守卫直接绑定在控制器与方法上，改路由时装饰器一起动，心智负担更小。

同一请求里可以既有中间件又有守卫，二者不互斥。常见组合是中间件解析或挂载轻量上下文，守卫基于上下文加路由元数据做决策。

## 小结

`Guard` 实现 `CanActivate`，在管道与控制器之前决定请求是否放行，适合认证与授权类策略。
`canActivate` 可同步、异步或返回 `Observable`，返回 `false` 或抛出 `401`、`403` 类异常都会中断后续管线。
登录校验与权限校验宜分层，并在使用 `JWT` 时保证策略、`AuthGuard`、模块注册与公开路由白名单在命名和密钥上保持一致。

把本篇与中间件、管道、拦截器、`ExecutionContext`、`AOP` 几篇连起来读，可以拼出完整的请求处理图景，并在新增接口时快速判断逻辑应落在哪一层。
