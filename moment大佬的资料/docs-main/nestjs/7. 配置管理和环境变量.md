上一节说的是"请求体不能直接信"。这一节是同一类思路换到另一条入口上：

系统跑起来以后要用的那些值，也不该在业务文件里随手写死、随手读环境变量。

很多人习惯把数据库地址、端口、第三方密钥、功能开关直接写进代码。短期省事，一进真实环境就会撞上：

- 本地、测试、预发、生产参数各不相同
- 密钥不能进 Git 历史
- 同一份镜像要在不同环境里换参
- 配置一多就分不清谁必填、谁有默认
- 到处 `process.env.XXX`，排查时无从找起

配置管理的目标就不只是"能读到环境变量"，而是：

- 来源清楚，最好只有一个读出口
- 结构可拆，按领域归类而不是一锅粥
- 环境可切换，代码不动、参数动
- 关键项在进程刚起来时就验完，别等业务跑一半才爆

可以把环境变量看成运行时的外部输入。请求进门前有 `DTO` 和管道，配置进应用前也应该有统一入口和可选的启动校验。

本地开发时，`.env` 一类文件常长这样（仅示意，真实密钥勿提交）：

```text
PORT=3000
DATABASE_URL=postgres://localhost:5432/nest_app
JWT_SECRET=dev-only-secret
REDIS_HOST=127.0.0.1
```

这些键不应在业务类里散落读取。更稳的做法是交给 `ConfigModule` 集中加载，再由 `ConfigService` 往外提供。

## 仓库里该出现什么

在动 `ConfigModule` 之前，先把仓库习惯立住：

- `.env`、`.env.local`、`.env.production` 等含真实密钥的文件，写进 `.gitignore`，不要提交
- 另起一个 `.env.example`（或 `.env.template`），只列键名和假值、简短注释，方便同事和 CI 知道要配哪些变量

这样新人克隆仓库后，复制一份 `.env.example` 改名成 `.env` 再填本地值，不会误把空仓库当成"不用配也能跑"。

## `ConfigModule` 与 `ConfigService`

`NestJS` 官方推荐用 `@nestjs/config` 里的 `ConfigModule` 加载变量、`ConfigService` 读取变量。

安装：

```shell
pnpm add @nestjs/config
```

在根模块里注册，最常见是全局一份，避免每个特性模块再 `import` 一遍：

```typescript
import { Module } from "@nestjs/common";
import { ConfigModule } from "@nestjs/config";

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      // 需要时可在 .env 里写 ${OTHER_VAR} 做简单展开
      expandVariables: true,
    }),
  ],
})
export class AppModule {}
```

`isGlobal: true` 表示把配置模块挂成全局，任意 `Provider` 里注入 `ConfigService` 即可，不必再导入 `ConfigModule`。

业务里读取时，尽量走 `ConfigService`，少碰裸的 `process.env`，这样以后换加载方式（例如只走容器注入）时改动面小：

```typescript
import { Injectable } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";

@Injectable()
export class AppService {
  constructor(private readonly configService: ConfigService) {}

  getAppInfo(): { port: number; nodeEnv: string } {
    const port = this.configService.get<number>("PORT", 3000);
    const nodeEnv = this.configService.get<string>("NODE_ENV", "development");

    return { port, nodeEnv };
  }
}
```

`get` 的第二个参数是默认值，键不存在时退回该值，省得在业务里写一长串 `??`。

对数据库连接串、主密钥这类缺了就不该启动的项，更适合用 `getOrThrow`，让应用在启动阶段直接失败，而不是带着 `undefined` 跑进业务：

```typescript
const databaseUrl = this.configService.getOrThrow<string>("DATABASE_URL");
```

可选配置再给默认，必填配置用 `getOrThrow`，分工会清晰很多。

`main.ts` 里监听端口时，同样可以接配置，避免魔法数字：

```typescript
import { ValidationPipe } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";

async function bootstrap(): Promise<void> {
  const app = await NestFactory.create(AppModule);
  const configService = app.get(ConfigService);
  const port = configService.get<number>("PORT", 3000);

  // 若上一篇已接入 ValidationPipe，可保留；否则可删
  app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }));
  await app.listen(port);
}

void bootstrap();
```

单元测试里若不想读真实 `.env`，可以在测试模块里 `ConfigModule.forRoot({ ignoreEnvFile: true, load: [...] })` 或覆写 `process.env`，避免测试依赖本机文件。

## 用常量收起环境变量键名

除了不要用裸的 `process.env` 读值，键名字符串本身也不建议在业务里到处手写。拼错一个字母，运行时只能等到 `undefined` 才暴露。

更稳的做法是按领域建一份「键名表」，值仍然是环境里的真实 key，代码里只引用常量。例如邮件相关：

```typescript
export const EmailConfigEnum = {
  EMAIL_HOST: "EMAIL_HOST",
  EMAIL_PORT: "EMAIL_PORT",
  EMAIL_AUTH_USER: "EMAIL_AUTH_USER",
  EMAIL_AUTH_PASS: "EMAIL_AUTH_PASS",
} as const;
```

注入 `ConfigService` 后统一这样取：

```typescript
const host = this.configService.get<string>(EmailConfigEnum.EMAIL_HOST);
const port = this.configService.get<number>(EmailConfigEnum.EMAIL_PORT, 465);
```

好处很实在：

- 改名只改键名表一处，配合全局搜索即可
- 编辑器能补全，减少手滑
- 和 `Joi.object({ [EmailConfigEnum.EMAIL_HOST]: Joi.string().required() })` 一类校验也能共用同一套名字，避免 schema 与业务各写一串

可以按模块拆多份枚举或 `const` 对象，例如 `RedisConfigEnum`、`JwtConfigEnum`，和后面的 `registerAs` 命名空间并不冲突，前者管「环境变量原名」，后者管「载入后的结构化树」。

## 启动时校验环境变量

默认值救不了"必填缺失"。`ConfigModule` 支持在应用起来前用 `Joi` 等库做一次整表校验，不通过则进程直接退出，错误信息集中在启动日志里。

安装：

```shell
pnpm add joi
```

示例：强制要求数据库地址，端口有默认，运行环境只允许若干取值：

```typescript
import { Module } from "@nestjs/common";
import { ConfigModule } from "@nestjs/config";
import * as Joi from "joi";

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      validationSchema: Joi.object({
        NODE_ENV: Joi.string()
          .valid("development", "test", "production")
          .default("development"),
        PORT: Joi.number().port().default(3000),
        DATABASE_URL: Joi.string().uri().required(),
      }),
      validationOptions: {
        allowUnknown: true,
        abortEarly: false,
      },
    }),
  ],
})
export class AppModule {}
```

若前面已经为邮件、Redis 等建了键名常量对象，校验层尽量用同一套名字（或从常量拼出 `Joi` 的键），避免 schema 里写一串字面量、`get` 里又写另一串。

`abortEarly: false` 时一次能多报几条错误，联调更省事。`allowUnknown: true` 表示允许多余环境变量存在（常见于平台自动注入的键），若你希望多一个键就报错，可以改成 `false` 并配合白名单式 schema，对接口边界特别严的系统会用这种风格。

## 多环境与 `envFilePath`

真实项目里至少会区分本地、测试、预发、生产。代码共用一套，换的是配置。

`ConfigModule` 可以按文件名顺序加载，靠后的覆盖靠前的：

```typescript
import { Module } from "@nestjs/common";
import { ConfigModule } from "@nestjs/config";

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      envFilePath: [
        `.env.${process.env.NODE_ENV ?? "development"}`,
        ".env",
      ],
    }),
  ],
})
export class AppModule {}
```

含义大致是：先读环境专用文件（例如 `.env.development`），再读通用 `.env`，后者填默认值、前者填覆盖值。

### 只准备开发与生产两份文件时

有的仓库只维护 `.env.development` 和 `.env.production`，不想为 `NODE_ENV` 的每一种取值各放一个文件。可以在 `AppModule` 外先把环境收成两档，再拼 `envFilePath`：

```typescript
import { Module } from "@nestjs/common";
import { ConfigModule } from "@nestjs/config";

const NODE_ENV =
  process.env.NODE_ENV === "development" ? "development" : "production";

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      expandVariables: true,
      envFilePath: `.env.${NODE_ENV}`,
    }),
  ],
})
export class AppModule {}
```

这样除 `development` 以外的取值（含未设置时被工具设成的 `production`、或 CI 里的 `test`）都会落到 `.env.production` 上，文件数量最少，心智也简单。

要特别注意：`Jest` 等工具常把 `NODE_ENV` 设成 `test`。若仍用上面二选一写法，单测进程会去读 `.env.production`，往往不是你想要的。可以任选其一做补丁：

- 测试模块里 `ConfigModule.forRoot({ ignoreEnvFile: true })` 并手动 `process.env` 或 `load` 假配置
- 把归一化改成三档：`development`、显式的 `test`、其余走 `production`
- 或继续用数组形式，为 `test` 单独提供 `.env.test` 并放在靠前位置

若既想少文件、又要公共默认值，可以把路径改成数组，例如 `[".env", `.env.${NODE_ENV}`]`，先加载无后缀的基底，再让环境专用文件覆盖。

常见文件名习惯：

```text
.env
.env.development
.env.test
.env.production
```

注意：`process.env.NODE_ENV` 在进程启动时就已确定，本地跑 `nest start` 前若未设置，多数工具会默认成 `development`。生产环境应由部署脚本或容器显式设置，不要依赖偶然默认值。

再次强调：生产密钥应由编排平台、密钥管理或 CI 注入进环境变量，而不是把填好真值的 `.env.production` 推进 Git。文件可以存在运维私有机密存储里，不进仓库。

## 按领域拆分与 `registerAs`

键少时，扁平的 `PORT`、`DB_HOST` 还能记。键一多，就需要按领域拆文件，和模块化一样按边界收纳。

除了 `export default () => ({ app: {...} })` 这种返回对象的工厂，官方更推荐用 `registerAs` 带上命名空间，避免不同文件里的键不小心顶撞：

```typescript
import { registerAs } from "@nestjs/config";

/** 应用级运行参数 */
export default registerAs("app", () => ({
  port: Number(process.env.PORT ?? 3000),
  nodeEnv: process.env.NODE_ENV ?? "development",
}));
```

```typescript
import { registerAs } from "@nestjs/config";

/** 数据库连接相关 */
export default registerAs("database", () => ({
  host: process.env.DB_HOST ?? "127.0.0.1",
  port: Number(process.env.DB_PORT ?? 5432),
  username: process.env.DB_USERNAME ?? "postgres",
  password: process.env.DB_PASSWORD ?? "",
  name: process.env.DB_NAME ?? "nest_app",
}));
```

在 `AppModule` 里统一 `load`：

```typescript
import { Module } from "@nestjs/common";
import { ConfigModule } from "@nestjs/config";
import appConfig from "./config/app.config";
import databaseConfig from "./config/database.config";

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      load: [appConfig, databaseConfig],
    }),
  ],
})
export class AppModule {}
```

读取时用命名空间加点路径，例如：

```typescript
const port = this.configService.get<number>("app.port");
const dbHost = this.configService.get<string>("database.host");
```

好处和模块化类似：一眼知道某个值属于应用、数据库还是缓存，后续加校验、写测试、做迁移都对着固定命名空间下手。

若某个特性模块只想暴露自己的配置，也可以用 `ConfigModule.forFeature` 做局部注册，避免根模块无限膨胀，具体写法可查当前版本的官方文档。

## 异步配置与自定义工厂

少数场景下，配置不是立刻可读的文件或环境变量，而要异步拉取（例如启动时向密钥服务要一把短期凭证）。这时可以用返回 `Promise` 的配置工厂，或在 `forRootAsync` 里用 `useFactory` 注入别的 `Provider` 再拼出配置对象。

这类写法略长，但原则不变：业务代码仍然只依赖 `ConfigService` 或你封装的配置类，不把"怎么拿到密钥"的细节散在控制器里。

## 配置中心与演进路径

实例一多、服务一拆，单靠每台机器上的 `.env` 往往不够。这时所谓"配置中心"不必指某个具体产品，而是一种意识：

配置继续外移，由平台集中管理、审计和下发。

常见形态包括：

- 容器或 `PaaS` 注入的环境变量
- `Kubernetes` 的 `ConfigMap` 与 `Secret`
- 云厂商参数存储、专用配置服务
- 密钥系统如 `Vault`

对 `NestJS` 应用来说，关键是读配置只走统一抽象。今天用 `.env` 加 `ConfigModule`，明天换容器注入，后天接集中配置，只要注入点不变，业务层改动就小。

不是每个项目都要一步走到最右。更常见的是小团队先把演进图里的 C、D、E 做扎实，规模上来再接 F。

## 小结

环境变量和请求参数一样，都是运行时从外部进来的输入，值得用同一套认真劲去对待。读值尽量走 `ConfigService`，键名用分领域的常量对象收一层，仓库里只留 `.env.example`、敏感文件不进版本控制，多环境则靠文件顺序、`NODE_ENV` 与脚本的配合或平台注入来切换，并记得单测里 `NODE_ENV=test` 和二档归一化叠在一起时可能误读生产文件。结构上要按领域拆配置，必要时用 `registerAs` 做命名空间，必填项在启动阶段用校验或 `getOrThrow` 卡住，等业务跑起来再发现缺配置就晚了。项目变大以后，再把本地文件逐步换成集中配置与密钥托管，只要业务始终只依赖统一的配置入口，迁移就不会推倒重来。
