事务的价值不在于把 `SQL` 写得更花，而在于把一组必须同时成立的读写动作收进一个可控的整体。

当多个请求同时访问同一批数据时，最常见的失败形态有两种，一种是写了一半就崩了，另一种是读到的内容在我处理期间发生了变化。事务把这些风险收口成明确的结果，要么全部成功，要么全部回滚。除此之外，事务还通过 `isolation level` 去约束并发读写时的可见性边界，`commit` 之后发生的写入也会被持久化保留下来。

可以把事务理解成四件事：

- 原子性决定了失败时如何回滚
- 隔离性决定了并发时读到的是什么状态
- 一致性往往要靠业务不变量配合数据库约束一起守住
- 持久性决定了提交之后不会凭空丢失

## 事务的基本使用

事务通常发生在边界上。这里说的边界不是指你写了多少次 `update`，而是指这段逻辑在业务语义上必须同步成立。

一个典型边界是资金转账。从余额校验开始，扣减转出方余额，增加接收方余额，再记录一条账务流水，整个过程中无论哪一步失败，数据都应该回到业务看起来仍然成立的状态，也就是全部撤销。

实践中事务更像一把短时手术刀。事务内部只做数据库相关的读取、计算与写入，尽量不要在事务里做外部 `HTTP` 请求、文件读写或长时间等待，这样可以缩短持锁时间，降低锁等待与死锁的概率。如果某些校验依赖事务外数据，先在事务之外把校验条件准备好，回到事务里只做最终一致的读取与写入。

在实现上，建议遵循几条顺序化原则：

- 先在事务内部完成所有需要相互一致的读取与计算
- 不要在事务外部读取事务内要依赖的脏数据后再回到事务中写回
- 把能引发失败的条件写进同一个异常路径里，让回滚自然发生

下面用 `TypeORM` 的 `DataSource.transaction` 展示一种常见写法。业务逻辑被包在回调里，由框架自动处理提交或回滚，回调一旦抛错，内部的所有写入都会被撤销。

```typescript
import { DataSource, EntityManager } from "typeorm";

async function transfer(
  dataSource: DataSource,
  fromId: string,
  toId: string,
  amount: number,
): Promise<void> {
  await dataSource.transaction(async (manager: EntityManager) => {
    const from = await manager.findOneByOrFail(Wallet, { id: fromId });
    if (from.balance < amount) {
      throw new Error("insufficient funds");
    }

    const to = await manager.findOneByOrFail(Wallet, { id: toId });

    from.balance -= amount;
    to.balance += amount;

    await manager.save(Wallet, from);
    await manager.save(Wallet, to);

    await manager.save(LedgerEntry, {
      fromId,
      toId,
      amount,
      type: "transfer",
    });
  });
}
```

真实项目里光靠这种写法还不够。余额校验在并发转账下会出现先读后写的时间窗，后面讲锁与隔离级别时会回到这个例子。

## 事务中的失败与重试

事务并不是为了保证永不失败，而是为了把失败收敛成可预期的错误边界。并发冲突下常见的失败来源有几类：

- 死锁，数据库检测到循环等待时会主动回滚其中一个事务
- 锁等待超时，比如 `MySQL` 的 `innodb_lock_wait_timeout`
- 可串行化隔离级别下的校验失败，`PostgreSQL` 会返回 `serialization_failure`

这几类错误通常都是可恢复的，重试一次往往就能通过。落地时建议把重试拆成两层来思考。

第一层是识别可重试错误。实现里需要根据底层驱动返回的错误类型或错误码判断，比如 `MySQL` 的 `ER_LOCK_DEADLOCK`、`PostgreSQL` 的 `40P01` 与 `40001`，都属于典型的可重试错误。不要对所有异常都盲目重试，否则会把不该继续的错误也反复放大。

第二层是重试时的幂等保护。重试会重复执行写操作，业务层必须能识别这是同一次请求，不然会出现重复副作用。常见做法是在用例入口生成唯一的请求标识，写入时把它作为幂等键或唯一约束的一部分去重。

下面给一个通用的重试外壳。`isRetryableDbError` 只是占位，真实项目需要接上对应数据库的错误判断逻辑，`baseDelayMs` 配合指数退避可以避免重试风暴。

```typescript
import { DataSource, EntityManager } from "typeorm";

function sleep(ms: number): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

function isRetryableDbError(_err: unknown): boolean {
  return false;
}

interface RetryOptions {
  retries: number;
  baseDelayMs: number;
}

async function withTransactionRetry<T>(
  dataSource: DataSource,
  work: (manager: EntityManager) => Promise<T>,
  options: RetryOptions,
): Promise<T> {
  const { retries, baseDelayMs } = options;

  for (let attempt = 0; attempt <= retries; attempt += 1) {
    try {
      return await dataSource.transaction(async (manager) => work(manager));
    } catch (err) {
      if (attempt >= retries || !isRetryableDbError(err)) throw err;
      const delay = baseDelayMs * Math.pow(2, attempt);
      await sleep(delay);
    }
  }

  throw new Error("unreachable");
}
```

这个外壳只解决框架层面的重试，业务层的幂等依旧需要自己保障。

## 并发更新问题

并发时事务到底能不能救场，取决于你遭遇的是哪一种读写交错问题。常见问题可以按现象归成四类：

- 脏读，读取到了另一个事务尚未提交的中间状态
- 不可重复读，同一个事务里重复读取同一行，两次读到的结果不同
- 幻读，按条件读取到的集合在两次读取之间发生了新增或删除
- 丢失更新，两个事务都基于旧值计算，后提交的写覆盖了先提交的改动

每一类现象对应的常见处理方向不同，可以先用一张表快速对照。

| 并发现象   | 它在说什么               | 常见处理方向                            |
| ---------- | ------------------------ | --------------------------------------- |
| 脏读       | 能看到未提交结果         | 提高 `isolation level` 到读已提交或更高 |
| 不可重复读 | 同一行多次读取结果不一致 | 可重复读，或在关键行加锁                |
| 幻读       | 条件查询的集合会变       | 串行化隔离，或使用范围锁、间隙锁        |
| 丢失更新   | 后写覆盖前写             | 乐观锁、悲观锁，或用原子更新语句        |

这里要强调一点，提高隔离级别并不会自动让业务正确，它只是减少了某些交错读写的可能性。业务仍然要定义清楚不变量，比如库存不能为负、订单状态只能从待支付流转到已支付、订单总金额必须等于明细求和，这些规则数据库并不知道。

还有一个容易被忽略的细节，不同数据库对同一隔离级别的实现并不一致。`MySQL InnoDB` 在可重复读级别下会通过间隙锁阻止范围内的插入，对大部分幻读场景已经够用。`PostgreSQL` 的可重复读是基于快照的，遇到写写冲突会直接报错让事务重试。写代码时最好按具体数据库的行为来选，而不是只看标准里的定义。

## 如何选 isolation level

选得太低会让并发异常更容易出现，选得太高会增加冲突检测与重试成本。可以按下面这个顺序判断：

- 先把读已提交或同等强度当作基线，它解决了脏读，性能损耗也可控
- 需要在同一事务内对同一批记录多次读取并保持视图一致时，升到可重复读
- 对并发顺序有强要求，且愿意承担冲突重试成本时，再考虑可串行化

不管隔离级别怎么选，都要在事务内部把读取与计算做完整，把写入条件写得更明确，避免事务里出现先读后判断再写、但判断依赖外部变量的情况，这种写法在任何隔离级别下都可能被并发绕过。

## 乐观锁 悲观锁基础

当你遇到并发写同一条数据这种冲突时，锁的选择往往决定了系统的吞吐与失败重试成本。

乐观锁的思路是允许并发读取，提交写回时再检查是否发生过冲突。典型做法是加一个版本号列或时间戳列，更新时带上旧版本作为条件，只要版本已变化就拒绝写入并交由业务重试。它适合冲突少、读多写少的场景。

悲观锁的思路是先假设冲突必然发生，在读取关键行时就把相关行锁住，直到事务结束才释放。它能减少失败重试，但代价是吞吐下降，容易引入锁等待甚至死锁风险。它适合冲突密集、单次操作计算依赖强一致中间结果的场景。

下面这段代码演示乐观锁基于版本号的 `CAS` 更新。它把 `id` 与 `expectedVersion` 一起作为 `where` 条件，只有版本匹配时才会命中行并完成更新。

```typescript
import { EntityManager } from "typeorm";

type UpdateResult = { ok: true } | { ok: false; reason: "conflict" };

async function updateWithOptimisticLock(
  manager: EntityManager,
  id: string,
  expectedVersion: number,
  newBalance: number,
): Promise<UpdateResult> {
  const result = await manager.update(
    Wallet,
    {
      id,
      version: expectedVersion,
    },
    {
      balance: newBalance,
      version: expectedVersion + 1,
    },
  );

  if (!result.affected || result.affected === 0) {
    return { ok: false, reason: "conflict" };
  }

  return { ok: true };
}
```

`TypeORM` 的实体上如果声明了 `@VersionColumn`，保存时框架会自动生成带版本判断的 `update`，不需要每次手写 `where`。

悲观锁常见实现是使用 `SELECT ... FOR UPDATE` 或 `ORM` 对应的行锁 `API`。读取时相关行就被锁住，其他并发事务只能等待，这样可以在事务内基于当前值安全地做计算。

```typescript
import { EntityManager } from "typeorm";

async function updateWithPessimisticLock(
  manager: EntityManager,
  id: string,
  delta: number,
): Promise<void> {
  const wallet = await manager
    .createQueryBuilder(Wallet, "w")
    .setLock("pessimistic_write")
    .where("w.id = :id", { id })
    .getOneOrFail();

  const next = wallet.balance + delta;
  if (next < 0) {
    throw new Error("balance cannot be negative");
  }

  wallet.balance = next;
  await manager.save(Wallet, wallet);
}
```

这段代码必须放在事务里执行，否则 `setLock` 不会生效。使用悲观锁时特别要注意锁顺序和锁粒度，否则很容易触发死锁：

- 以确定性的顺序获取锁，比如按主键升序逐个锁住，避免不同事务用相反的顺序加锁造成循环等待
- 给锁定条件加上能走索引的约束，避免意外锁住大量行甚至整表
- 控制事务内的工作量，减少持锁时间

如何选锁可以用一条经验判断，冲突较少且允许重试时优先考虑乐观锁，冲突较多且单次操作依赖强一致中间结果时考虑悲观锁。无论选哪种，都要配合重试策略来用，重试不是简单重复执行，而是要带上退避、限制次数，必要时把失败转为可恢复的业务状态让前端去引导用户。

## 不要把所有一致性问题都交给数据库

数据库能保证的是局部数据的一致性边界，它解决不了跨系统的整体一致性，也无法自动修复业务层面的逻辑漏洞。更可靠的做法是把一致性拆成几类能力分别落地：

- 用约束与不变量把状态合法性固定下来，比如唯一约束、外键约束、`CHECK` 约束，以及应用侧的状态机规则
- 把写操作设计成幂等，失败重试时不会产生重复副作用，常见做法是用幂等键或基于业务主键的去重
- 跨服务的流程里把补偿与重试写进方案，而不是只依赖数据库事务
- 对外部资源使用一致性协议，比如消息队列配合事务外发，避免本地提交了但通知没发出去的断裂

这四条里最常见、也最容易落地的是事务外发，也就是 `outbox` 模式。它的工作方式可以用一张图说明。

![20260421112537](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260421112537.png)

业务状态与待发布事件写在同一个事务里，提交成功后才会被后台任务读取并投递。每个事件都带有唯一的 `eventId`，消费者按 `eventId` 去重，把至少一次投递变成对业务可接受的一致性。

下面是 `outbox` 投递循环的一个最小实现，它只描述核心流程，真实项目里还需要加失败重试、投递节流和监控。

```typescript
interface OutboxEvent {
  id: string;
  type: string;
  payload: Record<string, unknown>;
  status: "pending" | "sent";
}

interface OutboxRepository {
  insertPending(event: OutboxEvent): Promise<void>;
  takePendingBatch(limit: number): Promise<OutboxEvent[]>;
  markSent(id: string): Promise<void>;
}

async function publishOutbox(
  repo: OutboxRepository,
  sendToQueue: (event: OutboxEvent) => Promise<void>,
): Promise<void> {
  const events = await repo.takePendingBatch(50);
  for (const event of events) {
    await sendToQueue(event);
    await repo.markSent(event.id);
  }
}
```

如果流程跨多个服务、需要多步编排，单纯 `outbox` 还不够，这时可以把事务的思路延伸成 `saga`。每一步都提供明确的正向动作与补偿动作，整体在失败与重试下仍能收敛到一致的业务结果。`outbox` 负责单次写操作与事件发布的原子性，`saga` 负责跨服务流程的最终一致性，这两者通常是配合使用而不是二选一。

可以用一句话收尾。数据库负责让一段本地写要么都发生要么都不发生，应用负责让业务语义在重试、并发、故障下仍然保持成立。

## 小结

把事务用在必须同步成立的边界上，让隔离级别与锁策略覆盖并发读写的风险。让重试具备识别能力并依赖幂等，不要把重试做成无脑循环。跨系统的一致性靠 `outbox` 与 `saga` 把失败路径也纳入方案，而不是只把希望寄托在数据库身上。
