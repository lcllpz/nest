前面几篇里已经反复出现 `Controller`、`Service`、`Provider`、`Module`。你会看到一种固定写法：类里很少自己 `new` 别的类，多半只在构造函数里声明"我需要谁"，剩下交给框架。

这就是依赖注入，一句白话版是：依赖不由自己造，由外部在创建你时塞进来。

对比传统写法：

```typescript
class UsersController {
  private readonly usersService = new UsersService();
}
```

控制器既知道要谁，又知道怎么造，实现被绑死了。换成 `NestJS` 常见形态：

```typescript
import { Controller, Get } from "@nestjs/common";
import { UsersService } from "./users.service";

@Controller("users")
export class UsersController {
  constructor(private readonly usersService: UsersService) {}

  @Get()
  findAll(): string[] {
    return this.usersService.findAll();
  }
}
```

`UsersController` 只声明依赖 `UsersService`。实例何时创建、是否单例、是否还有下游依赖，都由容器按图索骥，不用手写装配顺序。

分工可以记成两句：类负责说清依赖，框架负责创建和注入。结构会松很多。

## `IoC` 容器是什么

`IoC` 是 `Inversion of Control` 的缩写，常译成"控制反转"。不用先背定义，先看变化：以前依赖在类内部 `new` 出来，现在创建和装配收拢到框架里的统一容器。

在 `NestJS` 里这就是 `IoC` 容器。它登记有哪些 `Provider`、怎么实例化、依赖图怎么走、何时把实例塞进构造函数，并管默认情况下的生命周期与复用。

`UsersController` 要 `UsersService`，`UsersService` 又要 `ConfigService` 时，你不需要手写"先 `Config` 再 `Users` 再 `Controller`"。容器会沿依赖链自动解析。

关系可以记成：依赖注入是组织方式，`IoC` 容器是跑在背后的基础设施。缺了容器，大规模自动装配很难落地，缺了注入思想，容器也没有方向。

## 为什么值得用依赖注入

若只当成"少写几个 `new`"，就小看它了。它主要换的是可维护性。

耦合会降下来。自己 `new` 时，既依赖对方接口，又依赖对方具体类和构造方式，换实现往往要改调用方。注入后，边界在构造函数类型上，替换注册即可。

测试会好做。数据库、缓存、第三方 HTTP 在单元测试里常要换成 mock 或桩，通过模块里改 `Provider` 或测试工具里的 `overrideProvider`，比在每个测试里改业务代码干净得多。

扩展时改动面更小。本地实现换远程、旧组件换新组件，只要 token 与模块边界清楚，动注册表往往比动全目录的 `new` 安全。

类的职责也更纯，不必兼管"依赖从哪来"。

例如通知服务一开始只是占位实现：

```typescript
import { Injectable } from "@nestjs/common";

@Injectable()
export class NotificationService {
  send(message: string): string {
    return `已发送（示意）：${message}`;
  }
}
```

后面换成邮件、短信或多实现并行，只要在模块里调整 `providers` 或工厂，调用方构造函数签名可以保持稳定。

## 构造函数注入与 `@Inject`

`NestJS` 里最推荐的是构造函数注入：依赖全在参数列表上，一眼可读，对象构造完成时依赖已齐。

```typescript
import { Injectable } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";

@Injectable()
export class UsersService {
  constructor(private readonly configService: ConfigService) {}

  getUploadDir(): string {
    return this.configService.get<string>("UPLOAD_DIR", "/tmp");
  }
}
```

它不容易出现"用到一半才发现依赖没赋值"的问题，也方便静态分析和重构。

框架还支持属性注入，但不作为默认选项，可读性和不可变习惯都不如构造函数注入。

有些依赖在部分环境下不存在，可以配合 `@Optional()`，容器找不到时注入 `undefined`，业务里要自己判空：

```typescript
import { Injectable, Optional } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";

@Injectable()
export class UsersService {
  constructor(
    @Optional() private readonly configService?: ConfigService,
  ) {}

  getUploadDir(): string {
    return this.configService?.get<string>("UPLOAD_DIR", "/tmp") ?? "/tmp";
  }
}
```

当注入目标不是类类型，而是自定义 token（字符串、符号或抽象接口在运行时的 token），要在参数前加 `@Inject(token)`。

字符串 token 简单但容易和别的库撞名，稍大一点的项目更常用 `Symbol` 或集中导出的常量：

```typescript
export const LOGGER_TOKEN = Symbol("LOGGER");
```

```typescript
import { Inject, Injectable } from "@nestjs/common";
import { LOGGER_TOKEN } from "./logger.token";

export interface AppLogger {
  log(message: string): void;
}

@Injectable()
export class ReportService {
  constructor(@Inject(LOGGER_TOKEN) private readonly logger: AppLogger) {}

  generate(): void {
    this.logger.log("生成报表");
  }
}
```

模块里用 `useClass` 或 `useValue` 把 `LOGGER_TOKEN` 绑到具体实现即可。

## `Provider` 与模块边界

容器只会解析它已经知道的 `Provider`。`@Injectable()` 只表示"这类可被容器管理"，真正进图要靠模块的 `providers`。

```typescript
import { Module } from "@nestjs/common";
import { UsersController } from "./users.controller";
import { UsersService } from "./users.service";

@Module({
  controllers: [UsersController],
  providers: [UsersService],
})
export class UsersModule {}
```

别的模块要用 `UsersService`，需要在本模块 `exports`，对方再 `imports` 你的模块：

```typescript
import { Module } from "@nestjs/common";
import { UsersService } from "./users.service";

@Module({
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}
```

```typescript
import { Module } from "@nestjs/common";
import { UsersModule } from "../users/users.module";
import { AuthService } from "./auth.service";

@Module({
  imports: [UsersModule],
  providers: [AuthService],
})
export class AuthModule {}
```

所以注入不是"全局随便 `get`"，而是跟模块的 `providers`、`exports`、`imports` 绑在一起。

若某个模块的配置、日志等几乎人人都要用，可以谨慎使用 `@Global()` 把模块标成全局，减少重复 `imports`。全局模块一多，依赖从哪来会变模糊，只适合真正横切且稳定的能力。

## 作用域：默认单例与请求级实例

默认情况下，同一 `Provider` 在单个 `Nest` 应用进程里通常是单例，多次注入拿到的是同一实例，适合无状态服务。

若依赖里要放请求级状态（例如每个 HTTP 请求一份上下文），可以把作用域改成按请求实例化：

```typescript
import { Injectable, Scope } from "@nestjs/common";

@Injectable({ scope: Scope.REQUEST })
export class RequestScopedService {}
```

`Scope.TRANSIENT` 则每次注入都新建实例。`REQUEST` 与 `TRANSIENT` 比默认单例创建更频繁，要确认确有 per-request 或 per-inject 状态需求，不要默认把服务都改成 `REQUEST`。

## `useClass`、`useValue`、`useFactory`、`useExisting`

类直接写在 `providers` 数组里是最简形式。复杂一点时，用对象形式描述"token 怎么兑现"。

### `useClass`

token 指向一个具体类，由容器负责 `new`。适合接口与实现分离、或按环境换实现。

```typescript
import { Injectable, Module } from "@nestjs/common";
import { LOGGER_TOKEN } from "./logger.token";

@Injectable()
class ConsoleLogger {
  log(message: string): void {
    console.log(message);
  }
}

@Module({
  providers: [
    {
      provide: LOGGER_TOKEN,
      useClass: ConsoleLogger,
    },
  ],
})
export class AppModule {}
```

### `useValue`

直接注入已有对象：常量、第三方客户端实例、测试里的 mock。

```typescript
import { Module } from "@nestjs/common";

const appConfig = {
  appName: "nest-demo",
  timeout: 5000,
};

@Module({
  providers: [
    {
      provide: "APP_CONFIG",
      useValue: appConfig,
    },
  ],
})
export class AppModule {}
```

### `useFactory`

根据配置或其它 `Provider` 动态拼出实例，`inject` 数组与工厂参数顺序一致。

```typescript
import { Module } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";

@Module({
  providers: [
    {
      provide: "UPLOAD_OPTIONS",
      inject: [ConfigService],
      useFactory: (configService: ConfigService) => ({
        dir: configService.get<string>("UPLOAD_DIR", "/tmp"),
        maxSize: configService.get<number>("UPLOAD_MAX_SIZE", 5 * 1024 * 1024),
      }),
    },
  ],
})
export class AppModule {}
```

### `useExisting`

两个 token 共享同一实例，不额外 `new` 一份。

```typescript
import { Injectable, Module } from "@nestjs/common";
import { LOGGER_TOKEN } from "./logger.token";

@Injectable()
export class LoggerService {
  log(message: string): void {
    console.log(message);
  }
}

@Module({
  providers: [
    LoggerService,
    {
      provide: LOGGER_TOKEN,
      useExisting: LoggerService,
    },
  ],
})
export class AppModule {}
```

| 方式          | 适合场景              | 核心特点                |
| ------------- | --------------------- | ----------------------- |
| `useClass`    | token 映射到类        | 容器创建实例            |
| `useValue`    | 常量、现成对象、mock  | 不经过构造              |
| `useFactory`  | 依赖配置或其它依赖组装 | `inject` 与参数顺序对应 |
| `useExisting` | 别名                  | 与已有 `Provider` 同实例 |

## 循环依赖与 `forwardRef`

两个类或两个模块互相依赖时，静态解析会碰到环。`Nest` 提供 `forwardRef(() => Target)` 延迟解析引用，用在模块 `imports` 或构造函数 `@Inject(forwardRef(() => Service))` 上。

环能拆开最好还是拆边界（抽出第三块、事件、接口），`forwardRef` 是补丁而不是架构目标，环多了图会难懂。

## 测试与运行时解析

单测里常用 `Test.createTestingModule` 编译子图，并用 `overrideProvider` 把真实实现换成 mock，而不改业务类源码：

```typescript
import { Test } from "@nestjs/testing";
import { UsersService } from "./users.service";
import { EmailService } from "./email.service";

const mockEmail = { send: jest.fn() };

const moduleRef = await Test.createTestingModule({
  providers: [UsersService, EmailService],
})
  .overrideProvider(EmailService)
  .useValue(mockEmail)
  .compile();
```

少数场景要在请求处理之外按名字取实例，可以注入 `ModuleRef` 做程序化解析，日常业务仍优先构造函数注入，避免到处 `get` 把图搅乱。

## 小结

依赖注入把"谁创建谁"从业务类里挪出去，交给 `IoC` 容器按模块注册的 `Provider` 装配。构造函数注入是默认首选，`@Inject` 与 token 解决非类依赖，`@Optional` 处理可有可无的项。作用域决定单例、请求级还是每次新建，四种自定义注册覆盖类、常量、工厂与别名。模块的 `exports` 与 `imports` 划定可见范围，循环依赖用 `forwardRef` 应急、长期仍应削环。测试通过覆盖 `Provider` 换实现，既验证业务又不动生产代码结构。对象声明需要什么，容器决定从哪来、怎样连成片，这正是 `NestJS` 能把控制器、守卫、管道、拦截器、过滤器等挂进同一张网的底层原因。
