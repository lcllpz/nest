装饰器和元数据把结构写进源码里。请求一旦进入执行链，`Guard`、`Interceptor`、异常过滤器这类横切能力要在不钻进业务方法内部的前提下读请求、认路由、写响应，靠的是同一套运行时入口：`ExecutionContext` 和 `ArgumentsHost`。

可以把 `ExecutionContext` 想成一句白话：当前这段横切逻辑跑在什么现场、对着哪个类哪个方法、底层宿主是 HTTP 还是别的。

一次 HTTP 请求进控制器之前，`Guard` 往往要知道请求对象在哪、命中的是谁、`Authorization` 怎么取。这些都不是业务返回值，而是执行现场。`ExecutionContext` 把现场收成统一 API，横切代码先接上下文，再 `switchToHttp()` 一类方法落到具体协议对象上。

```typescript
import { CanActivate, ExecutionContext, Injectable } from "@nestjs/common";

@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest<{ headers: { authorization?: string } }>();
    const token = request.headers.authorization;

    return Boolean(token);
  }
}
```

重点不是鉴权规则写得多漂亮，而是横切层不直接假设自己手里已经是 `request`，而是先拿上下文，再切换协议。记住一句就够：`ExecutionContext` 是横切逻辑观察当前现场的统一入口。

## 多协议抽象：HTTP、`RPC`、`WebSocket`

若所有横切 API 都写死成 `request`、`response`，`Guard` 就很难复用到微服务消息或 `WebSocket`。`Nest` 选择先做协议无关的上下文，再在各适配器里填满底层对象。

- `switchToHttp()` 后通常是 `getRequest()`、`getResponse()`（具体类型随 `Express` 或 `Fastify` 驱动而变）
- `switchToRpc()` 后是 `getData()`、`getContext()` 等
- `switchToWs()` 后是 `getClient()`、`getData()`

`context.getType()` 返回当前链路类型字符串，常见为 `"http"`、`"rpc"`、`"ws"`。先分支再切换，避免在 `WS` 链路里误调 `getRequest()`。

```typescript
import { CanActivate, ExecutionContext, Injectable } from "@nestjs/common";

@Injectable()
export class ContextAwareGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const type = context.getType();

    if (type === "http") {
      const request = context.switchToHttp().getRequest();
      return Boolean(request.headers?.authorization);
    }

    if (type === "rpc") {
      const data = context.switchToRpc().getData<unknown>();
      return data !== undefined && data !== null;
    }

    if (type === "ws") {
      const client = context.switchToWs().getClient<unknown>();
      return Boolean(client);
    }

    return false;
  }
}
```

抽象优先，上层横切一套，底层多种传输往里适配，这是 `Nest` 不像"纯 HTTP 小框架"的重要原因之一。

## `ArgumentsHost` 与 `ExecutionContext` 的分工

`ArgumentsHost` 更底层：回答这一组调用参数从哪来、怎么取。它提供 `switchToHttp()` 等切换方法，也提供 `getArgs()`、`getArgByIndex()`，用来拿到传给当前处理函数（例如控制器方法）的原始参数数组。多数业务横切用不到这么细，但在极少数要和管道入参顺序打交道时有用。

`ExecutionContext` 在 `ArgumentsHost` 之上多了执行目标：`getClass()` 当前类、`getHandler()` 当前方法、`getType()` 当前协议类型。因此：

- 过滤器往往只关心怎么把错误写回客户端，`catch(exception, host: ArgumentsHost)` 就够
- `Guard`、`Interceptor` 常要认方法、读元数据，`ExecutionContext` 更合适

异常过滤器示例（与第 8 篇风格一致，响应体可再扩展）：

```typescript
import {
  ArgumentsHost,
  Catch,
  ExceptionFilter,
  HttpException,
  HttpStatus,
} from "@nestjs/common";
import type { Request, Response } from "express";

@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost): void {
    const ctx = host.switchToHttp();
    const request = ctx.getRequest<Request>();
    const response = ctx.getResponse<Response>();

    const status =
      exception instanceof HttpException
        ? exception.getStatus()
        : HttpStatus.INTERNAL_SERVER_ERROR;

    response.status(status).json({
      path: request.url,
      message: "服务异常",
    });
  }
}
```

快速区分可以记：只要参数宿主用 `ArgumentsHost`，还要类、方法、协议类型用 `ExecutionContext`。

在类型关系上，日常用到的 `ExecutionContext` 实例同时带有 `ArgumentsHost` 的能力，不必死记继承树，只要知道谁多提供了 `getClass` 与 `getHandler`。

## `GraphQL` 与特殊上下文

使用 `@nestjs/graphql` 时，横切里拿到的仍是 `ExecutionContext`，但 `switchToHttp()` 往往不是主战场。解析器运行在另一套执行图上，常用做法是 `GqlExecutionContext.create(context)` 取出 `GraphQL` 专属的 `req`、`args` 等。HTTP 守卫不能原样复制粘贴到 `GraphQL` 解析器守卫里，需要按文档换入口。本篇以 HTTP 为主，记住"协议不同，切换方式不同"即可。

## `Guard` 里结合 `Reflector` 与元数据

上一篇的 `SetMetadata` 要在运行时读回来，依赖 `Reflector` 和 `ExecutionContext` 上的 `getHandler()`、`getClass()`。方法级与类级元数据合并时，常用 `getAllAndOverride`。

```typescript
import { CanActivate, ExecutionContext, Injectable } from "@nestjs/common";
import { Reflector } from "@nestjs/core";
import { IS_PUBLIC_KEY } from "./public.decorator";

@Injectable()
export class AuthGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);

    if (isPublic) {
      return true;
    }

    const request = context.switchToHttp().getRequest<{ headers: { authorization?: string } }>();
    return Boolean(request.headers.authorization);
  }
}
```

这样 `Guard` 同时用到了现场（请求头）和结构声明（是否 `@Public()`），两条线在上下文上交汇。

## `Interceptor` 与执行现场

`Interceptor` 包裹 `next.handle()`，既可能读请求，也要知道当前是哪个控制器的哪个方法，便于日志、耗时、幂等等。`getClass()`、`getHandler()` 就是为这种需求准备的。

```typescript
import {
  CallHandler,
  ExecutionContext,
  Injectable,
  Logger,
  NestInterceptor,
} from "@nestjs/common";
import { Observable, tap } from "rxjs";

@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  private readonly logger = new Logger(LoggingInterceptor.name);

  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const controller = context.getClass().name;
    const handler = context.getHandler().name;
    const started = Date.now();

    return next.handle().pipe(
      tap(() => {
        this.logger.log(`${controller}.${handler} 耗时 ${Date.now() - started}ms`);
      }),
    );
  }
}
```

`Guard` 偏放行与否，`Interceptor` 偏环绕增强，二者都依赖上下文认路。

## 管道为什么不谈 `ExecutionContext`

`Pipe` 的 `transform(value, metadata)` 接收的是当前参数值和 `ArgumentMetadata`，而不是完整的 `ExecutionContext`。这是刻意收窄：管道专注校验与转换单参，不认全请求现场。要在管道里读 `req`，通常需要别的方式（例如自定义装饰器先把 `req` 注入参数再交给管道），默认模型不要和 `Guard` 混成同一套 API。

## 小结

`ArgumentsHost` 提供协议切换与原始参数入口，`ExecutionContext` 在此基础上给出当前类、方法与 `getType`，让横切逻辑在 HTTP、`RPC`、`WebSocket` 上共用一套形状。过滤器多数只要 `ArgumentsHost` 写回响应，`Guard` 与 `Interceptor` 用 `ExecutionContext` 认路由、读元数据、包一层 `RxJS`。`GraphQL` 要换专用辅助从同一 `ExecutionContext` 里拆字段。管道走另一套窄接口。装饰器与元数据写清结构，容器组装对象，执行上下文告诉横切层你现在站在哪条链路的哪个点上，三层合起来才是 `Nest` 常见的读代码方式。
