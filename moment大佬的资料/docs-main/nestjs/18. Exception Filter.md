业务代码、守卫、管道、拦截器里随时可能抛出异常。若不做集中处理，默认行为会依赖底层适配器与异常类型，导致客户端拿到的字段结构不稳定。异常过滤器把“异常 -> `HTTP` 响应”的映射从业务里抽出来，在出口处统一状态码、响应体形状、日志与监控埋点。

`ExceptionFilter` 接口只有一个 `catch(exception, host)` 方法。`exception` 是抛出的值，类型上往往是 `unknown`，需要自己做类型收窄。`host` 是 `ArgumentsHost`，用来切到 `HTTP`、`RPC` 或 `WebSocket` 宿主；在 `HTTP` 场景里通常 `switchToHttp()` 取 `Response` 与 `Request`，然后决定状态码与响应体。

只有没有在业务分支里被 `try/catch` 吃掉的异常，冒泡到框架层后才会触发异常过滤器。若服务层吞掉异常并返回默认值，过滤器不会介入。很多“以为全局过滤器能兜住，实际没兜住”的问题，都来自这里。

过滤器与拦截器里的 `catchError` 不同。拦截器多在返回路径上做 `Observable` 算子，过滤器面向请求生命周期末尾的异常出口。二者可以并存，但要避免重复写响应，避免 `headers already sent`。

你可以用下面的清单自检：什么时候会走到异常过滤器，什么时候不会。

- 控制器或服务把异常 catch 并返回默认值：通常不会触发异常过滤器
- 拦截器把异常转换成正常的 Observable：异常过滤器通常不会再看到这次异常
- 确实抛出了异常并一路冒泡到框架层：最终会命中异常过滤器

## 捕获异常并统一处理

`@Catch()` 不带参数时表示捕获任意异常，带参数时可以收窄到若干类型，例如只处理 `HttpException` 子类。多个过滤器同时存在时，更具体的匹配优先，具体规则以当前版本文档为准。

`HttpException` 及其子类携带 `getStatus()` 与 `getResponse()`，`getResponse()` 可能是字符串，也可能是带 `message`、`error` 字段的对象。非 `HTTP` 异常需要自己映射成状态码，常见做法是未知错误统一 `500`，日志里记录完整堆栈，响应里只给泛化文案。

下面是一个只处理 `HttpException` 的骨架，响应体形状可按团队规范替换。类型收窄用 `instanceof`，避免把任意对象当 `HttpException` 用。

```typescript
import {
  ArgumentsHost,
  Catch,
  ExceptionFilter,
  HttpException,
} from "@nestjs/common";
import type { Response } from "express";

@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost): void {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const status = exception.getStatus();
    const body = exception.getResponse();

    response.status(status).json({
      statusCode: status,
      error: typeof body === "string" ? body : body,
      path: ctx.getRequest<{ url?: string }>().url ?? "",
    });
  }
}
```

这个骨架里有两个关键点。第一，用 `@Catch(HttpException)` 直接收窄异常类型，避免把任何对象都当成 `HttpException` 处理。第二，响应体尽量只拼“稳定字段”，不要把 `getResponse()` 的原始结构原样返回给前端。

若还要兜底所有未分类错误，可以另写一个 `@Catch()` 不带参数的过滤器，或在一个过滤器里用 `instanceof` 分支。注意两个全局过滤器都挂在同一条链上时，谁先命中谁处理，避免重复发送响应。

## 全局异常过滤器设计

全局注册常见两种方式。一是在 `main.ts` 里 `app.useGlobalFilters(new SomeFilter())`。二是用 `APP_FILTER` 提供者配合依赖注入，让过滤器能注入配置服务、日志服务等。

过滤器解析顺序与守卫、管道不同。一般是从路由到控制器再到全局，也就是最靠近具体路由的过滤器先执行。如果局部过滤器已经写出响应，外层全局过滤器不应再重复处理；如果局部没有处理，异常会继续向外层传递，直到全局兜住。

全局过滤器适合放统一日志、统一监控和统一 `JSON` 结构。局部过滤器适合某个控制器返回专用错误码，或在特定协议下做不同的异常映射。

如果你只想要一个“可注入”的全局过滤器，也可以先从 `APP_FILTER` 这条线上入手；等你熟悉顺序和覆盖关系，再决定哪些逻辑下沉到局部过滤器。

## 业务异常与系统异常如何区分

业务异常指在预期之内的失败路径，例如库存不足、重复提交、权限不足。这类错误通常有稳定业务码，前端要根据码做提示或跳转。实现上可以用 `HttpException` 子类，或自定义类继承 `HttpException`，在 `getResponse()` 里带上 `code` 字段，与 `HTTP` 状态码区分。

系统异常指未预料的缺陷、下游超时、数据库断连、空指针。这类错误对客户端一般只暴露 `500` 或统一服务不可用码，详细信息进日志与告警系统。不要把数据库异常原文直接返回给前端，避免泄露表名或连接信息。

工程里可以约定一套基类，例如 `AppHttpException`，构造时固定业务码、用户可见文案、可选 `HTTP` 状态。服务层只抛这类异常，全局过滤器负责把 `AppHttpException` 映射成稳定 `JSON`，把普通 `Error` 映射成 `500` 且隐藏细节。

最小示意可以长这样：

```typescript
import { HttpException, HttpStatus } from "@nestjs/common";

export class AppHttpException extends HttpException {
  constructor(code: string, message: string, status: HttpStatus = HttpStatus.BAD_REQUEST) {
    super({ code, message }, status);
  }
}
```

区分时不必追求物理上的两个类，关键是事先约定哪些字段可给前端、哪些只能进日志。业务异常走显式字段，系统异常在过滤器里统一脱敏。

## 面向前端的稳定错误格式

前端更希望拿到可预测键名、可枚举错误码，而不是随 `HttpException` 构造参数变化的嵌套结构。常见做法是在全局过滤器里定义唯一外壳，例如 `success`、`code`、`message`、`payload`、`requestId`，其中 `payload` 只在需要时附加校验错误列表。`requestId` 通常来自请求头或请求级上下文，用于前后端联调与定位日志。

`ValidationPipe` 抛出的校验错误往往是数组，过滤器里需要单独分支，把 `message` 整理成数组或扁平字符串，并把“字段名与约束”放进 `payload`，前端表单才能对齐到具体输入项。`HTTP` 状态码可以保留在 `HTTP` 头上，也可以冗余进 `body` 里，团队要统一约定。

下面是一个合并了 `HttpException` 与未知分支的示意，生产环境应对 `isProduction` 隐藏堆栈与内部信息。

```typescript
import {
  ArgumentsHost,
  Catch,
  ExceptionFilter,
  HttpException,
  HttpStatus,
} from "@nestjs/common";
import type { Response } from "express";

@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly isProduction = process.env.NODE_ENV === "production";

  catch(exception: unknown, host: ArgumentsHost): void {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<{ url?: string; headers: { 'x-request-id'?: string } }>();

    let status = HttpStatus.INTERNAL_SERVER_ERROR;
    let code = "INTERNAL_ERROR";
    let message = "服务暂时不可用";
    let details: unknown;

    if (exception instanceof HttpException) {
      status = exception.getStatus();
      const body = exception.getResponse();
      message =
        typeof body === "string"
          ? body
          : ((body as { message?: string | string[] }).message?.toString() ??
            message);
      code = `HTTP_${status}`;
    } else if (exception instanceof Error) {
      if (!this.isProduction) {
        details = exception.stack;
      }
    }

    response.status(status).json({
      success: false,
      code,
      message,
      path: request.url ?? "",
      requestId: request.headers["x-request-id"] ?? "",
      timestamp: new Date().toISOString(),
      ...(this.isProduction ? {} : { details }),
    });
  }
}
```

国际化场景下，`message` 可以只返回错误码，由前端根据 `code` 查表；也可以在过滤器里读取请求头（例如 `Accept-Language`）选择文案。不要把敏感操作是否成功的细节写在可被枚举的码上，避免被恶意扫描。

## 小结

异常过滤器把“异常 -> `HTTP` 响应”的映射集中处理，只对未在业务里捕获、一路冒泡到框架层的异常生效；而拦截器里的 `catchError` 更偏向返回路径上的 Observable 处理。配合 `@Catch` 收窄异常类型，你可以用 `useGlobalFilters` 或 `APP_FILTER` 注册全局过滤器，并利用“路由近到远”的覆盖顺序实现局部定制。

在内容上，把业务异常与系统异常在对外字段上区分：系统异常对外脱敏、对内保留完整日志；同时让前端拿到稳定的错误外壳与统一业务码，并根据环境控制是否暴露堆栈等细节。

把本篇与中间件、管道、守卫、拦截器、`ExecutionContext` 几篇连起来读，可以完整覆盖请求从进入到错误返回的闭环。
