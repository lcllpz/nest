上一章接入大模型时提到，`invoke` 可以传一条字符串或一组消息，多轮对话和工具调用都依赖这套统一结构。

裸调各家 API 时消息格式不一：有的用 `role: "user"`，有的用 `type: "human"`，工具结果也塞在不同字段里。LangChain 用一套抽象把"谁说的、说了什么、有没有附带工具调用或结果"收敛成几种标准消息类型，换模型或厂商时业务层只和这套结构打交道，底层由各厂商集成自动转换。链与 `Agent` 的"历史"本质就是消息数组，统一结构之后，拼接、截断、注入系统提示都会简单很多。

## 消息的基本构成

一条消息包含三个部分：

- 角色：谁发的（系统、用户、模型、工具）
- 内容：正文或多模态负载（纯文本或按块组成的内容数组）
- 元信息：可选字段，如 `token` 用量、工具调用的 `id`、模型名称等

不同角色对应不同的消息类，内容可以是纯文本，也可以是按块组成的多模态或结构化负载。先记住这一点，再看四种类型会清晰不少。

## 四种标准消息类型

下图从上到下是四种消息类型与对应角色，图下方是一次带工具调用的对话里消息的典型顺序。

![20260309102428](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260309102428.png)

### SystemMessage

系统消息用来定调模型的行为和身份，通常只发一次、放在对话最前面，例如"你是一个把英文翻译成中文的助手"或"回答要简短、专业"。

多数模型会把 `system` 和 `user`、`assistant` 分开处理，有的还会给系统消息更长上下文或更高权重。规则、风格和约束适合写在系统提示里，不必塞进一条很长的用户消息。

```typescript
import { SystemMessage } from "@langchain/core/messages";

const sys = new SystemMessage('你是一个只回答"是"或"否"的助手。');
```

传给 `invoke` 时，LangChain 会把它转成当前模型 API 需要的格式（例如 OpenAI 的 `role: "system"`），业务层不用关心底层差异。

### HumanMessage

用户消息对应人类的一轮输入。单轮时可以直接 `invoke("一句话")`，LangChain 会自动包成一条 `HumanMessage`。多轮时显式 new 一个 `HumanMessage`，或传 `{ role: "user", content: "..." }` 即可。

```typescript
import { HumanMessage } from "@langchain/core/messages";

const userMsg = new HumanMessage("北京今天天气怎么样？");
```

若要支持多模态（文字加图片），把 `content` 改成内容块数组，做法见后文"内容块与多模态"一节。

### AIMessage

模型每轮回复对应一条 `AIMessage`，`invoke` 的返回值就是它。流式时先拿到的是 `AIMessageChunk`，拼成完整消息后结构和 `AIMessage` 一致。

常用字段有三个：

- `content`，正文字符串，普通对话里就是模型说的话
- `tool_calls`，模型决定调工具时会出现一至多个调用请求，每项包含工具名、参数和 `tool_call_id`
- `usage_metadata`，`token` 用量（`input_tokens`、`output_tokens`），方便算成本

调试时把这三个打出来，就能看出模型是在正常回话还是在请求调工具：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const aiMsg = await model.invoke([new HumanMessage("用一句话介绍你自己。")]);

console.log(aiMsg.content);
console.log(aiMsg.tool_calls);
console.log(aiMsg.usage_metadata);
```

没有工具调用时 `tool_calls` 是空数组。有的话，对每个 `tool_call` 执行工具，把结果写成 `ToolMessage` 追回消息列表，再 `invoke` 一次。

### ToolMessage

模型通过 `tool_calls` 请求调工具之后，你得把执行结果以工具消息的形式写回对话，模型才能根据结果生成下一句。

每条 `ToolMessage` 都要带上对应的 `tool_call_id`，和 `AIMessage.tool_calls[].id` 一一对应，模型靠这个把结果和某次调用对上。内容一般是字符串（例如"北京晴，25°C"），部分厂商也支持结构化内容。

用 LangChain 的 `tool()` 定义并执行工具时，返回值已经是一条带好 `tool_call_id` 的 `ToolMessage`，直接 `push` 进消息数组即可：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { tool } from "@langchain/core/tools";
import { HumanMessage } from "@langchain/core/messages";
import type { BaseMessage } from "@langchain/core/messages";
import { z } from "zod";

const getWeather = tool(
  (input: { location: string }): string => `${input.location} 天气晴朗，25°C。`,
  {
    name: "get_weather",
    description: "查询某地的当前天气状况。",
    schema: z.object({
      location: z.string().describe("要查询天气的城市名称"),
    }),
  },
);

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const modelWithTools = model.bindTools([getWeather]);
const messages: BaseMessage[] = [new HumanMessage("波士顿天气怎么样？")];

const aiMsg = await modelWithTools.invoke(messages);
messages.push(aiMsg);

for (const toolCall of aiMsg.tool_calls ?? []) {
  const toolResult = await getWeather.invoke(toolCall);
  messages.push(toolResult);
}

const finalResponse = await modelWithTools.invoke(messages);
console.log(finalResponse.content);
```

如果不用 LangChain 执行工具、而是从别处拿到结果再写回对话，就要手动 new 一条 `ToolMessage`，自己填好 `tool_call_id` 和 `content`：

```typescript
import { ToolMessage } from "@langchain/core/messages";

const toolResult = new ToolMessage({
  tool_call_id: "call_abc123",
  content: "北京：晴，25°C",
});
```

## 工具参数的结构化

消息解决"谁说的、说了什么"；工具调用还涉及另一类结构化：模型要调用的函数需要哪些参数、各是什么类型。

LangChain 里定义工具时，通常用 `zod` 描述这份参数 schema。`zod` 是 TypeScript 中常用的 schema 库，既可做运行时校验，又能被 LangChain 转成模型 API 所需的 `JSON Schema`；模型返回的 `tool_calls[].args` 会按该 schema 填写，执行前也可用同一套 schema 做校验。

```typescript
import { tool } from "@langchain/core/tools";
import { z } from "zod";

const getWeather = tool(
  (input: { location: string; unit?: "c" | "f" }): string => {
    const temp = input.unit === "f" ? "77°F" : "25°C";
    return `${input.location}：晴，${temp}`;
  },
  {
    name: "get_weather",
    description: "查询某地的天气。",
    schema: z.object({
      location: z.string().describe("要查询天气的地点"),
      unit: z
        .enum(["c", "f"])
        .optional()
        .describe("温度单位，c 为摄氏度，f 为华氏度"),
    }),
  },
);
```

`schema` 用 `z.object` 声明参数形状，`.describe()` 会变成 `JSON Schema` 的字段说明，模型根据这些描述决定是否调用工具以及如何填写参数。

消息结构化与工具参数结构化是一体的：消息承载对话与工具结果，`zod` schema 承载工具入参约定，整条链路从用户输入到工具调用再到 `ToolMessage` 都类型清晰、可校验。

## 简单对象与类实例

LangChain 既接受 `HumanMessage`、`AIMessage` 等类实例，也接受 `role + content` 的简单对象，内部会统一成标准消息再发给模型。两种写法等价，对应关系如下：

| 简单对象 `role` | 对应类          | 说明                                                         |
| --------------- | --------------- | ------------------------------------------------------------ |
| `"system"`      | `SystemMessage` | 内容为字符串或 `content` 数组                                |
| `"user"`        | `HumanMessage`  | 内容为字符串或 `content` 数组                                |
| `"assistant"`   | `AIMessage`     | 内容为字符串或 `content` 数组，可附带 `tool_calls`           |
| `"tool"`        | `ToolMessage`   | 还需带 `tool_call_id`，否则模型无法将结果与 `tool_call` 关联 |

使用工具执行结果做 `messages.push(result)` 时，LangChain 已填好 `tool_call_id` 等字段，不需要手动处理。

业务代码中推荐用类实例，可读性和 TypeScript 类型提示更好；从数据库或接口反序列化历史消息时，用简单对象更方便。

## 内容块与多模态

多数情况下 `content` 是一个字符串。在多模态或结构化场景中，一条消息的内容可由多个块组成，每块有类型和负载。常见的 `type` 包括：

| `type`        | 说明                                            |
| ------------- | ----------------------------------------------- |
| `"text"`      | 普通文本块                                      |
| `"image_url"` | 图片链接，模型读取后解析图像内容                |
| `"image"`     | `base64` 编码的图片数据                         |
| `"reasoning"` | 模型的推理过程（部分模型如 DeepSeek-R1 会输出） |

带图片的用户消息：

```typescript
import { HumanMessage } from "@langchain/core/messages";

const userMsg = new HumanMessage({
  content: [
    { type: "text", text: "这张图里有什么？" },
    {
      type: "image_url",
      image_url: { url: "https://example.com/image.jpg" },
    },
  ],
});
```

模型若支持多模态会按块解析并生成回复。对于支持推理输出的模型，返回的 `AIMessage` 中 `content` 也可能是块数组，其中 `type: "reasoning"` 的块是模型的思考过程，`type: "text"` 的块是最终答案，可以分别提取使用。

## 对话顺序与历史

模型看到的上下文就是 `messages` 数组的先后顺序。常见的排列方式有两种：

普通对话：

```
SystemMessage → HumanMessage → AIMessage → HumanMessage → AIMessage → ...
```

带工具调用的对话：

```
SystemMessage → HumanMessage → AIMessage（含 tool_calls）→ ToolMessage → AIMessage（最终回复）
```

一旦 `AIMessage` 里有 `tool_calls`，必须先追加对应的 `ToolMessage`，再发下一轮请求，模型才能根据工具结果生成最终回复。`tool_call_id` 的对应关系也不能弄错，否则模型无法识别哪条结果对应哪次工具调用。

下图展示了带工具调用时，消息数组的完整追加顺序：

![20260215010313](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260215010313.png)

在链或 `Agent` 里通常维护一个 `messages` 数组：用户说一句就 `push` 一条 `HumanMessage`，模型回一句就 `push` 一条 `AIMessage`，工具调用则按上述方式追加 `ToolMessage`。消息过多时再按需做长度截断或摘要，避免超出模型上下文窗口。

## 小结

消息是 LangChain 中对话与工具调用的基本单位，四种类型各司其职：

- `SystemMessage`：定角色与规则，放在对话最前面
- `HumanMessage`：用户的每一轮输入
- `AIMessage`：模型的回复，可附带 `tool_calls` 请求执行工具
- `ToolMessage`：工具执行结果，必须带 `tool_call_id` 与 `tool_call` 对应

写法上可用类实例，也可用 `role + content` 的简单对象；内容可为纯文本，也可为多块组成的多模态或推理块。工具参数用 `zod` 定义 schema，由 LangChain 转成 API 所需的 `JSON Schema`，执行时还可做运行时校验。

掌握这套结构后，换模型、维护对话历史、接入工具都会更顺畅，后续链与 `Agent` 章节也会持续用到这些类型。
