前面几章把 Function Call 的约定、消息结构和单工具绑定都跑通了。真正做 Agent 时，往往需要给模型配备多个工具：查天气、搜网页、查数据库、发邮件等。模型根据用户意图和工具描述决定调用哪一个、传什么参数；应用负责执行并把结果塞回对话。

本章在上一章单工具绑定的基础上，接入两个真实的外部 API（和风天气 + Tavily 网页搜索），讲清楚多工具的定义、绑定、循环执行和错误处理，最后给出一份可直接运行的完整示例。

## 工具在 Agent 里扮演什么角色

可以把 Agent 想成"会推理的人"。模型负责理解问题、决定下一步、组织答案。工具负责模型做不到的事，例如查实时数据、改数据库、调外部服务。

没有工具时，模型只能靠训练数据里的知识回答，无法知道"今天北京的天气"或"Next.js 昨天发布了什么"。

配备工具之后，模型在需要时会输出 `tool_calls`，应用执行对应逻辑并把结果以 `ToolMessage` 的形式回传，模型再根据结果继续推理或给出最终回复。

给 Agent 配备工具，本质上就是三件事：

- 定义好一组工具（名称、描述、参数、执行逻辑）
- 用 `bindTools` 绑到聊天模型上
- 在对话循环里处理 `tool_calls` 与 `ToolMessage`

## 接入工具

本章以两个工具为例，把定义、绑定和对话循环串起来。一个是和风天气（QWeather）API，可查指定城市未来三天的天气预报。另一个是 Tavily，面向 AI Agent 的网页搜索 API，返回结构化搜索结果。

### 天气查询（和风）

和风天气的查询分两步，先用城市名查出 `locationId`，再用 `locationId` 查天气预报。把这两步封装进一个 `tool()`，对模型只需传城市名即可：

```typescript
import { tool } from "@langchain/core/tools";
import { z } from "zod";

const QWEATHER_HOST = "https://ph5u9wjwrf.re.qweatherapi.com";
const QWEATHER_KEY = "8a6ce52a897c438dac4bc2c80aeb7a38";

interface QWeatherLocation {
  id: string;
  name: string;
  adm1: string;
  adm2: string;
}

interface QWeatherDaily {
  fxDate: string;
  textDay: string;
  textNight: string;
  tempMax: string;
  tempMin: string;
}

async function fetchLocationId(cityName: string): Promise<QWeatherLocation> {
  const url = `${QWEATHER_HOST}/geo/v2/city/lookup?location=${encodeURIComponent(cityName)}&range=cn&number=1&key=${QWEATHER_KEY}`;
  const res = await fetch(url);
  if (!res.ok) throw new Error(`城市查询 HTTP 错误：${res.status}`);
  const data = await res.json();
  if (data.code !== "200") throw new Error(`城市查询失败，错误码：${data.code}`);
  if (!data.location?.length) throw new Error(`未找到城市：${cityName}`);
  return data.location[0] as QWeatherLocation;
}

const getWeather = tool(
  async (input: { location: string }): Promise<string> => {
    const location = await fetchLocationId(input.location);
    const url = `${QWEATHER_HOST}/v7/weather/3d?location=${location.id}&key=${QWEATHER_KEY}`;
    const res = await fetch(url);
    if (!res.ok) throw new Error(`天气查询 HTTP 错误：${res.status}`);
    const data = await res.json();
    if (data.code !== "200") throw new Error(`天气查询失败，错误码：${data.code}`);
    const daily = data.daily as QWeatherDaily[];
    const forecast = daily
      .map(
        (d) =>
          `${d.fxDate} 白天:${d.textDay} 夜间:${d.textNight} ${d.tempMin}~${d.tempMax}℃`,
      )
      .join("\n");
    return `${location.name}（${location.adm1} ${location.adm2}）未来三天：\n${forecast}`;
  },
  {
    name: "get_weather",
    description:
      "根据城市名称查询未来三天天气预报，返回每天的天气状况和气温范围。适合回答天气相关的问题。",
    schema: z.object({
      location: z.string().describe("城市名称，如北京、上海、深圳"),
    }),
  },
);
```

### 网页搜索（Tavily）

Tavily 面向 AI Agent 的搜索 API，返回结构化摘要而不是原始网页，适合直接放进 `ToolMessage`：

```typescript
import { TavilySearch } from "@langchain/tavily";

const TAVILY_KEY = "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";

const internetSearch = tool(
  async (input: {
    query: string;
    maxResults?: number;
    topic?: "general" | "news" | "finance";
  }): Promise<string> => {
    const tavilySearch = new TavilySearch({
      maxResults: input.maxResults ?? 5,
      tavilyApiKey: TAVILY_KEY,
      topic: input.topic ?? "general",
    });
    return await tavilySearch._call({ query: input.query });
  },
  {
    name: "internet_search",
    description:
      "在互联网上搜索最新信息，适合查询实时资讯、新闻、技术动态、人物介绍等。当问题涉及近期发生的事或模型知识截止日期之后的内容时，优先使用此工具。",
    schema: z.object({
      query: z.string().describe("要搜索的问题或主题，尽量具体"),
      maxResults: z
        .number()
        .optional()
        .default(5)
        .describe("返回结果条数，默认 5"),
      topic: z
        .enum(["general", "news", "finance"])
        .optional()
        .default("general")
        .describe("搜索主题类别：general 通用、news 新闻、finance 金融"),
    }),
  },
);
```

## 绑定多个工具

定义好工具后，把它们放进数组、一起传给 `model.bindTools()`。绑完后的模型在每轮 `invoke` 时都知道这些工具可用，会根据用户输入和工具描述自主决定调用哪个：

```typescript
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const tools = [getWeather, internetSearch];
const modelWithTools = model.bindTools(tools);
```

`bindTools` 的返回值是一个"带工具的模型"，后续调用方式和普通模型完全一样（`invoke`、`stream`、`batch`），只是当模型决定调用工具时，返回的 `AIMessage` 里会带 `tool_calls`。

## Agent 调用循环

有了带工具的模型，还需要一个循环来处理"模型返回 `tool_calls` → 执行工具 → 把结果写回对话 → 再次调用模型"的来回过程，直到模型不再需要工具、给出最终回复为止。

![20260319142150](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260319142150.png)

上图顺时针展示了 Agent 调用循环的六个环节：用户消息进入，模型判断是否需要工具，按工具名路由到执行函数，处理成功与失败两种情况，把 `ToolMessage` 写回对话，最后检查轮次限制决定继续还是退出。

核心循环逻辑如下：

```typescript
import type { BaseMessage } from "@langchain/core/messages";
import { HumanMessage, ToolMessage } from "@langchain/core/messages";

const toolsByName = Object.fromEntries(tools.map((t) => [t.name, t]));
const MAX_TOOL_ROUNDS = 5;

async function runAgent(userInput: string): Promise<string> {
  const messages: BaseMessage[] = [new HumanMessage(userInput)];
  let aiMsg = await modelWithTools.invoke(messages);
  let rounds = 0;

  while (aiMsg.tool_calls?.length && rounds < MAX_TOOL_ROUNDS) {
    rounds++;
    messages.push(aiMsg);

    for (const tc of aiMsg.tool_calls) {
      const toolFn = toolsByName[tc.name];
      try {
        if (toolFn) {
          const result = await toolFn.invoke(tc);
          messages.push(result);
        } else {
          messages.push(
            new ToolMessage({
              content: `未知工具：${tc.name}`,
              tool_call_id: tc.id,
            }),
          );
        }
      } catch (err: unknown) {
        const message = err instanceof Error ? err.message : String(err);
        messages.push(
          new ToolMessage({
            content: `工具执行失败：${message}`,
            tool_call_id: tc.id,
          }),
        );
      }
    }

    aiMsg = await modelWithTools.invoke(messages);
  }

  if (aiMsg.tool_calls?.length) {
    console.warn(`已达最大工具轮数（${MAX_TOOL_ROUNDS} 轮），强制返回当前回复。`);
  }

  return aiMsg.content as string;
}
```

循环里用 `toolsByName` 把 `tool_call.name` 映射到具体工具实例，避免每次都遍历数组查找。`MAX_TOOL_ROUNDS` 是安全阀，防止模型反复请求工具导致死循环。一旦某轮没有 `tool_calls`，说明模型已经拿到足够的信息，不再需要工具，循环自然结束。

## 工具命名与描述规范

模型完全依赖"工具名、描述、参数 schema"来决定是否调用、调哪个、参数填什么，这三样写得好，误用和漏用会少很多。

名称建议用简短、见名知意的英文或下划线形式，与代码里注册的名字一致，例如 `get_weather`、`search_database`、`send_email`。避免用容易混淆的命名（例如两个都叫 `search`，一个搜网页、一个搜库）。工具名一旦对外暴露给模型就不要随意改，否则历史对话里若出现过旧名字，模型可能请求已不存在的工具，导致"未知工具"错误。

描述要写清两件事：这个工具在什么场景下该被调用，以及它具体做了什么。"根据城市名称查询未来三天天气预报"就比"查天气"更明确；"在互联网上搜索最新信息，适合查询近期发生的事"会让模型在遇到时效性问题时主动用它，而不是用训练数据瞎猜。工具有副作用（如发邮件、写数据库）时，也建议在描述里点明，方便模型判断是否适合在当前对话里使用。

## 参数 schema 设计要点

`schema` 既约束模型输出的 `arguments`，也方便在执行前做校验。类型要和实际执行函数一致：要数字就 `z.number()`，要字符串就 `z.string()`，不然模型填了字符串而我们当数字用，容易在运行时出错。

几个常用模式：

- 必填参数：`z.string()`、`z.number()` 不加 `.optional()`
- 可选参数：`.optional()` 或 `.default(值)` — 有默认值时优先用 `.default()`，模型不填时自动补上
- 枚举约束：`z.enum(["a","b","c"])`，能显著减少模型填出非法值的概率
- 每个字段加 `.describe()`：这段文字会变成 `JSON Schema` 的 `description`，模型填参时的主要参考依据

复杂参数（嵌套对象、数组）用 `z.object`、`z.array` 逐层定义，必要时对数组元素也加 `.describe()`，模型会更容易填对。

执行前也可以用同一份 `schema` 做运行时校验：`schema.safeParse(args)`，通不过则把校验错误信息放进 `ToolMessage` 的 `content` 回传给模型，让它在回复里提示用户修正。

## 错误处理与回传

工具内部可能抛错：网络超时、参数不合法、权限不足、第三方 API 返回错误码等。若直接让错误冒泡，整个对话循环会中断。更稳妥的做法是 `try-catch`：把错误信息写成字符串放进 `ToolMessage` 的 `content` 回传，并带上对应的 `tool_call_id`。模型看到"工具执行失败：..."后，会在回复里告诉用户失败原因或建议重试，整个对话不会中断。

有三种情况需要处理：

- 工具执行成功：用 `tool.invoke(tc)` 得到 LangChain 封装好的 `ToolMessage`，直接 `push`
- 工具名未知：手动构造 `ToolMessage`，`content` 写 `"未知工具：xxx"`，带上 `tool_call_id`
- 工具执行抛错：`catch` 里构造 `ToolMessage`，`content` 写错误信息，带上 `tool_call_id`

这样一轮里即使某个工具失败，其他工具的结果仍会进消息，模型也能基于"部分成功、部分失败"给出合理回复。

## stream 场景下的注意点

用 `modelWithTools.stream(messages)` 时，模型会流式输出 `content` 和 `tool_calls`。要执行工具，必须等当前 `stream` 全部结束、拿到完整的 `AIMessage`（包含完整的 `tool_calls`）后，再按前面的循环处理。不能在半途根据"已经 stream 出的部分 `tool_calls`"就去执行，否则可能拿到不完整的参数。

实践上通常是：流式用来展示"模型正在思考……"或逐字输出文字部分；真正的工具执行仍是同步等完整回复后再做。

## toolChoice 按需控制

默认 `bindTools` 后，模型会根据输入和描述"自动"决定是否调用工具（相当于 API 层的 `tool_choice: "auto"`）。有时需要在某轮强制使用工具，可以在 `invoke` 时传入第二参数指定 `toolChoice`：

- `toolChoice: "any"`：本轮必须调用至少一个工具，具体选哪个由模型定
- `toolChoice: "get_weather"`：强制调用名为 `get_weather` 的工具（传工具名字符串）
- `toolChoice: "none"`：本轮禁止调用工具，只生成文字

例如，某轮确定用户在问天气，可以强制调用：

```typescript
const aiMsg = await modelWithTools.invoke(messages, {
  toolChoice: "get_weather",
});
```

"配备哪些工具"和"本轮是否使用工具"可以分开控制，让 Agent 的行为更可预期。

## 多工具与上下文长度

工具越多，发给模型的工具定义就越多，会占用一部分上下文窗口。若工具超过二十个，可以考虑按场景分组、只绑定当前场景需要的子集，或把描述写精炼一些，避免重复啰嗦。

工具返回的内容尽量简洁可读。若必须返回大段数据（如一大段 JSON 或日志），可在工具内部先做摘要再塞进 `ToolMessage`，既省 `token` 也方便模型抓住重点。天气工具里就做了这件事：把 API 返回的原始 JSON 整理成"日期 天气 气温"这种一行一天的格式，而不是把整个 JSON 字符串直接回传。

## 小结

给 Agent 配备工具的核心流程：

- 用 `tool()` 定义每个工具，包含名称、描述、`zod` schema 和执行函数
- 用 `model.bindTools([...])` 把多个工具绑到模型上
- 在对话循环里按 `tool_call.name` 路由到对应工具，执行后把结果以 `ToolMessage` 追回 `messages`，再继续 `invoke`，直到没有新的 `tool_calls`
- 用 `MAX_TOOL_ROUNDS` 防止死循环，用 `try-catch` 把工具错误包装成 `ToolMessage` 而不是让对话中断
- 名称和描述写清楚，参数 schema 设计好，是减少模型误用工具的最有效手段

工具是 Agent 的"手"，配备好了，Agent 才能完成查数据、调 API 等动作。下一章会在这套工具配备的基础上，把整条 Agent 循环搭成一个完整的智能体。
