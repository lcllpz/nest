`Plan-and-Execute` 即先列清单再逐条完成，先根据用户目标产出一份任务分解清单（`Plan`），再按清单逐项执行（`Execute`），每步结果作为后续步骤的上下文，必要时可回头修订计划。这种思路适合步骤较多、结构清晰的任务，与 `ReAct` 的边想边做形成互补。

`LangChain` 在 [Planning Agents](https://blog.langchain.com/planning-agents/) 一文中把这类设计归纳为规划式 Agent，多步工作流往往更快、更省、完成率也更好。更快是因为子任务可用更小或专用模型，更省是因为大模型主要用在出计划和最终汇总，更好是因为规划阶段逼模型先想清整条链路。

## 两个阶段分别做什么

`Plan-and-Execute` 把一次任务拆成只出计划和按计划做两块。规划阶段只产出步骤清单、不执行任何动作，执行阶段才按清单从第一步跑到最后一步，每步的产出会作为下一步的输入。

### 规划阶段

规划阶段只做一件事：根据用户目标产出一份可执行的任务分解清单，不跑工具、不写代码。清单通常是一组有序步骤，每步用一两句话描述要完成什么。例如用户目标是帮我设计一个用户注册接口，清单可能是三步：第一步收集需求并明确入参与出参，第二步设计 `REST` 接口与数据模型，第三步给出至少一个请求与响应的代码示例。清单的粒度由你在 `prompt` 里约束，可以粗到只列三到五步，也可以细到每步对应一次工具调用或一次子任务。

输入只有用户目标，外加可选的约束（如不超过五步、必须包含测试）。输出是结构化的步骤列表，可以要求模型用 JSON 数组或 Markdown 列表输出，再由解析器抽成数组，供执行阶段按序消费。若希望用户能在执行前参与修订，可以在规划完成后把清单展示给用户，允许增删改步骤后再进入执行阶段。

### 执行阶段

执行阶段拿着规划产出的步骤列表，从第一步开始逐条执行，每步的交付物会写回状态，作为下一步的上下文。

对每一步，会把当前步骤描述以及前面各步的执行结果一起喂给模型，必要时再挂上工具，让模型产出这一步的交付物。执行阶段可以每一步只调一次模型、只生成一段文本，也可以在单步内再跑一轮 `ReAct` 循环，例如这一步需要先查库再算数再写结论。这样 `Plan-and-Execute` 负责整体顺序和步骤边界，`ReAct` 负责单步内的多轮推理与工具调用，两者组合很常见。

## 计划与执行如何衔接

规划产出的步骤列表要能被执行阶段按序消费，两边共享同一份状态，状态里记录当前进度和每步的产出，执行器根据状态决定下一步做什么、往上下文里塞什么。

状态里至少要能回答三件事：当前执行到第几步、每一步在做什么（输入描述）、以及每一步已经产出了什么（结果）。典型实现可以用这样的结构：

- `plan`：步骤列表，例如 `["收集需求", "设计接口", "给出代码示例"]`
- `pastSteps`：已完成的步骤与结果对，例如 `[["收集需求", "需求：..."], ["设计接口", "接口设计：..."]]`，执行器在跑当前步时会把已有结果拼进 `prompt`，让模型知道前面做了什么
- `response`：最终回答，一旦有值就结束整个流程

执行循环就是：只要还有未完成的步骤，就取出当前步骤描述和已有的 `pastSteps`，调一次模型（或带工具的 Agent）完成这一步，把结果写进 `pastSteps`，再推进到下一步，直到全部步骤跑完或重规划决定直接作答。

## 流程总览

从用户给出目标到拿到最终回答，整条链路先跑规划得到步骤列表，再进入执行循环按序做完每一步并把结果写回状态，最后汇总成回复。若某步执行失败或明显偏离目标，可以触发重规划，从当前步或上一步继续。如下图所示。

## 与 ReAct 的对比与组合

`ReAct` 是每一步只决定下一步做什么，适合依赖中间结果、需要边查边想的任务，但步骤一多容易漏步或重复。`Plan-and-Execute` 则是先把所有步骤想好、再按表执行，适合步骤边界清晰、顺序相对固定的任务，整体路径一目了然，也便于做进度展示和断点续跑。

两者可以组合：用 `Plan-and-Execute` 做顶层编排，用 `ReAct` 做单步内的推理。比如写一份技术方案，先规划成需求、设计、示例、测试四步，执行时每一步内部再根据需要调用搜索、查文档等工具，用 `ReAct` 循环直到该步产出满意结果，再进入下一步。这样既避免 `ReAct` 在长链条里跑偏，又避免 `Plan-and-Execute` 在单步内缺乏灵活工具调用。

相对 `ReAct` 每轮都要调一次大模型，规划式设计通常更快、更省、完成率也更好。更快是因为执行子任务时不必每次都问大模型，更省是因为大模型主要用在出计划和最终汇总，子步骤可用小模型或专用模型，更好是因为规划阶段强迫模型先想清整条链路，本身就能提高完成率。

## 规划式 Agent 的几种形态

同属先计划再执行的思路，工程里还会遇到两种进阶形态，分别解决每步都要调 `LLM`、以及只能串行跑得慢这两类问题。

经典 `Plan-and-Execute`（即本文主讲的形态）的规划器产出一份有序步骤列表，执行器按序跑每一步，执行完后做一次重规划，决定是直接给最终回复还是再生成后续计划。缺点是每步执行时通常都要一次 LLM 调用，且没有变量引用。

`ReWOO`（Reasoning WithOut Observations）的规划器输出的不是纯步骤描述，而是穿插 Plan（推理说明）和 E#（执行行），且允许用 `#E1`、`#E2` 等形式引用前面步骤的结果。执行器按序跑任务并把结果赋给对应变量，后续步骤调用时把变量替换成真实结果，无需每步都重新问大模型下一步该干什么。代价是仍为串行执行。如下图所示。

![20260327135644](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327135644.png)

用户请求经规划器一次生成含变量引用的任务列表，执行器串行跑完后由求解器汇总成最终回答。

`LLMCompiler` 的规划器输出带依赖关系的任务 DAG（有向无环图），不互相依赖的任务可以并行跑，论文里能带来约 3.6 倍的加速。适合多工具、多 API 且可并行的场景。如下图所示。

![20260327135908](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327135908.png)

规划器流式输出任务依赖图，无依赖的任务并行执行，Joiner 决定直接回答还是补充新任务。

选型时，步骤少、顺序固定的用经典 `Plan-and-Execute` 即可，需要少调 LLM 又能引用前步结果时考虑 `ReWOO`，追求延迟和并行时再考虑 `LLMCompiler`。`LangGraph` 对这三种都有参考实现。

## 何时需要重规划

执行过程中出现以下几种情况时，可以考虑触发重规划：

- 某一步执行失败或超时，重试仍无法继续，需要调整后续步骤或把当前步拆细
- 执行结果明显偏离用户目标，由规则或人工判定后，用当前状态和失败信息重新跑一遍规划，得到修订后的步骤列表，从当前步或上一步继续
- 用户主动要求改计划，直接带着新意图重新规划，用新计划替换旧计划后继续执行

重规划不宜太频繁，否则成本和延迟都会上去。通常会对连续重规划次数或单任务总规划次数设上限，超过后走降级逻辑，例如返回当前已有结果并提示用户简化目标。

## 适合与不适合的场景

`Plan-and-Execute` 适合任务本身结构清晰、能拆成若干顺序步骤的场景，例如：

- 写一份多章节的技术方案（先需求、再设计、再示例、再测试或部署）
- 做一轮端到端的数据分析（先取数、再清洗、再统计、再可视化或总结）
- 搭建一个包含多模块的脚手架项目（先目录结构、再核心依赖、再配置、再示例代码）

优点是整体结构一开始就比较清楚，用户能在规划阶段预览和调整步骤，进度和断点也好做。缺点是，若任务在执行过程中频繁变化或依赖大量中间反馈，早期计划可能经常要重写，带来额外调用和延迟。对一步就能搞定的简单问题，先规划再执行反而多一轮成本，不如直接用 `ReAct` 或单次生成。

## 错误与边界处理

某一步执行失败时，可以把错误信息写进该步的结果，并决定是重试当前步、跳过该步继续、还是触发重规划。若步骤依赖前序结果，解析或校验失败时应优先重做上一步或重规划，而不是强行进入下一步。

对规划阶段产出的步骤数量应设上限（例如最多 10 步），避免模型输出过长或不可执行的清单。执行阶段同样应对单步耗时和总步数设限，防止卡死或成本过高。若允许用户在执行前修订计划，需要把用户确认后的计划作为唯一权威来源，执行阶段只按这份计划跑，重规划时再覆盖。

## 完整代码示例

下面用 `LangGraph` 的 `StateGraph` 把整个 `Plan-and-Execute` 流程串起来：规划节点、执行节点（内部跑 `ReAct` 循环）、重规划节点，三者通过共享状态衔接，重规划节点根据当前状态决定继续执行还是直接作答。

工具只用了两个本地工具（计算器和获取当前日期），无需任何外部 API，clone 下来直接跑。执行节点使用 `ToolNode` 配合手写的 `ReAct` 循环，不依赖任何已废弃的 prebuilt API。

```typescript
import "dotenv/config";
import { JsonOutputToolsParser } from "@langchain/core/output_parsers/openai_tools";
import {
  AIMessage,
  HumanMessage,
  type BaseMessage,
} from "@langchain/core/messages";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { tool } from "@langchain/core/tools";
import type { RunnableConfig } from "@langchain/core/runnables";
import { Annotation, END, START, StateGraph } from "@langchain/langgraph";
import { ToolNode } from "@langchain/langgraph/prebuilt";
import { ChatOpenAI } from "@langchain/openai";
import { z } from "zod";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
  temperature: 0,
});

// ----- 共享状态定义 -----
// plan 存步骤列表，pastSteps 存已完成步骤与结果，response 有值时流程结束
const PlanExecuteState = Annotation.Root({
  input: Annotation<string>({
    reducer: (x, y) => y ?? x ?? "",
  }),
  plan: Annotation<string[]>({
    reducer: (x, y) => y ?? x ?? [],
  }),
  pastSteps: Annotation<[string, string][]>({
    reducer: (x, y) => x.concat(y),
  }),
  response: Annotation<string>({
    reducer: (x, y) => y ?? x,
  }),
});

type PlanExecuteStateType = typeof PlanExecuteState.State;

// ----- 工具（本地即可跑，无需外部 API）-----
const calculator = tool(
  (input: { expression: string }) => {
    const sanitized = input.expression.replace(/[^\d+\-*/().\s]/g, "");
    if (sanitized.length === 0) return "Invalid expression.";
    try {
      const value = Function(`"use strict"; return (${sanitized})`)();
      return String(value);
    } catch {
      return "Calculation error.";
    }
  },
  {
    name: "calculator",
    description: "计算数学表达式，例如 '25 * 4 + 17' 或 '(10 + 5) / 3'。",
    schema: z.object({ expression: z.string() }),
  },
);

const getCurrentDate = tool(() => new Date().toISOString().slice(0, 10), {
  name: "get_current_date",
  description: "获取今天的日期，格式为 YYYY-MM-DD。",
  schema: z.object({}),
});

const tools = [calculator, getCurrentDate];
const toolNode = new ToolNode(tools);
const modelWithTools = model.bindTools(tools);

// ----- 执行节点：手写 ReAct 循环 -----
// 调用模型 → 有 tool_calls 就用 ToolNode 执行 → 追加结果 → 再次调用 → 直到无 tool_calls
async function runAgentLoop(
  task: string,
  config?: RunnableConfig,
): Promise<string> {
  const messages: BaseMessage[] = [new HumanMessage(task)];

  for (let i = 0; i < 10; i++) {
    const response = (await modelWithTools.invoke(
      messages,
      config,
    )) as AIMessage;
    messages.push(response);

    if (!response.tool_calls || response.tool_calls.length === 0) {
      return typeof response.content === "string"
        ? response.content
        : JSON.stringify(response.content);
    }

    const toolResults = await toolNode.invoke({ messages });
    for (const msg of toolResults.messages) {
      messages.push(msg);
    }
  }

  return "Reached max iterations without final answer.";
}

// ----- 规划节点：用 tool-calling 输出结构化步骤列表 -----
const planTool = tool(() => {}, {
  name: "plan",
  description: "输出任务分解的步骤列表。",
  schema: z.object({
    steps: z
      .array(z.string())
      .describe("按顺序排列的步骤列表，每步一句话描述要完成什么"),
  }),
});

const plannerPrompt = ChatPromptTemplate.fromTemplate(
  `针对以下目标，制定一个简单的分步计划。
每一步应是独立任务，按顺序正确执行后即可得到答案。不要添加多余步骤。
最后一步的结果即为最终答案。确保每一步都包含所需信息，不要跳过步骤。
你必须调用 plan 工具，并传入步骤列表。

目标：{objective}`,
);

const planner = plannerPrompt
  .pipe(model.bindTools([planTool]))
  .pipe(new JsonOutputToolsParser());

// ----- 重规划节点：根据已完成进度决定继续还是直接作答 -----
const responseTool = tool(() => {}, {
  name: "response",
  description: "信息足够时直接回复用户。",
  schema: z.object({
    response: z.string().describe("面向用户的最终回答"),
  }),
});

const replannerPrompt = ChatPromptTemplate.fromTemplate(
  `针对以下目标，制定一个简单的分步计划。
每一步应是独立任务，按顺序正确执行后即可得到答案。不要添加多余步骤。

你的目标是：
{input}

你原来的计划是：
{plan}

你目前已完成以下步骤：
{pastSteps}

请根据以上情况更新计划。若已无需更多步骤、可以回复用户，请使用 response 工具给出最终回复。
否则，请填写剩余计划。只保留仍需执行的步骤，不要把已经做过的步骤再写进计划。`,
);

const replanner = replannerPrompt
  .pipe(model.bindTools([planTool, responseTool]))
  .pipe(new JsonOutputToolsParser());

// ----- Graph 节点函数 -----
type ToolCallOutput = {
  type?: string;
  name?: string;
  args?: Record<string, unknown>;
};

async function planStep(
  state: PlanExecuteStateType,
): Promise<Partial<PlanExecuteStateType>> {
  const output = (await planner.invoke({
    objective: state.input,
  })) as ToolCallOutput[];
  const steps = (output[0]?.args as { steps?: string[] })?.steps ?? [];
  return { plan: steps };
}

async function executeStep(
  state: PlanExecuteStateType,
  config?: RunnableConfig,
): Promise<Partial<PlanExecuteStateType>> {
  if (state.plan.length === 0) return { pastSteps: [], plan: [] };
  const task = state.plan[0];
  const result = await runAgentLoop(task, config);
  return {
    pastSteps: [[task, result]],
    plan: state.plan.slice(1),
  };
}

async function replanStep(
  state: PlanExecuteStateType,
): Promise<Partial<PlanExecuteStateType>> {
  const output = (await replanner.invoke({
    input: state.input,
    plan: state.plan.join("\n"),
    pastSteps: state.pastSteps
      .map(([step, result]) => `${step}: ${result}`)
      .join("\n"),
  })) as ToolCallOutput[];

  const toolCall = output[0];
  if (!toolCall) return { plan: [] };

  const toolName = toolCall.name ?? toolCall.type;
  if (toolName === "response") {
    return {
      response: (toolCall.args as { response?: string })?.response ?? "",
    };
  }
  return { plan: (toolCall.args as { steps?: string[] })?.steps ?? [] };
}

// response 有值说明任务完成，无值说明继续执行
function shouldEnd(state: PlanExecuteStateType): "true" | "false" {
  return state.response ? "true" : "false";
}

// ----- 构建 StateGraph -----
// planner → agent → replan，replan 根据 response 是否有值决定结束或继续
const workflow = new StateGraph(PlanExecuteState)
  .addNode("planner", planStep)
  .addNode("agent", executeStep)
  .addNode("replan", replanStep)
  .addEdge(START, "planner")
  .addEdge("planner", "agent")
  .addEdge("agent", "replan")
  .addConditionalEdges("replan", shouldEnd, {
    true: END,
    false: "agent",
  });

const app = workflow.compile();

// ----- 运行 Demo -----
async function main() {
  const config = { recursionLimit: 50 };
  const inputs = {
    input:
      "请计算 (25 * 4) + 17 的结果，再告诉我今天的日期，并用一句话总结这两个信息。",
  };

  console.log("=== Plan-and-Execute streaming ===\n");
  for await (const event of await app.stream(inputs, config)) {
    console.log(JSON.stringify(event, null, 2));
  }

  const final = await app.invoke(inputs, config);
  console.log("\n--- Final response ---");
  console.log(final.response);
}

main().catch(console.error);
```

三个节点通过 `PlanExecuteState` 共享状态，整条链路不需要手动编排调用顺序：

- `planStep` 调用规划器拆出步骤列表
- `executeStep` 每次只执行 `plan[0]`（取第一步），把结果追加到 `pastSteps` 并把已完成的步骤从 `plan` 里移除
- `replanStep` 读取当前状态决定是继续执行还是直接作答

执行节点内部的 `runAgentLoop` 是一个手写的 `ReAct` 循环：调用绑定了工具的模型，有 `tool_calls` 就用 `ToolNode` 执行并把结果追加进消息列表，没有 `tool_calls` 就直接返回。每个步骤最多循环 10 次，避免无限重试。

实际项目中可以把 `calculator` 和 `getCurrentDate` 换成真实的工具（查数据库、调 API 等），状态结构和图的连线方式完全不用改。

## 小结

`Plan-and-Execute` 的价值在于把想清楚再动手这件事做成了工程约束，规划阶段逼模型先看清全局，执行阶段每步有据可查，出了问题知道是哪一步、该怎么改。它和 `ReAct` 不是非此即彼的关系，工程里更常见的是两者组合：`Plan-and-Execute` 管整体节奏，`ReAct` 管单步内的工具调用与多轮推理。遇到复杂长链条任务时，先用规划把任务切成清晰的步骤边界，再在每步内部按需引入 `ReAct`，是目前最稳定也最实用的组合方式。
