前面的内容里，会话记忆、`Reflexion`、`Self-Critic` 多半把状态留在当前进程或当前会话里。换入口或重启后，偏好与背景容易一次清空。我们要回答的长期记忆问题是，怎样让 `agent` 跨会话、跨 `thread`、跨时间点仍能写入并取回对未来仍然有用的信息。

先把叙事路线摆顺。研究一侧，`MemGPT` 用换入换出把上下文比作 `RAM`、把外存比作磁盘；`CoALA` 再从认知科学里拆出四类记忆，给后续对照用的语言。工程一侧，在 `LangChain` 当前官方路径里，跨 `thread` 的持久读写主要落在 `LangGraph` 的 `store` 上。

在进入接口与配置之前，我们先立住长期记忆的对象边界。它关心的是时间轴上的可恢复性，典型落点是偏好、画像、可复用经验这类下一通对话仍应对齐生效的信息。它不等于把反思链或自评链拖得更长，也不等于单纯把上下文窗口撑宽。

如下。

当我们再遇到换入换出与记忆类型时，可以用一句判据对齐思路，先看这条机制到底是在扩展当前推理能看到的上下文，还是在扩展关掉窗口以后仍能被点名取回的那部分世界。

责任边界要收束到同一套框架里。短期侧是同一条 `thread` 里的图运行与快照，长期侧是跨 `thread` 仍可检索的文档集合，常见误区是把二者当成同一只抽屉。再来一段同口径备忘，方便我们之后对照 `checkpointer` 与 `store` 时不走样。

关于这两条支路，我们先记住一组判据就好。`checkpointer` 与图状态服务续跑与回放，换 `thread` 或换进程后断档往往属于边界内的结果。`store` 与 `namespace` 服务跨会话的点名取回，写入路径、分区与检索契约一旦含糊，下一通对话里就容易取回落空，根因常常和误以为已经落盘叠在一起。

## 短期记忆与长期记忆的本质区别

短期记忆是 `thread-scoped` 的：某一会话线程里的状态，经 `checkpointer` 落成 `checkpoint`，同一 `thread` 内多轮对话可以续写。会话结束或换了 `thread`，这些状态不再随手可得。

长期记忆不绑在某个 `thread` 上，而是落在自定的 `namespace` 里，任意后续会话都能按路径取回，适合放用户偏好、画像、历史经验、行为规则等跨会话信息。

`MemGPT` 把结构说得很直白：`in-context storage` 像 `RAM`，模型当下就能读；`external storage` 像磁盘，里面常见 `archival memory`（长期归档）与 `recall memory`（对话索引）。`agent` 用工具调用决定何时从外存换进上下文、何时换出落盘。这条主动调度线，正是长期记忆与整段聊天记录原封不动灌进向量库这类堆法之间的分水岭。

分工可以记两句：

- 短期记忆管的是当前这轮怎么把话说通。
- 长期记忆管的是这个 `agent` 对这个人或这个组织长期沉淀了什么。

## 长期记忆不是把聊天记录全部存下来

最值得单独强调的一点。

长期记忆的本质，是从交互里抽出仍值得复用的信息，整理成可检索的知识，而不是把每句对话原样入库。用户说喜欢简洁、常用 `TypeScript`、在做文档平台，这类适合进长期记忆。用户说刚去倒了杯水，多数时候不必记。

`LangMem` 把一次 `memory operation` 抽象成固定套路：给定一段对话和当前 `memory state`，让 `LLM` 判断如何扩展、合并、更新，再产出新的 `memory state`。这是主动提炼，不是被动堆日志。

抽取过猛与抽取不足都会伤效果：过猛会污染后续推理，过弱会漏掉高价值信号。取舍没有通用答案，官方也写明这是 `application-specific` 的。工程上要为写入设计触发条件与阈值，而不是交给模型自由发挥。

## 三类长期记忆

`CoALA` 借认知科学里的经典分法，把语言 `agent` 的长期记忆拆成三类。落地时好用，因为三类的存储形态、更新频率和检索方式本来就不一样。下面备忘给出一屏总览，细节在各小节展开。

![20260327150829](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327150829.png)

三类记忆围绕同一个长期记忆核心，各有边界，也各自对应更顺手的存储与检索方式。

### `Semantic memory`

事实、概念、用户偏好、稳定背景，最常见，主要服务个性化。

官方在 `semantic memory` 里又拆出两种形态。

`Profile` 适合结构化、当前态、需要一眼读出的画像：姓名、语言、时区、回答风格、专业背景。可以想成一张会原地覆盖的档案卡，建议用 `schema` 锁结构，用更新代替无限追加，读取时按 `key` 直取，不必每次做语义检索。

`Collection` 适合条目不断增长、条数不定、按需召回的便签式知识。难点在更新、失效、去重与合并。2025 年的 `A-MEM` 受 `Zettelkasten` 启发：新条目不只存原文，还生成上下文说明、关键词与指向旧条的链接，让记忆网能随时间改写，而不是只堆体积。这和单纯 `key-value` 堆栈的差别在于，新写入可以触发对旧记忆的修订。

### `Episodic memory`

重点不在用户是谁这一问本身，而在发生过什么、以及哪种做法真的奏效。`CoALA` 强调它保存的是情境与结果，相对 `semantic memory` 更关心完整情境、处理过程和为何有效。

典型例子：

- 某次检索策略在类似问题上跑通。
- 用户过去问工程题时，往往更认原理先行、附完整代码、再给落地步骤这一套节奏。
- 某类任务里某种工具调用顺序最稳。

适合做 `few-shot`、成功案例回放以及 `agent` 自改进。2025 年的 `AriGraph` 把 `episodic` 与 `semantic` 收成一张知识图谱：节点是概念与事件，边是关系，`agent` 需要跨概念推理时可以沿边走，而不是只靠孤立的向量近邻。

### `Procedural memory`

记的是 `agent` 该怎么做事，编码的是系统行为与响应习惯，起点常常是 `system prompt`，随后会随反馈慢慢改。

例如：

- 回答前先确认上下文够不够。
- 代码类问题优先给可运行的完整示例。
- 有用户资料时先读 `profile` 再定调。
- 某类任务固定走某条工具链。

它本质是可演化的系统规则。`LangMem` 里的 `prompt optimizer` 会沿交互轨迹看哪些规则真的有用，再去改 `system prompt`，这才是从经验里学做事方式，而不是只堆对话。

## 底层存储结构

长期记忆在实现层绕不开 `store`。每条记忆落成一个 `JSON` 文档，由两个维度一起定位，缺一则要么写不进、要么读不准。

- `namespace`，多级路径，管隔离边界与检索范围。
- `key`，文档标识，在同一 `namespace` 下唯一。

对外常见三个方法，语义上可以理解成写、点读、带条件的翻找。

- `put(namespace, key, value)` 写入或整文档覆盖，后写覆盖先写，需要我们自己在业务层做版本或合并策略。
- `get(namespace, key)` 按 `key` 精确读，适合 `profile` 这类结构稳定的对象。
- `search(namespace, { query, filter })` 在向量索引可用时做语义近邻，并叠加结构化过滤，适合条目多、边界软的集合类记忆。

本地开发用 `InMemoryStore` 即可，本质是内存结构，进程退出即丢，用来跑通读写与 `namespace` 设计最合适。上线可换成 `AsyncPostgresStore`、`Redis`、`MongoDB` 等实现，只要接口一致，上层不必重写，迁移时重点测的是并发写入与索引重建耗时，而不是方法名是否一致。若配置了 `embedding`，写入路径里通常会异步或同步向量化，`search` 才能按相似度排序，否则退化为过滤与扫描的组合，延迟模型会完全不同。向量服务的 `dims` 与所用模型必须一致，不要把聊天用的 `baseURL` 误配给另一套向量 `API`，否则会出现写入成功、检索却一直飘在错误子空间里的隐性故障。

下面演示 `InMemoryStore` 带索引时的典型写法（向量模型与密钥请换成运行环境里真实可用的组合）。示例里 `dims` 取 `1536` 是为了和常见 `text-embedding-3-small` 维度对齐，我们换模型时要同步改这一行。

```typescript
import { InMemoryStore } from '@langchain/langgraph';
import { OpenAIEmbeddings } from '@langchain/openai';

const embeddings = new OpenAIEmbeddings({
  model: 'text-embedding-3-small',
  // apiKey 与 configuration.baseURL 必须指向实际提供该 embedding 的服务
});

const store = new InMemoryStore({
  index: {
    embed: async (texts: string[]) => embeddings.embedDocuments(texts),
    dims: 1536,
  },
});

const userId = 'user_001';
const namespace = [userId, 'preferences'];

await store.put(namespace, 'coding-style', {
  language: 'TypeScript',
  answerStyle: '简洁直接',
  projectType: 'AI 文档平台',
  timezone: 'Asia/Shanghai',
  updatedAt: new Date().toISOString(),
});

const item = await store.get(namespace, 'coding-style');

const results = await store.search(namespace, {
  filter: { language: 'TypeScript' },
  query: '回答风格偏好',
});
```

## `namespace` 设计

`namespace` 决定隔离边界与检索粒度，是长期记忆设计里最先要想清的一层。路径过粗时，不同业务线会在同一个前缀下抢检索范围，噪声和权限风险一起上来。路径过细时，迁移与批量清理成本又会上去，因此我们要在粗粒度隔离与可运维性之间留一点缓冲段。

实用原则可以收成一句，把谁在用、在什么场景、存哪类数据，依次折成路径上的多级段。一般顺序是主体、租户或组织、记忆类型，必要时再补环境或版本段，避免把易变字段塞在最前面导致全路径频繁失效。

```text
('users', user_id, 'profile')        → 用户画像，原地更新，直接读取
('users', user_id, 'preferences')    → 偏好集合，语义搜索召回
('users', user_id, 'episodes')       → 历史经验，few-shot 复用
('orgs', org_id, 'shared_knowledge') → 组织共享知识
('projects', project_id, 'rules')    → 项目级行为规则
```

`LangMem` 支持动态 `namespace`，模板可写成由 `memories` 与占位 `user_id` 等组成的多段路径，运行时从 `config` 注入真实 `userId`，多租户很省事。还可以在同一路径下加收 `manual_memories` 段放用户主动标注，或单开助手侧路径放助手沉淀，权限与清理都更好做。落地时建议我们同时写清路径与 `key` 的命名规范文档，并在代码里集中封装常量，避免字符串散落在各工具实现里。

收益很直接，包括边界清楚、检索更准、迁移与清理更可控、多租户更安全四条。反过来，若路径设计漂移，最常见症状是工具里硬编码了旧前缀，线上却有一半流量写进新前缀，排查会拖很久。

## 在工具里读取长期记忆

长期记忆真正发力，多半在工具阶段。`agent` 在决定下一步前，用专门工具把当前用户或组织的长期信息拉出来，再影响生成。相比完全依赖模型在上下文中自己回忆，工具读回的是可验证的结构化结果，便于我们做缓存、超时与脱敏。

下面的 `getUserInfo` 根据 `runtime.context` 里的 `userId` 读 `store`，把结果交回模型。运行前需在环境里配置 `DEEPSEEK_API_KEY`，与示例中的 `baseURL` 指向一致。线上我们通常会为工具调用加超时与重试上限，并对返回体做字段级脱敏，避免把内部备注或运营标签原样喂回对话。

```typescript
import * as z from 'zod';
import { createAgent, tool, type ToolRuntime } from 'langchain';
import { InMemoryStore } from '@langchain/langgraph';
import { ChatOpenAI } from '@langchain/openai';

const model = new ChatOpenAI({
  model: 'deepseek-chat',
  apiKey: process.env.DEEPSEEK_API_KEY,
  configuration: { baseURL: 'https://api.deepseek.com' },
});

const store = new InMemoryStore();

const contextSchema = z.object({ userId: z.string() });

await store.put(['users'], 'user_001', {
  name: '张伟',
  language: '中文',
  answerStyle: '简洁直接',
  timezone: 'Asia/Shanghai',
});

const getUserInfo = tool(
  async (_, runtime: ToolRuntime<unknown, z.infer<typeof contextSchema>>) => {
    const { userId } = runtime.context;
    if (!userId) throw new Error('userId is required');

    const userInfo = await runtime.store.get(['users'], userId);
    return userInfo?.value ? JSON.stringify(userInfo.value) : '未找到用户信息';
  },
  {
    name: 'getUserInfo',
    description: '从长期记忆 store 中查询用户信息。',
    schema: z.object({}),
  },
);

const agent = createAgent({
  model,
  tools: [getUserInfo],
  contextSchema,
  store,
});

await agent.invoke(
  { messages: [{ role: 'user', content: '你对我有哪些了解？' }] },
  { context: { userId: 'user_001' } },
);
```

请求侧带上 `userId`，`agent` 在工具阶段执行 `getUserInfo` 取数，再融进最终回复。更常见的工程习惯，是在每次请求开头就把 `profile` 读进 `system prompt` 或 `runtime context`，让整段生成都带着用户背景，而不是等到模型想起要调工具才去取。若两种做法并存，我们要约定优先级，避免同一轮里既注入了完整画像、又在工具里重复拉回一大段相同字段，白白占掉上下文。

## 在工具里写入长期记忆

读之外还要在合适时机写回。例如用户说姓名、时区、或希望以后直接给结论，这类都适合沉淀。写入前最好在工具或上游管线里做一次轻量校验，例如时区是否落在已知枚举、偏好是否与产品允许的值冲突，减少脏数据进 `store` 后再靠人工清理。

下面的 `saveUserInfo` 在模型整理出结构化字段后写入 `store`。密钥同样建议走环境变量，勿把真实 `apiKey` 写进仓库。若同一 `userId` 会有并发会话，我们还要评估写入是否需要乐观版本号或后台合并任务，否则最后一条会话覆盖前一条会话的整文档，用户会感到偏好忽隐忽现。

```typescript
import * as z from 'zod';
import { tool, createAgent, type ToolRuntime } from 'langchain';
import { InMemoryStore } from '@langchain/langgraph';
import { ChatOpenAI } from '@langchain/openai';

const model = new ChatOpenAI({
  model: 'deepseek-chat',
  apiKey: process.env.DEEPSEEK_API_KEY,
  configuration: { baseURL: 'https://api.deepseek.com' },
});

const store = new InMemoryStore();

const contextSchema = z.object({ userId: z.string() });

const UserInfo = z.object({
  name: z.string().optional(),
  language: z.string().optional(),
  answerStyle: z.string().optional(),
  timezone: z.string().optional(),
  expertise: z.string().optional(),
});

const saveUserInfo = tool(
  async (
    userInfo: z.infer<typeof UserInfo>,
    runtime: ToolRuntime<unknown, z.infer<typeof contextSchema>>,
  ) => {
    const { userId } = runtime.context;
    if (!userId) throw new Error('userId is required');

    const existing = await runtime.store.get(['users'], userId);
    const merged = { ...(existing?.value ?? {}), ...userInfo };
    await runtime.store.put(['users'], userId, merged);

    return '用户信息已保存。';
  },
  {
    name: 'save_user_info',
    description:
      '当用户透露姓名、偏好或背景信息时，调用此工具将其保存到长期记忆中。',
    schema: UserInfo,
  },
);

const agent = createAgent({
  model,
  tools: [saveUserInfo],
  contextSchema,
  store,
});

await agent.invoke(
  {
    messages: [
      {
        role: 'user',
        content: '我叫李明，时区是上海，以后回答我请用中文，直接给结论。',
      },
    ],
  },
  { context: { userId: 'user_001' } },
);

const result = await store.get(['users'], 'user_001');
console.log(result?.value);
```

先读后合并（`{ ...existing, ...newInfo }`）是常见工程细节，用来避免新写入抹掉旧字段，让 `profile` 走累积演进而不是每轮重置。若某些字段需要显式删除，我们还要另建删除语义或哨兵值，单靠合并对象有时会留下幽灵字段。

读写合上环之后，用户边聊边沉淀偏好，下次会话就能直接受益。建议我们在网关或审计日志里顺带记一笔写入来源，例如来自哪次会话、哪条工具调用，后续做误写回滚或合规取证会轻松很多。

## 两种写入时机

官方把 `memory formation` 拆成两条路径，我们按场景混用即可，不必二选一绑死。

`hot path` 指当前对话里当场写。`agent` 判断某条信息值得固化，就立刻调记忆工具落盘。`LangMem` 的 `create_manage_memory_tool` 面向这条路径，可对单条做增删改。收益是马上生效，同一轮里就能刷新 `profile`，用户立刻感到被记住。代价是拉长当次请求的尾延迟，也更容易出现该记没记、不该记乱记两类失误。典型适用场景包括用户口头要求记住、或偏好信号很强且歧义低。

`background` 指事后反思再入库，文档里有时叫 `subconscious formation`。会话结束或静默一段时间后，由单独任务扫整段对话做抽取与合并。`LangMem` 的 `create_memory_store_manager` 走这条线，并可配 `ReflectionExecutor` 做 `debounce`，避免每条消息都打一次后台管线。收益是在线路径更轻，更容易做跨轮模式识别与去重。代价是写入有滞后，当前轮未必立刻读到刚聊过的细枝末节。典型适用场景包括从长对话里汇总偏好、沉淀项目结论、抽取 `episodic` 经验条。

工程上我们常把强信号交给 `hot path`，把弱信号或需要跨轮汇总的交给 `background`，在延迟与质量之间换档位。两条路径写入触发点不同，终点仍是同一个 `store`，因此要保证 `namespace` 与 `key` 约定一致，否则会出现两条管线争写同一文档、后写覆盖先写而不自知的情况。下面备忘左侧对应热路径、右侧对应事后提炼，可与上文两条路径逐条对上。

![20260327163123](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327163123.png)

热路径与后台路径最终都收敛到 `store`。差别主要在写入触发点、在线尾延迟、以及我们愿意为当场一致性付出多少额外工具调用。

## 三种检索方式

写只是第一步，读决定体验。官方常见三条路，对应三类问题，我们分开记不容易混。

`Direct access` 按 `namespace` 与 `key` 直取，最适合结构稳定的 `profile`。常见做法是每次用户发消息前，先按用户维度取出画像，塞进 `system prompt` 或 `runtime context`。延迟低，结果确定，也便于做权限校验，因为路径本身就是访问控制的一部分。要留意的是，直取不会自动帮我们判断这条画像是否过期，过期策略仍要在写入侧维护。

`Semantic search` 按语义近邻找，最适合条目多、边界软的 `collection` 与 `episodic memory`。用户问上次某次编辑建议时，我们就去对应 `namespace` 调 `search`，用 `query` 与 `filter` 叠语义与结构化条件。成本与延迟通常高于直取，因此更依赖合理的 `top_k`、截断与重排，否则容易把噪声段落整段搬进上下文。

`Metadata filtering` 按属性收窄范围，例如只要某用户、某场景、某类记忆。`namespace` 设计得好，本身就是最自然的元数据过滤器，很多团队会把租户、产品线、记忆类型直接编进路径，减少后置过滤压力。

实践里我们常做组合。默认用 `direct access` 拉 `profile`，任务需要历史经验时再 `semantic search` 拉 `episode`，最后把检索结果压缩成短 bullet 再注入 `prompt`。不要把检索结果整包塞进上下文，噪声一多，模型反而分心，token 账单也会上去。下面备忘把三条检索与注入位置放在一屏，便于和上文三条路对照。

![20260327164644](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327164644.png)

三条检索在 `store` 上各管一段需求。经压缩后再分流进 `system prompt` 与 `runtime context`，避免一次塞满上下文，也便于我们事后对照日志判断到底是哪一路召回在拖慢或带偏当次回答。

## 整体架构关系

从更高一层看，长期记忆是挂在 `agent` 工具链下面的一块基础设施，和模型权重解耦，却和每次请求的上下文装配强相关。下面备忘把请求、推理、工具读写与两条写入节奏放在一条时间线上，方便我们和后文代码示例对齐。

![20260327164718](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327164718.png)

新请求进来后，`agent` 在推理中通过工具访问 `store`，按任务挑需要的记忆，再决定要不要调用写入工具。一次运行里可以多次读、少量写，也可以把写主要留给离线管线。结束后，`hot path` 工具往往同步写回要点，`background` 异步提炼再入库，两者并行时更要盯住冲突与幂等。长期记忆因此更像随任务流动的知识库，而不是一张静态大表，观测上建议至少记录每次读写的 `namespace`、耗时与命中条数，方便我们回滚错误写入。

## 四层记忆模块的落地结构

不要一上来做一张万能表。更稳的是按职责拆四层，各层有清晰 `schema` 与检索策略，迁移时也可以按层渐进启用。

第一层 `Profile` 偏稳定画像，字段常见为 `name`、`language`、`timezone`（例如 `Asia/Shanghai`）、`answerStyle`、`expertise`。原地更新优于无限追加，读时按 `key` 直取。官方常建议对 `profile` 关插入只留更新，例如 `enable_inserts=false`，避免同键多条堆叠。若和业务身份强绑定，还要提前想好注销与导出，以免合规阶段返工。

第二层 `Preference Collection` 管偏好与长期便签，例如喜欢的讲解方式、常用技术栈、产品硬约束。这一层更适合 `collection` 加语义召回。若借鉴 `A-MEM`，写入时带关键词与指向旧条的关联，检索会稳很多，也便于后续合并与失效。

第三层 `Episode` 管成功经验，例如某类 `bug` 排查套路、某类任务的最佳操作顺序、某条工具链更省步数。适合做 `few-shot` 与策略回放。写入时建议打任务类型标签，召回时不只靠向量。若知识关系复杂，可参考 `AriGraph`，用图组织 `episode`，让跨概念推理能沿边走。

第四层 `Procedural Rules` 管行为规则与 `prompt` 优化结果，例如代码题先给可运行示例、模糊请求先轻量澄清。它会随反馈演化，`LangMem` 的 `prompt optimizer` 主要维护这一层。与 `Profile` 的差别在于，这一层更像系统策略，变更频率与审计需求往往更高，适合单独做版本号或变更记录。

## 该记什么、不该记什么

这是最容易做坏的一环。我们落笔前先问三个问题，这条信息在七天后是否仍应生效、是否会被多次检索、以及一旦写错有没有低成本纠正路径。三个答案里至少有两个偏肯定，再进长期记忆通常更稳。

| 建议记                           | 建议不记                               |
| -------------------------------- | -------------------------------------- |
| 长期稳定偏好（语言、风格、时区） | 一次性闲聊（例如刚吃完饭）             |
| 身份与背景（职业、项目、技术栈） | 常变却没有更新策略的字段               |
| 未来高概率复用的约束             | 与业务无关的噪声                       |
| 高价值成功经验与流程             | 敏感信息（除非有权限、合规与删除方案） |
| 系统行为与风格修正               | 无复用价值的瞬时状态                   |

`A-MEM` 也讨论过质量，相关性不只是向量相似度，还要考虑重要性与使用频率。常被触发的条目权重大，过时条目应触发修订或归档，而不是无限堆高。我们也可以把使用频率当作清理任务的输入，让冷条目自然降权或合并，避免记忆库体积与噪声同步膨胀。

## 总结

`Long-term memory` 的主线可以收成几条，下面用无序列表收束。

- 分类。`semantic memory` 管用户是谁、知道什么；`episodic memory` 管发生过什么、哪种做法有效；`procedural memory` 管 `agent` 该怎么做事。三类不要混在同一个写入策略里。
- 生命周期。对话里识别高价值信号，`hot path` 当场写或 `background` 事后提炼；写入按 `namespace` 分层；读取按直取、语义、过滤组合使用，再压缩进 `prompt`，影响当次输出。
- 落地分层。`Profile` 管稳定画像，`Preference Collection` 管偏好便签，`Episode` 管成功经验，`Procedural Rules` 管行为规则。每层有自己的 `schema`、写入与检索策略，避免一张表包一切。

实践中还容易把下面几件事想反，设计时可直接对照第二列纠偏。

| 误区                               | 更贴近事实的说法                                                                   |
| ---------------------------------- | ---------------------------------------------------------------------------------- |
| 记忆越多越好                       | 召回噪声会拖垮当次生成，成本与延迟也会上去                                         |
| 长期记忆等于向量库                 | `store` 同时支持直读、语义搜、元数据过滤，`profile` 往往直读更快                   |
| 存了 `agent` 就会自动用            | 要写清何时写、何种 `schema`、何时读、读多少、怎么注入，这才是 `memory engineering` |
| `profile` 与 `collection` 混一张表 | 读写模式不同，应分 `namespace`，最好也分 `schema`                                  |

起步不必四层齐上，不必一次把四层模块全部铺满。先从 `Profile` 做起，让 `agent` 记住语言偏好与时区（例如 `Asia/Shanghai`），再视需要叠上 `Preference Collection` 与 `Episode`。当某类任务反复犯同类错时，再引入 `Procedural Rules`，通常性价比最高。

就定义而言，`long-term memory` 并不是把完整聊天记录原样囤积，而是让 `agent` 在不额外训练模型的前提下，逐步积累用户知识、任务经验与行为规则，并让输出越来越贴场景。
