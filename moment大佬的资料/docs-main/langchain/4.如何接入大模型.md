在前面内容里，我们已经看过裸调 API 容易卡在哪里、`LangChain` 在工程里能帮你省什么事，以及 `Function Call` 是怎么跑通的。

然后接下来进入写应用的阶段，本章先把大模型接稳，把厂商、密钥、模型名集中配好，让链、`Agent`、工具都共用同一个 `model` 实例，别在业务代码里东一块 `baseURL`、西一个模型名字符串。接下来会依次写初始化方式、各家怎么接、常用参数怎么设，以及 `invoke`、`stream`、`batch` 三种调用和工具绑定的大致写法。

## 统一接入多家厂商

LangChain 用聊天模型抽象（`ChatOpenAI`、`ChatAnthropic` 等）封装不同厂商的 API。只要厂商提供与 OpenAI 兼容的 HTTP 接口（DeepSeek、智谱、月之暗面等均如此），就可以用 `ChatOpenAI` 配不同的 `configuration.baseURL` 和 `model` 接入，不需要为每家写一套调用逻辑。

后续链或 `Agent` 里统一依赖这个 `model` 实例，换厂商时改配置即可，业务代码不用动。

以 DeepSeek 为例：

```typescript
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: {
    baseURL: "https://api.deepseek.com",
  },
});
```

`configuration.baseURL` 指向 DeepSeek 的 API 地址，`model` 决定用哪个模型，`apiKey` 负责鉴权。封装好这个 `model` 之后，所有后续调用都通过它完成。切换到其他兼容 OpenAI 接口的厂商，只需改这三个参数。

## 两种初始化方式

### 直接实例化模型类

直接用各厂商的模型类传参，适合在代码里显式配置 `baseURL`、`apiKey`、`model` 的场景，也是对接 DeepSeek 等兼容厂商时最常用的方式，上面的 DeepSeek 例子就是这种。

### initChatModel

另一种是 `initChatModel`，按"模型名"或"厂商:模型名"字符串从环境变量读密钥并初始化，适合多模型对比、或通过配置文件动态切换的场景：

```typescript
import { initChatModel } from "langchain/chat_models/universal";

const model = await initChatModel("gpt-4.5", {
  temperature: 0.7,
  maxTokens: 1000,
});
```

也可以带厂商前缀明确指定，例如 `"openai:gpt-5.4"`、`"anthropic:claude-sonnet-4-6"`。用 `initChatModel` 时，对应厂商的环境变量（如 `OPENAI_API_KEY`）必须提前设好。

两种方式得到的都是同一套接口，都支持 `invoke`、`stream`、`batch`，按需选择即可。

## 支持的厂商

LangChain 支持主流模型厂商，每个厂商有独立的集成包和对应的 `Chat*` 类：

| 厂商          | 安装包                    | 模型类                   | 环境变量               | 模型名示例                     |
| ------------- | ------------------------- | ------------------------ | ---------------------- | ------------------------------ |
| OpenAI        | `@langchain/openai`       | `ChatOpenAI`             | `OPENAI_API_KEY`       | `gpt-5.4`                      |
| DeepSeek      | `@langchain/openai`       | `ChatOpenAI`             | 自定义 `apiKey`        | `deepseek-chat`                |
| Anthropic     | `@langchain/anthropic`    | `ChatAnthropic`          | `ANTHROPIC_API_KEY`    | `claude-sonnet-4-6`            |
| Google Gemini | `@langchain/google-genai` | `ChatGoogleGenerativeAI` | `GOOGLE_API_KEY`       | `gemini-2.5-pro`、`gemini-2.5-flash` |
| Azure OpenAI  | `@langchain/openai`       | `AzureChatOpenAI`        | `AZURE_OPENAI_API_KEY` | `gpt-5.4`                      |
| AWS Bedrock   | `@langchain/aws`          | `ChatBedrockConverse`    | AWS 凭证               | `anthropic.claude-sonnet-4-6`  |

模型是 `Agent` 的"推理引擎"，决定何时调工具、如何理解结果、何时给出最终回答。不同模型在复杂指令遵循、结构化推理、长上下文处理上各有强弱，建议按任务多试几种。

## 常用参数

初始化时支持一批通用参数，用于控制模型行为和资源消耗：

| 参数          | 说明                                                                                  |
| ------------- | ------------------------------------------------------------------------------------- |
| `model`       | 具体模型名，有的写法支持"厂商:模型"一起指定                                           |
| `apiKey`      | 访问该厂商 API 的密钥，建议从环境变量读取，不要写死                                   |
| `temperature` | 输出随机程度，0～2 之间。创意写作可调高（0.8～1.2），分类、抽取、翻译宜调低（0～0.3） |
| `maxTokens`   | 限制最多生成多少 `token`，控制回复长度与成本                                          |
| `timeout`     | 单次请求最长等待时间（毫秒），网络不稳时可适当加大                                    |
| `maxRetries`  | 请求失败时的最多重试次数，用于自动恢复偶发故障                                        |

用 `initChatModel` 时，这些参数作为第二个参数对象传入：

```typescript
import { initChatModel } from "langchain/chat_models/universal";

const model = await initChatModel("claude-3-7-sonnet-latest", {
  temperature: 0.7,
  maxTokens: 1000,
  timeout: 30_000,
  maxRetries: 3,
});
```

## 三种调用方式

模型必须被"调用"才会生成输出。三种方式分别对应不同的使用场景：

![20260309102019](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260309102019.png)

上图左中右依次展示了三种调用方式的特点：`invoke` 等待完整结果后返回，`stream` 逐 token 边生成边消费，`batch` 并发处理多条独立请求。选哪种取决于场景而不是性能偏好。

### invoke

最直接的调用方式，传入消息后等待完整回复，返回一条 `AIMessage`：

```typescript
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const response = await model.invoke("为什么鹦鹉会说话？");
console.log(response.content);
```

运行后控制台会打印模型的完整回复：

![20260214185437](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260214185437.png)

需要带系统提示或多轮对话时，传消息数组即可。消息可以是 LangChain 的消息类（`SystemMessage`、`HumanMessage`、`AIMessage`），也可以是简单的 `{ role, content }` 对象：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import {
  SystemMessage,
  HumanMessage,
  AIMessage,
} from "@langchain/core/messages";
import type { BaseMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const conversation: BaseMessage[] = [
  new SystemMessage("你是一个把英文翻译成中文的助手。"),
  new HumanMessage("Translate: I love programming."),
  new AIMessage("我喜欢编程。"),
  new HumanMessage("Translate: I love building applications."),
];

const response = await model.invoke(conversation);
console.log(response.content);
```

`invoke` 的返回是 `AIMessage`，不是字符串。要拿到文字内容需要访问 `.content`。

### stream

需要边生成边展示时，用 `stream` 代替 `invoke`。它返回一个异步迭代器，每次 yield 一块 `AIMessageChunk`，适合在界面上逐字展示或做实时管道处理：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import type { AIMessageChunk } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const stream = await model.stream("为什么鹦鹉的羽毛是彩色的？");
let full: AIMessageChunk | null = null;

for await (const chunk of stream) {
  full = full ? full.concat(chunk) : chunk;
  process.stdout.write(chunk.text ?? "");
}
```

循环结束后 `full` 就是完整的消息，结构和 `invoke` 的返回一致，可以继续放进对话历史使用。

如果需要按语义事件处理（而不是原始 chunk），用 `streamEvents`。它按三类事件推送：`on_chat_model_start`（模型开始）、`on_chat_model_stream`（每来一块 token 触发一次）、`on_chat_model_end`（整条回复结束），适合做进度提示、统计或按事件类型分发：

```typescript
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const events = await model.streamEvents("你好", { version: "v2" });

for await (const event of events) {
  if (event.event === "on_chat_model_start") {
    console.log("开始，输入:", event.data.input);
  }
  if (event.event === "on_chat_model_stream") {
    process.stdout.write(event.data.chunk.text ?? "");
  }
  if (event.event === "on_chat_model_end") {
    console.log("\n完整回复:", event.data.output.text);
  }
}
```

注意第二个参数里必须带 `version: "v2"`，否则会报错。运行后终端会先逐 token 打印内容，最后输出完整回复：

![20260214190940](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260214190940.png)

在 `LangGraph` 等 `Agent` 框架里，即使使用 `model.invoke()`，只要整体以流式模式运行，LangChain 也会在内部走流式并触发对应回调，便于在界面上边生成边显示。

### batch

多条彼此独立的请求并发提交时，用 `batch` 比在循环里多次 `invoke` 更高效：

```typescript
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const responses = await model.batch([
  "为什么鹦鹉的羽毛是彩色的？",
  "飞机是怎么飞的？",
  "什么是量子计算？",
]);

for (const response of responses) {
  console.log(response.content);
}
```

返回数组与输入顺序一致，每一项都是完整的 `AIMessage`。三条请求并行发出，因此你会看到多条回复陆续输出，而不是等第一条结束才有第二条：

![20260214191559](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260214191559.png)

批量很大时，通过第二个参数里的 `maxConcurrency` 限制并发数，避免触发厂商限流：

```typescript
const responses = await model.batch(listOfInputs, {
  maxConcurrency: 5,
});
```

## 工具调用

LangChain 用 `bindTools` 把工具绑到模型上，省去了手写 `tools` 数组和轮询 `tool_calls` 的循环。整个流程和第三章的 Function Call 一样：模型返回 `tool_calls` → 应用执行工具 → 把结果写回对话 → 再调模型。

![20260214195817](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260214195817.png)

工具用 `tool()` 定义，传入执行函数和工具描述（名称、`zod` schema、用途说明），LangChain 会自动把它转成当前模型所需的 `tools` 格式：

```typescript
import { tool } from "@langchain/core/tools";
import { z } from "zod";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const getWeather = tool(
  (input: { location: string }): string => `${input.location} 天气晴朗，25°C。`,
  {
    name: "get_weather",
    description: "查询某地的当前天气状况。",
    schema: z.object({
      location: z.string().describe("要查询天气的城市名称"),
    }),
  },
);

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const modelWithTools = model.bindTools([getWeather]);

const messages = [new HumanMessage("波士顿天气怎么样？")];

const aiMsg = await modelWithTools.invoke(messages);
messages.push(aiMsg);

for (const toolCall of aiMsg.tool_calls ?? []) {
  const result = await getWeather.invoke(toolCall);
  messages.push(result);
}

const finalResponse = await modelWithTools.invoke(messages);
console.log(finalResponse.content);
```

运行结果如下图所示：

![20260215003328](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260215003328.png)

与第三章手写的裸调相比，LangChain 帮你处理了几件事：自动把 `tool()` 定义转成当前模型所需的 `tools` 格式，执行工具后自动生成带 `tool_call_id` 的 `ToolMessage`，结构化错误也会自动包装成合法的回传消息。

多轮或多工具时做法一样：只要有 `tool_calls` 就执行并追加结果，再 `invoke`，直到模型不再返回 `tool_calls`。若想强制走某工具，可传 `toolChoice: "any"` 或 `toolChoice: "工具名"`。

## 进阶能力

以下能力入门时不一定用到，做到一定阶段会很有用。

### 模型档案

部分聊天模型会暴露 `.profile`，里面是该模型的能力描述，包括 `maxInputTokens`（最大输入 `token` 数）、`toolCalling`（是否支持工具调用）、`imageInputs`（是否支持图像输入）等。

应用可以根据 `profile` 动态决策：例如上下文快满时自动触发摘要压缩，或根据任务类型选择合适的模型。若某模型的 `profile` 不准确，可在实例化时传入自定义 `profile` 对象覆盖。

### 多模态

部分模型支持图像、音频等非文本输入。输入时通过消息的 `content blocks` 传入（一条消息里同时包含文本块和图片块），LangChain 在不同厂商间提供统一的格式，同时兼容各厂商原生格式。若模型支持生成图像，返回的 `AIMessage` 的 `contentBlocks` 里会有 `type: "image"` 的块。

### 推理过程

部分模型（如 `DeepSeek-R1`、`Claude 3.7` 的扩展思考模式）会先输出推理链再给出答案。无论流式还是非流式，都可以从 `contentBlocks` 里筛出 `type === "reasoning"` 的块查看中间思考过程。有的厂商还支持配置推理强度（`low`、`high`）或推理 `token` 预算，用于在质量与延迟之间做权衡。

### 本地模型

若希望数据不出本机，或要使用自训练模型，可以用 `Ollama` 等集成在本地运行聊天模型。安装好 `Ollama` 并拉取模型后，用 LangChain 的 Ollama 集成包即可像调用云端模型一样 `invoke`、`stream`，无需 `apiKey`，走本地端口。

### Prompt 缓存

不少厂商支持对重复前缀做缓存以降低延迟和成本。有的隐式生效（相同前缀自动命中），有的需要显式标记可缓存边界。缓存通常有最小 `token` 数要求，低于阈值不会启用；命中后的用量信息会体现在返回的 `usage_metadata` 里。

### Token 用量追踪

大多数厂商会在返回的 `AIMessage` 上附带 `token` 使用信息，位于 `usage_metadata` 或 `response_metadata` 里，包含 `input_tokens`、`output_tokens` 等字段。流式场景下，部分 API 需要额外开启某一选项才会在流中携带用量，关心按 `token` 计费或限流时需要确认当前集成的支持情况。

### RunnableConfig

`invoke`、`stream`、`batch` 都可以传第二个参数 `config`，常用字段如下：

| 字段             | 说明                                   |
| ---------------- | -------------------------------------- |
| `runName`        | 给这次调用起名，便于在日志或追踪里区分 |
| `tags`           | 字符串数组，给运行打标签               |
| `metadata`       | 任意键值对，会向下传递到所有子运行     |
| `callbacks`      | 回调处理器，用于监控或埋点             |
| `maxConcurrency` | 在 `batch` 时限制并发请求数            |
| `recursionLimit` | 链或图的最大递归深度，防止死循环       |

配合 `LangSmith` 做追踪，或自己做日志与限流时，这些字段会经常用到。

## 小结

接入大模型时，用 `ChatOpenAI` 等聊天模型类配置 `baseURL`、`model`、`apiKey`，可以统一对接多家兼容 OpenAI 接口的厂商，换厂商只改配置，业务代码不用动。

调用上掌握三种方式就够覆盖大部分场景：

- `invoke`：一次性返回完整 `AIMessage`，适合普通问答
- `stream`：逐 token 返回 `AIMessageChunk` 迭代器，适合长文、实时界面；需要按事件处理时用 `streamEvents`
- `batch`：并发处理多条独立请求，适合批量任务，用 `maxConcurrency` 控制并发

工具调用用 `bindTools` 绑定 `zod` 定义的工具，跑完"模型 → 工具 → 再模型"的循环即可，不需要手写 `tools` 数组和 `tool_calls` 轮询逻辑。

参数上重点用好 `temperature`、`maxTokens`、`timeout`、`maxRetries`；进阶需求按需查看模型档案、多模态、推理过程、本地模型、Prompt 缓存和 `RunnableConfig` 的相关文档。
