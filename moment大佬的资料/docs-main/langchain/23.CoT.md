`CoT`（Chain of Thought，思维链）来自提示工程里的一类做法，核心只有一句话，在给出最终结论之前，让模型用自然语言把中间步骤写出来，像纸上演算一样从已知推到未知。它不额外发明工具循环，主要改的是单次或多轮生成里上下文长什么样，以及你能从模型输出里抠出哪些可解析片段。

上一篇总览把推理结构和运行时编排分成两层。本篇只盯第一层，说清楚提示怎么写、和 `Agent` 里各段怎么叠、哪些场景反而不要硬上，以及工程里常见的坑。

## 它在解决什么问题

很多任务直接要一句答案时，模型容易跳步或胡猜。显式要求先推理再结论，相当于给模型脚手架，让它把依据摆出来。常见收益有这些。

- 算术、逻辑、多约束推理类问题，逐步写往往比跳步答更稳（也依赖具体模型与题型）
- 中间步骤给人看，便于核对错在哪，适合合规留痕、教学讲解、客服话术复核
- 解析出结构化片段后，流水线里可以接程序校验（验算、查规则、敏感词扫描）

也要心里有数，`CoT` 不是银弹。链写得再漂亮，也可能是流畅的胡扯，听起来每一步都有道理，最后结论仍错。所以不能把它当成可证明正确，只能当成提高可控性与可诊断性的手段。

代价同样实在，输出变长会带来 `token` 与延迟上升。推理段落若写成大段散文，还会挤占工具返回、检索片段、用户历史等更有信息密度的内容，需要在提示里收束长度。

## 一次生成为什么常常够用

很多业务场景里，用户只问一次，模型也只答一次。这时 `CoT` 完全落在单次 `chat` 或 `completion` 里，流程可以想得很简单。

## 提示里常见的几种用法

零样本 `CoT` 最省事，在指令末尾加一句请逐步推理再给出答案，或英文里常见的 `Let's think step by step`。模型会自己分段，格式不一定统一，适合先验证题型是否吃这一套。

少样本 `CoT` 在 `system` 或 `user` 里放一两个完整例题，例题里已经带好步骤 1、步骤 2、因此……的节奏。模型会模仿分段方式，格式稳定性通常好于零样本，适合你要固定审计口径或内部模板时。

分块强制输出适合要程序解析的场景。约定少见分隔符，例如先输出 `### REASONING` 再输出 `### ANSWER`，或要求只输出一段 `JSON`，字段名固定为 `reasoning` 与 `answer`。后面用切分或 `JSON.parse` 拆开。分隔符要选模型平时不会随手打出来的串，解析失败要有兜底。

还可以在同一提示里加一层结构，例如先列出已知条件与未知量，再分步计算，最后写结论。这比空泛的想一想更可执行，也更容易在日志里对齐你的检查清单。

若题目语言是中文，推理也尽量要求中文。混用语言时，模型有时会在术语和数字上对齐变差，除非你刻意要英文审计底稿。

## 和进阶技巧怎么搭配（可选）

自洽性（`self-consistency`）一类做法是，对同一题用偏高温度采样多条完整 `CoT`，再对最终答案做投票或选多数。成本按采样次数线性涨，适合高价值、低频次、又能自动判对错的题型（例如选择题、有标准答案的计算）。日常对话里要慎用，除非你愿意为准确率付多倍 `token`。

自动示例挖掘、动态挑少样本例题，属于 `CoT` 外围的提示优化话题，落地依赖离线数据与评测，这里不展开。你只要记住，少样本质量比数量更重要，一两个坏例题会把整条链带偏。

## 和 Agent 里各块怎么叠

在 `ReAct` 里，每一轮的 `Thought` 本质上就是一小段链式推理，只是被工具返回打断了多次。整条对话串起来，是一条推理与观察交织的长链。总览与 `26.ReAct` 专文里对 `Thought` 有细讲。这里只强调一点，`CoT` 不必锁死在单次 `completion` 里，也可以活在多轮循环的每一跳里。

在 `Plan-and-Execute` 里，计划阶段常常先列步骤再执行。你可以在同一刀里要求每一步下面用一句话写依据，计划列表本身就带一层 `CoT`。执行阶段跑偏时，对照计划里的理由排查会快很多。

在 `Reflexion` 或 `Self-Critic` 里，反思段落、批评段落同样是自然语言链。它们不负责分叉搜索，做的是对已有轨迹做文字推敲，和本篇说的单链是同一类技术在不同节点复用。

若产品上要给人看推理过程，记得做脱敏与裁剪。完整 `CoT` 可能含内部字段名、用户别称、不该外露的策略说明，展示层应只放行经过过滤的摘要。

## 什么时候不太适合硬上

下面几类情况，先别默认套长链，往往更划算。

- 答案极短、对错一眼可判，例如分类标签、单一实体抽取，长推理多半是噪音
- 延迟或 `token` 预算极紧，且业务允许错一点就人工兜底
- 真实数据必须以工具或数据库为准，模型在链里推出来的事实一律不能信，除非后面有硬校验

反过来，数学题、逻辑题、多步政策解读、要写清为什么这样答的 `B` 端场景，通常值得先试验 `CoT`。

## 工程上建议盯住的点

长度上限写进提示或后处理，例如推理不超过 8 句或不超过固定字数。否则遇到啰嗦模型，账单会飘。

推理与事实分开记账。该查数、该查库的步骤仍应走工具或检索，不要指望模型在想象出来的步骤里编对实时数据。链里出现的数字、日期、人名，只要会进对外文案，就要么引用工具返回值，要么事后校验。

解析必须容错。模型会漏标题、换标记、多打一个空行。常见兜底是整段当答案、或带更短 `system` 重试一次。重试仍失败就降级为不展示推理、只展示结论，并打监控告警。

温度与模型选择会改变 `CoT` 的稳定性。推理类任务多数团队会把温度压低，减少跳步和自相矛盾。换了底座模型，旧的少样本示例最好重新抽一小批题做回归。

## TypeScript 示意 分隔符与 JSON 两种拆法

下面两段都是一次请求并拆出推理与结论，便于接到你自己的网关或 `Agent` 节点里。把客户端换成你在用的供应商 `SDK` 即可，重点在消息约定与解析边界。

第一段用分隔符，兼容面宽，解析逻辑要自己维护。

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
});

const ANSWER_MARK = "---ANSWER---";

function splitReasoningAndAnswer(raw: string): {
  reasoning: string;
  answer: string;
} {
  const text = raw.trim();
  const idx = text.indexOf(ANSWER_MARK);
  if (idx === -1) {
    return { reasoning: "", answer: text };
  }
  const after = text
    .slice(idx + ANSWER_MARK.length)
    .replace(/^\s*\n?/, "")
    .trim();
  return {
    reasoning: text.slice(0, idx).trim(),
    answer: after,
  };
}

export async function cotWithDelimiter(userQuestion: string): Promise<{
  reasoning: string;
  answer: string;
}> {
  const completion = await client.chat.completions.create({
    model: "gpt-4.5",
    messages: [
      {
        role: "system",
        content: [
          "先用中文写出完整推理，短句为主，不要编造未给出的数据。",
          "推理结束后单独起一行，该行只含分隔标记 ---ANSWER---。",
          "从下一行起只写最终结论，不要重复推理过程。",
        ].join(""),
      },
      { role: "user", content: userQuestion },
    ],
    temperature: 0.2,
  });

  const raw = completion.choices[0]?.message?.content ?? "";
  return splitReasoningAndAnswer(raw);
}
```

第二段在支持 `JSON` 对象模式的模型上，用字段约束代替分隔符，解析更干净，记得在 `system` 里明确写出键名。

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
});

interface CotJsonPayload {
  reasoning: string;
  answer: string;
}

export async function cotWithJsonMode(
  userQuestion: string,
): Promise<CotJsonPayload> {
  const completion = await client.chat.completions.create({
    model: "gpt-4.5",
    messages: [
      {
        role: "system",
        content: [
          "只输出一个 JSON 对象，不要 Markdown，不要代码块。",
          "键名必须是 reasoning 与 answer，均为字符串。",
          "reasoning 用中文分步写清依据，answer 只写最终给用户的结论。",
        ].join(""),
      },
      { role: "user", content: userQuestion },
    ],
    temperature: 0.2,
    response_format: { type: "json_object" },
  });

  const raw = completion.choices[0]?.message?.content?.trim() ?? "{}";
  const parsed = JSON.parse(raw) as CotJsonPayload;
  if (
    typeof parsed.reasoning !== "string" ||
    typeof parsed.answer !== "string"
  ) {
    throw new Error("Invalid CoT JSON shape");
  }
  return parsed;
}
```

若你用的是带 `json_schema` 的接口，可以把字段类型和必填项写进模式里，由服务端做强校验，比单纯 `json_object` 更严。

## 小结

`CoT` 是单链推理形态里最容易落地的一块，几乎任何流水线都能先插一层试试效果。它的边界也很清楚，擅长可解释与可诊断，不擅长代替工具与外部事实，更不能保证链上每一步都真对。

需要同一层上并行试多条路、再系统比较优劣时，单链就不够用，下一篇 `24.ToT` 专文讲树形分叉与评估，再往后是 `25.GoT`。
