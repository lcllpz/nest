前面的 `LCEL` 内容中我们已经把组合语法讲得很完整，这一篇不再重复 `LCEL` 入门，而是专注回答一个更实战的问题，为什么我们已经会写 `pipe` 了，链路在生产里还是容易出问题。答案通常都在 `Runnable` 本身。

`Runnable` 是统一执行协议，`LCEL` 是组合表达方式。两者关系可以简单理解为，`LCEL` 负责描述流程，`Runnable` 负责真正执行流程。如果我们只会写组合而不理解执行语义，链路看起来能跑，上线后却容易在并发、流式、重试和可观测性上失控。

## 先和前文 LCEL 对齐边界

为了避免概念混淆，先把最常见的误区厘清：

- `LCEL` 不是运行时引擎，它主要解决声明式组合问题
- `Runnable` 不是某一个类，而是一套统一协议和行为约定
- `pipe` 写通不代表工程上可用，还要考虑重试、超时、回退、追踪
- 链路稳定性的核心不只是 prompt 质量，而是输入输出 shape 的确定性

前文 `LCEL` 解决的是怎么把流程拼起来，这一篇 `Runnable` 解决的是拼好的流程如何稳定执行。

## Runnable 的执行心智模型

可以把 `Runnable` 理解成统一的执行插槽。每个节点都遵循同一套输入输出约定，接收上游结果、产出下游可消费的数据。`Prompt`、`Model`、`Parser`、自定义函数虽然职责不同，但只要都实现 `Runnable` 协议，就能像积木一样稳定串接。

如下图所示。

![20260416152659](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260416152659.png)

图里展示的是最典型的串行执行链路，输入从左到右依次流经多个 `Runnable` 节点，最终得到输出。我们可以把它看成一条标准装配线，节点内部实现可以完全不同，但外部调用方式保持一致。

这个模型的关键价值是工程一致性。统一协议意味着我们可以在任意节点复用同一套能力，比如重试、超时、并发、流式输出、日志追踪，而不需要为不同组件分别实现一套调用逻辑。

很多文章会把 `invoke`、`ainvoke`、`batch`、`stream`、`transform` 写成 `API` 清单。更实用的理解方式是按目标选方法，而不是按名称背方法。

| 方法        | 典型目标         | 优先指标      | 常见误区                      |
| ----------- | ---------------- | ------------- | ----------------------------- |
| `invoke`    | 先跑通最短链     | 正确性        | 过早优化并发导致定位困难      |
| `ainvoke`   | 并发协调多个任务 | 资源利用率    | 把 `Promise.all` 当成容错方案 |
| `batch`     | 离线高吞吐处理   | 吞吐、成本    | 并发拉满触发限流              |
| `stream`    | 在线体验优化     | 首 token 延迟 | 中间节点把流聚合后再吐出      |
| `transform` | 流中间加工       | 增量处理精度  | 没有 chunk 边界意识           |

方法选择本质是目标函数选择。不同场景下我们要优化的指标不同，最优调用方式也就不同。下面先从最常用、最容易落地的 `invoke` 起步。

## 从零到一的 invoke 起步链

我们在项目里新增一条能力链时，建议总是先从 `invoke` 开始。这样可以最快确认提示词、模型、解析器、数据 shape 是否对齐。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个架构解释助手，请用简洁语言回答。"],
  ["human", "{question}"],
]);

const chain = prompt.pipe(model).pipe(new StringOutputParser());

async function main(): Promise<void> {
  const result = await chain.invoke({
    question: "Runnable 和 LCEL 的关系是什么",
  });
  console.log(result);
}

main().catch(console.error);
```

这段代码真正有价值的地方不是简短，而是它把最关键的 shape 链路固定住了。

## 服务端并发场景的 ainvoke

`ainvoke` 的重点不是让一次调用更快，而是让我们可以优雅地组织并发任务。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const summaryChain = ChatPromptTemplate.fromMessages([
  ["system", "请输出两句话摘要。"],
  ["human", "{text}"],
])
  .pipe(model)
  .pipe(new StringOutputParser());

const riskChain = ChatPromptTemplate.fromMessages([
  ["system", "请识别这段文本里的技术风险点，最多 3 条。"],
  ["human", "{text}"],
])
  .pipe(model)
  .pipe(new StringOutputParser());

async function main(): Promise<void> {
  const input = {
    text: "我们计划把多条业务链路统一到 Runnable，并在高峰期支持 stream 与 batch 共存。",
  };

  const [summary, risks] = await Promise.all([
    summaryChain.ainvoke(input),
    riskChain.ainvoke(input),
  ]);

  console.log("summary:", summary);
  console.log("risks:", risks);
}

main().catch(console.error);
```

实战建议是给并发任务加超时和降级策略，不要把 `Promise.all` 当作异常处理机制。

## batch 的吞吐优化与并发治理

`batch` 适合离线评估、批量改写、批量抽取。这个场景里我们追求的是单位时间处理量和稳定成功率。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const chain = ChatPromptTemplate.fromMessages([
  ["system", "请将中文翻译成英文，只输出结果。"],
  ["human", "{text}"],
])
  .pipe(model)
  .pipe(new StringOutputParser());

async function main(): Promise<void> {
  const inputs = [
    { text: "Runnable 让执行协议统一" },
    { text: "LCEL 让数据流组合更清晰" },
    { text: "batch 更关注吞吐" },
    { text: "stream 更关注体验" },
  ];

  const outputs = await chain.batch(inputs, { maxConcurrency: 2 });
  outputs.forEach((item, idx) => {
    console.log(`${idx + 1}. ${item}`);
  });
}

main().catch(console.error);
```

并发建议通常是从 1、2、4、8 逐步压测，不要直接跳到很高值。

## stream 的体验优化和断流排查

`stream` 的收益是首 token 更快返回，用户体感会显著提升，特别是答案较长时。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const chain = ChatPromptTemplate.fromMessages([
  ["system", "请按 3 段解释概念，每段一句话。"],
  ["human", "{topic}"],
])
  .pipe(model)
  .pipe(new StringOutputParser());

async function main(): Promise<void> {
  const stream = await chain.stream({
    topic: "为什么 Runnable 是 LangChain 的执行基础",
  });

  for await (const chunk of stream) {
    process.stdout.write(chunk);
  }
  console.log("\n--- stream done ---");
}

main().catch(console.error);
```

如果我们看到最后才一次性输出，优先检查中间节点是否做了全文拼接，比如某个 `RunnableLambda` 把所有 chunk 收集完才返回。

`stream` 成功与否通常取决于整条链是否保持增量语义，而不只取决于模型本身。

## transform 的进阶使用

`transform` 适合对流进行中间加工，比如边输出边标记、边输出边过滤。它的设计重点是 chunk 粒度与状态累积。

下面示例用 `RunnableLambda` 模拟了一个对文本流做前缀标记的场景。

```typescript
import { RunnableLambda } from "@langchain/core/runnables";

const markChunk = new RunnableLambda({
  func: async (input: string): Promise<string> => `[chunk] ${input}`,
});

async function main(): Promise<void> {
  const result = await markChunk.invoke("runnable-transform-demo");
  console.log(result);
}

main().catch(console.error);
```

真实项目里如果我们要操作流对象，请先保证每个 chunk 是自描述的，避免后续阶段无法还原上下文。

## Runnable 的工程增强层

我们在本地跑通链路后，下一步是补齐工程增强能力，重点是可观测性与韧性。

### withConfig 用于可观测性

在这段代码里，我们通过给链路加稳定的追踪元数据，把一次调用从能执行升级为可追踪、可定位、可复盘。重点看 `.withConfig()` 里的 `runName` 和 `tags`，这两个字段会直接影响我们在日志平台里的检索效率、聚合分析和问题回放速度。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const chain = ChatPromptTemplate.fromMessages([
  ["system", "你是一个可观测性助手。"],
  ["human", "{question}"],
])
  .pipe(model)
  .pipe(new StringOutputParser())
  .withConfig({
    runName: "runnable-observable-chain",
    tags: ["runnable", "observe", "v1"],
  });

async function main(): Promise<void> {
  const result = await chain.invoke({
    question: "如何给链路加追踪信息",
  });
  console.log(result);
}

main().catch(console.error);
```

如果我们在生产环境里统一约定 `runName` 命名规则和 `tags` 维度，比如按业务线、场景、版本打标，后续排查跨链路问题会快很多。反过来，如果每条链都随意命名，日志虽然有了，但可观测性价值会明显打折。

### withRetry 用于韧性控制

这一段示例展示的是在高失败概率节点上做定点重试，而不是全链路重试。核心看点是 `model.withRetry({ stopAfterAttempt: 3 })`，它明确了最大尝试次数，避免在异常风暴下把上游服务压垮。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const chain = ChatPromptTemplate.fromMessages([
  ["system", "请稳定回答。"],
  ["human", "{text}"],
])
  .pipe(model.withRetry({ stopAfterAttempt: 3 }))
  .pipe(new StringOutputParser());

async function main(): Promise<void> {
  const result = await chain.invoke({
    text: "解释一下为什么重试需要限制次数",
  });
  console.log(result);
}

main().catch(console.error);
```

生产实践中我们建议把重试策略放在高失败概率节点，不要全链无差别重试。除此之外，建议结合超时和熔断一起设计，避免重试把瞬时故障放大成系统性抖动。

## 上线前复盘和自查

为了避免清单式阅读负担，我们可以把常见问题收敛成三组上线前必查项：

- 执行语义是否闭环，输入输出 `shape` 是否逐层可验证，末尾是否有明确解析器
- 韧性策略是否闭环，高失败节点是否有重试上限、超时边界、失败隔离
- 可观测性是否闭环，核心链路是否配置 `runName`、`tags` 并记录业务上下文

如果这三组检查没有通过，线上常见故障通常会集中在四个方向：

- `stream` 链路被中间节点聚合，最终失去增量输出能力
- `batch` 并发设置过高，触发上游限流并放大失败率
- 只覆盖成功路径，故障时没有降级、兜底或熔断策略
- 业务逻辑全部压给模型，缺少 `RunnableLambda` 的可控处理层

## 总结

如果说前文 `LCEL` 教我们如何写出一条可读的数据流，这一篇 `Runnable` 讲的是如何把这条数据流变成可上线的执行系统。真正有经验的区别不在于我们会不会 `pipe`，而在于我们是否能稳定处理并发、流式、重试、观测和失败路径。

落到工程实践里，我们可以用一句话概括 `Runnable` 的价值，它是统一执行协议，`LCEL` 是组合表达方式，两者配合后，`Prompt`、`Model`、`Retriever`、`Parser` 和自定义函数都能进入同一执行模型，并共享 `invoke`、`ainvoke`、`batch`、`stream` 这套调用语义。

真正可上线的标准不是跑通一次，而是持续稳定。我们通常先用 `invoke` 固定最短链，再按目标切到 `stream` 或 `batch`，同时补齐 `withConfig`、`withRetry` 和失败路径治理，最后用上线前的三组检查确保链路可观测、可恢复、可演进。
