# LangChain 里怎么用 GoT 思维图

`GoT`（Graph of Thoughts，思维图）把每个中间结论看成图上的一个节点，边表示依赖、推导关系，也可以表示从多个节点合并出一个新结论。它不再要求是树，因为真实推理里经常出现合并、重用、交叉引用，树的分叉结构会把重复内容拷很多份，图则允许共享子结论。

前两篇分别讲了单链 `23.CoT` 和树形 `24.ToT`。本篇讲图形态解决什么问题、三类边怎么用、工程上主要难在哪里，以及在实际项目里怎么收敛落地范围。

## 树哪里不够用

树结构的硬约束是每个节点只能有一个父节点。一旦出现 A、B 两条推导线要在 C 汇合，树通常只剩两种写法：

- 复制一份上游内容，分别挂到两条分支里
- 强行选一个主父节点，另一条线改成文字补充

这两种写法都别扭，前者冗余，后者不直观。图结构更贴近真实推理过程，因为它能直接表达 C 同时依赖 A 和 B，也支持一个子结论被多个后继复用。以下任务用图会更自然：

- 长论证中的多线汇合，例如技术方案里安全、性能两条分析线最终汇到同一结论
- 多源信息综合，例如不同检索片段先各自压缩，再输入同一个聚合节点
- 带版本迭代的写作，例如节点 A 迭代成 A'，后续节点直接引用 A'

## 边上一般承载什么语义

不同实现里边的含义不完全统一，但固定一小套边类型会让解析和可视化轻松很多。常见几类如下。

- 推导边：从前提节点指向结论节点，表示由 A 得出 B。最常见，几乎所有节点之间都可以建一条。
- 合并边：多个节点共同指向一个聚合节点，表示综合或投票。例如来自不同角度的三段分析，最终汇入一个综合结论节点。
- 改进边：从旧版结论节点指向修订版节点，表示迭代而不是分叉。例如初稿节点 A 修订后变成 A'，后续节点引用 A' 而不是 A，但 A 仍保留在图里用于 diff 或回滚。
- 引用边：某个节点直接借用另一个节点的内容，不重新生成。语义上是我依赖你说的那件事，主要解决上下文复用，不产生新的推导。

在代码里固定 `EdgeKind` 枚举，把含义锁死，后面解析日志、做可视化、写一致性检查都会方便很多。不要在运行时让模型自由发明边类型，否则下游处理会一团糟。

理解图结构最快的方式是先画一个具体例子。假设任务是给一个新产品写上线检查报告，拆成以下几个节点。

A 和 B 各自生成，再合并进 C 与 D，C 和 D 同时输入最终报告节点。若用树，D 只能挂在 A 或 B 下，另一个的贡献只能复制过来。

## 生成节点的时机和顺序

图不一定要等所有前驱节点全部生成完才继续。常见两种推进策略。

拓扑顺序生成，先做拓扑排序，按依赖层级逐批生成，没有前驱的节点先做，所有前驱都就绪的节点再做。适合依赖关系固定、不会在中途动态新增节点的场景。

增量扩展，每生成一个节点后，由模型或规则判断是否要新增后续节点或新增一条边。灵活但难预测规模，必须配严格的预算。若每步都可能长出新节点，很容易在不知不觉间把图扩到上百个节点，成本和上下文窗口双超。

两种策略可以组合，初期用拓扑顺序把主干节点做出来，中期在若干关键节点上允许一次增量扩展，然后锁图转成单条 `CoT` 做最后综合。这样既有图的表达力，又不会让生成过程失控。

## 工程上的主要难点

状态空间比树更大。若没有强剪枝或强预算，节点和边会快速膨胀，上下文窗口和费用都吃不消。常见安全阀包括最大节点数、最大边数、以及单次扩展最多能新增几条边，三个数字写进配置并在每步检查。

一致性维护更难。合并两个节点时，若二者隐含前提冲突，图里会出现看似连通、实则矛盾的结构。如果你不做冲突检测，模型在读入多个前驱的上下文时会悄悄调和矛盾，给你一个措辞圆滑但实质偏的结论。检测手段有两类，一是在合并节点前专门跑一次前驱一致性检查的短提示，二是维护一个结构化的已承诺事实列表，每次生成新节点前先与列表比对。

可视化与调试成本高。链和树还能在日志里缩进打印，任意有向图往往需要专门工具或离线导出才能看清。推荐在开发阶段把图序列化成 `DOT` 格式，用 `Graphviz` 或在线渲染工具先确认结构，再交付代码。

因此常见落地策略是缩小范围，不要一上来就做全任务全图展开。例如只在规划层维护一张小图表示模块依赖，执行层仍用 `ReAct` 单链跑工具。又例如在离线审计里根据日志事后构图，在线路径仍保持简单。

## TypeScript 图数据结构与工具函数

下面把最小图模型扩充成带拓扑排序、环检测、一致性检查占位的版本，便于直接接到生成循环里。

````typescript
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { ChatSiliconFlow } from "./tot.ts";
import { z } from "zod";

export type EdgeKind = "derive" | "merge" | "revise" | "ref";

export interface GraphThoughtNode {
  id: string;
  content: string;
  meta?: Record<string, unknown>;
}

export interface GraphThoughtEdge {
  id: string;
  fromId: string;
  toId: string;
  kind: EdgeKind;
}

export interface ThoughtGraph {
  nodes: Map<string, GraphThoughtNode>;
  edges: GraphThoughtEdge[];
}

export interface GoTSearchConfig {
  maxNodes: number;
  maxEdges: number;
}

export function emptyGraph(): ThoughtGraph {
  return { nodes: new Map(), edges: [] };
}

export function predecessors(
  g: ThoughtGraph,
  nodeId: string,
): GraphThoughtNode[] {
  const ids = new Set(
    g.edges.filter((e) => e.toId === nodeId).map((e) => e.fromId),
  );
  return [...ids]
    .map((id) => g.nodes.get(id))
    .filter((n): n is GraphThoughtNode => n !== undefined);
}

export function successors(
  g: ThoughtGraph,
  nodeId: string,
): GraphThoughtNode[] {
  const ids = new Set(
    g.edges.filter((e) => e.fromId === nodeId).map((e) => e.toId),
  );
  return [...ids]
    .map((id) => g.nodes.get(id))
    .filter((n): n is GraphThoughtNode => n !== undefined);
}

export function topoSort(g: ThoughtGraph): string[] {
  const inDegree = new Map<string, number>();
  for (const id of g.nodes.keys()) inDegree.set(id, 0);
  for (const e of g.edges) {
    inDegree.set(e.toId, (inDegree.get(e.toId) ?? 0) + 1);
  }

  const queue: string[] = [];
  for (const [id, deg] of inDegree) {
    if (deg === 0) queue.push(id);
  }

  const result: string[] = [];
  while (queue.length > 0) {
    const curr = queue.shift()!;
    result.push(curr);
    for (const succ of successors(g, curr)) {
      const newDeg = (inDegree.get(succ.id) ?? 1) - 1;
      inDegree.set(succ.id, newDeg);
      if (newDeg === 0) queue.push(succ.id);
    }
  }

  if (result.length !== g.nodes.size) {
    throw new Error("ThoughtGraph contains a cycle");
  }
  return result;
}

export function addNode(g: ThoughtGraph, node: GraphThoughtNode): void {
  if (g.nodes.has(node.id)) {
    throw new Error(`Node ${node.id} already exists`);
  }
  g.nodes.set(node.id, node);
}

export function addEdge(g: ThoughtGraph, edge: GraphThoughtEdge): void {
  if (!g.nodes.has(edge.fromId)) {
    throw new Error(`Source node ${edge.fromId} not found`);
  }
  if (!g.nodes.has(edge.toId)) {
    throw new Error(`Target node ${edge.toId} not found`);
  }
  if (edge.fromId === edge.toId) {
    throw new Error(`Self-loop is not allowed: ${edge.fromId} -> ${edge.toId}`);
  }
  const duplicated = g.edges.some(
    (e) =>
      e.fromId === edge.fromId && e.toId === edge.toId && e.kind === edge.kind,
  );
  if (duplicated) {
    throw new Error(
      `Duplicated edge detected: ${edge.fromId} -> ${edge.toId} (${edge.kind})`,
    );
  }
  g.edges.push(edge);
}

export function toDot(g: ThoughtGraph): string {
  const lines: string[] = ["digraph ThoughtGraph {"];
  for (const [id, node] of g.nodes) {
    const label = node.content
      .slice(0, 36)
      .replace(/"/g, "'")
      .replace(/\n/g, " ");
    lines.push(`  "${id}" [label="${label}…"];`);
  }
  for (const e of g.edges) {
    lines.push(`  "${e.fromId}" -> "${e.toId}" [label="${e.kind}"];`);
  }
  lines.push("}");
  return lines.join("\n");
}

function assertBudget(g: ThoughtGraph, cfg: GoTSearchConfig): void {
  if (g.nodes.size > cfg.maxNodes) {
    throw new Error(`node budget exceeded: ${g.nodes.size} > ${cfg.maxNodes}`);
  }
  if (g.edges.length > cfg.maxEdges) {
    throw new Error(
      `edge budget exceeded: ${g.edges.length} > ${cfg.maxEdges}`,
    );
  }
}

function validateGraph(g: ThoughtGraph): void {
  for (const e of g.edges) {
    if (!g.nodes.has(e.fromId)) {
      throw new Error(`invalid edge ${e.id}: missing source ${e.fromId}`);
    }
    if (!g.nodes.has(e.toId)) {
      throw new Error(`invalid edge ${e.id}: missing target ${e.toId}`);
    }
    if (e.fromId === e.toId) {
      throw new Error(`invalid edge ${e.id}: self-loop on ${e.fromId}`);
    }
  }
}

let edgeSeq = 0;
function nextEdgeId(): string {
  edgeSeq += 1;
  return `e-${edgeSeq}`;
}

export function buildDemoPermissionGraph(rootQuestion: string): ThoughtGraph {
  edgeSeq = 0;
  const g = emptyGraph();
  addNode(g, { id: "root", content: rootQuestion });
  addNode(g, { id: "n-sec", content: "" });
  addNode(g, { id: "n-perf", content: "" });
  addNode(g, { id: "n-merge", content: "" });
  addNode(g, { id: "n-final", content: "" });

  addEdge(g, {
    id: nextEdgeId(),
    fromId: "root",
    toId: "n-sec",
    kind: "derive",
  });
  addEdge(g, {
    id: nextEdgeId(),
    fromId: "root",
    toId: "n-perf",
    kind: "derive",
  });
  addEdge(g, {
    id: nextEdgeId(),
    fromId: "n-sec",
    toId: "n-merge",
    kind: "merge",
  });
  addEdge(g, {
    id: nextEdgeId(),
    fromId: "n-perf",
    toId: "n-merge",
    kind: "merge",
  });
  addEdge(g, {
    id: nextEdgeId(),
    fromId: "n-merge",
    toId: "n-final",
    kind: "derive",
  });

  return g;
}

const nodeBodySchema = z.object({ content: z.string().min(1) }).strict();

const derivePrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    [
      "你是 GoT 推导节点。只输出 JSON，不要 markdown 代码块。",
      '格式 strictly: {{"content":"本步正文"}}',
    ].join("\n"),
  ],
  [
    "human",
    [
      "原始任务:\n{root_question}",
      "",
      "前置结论（单条）:\n{predecessor}",
      "",
      "本步写作要求:\n{instruction}",
    ].join("\n"),
  ],
]);

const mergePrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    [
      "你是 GoT 合并节点。把多条前驱合成一段综合结论，去重、对齐矛盾时写明取舍。只输出 JSON。",
      '格式 strictly: {{"content":"合并后的综合段落"}}',
    ].join("\n"),
  ],
  [
    "human",
    "原始任务:\n{root_question}\n\n下列前驱来自不同角度，请合并:\n{blocks}",
  ],
]);

function stripFences(s: string): string {
  let t = s.trim();
  if (!t.startsWith("```")) return t;
  t = t.replace(/^```[a-zA-Z0-9_-]*\s*/, "");
  t = t.replace(/\s*```\s*$/, "");
  return t.trim();
}

function pickJsonObject(raw: string): unknown {
  const clean = stripFences(raw);
  try {
    return JSON.parse(clean);
  } catch {
    const start = clean.indexOf("{");
    const end = clean.lastIndexOf("}");
    if (start < 0 || end < start) {
      throw new Error(`model output is not json: ${clean.slice(0, 120)}`);
    }
    const sliced = clean.slice(start, end + 1);
    return JSON.parse(sliced);
  }
}

async function invokeJsonNodeBody(
  model: ChatSiliconFlow,
  prompt: ChatPromptTemplate,
  vars: Record<string, string>,
): Promise<string> {
  const msgs = await prompt.formatMessages(vars);
  const res = await model.invoke(msgs, {
    response_format: { type: "json_object" },
  });
  const raw =
    typeof res.content === "string"
      ? res.content
      : JSON.stringify(res.content ?? "");
  const parsed = nodeBodySchema.parse(pickJsonObject(raw));
  return parsed.content.trim();
}

async function fillDerive(
  model: ChatSiliconFlow,
  rootQuestion: string,
  pred: GraphThoughtNode,
  instruction: string,
): Promise<string> {
  const text = await invokeJsonNodeBody(model, derivePrompt, {
    root_question: rootQuestion,
    predecessor: pred.content,
    instruction,
  });
  return text;
}

async function fillMerge(
  model: ChatSiliconFlow,
  rootQuestion: string,
  preds: GraphThoughtNode[],
): Promise<string> {
  const blocks = preds
    .map((p, i) => `--- 前驱 ${i + 1} (${p.id}) ---\n${p.content}`)
    .join("\n\n");
  const text = await invokeJsonNodeBody(model, mergePrompt, {
    root_question: rootQuestion,
    blocks,
  });
  return text;
}

function isMergeTarget(g: ThoughtGraph, nodeId: string): boolean {
  return g.edges.some((e) => e.toId === nodeId && e.kind === "merge");
}

export async function runGoTTopoFill(
  g: ThoughtGraph,
  rootQuestion: string,
  cfg: GoTSearchConfig,
  model: ChatSiliconFlow,
): Promise<void> {
  assertBudget(g, cfg);
  validateGraph(g);
  const order = topoSort(g);

  for (const id of order) {
    if (id === "root") continue;
    const node = g.nodes.get(id);
    if (!node) throw new Error(`missing node ${id}`);
    if (node.content.trim()) continue;

    const preds = predecessors(g, id);
    if (preds.length === 0) {
      throw new Error(`node ${id} has no predecessors`);
    }

    if (preds.length >= 2 || isMergeTarget(g, id)) {
      node.content = await fillMerge(model, rootQuestion, preds);
      continue;
    }

    const pred = preds[0]!;
    let instruction: string;
    if (id === "n-sec") {
      instruction =
        "从安全与权限风险角度写要点，markdown 短列表，不超过 5 条，每条一行以 - 开头";
    } else if (id === "n-perf") {
      instruction =
        "从性能与实现复杂度角度写要点，markdown 短列表，不超过 5 条，每条一行以 - 开头";
    } else if (id === "n-final") {
      instruction = "在综合结论之上，给出一句可执行的最终建议，单行，不要分点";
    } else {
      instruction = "承接前置结论，写出本节点应负责的下一层内容，保持简洁";
    }

    node.content = await fillDerive(model, rootQuestion, pred, instruction);
  }
}

async function main(): Promise<void> {
  const rootQuestion =
    "小型团队要给内部管理后台做权限模型（角色、资源、审计），请产出要点清单和一条可落地的最终建议。";

  const cfg: GoTSearchConfig = { maxNodes: 12, maxEdges: 20 };
  const g = buildDemoPermissionGraph(rootQuestion);
  assertBudget(g, cfg);
  validateGraph(g);

  const model = new ChatSiliconFlow({ verboseStream: false });

  console.log("[start] GoT topo fill, sort order:", topoSort(g).join(" → "));
  await runGoTTopoFill(g, rootQuestion, cfg, model);

  console.log("\n[dot]\n");
  console.log(toDot(g));

  const finalNode = g.nodes.get("n-final");
  console.log("\n[final n-final]\n");
  console.log(finalNode?.content ?? "(missing)");
}

main().catch((e) => {
  console.error("[fatal]", e);
  process.exitCode = 1;
});
````

到这里这份 `GoT` 示例已经可以直接跑通，终端里看到以下三段标记通常就说明主链路正常：

- `[start] GoT topo fill`，说明拓扑顺序计算成功，准备按依赖填充节点
- `[dot]`，说明图已经生成完成，并导出了可视化用的 `DOT` 文本
- `[final n-final]`，说明最终收敛节点已经产出，可以直接拿来做落地建议

这版代码还专门补了两类稳定性处理：

- 提示词里的示例 `JSON` 改成双花括号，避免 `ChatPromptTemplate` 把 `content` 误判成模板变量，触发 `INVALID_PROMPT_INPUT`
- 增加图结构校验和 `JSON` 解析兜底，模型偶发输出前后包裹文本时也能尽量恢复

如果你在同一个终端看到很长的深度思考日志，通常是 `tot.ts` 的树搜索调试输出，不是这份 `got.ts` 的核心结果。`got.ts` 的关键观察点还是上面那三段固定标记。

`toDot` 输出的 `DOT` 字符串可以直接粘到 [Graphviz online](https://dreampuf.github.io/GraphvizOnline/) 或 `d3-graphviz` 渲染。开发阶段先看图结构，再看节点文本，会比只盯终端日志更容易定位问题。

## 和 `LangGraph` 名字里的图是什么关系

`LangGraph` 里的图是控制流图，节点往往是函数或 `Agent` 步骤，边是运行时的跳转与条件分支。`GoT` 里的图是认知结构图，节点是中间命题或内容草稿，边是推导与合并关系。

两者可以共存，例如在 `LangGraph` 里有一个叫 `expand_graph` 的节点，它读取当前的 `ThoughtGraph` 状态，生成几个新节点，更新图，然后把更新后的图写回到 `LangGraph` 的 `State`，由后续节点继续读取。但不要混为一谈，控制流里的边和认知图里的边语义完全不同，变量命名上最好用不同前缀区分。

## 适合与不适合

适合子结论需要多处复用、或必须显式合并多条线索的任务，例如复杂技术方案的依赖拆解、多源信息综合、带版本迭代的写作。也适合需要人工介入审这条推导路径是否合理的场景，因为图结构天然可以在任意节点上加批注或标记待审。

不适合能一条链或浅树说完的问题，强加图只会增加维护成本。也不适合没有清晰节点粒度的流程，若一个节点里该装什么都说不清楚，图会先碎成噪声点，上下文质量反而比单链差。还不适合对延迟要求非常高的在线场景，拓扑排序、一致性检查、图序列化都是额外的计算和 `IO`，要酌情放到后台或异步链路里做。

## 小结

`GoT` 是三类推理形态里表达力最强的一种，主要收益是合并与共享，主要代价是一致性与可视化的复杂度。多数产品的现实路径是先把 `CoT` 与受控的 `ToT` 或 `30.LATS` 跑稳，再在明确需要合并或复用子结论的模块引入小图，而不是一上来就全任务建图。
