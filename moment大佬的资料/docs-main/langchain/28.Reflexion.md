Reflection 是让模型在生成一轮结果之后，再发起一次审查调用，对自己的输出做批评和改进的提示策略。相比直接回答，多一层复盘可以把上一轮的错漏变成下一轮的显式约束，代价是额外的推理成本，换来的是更高的输出质量，更适合知识密集型任务，而不是对延迟敏感的实时查询。

`Reflexion` 是 `reflection` 家族里的一个具体架构，强调让反思有据可查。它要求模型结合工具执行结果、外部检索内容，给出引用并列举多了什么、缺了什么，把批评落地到结构化要点上，再据此决定是否重试、如何重试。

## 典型流程

Reflexion 的执行节奏分四步。

1. `actor` 节点根据当前提示给出初始结果，不做额外自检。
2. `execute_tools` 节点真实执行搜索、代码运行、数据库查询，把环境反馈写回状态。
3. `revisor` 节点基于工具结果与过往反思记忆，给出改进建议或直接改写下一版答案。
4. 根据步数、外部评估结果或 `revisor` 判断，决定是结束还是回到步骤 1 继续。

![20260324112654](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260324112654.png)

不少实现会在循环上加一层策略：只有在模型置信度较低，或外部评估没通过时才触发反思与重试；任务足够简单时则直接跳过，以换取更低的延迟和成本。

## 与简单 Reflection 的差别

最简单的 `reflection` 只有两个节点：一个生成节点产出答案，一个反思节点扮演老师角色给出点评，按固定轮数循环。

![20260324112728](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260324112728.png)

这种写法实现简单，但反思步骤没有接触任何真实世界反馈，结果有时并不比第一版明显更好。Reflexion 的核心改进是让反思必须依赖工具调用结果或检索文档，强制给出引用、明确列举多余部分和缺失部分，使批评更具体，下一轮修改也更有方向。

## 什么时候用

Reflexion 适合质量要求高、又能接受多试几轮的任务，例如复杂代码生成、数学推理、长文档写作、需要多步验证的规划。在这些场景里，一次生成往往很难到位，多一轮反思相当于给下一轮喂更好的上下文，而不是用同一套提示反复重跑靠随机采样碰运气。

以下几类场景不太值得引入：

- 实时对话或对延迟极其敏感的一次性查询，多一轮反思就多一轮延迟。
- 任务本身没有清晰的好与不好标准，反思回合容易空转，应先把评估指标说清楚。

## 反思回合的设计

反思回合本质上是一个审稿人角色，接收刚才的输出和期望标准，指出问题并给出改进建议。设计时有三点值得注意。

- 输入要明确：除本轮产出外，最好同时传入任务描述、成功标准或评分规则，若有外部工具结果也一并贴进去，让反思建立在真实反馈之上。
- 输出要可操作：最好拆成缺失点、多余点、引用或证据这类结构，而不是泛泛的可以更好。
- 控制篇幅：反思内容会进入下一轮上下文，太长会挤占有效窗口，可要求模型用条目式短句总结，必要时限制条目数量。

## 完整示例

下面用一个 TypeScript 工具函数生成 Demo，完整走一遍 Reflexion 的实现。问题是实现 `deepEqual` 函数，处理基础类型、对象、数组、`Date`、`RegExp`、`Map`、`Set`、`NaN`、循环引用，这类问题一次生成往往遗漏边界情况，正好能看出每轮反思的价值。代码分四个文件。

### 输出结构定义

用 `zod` 定义两个 schema。`AnswerQuestionSchema` 是初稿结构，包含 `answer`（正文）、`reflection`（拆成 `missing` 和 `superfluous` 两个字段的自我批评）、`search_queries`（下一步检索词）。`ReviseAnswerSchema` 在此基础上扩展 `references`，用于存放修订稿的引用链接。

```typescript
import { z } from "zod";

const ReflectionSchema = z.object({
  missing: z.string().describe("当前回答缺失了什么"),
  superfluous: z.string().describe("当前回答哪些地方是多余的"),
});

export const AnswerQuestionSchema = z.object({
  answer: z.string().describe("详细回答，可含代码示例，约 300 字"),
  reflection: ReflectionSchema,
  search_queries: z
    .array(z.string())
    .min(1)
    .max(3)
    .describe("1～3 条用于改进回答的搜索词"),
});

export const ReviseAnswerSchema = AnswerQuestionSchema.extend({
  references: z.array(z.string()).describe("支撑修订内容的参考链接"),
});

export type AnswerQuestion = z.infer<typeof AnswerQuestionSchema>;
export type ReviseAnswer = z.infer<typeof ReviseAnswerSchema>;
```

`reflection` 字段强制模型在生成答案的同时写下自我批评，`search_queries` 直接输出下一步要搜索的关键词，两者共同驱动后续的工具执行节点。`ReviseAnswer` 新增的 `references` 则要求修订时必须注明信息来源，避免反思变成模型的自说自话。

### Chain 定义

`firstResponderChain` 和 `revisorChain` 共用同一套 `promptTemplate`，区别只在于 `instruction` 变量：初稿要求写出完整可运行代码，修订要求结合检索结果改进并附上参考链接。两者都通过 `bindTools` 把对应的 schema 绑定为工具，`tool_choice: "required"` 强制模型必须以结构化 `tool_call` 的形式输出，不能返回普通文本。

```typescript
import { tool } from "@langchain/core/tools";
import {
  ChatPromptTemplate,
  MessagesPlaceholder,
} from "@langchain/core/prompts";
import { ChatOpenAI } from "@langchain/openai";
import type { BaseMessage } from "@langchain/core/messages";
import { AnswerQuestionSchema, ReviseAnswerSchema } from "./schemas.js";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const answerTool = tool(async () => "", {
  name: "AnswerQuestion",
  description: "回答用户问题",
  schema: AnswerQuestionSchema,
});

const reviseTool = tool(async () => "", {
  name: "ReviseAnswer",
  description: "根据检索结果修订回答",
  schema: ReviseAnswerSchema,
});

const promptTemplate = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是一名资深 TypeScript 工程师。

{instruction}

回答后，严格反思：哪些边界情况被遗漏了？哪些内容是多余的？给出 1～3 条具体的搜索词用于改进。`,
  ],
  new MessagesPlaceholder({ variableName: "messages" }),
]);

const FIRST_INSTRUCTION =
  "用 TypeScript 给出完整、可直接运行的代码实现，附上关键注释，约 300 字。";

const REVISE_INSTRUCTION =
  "结合检索结果改进上一版回答：补充缺失的边界情况，删除冗余内容，保留完整可运行的代码，文末附参考链接。";

export async function firstResponderChain(state: {
  messages: BaseMessage[];
}): Promise<BaseMessage[]> {
  const result = await promptTemplate
    .pipe(model.bindTools([answerTool], { tool_choice: "required" }))
    .invoke({ messages: state.messages, instruction: FIRST_INSTRUCTION });
  return [result];
}

export async function revisorChain(state: {
  messages: BaseMessage[];
}): Promise<BaseMessage[]> {
  const result = await promptTemplate
    .pipe(model.bindTools([reviseTool], { tool_choice: "required" }))
    .invoke({ messages: state.messages, instruction: REVISE_INSTRUCTION });
  return [result];
}
```

### 工具执行节点

`executeTools` 从上一条 `AIMessage` 里解析出 `search_queries`，对每个词调用 `mockResearch` 生成模拟结果，打包成一条 `ToolMessage` 写回状态。`extractArgs` 是关键的兼容层：优先读 `message.tool_calls`（标准格式），若为空则尝试从 `additional_kwargs.tool_calls` 解析，还支持部分模型返回的 XML 自定义格式。替换为真实搜索 API 时，只需修改 `mockResearch` 函数。

```typescript
import { BaseMessage, AIMessage, ToolMessage } from "@langchain/core/messages";

export interface ParsedArgs {
  answer?: string;
  reflection?: { missing?: string; superfluous?: string };
  search_queries?: string[];
  references?: string[];
  [key: string]: unknown;
}

interface RawToolCall {
  id?: string;
  function?: { arguments?: string };
}

function mockResearch(query: string): Record<string, unknown> {
  return {
    query,
    results: [
      {
        title: `[参考] ${query}`,
        content: `关于 ${query} 的参考资料摘要。实际场景可替换为真实搜索 API。`,
        url: `https://example.com/search?q=${encodeURIComponent(query)}`,
      },
    ],
  };
}

export function parseToolCallArguments(raw: string): ParsedArgs {
  try {
    return JSON.parse(raw) as ParsedArgs;
  } catch {
    // 部分模型（如 GLM 系列）使用自定义 XML 格式，fallback 解析
  }

  const result: Record<string, unknown> = {};
  const keys = [...raw.matchAll(/<arg_key>(.*?)<\/arg_key>/gs)].map(
    (m) => m[1],
  );
  const values = [...raw.matchAll(/<arg_value>(.*?)<\/arg_value>/gs)].map(
    (m) => m[1],
  );

  for (let i = 0; i < keys.length; i++) {
    const key = keys[i]?.trim();
    if (!key) continue;
    const val = values[i] ?? "";
    try {
      result[key] = JSON.parse(val);
    } catch {
      result[key] = val;
    }
  }

  return result as ParsedArgs;
}

export function extractArgs(message: AIMessage): ParsedArgs {
  if (message.tool_calls?.length) {
    return (message.tool_calls[0].args ?? {}) as ParsedArgs;
  }
  const raw = (message.additional_kwargs?.tool_calls ?? []) as RawToolCall[];
  if (raw.length && raw[0].function?.arguments) {
    return parseToolCallArguments(raw[0].function.arguments);
  }
  return {};
}

export async function executeTools(state: {
  messages: BaseMessage[];
}): Promise<ToolMessage[]> {
  const last = state.messages[state.messages.length - 1];
  if (!(last instanceof AIMessage)) return [];

  const args = extractArgs(last);
  const queries = args.search_queries ?? [];
  if (!queries.length) return [];

  const toolCallId =
    last.tool_calls?.[0]?.id ??
    (last.additional_kwargs?.tool_calls as RawToolCall[] | undefined)?.[0]
      ?.id ??
    "mock_id";

  const results = Object.fromEntries(queries.map((q) => [q, mockResearch(q)]));

  return [
    new ToolMessage({
      content: JSON.stringify(results),
      tool_call_id: toolCallId,
    }),
  ];
}
```

### 图结构与入口

图的结构是 `First_Draft → Execute_Tools → Revise → 条件判断`。状态里用 `revisionCount` 累加计数，每次 `Revise` 节点执行时加 1，达到 `MAX_ITERATIONS` 就结束。这比用 `ToolMessage` 数量计数更可靠，因为当 `Execute_Tools` 返回空结果时，`ToolMessage` 计数不会增长，会导致无限循环。`printArgs` 在每个节点完成后打印本轮的反思摘要，让执行过程可观测。

```typescript
import { BaseMessage, HumanMessage, AIMessage } from "@langchain/core/messages";
import { Annotation, END, START, StateGraph } from "@langchain/langgraph";
import { firstResponderChain, revisorChain } from "./chains.js";
import { executeTools, extractArgs } from "./tool_executor.js";

const MAX_ITERATIONS = 2;

const ReflexionState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (curr, next) => (curr ?? []).concat(next ?? []),
  }),
  revisionCount: Annotation<number>({
    reducer: (curr, next) => (curr ?? 0) + (next ?? 0),
    default: () => 0,
  }),
});

type ReflexionStateType = typeof ReflexionState.State;

const graph = new StateGraph(ReflexionState)
  .addNode("First_Draft", async (state: ReflexionStateType) => ({
    messages: await firstResponderChain(state),
  }))
  .addNode("Execute_Tools", async (state: ReflexionStateType) => ({
    messages: await executeTools(state),
  }))
  .addNode("Revise", async (state: ReflexionStateType) => ({
    messages: await revisorChain(state),
    revisionCount: 1,
  }))
  .addEdge(START, "First_Draft")
  .addEdge("First_Draft", "Execute_Tools")
  .addEdge("Execute_Tools", "Revise")
  .addConditionalEdges(
    "Revise",
    (state: ReflexionStateType) =>
      state.revisionCount >= MAX_ITERATIONS ? END : "Execute_Tools",
    ["Execute_Tools", END],
  )
  .compile();

function printArgs(messages: BaseMessage[]) {
  const last = messages[messages.length - 1];
  if (!(last instanceof AIMessage)) return;
  const args = extractArgs(last);
  if (args.reflection?.missing)
    console.log(`  缺失：${args.reflection.missing}`);
  if (args.reflection?.superfluous)
    console.log(`  冗余：${args.reflection.superfluous}`);
  if (args.search_queries?.length)
    console.log(`  搜索词：${args.search_queries.join("、")}`);
  if (args.references?.length)
    console.log(`  参考：${args.references.join(" | ")}`);
}

async function main() {
  const query =
    "用 TypeScript 实现一个 deepEqual 函数，比较两个任意值是否深度相等，" +
    "需要处理：基础类型、对象、数组、Date、RegExp、Map、Set、NaN、循环引用。" +
    "给出完整可运行的带类型注解的代码。";

  console.log(`问题: ${query}\n`);
  console.log("─".repeat(60));

  let allMessages: BaseMessage[] = [new HumanMessage(query)];
  let revisions = 0;

  for await (const chunk of await graph.stream(
    { messages: allMessages },
    { recursionLimit: 20, streamMode: "updates" as const },
  )) {
    const [nodeName, update] = Object.entries(chunk)[0] as [
      string,
      { messages?: BaseMessage[]; revisionCount?: number },
    ];
    const newMessages = update.messages ?? [];
    allMessages = allMessages.concat(newMessages);

    switch (nodeName) {
      case "First_Draft":
        console.log("\n[First_Draft] 初稿生成完成");
        printArgs(allMessages);
        break;
      case "Execute_Tools":
        console.log(
          `\n[Execute_Tools] 检索完成，返回 ${newMessages.length} 条工具结果`,
        );
        break;
      case "Revise":
        revisions += 1;
        console.log(`\n[Revise] 第 ${revisions} 轮修订完成`);
        printArgs(allMessages);
        break;
    }
  }

  const last = allMessages[allMessages.length - 1] as AIMessage;
  const args = extractArgs(last);
  const answer =
    args.answer ?? (typeof last?.content === "string" ? last.content : "");

  console.log("\n" + "─".repeat(60));
  console.log("最终答案:\n");
  console.log(answer || "未能解析最终答案");
}

main().catch(console.error);
```

运行后，终端输出大致如下。可以看到每轮 `[Revise]` 都给出了具体的缺失与冗余反馈，最终答案比初稿明显更完整。

## 和其他模式的关系

Reflexion 可以嵌套在 `Subagent` 内部：主 `Agent` 把写代码交给 `Subagent`，`Subagent` 内部采用 Reflexion 先产出再反思，把相对满意的一版交回主 `Agent`，既保留了分工清晰的结构，又在子任务内部拉高了单次产出的质量。

在整个 `reflection` 家族里，Reflexion 处于中间位置。

- 轻量任务：简单的生成加反思就够用，先生成一版，再用另一个节点点评，按固定轮数或简单条件结束。
- 激进探索：可以考虑 `Language Agent Tree Search`（`LATS`），把反思与蒙特卡洛树搜索结合，在多条候选轨迹之间做决策。

![20260324142451](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260324142451.png)

`LATS` 把推理、规划和反思统一进一套搜索框架，选择时用类似 `UCT` 的公式在当前期望奖励与探索新路径之间做平衡，适合代码生成等需要多分支探索的任务，但对打分和奖励设计比较敏感。

三种架构从简到繁：简单 `reflection` 只做自我点评、无外部反馈；Reflexion 引入工具结果和结构化批评；`LATS` 进一步加入树状搜索与奖励回传，在有高质量环境反馈时能更好地区分不同轨迹并选出最优路径。

## 小结

Reflexion 的核心是一个带记忆的执行、反思、改进闭环，而不是单纯的多轮对话。它和普通的多问几次的本质区别在于：每一轮的反思内容是结构化的、有据可查的，下一轮拿到的是明确的缺什么和改什么，而不是泛泛的还可以更好之类点评。

从这一节的实现来看，有几个地方值得记住。

第一，结构化输出是 Reflexion 的基础。把 `answer`、`reflection`、`search_queries` 用 schema 约束下来，才能保证反思内容真正进入后续节点，而不是藏在模型的自由文本里谁也用不上。

第二，终止条件要显式管理。用 `ToolMessage` 计数来判断是否结束，在工具返回空结果时会无限循环。在状态里单独维护 `revisionCount` 是更可靠的做法，循环次数一目了然，也方便后续扩展（例如加一条外部评分通过就提前终止的条件）。

第三，工具执行层要做兼容。不同模型（包括通过 OpenAI 兼容接口调用的第三方模型）返回 `tool_calls` 的格式不完全一致，有些在 `message.tool_calls`，有些在 `additional_kwargs.tool_calls`，还有个别模型用 XML 格式。`extractArgs` 把这些差异屏蔽掉，上层代码不用关心底层格式。

第四，`mockResearch` 替换为真实 API 时，整体结构不需要改动。把 `executeTools` 里的 `mockResearch` 换成真实的搜索调用（比如 Tavily、Bing Search、自建向量库检索），就能让反思建立在真实外部信息上，而不是模型自身的记忆。

最后，把每轮的生成、反思、工具结果整条轨迹存下来，是 Reflexion 比较容易被忽视的价值：这些轨迹本身就是高质量的示例数据，可以用来微调模型、构建 `few-shot` 提示，或者直接复用到相似任务里，让系统越用越稳。
