# LangChain 里 Agent 怎么思考，推理形态与运行时编排一张图说清

前面几章更多是在讲 Agent 放在哪里跑、挂什么工具、如何拆成多 Agent 协作。除了这些结构层面的问题，还有一条容易被忽略的维度，就是 Agent 在内部到底怎么思考，它是一步到位直接给答案，还是显式地先规划、再行动、做完了再复盘。

可以把相关讨论拆成两层，读起来会清晰很多。

第一层是推理在上下文里长什么样，也就是中间结论要不要写出来、要不要在多条候选之间比较、这些结论之间是链状、树状还是更一般的图状。论文里常见的名字包括 `CoT`、`ToT`、`GoT`，它们描述的是搜索空间与中间结果的组织方式。

第二层是运行时怎么编排，也就是谁来调用工具、什么时候重试、谁来当裁判。`ReAct`、`Plan-and-Execute`、`Reflexion`、`Self-Critic` 以及后面的 `LATS` 等，都属于这一层。两层经常叠在一起用，例如 `ReAct` 里每一轮的 `Thought` 可以看成一段很短的 `CoT`，而需要多路径尝试时又会落到树搜索一类机制上。

这类思考模式本质上是对 `LLM` 推理流程的编排方式，决定了 Agent 处理复杂任务时的稳定性与可控性。本章先做总览，把第一层的三类经典形态和第二层的几种常见编排都摆到同一张地图上。链、树、图三种形态各有专文展开（`23.CoT`、`24.ToT`、`25.GoT`），运行时编排从 `26.ReAct` 起逐篇细讲。

## 为什么需要显式的思考模式

最简单的调用方式是用户发问，模型调用一次，返回文本。这在简单问答里够用，但一旦任务需要多步工具调用、需要根据中间结果调整策略，甚至需要从失败中学习，单次推理就很容易跑偏或卡死。显式的思考模式要做的事情，就是把推理和执行拆成明确的步骤，让规划、行动、复盘各自成为流程里的一环，使整个过程可观测、可干预、可复现。

只依赖模型在单次生成里默默想完往往不够。把中间结构写出来或拆成多步，既方便人查日志，也方便加规则、加预算、在关键节点插审核。

## ReAct 边想边做

`ReAct`（Reasoning + Acting）让 Agent 在一个固定循环里运转：先根据当前信息做一步推理（`Thought`），再决定调用哪个工具、传什么参数（`Action`），执行完后把工具返回的结果当作观察（`Observation`），再进入下一轮 `Thought`，直到模型认为可以给出最终答案为止。

典型流程如下。

![20260320181928](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320181928.png)

适合需要频繁查数据、计算、调 `API` 的任务，比如查完天气再推荐穿衣、先搜索再汇总这类场景。灵活是它最大的优点，模型可以根据每一步的观察结果随时调整后续动作。它和 `CoT` 的天然连接在于，每一轮 `Thought` 就是一小段可见推理。若任务步骤很多、依赖关系复杂，单靠局部推理有时会漏步骤或重复执行，这时往往需要和先规划再执行的模式配合使用，或在更外层引入树搜索一类机制。更细的约定与消息形态见 `26.ReAct` 专文。

## Plan-and-Execute 先规划后执行

`Plan-and-Execute` 把流程拆成两段：先让模型根据用户目标列出一份完整的步骤列表（`Plan`），再按顺序逐条执行（`Execute`）。每步的执行结果会作为上下文传给后续步骤，必要时也可以让模型根据执行情况回头调整计划。

典型流程如下。

![20260320181948](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320181948.png)

适合任务结构清晰、步骤数大致可预见的场景，比如写一份报告，先查资料、再列提纲、再写各节、最后润色。整体路径清晰，便于做进度展示和断点续跑。计划一旦偏差或环境变化较大，可能需要引入重规划机制，或者与 `ReAct` 结合，在单步内再做局部推理。计划阶段本身也可以要求模型输出链式依据，相当于在 `Plan` 这一刀上叠一层 `CoT`。

## Reflexion 执行后复盘重试

`Reflexion` 在任务跑完一轮之后，多留一步反思（`Reflection`）：让模型根据最终结果或中间失败，总结哪里出了问题、下次该怎么改，并把这段反思存下来作为经验，用于下一轮重试或后续类似任务。Agent 不再只是执行完就结束，而是显式地从每一次失败里积累经验。

典型流程如下。

![20260320182010](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320182010.png)

适合允许重试、且希望同类错误不要反复出现的场景，比如多轮调试、自动修 Bug、反复执行直到满足条件的任务。代价是多一轮甚至多轮执行加反思的调用，成本和延迟都会上升，一般会对重试次数或反思深度设置上限。它和 `ToT`、`LATS` 的共同点是要对哪条路更好形成判断，只是 `Reflexion` 更偏单轨迹事后改，而不是同步维护多分支搜索树。

## Self-Critic 内部裁判审核

`Self-Critic` 在关键输出节点引入一个裁判角色：可以是同一个模型换一套 `prompt` 做自检，也可以是专门的小模型或规则引擎，对当前步的输出做审核、打分或安全检查。只有通过检查才进入下一步，否则要求模型重新生成或走降级逻辑。

适合对安全性、格式、合规性要求高的环节，比如对外回复前的敏感信息过滤、代码生成后的语法与规范检查。它的优点是能明显提高输出的稳定性和可控性，缺点是多一次甚至多次模型调用，且裁判本身的设计（打分标准、通过阈值）需要根据业务情况持续调优。树搜索类方法里常见的节点打分有时就会用同一套 `Self-Critic` 思路来实现，只是调用频率和上下文组织不同。

## 小结

在实际工程里，这些模式往往是组合使用的。比较常见的做法是先用 `Plan-and-Execute` 把大任务拆成若干步骤，再在每一步内部用 `ReAct` 去调工具、查数据，对关键步骤挂一层 `Self-Critic` 做审核，对允许重试的流程在结束后加一层 `Reflexion` 做复盘。若问题需要多路径探索，再在子问题或全局外层引入 `LATS` 或受控的 `ToT` 式分叉，而不是一上来就全图展开。

推理形态这一侧，`CoT` 适合作为默认的把话说清楚手段，嵌在计划、工具前推理、反思文字里。`ToT` 与 `GoT` 适合在明确需要比较多条中间路线、或需要合并子结论时使用，但要配好预算与评估，避免分叉失控。
