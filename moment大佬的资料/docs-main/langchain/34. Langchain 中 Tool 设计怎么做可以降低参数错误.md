# LangChain 中 Tool 设计如何降低参数错误

在 `Agent` 系统里，`Tool` 参数错误几乎是最常见的故障来源。模型本身会推理，但它对参数边界、业务约束、线上风险并不天然敏感。想把出错率压下去，关键不是多写提示词，而是把 `Tool` 设计成一个可收敛、可验证、可恢复的调用接口。

这篇文章会围绕一个核心目标展开，如何让模型在复杂任务里也能稳定传对参数，并在失败时快速自修复，不把问题放大成链路故障。

## 参数错误为什么高发

先看几个典型场景：

- 参数字段过多，模型在多轮推理后容易漏填或错填
- 字段语义不清，比如 `type`、`mode` 这类抽象命名，模型难以稳定映射
- 缺少强校验，错误参数直接进下游接口，导致更隐蔽的业务异常
- 失败响应不可修复，只返回调用失败，模型不知道该改哪里
- 生产保障缺失，没有幂等、超时、重试策略，单次错误演变成系统抖动

所以问题的本质不是模型能力不够，而是接口契约太松、失败路径设计太弱。

参数错误高发本质是接口设计问题而不是模型能力问题，想降低错误率就要先收敛输入契约，再设计失败恢复路径。

## 一层收敛 先把输入空间压小

降低参数错误的第一步是 `schema` 收敛，也就是减少模型可犯错的自由度。

实践原则可以归纳为四条：

- 字段少，能由系统推导的就不要让模型传
- 类型硬约束，`string`、`number`、`boolean` 明确，不用模糊结构
- 枚举有限，业务状态尽量用 `enum`，不要让模型自由生成
- 短参数优先，长文本放上下文，不要塞进结构化参数

下面是一个更稳妥的 `Tool` 入参示例。这个示例使用 `zod` 收敛字段，并通过描述把模型引导到正确输入区间。

```typescript
import { z } from "zod";

export const CreateTicketInputSchema = z.object({
  title: z.string().min(5).max(80).describe("工单标题，简短明确"),
  priority: z.enum(["low", "medium", "high"]).describe("优先级枚举值"),
  category: z.enum(["billing", "tech", "account"]).describe("问题分类"),
  userId: z.string().uuid().describe("用户唯一标识 UUID"),
});

export type CreateTicketInput = z.infer<typeof CreateTicketInputSchema>;
```

这段设计的重点不是语法，而是输入边界被固定住了。模型只能在有限空间里选值，错误概率会显著下降。上线前建议把每个字段再过一遍，问自己一句，这个字段是否真的必须由模型提供。

`schema` 收敛是在源头减少错误空间，而字段最小化、类型明确化、枚举化是最有效的三件事。

## 二层验证 在 Tool 侧做严格拦截

只做 `schema` 还不够，`Tool` 执行前必须做强校验，保证错误不会穿透到外部系统。

推荐做法是把校验拆成两段：

- 结构校验，判断类型、必填、长度、枚举是否合法
- 业务校验，判断数据是否存在、状态是否允许、权限是否匹配

示例代码如下。这里刻意把错误信息结构化，目的是让模型能看懂并执行下一步修复。

```typescript
type ToolValidationError = {
  code: "VALIDATION_ERROR";
  field: string;
  reason: string;
  expected: string;
  retryable: true;
};

function toValidationError(
  field: string,
  reason: string,
  expected: string,
): ToolValidationError {
  return {
    code: "VALIDATION_ERROR",
    field,
    reason,
    expected,
    retryable: true,
  };
}

export function validateCreateTicketInput(raw: unknown) {
  const parsed = CreateTicketInputSchema.safeParse(raw);
  if (!parsed.success) {
    const issue = parsed.error.issues[0];
    return {
      ok: false as const,
      error: toValidationError(
        issue.path.join("."),
        issue.message,
        "请修正该字段后重试，不要改动其他字段",
      ),
    };
  }
  return { ok: true as const, data: parsed.data };
}
```

这里的关键点是可修复。如果错误返回只写 `invalid input`，模型通常会整体重写参数，甚至改掉本来正确的字段。结构化错误会把修复动作约束到单一字段，稳定性明显更好。

校验必须分层，因为结构合法不代表业务合法，同时错误返回必须可修复，这决定了模型能否稳定进行二次纠错。

## 三层恢复 失败后让模型只修参数

很多团队已经有校验，但依然故障频发，原因是失败后的策略不完整。正确做法是设计一条三层恢复链路：

- 第一次失败，返回字段级错误，让模型只改参数
- 第二次失败，降级到安全模板或半自动流程
- 连续失败，转人工兜底并记录样本，进入后续优化闭环

可以把这条链路理解成自动纠错优先、风险扩散受限。下面这段流程图用于描述推荐路径。

这个流程的价值在于每次失败都能在本层被吸收，不会直接冲击数据库、支付、发消息等高风险下游系统。

恢复链路的目标是把失败局部化并阻止问题扩散到下游，执行策略应当是自动修复优先、降级次之、人工兜底保底。

## 生产环境必须补齐的四个保障

如果要真正上线，除了参数校验，还要补齐工程层面的保护网：

- 幂等键，同一次业务请求只能生效一次，避免重试造成重复写入
- 超时控制，每个 `Tool` 有明确超时，避免调用卡死拖垮链路
- 重试策略，仅对可重试错误做指数退避，避免无脑重试放大故障
- 回滚机制，跨系统写入失败时可以补偿，保证数据一致性

下面给一个简化执行器示例，展示如何把这些策略放进同一层：

```typescript
interface ToolCallContext {
  requestId: string;
  timeoutMs: number;
  idempotencyKey: string;
}

async function executeWithGuard<T>(
  ctx: ToolCallContext,
  fn: () => Promise<T>,
): Promise<T> {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), ctx.timeoutMs);

  try {
    // 这里可接入幂等层缓存或数据库去重
    const result = await fn();
    return result;
  } finally {
    clearTimeout(timer);
  }
}
```

这段代码并不完整，但体现了设计方向，参数正确只是起点，线上稳定取决于完整的执行保护。

生产稳定性来自幂等、超时、重试、回滚的协同，这些保障应统一放到执行器层而不要散落在各个 `Tool` 中。

## 具体代码实现 一个可落地的 LangChain Tool 防错方案

下面给出一套可以直接落地的示例实现。这个实现覆盖了参数收敛、校验、重试、降级和可观测日志，适合作为项目模板起点。

```typescript
import { tool } from "@langchain/core/tools";
import { z } from "zod";

const CreateTicketInputSchema = z.object({
  title: z.string().min(5).max(80),
  priority: z.enum(["low", "medium", "high"]),
  category: z.enum(["billing", "tech", "account"]),
  userId: z.string().uuid(),
  requestId: z.string().min(8),
});

type CreateTicketInput = z.infer<typeof CreateTicketInputSchema>;

type ToolErrorCode =
  | "VALIDATION_ERROR"
  | "BUSINESS_ERROR"
  | "TIMEOUT_ERROR"
  | "DOWNSTREAM_ERROR";

type ToolError = {
  code: ToolErrorCode;
  message: string;
  retryable: boolean;
  field?: string;
  expected?: string;
};

type ToolResult =
  | { ok: true; data: { ticketId: string; status: "created" } }
  | { ok: false; error: ToolError };

class IdempotencyStore {
  private readonly done = new Set<string>();

  has(key: string): boolean {
    return this.done.has(key);
  }

  set(key: string): void {
    this.done.add(key);
  }
}

const idempotencyStore = new IdempotencyStore();

function sleep(ms: number): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

async function withRetry<T>(
  task: () => Promise<T>,
  maxRetries: number,
  baseDelayMs: number,
): Promise<T> {
  let lastError: unknown;
  for (let i = 0; i <= maxRetries; i += 1) {
    try {
      return await task();
    } catch (error) {
      lastError = error;
      if (i === maxRetries) break;
      const delay = baseDelayMs * 2 ** i;
      await sleep(delay);
    }
  }
  throw lastError;
}

async function createTicketInDownstream(
  input: CreateTicketInput,
): Promise<{ ticketId: string }> {
  // 模拟下游接口调用
  await sleep(80);
  if (input.category === "billing" && input.priority === "high") {
    // 模拟瞬时失败，重试后可恢复
    throw new Error("temporary downstream 503");
  }
  return { ticketId: `TICKET-${Date.now()}` };
}

function validateBusiness(input: CreateTicketInput): ToolResult | null {
  if (input.title.includes("测试测试测试")) {
    return {
      ok: false,
      error: {
        code: "BUSINESS_ERROR",
        message: "title 含有重复无意义文本",
        retryable: true,
        field: "title",
        expected: "请使用真实问题描述，并保持 5-80 字",
      },
    };
  }
  return null;
}

async function executeCreateTicket(
  input: CreateTicketInput,
): Promise<ToolResult> {
  if (idempotencyStore.has(input.requestId)) {
    return {
      ok: true,
      data: { ticketId: `REUSED-${input.requestId}`, status: "created" },
    };
  }

  const businessError = validateBusiness(input);
  if (businessError) return businessError;

  const timeoutMs = 1200;
  const timeoutPromise = new Promise<never>((_, reject) => {
    setTimeout(() => reject(new Error("timeout")), timeoutMs);
  });

  try {
    const result = await Promise.race([
      withRetry(() => createTicketInDownstream(input), 2, 100),
      timeoutPromise,
    ]);
    idempotencyStore.set(input.requestId);
    return {
      ok: true,
      data: { ticketId: result.ticketId, status: "created" },
    };
  } catch (error) {
    const message = error instanceof Error ? error.message : "unknown error";
    if (message === "timeout") {
      return {
        ok: false,
        error: {
          code: "TIMEOUT_ERROR",
          message: "下游调用超时",
          retryable: true,
          expected: "请稍后重试，保持原参数不变",
        },
      };
    }
    return {
      ok: false,
      error: {
        code: "DOWNSTREAM_ERROR",
        message: message,
        retryable: true,
        expected: "请只重试，不要修改已通过校验的字段",
      },
    };
  }
}

export const createTicketTool = tool(
  async (rawInput: unknown): Promise<string> => {
    const parsed = CreateTicketInputSchema.safeParse(rawInput);
    if (!parsed.success) {
      const issue = parsed.error.issues[0];
      return JSON.stringify({
        ok: false,
        error: {
          code: "VALIDATION_ERROR",
          message: issue.message,
          retryable: true,
          field: issue.path.join("."),
          expected: "请仅修正该字段，不要重写全部参数",
        } satisfies ToolError,
      });
    }

    const result = await executeCreateTicket(parsed.data);
    return JSON.stringify(result);
  },
  {
    name: "create_ticket",
    description: "创建客服工单，失败时返回可修复错误",
    schema: CreateTicketInputSchema,
  },
);
```

这段代码可以直接作为最小可运行模板。你只需要替换 `createTicketInDownstream` 为真实业务调用，就能在项目里快速落地。

如果你希望让模型严格执行只修字段、不改思路，可以给 `Agent` 增加如下失败修复提示：

```typescript
const repairPrompt = `
当 Tool 返回 error.code 为 VALIDATION_ERROR 或 BUSINESS_ERROR 时：
1. 只修改 error.field 对应参数
2. 不改动其他已正确字段
3. 重试时保留同一个 requestId
`;
```

这个提示的作用是把修复动作显式化。模型在二次调用时会更稳定，不容易把正确参数一起改坏。

完整落地需要把 `schema`、执行器、重试策略、错误协议放在一套实现里，而代码里最关键的是返回结构化错误来驱动模型做最小修改。

## 总结

把 `LangChain Tool` 设计好，关键不是写更多提示词，而是搭建一条完整防线：

- 用 `schema` 收敛输入，降低犯错空间
- 用双层校验阻断错误，返回可修复信息
- 用三层恢复控制失败扩散，保护下游系统
- 用幂等、超时、重试、回滚保障线上稳定

上线前我们可以把这四层防线当成统一检查项，从参数最小化、类型约束、字段级修复到失败样本回收逐项自查，避免多人协作时出现漏项。很多线上问题并不是模型能力不够，而是约束、修复、恢复、观测四层防线没有同时到位。只要把这四步做扎实，参数错误就会从高频线上故障变成可管理的常规异常。
