前两篇我们把 `Memory` 的类型选型和多用户隔离讲清楚了，再往下会遇到另一类问题，会话一长，上下文就装不下。模型的 `context window` 是硬约束，哪怕是 `128k` 的模型，真实项目里也经常碰到上下文超限或首 `token` 延迟显著变大的情况。最直觉的做法是粗暴截断只留最近几轮，但这样一剪刀下去，用户之前说过的名字、订单号、偏好设置全都丢了，模型回答立刻变得驴唇不对马嘴。

这一篇围绕一个具体问题展开，长对话场景下怎样把历史压小，又不把关键信息一起压没。结论可以先给出来，好的压缩一定是分层的，对话流、事实、引用要分别走不同的处理路径，而不是指望一个摘要把所有东西一股脑打包。

## 一把全砍掉为什么不行

最常见的错误就是简单保留最近 `k` 轮，比如只留最新 10 轮原文。这种方式在某些场景确实够用，比如工具类对话、主题切换频繁的搜索类产品，上下文本来就很短。但只要会话里出现下面几种情况里的任何一种，它就会立刻失效。

- 用户在会话开头自报身份、偏好、目标。这类信息出现得早，后面几十轮可能都用不到，但只要后面一问具体问题，它立刻又是关键输入，截断策略一上来就把这段剪掉了。

- 多轮任务里有跨多轮的稳定状态。订单号、会议时间、系统配置这种强事实往往只在某一轮说过一次，后面每一轮都得基于它回答，按时间裁剪很容易在某个轮次后把这条线索彻底丢掉。

- 用户提过某个引用或文档片段。后续提问全都建立在这段引用之上，比如他先贴了一段合同条款再连问五个问题，一旦原文被裁掉，模型就只能靠猜，回答的底子就没了。

一刀切的截断方案没能力区分这些强事实和普通闲聊，丢掉的很可能正好是后面要用的关键信息。压缩的第一性原则是，必须按信息性质分类处理，不能按时间统一裁剪。

## 分层处理的三条管道

实践里被反复验证有效的结构是下面这三条并行管道，分别处理不同性质的历史。

| 管道     | 处理对象         | 存储形态         | 更新频率         |
| -------- | ---------------- | ---------------- | ---------------- |
| 近期窗口 | 最近 `k` 轮原文  | 原始消息         | 每轮都在变       |
| 滚动摘要 | 中期对话主线     | 一段文本摘要     | 达到阈值时更新   |
| 事实存储 | 强事实与画像偏好 | 结构化字段或向量 | 仅在识别到时写入 |

这张表真正要看的不是字段本身，而是三条管道各自的分工非常清晰。

- 近期窗口负责刚刚说过什么，承担短期上下文的保真度。这一层最贴近当前这一轮对话，模型要接住用户正在说的话、正在做的动作，全靠它。

- 滚动摘要负责这次会话在聊什么，承担中期主线的连续性。它不追求逐字准确，只负责把几十轮对话压成一段话，让模型知道对话走到现在这一步的主线是什么。

- 事实存储负责这个用户是谁以及之前发生过什么，承担长期可靠召回。它是跨会话也能用的那一层，用户下次再进来模型还能认得出他，靠的就是这里。

分层的好处是每一层可以独立优化。摘要更新策略出问题，不会影响事实库；事实召回跑偏，也不会破坏近期对话的连贯性。

## 近期窗口的裁剪细节

近期窗口看上去最简单，用 `trimMessages` 就能做，但细节上有两个坑必须避开。

- 不要只看消息条数，要按 `token` 算。不同模型的 `token` 成本差异很大，中英文混排时更离谱，按条数估算经常会出现看着没超实际已经爆了的情况，按 `token` 才是真正对着模型 `context window` 在做裁剪。

- 不要裁掉系统消息和工具调用链。`SystemMessage` 定义了整段对话的规则，把它剪掉模型就换了个性格。更危险的是裁断一个 `AIMessage` 的 `tool_calls`，调用链一旦只剩一半，模型下一步逻辑会直接断掉，甚至报错。

下面这段示例展示按 `token` 裁剪并保留系统消息的做法。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { trimMessages, BaseMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const trimmer = trimMessages({
  maxTokens: 1200,
  strategy: "last",
  tokenCounter: model,
  includeSystem: true,
  startOn: "human",
  allowPartial: false,
});

async function buildRecent(history: BaseMessage[]): Promise<BaseMessage[]> {
  return trimmer.invoke(history);
}
```

这里 `strategy: "last"` 表示从最新一条往前保留，`startOn: "human"` 确保裁剪后的序列从用户消息开始，这样模型接入上下文时角色顺序是合法的。`allowPartial: false` 禁止把单条消息切一半，避免出现断句的 `AIMessage` 让后续调用变得不稳。

## 滚动摘要的更新策略

滚动摘要本质上是一个状态机，状态包括现有摘要文本和一个待压缩的对话队列。关键设计集中在三处。

- 触发条件。常见做法是近期窗口超阈值时触发，也可以按时间间隔定时触发。两种方式可以组合，前者保证上下文不会爆，后者保证久不说话也有摘要沉淀。触发条件太宽，摘要会频繁调用模型浪费 `token`；太严，近期窗口又容易来回抖动。

- 提示词。摘要提示词要显式列出需要保留的信息类型，而不是让模型自己决定。只说请压缩成一段话，模型大概率会把订单号、金额这类强事实当作无关细节抹掉，生成的摘要读起来很顺，但关键信息已经没了。

- 叠加方式。新摘要应该是对老摘要加新增对话的重写，而不是简单拼接。如果只是拼接，摘要会越滚越长，时间一久就和原文差不多重。重写的方式让模型每次都在已有摘要的基础上收敛，而不是再往后接一段。

下面这段示意把这三条约束写进一个 `summarize` 函数里，便于看到关键参数在哪里。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import {
  BaseMessage,
  SystemMessage,
  HumanMessage,
} from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const SUMMARY_PROMPT = `请根据已有摘要与新增对话，重写一段不超过 200 字的摘要。
保留以下信息类型，丢失其中任何一类都视为不合格：
1. 用户画像 身份、角色、偏好
2. 任务目标 当前会话在做什么
3. 关键事实 订单号、金额、时间、地点、配置项
4. 进度状态 已完成步骤、未完成步骤、阻塞点
不要添加原文中不存在的信息。`;

async function updateSummary(
  prevSummary: string,
  newTurns: BaseMessage[],
): Promise<string> {
  const transcript = newTurns
    .map((m) => `${m._getType()}: ${m.content}`)
    .join("\n");
  const res = await model.invoke([
    new SystemMessage(SUMMARY_PROMPT),
    new HumanMessage(
      `已有摘要：${prevSummary || "空"}\n\n新增对话：\n${transcript}`,
    ),
  ]);
  return String(res.content);
}
```

提示词里最有价值的部分是那四类显式约束。很多团队写的摘要提示词只说请压缩成一段话，模型就会按自己的审美挑内容，把数字、订单号这类强事实当成可省略的细节砍掉，最后摘要读着很顺，信息却全漏了。显式列出需要保留的类型，是控制摘要漂移最便宜的办法。

另一个细节是摘要本身不要越滚越长。建议给摘要也设置一个 `token` 上限，超过就触发重写压缩，否则几十轮之后摘要会膨胀到和原文差不多长，压缩就失去意义。

## 事实存储应该长什么样

事实存储是这套结构里最被低估的一层。很多团队只做了摘要就以为长期记忆解决了，结果下一次会话里模型又要问一遍用户是谁、之前买了什么。事实存储的价值就在于把这些长期稳定的信息从对话流里拎出来，单独持久化，和对话流解耦。

事实存储通常有两种形态，结构化字段和向量片段，两者并不互斥，实际项目里经常同时用。

- 结构化字段适合确定性强的信息，比如 `userId`、`region`、`preferences.notificationChannel`。它的优点是读取快、查询准，业务代码可以直接拿来做路由、做风控。

- 向量片段适合表达更复杂的事实，比如用户上次描述过的具体问题、过往投诉的大致内容。这类信息字段化成本太高，向量化后按语义召回更实用，准确度也够。

事实抽取环节需要一个独立的调用，可以和主回答并行跑。下面这段示例展示用 `zod` 约束抽取结构。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { z } from "zod";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const FactSchema = z.object({
  profile: z
    .object({
      nickname: z.string().optional(),
      region: z.string().optional(),
      role: z.string().optional(),
    })
    .optional(),
  preferences: z.array(z.string()).optional(),
  keyEvents: z
    .array(
      z.object({
        description: z.string(),
        ts: z.number(),
      }),
    )
    .optional(),
});

type Fact = z.infer<typeof FactSchema>;

const extractor = model.withStructuredOutput(FactSchema, {
  name: "extract_fact",
});

async function extractFact(userMessage: string): Promise<Fact> {
  return await extractor.invoke(
    `从用户发言里抽取画像与关键事实，没出现的字段保持为空：\n${userMessage}`,
  );
}
```

这段代码的关键点不是 `schema` 本身，而是两条约束。一是字段必须允许选择性填写，没识别到就留空，不让模型瞎猜。二是抽取结果要和已有事实做合并而不是覆盖，避免新会话把老偏好洗掉。合并逻辑通常放在业务层，按字段级别做冲突检测，冲突时要么以更新时间更近的为准，要么走人工介入。

抽取出来的事实写入存储后，下一轮回答时通过向量检索或字段读取的方式召回，作为上下文里的独立小节注入。这样模型看到的不是一团摘要，而是清晰分块的画像、偏好、事实，效果比混在摘要里稳得多。

## 按任务阶段做二次裁剪

前面三条管道解决了大部分场景，还有一个进阶技巧值得单独拿出来说。对那些本身就有明显阶段划分的任务，比如工单处理、下单流程、问诊流程，可以做任务阶段级的裁剪。每进入一个新阶段，就把上一阶段的详细对话压缩成结构化的阶段结果，并写入事实存储。

比如客服工单的阶段结果可以长下面这样。

```typescript
interface StageResult {
  stage: "intake" | "diagnose" | "solve" | "closed";
  summary: string;
  decisions: string[];
  refs: { id: string; snippet: string }[];
  nextStage?: string;
}
```

每个阶段结束时，前一阶段的原文对话就可以从上下文里撤掉，只保留这一段结构化结果。相比通用滚动摘要，这种按阶段压缩的方式信息密度更高，因为阶段边界本来就是业务天然的分割点，摘要不会跨阶段混流，回溯起来也方便。

这里的 `refs` 字段特别值得保留，它是事后可追溯的关键。只靠摘要时，出现争议没法复查；有了引用链，就能回到原文段落去核对。这也是客服、金融、医疗这类需要审计的场景必须加的一层。

## 评估怎么做才能防止摘要漂移

压缩策略上线后，一定要有评估手段。没有评估，摘要提示词每改一次、模型版本每升一次，都可能引入信息丢失，线上短时间还看不出来，等用户来骂就晚了。

推荐的评估方法是构造一个回答一致性测试集，构造方式很简单。

- 准备 `N` 组长会话，每组都有清晰的已知事实清单。事实清单要覆盖身份、偏好、订单号、时间、地点这类强事实，数量不用多，十几组就能把主要情况覆盖住。

- 同时用完整历史和压缩后的上下文各跑一次问答。问答的题目要故意踩在强事实上，比如直接问用户的订单号、之前选过什么偏好，让压缩策略在这些关键点上接受检验。

- 对比两种上下文下的回答是否在关键事实上一致。一致性判断可以用关键字比对，也可以再让一个模型当裁判打分，重点看强事实有没有漂移或丢失。

这套方法不依赖人工打分，可以做成 `CI` 环节的一部分。一旦压缩策略有退化，测试集上的一致性分数会立刻下降，问题就能在上线前拦住。

## 总结

长对话的历史压缩不是一个摘要接口就能解决的问题，真正的答案是把历史按信息性质拆开，每一类信息走自己最合适的那条通道。

- 对话流用摘要维护主题与进度。它承担阅读连贯性，让模型知道这次对话走到现在大概讲过什么，不追求字字准确。

- 关键事实用结构化字段或向量记忆单独存。它承担长期可靠召回，保证订单号、偏好这类强事实不被摘要抹平，下次会话还能拿回来。

- 近期原文保留最近 `k` 轮或按 `token` 裁剪。它承担短期保真度，让模型接住用户正在说的这一轮，不会因为压缩丢掉刚刚发生的细节。

- 可追溯场景要留引用链或原文片段 `id`。它承担审计和复查，出现争议时可以回到原文核对，这是客服、金融、医疗类场景的底线要求。

把这几条组合起来就是前文那张流程图。落到工程上还要记住三件事，摘要提示词显式列出需要保留的信息类型、摘要更新尽量异步、事实抽取要做字段级合并而不是覆盖。再配上一套回答一致性的评估集，这套长对话压缩方案就能稳稳地跑下去。
