# LangChain 里如何给 Agent 添加 Skills，并用渐进式披露实现按需能力包

前面我们一起把 Agent 接工具、拆子任务、做多轮协作这些基础能力拆开看过。场景一复杂，能力就会越堆越多，系统提示词往往也越写越长，上下文里塞满这一轮根本用不到的规则、模板和示例，模型反而更容易分心，`token` 花费也跟着上去。

这一篇要聊的 Skills，就是为缓解这种上下文膨胀准备的一种做法。它有点像插件、能力包，不是单独再拉一名 Agent，而是一块可按需加载的能力模块，可以挂在某个 Agent 身上，也可以在多个 Agent 之间共享。

和把所有能力一次性写进提示词不同，Skills 的核心思路是先列目录、再按需展开。Agent 启动时只读各项技能的摘要，真正要用到某项技能时再加载完整内容，官方文档把这种节奏称为渐进式披露。好处很直接，窗口里少装用不着的长文，推理更集中，花费也更可控。

## Skills 解决什么问题

一个功能完善的 Agent 往往具备很多专业方向，但在某一轮对话里，用户的问题通常只涉及其中一两块。如果把所有技能的提示词、模板、示例全部塞进系统提示词，上下文会迅速膨胀，模型反而容易在一大堆无关内容里迷失方向。

Skills 特别适合下面这几种情况：

- 一个 Agent 有很多专业方向，但每次对话只用到其中一小部分
- 不同团队分别维护自己的领域知识，希望用技能目录的形式统一接入
- 某个复杂能力需要在多个项目里复用，打包成独立技能模块更容易管理
- 系统里的技能数量会随业务增长，需要一种动态扩展的机制

例如一个前端工程助手，可以挂三组技能：负责组件拆分和页面改版方案的 `component-designer`、负责无障碍和性能问题检查的 `accessibility-checker`、负责代码审查和规范核查的 `code-reviewer`。助手启动时只读取三组技能的摘要，收到具体请求后再加载对应的完整内容。

## Skills 的文件结构

一项技能通常对应一个独立目录，里面至少有一个 `SKILL.md` 文件。`SKILL.md` 用 frontmatter 描述技能的核心信息，包括技能名称、一句话说明、触发场景关键词、以及约束条件。frontmatter 之后是技能的完整内容，可以包含查询模板、脚本片段、示例输入输出等。Agent 启动时只读取 frontmatter，需要深入用这项技能时再读完整内容。

目录结构通常是这样的：

```
skills/
  component-designer/
    SKILL.md
    examples/
  accessibility-checker/
    SKILL.md
    checklists/
  code-reviewer/
    SKILL.md
    rules/
```

## 渐进式披露

渐进式披露的核心是两步：先用 `list_skills` 工具拿到所有技能的 frontmatter 摘要，再用 `load_skill` 工具按需加载某项技能的完整内容。Agent 不需要提前知道所有技能的细节，只需要在收到用户请求后，先扫一眼目录，再决定要展开哪一本。如下图所示。

![20260320152450](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320152450.png)

先查目录再按需加载，让 Agent 每次只携带当前任务真正需要的技能内容。

`list_skills` 只读 frontmatter，速度快、消耗 `token` 少。`load_skill` 才读完整内容，只在真正需要时调用一次。

## 如何实现 Skills

实现 Skills 的核心是两个工具，一个负责扫描目录、返回 frontmatter 列表，另一个负责加载某项技能的完整内容。有了这两个工具，Agent 就可以自主决定什么时候查目录、什么时候展开技能。

```ts
import { tool } from "@langchain/core/tools";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { ChatOpenAI } from "@langchain/openai";
import { z } from "zod";
import * as fs from "fs";
import * as path from "path";

const llm = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const listSkills = tool(
  async () => {
    const skillsDir = path.join(process.cwd(), "skills");
    const entries = fs.readdirSync(skillsDir, { withFileTypes: true });
    const summaries: string[] = [];

    for (const entry of entries) {
      if (!entry.isDirectory()) continue;
      const skillFile = path.join(skillsDir, entry.name, "SKILL.md");
      if (!fs.existsSync(skillFile)) continue;

      const content = fs.readFileSync(skillFile, "utf-8");
      const match = content.match(/^---\n([\s\S]*?)\n---/);
      if (match) summaries.push(match[1].trim());
    }

    return summaries.length > 0
      ? `可用技能（共 ${summaries.length} 项）：\n\n${summaries.join("\n\n---\n\n")}`
      : "当前没有可用技能。";
  },
  {
    name: "list_skills",
    description:
      "列出所有可用技能的摘要信息（名称、适用场景、约束）。不确定用哪项技能时先调用这个。",
    schema: z.object({}),
  },
);

const loadSkill = tool(
  async ({ skillName }: { skillName: string }) => {
    const skillFile = path.join(process.cwd(), "skills", skillName, "SKILL.md");
    if (!fs.existsSync(skillFile)) {
      return `技能 ${skillName} 不存在，请先用 list_skills 查看可用技能列表。`;
    }
    return fs.readFileSync(skillFile, "utf-8");
  },
  {
    name: "load_skill",
    description:
      "加载某项技能的完整内容，包括详细提示词、模板和示例。确认要深入使用某项技能后再调用。",
    schema: z.object({
      skillName: z
        .string()
        .describe("技能目录名，例如 sql-analyst、code-reviewer"),
    }),
  },
);

const agent = createReactAgent({
  llm,
  tools: [listSkills, loadSkill],
  prompt:
    "你是一名前端工程助手。收到请求时，先用 list_skills 查看有哪些可用技能，再用 load_skill 加载对应技能的完整内容，然后按技能里的要求完成任务。",
});

const result = await agent.invoke({
  messages: [
    {
      role: "user",
      content:
        "我有一个课程列表页面，需要在移动端和桌面端都体验良好，技术栈是 React，帮我给一套组件拆分方案。",
    },
  ],
});

console.log(result.messages.at(-1)?.content);
```

如果技能目录很大，还可以给 `list_skills` 加一个 `category` 参数，支持按分类过滤，避免返回太多不相关的摘要。

## Skills 和 Subagent 的区别

Skills 和 Subagent 解决的是不同粒度的问题，两者并不冲突。

Subagent 是一整个独立的 Agent，有自己的模型、工具和中间件，适合这个子任务足够复杂、需要独立的推理循环的情况。Skills 更偏向在同一个 Agent 内部动态切换行为、注入专属知识，不引入新的 Agent，只是在提示词和模板层面做差异化。

如果需要完整的上下文隔离、不同的工具权限，或者某个子任务的复杂度已经需要自己的一套推理循环，Subagent 更合适。反过来，如果只是同一个 Agent 需要在多个专业方向之间灵活切换，Skills 比拆出多个 Subagent 更轻量。两者也可以结合，一个 Subagent 本身也可以挂载一组 Skills。

## 小结

Skills 是把能力拆成可按需加载模块的一种工程实践，解决的是上下文膨胀和知识组织的问题。它不引入新的 Agent，只是在同一个 Agent 内部用两个工具（`list_skills`、`load_skill`）实现了目录扫描和内容懒加载，让模型每次只携带当前任务真正需要的知识。

落地时从最简单的两个工具入手，先打通流程，再随业务扩展技能目录。`SKILL.md` 的 frontmatter 格式要统一，`description` 和 `triggers` 字段写得越清晰，Agent 的路由决策就越准确。
