如果你读过 LangChain 最近的文章，或者看过 Anthropic 关于长时运行 agent 的工程实践，你一定见过一个词：`harness`。它不是某个具体的库，也不是某个配置项，而是一个描述"agent 应该长什么样"的工程概念。

理解 `harness`，是从会调模型走向能做真实 agent 产品的关键一步。

## 模型和 Agent 到底差在哪里

很多人第一次做 AI 应用，写的是这样的代码：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const response = await model.invoke([
  new HumanMessage("帮我把这个函数重构一下"),
]);
console.log(response.content);
```

这当然能跑，但它只是一个"问答机器"。你每次调用，它都不知道上下文是什么，不知道项目里有哪些文件，不知道上一步做了什么，做完了也没地方保存结果，出错了更无法恢复。

`LangChain` 对这件事有一句很直白的表述：一个裸模型不是 `agent`，必须被 `harness` 包起来，才拥有状态、工具执行、反馈循环和约束。

所以可以把 agent 理解为 Model 加上 Harness，也就是 Agent = Model + Harness。

模型负责推理，`harness` 负责把模型变成一个能持续干活的系统。

## 什么是 harness

`harness` 这个词本义是"挽具"，指套在马身上、让马能拉动车的那套系统。在 agent 语境里，它的含义延伸为：把模型包起来、赋予它行动能力的那层工程外壳。

`LangChain` 的官方定义是：`harness` 是"模型本身之外的所有代码、配置和执行逻辑"，包括状态管理、工具执行、反馈循环、上下文工程、安全约束等。Anthropic 在讲长时运行 agent 时也强调，模型要跨多个上下文窗口工作，必须依赖外部系统去桥接记忆、状态和执行——这个外部系统，就是 `harness`。

你可以用这个类比理解：

- 模型像"大脑"，负责理解、推理、生成
- `harness` 像"身体 + 工具腰带 + 工作台"，负责上下文、工具调用、状态保存、任务分派、失败恢复、人工审批

没有 `harness`，模型只是一个对话框。有了 `harness`，模型才更像一个能持续开展任务、能被信任的工程系统。

行业里常说 Agent 等于模型加上 `harness`：模型负责理解与推理，外层系统负责提示契约、工具执行、状态与产物、上下文管理、任务编排以及安全与人工审批。把两者叠在一起，才更像能持续干活的可执行 Agent。

## harness 由哪些层组成

一个完整的 `harness` 本质上是多层能力协作系统，不是某个单一模块。

![20260327171836](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327171836.png)

harness 可以理解为九个能力一起协作：提示词与规则定义行为契约，上下文管理负责摘要与裁剪避免失忆，状态层负责进度与 `artifacts` 持久化，工具层提供读写执行与检索能力，任务编排层负责拆解任务与调度子 agent，执行环境层负责沙箱隔离副作用，人工介入层拦截高风险动作，安全边界层在工具和环境里硬限制权限，可观测层记录调用链路、耗时、成本和失败原因。最容易踩的坑是把 `harness` 误当成 `prompt` 工程，实际上 `prompt` 只定义模型怎么想，真正决定 agent 能否稳定交付的是上下文、状态、路由、边界和审计这些工程层。

下面这张表保留了最常见的问题现象和对应根因：

| 问题现象                           | 根因                   |
| ---------------------------------- | ---------------------- |
| 任务做着做着上下文爆了             | 上下文管理层缺失       |
| 任务中断后无法恢复，只能重来       | 状态层缺失             |
| Agent 改了文件但用户不知道改了什么 | 人工介入层、审批流缺失 |
| 多步骤任务越做越乱、方向跑偏       | 任务编排层缺失         |
| 模型"幻觉"直接写坏了文件           | 安全边界层缺失         |
| 子任务结果没整合，token 浪费严重   | 子 agent 管理缺失      |
| 模型明明能推理，但就是不会做       | 工具层没给它合适的"手" |
| 执行结果无法审计，出了问题查不到   | 状态层、可观测层缺失   |
| 工具失败后 agent 不知道怎么处理    | 错误处理策略缺失       |

这也是为什么大家现在说"很多 agent 的真实瓶颈不是模型不够聪明，而是外层系统没搭好"——换更强的模型并不能解决上述问题，把 `harness` 搭好才能。

## harness 和相关概念的关系

这几个词是现在最容易混淆的，理清楚非常重要。

![20260327172153](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327172153.png)

### harness 和 prompt 的关系

`prompt` 是给模型的指令契约，解决的是"模型应该怎么思考、怎么回答、怎么调用工具"。但 `prompt` 本身不会保存状态，不会执行工具，不会处理失败恢复。

`harness` 包含 `prompt`，但远不止 `prompt`。可以这样理解：`prompt` 是 `harness` 的"策略层输入"，`harness` 是 `prompt` 的"执行环境"。只做 `prompt` 工程，最多让模型回答得更好；做好 `harness` 工程，才能让 agent 真正干活。

### harness 和 workflow 的关系

`workflow` 是预先设计好的执行流程，步骤相对固定，比如：抽取标题 → 生成摘要 → 生成配图提示词 → 导出。它偏向确定性。

`harness` 不等于 `workflow`，而是承载 `workflow` 和 agent 行为的运行外壳。同一个 `harness` 里可以跑固定 `workflow`，也可以跑开放式 agent。`workflow` 告诉系统"走哪条路"，`harness` 提供"走路需要的腿和工具"。

### harness 和 framework、runtime 的关系

这三个层各司其职：

- `framework`（如 `LangChain`）：解决"怎么更方便地开发 agent"，提供抽象、接口和统一调用方式
- `runtime`（如 `LangGraph`）：解决"agent 怎么在生产环境持续运行"，处理 durable execution、streaming、持久化、失败恢复
- `harness`（如 `Deep Agents`）：解决"模型手里有哪些能力、在什么边界内干活"，带着预置工具、预置 `prompt`、subagent 机制、文件系统、上下文工程

实际系统里这三层通常是叠起来的：用 `framework` 写逻辑，用 `runtime` 负责长期运行，用 `harness` 给模型配齐能力。`Deep Agents` 建在 `LangGraph` 之上，就是这个原因。

## 一次完整的 harness 执行过程

假设你要做一个"文档改写 agent"，用户说：

> 帮我把这篇文章改成公众号风格，补一个标题，顺便配 3 张图的提示词

一个完整的 `harness` 会这样工作：

![20260327173006](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327173006.png)

每一步都不是"把用户话直接喂给模型"。

第一步，接收任务时，`harness` 不只是透传输入，而是把任务写进状态，创建 `todo` 列表：改风格、补标题、生成 3 个配图提示词。这样模型有了清晰的执行计划，而不是"看着办"。

第二步，上下文打包时，`harness` 负责把当前文档内容、选区、风格偏好、历史修改记录组装成模型输入，而不是只传一句模糊需求。这个组装过程会控制 token 量，把最相关的信息排在前面。

第三步，工具执行时，模型调用 `read_file` 读取文档、调用 `rewrite_section` 改段落、调用 `generate_image_prompt` 生成配图提示词，`harness` 负责执行这些工具并把结果回填。

第四步，遇到"写回正式文档"这类高风险操作时，`harness` 暂停执行，把修改提案推给用户审批，确认后才正式落盘。

第五步，任务完成后，`harness` 把所有产出物归档：修改后的文档、标题候选、配图提示词、执行日志，后续如果要回溯或重做，都有迹可查。

从整个执行过程可以看出，`harness` 里最复杂的不是单步逻辑，而是各步之间的状态流转、上下文衔接和异常处理。一旦工具失败或模型输出不符合预期，系统需要知道"下一步该怎么办"，而不是直接崩溃。

如下图所示。

![20260327174522](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327174522.png)

接收任务到归档产出的六步主链，审批拒绝后折返推理形成安全闭环。

## 用 LangGraph 实现一个可运行的 harness Demo

在前面的内容中，我们已经了解了 `harness` 的核心概念和分层思路。接下来这一节按 `state.ts`、`tools.ts`、`graph.ts`、`index.ts` 的顺序给出完整代码，并对应说明每个文件在 `harness` 里的职责。

先看状态层，`state.ts` 负责持久化执行状态，让图在中断后可以恢复，也能在结束后归档产物。

```typescript
import { Annotation } from "@langchain/langgraph";
import { BaseMessage } from "@langchain/core/messages";

export type Phase = "planning" | "executing" | "awaiting_approval" | "done";

export interface Artifact {
  name: string;
  content: string;
  createdAt: string;
}

export const HarnessState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (x, y) => x.concat(y),
    default: () => [],
  }),
  phase: Annotation<Phase>({
    reducer: (_, y) => y,
    default: () => "planning",
  }),
  todos: Annotation<string[]>({
    reducer: (_, y) => y,
    default: () => [],
  }),
  artifacts: Annotation<Artifact[]>({
    reducer: (x, y) => x.concat(y),
    default: () => [],
  }),
  retryCount: Annotation<number>({
    reducer: (_, y) => y,
    default: () => 0,
  }),
});
```

这里最关键的是 `messages`、`phase`、`artifacts` 三个字段。前两者保证流程连续性，后者保证最终结果可追踪。`retryCount` 则给失败重试预留了状态位。

再看工具层，`tools.ts` 负责提供可调用动作，重点是把高风险写入动作显式化，交给编排层处理审批。

```typescript
import { tool } from "@langchain/core/tools";
import { z } from "zod";

export const virtualFS: Record<string, string> = {
  "src/utils.ts": `// 用户模块工具函数

export async function fetchUserData(userId) {
  // 缺少类型注解，缺少 try/catch 错误处理
  const response = await fetch('/api/users/' + userId)
  return response.json()
}

export function calculateAge(birthYear) {
  // getFullYear 后面漏了括号，导致 NaN
  const currentYear = new Date().getFullYear
  return currentYear - birthYear
}

export function formatName(firstName, lastName) {
  // 当两者都是空字符串时，仍然返回 " "（多余空格）
  return firstName + ' ' + lastName
}

export function parseConfig(jsonStr) {
  // 缺少错误处理，JSON.parse 失败时会直接抛出
  return JSON.parse(jsonStr)
}
`,
  "src/types.ts": `// 类型定义文件
export interface User {
  id: string;
  name: string;
  age: number;
}

export interface Config {
  apiUrl: string;
  timeout: number;
}
`,
};

export const listFilesTool = tool(
  async () => {
    const files = Object.keys(virtualFS)
      .map((f) => `  - ${f}`)
      .join("\n");
    return `项目中可访问的文件：\n${files}`;
  },
  {
    name: "list_files",
    description: "列出项目中所有可访问的文件路径",
    schema: z.object({}),
  },
);

export const readFileTool = tool(
  async ({ path }: { path: string }) => {
    if (virtualFS[path]) {
      return `文件 "${path}" 的内容：\n\`\`\`typescript\n${virtualFS[path]}\`\`\``;
    }
    const available = Object.keys(virtualFS).join("、");
    return `错误：文件 "${path}" 不存在。可用文件：${available}`;
  },
  {
    name: "read_file",
    description: "读取项目文件内容，只读操作，无需审批可直接执行",
    schema: z.object({
      path: z.string().describe("文件路径，例如 src/utils.ts"),
    }),
  },
);

export const writeFileTool = tool(
  async ({ path, content }: { path: string; content: string }) => {
    virtualFS[path] = content;
    return `文件 "${path}" 已成功写入，共 ${content.length} 字符。`;
  },
  {
    name: "write_file",
    description:
      "向文件写入内容。这是高风险操作，harness 会在执行前自动暂停并等待用户确认。",
    schema: z.object({
      path: z.string().describe("目标文件路径"),
      content: z.string().describe("要写入的完整文件内容"),
    }),
  },
);

export const tools = [listFilesTool, readFileTool, writeFileTool];
```

这层的核心不是工具数量，而是工具语义边界。`write_file` 一旦被模型命中，就会触发图路由里的审批逻辑，这是安全边界的落点。

下面是编排层核心，`graph.ts` 负责把状态、工具、审批、归档串成可恢复的执行图。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { StateGraph, interrupt, MemorySaver } from "@langchain/langgraph";
import { ToolNode } from "@langchain/langgraph/prebuilt";
import {
  AIMessage,
  AIMessageChunk,
  BaseMessage,
  HumanMessage,
  SystemMessage,
  ToolMessage,
} from "@langchain/core/messages";
import { HarnessState, Phase } from "./state.js";
import { tools } from "./tools.js";

const model = new ChatOpenAI({
  model: "Pro/zai-org/GLM-5",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
});

const modelWithTools = model.bindTools(tools);
const toolNode = new ToolNode(tools);

const SYSTEM_PROMPT = `你是一个专业的 TypeScript 代码审查 agent。

你的任务是：
1. 读取指定的 TypeScript 文件，分析其中的 bug 和代码质量问题
2. 生成修复后的完整文件内容
3. 使用 write_file 工具提交改进后的代码（系统会自动暂停等待用户确认）

工作规范：
- 先用 list_files 了解项目结构，再用 read_file 读取目标文件
- 读取操作无需审批，可以直接执行
- 写入操作是高风险操作，必须通过 write_file 工具，不要在回复里直接输出代码作为"写入"
- 改进代码时要保留原有逻辑，只修复错误、补充类型注解、添加错误处理
- 每次只处理一个文件，处理完后给出清晰的修改说明`;

function trimContext(
  messages: BaseMessage[],
  maxNonSystem = 20,
): BaseMessage[] {
  const systemMsgs = messages.filter((m) => m._getType() === "system");
  const otherMsgs = messages.filter((m) => m._getType() !== "system");
  if (otherMsgs.length <= maxNonSystem) return messages;
  return [...systemMsgs, ...otherMsgs.slice(-maxNonSystem)];
}

async function callModel(
  state: typeof HarnessState.State,
): Promise<Partial<typeof HarnessState.State>> {
  const messagesWithSystem: BaseMessage[] =
    state.messages.length === 0 || state.messages[0]._getType() !== "system"
      ? [new SystemMessage(SYSTEM_PROMPT), ...state.messages]
      : state.messages;

  let response: AIMessageChunk | undefined;
  process.stdout.write("\n[模型] ");

  for await (const chunk of await modelWithTools.stream(
    trimContext(messagesWithSystem),
  )) {
    if (typeof chunk.content === "string" && chunk.content) {
      process.stdout.write(chunk.content);
    } else if (Array.isArray(chunk.content)) {
      for (const part of chunk.content) {
        if (typeof part === "object" && part !== null && "text" in part) {
          process.stdout.write(String(part.text));
        }
      }
    }
    response = response ? (response.concat(chunk) as AIMessageChunk) : chunk;
  }

  process.stdout.write("\n");
  if (!response) throw new Error("模型 stream 未返回任何内容");

  if (response.tool_calls && response.tool_calls.length > 0) {
    const names = response.tool_calls.map((tc) => tc.name).join("、");
    console.log(`  → 工具调用: ${names}`);
  }

  return { messages: [response], phase: "executing" as Phase };
}

function routeAfterModel(state: typeof HarnessState.State): string {
  const last = state.messages[state.messages.length - 1] as AIMessage;
  if (!last.tool_calls || last.tool_calls.length === 0) return "save_artifacts";
  if (last.tool_calls.some((tc) => tc.name === "write_file"))
    return "human_review";
  return "tools";
}

async function humanReview(
  state: typeof HarnessState.State,
): Promise<Partial<typeof HarnessState.State>> {
  const last = state.messages[state.messages.length - 1] as AIMessage;
  const pendingWrites =
    last.tool_calls?.filter((tc) => tc.name === "write_file") ?? [];

  const decision = interrupt({
    type: "write_approval",
    message: "Agent 请求写入文件，是否允许？（approve / reject）",
    pendingWrites: pendingWrites.map((tc) => ({
      path: tc.args.path,
      contentLength: String(tc.args.content).length,
    })),
  }) as string;

  if (decision === "reject") {
    return {
      messages: [
        new HumanMessage("用户拒绝了这次写入操作，请重新思考改进方案。"),
      ],
      phase: "executing" as Phase,
    };
  }

  return { phase: "executing" as Phase };
}

function routeAfterReview(state: typeof HarnessState.State): string {
  return state.messages[state.messages.length - 1]._getType() === "human"
    ? "model"
    : "tools";
}

function routeAfterTools(state: typeof HarnessState.State): string {
  const recentToolMsgs: ToolMessage[] = [];
  const msgs = state.messages;

  for (let i = msgs.length - 1; i >= 0; i--) {
    if (msgs[i]._getType() === "tool") {
      recentToolMsgs.unshift(msgs[i] as ToolMessage);
    } else {
      break;
    }
  }

  const hasError = recentToolMsgs.some((m) => m.status === "error");
  if (hasError && state.retryCount >= 3) return "save_artifacts";
  return "model";
}

async function saveArtifacts(
  state: typeof HarnessState.State,
): Promise<Partial<typeof HarnessState.State>> {
  return {
    artifacts: [
      {
        name: "agent-final-output",
        content: String(state.messages[state.messages.length - 1].content),
        createdAt: new Date().toISOString(),
      },
    ],
    phase: "done" as Phase,
  };
}

export function buildHarness() {
  const graph = new StateGraph(HarnessState)
    .addNode("model", callModel)
    .addNode("tools", toolNode)
    .addNode("human_review", humanReview)
    .addNode("save_artifacts", saveArtifacts)
    .addEdge("__start__", "model")
    .addConditionalEdges("model", routeAfterModel, {
      tools: "tools",
      human_review: "human_review",
      save_artifacts: "save_artifacts",
    })
    .addConditionalEdges("tools", routeAfterTools, {
      model: "model",
      save_artifacts: "save_artifacts",
    })
    .addConditionalEdges("human_review", routeAfterReview, {
      model: "model",
      tools: "tools",
    })
    .addEdge("save_artifacts", "__end__");

  return graph.compile({ checkpointer: new MemorySaver() });
}
```

这份代码里有两个关键点最容易忽视。第一，`interrupt` 不是日志提示，而是执行暂停点。第二，`routeAfterTools` 用 `ToolMessage.status` 判断失败，比搜索字符串 `error` 更可靠。

最后看入口层，`index.ts` 负责消费整张图的流式事件并处理审批恢复。

```typescript
import { HumanMessage, ToolMessage } from "@langchain/core/messages";
import { Command } from "@langchain/langgraph";
import { buildHarness } from "./graph.js";
import { Artifact } from "./state.js";

function banner(title: string) {
  const line = "─".repeat(55);
  console.log(`\n${line}\n  ${title}\n${line}`);
}

function printChunk(chunk: Record<string, unknown>) {
  for (const [node, update] of Object.entries(chunk)) {
    if (node === "tools") {
      const msgs = (update as { messages?: unknown[] }).messages ?? [];
      for (const msg of msgs) {
        const tm = msg as ToolMessage;
        if (tm._getType?.() === "tool") {
          const content = String(tm.content);
          const display =
            content.length > 500 ? content.slice(0, 500) + "\n  ..." : content;
          console.log(`\n[工具结果 / ${tm.name}]\n${display}`);
        }
      }
    }
    if (node === "save_artifacts") {
      const arts = (update as { artifacts?: Artifact[] }).artifacts ?? [];
      if (arts.length > 0) console.log(`\n[归档] 共 ${arts.length} 个产出物`);
    }
  }
}

type StreamInput = { messages: HumanMessage[] } | InstanceType<typeof Command>;

async function runStream(
  app: ReturnType<typeof buildHarness>,
  input: StreamInput,
  config: Record<string, unknown>,
): Promise<{ interrupted: boolean; interruptValue: unknown }> {
  for await (const chunk of await app.stream(
    input as Parameters<typeof app.stream>[0],
    {
      ...config,
      streamMode: "updates",
    } as Parameters<typeof app.stream>[1],
  )) {
    const c = chunk as Record<string, unknown>;
    if ("__interrupt__" in c) {
      const interrupts = c.__interrupt__ as Array<{ value: unknown }>;
      return { interrupted: true, interruptValue: interrupts[0]?.value };
    }
    printChunk(c);
  }
  return { interrupted: false, interruptValue: null };
}

async function askUserApproval(interruptValue: unknown): Promise<string> {
  console.log("\n╔══════════════════════════════════════════╗");
  console.log("║         ⚠️   等待人工审批                  ║");
  console.log("╚══════════════════════════════════════════╝");

  if (typeof interruptValue === "object" && interruptValue !== null) {
    const iv = interruptValue as Record<string, unknown>;
    console.log(`\n  ${iv.message}`);
    const writes = iv.pendingWrites as Array<{
      path: string;
      contentLength: number;
    }>;
    writes?.forEach((w) =>
      console.log(`  文件: ${w.path}  (${w.contentLength} 字符)`),
    );
  }

  const decision = "approve";
  console.log(`\n  [用户决策] → ${decision}`);
  return decision;
}

async function main() {
  banner("Agent Harness 示例 — 代码审查 Agent");

  const app = buildHarness();
  const config = {
    configurable: { thread_id: "code-review-1" },
    recursionLimit: 30,
  };

  banner("启动任务");
  const { interrupted, interruptValue } = await runStream(
    app,
    {
      messages: [
        new HumanMessage(
          "请帮我审查并修复 src/utils.ts 文件中的所有 bug，包括缺少类型注解、错误处理和逻辑错误，然后把修复后的版本写回文件。",
        ),
      ],
    },
    config,
  );

  if (interrupted) {
    const decision = await askUserApproval(interruptValue);
    banner("恢复执行");
    await runStream(app, new Command({ resume: decision }), config);
  }

  banner("执行完成");

  const finalState = await app.getState(config);
  const artifacts = (finalState.values.artifacts ?? []) as Artifact[];

  if (artifacts.length > 0) {
    for (const [i, a] of artifacts.entries()) {
      console.log(`\n  [产出物 ${i + 1}] ${a.name}  (${a.createdAt})`);
      const display =
        a.content.length > 600
          ? a.content.slice(0, 600) + "\n  ..."
          : a.content;
      console.log(`\n${display}`);
    }
  } else {
    const msgs = (finalState.values.messages ?? []) as Array<{
      _getType: () => string;
      content: unknown;
    }>;
    const lastAI = [...msgs].reverse().find((m) => m._getType() === "ai");
    if (lastAI) console.log(`\n${lastAI.content}`);
  }

  console.log(`\n${"─".repeat(55)}\n`);
}

main().catch(console.error);
```

入口层最大的价值是把一次任务自然拆成两段运行：先跑到中断，再带审批结果恢复。这个结构让模型流、工具流、审批流都在同一事件通道里，后续接前端和监控系统时会非常顺滑。

## 小结

> `harness` 的本质是把模型放进一套可控、可恢复、可审计的执行系统。

这套系统里，`state.ts` 负责记住上下文和进度，`tools.ts` 负责提供可约束的动作，`graph.ts` 负责路由和边界，`index.ts` 负责把执行过程实时暴露给外部。模型能力决定上限，但真正决定 agent 能不能稳定交付的，是外层工程是否扎实。沿着这套 Demo 往前做，把沙箱文件系统、审批回调、结构化事件流和归档审计补齐，就可以从教学样例平滑过渡到可上线原型，这就是 `harness engineering` 的核心价值。
