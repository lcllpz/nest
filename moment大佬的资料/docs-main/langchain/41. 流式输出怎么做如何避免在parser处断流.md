很多人第一次做流式输出，脑子里的链路都是同一套。前端发请求，后端打开模型的 `streaming`，通过 `SSE` 或 `WebSocket` 把 `token` 一段段推给前端，前端做个打字机效果。这套方案在普通聊天里完全够用，它只解决一件事，让模型生成的文本不要等到最后一起返回，而是边生成边展示。

但到了 Agent 场景，这种理解就明显不够用。Agent 不只是在吐答案，它还在拆任务、选工具、调工具、处理返回、失败重试、调整路径，有时候还要停在某个节点等用户确认。这时候系统真正需要推给前端的，已经不只是模型吐出的文本，而是整个运行过程本身。问题也从模型能不能边生成边吐 `token`，变成了 Agent 运行时能不能把关键状态、步骤、工具事件、阶段结果都持续暴露出去。这两个问题看起来都叫流式输出，实际复杂度完全不在一个档次。

## 普通 SSE 和 Agent 流式根本不是一个量级

最基础的 `SSE` 聊天本质上是一条单通道数据流。模型不断生成 `token`，服务端往前端推，前端把 `token` 拼成一段不断增长的文本。它解决的问题很单纯，让用户在等待的时候有东西在动，知道模型没卡死。至于系统内部到底在干嘛，前端其实什么都不知道，它只知道字还在出来。

Agent 就不一样。一次 Agent 任务里通常要经历好几个不同性质的阶段，先理解任务、再决定搜资料、调 `search` 工具、拿到结果、整理信息、发现不够再调另一个工具、某个工具超时、系统触发重试、最后生成一份结构化结论。这里面真正需要实时展示的，已经不是模型吐出的那句话，而是系统现在在干什么、为什么停下来、是在思考还是在查资料还是在重试。

如果这时候还只把流式理解成模型 `token` 的 `SSE` 输出，前端就会得到一种很割裂的体验。模型先输出一句话，然后突然停住，界面没有任何反馈，几秒后又继续。从系统角度看这几秒可能是 Agent 在跑工具，但对用户来说就像系统卡死。普通流式解决的是文本等待感，Agent 流式解决的是运行过程可见性，后者的复杂度要高一个数量级。

## Agent 流式真正要推的五层信息

Agent 的流式输出不是单一文本流，拆开来看大致能分成五层，每一层承载的信息性质不一样，该去的前端区域也不一样。

- 模型文本流，就是大家平时说的打字机效果那一层。模型一边想一边把回答吐出来，比如我先帮你分析一下、从当前文档来看、综合上面的信息可以得出。这部分按 `token` 或 `chunk` 持续推给前端最合适，让正文区域有东西在动，用户心里的等待焦虑就会降下来。不过要清楚，它只是 Agent 流式里最表层的那一层，远不是全部。

- 步骤状态流，用来回答系统现在跑到哪一步了这个问题。Agent 在一次任务里会经过好几个阶段，任务分析、工具选择、资料检索、结果整理、最终回答，这些阶段切换不该混进正文，但放进一个独立的步骤区或时间线区就刚刚好。用户很多时候并不关心模型写了多少字，他关心的是你现在到底是在查资料，还是已经快要给结论了。

- 工具调用流，这是 Agent 和普通 `SSE` 拉开距离的地方。一旦走到 `tool calling`，背后就会冒出一整串运行事件，准备调哪个工具、传了什么参数、开始执行没、跑了多久、拿回多少结果、摘要是什么、失败了没、要不要重试。这些事件如果全藏起来，用户看到的就是我来帮你查一下、空白几秒、我查到了以下内容，中间那几秒对他就是黑盒，而 Agent 体验里黑盒越多，用户越没安全感。比较舒服的做法是模型一决定调工具，前端立刻亮出一条事件说正在调用 `search_docs`，执行中持续更新进度，超时就提示正在重试，彻底失败就显示已降级为基于已有上下文回答。重点不是把技术细节一股脑倒给用户，而是让关键执行节点保持可见、可理解、可解释。

- 系统状态流，平时最容易被忽略的一层。很多人做流式只盯着业务内容，系统状态完全不推。但 Agent 跑一次任务通常比普通对话久得多，只要中间停顿超过两三秒，用户第一反应就是挂了吗、网断了吗。这时候正在等待工具响应、某个节点执行偏慢、触发自动重试、达到最大重试次数、切到了备用工具、进入人工确认节点、用户已中断、任务已恢复这类信号就特别值钱。它们不是正文，也不是工具结果，但能把沉默时刻转成可见状态，让流式从一个视觉动画升级成完整的运行态反馈。

- 结构化结果流，专门给下游消费用的那一份。有些 Agent 任务跑完之后，除了给用户看一段自然语言回答，还要吐一份 `JSON` 结论对象、工具结果汇总、表单填充、执行元数据、审批结果、提取字段集合。这种东西和正文 `token` 搅在一起流毫无意义，更顺手的做法是正文照常流式展示，整个任务结束再单独发一个 `final_result` 或 `structured_result` 事件。展示流和结构化流解耦，后面讲怎么避免 `parser` 断流时还会再提到，这是关键一招。

## 用带类型事件流替代纯文本流

把这五层串起来看就能得到一个很清楚的结论，Agent 的流式协议不该是一段不断追加的字符串，而应该是一种带类型的事件流协议。后端推给前端的不再是一堆拼起来的文本，而是类似下面这样的事件序列。

```typescript
type StreamEvent =
  | { type: "agent.step.start"; step: string; message: string }
  | { type: "agent.step.finish"; step: string; message: string }
  | {
      type: "agent.status";
      status: "retrying" | "degraded" | "waiting";
      message: string;
    }
  | { type: "tool.start"; tool: string; args?: Record<string, unknown> }
  | { type: "tool.progress"; tool: string; message: string }
  | { type: "tool.result"; tool: string; summary: string }
  | { type: "tool.error"; tool: string; reason: string }
  | { type: "message.delta"; content: string }
  | { type: "final"; content: string }
  | { type: "structured_result"; data: Record<string, unknown> };

const sampleStream: StreamEvent[] = [
  { type: "agent.step.start", step: "plan", message: "开始分析任务" },
  { type: "agent.step.finish", step: "plan", message: "任务分析完成" },
  { type: "tool.start", tool: "search_docs", args: { query: "流式输出" } },
  {
    type: "tool.progress",
    tool: "search_docs",
    message: "已检索 12 个候选结果",
  },
  { type: "tool.result", tool: "search_docs", summary: "返回 3 条高相关内容" },
  { type: "message.delta", content: "根据搜索结果，这里主要有三个关键点..." },
  { type: "agent.status", status: "retrying", message: "搜索超时，正在重试" },
  { type: "final", content: "综合来看，最终结论是..." },
  { type: "structured_result", data: { score: 0.92, sourceCount: 3 } },
];
```

上面这段只是示意。事件名、`payload` 结构、`message` 文案都可以按自己的产品去调整，真正值钱的是这套思路。有了带 `type` 的事件类型之后，前端就能按类型分区域渲染，`message.delta` 进正文区，`tool.start` 和 `tool.result` 进工具日志区，`agent.step.*` 进步骤时间线，`agent.status` 进状态提示栏，`structured_result` 交给业务逻辑或折叠详情区。这样做的本质是把 Agent 的执行过程从黑盒变成半透明盒，用户看到的不再是一个会打字的聊天框，而是一个真正的 Agent 执行界面。

## Parser 为什么总是把流掐死

有了事件协议之后，再讲 `parser` 断流就自然了。普通模型 `SSE` 里 `parser` 断流已经是老问题，到了 Agent 里会更严重，因为 Agent 输出的内容类型更多、更杂、更不稳定。

举一个常见场景。某一时刻模型可能只吐出了一半工具参数。

```text
{
  "tool": "search_docs",
  "args": {
    "query": "流式输出
```

对流式展示来说这段已经有价值了，系统完全可以告诉用户 Agent 正在准备搜索流式输出。但对 `parser` 来说，这段既不是合法 `JSON`，也过不了严格的 `schema` 校验。如果链路里有节点坚持必须 `parse` 成完整的工具调用对象才能继续，这段流就只能先缓存，直到模型把剩余部分吐完。

`parser` 和 `streaming` 天然是有张力的，`parser` 想要完整闭合的数据，`streaming` 提供的是中间过程数据。偏偏 Agent 场景里经常会在中途穿插各种解析动作，解析 `tool call`、解析中间结构、做 `schema` 校验、把中间结果映射成业务对象、对步骤结果做标准化。这些动作越多，链路里越容易形成必须等完整结果的节点。只要有一个节点把数据攒住，前面已经流动起来的 `token`、工具事件、阶段状态就会被重新缓存，整条流最终退化成一次性返回。

所以在 Agent 里，不要让 `parser` 站到实时链路中央。更合理的拆法是下面这几条，核心思路就是让流式链路和解析链路各走各的，不要搅在一起。

- 实时链路只推事件和增量。模型 `token` 走 `message.delta`，工具生命周期走 `tool.*`，步骤切换走 `agent.step.*`，系统状态走 `agent.status`。这条链路上不做任何可能阻塞的严格解析，只做最低限度的透传和分发，确保数据一产生就能尽快到达前端。

- 严格解析放到阶段结束之后。真正需要 `JSON` 对象、`schema` 校验、业务结构化的节点，统一挪到某个阶段结束、某个工具调用完成、整个任务收尾这些明确的时间点再去做。阶段没结束之前允许半成品存在，阶段一结束再把完整的数据丢给 `parser`，两边互不干扰。

- 结构化结果用单独事件返回。不要让结构化产物和正文 `token` 抢同一条通道，专门留一个 `structured_result` 事件给下游消费方用，展示流只管展示，结构化流只管结果，订阅方也能按需只听自己关心的那一路。

- 必须解析的中间对象要显式产出。如果某个中间环节一定要拿到解析后的对象，比如已经确认的工具调用参数，就让编排层在确定之后显式发一个事件出来，而不是让下游 `parser` 从自然语言文本里临时抠。显式事件比字符串解析稳定得多，也不会因为模型换一种措辞就挂掉。

一句话概括就是，流式负责过程可见，解析负责结果可靠，这两件事不要混成同一条链路。

## 把 Agent 流式拆成三层去落地

把前面所有内容工程化之后，Agent 的流式设计可以拆成三层，从下往上逐层加工。

![20260424114459](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260424114459.png)

最下面是模型层 `streaming`。这一层只关心一件事，模型本身有没有增量返回能力，能不能按 `token`、`chunk` 或 `partial message` 持续吐数据。它是最底层的能力，不是最终的用户体验，没有它后面两层都无从谈起，但只有它远远不够。

中间是运行时编排层，这是 Agent 流式真正的主战场，它的工作是把模型、工具、状态机、重试逻辑、步骤推进这些内部动作统一转成前端能消费的事件。模型 `token` 转成 `message.delta`，步骤切换转成 `step.start` 和 `step.finish`，工具生命周期转成 `tool.start`、`tool.progress`、`tool.result`、`tool.error`，系统异常和重试转成 `status`、`retry`、`warning`，最终结果转成 `final` 和 `structured_result`。Agent 流式不是模型直接把 `SSE` 丢给前端，而是编排层把整个执行过程包装成事件流再对外输出。

最上面是前端渲染层。前端接到事件后不该只做字符串拼接，而是把 UI 拆成几个区域，分别消费不同类型的事件，下一节专门讲这一层怎么分。

## 前端界面要按事件类型分区

前端这一层单独拎出来讲，是因为很多团队做到事件协议这一步就停了，UI 依然只有一个聊天框，后端吭哧吭哧分出来的那些事件类型在前端全浪费掉了。一个成熟的 Agent 界面至少要分下面这几块。

- 主回答区域，用来展示模型最终要对用户说的话，也就是 `message.delta` 拼接出来的正文。这一块可以保留打字机效果，因为它最贴合用户对 `AI` 在回答我的直觉，是整个界面的主角。

- 步骤区域，用来展示 Agent 当前跑到哪一步，分析需求、搜索资料、整理候选、生成回答、输出结论这种节奏感很强的过程就适合放在这里。它能明显降低复杂任务里的不确定感，让用户知道系统有在往前走，而不是卡在某个地方。

- 工具日志区域，用来展示工具调用细节，通常要做成摘要可见、细节可折叠的形式。默认只显示已调用搜索工具、已返回 3 条结果、正在重试这类摘要，用户想看细节再展开。千万不要一上来就把原始工具响应全砸到界面上，那种信息密度对绝大多数用户都是干扰。

- 状态提示区域，用来展示正在等待响应、重试中、已降级、需要你确认、已取消这类信号。这一块在长任务里尤其重要，它负责填补内容暂时不增长的空白期，让用户在那几秒不会误判系统已经死了。

- 最终结果区域，任务结束之后统一展示最终回答、摘要、结构化结果、来源引用、执行概览。它的作用是给整个过程画一个明确的收尾，让用户知道任务到此结束，也方便他回头查看这次跑了些什么、结果来自哪里。

这五块区域各司其职，事件协议的价值才真正落到用户看得见的地方。

## 工具调用场景必须回答的三个问题

前端分区之外，还有一个工程要求要单独点出来。Agent 里只要涉及 `tool calling`，流式设计至少要回答三个问题，否则体验一定会出问题。

- 工具开始了吗。如果开始了就必须尽早告诉用户，一旦模型停顿用户就会误以为系统卡死。这要求编排层在发出工具调用请求的同一时刻就发一条 `tool.start` 事件，不要等工具返回再说，因为那时候等待已经发生了。

- 工具执行得怎么样了。如果耗时较长，要提供最少但足够理解的进度信息，否则等待就会变成黑盒。对于有明确进度的工具可以发 `tool.progress`，对于不透明的外部调用至少要定时发一条心跳状态出来，让前端知道还在跑，哪怕这条心跳里没有任何新信息。

- 工具失败了怎么办。失败不能被内部吞掉，用户最好知道失败了、系统是否正在重试、是否已经降级、是否需要他介入。这几个信号通过 `tool.error` 和 `agent.status` 的组合就能传递清楚，让用户在失败发生的第一时间就看到后续动作，而不是在界面上盯着一个停住的状态发呆。

这三个问题答清楚，`tool calling` 场景下的流式体验就不会再出现突然安静几秒的情况。

## 总结

回到开头那个区分，普通聊天流式解决的是文本等待感，Agent 流式解决的是运行过程可见性，两者不是一个量级的问题。做 Agent 的时候如果还停留在模型 `token` 的 `SSE` 输出，那这个流式方案大概率是不完整的。

一个完整的 Agent 流式方案可以归纳成三条。

- 协议层走事件流，不走纯文本流。用 `message.delta`、`step.*`、`tool.*`、`status`、`final`、`structured_result` 这类带类型事件承载不同性质的信息，让每种事件都有自己明确的语义，而不是让前端从一坨文本里猜现在在发生什么。

- 链路层把流式和解析拆开。实时链路只推事件和增量内容，严格的 `parser` 和 `schema` 校验放到阶段结束或任务完成之后再做，不要让 `parser` 站在实时链路中央。一旦中间有节点必须等完整数据才肯放行，前面跑起来的流就会被重新攒成一次性返回。

- 界面层按事件类型分区。正文、步骤、工具日志、状态提示、最终结果各走各的区域，后端辛辛苦苦分好的事件类型要在界面上看得见，用户才会真正感觉到 Agent 在干活，而不只是看到字在动。

把这三条落到位，用户在 Agent 里感受到的就不再是模型说到一半突然停住再继续，而是整个执行过程都在持续可见。流式输出这件事做到这一步，才算真正做完。
