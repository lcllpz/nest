前面的内容里，我们已经接好大模型、用结构化消息把对话摆清楚、给模型配上多工具，并跑通一整轮"模型返回 `tool_calls`、执行工具、把结果写成 `ToolMessage` 追进消息、再调模型"的循环。

本系列里所说的 `Agent`，指的就是这条闭环。用户话进来，模型负责理解与拍板，需要时走工具拿外部信息，再把结果纳入下一轮推理，直到落成最终回答。

接下来这一章把模型、消息和工具串在一起，从零搭一个能跑的 `Agent`，亲手走完整条链路。

## 智能体在这里指什么

日常用语里的"智能体"含义很广，可以是多步规划、记忆、反思等。在 LangChain 的入门语境下，我们先把它收窄成一种能根据用户输入决定是否调工具、调哪些，再把工具结果纳入下一轮推理的对话循环。模型是大脑，工具是手脚，消息数组是对话与工具结果的载体，三者齐备、循环跑通，就是一个最简可用的 Agent。

构建第一个智能体，只需要做三件事：

- 用前面学过的方式接入一个聊天模型
- 用 `tool()` 和 `bindTools` 给模型配好一组工具
- 写一段循环逻辑，有 `tool_calls` 就按名称执行、把结果写成 `ToolMessage` 追进消息、再 `invoke`，直到模型不再返回 `tool_calls`、只给最终文本

下面按这个顺序把代码串起来。

## 第一步 定义工具并绑定到模型

工具定义和绑定的写法和第 6 章一致，这里直接复用两个真实工具，即查天气的 `getWeather`（和风天气 API）和搜互联网的 `internetSearch`（Tavily）。把这两个工具放进数组传给 `model.bindTools()`，再建好 `toolsByName` 映射表，后面循环里按名称查找执行。下面是一段完整示例，可直接放进项目里用。

```typescript
import { tool } from "@langchain/core/tools";
import { TavilySearch } from "@langchain/tavily";
import { z } from "zod";
import { ChatOpenAI } from "@langchain/openai";

const QWEATHER_HOST = "https://ph5u9wjwrf.re.qweatherapi.com";
const QWEATHER_KEY = "你的和风天气密钥";
const TAVILY_KEY = "你的 Tavily 密钥";

interface QWeatherLocation {
  id: string;
  name: string;
  adm1: string;
  adm2: string;
}

interface QWeatherDaily {
  fxDate: string;
  textDay: string;
  textNight: string;
  tempMax: string;
  tempMin: string;
}

async function fetchLocationId(cityName: string): Promise<QWeatherLocation> {
  const url = `${QWEATHER_HOST}/geo/v2/city/lookup?location=${encodeURIComponent(cityName)}&range=cn&number=1&key=${QWEATHER_KEY}`;
  const res = await fetch(url);
  if (!res.ok) throw new Error(`城市查询 HTTP 错误：${res.status}`);
  const data = await res.json();
  if (data.code !== "200")
    throw new Error(`城市查询失败，错误码：${data.code}`);
  if (!data.location?.length) throw new Error(`未找到城市：${cityName}`);
  return data.location[0] as QWeatherLocation;
}

const getWeather = tool(
  async (input: { location: string }): Promise<string> => {
    const location = await fetchLocationId(input.location);
    const url = `${QWEATHER_HOST}/v7/weather/3d?location=${location.id}&key=${QWEATHER_KEY}`;
    const res = await fetch(url);
    if (!res.ok) throw new Error(`天气查询 HTTP 错误：${res.status}`);
    const data = await res.json();
    if (data.code !== "200")
      throw new Error(`天气查询失败，错误码：${data.code}`);
    const daily = data.daily as QWeatherDaily[];
    const forecast = daily
      .map(
        (d) =>
          `${d.fxDate} 白天:${d.textDay} 夜间:${d.textNight} ${d.tempMin}~${d.tempMax}℃`,
      )
      .join("\n");
    return `${location.name}（${location.adm1} ${location.adm2}）未来三天：\n${forecast}`;
  },
  {
    name: "get_weather",
    description:
      "根据城市名称查询未来三天天气预报，返回每天的天气状况和气温范围。适合回答天气相关的问题。",
    schema: z.object({
      location: z.string().describe("城市名称，如北京、上海、深圳"),
    }),
  },
);

const internetSearch = tool(
  async (input: {
    query: string;
    maxResults?: number;
    topic?: "general" | "news" | "finance";
  }): Promise<string> => {
    const tavilySearch = new TavilySearch({
      maxResults: input.maxResults ?? 5,
      tavilyApiKey: TAVILY_KEY,
      topic: input.topic ?? "general",
    });
    return await tavilySearch._call({ query: input.query });
  },
  {
    name: "internet_search",
    description:
      "在互联网上搜索最新信息，适合查询实时资讯、新闻、技术动态、人物介绍等。当问题涉及近期发生的事或模型知识截止日期之后的内容时，优先使用此工具。",
    schema: z.object({
      query: z.string().describe("要搜索的问题或主题，尽量具体"),
      maxResults: z
        .number()
        .optional()
        .default(5)
        .describe("返回结果条数，默认 5"),
      topic: z
        .enum(["general", "news", "finance"])
        .optional()
        .default("general")
        .describe("搜索主题类别：general 通用、news 新闻、finance 金融"),
    }),
  },
);

const tools = [getWeather, internetSearch];
const toolsByName = Object.fromEntries(tools.map((t) => [t.name, t]));

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "你的 DeepSeek 密钥",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const modelWithTools = model.bindTools(tools);
```

密钥建议从环境变量读取，不要写进版本库。`bindTools` 返回的 `modelWithTools` 在每次 `invoke` 时都能看到这一组工具，会根据用户输入和工具描述自己决定调哪个。对模型来说，它只根据名称和描述做决策，具体怎么调 API、怎么处理返回，都封装在工具函数里，模型不关心。

下图是"工具定义、绑定模型、模型决策"这一准备阶段的整体结构。

![20260314235339](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260314235339.png)

图里从上到下依次是工具三要素（名称、描述、schema）、工具数组与 `toolsByName` 映射、`bindTools` 把工具挂到模型的过程，以及模型在每次 `invoke` 时按工具描述做决策。最下面一层表示 API 调用细节都封在工具函数内部，对模型不可见。

## 第二步 实现 Agent 循环

Agent 的核心就是这段循环。先拿用户输入调一次模型，若返回的 `AIMessage` 里有 `tool_calls`，就把这条 `AIMessage` 和每个工具的执行结果（`ToolMessage`）依次追进 `messages`，再调一次模型，如此重复直到没有新的 `tool_calls`。同时加一个最大轮数（例如 5 轮），防止模型一直请求工具导致死循环。

下面把这段逻辑封装成 `runAgent(userInput)`，便于后面复用和测试。

```typescript
import type { BaseMessage } from "@langchain/core/messages";
import { HumanMessage, ToolMessage } from "@langchain/core/messages";

const MAX_TOOL_ROUNDS = 5;

async function runAgent(userInput: string): Promise<string> {
  const messages: BaseMessage[] = [new HumanMessage(userInput)];
  let aiMsg = await modelWithTools.invoke(messages);
  let rounds = 0;

  while (aiMsg.tool_calls?.length && rounds < MAX_TOOL_ROUNDS) {
    rounds++;
    messages.push(aiMsg);

    for (const tc of aiMsg.tool_calls) {
      const toolFn = toolsByName[tc.name];
      try {
        if (toolFn) {
          const result = await toolFn.invoke(tc);
          messages.push(result);
        } else {
          messages.push(
            new ToolMessage({
              content: `未知工具：${tc.name}`,
              tool_call_id: tc.id,
            }),
          );
        }
      } catch (err: unknown) {
        const message = err instanceof Error ? err.message : String(err);
        messages.push(
          new ToolMessage({
            content: `工具执行失败：${message}`,
            tool_call_id: tc.id,
          }),
        );
      }
    }

    aiMsg = await modelWithTools.invoke(messages);
  }

  if (aiMsg.tool_calls?.length) {
    console.warn(
      `已达最大工具轮数（${MAX_TOOL_ROUNDS} 轮），强制返回当前回复。`,
    );
  }

  return aiMsg.content as string;
}
```

循环里用 `toolsByName` 把 `tool_call.name` 映射到具体工具实例，不用每次遍历数组查找。`MAX_TOOL_ROUNDS` 是安全阀，防止模型反复请求工具造成死循环。某轮一旦没有 `tool_calls`，说明模型已经拿到足够信息，循环自然结束。

遇到"未知工具名"或"工具执行抛错"时，都用带对应 `tool_call_id` 的 `ToolMessage` 把错误信息回传给模型，模型可以在回复里说明或道歉，整轮对话不必直接报错退出。

整个循环的流向如下图所示。

![20260315093647](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260315093647.png)

从用户输入触发到模型判断、工具执行、轮数检查、最终回复的完整 Agent 循环。

每个 `ToolMessage` 都要带上对应的 `tool_call_id`，模型靠它区分"哪条结果对应哪次工具请求"。`tool_call_id` 对不上，模型会困惑，回复容易答非所问。

## 第三步 接上入口并跑一次

在脚本末尾传入一个问题并打印助手回复即可。例如同时问天气和搜索，模型会先产生两个 `tool_calls`（或分两轮），执行完后根据工具结果生成一段总结。

```typescript
const userQuestion =
  "深圳今天天气怎么样？另外帮我搜一下 LangChain 最近有什么新动态。";
console.log("用户：", userQuestion);
console.log("---");

runAgent(userQuestion).then((reply) => {
  console.log("助手：", reply);
});
```

运行后终端会先打出用户提问和分隔线，再打出助手的回复。回复里会包含深圳的真实天气预报（来自 `get_weather`）和 LangChain 的搜索摘要（来自 `internet_search`），说明两个工具都被正确调用并把结果汇总到了一段自然语言里。

若要换问题，改 `userQuestion` 即可；若要增加新工具，往 `tools` 数组里加一项，`toolsByName` 的构建逻辑不用动。

## 整条链路长什么样

从用户输入到最终回复，消息在"用户、模型、工具"之间的流转可以概括成下图。用户消息进入后，模型可能先返回 `tool_calls`；应用按名称执行工具、把每条结果写成 `ToolMessage` 追进 `messages`，再调模型；如此循环直到模型不再返回 `tool_calls`，此时最后一条 `AIMessage` 的 `content` 就是面向用户的最终回复。

![20260215113705](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260215113705.png)

理解这条链路后，排查"模型为什么没用工具结果"或"为什么重复调同一个工具"会容易很多：重点看 `messages` 里 `AIMessage` 与 `ToolMessage` 的顺序，以及每条 `ToolMessage` 的 `tool_call_id` 是否与前面的 `tool_calls[].id` 一致。

## 加上系统提示来定调 Agent 的行为

默认情况下模型只根据工具描述和用户输入决定行为。若希望 Agent 固定某种风格或约束（例如"回答要简短"、"只讨论技术话题"），可以在 `messages` 最前面加一条 `SystemMessage`。模型会把系统消息当作全局设定，后续用户消息与工具结果都在这个设定下理解。

下面在 `runAgent` 里把第一条消息改成"系统消息 + 用户消息"，这样每次调用都会带上同一句系统提示：

```typescript
import { SystemMessage } from "@langchain/core/messages";

async function runAgent(userInput: string): Promise<string> {
  const messages: BaseMessage[] = [
    new SystemMessage(
      "你是一个助手，负责查询天气和搜索互联网信息。回答请简洁，用一到两句话概括结果。",
    ),
    new HumanMessage(userInput),
  ];
  let aiMsg = await modelWithTools.invoke(messages);
  let rounds = 0;
  // while 循环和工具执行部分与第二步一致，原样接上即可
}
```

系统提示可以按需求写长写短，例如禁止幻觉、要求引用工具结果、或限定领域，都能通过这一段统一约束。工具描述解决的是"模型知道有哪些工具、什么时候用"，而系统提示解决的是"模型整体应该以什么态度和风格去使用这些工具"，两者配合才能让 Agent 行为可预期。

## 可选：流式输出最终回复

目前我们是等整条 Agent 跑完（包括所有工具轮次）再一次性返回 `aiMsg.content`。若希望最后一轮模型的文字回复能边生成边输出，可以在"已经没有 `tool_calls`、准备输出最终回复"的那一轮改用 `modelWithTools.stream(messages)`，在循环里打印或推送每个 chunk 的文本；流式结束后，再根据需求决定是否把拼好的完整消息追进 `messages` 以便后续多轮对话。

有一点需要注意：工具调用那一轮必须等完整的 `tool_calls` 收齐后再执行，不能根据流式过程中不完整的 chunk 去调工具，否则参数可能不完整或错误。实践上通常的做法是，流式只用于最终回复展示阶段，工具执行阶段仍然是同步等待完整结果。

## 小结

构建第一个智能体，就是把"接入模型、配备工具、消息循环"三件事拼在一起。

用 `ChatOpenAI`（或其它聊天模型类）接入大模型，用 `tool()` 定义一组工具并用 `bindTools` 绑到模型上；在应用里维护一个 `messages` 数组，先塞入用户输入（可选加一条 `SystemMessage`），然后循环：若模型返回 `tool_calls`，就按名称执行、把结果写成 `ToolMessage` 追进 `messages`、再 `invoke`，直到没有新的 `tool_calls`，最后一条 `AIMessage` 的 `content` 即为最终回复。

注意最大工具轮数、未知工具名与执行失败时用 `ToolMessage` 回传错误信息，这样一个可复用的 Agent 入口（如 `runAgent`）就搭好了。

后续若要扩展，只需增加工具定义、在 `tools` 和 `toolsByName` 里注册，或调整系统提示与模型参数即可。
