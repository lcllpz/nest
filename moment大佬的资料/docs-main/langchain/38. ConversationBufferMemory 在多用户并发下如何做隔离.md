上一篇我们把 `Memory` 的几种类型和选型逻辑梳理清楚了，接下来要解决另一个在生产里几乎必然会踩的问题，多个用户同时在用，`Memory` 该怎么隔离。这个问题在本地 `demo` 里完全看不出来，因为只有一条会话。一旦服务上线，多个会话交错进来，稍有不慎就会出现两个经典故障：

- 串话，用户 A 的对话内容被带进了用户 B 的上下文
- 数据泄露，一个会话的敏感字段被另一个会话读到

这两类问题看起来像偶发 `bug`，实际上都是隔离模型没设计好。这一篇围绕一个核心问题展开，在多用户并发下，怎样让 `Memory` 的读写既高效又隔离得住。

## 串话和泄露是怎么发生的

先看一个很常见的错误写法。很多人第一次写带记忆的服务时，会在模块顶层直接初始化一个全局 `Memory`，然后所有请求复用它。

```typescript
import { BufferMemory } from "langchain/memory";

const memory = new BufferMemory();

export async function handleRequest(input: string): Promise<string> {
  await memory.saveContext({ input }, { output: "..." });
  return "ok";
}
```

这段代码在单用户本地测试时一切正常。上线之后问题马上就来：

- `memory` 是模块级单例，所有请求共享同一块状态
- 用户 A 写入后立刻被用户 B 读到，历史混在一起
- 请求之间没有顺序保证，写入顺序还可能乱

表面上是多线程安全问题，本质上是作用域设计错了。`Memory` 本来就应该和一次会话绑定，而不是和进程生命周期绑定。

## 隔离的第一原则是按 key 取实例

正确的做法是把 `Memory` 的作用域收到会话一级。每一次请求进来，先从某个标识里取出会话键，再按这个键拿到对应的 `Memory` 实例。常见的会话键有三种：

- `userId`，按用户隔离，适合单会话多页面但共享记忆的场景
- `sessionId`，按会话隔离，适合一个用户可以开多个独立对话
- `threadId`，按线程隔离，常见于 `LangGraph` 里的持久化状态

生产里通常是 `userId + sessionId` 组合成唯一键，既能做权限控制，又能支持多会话。下面这段示意展示了最小的按键取实例思路。

```typescript
import { ChatMessageHistory } from "@langchain/community/stores/message/in_memory";

const store = new Map<string, ChatMessageHistory>();

function getHistory(sessionKey: string): ChatMessageHistory {
  if (!store.has(sessionKey)) {
    store.set(sessionKey, new ChatMessageHistory());
  }
  return store.get(sessionKey)!;
}
```

这段代码的关键点不是 `Map`，而是 `sessionKey` 的生成规则和存储介质。换成 `Redis` 或数据库，函数签名可以不变，只是 `getHistory` 内部从进程内读取换成跨进程读取。

把它接进 `LCEL` 链路，官方推荐的写法是 `RunnableWithMessageHistory`。它是对上面这套逻辑的标准封装，避免我们在每条链里都手写一遍。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate, MessagesPlaceholder } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import { RunnableWithMessageHistory } from "@langchain/core/runnables";
import { ChatMessageHistory } from "@langchain/community/stores/message/in_memory";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一位客服助手，请结合历史对话回答用户。"],
  new MessagesPlaceholder("history"),
  ["human", "{input}"],
]);

const chain = prompt.pipe(model).pipe(new StringOutputParser());

const store = new Map<string, ChatMessageHistory>();

const withHistory = new RunnableWithMessageHistory({
  runnable: chain,
  getMessageHistory: (sessionId: string) => {
    if (!store.has(sessionId)) store.set(sessionId, new ChatMessageHistory());
    return store.get(sessionId)!;
  },
  inputMessagesKey: "input",
  historyMessagesKey: "history",
});

async function main(): Promise<void> {
  await withHistory.invoke(
    { input: "我叫小白" },
    { configurable: { sessionId: "u-1:s-1" } },
  );
  const res = await withHistory.invoke(
    { input: "你记得我是谁吗" },
    { configurable: { sessionId: "u-1:s-1" } },
  );
  console.log(res);
}

main().catch(console.error);
```

这里的 `sessionId` 就是前面说的会话键。只要每次请求都带对 `sessionId`，链路内部拿到的就是独立的历史，不会相互污染。

## 进程内 Map 只能撑到 demo 阶段

上面那段示例里，`Map` 放在进程内，一旦服务重启所有历史就没了，开多副本还会出现不同实例状态不一致的问题。生产里必须把存储换成跨进程的持久化介质。常见的三种选择和适用场景如下。

| 介质         | 适合场景                  | 主要代价               |
| ------------ | ------------------------- | ---------------------- |
| `Redis`      | 活跃会话、高频读写        | 需要自己做持久化和备份 |
| 关系型数据库 | 审计、可追溯、低频读写    | 延迟高，不适合每轮写入 |
| 文档数据库   | 历史结构半结构化、查询灵活 | 成本介于两者之间       |

大多数线上项目的默认答案是 `Redis` 加数据库分层。热数据放 `Redis`，会话结束或定时归档到数据库。下面给一段基于 `Redis` 的会话存储骨架，便于看到替换点在哪里。

```typescript
import { createClient, RedisClientType } from "redis";
import { BaseChatMessageHistory } from "@langchain/core/chat_history";
import { BaseMessage, mapStoredMessagesToChatMessages, mapChatMessagesToStoredMessages } from "@langchain/core/messages";

class RedisChatHistory extends BaseChatMessageHistory {
  lc_namespace = ["langchain", "memory", "redis"];

  constructor(
    private readonly client: RedisClientType,
    private readonly key: string,
    private readonly ttlSeconds: number,
  ) {
    super();
  }

  async getMessages(): Promise<BaseMessage[]> {
    const raw = await this.client.lRange(this.key, 0, -1);
    const stored = raw.map((item) => JSON.parse(item));
    return mapStoredMessagesToChatMessages(stored);
  }

  async addMessage(message: BaseMessage): Promise<void> {
    const [stored] = mapChatMessagesToStoredMessages([message]);
    await this.client.rPush(this.key, JSON.stringify(stored));
    await this.client.expire(this.key, this.ttlSeconds);
  }

  async clear(): Promise<void> {
    await this.client.del(this.key);
  }
}

const redis: RedisClientType = createClient({ url: "redis://127.0.0.1:6379" });
await redis.connect();

function getRedisHistory(sessionKey: string): RedisChatHistory {
  return new RedisChatHistory(redis, `chat:history:${sessionKey}`, 60 * 60 * 24);
}
```

这段骨架有三个细节值得注意。第一，`key` 的命名一定要带前缀和会话键，方便运维按前缀清理。第二，每次写入都刷新 `TTL`，让活跃会话自动续期。第三，所有读写都过 `mapStoredMessagesToChatMessages` 这类官方方法，避免序列化格式不一致。

## TTL 和清理策略怎么设计

会话数据堆在 `Redis` 里不处理，内存很快就撑爆。清理策略要在三个维度上同时起作用。

- 活跃性过期，每次读写都续期，不活跃的会话到期自动淘汰
- 硬上限过期，给每个 `key` 设置一个绝对最大寿命，避免 `TTL` 被无限续期
- 主动归档，会话结束或达到一定长度时，把数据搬到数据库并清掉 `Redis` 里的 `key`

硬上限常被忽略。只有活跃续期的系统，会在长尾用户身上积累大量僵尸会话，表面还活着，实际再也不会被访问。推荐做法是给每条会话的第一条消息里记录 `createdAt`，写入时如果超过上限就直接截断或归档。

归档链路也不只是数据层的事。对需要可追溯的业务，比如客服、金融、医疗，归档时要记录完整的原文加引用链，不能只留摘要。否则事后复查时会发现摘要丢了关键字段，对方还咬住当时说过什么，争议很难化解。

## 线程安全和共享可变状态

讨论并发绕不开线程安全。`Node.js` 单进程里的异步并发，其实不会像多线程那样需要加锁，真正的风险在两个地方。

- 多个请求几乎同时操作同一 `sessionKey`，比如两次写入都发生在摘要计算过程中
- 多个副本实例同时读写同一 `Redis key`，顺序不可控

对第一类，最简单的做法是让同一 `sessionKey` 的请求串行处理。可以用一个内存级锁，或者在 `Redis` 层加分布式锁。对第二类，推荐用 `Redis` 的 `list` 结构并配合 `append only` 语义，所有写入只能追加，读取时按时间戳排序。不要在 `Redis` 里用 `GET` 加 `SET` 这种读后改写的模式，容易丢写。

下面是一段基于 `Redis` 的会话级锁示例，用于保护摘要更新这种非幂等操作。

```typescript
import type { RedisClientType } from "redis";

async function withSessionLock<T>(
  redis: RedisClientType,
  sessionKey: string,
  ttlMs: number,
  fn: () => Promise<T>,
): Promise<T> {
  const lockKey = `lock:session:${sessionKey}`;
  const token = `${Date.now()}-${Math.random()}`;
  const ok = await redis.set(lockKey, token, { NX: true, PX: ttlMs });
  if (!ok) throw new Error("session is busy, please retry");
  try {
    return await fn();
  } finally {
    const current = await redis.get(lockKey);
    if (current === token) await redis.del(lockKey);
  }
}
```

这段锁实现并没有追求极致完善，重点是两件事。第一，用 `NX + PX` 保证原子抢锁和自动过期。第二，释放锁时要校验 `token`，避免误删别人的锁。真正做高可用锁可以考虑 `Redlock`，但大多数会话级写入用这种轻量锁已经够用。

读多写多的场景还要留意一个细节，不要在锁里做任何模型调用。模型调用延迟不可控，锁里放慢操作会让同会话的后续请求全部排队等待。比较好的拆法是把模型调用放在锁外，只把最终写入动作放在锁内。

## 在 LangGraph 里天然就是隔离的

如果项目已经走到了 `LangGraph`，隔离问题其实更轻。`LangGraph` 的 `checkpointer` 天生就是按 `thread_id` 组织的，每个线程自己一份状态，互不干扰。

```typescript
import { StateGraph, MessagesAnnotation } from "@langchain/langgraph";
import { MemorySaver } from "@langchain/langgraph";
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const graph = new StateGraph(MessagesAnnotation)
  .addNode("chat", async (state) => {
    const res = await model.invoke(state.messages);
    return { messages: [res] };
  })
  .addEdge("__start__", "chat")
  .addEdge("chat", "__end__")
  .compile({ checkpointer: new MemorySaver() });

await graph.invoke(
  { messages: [{ role: "user", content: "你好" }] },
  { configurable: { thread_id: "u-1:s-1" } },
);
```

`thread_id` 的角色就是前面的 `sessionKey`，`checkpointer` 负责按线程落盘。`MemorySaver` 是进程内实现，生产里换成 `PostgresSaver`、`RedisSaver` 这类持久化版本即可。好处是隔离、持久化、断点续跑都由框架统一处理，不用再自己拼装一套。

## 总结

多用户并发下的 `Memory` 隔离看起来是个工程细节，往深处挖其实就是三件事。

- 作用域控制，每条会话一份实例，`Memory` 的生命周期不能和进程绑在一起
- 存储选型，进程内只能做 `demo`，生产必须上 `Redis` 或数据库，并设计好 `TTL` 和归档
- 并发保护，同一会话的非幂等写入要串行，跨副本写入要用原子操作

把这三件事放到一张图里就是前面那张全景图。现实工程里还会叠加权限、审计、合规这些额外约束，但核心的隔离模型不会变。

理解清楚这三件事之后，新写或重构一个带记忆的服务，上线前可以对着下面几项逐条过一遍，基本就能堵住多用户并发场景下的主要坑。

- 会话键是否全链路贯穿，日志、监控、错误上下文里都带得上
- 存储层是否可持久化、多副本一致，能不能抗单点故障
- 是否同时有活跃 `TTL` 和硬上限两层过期，会话既不会泄露也不会堆积
- 并发写入有没有做串行保护，摘要更新这类非幂等操作有没有加锁
- 链路里有没有任何模块级共享的 `Memory` 单例或者误用的全局变量
- 归档链路是否保留了可追溯所需的原文，而不是只留一份摘要

这几条都通过，串话和泄露这两个经典故障基本就不会再出现。剩下的失效模式大多和业务逻辑挂钩，比如权限校验不到位、`sessionId` 生成规则不安全，这些就已经不在 `Memory` 本身的范畴里了，属于业务侧另外要处理的事。
