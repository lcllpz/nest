前面的内容里讲到的 `Agent` 已经能接用户输入、触发工具调用、把结果写回对话并给出最终回复，但它还有一个很现实的缺口。每次调用 `runAgent(userInput)`，函数内部都会重新创建只包含本轮那句话的 `messages`，模型看不到之前聊过什么。比如先问 深圳天气怎么样，再接一句 那上海呢，模型只能按字面理解， 那 指的语境就丢了。

要让 `Agent` 真正像在聊天，就需要在多次调用之间保留并复用同一份对话历史，也就是给 `Agent` 加上会话记忆。接下来先讲清 `Agent` 为什么会失忆，记忆应该落在链路的哪一层。然后给出两种实现：一种由调用方维护 `messages` 并逐轮追加新消息，另一种用会话 ID 区分多用户与多会话，让每个会话各自拥有一份历史。最后再讨论历史变长后的截断策略，避免超出上下文窗口，或把 token 成本拉得过高。

## 为什么 Agent 会失忆

大模型 API 是无状态的，每次请求都是一次独立的 HTTP 调用，请求体里带什么消息，模型就看到什么，服务端不会替你保存上一次的对话。这是大模型 API 的设计约定，不是 bug。记住上文完全是应用层的责任，要么在每次请求里把需要的历史消息一并塞进 `messages`，要么接受模型永远只看到当前这一句。

在前面的 `runAgent` 示例里，`messages` 在函数内部声明，初始值只包含当前用户输入：

```typescript
async function runAgent(userInput: string): Promise<string> {
  const messages: BaseMessage[] = [new HumanMessage(userInput)];
  // ...
}
```

两次连续调用时，模型实际收到的 `messages` 是这样的：

```
// 第一次调用：问深圳天气
messages = [
  HumanMessage("深圳今天天气怎么样？")
]

// 第二次调用：追问上海
messages = [
  HumanMessage("那上海呢？")   // 上一轮的所有内容全部消失
]
```

模型在第二次请求里看不到任何关于深圳的内容。此时 那 指的语境只能按字面猜测。于是像 那上海呢、再搜一次、把刚才两个城市的气温比一下 这类追问就很难自然接上。

会话记忆要做的，就是让多轮调用共享同一份 `messages`，让后续请求自动携带上下文。

## 会话记忆在链路中的位置

带会话记忆的 Agent 处理一条用户消息的完整链路如下：

1. 收到用户输入，取出或初始化该会话的 `messages`。
2. 把本条输入包装成 `HumanMessage` 追加到 `messages`。
3. 带着整份 `messages` 调用模型，可能触发多轮工具调用。
4. 把本轮产生的 `AIMessage`、`ToolMessage` 等全部追加到 `messages`。
5. 返回最终回复，`messages` 保持最新状态，等待下一轮。

会话记忆的本质就是一个"按会话维护的消息数组"。调用模型前，它是本轮的上下文输入；调用模型后，它更新为本轮结束后的新状态。

如下图所示。

![20260315185711](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260315185711.png)

每轮产生的消息追进同一份列表，不同会话各自维护独立历史。

实现上有两种常见做法。第一种由调用方持有一份 `messages`，每次调用时传入，函数跑完后数组里已追加了本轮所有新消息。第二种在应用层维护一个"会话 ID → 消息列表"的映射，根据请求里的会话 ID 取出对应列表、追加、再写回。下面先讲第一种，再讲第二种。

## 做法一：由调用方维护 messages

思路很直接，不再在 `runAgent` 内部新建 `messages`，而是让调用方传入当前会话已有的消息列表。函数在本轮只负责追加本条用户消息、跑完模型与工具循环、把本轮产生的新消息全部追进这份列表，最后返回最终回复。调用方只要在多次调用之间复用同一个数组，就自然形成了会话记忆。

工具定义与第 6、7 章一致，直接复用 `getWeather` 和 `internetSearch`。核心改动在函数签名，从 `runAgent(userInput)` 改为 `runAgentWithMemory(messages, userInput)`，`messages` 由调用方维护并传入，函数内部不再 `new` 新数组。

```typescript
import { tool } from "@langchain/core/tools";
import { TavilySearch } from "@langchain/tavily";
import { z } from "zod";
import { ChatOpenAI } from "@langchain/openai";
import type { BaseMessage } from "@langchain/core/messages";
import { HumanMessage, SystemMessage, ToolMessage } from "@langchain/core/messages";

// getWeather 和 internetSearch 的完整定义见第 6 章，此处直接使用
// ...

const tools = [getWeather, internetSearch];
const toolsByName = Object.fromEntries(tools.map((t) => [t.name, t]));

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "你的 DeepSeek 密钥",
  configuration: { baseURL: "https://api.deepseek.com" },
});
const modelWithTools = model.bindTools(tools);

const MAX_TOOL_ROUNDS = 5;

async function runAgentWithMemory(
  messages: BaseMessage[],
  userInput: string,
): Promise<string> {
  messages.push(new HumanMessage(userInput));

  let aiMsg = await modelWithTools.invoke(messages);
  let rounds = 0;

  while (aiMsg.tool_calls?.length && rounds < MAX_TOOL_ROUNDS) {
    rounds++;
    messages.push(aiMsg);

    for (const tc of aiMsg.tool_calls) {
      const toolFn = toolsByName[tc.name];
      try {
        if (toolFn) {
          const result = await toolFn.invoke(tc);
          messages.push(result);
        } else {
          messages.push(
            new ToolMessage({
              content: `未知工具：${tc.name}`,
              tool_call_id: tc.id,
            }),
          );
        }
      } catch (err: unknown) {
        const message = err instanceof Error ? err.message : String(err);
        messages.push(
          new ToolMessage({
            content: `工具执行失败：${message}`,
            tool_call_id: tc.id,
          }),
        );
      }
    }

    aiMsg = await modelWithTools.invoke(messages);
  }

  if (aiMsg.tool_calls?.length) {
    console.warn(`已达最大工具轮数（${MAX_TOOL_ROUNDS} 轮），强制返回当前回复。`);
  }

  messages.push(aiMsg);
  return aiMsg.content as string;
}
```

注意最后多了一行 `messages.push(aiMsg)`。第 7 章的 `runAgent` 在拿到最终回复后直接返回，不把这条 `AIMessage` 写进列表。而这里必须把它追进去，这样下一轮 `invoke` 时模型才能看到"上一轮助手说了什么"，指代和追问才能正常工作。这是改造时最容易漏掉的一步。

调用方维护一个 `sessionMessages` 数组，在多次 `chat()` 之间复用它：

```typescript
const sessionMessages: BaseMessage[] = [
  new SystemMessage(
    "你是一个助手，负责查询天气和搜索互联网信息。回答请简洁，用一到两句话概括结果。",
  ),
];

async function chat(userInput: string): Promise<string> {
  console.log("用户：", userInput);
  const reply = await runAgentWithMemory(sessionMessages, userInput);
  console.log("助手：", reply);
  console.log("---");
  return reply;
}

chat("深圳今天天气怎么样？")
  .then(() => chat("那上海呢？"))
  .then(() => chat("刚才两个城市里哪个更热？"));
```

三次对话跑完后，`sessionMessages` 的状态大致如下：

```
SystemMessage    "你是一个助手…"
HumanMessage     "深圳今天天气怎么样？"
AIMessage        [tool_calls: getWeather("深圳")]
ToolMessage      "深圳今日晴，28℃…"
AIMessage        "深圳今天晴，气温约 28℃。"
HumanMessage     "那上海呢？"
AIMessage        [tool_calls: getWeather("上海")]
ToolMessage      "上海今日多云，24℃…"
AIMessage        "上海今天多云，气温约 24℃。"
HumanMessage     "刚才两个城市里哪个更热？"
AIMessage        "深圳更热，28℃ 比上海 24℃ 高约 4 度。"
```

模型在第三轮时能直接从历史里读出两座城市的气温，不需要再调工具，回答也更自然。

`SystemMessage` 放在数组首位有一个约定，它不属于任何一轮用户交互，是贯穿整个会话的角色设定，做截断时应当始终保留。

## 做法二：按会话 ID 存多份历史

单用户、单会话时，一个全局的 `sessionMessages` 就够用。若有多用户或同一用户开了多个对话（例如多个标签页、多个客服会话），就需要按会话区分历史，否则所有对话会混在同一份 `messages` 里。

做法是给每个会话一个 ID，在应用里维护一个"会话 ID → 消息数组"的映射。每次请求带上会话 ID，根据 ID 取出对应的 `messages`，若没有则初始化一份（可带一条默认的 `SystemMessage`），然后追加用户消息、跑 Agent、把新消息追进该数组，最后返回回复。

```typescript
import type { BaseMessage } from "@langchain/core/messages";
import { SystemMessage } from "@langchain/core/messages";

const sessionStore = new Map<string, BaseMessage[]>();

function getOrCreateSession(sessionId: string): BaseMessage[] {
  if (!sessionStore.has(sessionId)) {
    sessionStore.set(sessionId, [
      new SystemMessage(
        "你是一个助手，负责查询天气和搜索互联网信息。回答请简洁，用一到两句话概括结果。",
      ),
    ]);
  }
  return sessionStore.get(sessionId)!;
}

async function chatBySession(
  sessionId: string,
  userInput: string,
): Promise<string> {
  const messages = getOrCreateSession(sessionId);
  return runAgentWithMemory(messages, userInput);
}
```

实际接入 HTTP 接口时，`sessionId` 可从请求头、Cookie 或登录态里取：

```typescript
app.post("/chat", async (req, res) => {
  const { sessionId, message } = req.body as {
    sessionId: string;
    message: string;
  };
  const reply = await chatBySession(sessionId, message);
  res.json({ reply });
});
```

若用 LangChain 的 `InMemoryStore`，可以把"会话 ID → 序列化后的消息列表"存进去，需要时反序列化再传给 `runAgentWithMemory`。逻辑相同，只是存储从内存 `Map` 换成了可插拔的 store，便于将来替换成 Redis 或数据库。

多会话时还要考虑过期与清理。某会话超过一定时间没有新消息就该删掉或归档，否则内存会无限增长。可以在 `Map` 旁边再维护一份"会话 ID → 最后活跃时间"的映射，定时扫描并清理超过 TTL 的会话，例如下面这样：

```typescript
const sessionLastActive = new Map<string, number>();
const SESSION_TTL_MS = 30 * 60 * 1000; // 30 分钟无活动则过期

function getOrCreateSession(sessionId: string): BaseMessage[] {
  sessionLastActive.set(sessionId, Date.now());
  if (!sessionStore.has(sessionId)) {
    sessionStore.set(sessionId, [
      new SystemMessage(
        "你是一个助手，负责查询天气和搜索互联网信息。回答请简洁，用一到两句话概括结果。",
      ),
    ]);
  }
  return sessionStore.get(sessionId)!;
}

function cleanExpiredSessions(): void {
  const now = Date.now();
  for (const [id, lastActive] of sessionLastActive.entries()) {
    if (now - lastActive > SESSION_TTL_MS) {
      sessionStore.delete(id);
      sessionLastActive.delete(id);
    }
  }
}

// 每 5 分钟清理一次
setInterval(cleanExpiredSessions, 5 * 60 * 1000);
```

多会话下，请求进来后按会话 ID 查表、取消息或建新会话，再跑 Agent、写回，整条链路如下图所示。

## 历史变长后的截断策略

对话轮次一多，`messages` 会越来越长，带来两个问题。第一是可能超过模型上下文长度限制，请求直接报错。第二是 token 用量和费用会随历史长度线性增长。因此生产环境里通常不会无上限地保留全部历史，而是做截断或压缩。

常见做法有三种：

- 滑动窗口：只保留最近 N 条消息。若 `messages` 长度超过阈值，就只保留第一条（通常是 `SystemMessage`）和最后若干条，中间的删掉。实现最简单，但模型无法回答很久以前的细节。
- 按 token 数截断：统计当前 `messages` 的 token 总数，若超过设定上限（例如 8000），就从最旧的消息开始删，直到总 token 低于上限。比滑动窗口更精确，但需要额外引入 token 计数库。
- 摘要压缩：对较早的对话调一次模型做摘要，用一条摘要消息替代大段旧消息，再保留最近几轮原文。既控制长度，又保留一部分长期信息，实现成本相对高一些。

如下图所示。

![20260315194154](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260315194154.png)

三种策略在实现难度、信息保留度、成本控制上各有取舍。

下面是"只保留系统消息 + 最近 K 条消息"的滑动窗口实现。在调用 `runAgentWithMemory` 之前先做裁剪，若 `messages` 长度大于 `maxMessages`，就只保留第一条（系统消息）和最后 `maxMessages - 1` 条，避免数组无限增长：

```typescript
const MAX_MESSAGES = 30;

function trimMessages(
  messages: BaseMessage[],
  maxMessages: number = MAX_MESSAGES,
): void {
  if (messages.length <= maxMessages) return;
  const systemMsg = messages[0];
  const recent = messages.slice(-(maxMessages - 1));
  messages.length = 0;
  messages.push(systemMsg, ...recent);
}

async function runAgentWithMemory(
  messages: BaseMessage[],
  userInput: string,
): Promise<string> {
  trimMessages(messages);
  messages.push(new HumanMessage(userInput));
  // ... 其余部分与前面一致
}
```

若消息平均体积较大，可以改成按 token 数截断。下面用 `tiktoken` 统计 token，超过上限后从最旧的非系统消息开始逐条删除：

```typescript
import { encodingForModel } from "tiktoken";

const TOKEN_LIMIT = 8000;

function countTokens(messages: BaseMessage[]): number {
  const enc = encodingForModel("gpt-4o");
  return messages.reduce((total, msg) => {
    const text = typeof msg.content === "string" ? msg.content : JSON.stringify(msg.content);
    return total + enc.encode(text).length + 4; // 每条消息约 4 个 token 的格式开销
  }, 0);
}

function trimMessagesByToken(messages: BaseMessage[]): void {
  while (countTokens(messages) > TOKEN_LIMIT && messages.length > 2) {
    // 保留 index 0 的 SystemMessage，从 index 1 开始删
    messages.splice(1, 1);
  }
}
```

`trimMessages` 和 `trimMessagesByToken` 都放在追加用户消息之前调用，这样每轮都先裁掉过旧的消息再追加新的一轮。

三种策略并不互斥。实际项目里可以组合使用，例如先做一次 token 统计，若超过阈值就对最老的若干轮做摘要，再保留最近 N 轮原文，这样兼顾了成本控制与信息密度。

## 小结

第 7 章的 Agent 每次调用都从零开始建 `messages`，模型看不到之前的对话，无法做多轮追问和指代。会话记忆就是在多次调用之间共享同一份消息列表。每轮把用户输入追进这份列表，用整份列表跑完模型与工具循环，再把本轮产生的 `AIMessage` 与 `ToolMessage` 追进同一份列表，下一轮就能带着完整上文继续。

实现上可以只由调用方维护一个 `messages` 数组，每次调用 `runAgentWithMemory(messages, userInput)` 并复用该数组。注意在函数末尾把最终的 `AIMessage` 也追进 `messages`，否则下一轮模型看不到上一轮的助手回复。若有多会话，就用会话 ID 映射到多份 `messages`（例如内存 `Map`、`InMemoryStore` 或 Redis），按会话取用与更新，不同会话的历史互不干扰。同时建议维护会话最后活跃时间，定期清理超时的会话，避免内存无限增长。

历史变长后要做截断或压缩。常用方式有三种，分别是只保留最近 N 条消息、按 token 上限裁剪、对旧对话做摘要再保留最近几轮原文。三种策略可按场景组合使用。加上会话记忆之后，Agent 就具备了会话内的短期记忆，能连贯地处理多轮对话和依赖上下文的追问。
