前面几章里，工具用 `zod` 描述的是"模型要传给工具的入参长什么样"，属于输入侧的结构化。还有另一类需求，你希望模型直接返回一段可程序化处理的数据，例如一段摘要里必须包含"标题、要点列表、结论"三个字段，或从用户句子里抽出的"人名、日期、事件"要落进固定字段，方便你写库、调接口、做校验。这类需求关心的是模型输出长什么样，也就是让 AI 返回可控的结构化数据。

本章先说明为什么光靠自然语言不够、结构化输出和工具参数的区别，再介绍两种常见做法，一是在提示里塞进格式说明、用解析器把模型回复转成对象，二是用聊天模型自带的 `withStructuredOutput`，让模型按你给的 schema 直接产出可解析的结果。两种方式都会用到 `zod`，和工具那边的用法一脉相承。

## 为什么需要可控的结构化输出

普通调用里，`invoke` 返回的 `AIMessage.content` 是一段自然语言。若你要把"北京今天 25 度、晴"存进数据库的"城市、温度、天气"字段，或把"张三、2024 年 3 月、签约"填进表单，就得自己在代码里解析这段文字。用正则或字符串切分非常脆弱，模型换一种说法或多打一个标点就可能解析失败。字段一多、嵌套一深，维护成本也会很高。

若能在请求时就约定"模型必须按某种结构返回"，例如一个 JSON，且形状由你定的 schema 描述，那么下游拿到手的就是已经按字段分好的对象，直接做类型校验、入库或转发 API 即可。这样既减少解析错误，又便于和 TypeScript 类型、数据库表结构对齐。

和工具调用的区别在于，工具那边是"模型调用我们时，我们要求参数符合 schema"，这里则是"我们调用模型时，要求模型的回复符合 schema"，一个管输入、一个管输出。

## 两种实现思路

要让模型输出可控，大致有两种路子：

- "提示加解析"：在系统或用户消息里明确写出"请用如下 JSON 格式回复"，并把格式说明（例如由 schema 生成的说明文本）塞进提示。模型仍返回普通文本，你在应用里用解析器（如 LangChain 的 `StructuredOutputParser`）把这段文本解析成对象。这种方式依赖模型"听话"地按格式写，若模型多输出了说明文字或漏了字段，解析可能失败，必要时可做重试或后处理。
- "模型层约束"：用聊天模型的 `withStructuredOutput(schema)` 得到一个 runnable，你 `invoke` 时传入消息，返回的已经不是原始 `AIMessage`，而是按 schema 解析好的对象。底层可能通过 function calling 或厂商的 JSON mode 等方式实现，因厂商和配置而异。这种方式通常更稳定，因为 API 或模型会尽量保证输出落在给定结构内。

## 格式说明放进提示并用解析器解析（做法一）

思路是用 `zod` 定义你期望的输出结构，用 `StructuredOutputParser.fromZodSchema`（或 `JsonMarkdownStructuredOutputParser.fromZodSchema`）生成一个解析器。解析器提供 `getFormatInstructions()`，返回一段给模型看的格式说明，把这段说明拼进你的系统或用户提示里，让模型按这个格式写，拿到 `AIMessage.content` 后再用解析器的 `parse()` 把字符串解析成对象。

整条链路如下图所示。

![20260315200123](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260315200123.png)

从 schema 定义到解析器、格式说明拼进提示、模型返回文本再 `parse` 成对象的完整七步。

下面定义一个"电影推荐"结构，包含标题、主演数组、年份、一句话简介。用 `JsonMarkdownStructuredOutputParser` 可以要求模型用 markdown 代码块包裹 JSON，解析时更稳一些：

```typescript
import { z } from "zod";
import { JsonMarkdownStructuredOutputParser } from "@langchain/core/output_parsers";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage, SystemMessage } from "@langchain/core/messages";

const movieSchema = z.object({
  title: z.string().describe("推荐的电影名称"),
  actors: z.array(z.string()).describe("主要演员，至少一位"),
  year: z.number().describe("上映年份，四位数字"),
  summary: z.string().describe("一句话剧情简介，不超过 50 字"),
});

const parser = JsonMarkdownStructuredOutputParser.fromZodSchema(movieSchema);
const formatInstructions = parser.getFormatInstructions();
```

拿到 `formatInstructions` 后，把它放进发给模型的系统消息里，告诉模型"必须按这个格式输出"：

```typescript
const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "你的 DeepSeek 密钥",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const systemContent = `你是一个电影推荐助手。用户说出喜好或场景时，你推荐一部电影，并严格按以下格式回复，不要输出格式以外的任何内容。

${formatInstructions}`;

async function recommendMovie(userInput: string) {
  const messages = [
    new SystemMessage(systemContent),
    new HumanMessage(userInput),
  ];

  const aiMsg = await model.invoke(messages);
  const text = typeof aiMsg.content === "string" ? aiMsg.content : "";

  try {
    const result = await parser.parse(text);
    return result;
  } catch {
    console.warn("解析失败，模型输出：", text);
    throw new Error("模型未按格式返回，请重试。");
  }
}

const result = await recommendMovie("我想看一部轻松的喜剧，适合周末晚上。");
console.log(result);
// { title: "...", actors: ["..."], year: 2020, summary: "..." }
```

`parse` 抛错时说明模型这次没有严格遵守格式，可以在业务里捕获后重试一次，或换用做法二以获得更稳定的输出。

若你希望整条链路写成"提示 → 模型 → 解析"的链，可以用 `RunnableSequence` 把模型和解析器串起来，这样一次 `invoke` 就直接拿到最终对象，中间不需要手动取 `content` 再调 `parse`。

## 用 withStructuredOutput 直接拿对象（做法二）

聊天模型（如 `ChatOpenAI`）上有 `withStructuredOutput(outputSchema, config?)` 方法。你传入一个 `zod` schema，它会返回一个新的 runnable：输入仍是消息或字符串，但输出不再是 `AIMessage`，而是按 schema 解析后的对象。不必自己在提示里写格式说明，也不必手写 `parse`，一次 `invoke` 就能拿到类型明确的结构。

沿用同一个 `movieSchema`，改用 `withStructuredOutput`：

```typescript
import { z } from "zod";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage, SystemMessage } from "@langchain/core/messages";

const movieSchema = z.object({
  title: z.string().describe("推荐的电影名称"),
  actors: z.array(z.string()).describe("主要演员，至少一位"),
  year: z.number().describe("上映年份，四位数字"),
  summary: z.string().describe("一句话剧情简介，不超过 50 字"),
});

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "你的 DeepSeek 密钥",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const structuredModel = model.withStructuredOutput(movieSchema);

const result = await structuredModel.invoke([
  new SystemMessage("你是一个电影推荐助手，根据用户需求推荐一部电影。"),
  new HumanMessage("我想看一部轻松的喜剧，适合周末晚上。"),
]);

console.log(result);
// { title: "...", actors: ["..."], year: 2020, summary: "..." }
```

`result` 的 TypeScript 类型会由 `movieSchema` 自动推断，可直接用于入库、传 API 或类型校验，不需要额外的类型声明。`withStructuredOutput` 内部会根据模型和配置选择实现方式（例如 function calling 或 JSON mode），你一般不需要关心细节，只要 schema 写清楚、每个字段带好 `.describe()` 即可。

若你除了解析后的对象，还想同时拿到原始的 `AIMessage`（例如记日志、做审计），可以在第二个参数里传 `{ includeRaw: true }`：

```typescript
const structuredModel = model.withStructuredOutput(movieSchema, {
  includeRaw: true,
});

const out = await structuredModel.invoke([
  new HumanMessage("推荐一部科幻片。"),
]);

console.log(out.parsed); // 按 schema 解析后的对象
console.log(out.raw); // 本次模型返回的完整 AIMessage
```

此时 `out` 为 `{ raw: BaseMessage, parsed: ... }`，两份数据都能用到，不必在"要对象"和"要原始消息"之间二选一。

## 嵌套 schema 的写法

实际场景里 schema 往往比"三个平铺字段"复杂，例如要抽取的实体有嵌套对象或数组。用 `zod` 可以逐层定义，和工具参数那边的写法完全一样：

```typescript
import { z } from "zod";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

// 从新闻文章里抽取结构化信息
const newsSchema = z.object({
  headline: z.string().describe("文章标题"),
  publishedAt: z
    .string()
    .describe("发布日期，格式 YYYY-MM-DD，无法确定则返回空字符串"),
  entities: z
    .array(
      z.object({
        name: z.string().describe("实体名称，如人名、公司名、地名"),
        type: z
          .enum(["person", "organization", "location", "other"])
          .describe("实体类型"),
      }),
    )
    .describe("文章中提到的关键实体列表"),
  sentiment: z
    .enum(["positive", "neutral", "negative"])
    .describe("文章整体情感倾向"),
  keyPoints: z.array(z.string()).describe("文章核心要点，每条一句话，至多三条"),
});

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "你的 DeepSeek 密钥",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const extractor = model.withStructuredOutput(newsSchema);

const article = `
  OpenAI 今日宣布与微软延长战略合作协议，双方将在 Azure 云平台上共同推进 GPT 系列模型的商业化部署。
  微软 CEO 萨提亚·纳德拉表示，此次合作将加速企业级 AI 应用的普及。
  分析师普遍对此持乐观态度，认为此举将进一步巩固两家公司在 AI 领域的领先地位。
`;

const extracted = await extractor.invoke([
  new HumanMessage(`请从以下新闻文章中抽取结构化信息：\n\n${article}`),
]);

console.log(extracted);
/*
{
  headline: "OpenAI 与微软延长战略合作",
  publishedAt: "",
  entities: [
    { name: "OpenAI", type: "organization" },
    { name: "微软", type: "organization" },
    { name: "Azure", type: "organization" },
    { name: "萨提亚·纳德拉", type: "person" }
  ],
  sentiment: "positive",
  keyPoints: ["OpenAI 与微软延长战略合作", "合作基于 Azure 云平台", "分析师持乐观态度"]
}
*/
```

嵌套层级越深、字段越多，`.describe()` 就越重要。模型完全依赖这些描述来理解每个字段的含义和期望格式。描述写得模糊，填出来的值就容易跑偏，描述里写清"格式 YYYY-MM-DD"这类细节，模型遵守的概率会高很多。

`z.enum()` 在这里也很有用，把可能的取值限定在有限集合里（例如情感只有 `positive`、`neutral`、`negative`），比让模型自由发挥文字要稳定得多。

## 输入 schema 与输出 schema 的关系

第 6 章里工具用 `zod` 定义的是"模型调用工具时传入的参数"，模型产出 `tool_calls`，里面的 `arguments` 要符合该 schema。本章的结构化输出则是"模型直接回复给我们的内容"要符合另一个 schema。两处都用 `zod`，一个约束输入、一个约束输出，在 LangChain 里会分别转成 JSON Schema 或格式说明发给模型。

工具 schema 约束的是模型调用工具时填入的参数，如下图所示。

两种 schema 都用 `zod` 定义，但一个针对 `tool_calls.arguments`，另一个针对 `content`，约束方向相反。

若一个链里既要用工具又要结构化回复，通常分两步。先按第 7、8 章跑完带工具的 Agent 循环，得到最终一条 `AIMessage`。若你希望这条回复再被解析成固定结构，可以只对这条消息再调一次 `withStructuredOutput` 的 runnable（或把 `content` 交给解析器），而不是在绑工具的同一模型上同时绑输出 schema。具体设计要看你是"先自由对话再结构化总结"还是"单轮就只要结构化结果"。

## 两种做法的适用场景

两种做法在稳定性、灵活性、对模型的要求上各有取舍，可以参考下表选择：

| 维度       | 提示约束加解析器                 | withStructuredOutput                 |
| ---------- | -------------------------------- | ------------------------------------ |
| 实现复杂度 | 中（需拼格式说明、处理解析失败） | 低（一行切换）                       |
| 输出稳定性 | 一般（模型可能不遵守格式）       | 高（API 层保证）                     |
| 模型要求   | 任意支持自然语言的模型           | 需支持 function calling 或 JSON mode |
| 类型推导   | 需自行声明类型                   | 由 schema 自动推断                   |
| 调试友好度 | 高（可直接看原始文本）           | 中（需加 includeRaw 才能看原始消息） |

优先选 `withStructuredOutput`，更稳定、代码更简洁。只有在模型不支持 function calling 或 JSON mode，或者你需要高度定制格式说明内容时，再考虑做法一。

## 小结

让 AI 返回可控的结构化数据，是为了下游能稳定地解析、入库或调 API，而不依赖从自然语言里抠字段。和工具参数的区别是：工具 schema 约束的是模型调用我们时的入参，输出 schema 约束的是模型回复给我们的内容，两者都用 `zod` 写，但约束方向相反。

两种做法的选用原则可以概括为两点：

- 优先用 `withStructuredOutput(zodSchema)`，直接 `invoke` 消息就拿到解析好的对象，必要时加 `{ includeRaw: true }` 同时保留原始 `AIMessage`
- 若模型不支持 function calling 或 JSON mode，或需要自定义格式说明文字，可用解析器做法，把 `getFormatInstructions()` 拼进系统提示，模型返回文本后再 `parser.parse()` 转成对象，注意处理解析失败的情况

无论哪种做法，schema 里每个字段都应该带 `.describe()`，嵌套结构用 `z.object`、`z.array` 逐层定义，枚举值用 `z.enum()` 限定范围，这样模型填出来的值更准确，下游出现意外类型的概率也会明显下降。
