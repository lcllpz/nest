前面几篇我们把 `Runnable`、`Tool`、`Memory` 这几条主线都讲清楚了，一条链路能写出来、能跑通，接下来自然要问的就是，它跑得到底怎么样。流式输出怎么接、慢请求出在哪一段、工具调用传了什么参数、模型本次消耗了多少 `token`，这些问题单靠返回值根本看不出来，必须从链路内部把事件捞出来。

`LangChain` 里解决这一类问题的机制是 `Callback`。很多人第一次接触它时会把它和日志混为一谈，其实 `Callback` 的定位更接近事件总线，它是一层统一的生命周期钩子，链路里的每个节点在开始、结束、出错时都会发出事件，应用层可以在这些事件上挂接流式转发、日志记录、指标上报、`tracing` 注入这几件事。这一篇围绕一个目标展开，把 `Callback` 的模型讲透，再把它在工程里最常用的几种落地姿势讲清楚。

## Callback 的事件模型

`Callback` 的本质是一套跨组件统一的事件协议。无论节点是 `Chain`、`LLM`、`Tool` 还是 `Retriever`，它们都遵循同一套事件约定，应用层只需要实现一个 `handler`，就能同时监听所有组件。

常见的事件类型可以按组件分组看。

| 组件        | 主要事件                                                        | 典型用途              |
| ----------- | --------------------------------------------------------------- | --------------------- |
| `LLM`       | `start`、`new_token`、`end`、`error`                            | 流式输出、延迟、成本  |
| `Chat Model` | `start`、`new_token`、`end`、`error`                           | 与 `LLM` 类似         |
| `Chain`     | `start`、`end`、`error`                                         | 链路级耗时、失败追踪  |
| `Tool`      | `start`、`end`、`error`                                         | 参数审计、失败归类    |
| `Retriever` | `start`、`end`、`error`                                         | 召回命中率、检索延迟  |
| `Agent`     | `action`、`finish`                                              | 决策路径、动作序列    |

整张表最值得注意的是 `new_token` 事件，它是流式输出的底层支撑。`Runnable` 的 `stream` 方法之所以能把 `token` 逐块吐给前端，就是因为 `Callback` 在模型推理阶段持续触发 `new_token`。

## Callback 在链路里的位置

把 `Callback` 放到 `LCEL` 链路的全景里看，它其实是一层横切事件层，平行于数据流向。

图里实线是数据流，虚线是事件流。数据流决定结果，事件流决定可观测性。两条线不互相干扰，但应用层的稳定性往往取决于事件流是否完整、是否能覆盖所有关键节点。

## 最小可运行的自定义 Handler

写一个 `Callback` 最直接的方式是实现 `BaseCallbackHandler` 接口，或者直接传一个带方法的普通对象给链路。下面这段示例把三个最有用的事件接上，模型开始、模型结束、工具结束。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import { BaseCallbackHandler } from "@langchain/core/callbacks/base";
import { LLMResult } from "@langchain/core/outputs";

class MetricsHandler extends BaseCallbackHandler {
  name = "metrics-handler";
  private startedAt = 0;

  async handleLLMStart(llm: unknown, prompts: string[]): Promise<void> {
    this.startedAt = Date.now();
    console.log("[llm:start]", { promptCount: prompts.length });
  }

  async handleLLMEnd(output: LLMResult): Promise<void> {
    const cost = Date.now() - this.startedAt;
    const usage = output.llmOutput?.tokenUsage;
    console.log("[llm:end]", { cost, usage });
  }

  async handleLLMError(err: unknown): Promise<void> {
    console.error("[llm:error]", err);
  }

  async handleToolEnd(output: string, runId: string): Promise<void> {
    console.log("[tool:end]", { runId, length: output.length });
  }
}

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个简洁回答助手。"],
  ["human", "{question}"],
]);

const chain = prompt.pipe(model).pipe(new StringOutputParser());

async function main(): Promise<void> {
  const handler = new MetricsHandler();
  const res = await chain.invoke(
    { question: "Callback 机制主要解决什么问题" },
    { callbacks: [handler] },
  );
  console.log(res);
}

main().catch(console.error);
```

这段代码有三个点值得注意。第一，`callbacks` 是作为 `RunnableConfig` 的一部分传入的，不是构造 `chain` 时写死的，这样不同请求可以挂不同的 `handler`。第二，每个事件方法都是异步的，应用层里涉及网络上报的逻辑不要在回调里阻塞太久，免得影响主链路。第三，`runId` 会在同一次调用的所有事件里保持一致，可以作为关联键把一次请求里的多个事件串起来。

## Callback 的三种挂接姿势

挂接 `Callback` 有三种写法，分别适合不同粒度。

- 请求级，放在 `invoke`、`stream`、`batch` 的 `config` 里，只影响本次调用
- 链路级，通过 `.withConfig({ callbacks: [...] })` 绑定到某条链，全局生效
- 组件级，在 `new ChatOpenAI({ callbacks: [...] })` 里传入，只影响这个模型实例

三种粒度可以叠加，同一次调用时 `Callback` 列表会被合并。生产里常见的分配方式是，请求级挂业务上下文相关的 `handler`，链路级挂固定的 `tracing`、`metrics`，组件级挂针对这个组件的特殊监控。

下面这段示例展示把固定的 `tracing` `handler` 绑到链上，再在请求级附加业务字段。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import { BaseCallbackHandler } from "@langchain/core/callbacks/base";

class TracingHandler extends BaseCallbackHandler {
  name = "tracing";
  async handleChainStart(): Promise<void> {
    console.log("[chain:start]");
  }
  async handleChainEnd(): Promise<void> {
    console.log("[chain:end]");
  }
}

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const chain = ChatPromptTemplate.fromMessages([
  ["system", "你是一个助手。"],
  ["human", "{question}"],
])
  .pipe(model)
  .pipe(new StringOutputParser())
  .withConfig({
    runName: "answer-chain",
    tags: ["prod", "v1"],
    callbacks: [new TracingHandler()],
  });

async function ask(userId: string, question: string): Promise<string> {
  return await chain.invoke(
    { question },
    { metadata: { userId }, tags: [`user:${userId}`] },
  );
}
```

这里的 `metadata` 和 `tags` 会跟随每一个事件传到 `handler` 里。应用层写 `handler` 时可以读出来用作上报维度，比如把 `userId` 打进日志，或者按 `tags` 过滤指标。

## 流式输出的常见误区

流式输出基本上就是 `new_token` 事件的典型应用，但线上经常踩坑。最高频的问题是用户反馈流式不生效，点了发送后要等一整段才出来。原因往往不在模型本身，而是链路中间的某个节点把 `chunk` 全部聚合了才返回。

排查这类问题的思路是，沿着链路往前走，看谁是第一个聚合点。下面这段示例展示用 `streamEvents` 直接订阅事件流，可以非常直观地看到每个节点的输出节奏。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const chain = ChatPromptTemplate.fromMessages([
  ["system", "请用三句话解释 Callback 机制。"],
  ["human", "{topic}"],
])
  .pipe(model)
  .pipe(new StringOutputParser());

async function main(): Promise<void> {
  const stream = chain.streamEvents(
    { topic: "LangChain Callback" },
    { version: "v2" },
  );
  for await (const evt of stream) {
    if (evt.event === "on_chat_model_stream") {
      process.stdout.write(evt.data.chunk?.content ?? "");
    }
    if (evt.event === "on_chain_start") {
      console.log(`\n[chain:start] ${evt.name}`);
    }
    if (evt.event === "on_chain_end") {
      console.log(`\n[chain:end] ${evt.name}`);
    }
  }
}

main().catch(console.error);
```

这段代码比传统的 `stream` 方法信息更丰富，它把链路里的所有组件事件都打平成一条事件流，每个事件都带 `name`、`run_id`、`metadata`。生产里调试流式问题时，用它定位比肉眼盯着日志快得多。

新写业务推荐优先用 `streamEvents(v2)`，它是 `LangChain JS` 当前的主推方式，等价的场景里能拿到的信息更完整。

## 可观测性的三条基本线

`Callback` 最常见的落地场景是做可观测性，可以从三个维度同时落。

- 日志，把关键字段写进结构化日志，重点是工具入参、模型响应摘要、错误堆栈
- 指标，把耗时、`token` 消耗、失败率打进 `Prometheus` 或等价系统
- 追踪，把 `runId`、`parentRunId`、`tags` 注入 `OpenTelemetry` 的 `span`

下面这段示例把三件事放到同一个 `handler` 里，便于看到它们的挂接位置。代码里的 `metrics` 和 `tracer` 只是占位，生产里换成你自己的埋点 `SDK` 即可。

```typescript
import { BaseCallbackHandler } from "@langchain/core/callbacks/base";
import { LLMResult } from "@langchain/core/outputs";

interface MetricsClient {
  observe(name: string, value: number, tags?: Record<string, string>): void;
}

interface Tracer {
  startSpan(name: string, attrs: Record<string, unknown>): { end: () => void };
}

class ObservabilityHandler extends BaseCallbackHandler {
  name = "observability";
  private starts = new Map<string, number>();
  private spans = new Map<string, { end: () => void }>();

  constructor(
    private readonly metrics: MetricsClient,
    private readonly tracer: Tracer,
  ) {
    super();
  }

  async handleChainStart(
    chain: unknown,
    inputs: Record<string, unknown>,
    runId: string,
    parentRunId: string | undefined,
    tags: string[] | undefined,
    metadata: Record<string, unknown> | undefined,
    runType: string | undefined,
    runName: string | undefined,
  ): Promise<void> {
    this.starts.set(runId, Date.now());
    const span = this.tracer.startSpan(runName ?? "chain", {
      runId,
      parentRunId,
      tags,
      metadata,
    });
    this.spans.set(runId, span);
  }

  async handleChainEnd(
    outputs: Record<string, unknown>,
    runId: string,
  ): Promise<void> {
    const cost = Date.now() - (this.starts.get(runId) ?? Date.now());
    this.metrics.observe("chain.duration_ms", cost);
    this.spans.get(runId)?.end();
    this.starts.delete(runId);
    this.spans.delete(runId);
  }

  async handleLLMEnd(output: LLMResult, runId: string): Promise<void> {
    const usage = output.llmOutput?.tokenUsage as
      | { totalTokens?: number }
      | undefined;
    if (usage?.totalTokens) {
      this.metrics.observe("llm.tokens_total", usage.totalTokens);
    }
  }

  async handleToolStart(
    tool: unknown,
    input: string,
    runId: string,
    parentRunId: string | undefined,
    tags: string[] | undefined,
    metadata: Record<string, unknown> | undefined,
    runName: string | undefined,
  ): Promise<void> {
    console.log("[tool:start]", { runId, runName, inputLength: input.length });
  }

  async handleToolError(err: unknown, runId: string): Promise<void> {
    this.metrics.observe("tool.error", 1);
    console.error("[tool:error]", { runId, err });
  }
}
```

这段代码把三件事放到同一个 `handler` 里主要是为了示意。真实项目里建议把日志、指标、追踪拆成三个 `handler`，挂到同一个链上。拆开的好处是职责清晰，比如出问题时可以只关掉追踪 `handler`，不影响日志。

注意 `handleLLMEnd` 里取 `tokenUsage` 的写法。不同模型提供方返回字段名不完全一致，生产前建议先打印一遍 `output.llmOutput`，确认字段位置再落点。

## 工具调用审计的典型用法

`Tool` 节点的事件在审计场景尤其有用。很多线上故障的真正原因不是模型本身不稳定，而是工具被调用时传了奇怪的参数，或者返回了意料之外的结构。把 `Tool` 事件里的入参、出参、耗时全部写进日志，事后复盘时可以直接定位到那一次调用。

```typescript
import { BaseCallbackHandler } from "@langchain/core/callbacks/base";

class ToolAuditHandler extends BaseCallbackHandler {
  name = "tool-audit";
  private toolInputs = new Map<string, string>();
  private toolStarts = new Map<string, number>();

  async handleToolStart(
    tool: unknown,
    input: string,
    runId: string,
  ): Promise<void> {
    this.toolInputs.set(runId, input);
    this.toolStarts.set(runId, Date.now());
  }

  async handleToolEnd(output: string, runId: string): Promise<void> {
    const cost = Date.now() - (this.toolStarts.get(runId) ?? Date.now());
    console.log(
      JSON.stringify({
        kind: "tool.audit",
        runId,
        cost,
        input: this.toolInputs.get(runId),
        output: output.length > 500 ? `${output.slice(0, 500)}...` : output,
      }),
    );
    this.toolInputs.delete(runId);
    this.toolStarts.delete(runId);
  }

  async handleToolError(err: unknown, runId: string): Promise<void> {
    console.error(
      JSON.stringify({
        kind: "tool.error",
        runId,
        input: this.toolInputs.get(runId),
        err: err instanceof Error ? err.message : String(err),
      }),
    );
    this.toolInputs.delete(runId);
    this.toolStarts.delete(runId);
  }
}
```

这段代码里有一个很常见的取舍，要不要把 `output` 全量落日志。完整日志便于排查，但日志成本不低，且可能写入敏感信息。实际项目里通常是做长度截断加脱敏，关键字段单独抽出来走审计日志，不要全文裸存。

## 常见坑点

`Callback` 写起来看似简单，线上踩的坑反而集中在几个固定位置。

- 在 `handler` 里做重活。事件是在主链路上触发的，里面跑网络请求、复杂计算会拖慢整条链。正确做法是在 `handler` 里只做轻量聚合，落盘或上报走异步队列
- 依赖 `this` 状态跨请求保留。`handler` 实例如果是全局复用的，不同请求的 `runId` 状态会互相影响。推荐按请求新建实例，或者用 `runId` 做 `Map` 隔离
- 漏处理 `error` 事件。只挂 `end` 不挂 `error`，失败路径的可观测性就没了。出错往往才是最需要看日志的时刻
- 事件顺序不是严格同步的。比如 `LLM start` 可能和 `Chain start` 顺序交错，不要依赖具体顺序做逻辑，而是按 `runId`、`parentRunId` 做层级还原
- 混用 `stream` 和 `Callback` 的事件。`streamEvents(v2)` 已经把事件流打平，如果同时再挂一组 `Callback`，容易出现重复上报

如果线上突然发现日志量暴涨或者链路延迟变大，多半是这几条里的某一条被触发了。

## 总结

`Callback` 是 `LangChain` 里最容易被低估的一层能力。它不改变数据流，但它决定了我们能看到多少信息。把它讲清楚其实就三件事：

- 事件模型，链路里的每个节点在 `start`、`end`、`error`、`new_token` 时机会发出事件
- 挂接方式，请求级、链路级、组件级三种粒度可以叠加，按需选粒度
- 工程落点，流式输出靠 `new_token`，可观测性拆成日志、指标、追踪三条线，审计专用的 `Tool` 事件单独处理

新写项目时，推荐直接用 `streamEvents(v2)` 做事件消费，旧的 `Callback` 接口更多用于补齐它未覆盖的定制场景。把链路的事件全量拿到手，生产里的故障定位、成本核算、性能优化才有实打实的数据依据，而不是靠猜。
