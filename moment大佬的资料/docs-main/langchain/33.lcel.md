`LCEL` 是 LangChain Expression Language 的缩写。

它本质不是一门新语言，而是 LangChain 用来声明式组合 `Runnable` 的写法。LangChain 推出它的原因很直接，老式链条虽然能用，但可组合性差，也很难统一保证 `batch`、`stream`、`async` 这些能力。

在 TypeScript 里，`LCEL` 的核心构件是这几个：

- `.pipe()`，把前一个 `Runnable` 的输出接到后一个 `Runnable`
- `RunnableSequence.from([...])`，显式构造串行链
- 对象映射与 `RunnableParallel`，并行生成多个字段
- `RunnablePassthrough`，原样透传输入
- `RunnableLambda`，把普通 `TypeScript` 函数包装成 `Runnable`

`Runnable` 是 LangChain 的基础工作单元，可以被 `invoke`、`batch`、`stream`、`transform`。理解了这个，再看 `LCEL` 就很自然了。

## 没有 LCEL 时你在写什么

先不引入任何 LCEL，用最朴素的方式实现一个翻译功能。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个专业翻译助手，请把用户输入翻译成英文。"],
  ["human", "{text}"],
]);

async function translateToEnglish(text: string): Promise<string> {
  const messages = await prompt.formatMessages({ text });
  const aiMessage = await model.invoke(messages);
  return aiMessage.content as string;
}

async function main() {
  const result = await translateToEnglish("我最近在学习 LangChain");
  console.log(result);
}

main().catch(console.error);
```

这段代码没有问题，逻辑也清晰。

它本质上已经是一条链，数据从输入字符串出发，经过 `prompt` 和 `model`，最后落到输出文本。只是这条链散落在函数调用里，没有被显式表达出来。

## 手写流程编排为何容易失控

在简单场景里，手写流程往往看起来没问题。真正的麻烦出现在流程变长之后，业务逻辑、模型调用逻辑、数据变换逻辑会逐渐混在一起。

先看一个更贴近真实项目的 `RAG` 问答流程。

![20260424133736](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260424133736.png)

下面这段代码故意不用 `LCEL`，完全采用手动编排，便于对比后面的 `LCEL` 写法。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { Document } from "@langchain/core/documents";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是一个问答助手。只允许根据给定的 context 回答。如果 context 不足，就明确说不知道。`,
  ],
  ["human", "Context:\n{context}\n\nQuestion:\n{question}"],
]);

// 模拟检索函数，实际项目里会是向量数据库
async function mockRetrieve(query: string): Promise<Document[]> {
  const knowledgeBase: Document[] = [
    new Document({
      pageContent:
        "LCEL 是 LangChain Expression Language 的缩写，用于声明式组合 Runnable。",
    }),
    new Document({
      pageContent:
        "在 TypeScript 中，LCEL 使用 .pipe() 串联步骤，等价于 Python 中的 | 操作符。",
    }),
    new Document({
      pageContent:
        "Runnable 是 LangChain 的基础工作单元，支持 invoke、batch、stream 等方法。",
    }),
  ];

  return knowledgeBase
    .filter((doc) =>
      doc.pageContent.toLowerCase().includes(query.toLowerCase().slice(0, 4)),
    )
    .slice(0, 2);
}

function formatDocs(docs: Document[]): string {
  return docs.map((doc) => doc.pageContent).join("\n\n");
}

async function answerQuestion(question: string): Promise<string> {
  // 1) 手动检索
  const docs = await mockRetrieve(question);

  // 2) 手动格式化文档
  const context = formatDocs(docs);

  // 3) 手动组织 prompt 输入
  const messages = await prompt.formatMessages({ question, context });

  // 4) 手动调用模型
  const aiMessage = await model.invoke(messages);

  // 5) 手动取文本
  return aiMessage.content as string;
}

async function main() {
  const result = await answerQuestion("LCEL 是什么？");
  console.log(result);
}

main().catch(console.error);
```

这段代码本身可以工作，但阅读和维护成本已经明显上来了。你需要持续在脑子里跟踪每一步的数据形态：

- `question` 是 `string`
- `docs` 是 `Document[]`
- `context` 是 `string`
- `prompt` 需要 `{ question, context }`
- `model.invoke()` 需要 `messages`
- `model` 返回的是 `AIMessage`

而且这还只是一个简化版 `RAG`。它真正的痛点不是"代码长"，而是下面这些工程问题：

- 数据流不够显式。读代码时需要不断切换上下文，确认当前变量到底是 `string`、`Document[]` 还是 `messages`。
- 扩展点不集中。想插入改写 `query`、重排检索结果、敏感词过滤、重试策略这类步骤，只能在函数里手动加逻辑，代码会越叠越厚。
- 流式能力难以统一。如果要做流式输出，通常需要逐层改调用方式，很难做到"整条链一键切换为流式"。
- 链路复用成本高。流程散落在函数细节里，不容易把整段能力封装成可复用对象，也不方便统一挂载 tracing 与 retry 配置。

## LCEL 的本质

`LCEL` 要解决的核心问题，不是让代码看起来更短，而是把原本散落在函数调用里的流程关系，提升成可组合、可复用、可观察的数据流对象。

先看手动编排时常见的写法：

```typescript
const docs = await retriever.invoke(question);
const context = formatDocs(docs);
const messages = await prompt.formatMessages({ question, context });
const aiMessage = await model.invoke(messages);
return aiMessage.content;
```

在 `LCEL` 里，同一件事可以被表达成一张声明式的数据流图：

![20260424133810](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260424133810.png)

这样改写之后，你得到的不只是语法变化，而是工程能力的提升：

- 数据流更显式。每个节点的输入输出关系是可读、可推断的，不用在函数细节里来回跳转。
- 组合更自然。你可以像拼积木一样插入或替换步骤，而不是大段改写原函数。
- 调用接口更统一。整条链天然支持 `invoke`、`batch`、`stream`，调用方式一致。
- 工程化能力更完整。链路可以整体复用、包装，也更容易接入 tracing、重试与配置管理。

## 最简单的 LCEL 链

先把前面的翻译流程改写成 `LCEL` 版本。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

  ["system", "你是一个专业翻译助手，请把用户输入翻译成英文。"],
  ["human", "{text}"],
]);

const parser = new StringOutputParser();

const chain = prompt.pipe(model).pipe(parser);

async function main() {
  const result = await chain.invoke({ text: "我最近在学习 LangChain" });

```

这个版本少掉的不是业务逻辑，而是手动胶水代码。你不再需要自己调用 `formatMessages`、手动执行 `model.invoke(messages)`、再从消息对象里取 `.content`。

更关键的是，你现在得到的是一个统一的 `Runnable` 对象：

```typescript
// 三种调用方式都可以直接用
await chain.invoke({ text: "你好" });
await chain.batch([{ text: "你好" }, { text: "再见" }]);
const stream = await chain.stream({ text: "给我讲个故事" });
```

## LCEL 写 RAG

接下来把前面的 `RAG` 手写流程改写成 `LCEL` 版本。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import {
  RunnableLambda,
  RunnablePassthrough,
  RunnableSequence,
} from "@langchain/core/runnables";
import { Document } from "@langchain/core/documents";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是一个问答助手。只允许根据给定的 context 回答。如果 context 不足，就明确说不知道。`,
  ],
  ["human", "Context:\n{context}\n\nQuestion:\n{question}"],
]);

// 模拟检索器，实际项目里替换成向量数据库的 retriever
const mockRetriever = new RunnableLambda({
  func: async (_query: string): Promise<Document[]> => {
    return [
      new Document({
        pageContent:
          "LCEL 是 LangChain Expression Language 的缩写，用于声明式组合 Runnable。",
      }),
      new Document({
        pageContent:
          "在 TypeScript 中，LCEL 使用 .pipe() 串联步骤，等价于 Python 中的 | 操作符。",
      }),
      new Document({
        pageContent:
          "Runnable 是 LangChain 的基础工作单元，支持 invoke、batch、stream 等方法。",
      }),
    ];
  },
});

// 文档格式化，把 Document[] 拼成 string
const formatDocs = new RunnableLambda({
  func: async (docs: Document[]): Promise<string> => {
    return docs.map((doc) => doc.pageContent).join("\n\n");
  },
});

const ragChain = RunnableSequence.from([
  {
    context: mockRetriever.pipe(formatDocs),
    question: new RunnablePassthrough<string>(),
  },
  prompt,
  model,
  new StringOutputParser(),
]);

async function main() {
  const result = await ragChain.invoke("LCEL 是什么？");
  console.log(result);
}

main().catch(console.error);
```

## 逐层拆解 RAG 链

这一段是 `LCEL` 思维方式最集中的示例，建议按数据流顺序来理解。

入口是 `await ragChain.invoke("LCEL 是什么？")`。输入是一个字符串，也就是用户问题本身。

第一层先做字段构造，结构如下：

```typescript
{
  context: mockRetriever.pipe(formatDocs),
  question: new RunnablePassthrough<string>(),
}
```

这一层不是普通 `object`，而是并行字段映射。`context` 分支会把原始问题送入 `mockRetriever` 和 `formatDocs`，产出上下文字符串。`question` 分支通过 `RunnablePassthrough` 保留原始问题。执行完这一层后，数据会变成：

```typescript
{
  context: "LCEL 是 LangChain Expression Language...\n\n在 TypeScript 中...",
  question: "LCEL 是什么？"
}
```

第二层是 `prompt`。它刚好消费 `{ context, question }` 这两个字段，所以可以无缝衔接。这里体现了 `LCEL` 的核心习惯，不是盯着某个模型在调用什么，而是持续维护每一步输入输出 `shape` 的匹配关系。

第三层是 `model`，把 `prompt` 产出的消息列表转换为 `AIMessage`。

第四层是 `StringOutputParser`，把 `AIMessage` 解析成纯字符串，便于后续展示、存储或继续处理。

到这里可以看到，学习 `LCEL` 的重点并不是机械记忆 `prompt`、`model`、`parser` 的串联顺序，而是持续关注每个节点的输入输出 `shape`。你需要明确当前节点产出什么、下一节点消费什么、哪些步骤负责并行展开字段、哪些步骤负责保留原始输入、哪些步骤用普通函数做数据转换。思维方式一旦切换到"设计数据流"而不是"调用某个模型"，后面的链路扩展与调试都会顺很多。

## RunnableLambda 的用法

在真实项目里，并不是所有步骤都该交给模型。很多环节本质上是确定性的业务处理：

- 拼文档
- 清洗字段
- 裁剪历史消息
- 把数据库结果转成 prompt 所需结构
- 解析 JSON 某几个字段

这时候就该用 `RunnableLambda`。它可以把普通 `TypeScript` 函数包装成 `Runnable`，让这类业务逻辑直接接入 `LCEL` 数据流。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import { RunnableLambda } from "@langchain/core/runnables";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个专业翻译助手，请把用户输入翻译成英文。"],
  ["human", "{text}"],
]);

// 把用户输入规范化，去首尾空格、合并多余空白、转成 prompt 所需 shape
const normalizeInput = new RunnableLambda({
  func: async (input: string): Promise<{ text: string }> => {
    const normalized = input.trim().replace(/\s+/g, " ");
    return { text: normalized };
  },
});

const chain = normalizeInput
  .pipe(prompt)
  .pipe(model)
  .pipe(new StringOutputParser());

async function main() {
  const result = await chain.invoke("   我最近  在学习   LangChain   ");
  console.log(result);
}

main().catch(console.error);
```

`RunnableLambda` 的价值在于，它把"模型步骤"和"业务步骤"放进同一条执行链。这样你可以把数据清洗、字段转换、规则处理和模型调用统一编排，而不是分散在链路外侧打补丁。

## RunnablePassthrough 的用法

`RunnablePassthrough` 的作用是原样透传输入，不做修改。

它在 `RAG` 里尤其常见。你一方面要把问题送去检索生成 `context`，另一方面又要把原问题保留下来传给 `prompt` 的 `{question}`。可以把它理解成 `LCEL` 里的"保留现场"能力。

除了透传，`RunnablePassthrough` 还支持 `.assign()`。你可以在保留原始输入的同时，补充新的字段，例如下面这样。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import { RunnablePassthrough } from "@langchain/core/runnables";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "请将以下文本翻译成{target_language}。"],
  ["human", "{text}"],
]);

// 透传原始输入，同时附加一个 target_language 字段
const addTargetLanguage = RunnablePassthrough.assign({
  target_language: async () => "英文",
});

const chain = addTargetLanguage
  .pipe(prompt)
  .pipe(model)
  .pipe(new StringOutputParser());

async function main() {
  const result = await chain.invoke({ text: "我最近在学习 LangChain" });
  console.log(result);
}

main().catch(console.error);
```

## RunnableParallel 与并行多字段

`LCEL` 不只支持线性串行链，也支持把一个输入并行展开成多个分支，最后收敛为对象结果。

下面这个需求很常见，输入一段文章，同时生成摘要、关键词，并保留原文。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import {
  RunnablePassthrough,
  RunnableParallel,
} from "@langchain/core/runnables";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const summaryPrompt = ChatPromptTemplate.fromMessages([
  ["system", "请将用户输入总结成 2-3 句话。"],
  ["human", "{text}"],
]);

const keywordPrompt = ChatPromptTemplate.fromMessages([
  ["system", "请提取用户输入的 5 个关键词，用逗号分隔。"],
  ["human", "{text}"],
]);

const summaryChain = summaryPrompt.pipe(model).pipe(new StringOutputParser());
const keywordChain = keywordPrompt.pipe(model).pipe(new StringOutputParser());

const articleAnalysisChain = RunnableParallel.from({
  summary: summaryChain,
  keywords: keywordChain,
  original: new RunnablePassthrough(),
});

async function main() {
  const result = await articleAnalysisChain.invoke({
    text: "LangChain 的 LCEL 让链式组合、流式输出和批量处理都变得更统一。开发者可以用 .pipe() 串联任意 Runnable，也可以用对象映射并行生成多个字段。",
  });

  console.log("摘要：", result.summary);
  console.log("关键词：", result.keywords);
}

main().catch(console.error);
```

这个例子说明，`LCEL` 可以自然表达"分叉处理、汇总输出"的数据流结构，而不只是单线串行执行。

## 流式输出

`LCEL` 在流式场景里的优势很直接。你不是只对底层 `model` 做流式调用，而是可以对整条链统一调用 `.stream()`。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个擅长讲故事的助手。"],
  ["human", "{topic}"],
]);

const chain = prompt.pipe(model).pipe(new StringOutputParser());

async function main() {
  console.log("开始流式输出：\n");

  const stream = await chain.stream({ topic: "给我讲一个关于程序员的小故事" });

  for await (const chunk of stream) {
    process.stdout.write(chunk);
  }

  console.log("\n\n流式输出完毕。");
}

main().catch(console.error);
```

如果不用 `LCEL`，流式通常只发生在模型层，前后处理步骤很难以统一方式表达。用了 `LCEL` 之后，你对整条链调用 `.stream()`，流式结果会沿着链路自然传递到最终输出。

## 批量处理

批量处理的思路和流式一致，直接对链调用 `.batch()` 即可。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "请将用户输入的中文翻译成英文，只输出翻译结果，不要解释。"],
  ["human", "{text}"],
]);

const chain = prompt.pipe(model).pipe(new StringOutputParser());

async function main() {
  const inputs = [
    { text: "你好，世界！" },
    { text: "今天天气不错" },
    { text: "我喜欢写 TypeScript" },
    { text: "LangChain 让 AI 应用开发变得更简单" },
  ];

  const results = await chain.batch(inputs, {
    maxConcurrency: 2, // 最多同时并发 2 个请求
  });

  results.forEach((result, index) => {
    console.log(`${inputs[index].text}  对应  ${result}`);
  });
}

main().catch(console.error);
```

不用 `LCEL` 时，批量调用通常要自己组织循环、并发控制、结果对齐和异常边界。`LCEL` 把这些共性能力收敛为统一接口，并通过 `maxConcurrency` 直接控制并发度。

## `.pipe()` 与 `RunnableSequence.from()` 的选择

在 `TypeScript` 里，这两种写法都很常见。先看链式 `.pipe()`：

```typescript
const chain = prompt.pipe(model).pipe(new StringOutputParser());
```

再看 `RunnableSequence.from([...])`：

```typescript
import { RunnableSequence } from "@langchain/core/runnables";

const chain = RunnableSequence.from([prompt, model, new StringOutputParser()]);
```

两者在能力上等价，区别主要在可读性和组织方式。实践里可以这样选：

- 简单线性流程用 `.pipe()`，读起来更紧凑
- 有并行字段（对象映射层）或者层级更复杂时用 `RunnableSequence.from([])`，视觉上更清晰

## 给链条加运行时配置

## 给链条加重试

`withRetry()` 可以给任意 `Runnable` 增加自动重试能力，不需要手写重复的 `try/catch` 与重试循环。

```typescript
import { ChatOpenAI } from "@langchain/openai";
```

## 用 RunnableBranch 实现条件分支

有时候链条需要根据输入走不同的路径，这时候可以用 `RunnableBranch`。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import { RunnableBranch } from "@langchain/core/runnables";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

interface UserInput {
  type: "translate" | "summarize" | "other";
  text: string;
}

const translatePrompt = ChatPromptTemplate.fromMessages([
  ["system", "请将以下内容翻译成英文。"],
  ["human", "{text}"],
]);

const summarizePrompt = ChatPromptTemplate.fromMessages([
  ["system", "请将以下内容总结成 2-3 句话。"],
  ["human", "{text}"],
]);

const defaultPrompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个通用助手，请回答用户的问题。"],
  ["human", "{text}"],
]);

const translateChain = translatePrompt
  .pipe(model)
  .pipe(new StringOutputParser());
const summarizeChain = summarizePrompt
  .pipe(model)
  .pipe(new StringOutputParser());
const defaultChain = defaultPrompt.pipe(model).pipe(new StringOutputParser());

const branch = RunnableBranch.from([
  [(input: UserInput) => input.type === "translate", translateChain],
  [(input: UserInput) => input.type === "summarize", summarizeChain],
  defaultChain,
]);

async function main() {
  const r1 = await branch.invoke({
    type: "translate",
    text: "我最近在学习 LangChain",
  });
  console.log("翻译结果：", r1);

  const r2 = await branch.invoke({
    type: "summarize",
    text: "LCEL 是 LangChain 的声明式组合写法，它让链条组合、流式输出和批量处理变得更统一，开发者可以用 .pipe() 串联任意 Runnable。",
  });
  console.log("摘要结果：", r2);
}

main().catch(console.error);
```

## LCEL 边界与避坑速记

判断是否该用 `LCEL`，可以抓住一个核心标准，任务是否是可预期的数据流编排。这个标准明确后，再看边界、踩坑和速记概念会更清晰。

`LCEL` 的执行方式是把任务拆成一条可预期的数据流，按固定节点顺序执行。`Agent` 的执行方式是基于当前状态做动态决策，在工具调用、观察反馈和重规划之间循环推进，直到完成目标。

![20260413155707](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260413155707.png)

`LCEL` 更适合这类场景：

- 固定步骤的数据处理
- 可预期数据流的 `RAG`
- 文本提取、分类、改写
- 多阶段内容处理
- 模型前后处理

`Agent` 的执行方式是基于当前状态动态决策，在工具调用、观察反馈和重规划之间循环推进，直到完成目标。

![20260413155914](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260413155914.png)

`Agent` 更适合这类场景：

- 工具调用循环（模型自主决定用什么工具）
- 模型自主决定下一步行动
- 需要复杂状态管理的长任务
- 循环决策逻辑

`LangChain` 当前的 `createReactAgent` 底层构建在 `LangGraph` 之上。换句话说，`LCEL` 负责稳定的数据流组合，`Agent` 负责带决策循环的动态执行，二者是分工关系，不是替代关系。

边界清楚之后，再重点规避下面这些高频错误：

- 只盯 `Prompt`，不盯 `Shape`。LCEL 最大的坑不是 `prompt` 写不好，而是没搞清上一步输出是什么、下一步输入要什么。调试时先打印每一步的输入输出，比直接盯着 `prompt` 更有效。
- 忘了 `parser`。很多 `chat model` 返回的不是纯字符串，而是消息对象 `AIMessage`。如果你直接对链输出做字符串操作，可能会得到 `[object Object]`，记得在链末尾加 `StringOutputParser`。
- 什么都往模型里塞。很多转换逻辑其实应该交给 `RunnableLambda` 或普通代码，而不是全部写在 `prompt` 里，把数据整理的逻辑放在代码层，既更可控，也更容易调试。
- 把 LCEL 当工作流引擎。LCEL 是组合 `Runnable` 的表达方式，不是完整的复杂状态编排引擎，真要做带循环、带决策、带复杂状态管理的 agent runtime，应该去用 `LangGraph`。
- 还在用老式链条。LangChain v2（`JavaScript` 版）里，很多旧式 `chains`（比如 `LLMChain`、`ConversationChain`）已经被 LCEL 替代，新项目直接用 LCEL 写。

最后用五个关键词做收束。理解这五个，`LCEL` 基本就算入门了。

- `Runnable`，一切可执行步骤的统一抽象，可以被 `invoke`、`batch`、`stream`
- `pipe`，串行连接，`TypeScript` 里 LCEL 最核心的写法
- `parallel`，把一个输入并行展开成多个字段，用对象映射或 `RunnableParallel` 表达
- `passthrough`，原样透传输入，不做修改，常用于 `RAG` 场景保留原始问题
- `lambda`，把普通 `TypeScript` 函数包装进链里，`RunnableLambda` 是最灵活的工具

## 借助 LCEL 实现前端知识库问答

前面讲的是概念和局部能力，这一节给一个完整可运行的落地版本。示例代码在 `example/lcel-complete-demo/src/index.ts`，目标是做一个前端技术问答链，输入问题后返回结构化 JSON 结果。

这条链的执行顺序是固定的：

- 接收 `question`
- 检索前端知识库文档
- 格式化为 `context`
- 送入 `prompt + model`
- 解析成结构化对象输出

完整可运行代码如下：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { JsonOutputParser } from "@langchain/core/output_parsers";
import { RunnableLambda, RunnablePassthrough } from "@langchain/core/runnables";
import { Document } from "@langchain/core/documents";

type Confidence = "high" | "medium" | "low";

interface RagAnswer {
  answer: string;
  confidence: Confidence;
  citations: string[];
  nextAction: string;
}

const model = new ChatOpenAI({
  model: "deepseek-ai/DeepSeek-V3",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
  temperature: 0,
});

const prompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是一个资深前端技术助手。请只根据给定 context 回答，并返回 JSON：
{{
  "answer": "前端向的简洁回答",
  "confidence": "high 或 medium 或 low",
  "citations": ["引用片段1", "引用片段2"],
  "nextAction": "下一步建议"
}}
如果 context 不足，请明确写信息不足，confidence 设为 low，nextAction 写继续排查方向。`,
  ],
  ["human", "Context:\n{context}\n\nQuestion:\n{question}"],
]);

const retriever = new RunnableLambda({
  func: async (question: string): Promise<Document[]> => {
    const kb: Document[] = [
      new Document({
        pageContent:
          "React 性能优化优先顺序通常是减少不必要渲染、稳定 props 引用、合理拆分组件和懒加载路由级模块。",
      }),
      new Document({
        pageContent:
          "在 React 中，列表渲染要提供稳定 key，避免使用 index 作为 key 导致状态错位和额外重渲染。",
      }),
      new Document({
        pageContent:
          "前端接口层建议统一封装 request client，在拦截器里处理 token、错误映射和重试策略，减少页面层重复逻辑。",
      }),
      new Document({
        pageContent:
          "TypeScript 在前端中应尽量避免 any，优先通过类型守卫、泛型和明确接口定义提升可维护性。",
      }),
      new Document({
        pageContent:
          "前端性能排查可按顺序进行：先看网络瀑布和资源体积，再看主线程长任务，再看 React 重渲染与 memo 失效点，最后针对热点做代码分割和缓存优化。",
      }),
      new Document({
        pageContent:
          "减少 React 不必要渲染的常用做法包括 React.memo、useMemo、useCallback、状态下沉与按需拆分组件。",
      }),
      new Document({
        pageContent:
          "统一请求层的价值是把鉴权、超时、重试、错误提示和埋点集中管理，避免页面里重复写样板代码。",
      }),
    ];
    const rules: Array<{ keyword: string; score: number }> = [
      { keyword: "react", score: 3 },
      { keyword: "渲染", score: 3 },
      { keyword: "key", score: 3 },
      { keyword: "性能", score: 3 },
      { keyword: "优化", score: 2 },
      { keyword: "请求", score: 3 },
      { keyword: "封装", score: 3 },
      { keyword: "拦截器", score: 2 },
      { keyword: "typescript", score: 3 },
      { keyword: "any", score: 3 },
      { keyword: "类型守卫", score: 3 },
      { keyword: "泛型", score: 2 },
    ];
    const normalizedQuestion = question.toLowerCase();

    const ranked = kb
      .map((doc) => {
        const normalizedDoc = doc.pageContent.toLowerCase();
        const score = rules.reduce((acc, rule) => {
          if (
            normalizedQuestion.includes(rule.keyword) &&
            normalizedDoc.includes(rule.keyword)
          ) {
            return acc + rule.score;
          }
          return acc;
        }, 0);
        return { doc, score };
      })
      .filter((item) => item.score > 0)
      .sort((a, b) => b.score - a.score)
      .slice(0, 4)
      .map((item) => item.doc);

    return ranked;
  },
});

const formatDocs = new RunnableLambda({
  func: async (docs: Document[]): Promise<string> => {
    if (docs.length === 0) return "没有检索到可用上下文。";
    return docs.map((doc, i) => `[${i + 1}] ${doc.pageContent}`).join("\n");
  },
});

const pickQuestion = new RunnableLambda({
  func: async (input: { question: string }): Promise<string> => input.question,
});

const inputBuilder = RunnablePassthrough.assign({
  context: pickQuestion.pipe(retriever).pipe(formatDocs),
  question: async (input: { question: string }): Promise<string> =>
    input.question,
});

const parser = new JsonOutputParser<RagAnswer>();

const chain = inputBuilder
  .pipe(prompt)
  .pipe(model.withRetry({ stopAfterAttempt: 3 }))
  .pipe(parser)
  .withConfig({
    runName: "lcel-complete-demo",
    tags: ["lcel", "rag", "json"],
  });

async function runInvoke(): Promise<void> {
  const q = "React 列表为什么不建议用 index 作为 key？";
  const result = await chain.invoke({ question: q });
  console.log("\n=== invoke ===");
  console.log(result);
}

async function runBatch(): Promise<void> {
  const questions = [
    { question: "如何减少 React 组件的不必要渲染？" },
    { question: "为什么前端项目里要统一封装请求层？" },
    { question: "TypeScript 在前端里怎么减少 any 的使用？" },
  ];
  const results = await chain.batch(questions, { maxConcurrency: 2 });
  console.log("\n=== batch ===");
  results.forEach((item, idx) => {
    console.log(`Q${idx + 1}: ${questions[idx].question}`);
    console.log(item);
  });
}

async function runStream(): Promise<void> {
  console.log("\n=== stream ===");
  const stream = await chain.stream({
    question: "请给我一个前端性能优化的排查顺序。",
  });
  const merged: RagAnswer = {
    answer: "",
    confidence: "low",
    citations: [],
    nextAction: "",
  };

  for await (const chunk of stream) {
    if (chunk.answer) merged.answer = chunk.answer;
    if (chunk.confidence) merged.confidence = chunk.confidence;
    if (Array.isArray(chunk.citations) && chunk.citations.length > 0) {
      merged.citations = chunk.citations;
    }
    if (chunk.nextAction) merged.nextAction = chunk.nextAction;
  }

  console.log(merged);
}

async function main(): Promise<void> {
  await runInvoke();
  await runBatch();
  await runStream();
}

main().catch(console.error);
```

这个示例里 `inputBuilder` 用 `RunnablePassthrough.assign` 把输入问题扩展成 `{ question, context }`，`context` 来自 `retriever.pipe(formatDocs)`。这样做的好处是数据 shape 始终清晰，后续节点只关心自己需要的字段。

此外，示例同时演示了三种调用方式：

- `invoke`：单问题调用，验证链路是否正确
- `batch`：批量问题并发调用，验证链路可复用性
- `stream`：流式调用，验证输出可渐进生成

运行代码后，最终输出结果如下图所示：

![20260413162412](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260413162412.png)

这里的输出结果真正体现的是 `LCEL` 的核心能力：同一条链只定义一次，就能用统一接口切换 `invoke`、`batch`、`stream` 三种执行模式，而不用重写流程。无论调用模式怎么变，输出 `shape` 都保持一致（`answer`、`confidence`、`citations`、`nextAction`），这正是声明式数据流在工程上的价值。

## 总结

没有 `LCEL` 时，我们也能完成 `Prompt`、`Model`、`Retriever`、`Parser` 的组合，但这些步骤通常散落在业务函数里。每一步都要手动维护输入输出结构，流式输出、批量处理、重试和复用也需要额外胶水代码。

`LCEL` 的价值不只是少写几行代码，而是把原本隐式的数据流提升为显式的 `Runnable` 组合。这样 `Prompt`、`Model`、`Retriever`、普通函数、解析器都能进入同一个执行模型，并统一获得 `invoke`、`batch`、`stream`、`async` 等能力。

当你开始用输入输出 `shape` 来设计链路，而不是只关注当前在调用哪个函数，就真正进入了 `LCEL` 的思维方式。
