# LangChain 中怎么用 LATS，用 MCTS 与反思打分做多路径搜索

`Language Agent Tree Search`（`LATS`）由 Zhou 等人提出，把蒙特卡洛树搜索（`MCTS`）与 `LLM` 的反思能力绑在一起。它不是在单条回答上跑错了再改，而是维护一棵搜索树，在多条候选推理链上并行试探，用反思打分告诉算法下一轮更值得从哪条支路接着写，最终挑出综合质量最好的一条轨迹。

若你刚读完 `24.ToT` 里对树形推理与评估的抽象，可以把本篇看成把「树」和「可执行的多轮生成」缝得更紧的一种落地形态，树里每个节点都是一段真实可续写的对话切片，而不是纸上的一条抽象思路。

和 `Reflexion`、`Plan-and-Execute` 同属反思与规划这一大类，但三者侧重点不同。`Reflexion` 是单条轨迹的事后复盘与重试，`Plan-and-Execute` 是先规划再逐步执行，`LATS` 则是多轨迹并行探索加分数沿父链回传，适合需要多路径尝试、且能对每条路径给出相对稳定质量信号的任务。代价是调用次数与尾延迟都会上去，因此更适合离线或异步场景。

## 核心思想

`LATS` 可以收成三句话。

- 不在一条轨迹上试错再改，而是在一棵搜索树上并行长出多条候选推理链，每次都从当前整棵树里挑一条最值得继续往下写的分支去扩写。
- 每条新写出来的回答都会走反思与打分，分数既要用来比较兄弟节点谁更好，也要沿父链往回传，更新沿途节点记在账上的总价值与访问次数，下一轮选择才有据可依。
- 下一轮具体扩谁，不是简单选当前平均分最高的叶子，而是用 `UCT` 这类规则在「已经验证过比较好的路」和「还没怎么试过的路」之间折中，避免眼睛只盯着一条看似不错、其实越走越窄的局部最优链。

`UCT` 的全称是 Upper Confidence Bound for Trees，读法上可以把它想成「带着好奇心的综合分」。一半看这条支路历史上平均反馈好不好，另一半给访问次数少的支路加一点补偿，免得整棵树只往第一次偶然走通的那条缝里钻。实现里对应 `Node` 的 `upperConfidenceBound` 方法，下面树示意里虚线框挑中的往往就是综合分最高、而不是单纯平均分最高的那个节点。

## 四个阶段与整体循环

每轮搜索分成四步，循环执行，直到判出已有满意答案或树太高收工。

- Select（选择）。在已有节点里挑出下一轮要接着写的那个节点。选择时既看这条支路历史上平均反馈好不好，也看这条支路被光顾过多少次，避免永远只在第一条候选上打转。实现上就是比较各候选子节点的 `upperConfidenceBound`。
- Expand（扩展）。从被选中的节点出发，用该节点对应的完整对话轨迹做上文，再调用 `N` 次生成模型，采样 `N` 条新的助手回答，每条挂成树里的一个新子节点。
- Reflect（反思与评估）。对每条新回答单独跑反思链，产出文字批评、数值分数、以及是否已经彻底解决问题的布尔标记，后面回传与早停都靠它。
- Backpropagate（反向传播）。把本回合的分数从叶子沿 `parent` 指针一路加到根，更新沿路每个节点的访问次数与累计价值，下一轮 Select 用的统计才是新鲜的。

进入四步循环之前，还有一步「铺根」。先用用户原题生成第一条助手回答，反思打分后把它挂成树根，后面才有兄弟可比、有父链可传。

## 树结构的直观理解

搜索会像种树一样展开。根上是原始问题，往下每一层都是该轮里并存的若干条助手回答，兄弟之间是同一步的不同口径，树越深表示推理链拉得越长。

下面备忘给出两轮扩展后的示意，括号里的两个数可以理解成「这一路累计反馈有多厚」与「这一路被选中扩展过几次」。虚线框表示下一轮 `Select` 打算接着写的节点，也就是各子节点里 `upperConfidenceBound` 暂时领先的那一个。

![20260324163053](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260324163053.png)

只看平均分，很容易盯死在当前分数略高的一条路上。`UCT` 会给访问次数少的分支额外加成，于是可能出现图上那种情况：节点 C 平均分不是整棵树最高，但综合得分仍然可能被选中，下一轮从它往下再长新枝。这样搜索不会只在 A 的子树上越钻越深，而是留出口去试 C 这类仍可能藏着高性价比答案的分支。若某个子节点还没被访问过，实现里通常直接用它的初始 `value` 当作综合分，既避免除零，也鼓励每个新节点至少被扩展一次。

### 全局树状态 `TreeState`

实现里常用一个小结构盯住树根和原始输入。`TreeState` 里通常有根指针 `root` 和用户原句 `input`。沿 `parent` 与 `children` 能从根溜到任意叶子，也能从任意子节点沿 `parent` 链爬回根做反向传播，把本轮反思得到的分数分摊到整条路径上的统计里。

`LATS` 不是只比较哪条回答当前更顺，而是把每轮新分数回写进整棵树的历史统计，再据此决定下一轮从哪里继续扩写。`Select`、`Expand`、`Reflect`、`Backpropagate` 这四步和代码里的 `bestChild` 选择逻辑，本质上都依赖这条持续更新的父链统计。

![20260416101134](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260416101134.png)

分数沿父链回写后，节点的访问次数与价值会同步变化，下一轮扩写入口也会随之调整。树不是一次生成后静态不变，而是每轮都在旧统计基础上继续修正，这也是 `LATS` 区别于单条轨迹反复重试的关键。

## 反思打分的结构

每次扩展出 N 个候选后，都要给每个候选产出可回传的评估结果，否则反向传播就没有稳定信号。反思链的输入是用户原始问题和当前候选回答，输出必须结构化，至少包含三样东西。

- 文字反思（`reflections`），对候选的充分性、多余内容、整体质量做批评与总结。后续扩写时可选地拼回上下文，让模型看见上一轮被指出了什么。
- 数值分数（`score`），常用 0 到 10 分，再压到 0 到 1 参与反向传播，并和访问次数一起进入节点选择，避免量纲和 `visits` 混在一起难比较。
- 是否已完全解决问题（`foundSolution`），布尔量，用于提前终止搜索，也会在实现里自底向上标记整条路径已得解，主循环可以立刻收工。

若任务有外部反馈，例如代码生成后跑单元测试、数学题有标准答案，应把外部结果一并喂给反思链。这样打分更可靠，搜索也更容易区分真正好的轨迹与看起来流畅但其实错了的轨迹。纯模型自评时，建议把评分档位的 rubric 写死写细，并在解析失败时落一个保守默认分，避免一次坏 JSON 把整轮搜索搅散。

```typescript
export interface Reflection {
  reflections: string;
  score: number;
  foundSolution: boolean;
}

export function normalizedScore(r: Reflection): number {
  return Math.max(0, Math.min(1, r.score / 10));
}
```

这个接口的关键不是字段多，而是字段职责清楚。`reflections` 负责可解释性，`score` 负责可比较性，`foundSolution` 负责可终止性，三者组合后主循环才能既会选、也会停。

## Node 类的实现

节点类维护消息列表、父指针、子节点列表、`value`、`visits`、反思结果、深度，以及该子树是否已包含得解的标记。`value` 与 `visits` 是 `Select` 与 `UCT` 的账本，`bestChild` 会在所有后代里比较 `upperConfidenceBound`，相当于在整片子森林里挑下一轮扩写入口。

建节点时若传入 `reflection.foundSolution === true`，会自底向上把整条路径标为已得解，方便 `shouldLoop` 提前结束。反向传播在构造函数里自动触发，有 `reflection` 就用 `normalizedScore` 从当前节点一路更新到根，这样新建叶子时父链统计立刻跟上，不必在业务代码里手写一层回传。

```typescript
export type Message = { role: string; content: string };

export interface TreeState {
  root: Node | null;
  input: string;
}

export class Node {
  messages: Message[];
  parent: Node | null;
  children: Node[];
  value: number;
  visits: number;
  reflection: Reflection | null;
  depth: number;
  private _isSolved: boolean;

  constructor(
    messages: Message[],
    reflection: Reflection | null = null,
    parent: Node | null = null,
  ) {
    this.messages = messages;
    this.parent = parent;
    this.children = [];
    this.value = 0;
    this.visits = 0;
    this.reflection = reflection;
    this.depth = parent ? parent.depth + 1 : 1;
    this._isSolved = reflection?.foundSolution ?? false;
    if (this._isSolved) this._markTreeAsSolved();
    if (reflection) this.backpropagate(normalizedScore(reflection));
  }

  get isSolved(): boolean {
    return this._isSolved;
  }

  upperConfidenceBound(c: number = 1): number {
    if (!this.parent) throw new Error("UCT 只适用于非根节点");
    if (this.visits === 0) return this.value;
    const avg = this.value / this.visits;
    const exploration =
      c * Math.sqrt(Math.log(this.parent.visits) / this.visits);
    return avg + exploration;
  }

  backpropagate(reward: number): void {
    let node: Node | null = this;
    while (node) {
      node.visits += 1;
      node.value = (node.value * (node.visits - 1) + reward) / node.visits;
      node = node.parent;
    }
  }

  getMessages(includeReflections: boolean): Message[] {
    if (includeReflections && this.reflection) {
      return [
        ...this.messages,
        {
          role: "user",
          content: `思考过程：${this.reflection.reflections}\n评分：${this.reflection.score}`,
        },
      ];
    }
    return [...this.messages];
  }

  getTrajectory(includeReflections = true): Message[] {
    const segments: Message[][] = [];
    let node: Node | null = this;
    while (node) {
      segments.push(node.getMessages(includeReflections));
      node = node.parent;
    }
    return segments.reverse().flat();
  }

  get bestChild(): Node | null {
    const descendants = this._getAllDescendants();
    if (descendants.length === 0) return null;
    return descendants.reduce((a, b) =>
      a.upperConfidenceBound() >= b.upperConfidenceBound() ? a : b,
    );
  }

  get height(): number {
    if (this.children.length === 0) return 1;
    return 1 + Math.max(...this.children.map((c) => c.height));
  }

  getBestSolution(): Node {
    const nodes = [this, ...this._getAllDescendants()];
    const terminalSolved = nodes.filter(
      (n) => n.children.length === 0 && n.isSolved,
    );
    if (terminalSolved.length === 0) {
      return nodes.reduce((a, b) => (a.value >= b.value ? a : b));
    }
    return terminalSolved.reduce((a, b) => (a.value >= b.value ? a : b));
  }

  private _getAllDescendants(): Node[] {
    const out: Node[] = [];
    const queue: Node[] = [...this.children];
    while (queue.length) {
      const n = queue.shift()!;
      out.push(n);
      queue.push(...n.children);
    }
    return out;
  }

  private _markTreeAsSolved(): void {
    let p: Node | null = this.parent;
    while (p) {
      (p as any)._isSolved = true;
      p = p.parent;
    }
  }
}
```

读这段实现可以先抓三件事。`upperConfidenceBound` 决定扩写入口怎么选，`backpropagate` 决定分数怎么回灌父链，`getBestSolution` 决定收尾时把哪条轨迹交给用户，这三处连起来就是 `LATS` 的最小闭环。

## 接入 LangChain 的完整实现

接下来我们用 `LATS` 的最小可运行链路完整串起来，包括 `reflectionChain`（反思打分）、`streamAnswer`（流式生成回答）、`generateInitialResponse`（初始响应）、`expand`（扩展）、`runLats`（主循环）。推荐按调用顺序阅读，先看模型与反思链，再看扩展和主循环，最后回头对照 `bestChild` 与 `upperConfidenceBound` 如何驱动下一轮扩写。

### 模型与反思链

反思链和回答链分开配置更稳。反思模型开启 `json_object` 响应格式，输出可以直接过 `zod` 校验；提示模板里的 `JSON` 示例用双花括号 `{{}}` 转义，避免被 `LangChain` 模板引擎误识别为变量。反思过程同样走流式输出，终端不会长时间静默。示例里的密钥统一从环境变量读取，避免真实 `apiKey` 落库。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { z } from "zod";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: process.env.DEEPSEEK_API_KEY,
  configuration: { baseURL: "https://api.deepseek.com" },
});

const ReflectionSchema = z.object({
  reflections: z.string().describe("对回答的详细批评与总结"),
  score: z.number().describe("0 到 10 的整数评分"),
  foundSolution: z.boolean().describe("true 表示该回答已完全解决问题"),
});

const reflectionPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是一位严格的评估专家，请对助手的回答进行深入反思，给出详细的评估。

评分标准：
- 0-3 分：回答跑题、错误明显或完全没有解决问题
- 4-6 分：部分解决，但有遗漏或表述不清
- 7-9 分：基本解决，细节可以更完善
- 10 分：完整、准确、清晰地解决了问题

请严格以 JSON 格式输出，不要添加任何其他文字：
{{"reflections": "详细评估内容", "score": 数字, "foundSolution": true或false}}`,
  ],
  [
    "human",
    `用户问题：{input}

助手回答：{candidate}

请对这个回答进行评估。`,
  ],
]);

const reflectionModel = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: process.env.DEEPSEEK_API_KEY,
  configuration: { baseURL: "https://api.deepseek.com" },
  modelKwargs: { response_format: { type: "json_object" } },
});

async function reflectionChain(
  input: string,
  candidate: string,
): Promise<Reflection> {
  const stream = await reflectionPrompt
    .pipe(reflectionModel)
    .stream({ input, candidate });
  let text = "";
  for await (const chunk of stream) {
    const part = typeof chunk.content === "string" ? chunk.content : "";
    process.stdout.write(part);
    text += part;
  }
  process.stdout.write("\n");

  try {
    const jsonMatch = text.match(/\{[\s\S]*\}/);
    const parsed = JSON.parse(jsonMatch ? jsonMatch[0] : text);
    const result = ReflectionSchema.parse(parsed);
    return {
      reflections: result.reflections,
      score: Math.min(10, Math.max(0, result.score)),
      foundSolution: result.foundSolution,
    };
  } catch {
    return { reflections: text, score: 3, foundSolution: false };
  }
}
```

这段代码做了三层防护：先强约束输出格式，再用 `zod` 校验字段，最后在解析失败时回落到保守默认分。这样即使某次反思输出不稳定，主循环也能继续跑，不会被一条异常响应直接打断。

### 流式生成回答

回答阶段同样走流式输出，逐 `token` 打印，让调用侧能实时感知生成状态，也便于排查模型卡顿是在生成端还是反思端。

```typescript
async function streamAnswer(messages: Message[]): Promise<string> {
  const stream = await model.stream(messages);
  let content = "";
  for await (const chunk of stream) {
    const text = typeof chunk.content === "string" ? chunk.content : "";
    process.stdout.write(text);
    content += text;
  }
  process.stdout.write("\n");
  return content;
}
```

`streamAnswer` 的职责很单一，边收边打、完整拼回字符串再返回。后面不管是初始回答还是扩展候选，都复用这一层，避免两套输出逻辑造成日志风格不一致。

### 初始响应

```typescript
async function generateInitialResponse(state: TreeState): Promise<TreeState> {
  console.log("\n[初始回答]\n");
  const content = await streamAnswer([
    {
      role: "system",
      content:
        "你是一位知识渊博、思维清晰的 AI 助手，请用中文回答用户的问题，力求准确、完整、易于理解。",
    },
    { role: "user", content: state.input },
  ]);

  console.log("\n[评估初始回答]\n");
  const reflection = await reflectionChain(state.input, content);
  console.log(
    `\n  评分：${reflection.score}/10，已解决：${reflection.foundSolution}`,
  );

  const root = new Node([{ role: "assistant", content }], reflection, null);
  return { root, input: state.input };
}
```

初始响应这一步相当于给树铺根。先生成第一条回答，再立刻评估并建 `root`，后面的所有扩展都从这个根出发，少了这步就没有可比较的父链统计。

### 扩展

`expand` 负责一轮真正的搜索增量。它先从当前树里取 `upperConfidenceBound` 最大的节点（实现里通过 `bestChild` 落在后代上），再基于该节点完整轨迹生成 N 个候选，并逐条跑反思。这里用顺序循环而非 `Promise.all`，是为了保持输出可读，避免多个候选与反思并发打印后互相打乱。子节点构造时会自动触发反向传播。

```typescript
async function expand(state: TreeState, N: number = 3): Promise<TreeState> {
  const root = state.root!;
  const best = root.bestChild ?? root;
  const trajectory = best.getTrajectory(false);
  const systemMsg: Message = {
    role: "system",
    content:
      "你是一位知识渊博的 AI 助手，请用中文回答用户问题，力求准确、完整。如果之前的回答有不足，请给出更好的答案。",
  };

  const candidates: string[] = [];
  for (let i = 0; i < N; i++) {
    console.log(`\n  [候选 ${i + 1}/${N}]\n`);
    const content = await streamAnswer([
      systemMsg,
      ...trajectory,
      { role: "user", content: state.input },
    ]);
    candidates.push(content);
  }

  for (let i = 0; i < candidates.length; i++) {
    console.log(`\n  [评估候选 ${i + 1}/${N}]\n`);
    const r = await reflectionChain(state.input, candidates[i]);
    console.log(`\n  → 评分：${r.score}/10，已解决：${r.foundSolution}`);
    best.children.push(
      new Node([{ role: "assistant", content: candidates[i] }], r, best),
    );
  }

  return state;
}
```

这段最关键的是两次循环分工。第一次循环只负责生候选，第二次循环只负责评估和挂子节点。职责拆开后，后续若要引入并发评估或批量打分，改动点会更清楚。

### 终止条件与主循环

`shouldLoop` 把收束条件独立出来，避免终止逻辑散落在主循环里。`runLats` 叠加最大迭代次数与实时日志，能在质量和成本之间保持可控。跑通示例前请在运行环境导出 `DEEPSEEK_API_KEY`，并确认与上面 `ChatOpenAI` 的 `baseURL` 指向同一推理服务。

```typescript
function shouldLoop(state: TreeState, maxHeight: number = 5): "expand" | "end" {
  const root = state.root;
  if (!root) return "end";
  if (root.isSolved) return "end";
  if (root.height > maxHeight) return "end";
  return "expand";
}

async function runLats(
  inputQuery: string,
  maxIterations: number = 3,
): Promise<string> {
  console.log("═".repeat(60));
  console.log("LATS 启动");
  console.log(`问题：${inputQuery}`);
  console.log("═".repeat(60));

  let state: TreeState = { input: inputQuery, root: null };
  state = await generateInitialResponse(state);
  if (!state.root) return "初始响应生成失败。";

  for (let i = 0; i < maxIterations; i++) {
    if (shouldLoop(state) === "end") {
      console.log(`\n[终止] 已找到满意解或达到最大深度，停止搜索`);
      break;
    }
    console.log(
      `\n${"─".repeat(60)}\n[迭代 ${i + 1}] 树高度：${state.root!.height}，开始扩展...`,
    );
    state = await expand(state);

    const allNodes = getAllNodes(state.root!);
    const bestScore = Math.max(...allNodes.map((n) => n.value));
    console.log(
      `\n[迭代 ${i + 1}] 扩展完成，当前最高节点价值：${bestScore.toFixed(3)}`,
    );

    if (state.root!.isSolved) {
      console.log(`[迭代 ${i + 1}] 已找到满意解，提前终止`);
      break;
    }
  }

  const solutionNode = state.root!.getBestSolution();
  const trajectory = solutionNode.getTrajectory(false);
  const last = trajectory[trajectory.length - 1];

  console.log("\n" + "═".repeat(60));
  console.log(`搜索结束，最优节点评分：${solutionNode.value.toFixed(3)}`);
  console.log("═".repeat(60));

  return last?.content ?? "未找到解答。";
}

function getAllNodes(root: Node): Node[] {
  const result: Node[] = [root];
  const queue: Node[] = [...root.children];
  while (queue.length) {
    const n = queue.shift()!;
    result.push(n);
    queue.push(...n.children);
  }
  return result;
}

async function main() {
  const question =
    "请解释什么是递归，并用 TypeScript 写一个用递归计算斐波那契数列的函数，同时说明它的时间复杂度。";
  const answer = await runLats(question, 3);
  console.log("\n最终答案：");
  console.log("─".repeat(60));
  console.log(answer);
}

main().catch(console.error);
```

主循环结束后通过 `getBestSolution` 统一收口，把最优节点对应的最后一条回答返回。这样调用侧不需要关心树内部结构，只接收一个稳定的最终答案字符串。

## 三种架构的对比

`LATS`、`Reflexion`、`Tree-of-Thoughts` 都属于多步推理家族，差别主要在「有没有树」「有没有逐节点反思」「分数会不会沿父链回去改账本」。下面三张备忘各自对应一种典型工作方式，读的时候可以对照我们前面写的四阶段循环，看信息是只在一条线上走，还是在整棵树上摊开来用。

`Reflexion` 是最直接的一种，在单条轨迹上执行、反思、记忆、重试，循环直到得解。它不维护兄弟分支，因此省调用，但也少了并行对比，错误类型若是「第一步方向就偏了」，要多轮复盘才扭得回来。

![20260327141619](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327141619.png)

备忘概括的是执行、事后反思、更新记忆、再次重试这一条直线，适合短任务或错误容易定位在最后一两步的情形。

`Tree-of-Thoughts` 在每一层并行展开多个候选，靠投票或打分剪枝，再沿幸存分支继续深入。它强调扇出与剪枝，通常不会在每个节点上做细颗粒度的反思回传，因此实现相对轻，但对「什么叫更好」的投票规则要求更明确。

![20260327142241](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327142241.png)

备忘概括的是扇形展开、并行候选、投票或打分后保留优胜者再往下长。

`LATS` 把 `MCTS` 的四个阶段收成主循环，用 `upperConfidenceBound` 这类综合分在子节点里挑扩写入口，每个新叶子的反思分再沿父链回传，整棵树的统计随迭代更新。它最重，但在「多步、易偏、又能打分」的问题上往往最敢花算力去换稳定性。

![20260327142723](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260327142723.png)

备忘概括的是选择扩写点、并行长出候选、逐条反思、分数回传，直到判出得解或触顶。

三者的核心区别如下。

| 维度     | Reflexion          | Tree-of-Thoughts   | LATS               |
| -------- | ------------------ | ------------------ | ------------------ |
| 搜索结构 | 单条轨迹           | 树，只向前扩展     | 树，有回传         |
| 反思方式 | 事后总结           | 无反思，仅投票     | 每节点打分反向传播 |
| 分支策略 | 无分支             | 并行生成、投票剪枝 | 综合分引导选择     |
| 终止条件 | 得解或超轮数       | 达到深度限制       | 得解或超深度       |
| 适合场景 | 短任务、错误可定位 | 有明确投票标准     | 多步复杂、能评分   |

如果只从工程落地角度选型，可以记成一句话：`Reflexion` 用最少调用换快速复盘，`Tree-of-Thoughts` 用并行候选换更稳的前向筛选，`LATS` 用更高算力预算换全局统计与回传纠偏。

实践里通常先问三件事：任务是否长链路且容易早期跑偏、是否存在可执行的中间评分、是否愿意为稳定性支付更多推理成本。三问里前两项越明确，越值得从 `Reflexion` 往 `LATS` 方向升级；若评分标准不清或预算受限，`Reflexion` / `Tree-of-Thoughts` 往往更实用。

一屏把单线复盘、扇出剪枝与树加回传三条路线的气质差异收在一起，便于和上文三张备忘图对照，并在选型时快速对齐成本与收益。

## 什么时候用 LATS

`LATS` 适合同时满足这几点的任务。

- 需要多步推理或多次尝试，单条轨迹容易跑偏，且我们愿意付出多倍 `LLM` 调用换更稳的终点。
- 能对轨迹或步骤给出质量分数，无论是模型自反思还是外部可验证信号，否则 `Select` 会在噪声里乱晃。
- 对结果质量的要求明显高于对尾延迟的要求，例如离线代码修复、夜间批处理、或人机协同里允许等一会儿再给答案。

典型场景包括复杂代码生成里多版并行、用单元测试当外部分数，多步数学推理里每步可验算，或检索加综合的长问答里对每条综合结果单独反思。不适合的是强实时对话、或任务根本给不出稳定好坏标准的情形。粗算成本时，可以按每轮大约一次扩写入口选择、加 `N` 次生成、再加 `N` 次反思来估 `API` 账单，心里要有数。

## 实现与落地注意点

效果对奖励质量非常敏感。完全依赖模型自反思打分时，噪声比较大，一旦有外部反馈，例如测试通过与否、人工标注、环境奖励，尽量接入反思与打分阶段，这样 `upperConfidenceBound` 挑出的扩写入口才和真实质量对齐。

另一个值得考虑的用法是把 `LATS` 当作离线数据生成器。先跑出一批高质量轨迹，再存成微调或偏好对齐数据，线上改用单条轨迹或小模型，在质量和延迟之间折中。自 `LATS` 提出以来，推理成本已明显下降，这种离线大树搜索、线上小模型推理的组合会越来越常见。

实现时要同时卡住最大深度与最大轮数，避免在无解或语义含糊的任务上无限展开。`maxHeight` 建议从 5 试起，`N` 从 3 到 5 试起，过大的 `N` 会让每轮调用数近似线性膨胀。反思模型建议单独配置并开启 `JSON` 模式，例如 `response_format: { type: 'json_object' }`，配合流式输出便于排错。若反思与生成共用同一速率限制，建议拆队列或拆账号，否则搜索会在等待反思时整体卡顿。

## 总结

`LATS` 把蒙特卡洛树搜索的探索习惯移植到 `LLM` 推理里，让智能体不再只依赖单条试错再改的轨迹，而是在多候选分支上并行推进，通过反思打分和带探索项的综合分选择，动态决定下一轮从哪条支路接着写。核心贡献是把质量评估嵌进搜索循环，每个节点的分数既影响当前扩哪，也通过反向传播影响整棵树的走向。

相比 `Reflexion` 的线性复盘，`LATS` 更容易系统性地跳出局部最优。相比 `Tree-of-Thoughts` 的无反思投票，`LATS` 的评分与回传对复杂任务通常有更高区分度。

实际落地里，奖励信号的质量决定效果上限。外部可验证反馈，例如单元测试、标准答案、环境奖励，通常比纯模型自反思更可靠。在质量优先、延迟可换的场景里，`LATS` 仍是当前 `LLM` 推理架构里较系统化的多路径搜索方案之一，前提是我们愿意为观测、调参与算力预算留足工程空间。
