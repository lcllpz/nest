前面几章里，Agent 拿到用户指令以后，可以自己决定调哪些工具、按什么顺序调、什么时候结束，整个过程几乎是"放养"状态。这样虽然高效，但一旦涉及写文件、删库、发邮件、调用付费接口这些高风险操作，就很容易让人发怵，万一模型想多了、理解错了，直接把动作落在生产环境上怎么办。

这一章可以配合安全护栏一起看，安全护栏负责"哪些事绝对不能做"和"输入输出要怎么过滤"，人机协同则是在执行高风险工具前把最后的决策权交还给人。

## Human-in-the-Loop 是什么

LangGraph 的人机协同通过两个原语实现：`interruptBefore` 和 `Command`。`interruptBefore` 让 Agent 在执行工具节点前暂停，暂停时图的状态由 `checkpointSaver` 持久化；外部系统拿到待审核的工具调用后，由人做出判断，再带着 `Command({ resume: ... })` 恢复执行。

整个流程可以概括成：

1. 每一轮模型返回后，Agent 图在进入工具节点前暂停。
2. `snapshot.messages` 的最后一条 `AIMessage` 里的 `tool_calls` 就是待审核的动作。
3. 把这些动作展示给审核人，收集决策。
4. 带着 `Command` 和同一个 `thread_id` 再次调用 Agent，继续执行。

## 三类审核决策

当中间件判断某个工具调用需要人工介入时，它会发出一个中断请求，请求里带着若干"待审核的动作"。对于每一个动作，审核者都需要给出一个决策，共三种类型。如下图所示。

![20260320091934](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320091934.png)

三种决策覆盖了审核人面对工具调用时的全部选择。

### approve 放行

`approve` 表示"这个工具调用按原样执行"，不改动任何参数。常见的使用场景有两类：

- 内容类：Agent 草拟了一封邮件、公告、回复，审核人看完觉得没问题，就同意发送。
- 运维类：Agent 生成了一条清理旧数据的 `SQL`，你确认库名、表名、条件都对，就允许执行。

```typescript
const response = {
  decisions: [
    {
      type: "approve" as const,
    },
  ],
};
```

`decisions` 数组顺序要和中断请求里列出的动作顺序一一对应。

### edit 修改再执行

`edit` 代表"我要改一改这个工具调用再执行"。比如模型已经准备好了删库的 `SQL`，但你只想删测试环境，或者想把时间窗口缩短，就可以在决策里带上 `editedAction`，替换工具名和参数。

`edit` 更适合做"轻微修订"而不是"完全换一个操作"，例如：

- 把 `DELETE` 的条件从"30 天前"改成"7 天前"。
- 把收件人由个人邮箱改成一个群组地址。
- 把写入文件的路径从生产目录改到沙箱目录。

```typescript
const response = {
  decisions: [
    {
      type: "edit" as const,
      editedAction: {
        name: "execute_sql",
        args: {
          query:
            "DELETE FROM records WHERE created_at < NOW() - INTERVAL '7 days';",
        },
      },
    },
  ],
};
```

改参数时要尽量克制，只做必要的收窄和纠正。如果完全改写了请求，模型后续可能会"看不懂自己刚刚做了什么"，重新规划步骤，甚至再次触发工具调用。

### reject 拒绝并反馈

`reject` 就是"这事不能这么干"，既不放行也不修改执行，而是给模型一条反馈信息，告诉它为什么不允许，以及希望它改怎么做。

```typescript
const response = {
  decisions: [
    {
      type: "reject" as const,
      message: "不能直接删除生产数据，请先给出备份方案再讨论。",
    },
  ],
};
```

这条 `message` 会作为 `ToolMessage` 写回到 Agent 的消息流里，帮助模型在后续轮次里修正行为。

如果同一轮中断里有多个动作要审核，就按顺序给出多条决策，每个动作对应一个。三种类型的选择参考：

- 不敏感但希望有"人眼过一遍"的操作，用 `approve`。
- 条件正确但需要略微收紧、改个参数的操作，用 `edit`。
- 模型明显理解错需求、或者动作本身不被业务允许，直接 `reject`，在 `message` 里告诉它"为什么不行"。

## 配置中断策略

`createReactAgent` 的 `interruptBefore` 选项控制在哪些节点前暂停。传入 `["tools"]` 表示任何工具调用前都会触发中断，这是最简单也最常用的配置。

```typescript
import { z } from "zod";
import { tool } from "@langchain/core/tools";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { MemorySaver, Command } from "@langchain/langgraph";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";
import type { AIMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "Pro/zai-org/GLM-5",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
  temperature: 0,
});

const writeFileTool = tool(
  async (input: { path: string; content: string }) => `已写入 ${input.path}`,
  {
    name: "write_file",
    description: "向指定路径写入文件内容。",
    schema: z.object({
      path: z.string().describe("文件路径"),
      content: z.string().describe("写入内容"),
    }),
  },
);

const executeSQLTool = tool(
  async (input: { query: string }) => `SQL 执行完成`,
  {
    name: "execute_sql",
    description: "在数据库中执行 SQL 语句。",
    schema: z.object({ query: z.string().describe("SQL 语句") }),
  },
);

const agent = createReactAgent({
  llm: model,
  tools: [writeFileTool, executeSQLTool],
  checkpointSaver: new MemorySaver(),
  interruptBefore: ["tools"], // 任何工具调用前暂停等待人工审批
});

const config = { configurable: { thread_id: "review-001" } };

// ── 第一段：触发推理，遇到工具调用时暂停 ────────────────────────────────────
const snapshot = await agent.invoke(
  { messages: [new HumanMessage("清空 /tmp/app.log 文件")] },
  config,
);

// snapshot.messages 最后一条 AIMessage 里的 tool_calls 是待审核的动作
const pendingMsg = snapshot.messages.at(-1) as AIMessage;
console.log("待审核的工具调用：", JSON.stringify(pendingMsg.tool_calls, null, 2));

// ── 第二段：人工确认后，带同一个 config 恢复执行 ─────────────────────────────
// interruptBefore 模式下直接传 null，LangGraph 从中断点继续跑
const finalResult = await agent.invoke(null, config);
console.log("最终回复：", finalResult.messages.at(-1)?.content);
```

第一段打印出待审核的工具调用（`write_file`，参数 `path=/tmp/app.log`），第二段恢复后模型确认执行并回复结果，如下图所示。

![20260320093323](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320093323.png)

如果只需要对特定工具中断、其他工具放行，在工具函数里调用 `interrupt()` 主动暂停即可，不用 `interruptBefore`：

```typescript
import { interrupt } from "@langchain/langgraph";

const executeSQLTool = tool(
  async (input: { query: string }) => {
    // 主动触发中断，暂停并把待审核信息传出去
    const decision = interrupt({ action: "execute_sql", query: input.query });
    if (decision !== "approved") return "已拒绝执行。";
    return `SQL 执行完成`;
  },
  {
    name: "execute_sql",
    description: "在数据库中执行 SQL 语句。",
    schema: z.object({ query: z.string() }),
  },
);
```

设计中断策略的基本原则：写操作（写文件、写库、发消息）一律中断，读操作默认放过，避免低风险动作频繁打扰审核人。

## 持久化与 checkpointer 配置

人机协同的前提是 Agent 能安全地"暂停"和"继续"。LangGraph 的中断机制把图的状态停在当前节点，要让这个状态跨请求保存下来，就必须配置 `checkpointSaver`：

- 在本地开发、测试环境，可以用 `MemorySaver` 或 `InMemorySaver`，把状态放在进程内存里。
- 在生产环境，推荐用 `AsyncPostgresSaver` 之类的持久化实现，把状态写进数据库，进程重启也不会丢失。

调用 Agent 时，还需要在 `config` 里提供一个 `thread_id`，把同一条对话线程的多次调用串在一起：

```typescript
import { HumanMessage } from "@langchain/core/messages";

const config = { configurable: { thread_id: "session-abc123" } };

const result = await agent.invoke(
  {
    messages: [new HumanMessage("Delete old records from the database")],
  },
  config
);
```

`thread_id` 可以用你自己的会话标识，例如用户会话 `id`、任务 `id` 等，只要后续恢复时用的是同一个值，`LangGraph` 就能从之前的中断点继续往下跑。

## 用 __interrupt__ 与 Command 恢复执行

当某次调用因为人机协同被中断时，`agent.invoke` 的返回值里会多出一个特殊字段 `__interrupt__`，包含了完整的审核请求数据：

- `action_requests`：待审核的工具调用，里面有工具名、参数、自动生成的描述等。
- `review_configs`：每个动作允许的决策类型，方便前端做表单校验。

审核完成后，通过 `Command` 把决策传回 Agent 继续执行，必须带同一个 `config`（同一个 `thread_id`），`LangGraph` 靠它定位中断点：

```typescript
import { Command } from "@langchain/langgraph";

// interruptBefore 模式：直接传 null，LangGraph 从中断点继续跑
await agent.invoke(null, config);

// 若工具函数里用了 interrupt() 并期待一个返回值，则用 Command
await agent.invoke(new Command({ resume: "approved" }), config);
```

恢复后 Agent 继续跑剩下的图节点，直到再次遇到中断或正常完成。整个调用是"两段式"：第一段触发中断并持久化状态，第二段带着决策恢复执行。

## 流式场景下的人机协同

如果你用的是 `agent.stream` 来获取流式输出，人机协同同样适用。把 `streamMode` 设成同时包含 `updates` 和 `messages`，在迭代过程中留意是否出现了中断事件：

```typescript
const config = { configurable: { thread_id: "session-abc123" } };

for await (const [mode, chunk] of await agent.stream(
  {
    messages: [
      { role: "user", content: "Delete old records from the database" },
    ],
  },
  { ...config, streamMode: ["updates", "messages"] }
)) {
  if (mode === "messages") {
    const [token] = chunk as [{ content?: string }];
    if (token.content) {
      process.stdout.write(token.content);
    }
  } else if (mode === "updates") {
    if ("__interrupt__" in chunk) {
      console.log("需要人工审核：", JSON.stringify(chunk.__interrupt__));
    }
  }
}
```

一旦检测到 `__interrupt__`，暂停前端输出，把中断里的工具调用信息展示给审核人。审核完成以后，再用同样的 `Command` 调用方式恢复执行，继续用流式接口拿到后续输出。

流式场景下的"两段式"结构和非流式一样：

- 第一段流式调用时，一边展示模型生成的思考，一边监听 `updates` 流里的中断事件。
- 一旦发现中断，结束当前流式循环，把中断信息保存下来，进入人工审核阶段。
- 审核完成后，带着 `resume.decisions` 再调用一次 `agent.stream`，继续从中断点往下跑，前端只需把第二段生成的内容接在第一段后面显示。

## 执行生命周期

每次 `agent.invoke` 触发中断时，LangGraph 把图的状态写进 `checkpointSaver`，当前节点挂起。`snapshot.messages` 最后一条 `AIMessage` 里的 `tool_calls` 就是待审核的动作，可以直接渲染进审核界面。审核完成后带 `Command` 恢复，中断点之后的节点继续执行，`ToolMessage` 写回对话，Agent 继续跑直到完成或再次中断。

如果需要多级审批或联动企业内部风控系统，可以直接基于 `interrupt()` 原语自己搭流程，不受任何固定中间件的限制。

## 小结

人机协同的核心是在模型"想要执行"和工具"真正执行"之间插入一道人工审批关卡。`interruptBefore: ["tools"]` 决定哪些节点前暂停，图的状态由 `checkpointSaver` 持久化，`snapshot.messages` 末尾的 `AIMessage.tool_calls` 暴露待审核动作，`Command({ resume: ... })` 带着决策恢复执行。

两件事要记住：

- 必须配 `checkpointSaver`，没有持久化就没有跨请求的中断恢复能力。
- 每次恢复时带上同一个 `thread_id`，LangGraph 靠它定位中断点。

实际落地时，先用 `interruptBefore: ["tools"]` 跑通整条链路，需要细粒度控制时再在特定工具里加 `interrupt()` 主动暂停，既能覆盖主要风险，又不会因为审核频率太高拖慢 Agent 的正常使用。
