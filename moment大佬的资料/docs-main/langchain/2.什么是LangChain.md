前一章我们看到，裸调 LLM `API` 会把编排、记忆、工具、提示等职责都推给开发者，每一块都要自己从头写。本章来看 `LangChain` 是如何应对这些问题的。

`LangChain` 是一个开源 LLM 编排框架，提供统一抽象和可复用组件，帮你快速把大模型与真实数据、工具、服务连接起来，构建聊天机器人、智能助手、`RAG` 应用、`Agent` 等各类 LLM 应用。

## LangChain 的来历与定位

`LangChain` 由 Harrison Chase 于 2022 年 10 月发布，最初只是一套简单的链式调用工具。此后迅速成长，到 2023 年 6 月，它曾是 GitHub 上增长最快的开源项目之一，同年完成数千万美元融资。

如今 `LangChain` 同时支持 Python 和 TypeScript，可作为几乎所有主流 LLM 的通用接口。设计哲学强调可组合性：每个组件都可以单独使用，也可以按需组合成更复杂的工作流。

## 为什么值得用

大模型本身不是独立应用程序，它是一个统计模型，必须与应用层、数据源、工作流配对才能产生实际价值。`ChatGPT` 不是 LLM 本身，而是一个使用 GPT 系列模型构建的聊天应用，背后有上下文管理、工具调用、安全过滤等大量工程逻辑在支撑。

用同一个单轮对话来对比两种写法，可以直观看出 `LangChain` 做了什么。

裸调方式：

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.DEEPSEEK_API_KEY,
  baseURL: "https://api.deepseek.com",
});

const response = await client.chat.completions.create({
  model: "deepseek-chat",
  messages: [{ role: "user", content: "你好！" }],
  stream: false,
});

console.log(response.choices[0].message.content);
```

`LangChain` 方式：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const response = await model.invoke([new HumanMessage("你好！")]);
console.log(response.content);
```

两段代码的效果一样，但 `LangChain` 版本拿到的 `model` 对象已经具备了统一接口。切换到 OpenAI 或 Claude，只需改初始化参数，后续所有调用代码不用动。这只是最表层的差异，随着功能需求增加，两者的差距会越来越大。

## 六项核心能力

`LangChain` 的价值体现在以下六个方面，从模型接入到工具调用，从数据检索到记忆管理，共同构成构建 LLM 应用所需的基础设施。

![20260309100646](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260309100646.png)

上图展示了六项能力的全貌：模型抽象统一接口、提示工程模板化管理、数据与检索支撑 `RAG` 链路、工具调用赋能 `Agent`、链与编排将各组件串联成流程、记忆与状态让应用具备上下文感知。

### 1. 模型抽象

`LangChain` 用统一接口封装多种 LLM，包括 `OpenAI`、`Anthropic`、`Hugging Face` 上的开源模型（如 `Llama`、`Mistral`、`Gemma`）等。切换模型通常只需改一行初始化配置，无需重写业务代码。

还可以同时使用多个模型：用轻量模型做意图分类，用大模型做长文本生成，在效果与成本之间灵活取得平衡。私有或自研模型可通过 `CustomLLM` 类包装接入，适配自建推理服务。

### 2. 提示工程

`PromptTemplate` 把提示内容参数化，将上下文、指令、`Few-shot` 示例、输出格式要求等作为变量注入，避免硬编码。

除基础模板外，还提供 `ChatPromptTemplate`（多轮对话模板）、`FewShotPromptTemplate`（动态少样本）等变体，支持结构化输出和格式约束。提示从散落的字符串升级为可版本化、可复用、可 A、B 测试的资产，改起来不再需要重新部署。

### 3. 数据与检索

`LangChain` 提供完整的 `RAG` 链路：

- 文档加载器：支持从 `Notion`、`Google Drive`、`YouTube`、`MongoDB`、`GitHub` 等数十种来源导入数据
- 文本分割器：按字符、句子、段落或语义边界切分，兼顾检索精度与速度
- 向量数据库：支持 50 种以上向量存储，本地可用 `Chroma`、`FAISS`，云端可用 `Pinecone`、`Weaviate`
- 检索器：接受查询字符串，返回语义相关文档，作为上下文注入模型

`RAG` 从私有或实时知识库中检索相关内容再交给模型回答，是 LLM 接入企业数据最常见的方式，显著提升回答的准确性和时效性。

### 4. 工具调用

`Tool` 抽象让 LLM 能够调用函数、API、搜索引擎等外部能力。工具以函数签名的形式暴露给模型，模型根据用户意图自主决定是否调用、传入哪些参数。

内置工具包括 `Google` 搜索（实时信息）、`Wolfram Alpha`（数学计算）、`OpenWeatherMap`（天气数据）等，也可以把任何自定义函数包装成工具暴露给模型。这种"模型决策 + 工具执行"的模式是构建 `Agent` 的基础。

### 5. 链与编排

`Chain` 将模型与其他组件组合起来，执行一系列功能。最基础的 `LLMChain` 只调用一个模型和一个提示模板；`SequentialChain` 可以将多个步骤串联，每步的输出作为下一步的输入，各步可以使用不同的提示、工具，甚至不同的模型。

更复杂的场景可以用 `LangGraph` 建模：把工作流表示为图（节点、边、状态），支持条件分支、并行执行、循环推理和多 `Agent` 协作，后续章节会详细展开。

### 6. 记忆与状态

LLM 默认无状态，对之前的对话没有记忆。`LangChain` 提供多种记忆策略：

- `ConversationBufferMemory`：保留完整对话历史，适合短对话
- `ConversationSummaryMemory`：只保留摘要，节省 `token`，适合长对话
- `ConversationBufferWindowMemory`：只保留最近 N 轮，兼顾成本与连贯性

记忆可与会话 ID 绑定，支持多用户、多线程场景。也可以与向量库结合，从历史对话中检索最相关的片段注入上下文，而不是简单保留最近几轮，实现更智能的长期记忆。

## 周边生态

除核心库外，`LangChain` 与以下工具协同，形成从开发到部署的完整链路：

- `LangSmith`：监控、调试和评估平台，自动追踪每一次模型调用，可视化链路执行过程，支持提示版本对比和数据集评测，是定位问题和优化效果的主要工具
- `LangGraph`：同门下的 Agent 编排框架，把工作流建模成有向图，支持有状态的多步推理、多 `Agent` 协作和人机协同，适合复杂 Agent 场景，后续章节重点讲解
- `LangGraph Platform`：部署和扩展 `Agent` 的托管平台，支持多租户、可视化编排和团队协作

三者各司其职：`LangChain` 负责构建，`LangSmith` 负责调试与评测，`LangGraph` 负责复杂工作流编排，`LangGraph Platform` 负责生产部署。

## 小结

`LangChain` 是 LLM 应用的胶水层，通过统一抽象把模型、提示、数据、工具、记忆等组件粘合在一起，让开发者专注业务逻辑，而不是重复造基础设施。

简单链式调用或轻量 `Agent` 用 `LangChain` 核心库即可，复杂、有状态的多步 `Agent` 更适合用 `LangGraph`，两者共用同一套组件体系，可以无缝组合。
