`Subagent` 可以理解成"被别的 `Agent` 当工具调用的 `Agent`"，它不直接面对用户，而是挂在主 `Agent` 名下由其统一调度。

最常见的结构是"Supervisor 加多名 Subagent"。`Supervisor` 负责和用户对话、拆解任务、维护对话历史，每名 `Subagent` 只专注在自己的领域完成子任务。

`Subagent` 有三个特点：

1. 通常不直接与用户交互，通过主 `Agent` 间接被调用。
2. 从主 `Agent` 视角看，`Subagent` 就是一项带结构化入参与出参的工具。
3. 设计上是无状态的，同次调用内部有推理历史，但不跨调用保留记忆，长程记忆集中在主 `Agent` 身上。

下图展示了用户、`Supervisor`、多名 `Subagent` 之间的关系与消息流向。

![20260320113940](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320113940.png)

用户始终只和 `Supervisor` 对话，路由决策也由它完成。`Subagent` 之间互不感知，各自把结果交还给 `Supervisor` 统一整理。主 `Agent` 可以在同一轮里并行调用多名 `Subagent`，每个子 `Agent` 的上下文彼此独立，有利于控制窗口长度和排查问题。

## 什么时候用 Subagent

适合引入的场景：

- 不同子任务需要完全不同的提示词和工具
- 单个子任务足够复杂，独立拆出更好维护
- 业务域天然分散，彼此上下文互不相关
- 需要上下文隔离，避免主 `Agent` 上下文持续膨胀

不太适合的场景：

- 一两步能搞定的小任务
- 子任务之间强依赖中间上下文
- 只是工具多了点，任务本身并不复杂
- 系统规模还很小，引入多 `Agent` 反而增加维护负担

经验判断信号只有一条，主 `Agent` 的系统提示词开始写满"如果场景 A 就这样、如果场景 B 就那样"，维护起来越来越费劲，就该考虑引入 `Subagent` 了。如下图所示。

## Subagent 需要哪些配置

每个 `Subagent` 有四个核心配置：

- `name`：简短清晰，方便路由和日志识别。
- `description`：说清在什么情况下调用，是路由决策最关键的配置，官方称 `subagent specs`。
- `system_prompt`：专属提示词，需反复强调"结论和必要上下文要全部带上"，因为主 `Agent` 只能看到最后一条回复。
- `tools`：该 `Subagent` 独享的工具列表。

可选增强项有三个：单独的 `model`（按任务成本选型）、专属中间件（安全护栏、审核流程）、按需加载的技能集。

`Subagent` 数量少时直接在主 `Agent` 提示词里列出即可，数量多时配套 `list_agents` 工具做动态发现。`Supervisor` 最终基于 `name` 和 `description` 完成路由。如下图所示。

![20260320114929](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320114929.png)

从四个必填配置、三个可选增强，到 `Supervisor` 基于 `name` 和 `description` 动态路由的完整配置全景。

## Supervisor 和 Subagent 怎么配合

两者职责划分很清晰。`Supervisor` 负责对话、拆解任务、汇总结果，并维护整个会话的长期历史。`Subagent` 专注在自己的领域完成子任务，把结论压缩成清晰输出交还，保持自己的上下文窗口干净。

代码层面通常三步走：先用工厂函数统一封装 `Subagent` 的提示词、工具和中间件，再用 `tool` 包一层挂进主 `Agent` 的工具列表，`Supervisor` 调用 `Subagent` 和调用普通工具没有区别。

上下文流动有两种做法：简单时只传任务描述、返回文本结果，复杂时传入部分对话历史并让子 `Agent` 通过结构化返回值写回共享状态。如下图所示。

![20260320115151](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320115151.png)

`Supervisor` 负责对话协调、`Subagent` 以工具形式被调用，以及上下文在两者之间传递的两种做法。

## Subagent 的实现模式

分发模式有两类：

- 一对一：每个 `Subagent` 对应独立工具，可精细控制输入输出、单独做权限和监控。
- 统一分发：只暴露一个 `task` 工具，内部按 `agentName` 路由，配套 `list_agents` 做发现，适合数量多或多团队维护的场景。

调用方式也有两种：

- 同步：等结果再继续推理，实现简单，长耗时任务会拖慢整体响应。
- 异步：先拿 `job_id` 响应用户，再通过启动、查询状态、拉取结果三类工具把结论拼回对话，适合长耗时任务。

工程上建议先跑通同步一对一，有长任务需求再扩展异步。

## 如何实现一个 Subagent

下面用一个前端改版助手的例子走完整个流程，演示"一个 `Subagent` 一个工具、同步调用"这种最常见的模式。代码分三步，先创建子 `Agent`，再用 `tool` 包一层，最后挂给 `Supervisor`。可以配合官方的 [Subagents 文档](https://docs.langchain.com/oss/javascript/langchain/multi-agent/subagents) 一起看。

### 第一步，定义 Subagent

用 `createAgent` 创建子 `Agent`，`prompt` 只写它这一块的职责范围，和 `Supervisor` 的系统提示词完全独立。同时用 `zod` 定义好期望接收的结构化入参，`techStack` 有默认值，`constraints` 是可选的，后面包进工具时直接复用这份 schema。

系统提示词里最好显式要求子 `Agent` "在最终回复里带上所有关键结论"，因为 `Supervisor` 只能看到子 `Agent` 的最后一条消息，中间的推理过程它是看不到的。

```typescript
import { createAgent, tool } from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";
import * as z from "zod";

const llm = new ChatDeepSeek({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
});

const frontendTaskSchema = z.object({
  pageDescription: z
    .string()
    .min(1, "页面描述不能为空")
    .describe("当前页面的用途、目标用户、主要内容"),
  techStack: z
    .enum(["react", "vue", "nextjs", "nuxt"])
    .default("react")
    .describe("前端技术栈，用于给出更贴近项目的建议"),
  constraints: z
    .string()
    .optional()
    .describe("设计或实现上的约束，比如必须兼容移动端、不能引入新依赖等"),
});

const frontendSubagent = createAgent({
  model: llm,
  prompt:
    "你是一名资深前端架构师，擅长做页面改版方案和组件拆分设计。根据用户描述给出可落地的实现思路和组件结构，最终回复里必须带上完整的改版建议，不能只说思路，要给出具体的结构和交互说明。",
  tools: [],
});
```

### 第二步，用 tool 把 Subagent 包一层

这一步是整个模式的关键。`Supervisor` 不直接感知子 `Agent` 的存在，它只知道有一个叫 `frontend_refactor_planner` 的工具可以调用。工具内部把结构化入参拼成子 `Agent` 能理解的消息格式，执行完后把最后一条回复抽出来返回。

`description` 里写清楚"在什么情况下调用"，这是 `Supervisor` 做路由决策的依据，比 `name` 本身更重要。

```typescript
const frontendRefactorTool = tool(
  async ({ pageDescription, techStack, constraints }) => {
    const systemPrompt = `你是资深前端架构师，主要技术栈是 ${techStack}。根据下面的页面描述和约束，给出一套可落地的改版方案，包括信息架构、组件拆分建议、关键交互说明，答案要贴近工程实践。`;

    const result = await frontendSubagent.invoke({
      messages: [
        { role: "system", content: systemPrompt },
        {
          role: "user",
          content: [
            "页面描述：",
            pageDescription,
            constraints ? `约束条件：${constraints}` : "",
          ]
            .filter(Boolean)
            .join(" "),
        },
      ],
    });

    return result.messages.at(-1)?.content;
  },
  {
    name: "frontend_refactor_planner",
    description:
      "根据页面描述和技术栈，生成一套前端页面改版方案和组件拆分建议。当用户提出页面改版、移动端适配、组件设计等需求时调用。",
    schema: frontendTaskSchema,
  },
);
```

### 第三步，创建 Supervisor 并对外暴露入口

`Supervisor` 只管和用户对话，系统提示词里说清楚遇到前端改版需求就调用 `frontend_refactor_planner`，路由决策交给模型自己根据工具的 `description` 判断。

```typescript
const supervisor = createAgent({
  model: llm,
  prompt:
    "你是总负责人 Agent，和用户直接对话。当用户的问题涉及前端页面改版、组件拆分、交互设计时，调用 frontend_refactor_planner 工具获取更细致的方案，并把结果整合成一份面向产品和前端同学都易懂的回复。",
  tools: [frontendRefactorTool],
});

export async function handleUserMessage(input: string) {
  const trimmed = input.trim();
  if (!trimmed) {
    throw new Error("用户问题不能为空");
  }

  const result = await supervisor.invoke({
    messages: [{ role: "user", content: trimmed }],
  });

  const finalMessage = result.messages.at(-1);
  return typeof finalMessage?.content === "string"
    ? finalMessage.content
    : finalMessage;
}

if (require.main === module) {
  const cliQuestion =
    process.argv.slice(2).join(" ").trim() ||
    "现在有一个课程列表页面，要在移动端和桌面端都体验良好，技术栈是 React，请给一套改版方案。";

  handleUserMessage(cliQuestion)
    .then((answer) => {
      console.log("用户需求:", cliQuestion);
      console.log("\n前端改版建议:", answer);
    })
    .catch((err) => {
      console.error("调用失败:", err);
      process.exitCode = 1;
    });
}
```

三步下来，子 `Agent` 专注在自己的领域工作，`Supervisor` 只看工具接口，`tool` 起到适配器的作用，把结构化入参翻译成子 `Agent` 能接受的消息，再把最后一条回复剥出来返回。后续扩展非常自然，按同样的套路加一个写代码的 `Subagent`、一个查数据库的 `Subagent`，整个"主管加专家团队"的架构就有了。

## 小结

`Subagent` 本质上是"被主 `Agent` 当工具调用的 `Agent`"，让你在不牺牲单一入口的前提下，把复杂系统拆成职责清晰的多个专家，每个专家在自己干净的上下文窗口里独立工作。

引入的信号只有一条，主 `Agent` 的系统提示词开始变成"如果 A 就这样、如果 B 就那样"的长说明书，维护起来越来越费劲，就该拆了。系统还小、任务还简单的时候，单 `Agent` 加普通工具往往更省事。

配置上最关键的是 `description`，写得含糊就等于白配。系统提示词里也要反复强调"把结论完整带进最后一条回复"，因为 `Supervisor` 只能看到最后那一条。

实现层面，从最简单的同步一对一模式入手，把子 `Agent` 用 `tool` 包一层挂进主 `Agent` 的工具列表，跑通之后再按需扩展，异步调用和统一分发都是在这个基础上叠加的。
