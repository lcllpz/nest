前面几篇分别讲了 `Subagent`、`Handoffs`、`Skills` 和 `Router`，各自钉住一类问题。`Subagent` 做专项分工，`Handoffs` 串多阶段，`Skills` 按需加载能力，`Router` 把请求派给合适的 `Agent`。

真落到业务里，很少只靠一种模式就能扛住。同一条用户消息可能要先进路由、再跑多阶段、中间拉起一批 `Subagent`、最后还要等人拍板。这时就要把这些积木拼成一张完整的执行图，这就是 `Custom Workflow` 要接的活儿。

`LangGraph` 里的 `StateGraph` 把每一步收成节点、把怎么走收成边，编译之后就是一张可跑的图。状态是整条链的燃料，节点读状态、写回增量，下一站在更新后的状态上接着跑。

## 什么时候需要自定义工作流

如果只是想在多个 `Agent` 之间做一次分流或接力，内置的 `Router` 和 `Handoffs` 往往够用。需要自己动手画 `Custom Workflow` 时，常见是下面几类情况。

- 一条请求要穿过多个彼此依赖的阶段，每段由不同 `Agent` 扛，下一步还取决于上一步的真实输出。
- 有些步骤可以并行跑完再汇总，不必硬串行，总耗时会好看很多。
- 流程里要带条件跳转或循环，例如审核不过要重生成、某步失败要走降级。
- 有一段子流程会在多个工作流里反复出现，想封装成子图挂载，而不是复制粘贴。

在纸上画业务流程时，只要出现菱形判断、并行分叉或回环箭头，多半就是在喊你上 `Custom Workflow`。

## 图的核心构成

`StateGraph` 可以拆成三块来想。

- 状态：整图共享的一份数据结构，用 `Annotation.Root` 声明字段和归并方式。节点返回的补丁会合并进状态，大家读写的都是同一份。
- 节点：异步函数，吃当前状态，吐出要更新的字段。可以是纯逻辑、一次 `Agent` 调用，或对外部服务的读写。
- 边：节点之间的连线。`addEdge` 固定下一跳，`addConditionalEdges` 按返回值挑路。

拼起来就是有向图。从 `START` 进场，顺着边走节点，到 `END` 收场。挂上 `checkpointer` 后，每个节点结束都会落快照，断点续跑、按节点重放才有凭据。

## 把前面的模式组合进图里

前几章的模式塞进图里时，大致会长成下面几种节点脸孔。

- `Router`：条件边节点，在 `addConditionalEdges` 里读状态里的分类结果，把流量扇到不同分支。
- `Handoffs`：可以收成子图，里面仍是阶段串联，父图眼里只是一个节点。
- `Subagent`：普通节点即可，一个节点一块专责，跑完把结果写进状态字段。
- `Skills`：在节点函数里按需 `list_skills`、`load_skill`，对图长什么样零侵入。
- 人机协同：中断点节点加上 `checkpointer`，人操作完用 `Command.resume` 接起来。

## 实战 用前端代码审查串起整条工作流

下面用前端代码审查助手从零搭一条 `Custom Workflow`。链路包括这几步：先分类请求，再并行跑三个专项审查 `Agent`（代码质量、安全、性能），最后由综合节点出一份报告。

### 第一步，定义工作流状态

状态是整图的数据总线，节点从里读入、往里写出。字段全集用 `Annotation.Root` 声明，某个字段怎么合并由对应 `Annotation` 的参数说了算。

```typescript
import { Annotation } from "@langchain/langgraph";

const ReviewState = Annotation.Root({
  userInput: Annotation<string>(),
  codeContent: Annotation<string>(),
  requestType: Annotation<"code_review" | "question" | "unknown">(),
  qualityResult: Annotation<string>(),
  securityResult: Annotation<string>(),
  performanceResult: Annotation<string>(),
  finalReport: Annotation<string>(),
  requiresApproval: Annotation<boolean>(),
});

type ReviewStateType = typeof ReviewState.State;
```

如果希望多个节点往同一字段里追加而不是覆盖，可以配 `reducer`：

```typescript
reviewComments: Annotation<string[]>({
  reducer: (prev, next) => [...prev, ...next],
  default: () => [],
}),
```

字段尽量拍扁，少搞深层嵌套，后面每个节点的输入输出才好对账。

### 第二步，定义工具和 Agent

每个专项审查背后各有一套工具和对应的 `Agent`。工具干脏活（真实项目里常对接 `AST` 分析或静态检查），`Agent` 把结果整理成人话。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { tool } from "@langchain/core/tools";
import { createAgent } from "langchain";
import { z } from "zod";

const baseModel = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const analyzeCodeQuality = tool(
  async ({ code }: { code: string }) => {
    return "检测到 2 处命名不规范、3 处函数超过 50 行，建议提取逻辑到自定义 Hook。";
  },
  {
    name: "analyze_code_quality",
    description: "分析代码质量：命名规范、函数复杂度、组件设计合理性。",
    schema: z.object({ code: z.string() }),
  },
);

const scanSecurity = tool(
  async ({ code }: { code: string }) => {
    return "发现 1 处 dangerouslySetInnerHTML 未过滤，建议接入 DOMPurify。未发现其他高危漏洞。";
  },
  {
    name: "scan_security",
    description: "扫描前端安全问题：XSS、敏感信息泄露、不安全的 DOM 操作。",
    schema: z.object({ code: z.string() }),
  },
);

const analyzePerformance = tool(
  async ({ code }: { code: string }) => {
    return "useEffect 依赖数组不完整，存在闭包陷阱。大列表未使用虚拟化，建议引入 react-window。";
  },
  {
    name: "analyze_performance",
    description: "分析前端性能问题：渲染优化、内存泄漏、不必要的重渲染。",
    schema: z.object({ code: z.string() }),
  },
);

const qualityAgent = createAgent({
  llm: baseModel,
  tools: [analyzeCodeQuality],
  systemPrompt:
    "你是代码质量专家，专注于命名规范、函数复杂度和组件设计。收到代码后调用 analyze_code_quality 工具分析，以条目形式给出改进建议。",
});

const securityAgent = createAgent({
  llm: baseModel,
  tools: [scanSecurity],
  systemPrompt:
    "你是前端安全专家，专注于 XSS、CSRF 和敏感信息泄露。收到代码后调用 scan_security 工具扫描，给出高优先级的安全加固建议。",
});

const performanceAgent = createAgent({
  llm: baseModel,
  tools: [analyzePerformance],
  systemPrompt:
    "你是前端性能专家，专注于渲染优化、内存管理和不必要的计算。收到代码后调用 analyze_performance 工具分析，给出可量化的优化建议。",
});
```

### 第三步，编写节点函数

节点函数签名统一：吃进 `ReviewStateType`，吐出 `Partial<ReviewStateType>`。这里把分类、并行审查、综合报告拆成三个节点。

```typescript
import { HumanMessage } from "@langchain/core/messages";

async function classifyRequest(
  state: ReviewStateType,
): Promise<Partial<ReviewStateType>> {
  const structuredModel = baseModel.withStructuredOutput(
    z.object({
      requestType: z.enum(["code_review", "question", "unknown"]),
    }),
  );

  const result = await structuredModel.invoke([
    {
      role: "system",
      content:
        "判断用户请求类型：code_review（包含代码需要审查）、question（纯技术问题）、unknown（其他）。",
    },
    { role: "user", content: state.userInput },
  ]);

  return { requestType: result.requestType };
}

async function runAllChecks(
  state: ReviewStateType,
): Promise<Partial<ReviewStateType>> {
  // 三个 Agent 并行执行，总耗时约等于最慢的那一个
  const [qualityResult, securityResult, performanceResult] = await Promise.all([
    qualityAgent
      .invoke({
        messages: [
          new HumanMessage(`请审查以下代码的质量：\n\n${state.codeContent}`),
        ],
      })
      .then((r) => String(r.messages.at(-1)?.content ?? "")),
    securityAgent
      .invoke({
        messages: [
          new HumanMessage(
            `请扫描以下代码的安全漏洞：\n\n${state.codeContent}`,
          ),
        ],
      })
      .then((r) => String(r.messages.at(-1)?.content ?? "")),
    performanceAgent
      .invoke({
        messages: [
          new HumanMessage(
            `请分析以下代码的性能问题：\n\n${state.codeContent}`,
          ),
        ],
      })
      .then((r) => String(r.messages.at(-1)?.content ?? "")),
  ]);

  return { qualityResult, securityResult, performanceResult };
}

async function synthesizeReport(
  state: ReviewStateType,
): Promise<Partial<ReviewStateType>> {
  const response = await baseModel.invoke([
    {
      role: "system",
      content: `综合三位专家的审查结果，生成一份结构化报告。报告包含三个等级：
- 必须修复：会导致安全漏洞或严重 Bug 的问题
- 建议优化：影响可维护性或性能的问题
- 已通过：符合规范的部分`,
    },
    {
      role: "user",
      content: `代码质量：${state.qualityResult}\n\n安全扫描：${state.securityResult}\n\n性能分析：${state.performanceResult}`,
    },
  ]);

  const finalReport = String(response.content ?? "");
  // 报告里出现必须修复这一档，说明有高危问题，需要人工介入
  const requiresApproval = finalReport.includes("必须修复");

  return { finalReport, requiresApproval };
}
```

### 第四步，组装图

节点就绪后，用 `StateGraph` 接线、上条件边，再 `compile` 成可执行应用。

```typescript
import { StateGraph, START, END, MemorySaver } from "@langchain/langgraph";

const workflow = new StateGraph(ReviewState)
  .addNode("classify", classifyRequest)
  .addNode("run_all_checks", runAllChecks)
  .addNode("synthesize", synthesizeReport)
  .addEdge(START, "classify")
  .addConditionalEdges("classify", (state) => {
    if (state.requestType === "code_review") return "run_all_checks";
    return END;
  })
  .addEdge("run_all_checks", "synthesize")
  .addEdge("synthesize", END);

const app = workflow.compile({ checkpointer: new MemorySaver() });
```

`MemorySaver` 负责把快照落在内存里，节点跑完状态就有迹可循。若要在综合之后加人工审核，可以再接一个带中断的节点，等人确认后用 `Command.resume` 往下走。

### 第五步，运行工作流

编译得到的 `app` 和普通异步入口类似，丢初始状态加 `config` 即可。

```typescript
const config = { configurable: { thread_id: "review-session-001" } };

const result = await app.invoke(
  {
    userInput: "请帮我审查这段 React 代码",
    codeContent: `function UserList({ users }) {
  const [data, setData] = useState(users);

  useEffect(() => {
    fetch('/api/users').then(r => r.json()).then(setData);
  }, []);

  return (
    <div dangerouslySetInnerHTML={{ __html: data.map(u => u.name).join(', ') }} />
  );
}`,
  },
  config,
);

console.log(result.finalReport);

if (result.requiresApproval) {
  console.log("发现必须修复的问题，已通知相关负责人。");
}
```

从分类到出报告，这条示例中间不插人。三个 `Agent` 并行跑，状态始终在 `ReviewState` 里流转，最后由综合节点收口。

## 并行执行

上面的例子在单个节点里用 `Promise.all` 并行拉了三个 `Agent`，实现成本最低。若并行单元之间有依赖，或并行个数要运行时才算出来，可以改用 `LangGraph` 的 `Send`，在图这一层做扇出。

`Send` 让你在条件边里返回一组并行任务，每条任务自带输入补丁，`LangGraph` 会调度、等齐，再汇合到后续节点。

```typescript
import { Send } from "@langchain/langgraph";

// 假设 domains 是动态确定的领域列表，例如 ["react", "css", "performance"]
.addConditionalEdges("classify", (state) => {
  if (state.requestType === "code_review") {
    return state.domains.map(
      (domain) => new Send("domain_agent", { ...state, currentDomain: domain }),
    );
  }
  return END;
})
```

审查维度不固定、或要按代码类型动态挑检查项时，`Send` 往往比在一个节点里手写调度更贴图的结构。

怎么选可以简单记两条。

- 并行单元固定、逻辑直白，用节点内的 `Promise.all`，代码短、心智负担小。
- 并行份数或每份输入要运行时决定，用 `Send` 扇出，图上一眼能看出分叉。

## 子图复用

同一段流程在多个父图里都要用，收成子图通常比复制节点堆栈更稳。子图本质是已 `compile` 的 `StateGraph`，父图 `addNode` 挂上去，外面仍是一个节点。

```typescript
// 把代码审查流程封装成可复用子图
const reviewSubgraph = new StateGraph(ReviewState)
  .addNode("run_all_checks", runAllChecks)
  .addNode("synthesize", synthesizeReport)
  .addEdge(START, "run_all_checks")
  .addEdge("run_all_checks", "synthesize")
  .addEdge("synthesize", END)
  .compile();

// 在主工作流里挂载子图，对外只是一个普通节点
const mainWorkflow = new StateGraph(ReviewState)
  .addNode("classify", classifyRequest)
  .addNode("code_review", reviewSubgraph)
  .addEdge(START, "classify")
  .addConditionalEdges("classify", (state) =>
    state.requestType === "code_review" ? "code_review" : END,
  )
  .addEdge("code_review", END)
  .compile({ checkpointer: new MemorySaver() });
```

子图状态类型最好与父图对齐。若做不到，可以用 `inputTransform`、`outputTransform` 做字段映射，子图内部仍可按自己的形状设计。

子图还能单测，用固定输入跑通后再挂进任意父图，复杂工作流里这是控复杂度的老办法。

## 调试与可观测性

节点一多，最怕不知道卡在第几步。`LangGraph` 配合 `checkpointer` 至少能看清状态和轨迹。

顺着 `getStateHistory` 可以翻到每个节点前后的状态切片：

```typescript
const history = app.getStateHistory({
  configurable: { thread_id: "review-session-001" },
});

for await (const snapshot of history) {
  console.log("下一个节点：", snapshot.next);
  console.log("当前状态：", snapshot.values);
}
```

某个节点输出不对时，可以直接改快照上的字段，再从对应检查点续跑，而不必整条链重放：

```typescript
// 获取当前状态快照
const currentState = await app.getState({
  configurable: { thread_id: "review-session-001" },
});

// 手动修正某个字段的值
await app.updateState(
  { configurable: { thread_id: "review-session-001" } },
  { qualityResult: "手动修正后的代码质量结果" },
);

// 从 synthesize 节点重新运行，不必重跑前面的节点
await app.invoke(null, {
  configurable: {
    thread_id: "review-session-001",
    checkpoint_id: currentState.config.checkpoint_id,
  },
});
```

开发阶段迭代单个节点时，这套手法能省下大量全链重跑时间。

## 小结

`Custom Workflow` 是把前面几种模式装进同一个容器的做法。`StateGraph` 用状态、节点、边搭台，你在上面接 `Router`、`Handoffs`、`Subagent` 和人机协同，就能对齐真实业务那张弯弯曲曲的流程图。

下面几点值得记住。

- 先设计状态，再写节点。字段是节点之间唯一的协议，想清楚了后面少返工。
- 节点尽量单一职责，测起来省事，换实现或调顺序也不牵一片。
- 能并行的不要硬串，独立 `Agent` 并行时，总时间接近最慢那条腿，而不是简单相加。
- 复杂子流程收成子图，单独开发、单独测试，父图仍保持清爽。
- 调试多靠 `checkpointer`，快照在手，从任意一步重跑都站得住。

系列里几种多 `Agent` 搭法，从最简单的工具调用，到用 `StateGraph` 自定义整条业务流，大体就铺到这里。真实项目里多半是几种模式混用，按图施工比凭感觉堆 `if` 要可维护得多。
