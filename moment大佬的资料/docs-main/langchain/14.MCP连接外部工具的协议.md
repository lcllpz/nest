前几章里，Agent 用的工具多半是在代码里直接用 `tool` 定义的，它们属于应用进程里的"内置工具"。但在真实项目里，你往往已经有一批命令行脚本、微服务，或是其他应用暴露出来的接口，希望被不同 Agent、不同 IDE、多个应用同时复用。这时问题就来了，你缺少一套统一的方式去描述"有哪些工具、参数长什么样、应该怎么调用"。

`MCP` 也就是 `Model Context Protocol`，就是为了解决这类问题而设计的开放协议。它把"应用如何向大模型暴露工具和上下文"这件事标准化，让工具能被发现、被描述、被一致地调用。在这个协议之上，`LangChain` 通过 `@langchain/mcp-adapters` 把 MCP 服务器上的工具接进 Agent，最终在 Agent 视角里，它们和本地定义的工具没有区别。

这一章会先把 MCP 的目标说清楚，再结合 `LangChain` 的实现梳理消息流和关键对象。最后会用一个简单的数学服务器示例演示如何自己实现 MCP 服务器，并补充几种传输方式与前几章内容的衔接关系。

## MCP 是什么、解决什么问题

`MCP` 是一套开放协议，用来统一描述和调用大模型可用的外部工具。只要遵守 MCP 规范，无论底层实现是什么，都可以被任意支持 MCP 的客户端发现和调用。协议抽象出的几个关键点：

- 工具列表：服务器需要能告诉客户端"有哪些工具"，每个工具的名字、描述、参数 schema 是什么。
- 调用方式：客户端调用一个工具时，要有统一的请求格式（工具名、参数），服务器返回也要有统一格式（内容列表等）。
- 传输层：客户端和服务器之间可以用 `stdio`、`http`、`sse` 等多种传输方式，对上层来说细节都由 MCP 客户端屏蔽。

有了 MCP 之后，同一个工具服务器可以同时被编辑器、不同 Agent 框架、甚至多个后端应用共享，每一侧只负责"按协议说话"，不用各自再实现一套"列工具、发请求、收结果"的胶水逻辑。

在 `LangChain` 里，`@langchain/mcp-adapters` 负责把 MCP 工具转换成 `LangChain` 的工具对象，Agent 只会看到一组普通 `tools`，不知道这些工具来自哪个 MCP 服务器。

### MCP 的核心对象与消息流

一次完整的 MCP 工具调用涉及三个参与者：

- MCP 服务器：实现具体业务逻辑的一方，维护工具列表，处理"列工具"与"调用工具"请求。
- MCP 客户端：实现协议细节的一方，在 `LangChain` 里就是 `MultiServerMCPClient`，负责和服务器通信、封装请求与解析返回。
- Agent 或应用：只关心"这有哪些工具可以用"和"调用后结果是什么"，不关心协议细节。

标准的消息流可以拆成四步：

1. MCP 客户端向服务器发"列工具"请求（`ListToolsRequestSchema`），服务器返回工具清单，每个工具包含 `name`、`description` 和 `inputSchema`。
2. Agent 根据模型决策选择工具名和参数，通过 MCP 客户端发"调用工具"请求（`CallToolRequestSchema`）。
3. 服务器路由到对应实现，执行后返回统一结构的 `content` 列表（通常是若干段 `type: "text"` 的文本）。
4. MCP 客户端把返回值转成 `LangChain` 能理解的工具结果，塞回 Agent 的消息流里，让模型在下一轮看到。

协议把"列工具"和"调工具"这两类交互标准化，使得你可以在不改业务代码的前提下，把同一个 MCP 服务器同时接入不同 Agent 框架，只需在各自环境里配置好 MCP 客户端即可。

## 在 LangChain 里使用 MCP 工具

接入步骤分三步：安装 `@langchain/mcp-adapters`，用 `MultiServerMCPClient` 连接一个或多个 MCP 服务器并拉取工具，把这些工具传给 `createAgent`。

```bash
pnpm add @langchain/mcp-adapters
```

安装完成后，可以先接入本章的数学服务器，跑通一条最小链路，再按需往里加更多服务器。

```typescript
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { MultiServerMCPClient } from "@langchain/mcp-adapters";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const mathServerPath = new URL("./math_server.js", import.meta.url).pathname;

const client = new MultiServerMCPClient({
  math: {
    transport: "stdio",
    command: "node",
    args: [mathServerPath],
  },
});

const tools = await client.getTools();
const agent = createAgent({ model, tools });

const result = await agent.invoke({
  messages: [{ role: "user", content: "帮我算一下 (3 + 5) * 12？" }],
});

console.log(result.messages?.at(-1)?.content);
await client.close();
```

`client.getTools()` 会向各个 MCP 服务器发起"列工具"请求，把得到的工具统一转成 `LangChain` 可用的工具对象。Agent 在运行时若决定调用某个工具，会通过 MCP 客户端向对应服务器发"调用工具"请求，并把返回结果放进对话循环里。

`MultiServerMCPClient` 默认每次工具调用都会新建一个 MCP `ClientSession`，执行完就清理，对大多数请求响应式的工具已经足够。如果你需要长连接或更细粒度的会话管理，可以参考 `LangChain` 文档里的 stateful sessions 章节，或者在 MCP 服务器一侧维护跨请求的会话信息。

## 自己实现一个 MCP 服务器

用官方的 `@modelcontextprotocol/sdk` 可以快速写一个 MCP 服务器。SDK 帮你处理大部分协议细节，你主要精力放在"有哪些工具"以及"每个工具如何处理请求"这两件事上。

下面是一个通过 `stdio` 通信的数学服务器，提供 `add` 和 `multiply` 两个工具，用 `McpServer` 封装配合 `zod` 定义参数 schema。

```typescript
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";

const server = new McpServer({
  name: "math-server",
  version: "0.1.0",
});

server.registerTool(
  "add",
  {
    description: "Add two numbers",
    inputSchema: z.object({
      a: z.number().describe("First number"),
      b: z.number().describe("Second number"),
    }),
  },
  async ({ a, b }: { a: number; b: number }) => {
    // 日志写到 stderr，避免干扰 MCP 协议的 stdout
    console.error("[math-server] add", { a, b, result: a + b });
    return {
      content: [{ type: "text", text: String(a + b) }],
    };
  },
);

server.registerTool(
  "multiply",
  {
    description: "Multiply two numbers",
    inputSchema: z.object({
      a: z.number().describe("First number"),
      b: z.number().describe("Second number"),
    }),
  },
  async ({ a, b }: { a: number; b: number }) => {
    console.error("[math-server] multiply", { a, b, result: a * b });
    return {
      content: [{ type: "text", text: String(a * b) }],
    };
  },
);

async function main(): Promise<void> {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("Math MCP server running on stdio");
}

main().catch((err) => {
  console.error("Math MCP server crashed:", err);
  process.exit(1);
});
```

这个服务器启动后，在 `MultiServerMCPClient` 的配置里增加一项 `math`，指定 `transport` 为 `stdio` 并写清 `command` 和 `args`，Agent 就可以通过 MCP 客户端复用这些数学工具。

从协议层面看，`McpServer` 帮你把 `ListTools` 与 `CallTool` 的细节封装掉了：

- "有哪些工具"：`registerTool` 里写出的 `zod` schema 会被 SDK 转换成标准 JSON Schema，让客户端知道如何构造参数。
- "怎么调这个工具"：工具对应的处理函数执行业务逻辑后，返回带 `type` 字段的 `content` 列表，便于上层统一解析。

可以按下面的代码写一个最小可运行的调用入口，支持从命令行传入提问。

```typescript
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { MultiServerMCPClient } from "@langchain/mcp-adapters";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const userQuestion =
  process.argv.slice(2).join(" ").trim() || "帮我算一下 (3 + 5) * 12？";

const mathServerPath = new URL("./math_server.js", import.meta.url).pathname;

const mcpClient = new MultiServerMCPClient({
  math: {
    transport: "stdio",
    command: "node",
    args: [mathServerPath],
  },
});

const tools = await mcpClient.getTools();
const agent = createAgent({ model, tools });

const result = await agent.invoke({
  messages: [{ role: "user", content: userQuestion }],
});

console.log("用户提问：", userQuestion);
console.log("模型最终回答：", result.messages?.at(-1)?.content);

await mcpClient.close();
```

实际跑起来之后，Agent 会先调用 `add` 把 3 和 5 相加，再用 `multiply` 把结果和 12 相乘，最终把计算过程和答案一起回复给用户。如下图所示。

![20260320212910](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320212910.png)

Agent 把一道算式拆成两次 MCP 工具调用、逐步求解的全过程。

如果希望通过 `http` 或 `sse` 对外暴露 MCP，可以用 `SSEServerTransport` 配合 `express` 等框架实现网络服务器。官方文档中提供了天气服务器完整示例，具体代码可以参考 `LangChain` MCP 文档。

## 传输方式与部署选择

MCP 把传输层从协议里抽象出来，常见有两大类选择，适合不同场景：

| 传输方式 | 通信方式 | 适合场景 | 特点 |
| --- | --- | --- | --- |
| `stdio` | 子进程标准输入输出 | 本地脚本、命令行工具 | 部署简单，权限可控，无需网络 |
| `http`、`sse` | HTTP 接口或 SSE 流 | 远程服务、云端 API | 支持水平扩展，语言无关，方便统一网关管理 |

在 `LangChain` 的 `MultiServerMCPClient` 里，只需要在配置里写清 `transport`、`url` 或 `command`、`args`，客户端会按协议和服务器对话。下面的例子同时接入了一个 `sse` 远程天气服务器和一个 `stdio` 本地数学服务器：

```typescript
import { MultiServerMCPClient } from "@langchain/mcp-adapters";

const client = new MultiServerMCPClient({
  weather: {
    transport: "sse",
    url: "http://localhost:8000/mcp",
  },
  math: {
    transport: "stdio",
    command: "node",
    args: ["/path/to/math_server.js"],
  },
});
```

具体每种传输方式支持的能力和限制，可以参考官方的 MCP Transport 文档，根据安全、部署和性能等需求选择合适的方案。

## 和前面章节的关系

从 `LangChain` 的角度看，MCP 工具一旦通过 `client.getTools()` 加载进来，就和本地用 `tool` 定义的工具没有本质区别，都可以被 `createAgent` 使用，也都能经过第 10 章介绍的中间件控制（例如 `toolCallLimitMiddleware`、`humanInTheLoopMiddleware`），同样会出现在上下文工程和安全护栏的视野里。不同之处只是，工具的实现和托管位置从应用进程内抽象到了一个或多个遵守 MCP 协议的外部服务上。

结合第 12 章的 Runtime 和第 13 章的上下文工程，可以做出一些常见的组合用法：

- 把用户 ID、环境配置和租户信息通过 runtime context 传给 MCP 工具，让服务器端按用户做权限控制或路由，也可以在服务器侧做审计与计费。
- 用中间件在调用 MCP 工具前后做日志、限流、重试或安全检查，就像对待本地工具一样，把跨系统调用也纳入统一的治理体系。
- 通过 MCP 统一接入多个异构系统（本地脚本、内部服务、第三方 API），在 Agent 里对外只暴露一套统一的工具列表，让模型在一个抽象层面做工具选择。

## 小结

MCP 重点解决的是工具复用和生态统一的问题。在 `LangChain` 里，用 `@langchain/mcp-adapters` 提供的 `MultiServerMCPClient` 可以同时连接多个 MCP 服务器，把它们提供的工具一并挂到 Agent 上。服务器侧则可以用 `@modelcontextprotocol/sdk` 快速定义和发布工具，通过 `stdio`、`http`、`sse` 等传输方式对外服务。

配合前几章介绍的中间件、Runtime 和上下文工程，可以在不改动业务系统内部实现的前提下，把现有能力通过 MCP 暴露给 Agent，让 Agent 在一套统一的安全与治理框架下连接到更多外部系统。
