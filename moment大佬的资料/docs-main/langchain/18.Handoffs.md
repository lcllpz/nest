Handoffs 把多轮对话建模成一张状态机，每个阶段有专属的提示词和工具。当一个阶段完成工作，通过接力工具把控制权和状态传给下一个阶段，用户始终感知为在和同一个客服对话。和 `Subagents` 的"主管调度专家"不同，Handoffs 强调的是系统当前在哪一步、什么条件下往哪里走，两者可以嵌套使用。下图展示了后面代码示例里三阶段客服流程的状态跳转：

## 什么时候用 Handoffs

适合的场景：

- 对话流程有明确阶段顺序，后面的阶段依赖前面信息的收集结果
- 不同阶段都需要直接和用户对话，而不是纯后台任务
- 需要在代码里显式追踪当前阶段，方便调试和监控

不适合的场景：

- 流程简单，没有明确阶段划分
- 各阶段需要并行推进而不是串行接力
- 只需要主 Agent 调度多个专家，那用 `Subagents` 更合适

两者是互补关系，Handoffs 管"阶段顺序和跳转"，`Subagents` 管"主管向下分发任务"，一个 `Supervisor` 内部完全可以用 Handoffs 在不同子阶段之间移动。

## 状态定义

Handoffs 的状态对象通常包含两类字段：阶段标记（如 `currentStep`）和业务数据（如 `warrantyStatus`、`issueType`）。每次新消息进来，系统根据状态决定用哪套提示词、开放哪些工具、处理完后是否接力给下一阶段。

用枚举约束阶段名，TypeScript 会在编译时帮你发现拼写错误：

```typescript
import { z } from "zod";
import { StateSchema } from "@langchain/langgraph";

const SupportStepSchema = z.enum([
  "warranty_collector",
  "issue_classifier",
  "resolution_specialist",
]);

const SupportState = new StateSchema({
  currentStep: SupportStepSchema.optional() as any,
  warrantyStatus: z.enum(["in_warranty", "out_of_warranty"]).optional() as any,
  issueType: z.enum(["hardware", "software"]).optional() as any,
} as any);
```

## 用 Command 完成接力

接力工具做两件事：更新状态里的 `currentStep`，同时返回 `Command` 告诉图执行器下一轮跳到哪个阶段。

必须同时写一条 `ToolMessage` 进消息历史，`tool_call_id` 要对应触发本次调用的 `AIMessage`。如果只更新状态不写 `ToolMessage`，模型下次读历史会发现一条没有结果的工具调用，轻则卡壳，重则产生幻觉。`ToolMessage` 的内容还可以写一句状态摘要（比如"已记录质保状态，进入问题分类阶段"），让下一阶段的模型不用从头读整条对话：

```typescript
import { tool, ToolMessage, type ToolRuntime } from "langchain";
import { Command } from "@langchain/langgraph";

const recordWarrantyStatus = tool(
  async (input, config: ToolRuntime<typeof SupportState.State>) => {
    return new Command({
      update: {
        messages: [
          new ToolMessage({
            content: `已记录质保状态：${input.status}，进入问题分类阶段。`,
            tool_call_id: config.toolCallId,
          }),
        ],
        warrantyStatus: input.status,
        currentStep: "issue_classifier",
      },
    });
  },
  {
    name: "record_warranty_status",
    description: "记录用户的质保状态，并跳转到问题类型识别阶段。",
    schema: z.object({
      status: z.enum(["in_warranty", "out_of_warranty"]),
    }),
  },
);

const recordIssueType = tool(
  async (input, config: ToolRuntime<typeof SupportState.State>) => {
    return new Command({
      update: {
        messages: [
          new ToolMessage({
            content: `已记录问题类型：${input.issueType}，进入解决方案阶段。`,
            tool_call_id: config.toolCallId,
          }),
        ],
        issueType: input.issueType,
        currentStep: "resolution_specialist",
      },
    });
  },
  {
    name: "record_issue_type",
    description: "记录问题类型（硬件或软件），并跳转到解决方案阶段。",
    schema: z.object({ issueType: z.enum(["hardware", "software"]) }),
  },
);
```

## 单 Agent 加中间件

只有一个 Agent 负责和用户对话，外面包一层 `middleware`，每次调用模型前动态注入当前阶段对应的提示词和工具。上下文天然连续，整条消息历史都在同一个 Agent 里，不需要在多个 Agent 之间传递消息。

### 定义阶段配置表

为每个阶段写好提示词和允许使用的工具，集中放进一张配置表。提示词里用 `{warrantyStatus}` 这类占位符，中间件注入时自动替换为状态里的实际值：

```typescript
const WARRANTY_COLLECTOR_PROMPT = `你是一名负责处理设备问题的客服助手。
当前阶段：保修状态确认。
先用简短友好的语气和用户打招呼，再用自然的对话方式确认"设备是否还在保修期内"，确认后使用 record_warranty_status 工具记录结果并进入下一阶段。`;

const ISSUE_CLASSIFIER_PROMPT = `你是一名负责处理设备问题的客服助手。
当前阶段：问题类型识别。用户质保状态：{warrantyStatus}。
引导用户描述遇到的问题，判断是硬件问题（摔坏、屏幕碎裂、按键失灵等）还是软件问题（应用闪退、系统卡顿等），无法判断时先追问澄清，确认后使用 record_issue_type 工具记录并进入下一阶段。`;

const RESOLUTION_SPECIALIST_PROMPT = `你是一名负责处理设备问题的客服助手。
当前阶段：给出解决方案。质保状态：{warrantyStatus}，问题类型：{issueType}。
软件问题：使用 provide_solution 给出分步骤的排查和解决方案。
硬件问题且在保修期：使用 provide_solution 说明保修维修流程。
硬件问题且已过保：使用 escalate_to_human 升级给人工售后。`;

type StepConfigEntry = {
  prompt: string;
  tools: ReturnType<typeof tool>[];
  requires: string[];
};

const STEP_CONFIG: Record<string, StepConfigEntry> = {
  warranty_collector: {
    prompt: WARRANTY_COLLECTOR_PROMPT,
    tools: [recordWarrantyStatus],
    requires: [],
  },
  issue_classifier: {
    prompt: ISSUE_CLASSIFIER_PROMPT,
    tools: [recordIssueType],
    requires: ["warrantyStatus"],
  },
  resolution_specialist: {
    prompt: RESOLUTION_SPECIALIST_PROMPT,
    tools: [provideSolution, escalateToHuman],
    requires: ["warrantyStatus", "issueType"],
  },
};
```

### 写中间件

读取 `currentStep`，取出对应配置，把状态里的实际值填进提示词占位符。`requires` 字段做前置校验，必要字段还没被收集就直接报错，防止阶段跳转时序出问题：

```typescript
import { createMiddleware } from "langchain";

const applyStepMiddleware = createMiddleware({
  name: "applyStep",
  stateSchema: SupportState,
  wrapModelCall: async (request, handler) => {
    const currentStep =
      (request.state.currentStep as string | undefined) ?? "warranty_collector";
    const stepConfig = STEP_CONFIG[currentStep];

    for (const key of stepConfig.requires) {
      if (request.state[key as keyof typeof request.state] === undefined) {
        throw new Error(`${key} 尚未设置，无法进入 ${currentStep} 阶段`);
      }
    }

    let systemPrompt = stepConfig.prompt;
    for (const [key, value] of Object.entries(request.state)) {
      systemPrompt = systemPrompt.replace(`{${key}}`, String(value ?? ""));
    }

    return handler({
      ...request,
      systemPrompt,
      tools: [...stepConfig.tools],
    });
  },
});
```

### 创建 Agent

接入 `MemorySaver` 做持久化，每个 `thread_id` 对应一个用户会话，每轮对话后状态自动保存，下一轮中间件能读到上一轮更新的 `currentStep`：

```typescript
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { MemorySaver } from "@langchain/langgraph";
import { HumanMessage } from "langchain";
import { randomUUID } from "crypto";

const llm = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const allTools = [
  recordWarrantyStatus,
  recordIssueType,
  provideSolution,
  escalateToHuman,
];

const agent = createAgent({
  model: llm,
  tools: allTools,
  middleware: [applyStepMiddleware],
  checkpointer: new MemorySaver(),
});

const threadId = randomUUID();
const config = { configurable: { thread_id: threadId } };

// 第一轮：进入 warranty_collector 阶段
let result = await (agent as any).invoke(
  { messages: [new HumanMessage("你好，我的手机屏幕碎了")] },
  config,
);

// 第二轮：工具把 currentStep 自动更新为 issue_classifier
result = await (agent as any).invoke(
  { messages: [new HumanMessage("在保修期内")] },
  config,
);

// 第三轮：工具把 currentStep 自动更新为 resolution_specialist
result = await (agent as any).invoke(
  { messages: [new HumanMessage("屏幕是摔碎的")] },
  config,
);
```

三轮对话下来，用户感知到的始终是"同一个客服"，但背后已经自动切过了三套提示词和工具。

### 完整示例

把上面三步拼在一起就是可以直接运行的完整代码：

```typescript
import { createMiddleware, createAgent } from "langchain";
import { z } from "zod";
import { randomUUID } from "crypto";
import { tool, ToolMessage, type ToolRuntime, HumanMessage } from "langchain";
import { Command, MemorySaver, StateSchema } from "@langchain/langgraph";
import { ChatOpenAI } from "@langchain/openai";

// ── 状态定义 ──────────────────────────────────────────────────────────────────

const SupportStepSchema = z.enum([
  "warranty_collector",
  "issue_classifier",
  "resolution_specialist",
]);
const WarrantyStatusSchema = z.enum(["in_warranty", "out_of_warranty"]);
const IssueTypeSchema = z.enum(["hardware", "software"]);

type SupportStep = z.infer<typeof SupportStepSchema>;
type SupportStateKeys = "currentStep" | "warrantyStatus" | "issueType";

const SupportState = new StateSchema({
  currentStep: SupportStepSchema.optional() as any,
  warrantyStatus: WarrantyStatusSchema.optional() as any,
  issueType: IssueTypeSchema.optional() as any,
} as any);

// ── 接力工具 ──────────────────────────────────────────────────────────────────

const recordWarrantyStatus = tool(
  async (input, config: ToolRuntime<typeof SupportState.State>) => {
    return new Command({
      update: {
        messages: [
          new ToolMessage({
            content: `已记录质保状态：${input.status}，进入问题分类阶段。`,
            tool_call_id: config.toolCallId,
          }),
        ],
        warrantyStatus: input.status,
        currentStep: "issue_classifier",
      },
    });
  },
  {
    name: "record_warranty_status",
    description:
      "Record the customer's warranty status and transition to issue classification.",
    schema: z.object({ status: WarrantyStatusSchema }),
  },
);

const recordIssueType = tool(
  async (input, config: ToolRuntime<typeof SupportState.State>) => {
    return new Command({
      update: {
        messages: [
          new ToolMessage({
            content: `已记录问题类型：${input.issueType}，进入解决方案阶段。`,
            tool_call_id: config.toolCallId,
          }),
        ],
        issueType: input.issueType,
        currentStep: "resolution_specialist",
      },
    });
  },
  {
    name: "record_issue_type",
    description:
      "Record the type of issue and transition to resolution specialist.",
    schema: z.object({ issueType: IssueTypeSchema }),
  },
);

const escalateToHuman = tool(
  async (input) => {
    return `Escalating to human support. Reason: ${input.reason}`;
  },
  {
    name: "escalate_to_human",
    description: "Escalate the case to a human support specialist.",
    schema: z.object({ reason: z.string() }),
  },
);

const provideSolution = tool(
  async (input) => {
    return `Solution provided: ${input.solution}`;
  },
  {
    name: "provide_solution",
    description: "Provide a solution to the customer's issue.",
    schema: z.object({ solution: z.string() }),
  },
);

// ── 阶段配置表 ────────────────────────────────────────────────────────────────

const WARRANTY_COLLECTOR_PROMPT = `你是一名负责处理设备问题的中文客服助手。

当前阶段：保修状态确认。

在这个阶段，你需要：
1. 先用简短友好的语气和用户打招呼。
2. 用自然的对话方式确认"设备是否还在保修期内"。
3. 使用工具 record_warranty_status 把用户的保修状态记录到系统中，然后进入下一阶段。

和用户对话时一定要口语化、自然，不要一次问太多问题，也不要一下子抛出多个选择题。`;

const ISSUE_CLASSIFIER_PROMPT = `你是一名负责处理设备问题的中文客服助手。

当前阶段：问题类型识别。
用户信息：当前保修状态为 {warrantyStatus}。

在这个阶段，你需要：
1. 引导用户用自己的话描述当前遇到的问题。
2. 判断这个问题更像是硬件问题（例如摔坏、屏幕碎裂、按键失灵等），还是软件问题（例如应用闪退、系统卡顿、性能变差等）。
3. 使用工具 record_issue_type 把问题分类结果记录到系统中，然后进入下一阶段。

如果你暂时无法判断是硬件还是软件，一定要先追问一两句澄清问题，再做分类。`;

const RESOLUTION_SPECIALIST_PROMPT = `你是一名负责处理设备问题的中文客服助手。

当前阶段：给出解决方案。
用户信息：保修状态为 {warrantyStatus}，问题类型为 {issueType}。

在这个阶段，你需要：
1. 如果是软件问题（issueType = "software"）：使用工具 provide_solution，给出分步骤的排查与解决方案，例如重启、更新系统、重装应用、清理缓存等。
2. 如果是硬件问题（issueType = "hardware"）：
   - 且用户在保修期内（warrantyStatus = "in_warranty"）：使用 provide_solution 清晰说明保修维修流程，例如需要准备的材料、预约或寄修方式、可能的处理时间等。
   - 且用户已过保（warrantyStatus = "out_of_warranty"）：使用工具 escalate_to_human，把案例升级给人工售后，由人工提供付费维修方案。

无论是哪种情况，回答时都要具体、清晰、友好，避免只给出一句话的笼统建议。`;

  prompt: string;
  tools: any[];
  requires: SupportStateKeys[];
};

const STEP_CONFIG: Record<SupportStep, StepConfigEntry> = {
  warranty_collector: {
    prompt: WARRANTY_COLLECTOR_PROMPT,
    tools: [recordWarrantyStatus],
    requires: [],
  },
  issue_classifier: {
    prompt: ISSUE_CLASSIFIER_PROMPT,
    tools: [recordIssueType],
    requires: ["warrantyStatus"],
  },
  resolution_specialist: {
    prompt: RESOLUTION_SPECIALIST_PROMPT,
    tools: [provideSolution, escalateToHuman],
    requires: ["warrantyStatus", "issueType"],
  },
};

// ── 中间件 ────────────────────────────────────────────────────────────────────

const applyStepMiddleware = createMiddleware({
  name: "applyStep",
  stateSchema: SupportState,
  wrapModelCall: async (request, handler) => {
    const currentStep: SupportStep =
      (request.state.currentStep as SupportStep | undefined) ??
      "warranty_collector";
    const stepConfig = STEP_CONFIG[currentStep];

    for (const key of stepConfig.requires) {
      if (request.state[key] === undefined) {
        throw new Error(`${key} must be set before reaching ${currentStep}`);
      }
    }

    let systemPrompt: string = stepConfig.prompt;
    for (const [key, value] of Object.entries(request.state)) {
      systemPrompt = systemPrompt.replace(`{${key}}`, String(value ?? ""));
    }

    return handler({
      ...request,
      systemPrompt,
      tools: [...stepConfig.tools],

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },

  recordWarrantyStatus,
  recordIssueType,
  provideSolution,
  escalateToHuman,

const agent = createAgent({
  model,
  tools: allTools,

const config = { configurable: { thread_id: threadId } };

  config,
);
for (const msg of result.messages) console.log(msg.content);

// 第二轮：工具把 currentStep 自动更新为 issue_classifier
result = await (agent as any).invoke(
  { messages: [new HumanMessage("在保修期内")] },
  config,
);
for (const msg of result.messages) console.log(msg.content);
console.log(`当前阶段：${result.currentStep}`);

// 第三轮：工具把 currentStep 自动更新为 resolution_specialist
result = await (agent as any).invoke(
  { messages: [new HumanMessage("屏幕是摔碎的")] },
  config,
);
for (const msg of result.messages) console.log(msg.content);
console.log(`当前阶段：${result.currentStep}`);

// 第四轮：进入解决方案阶段，模型根据 warrantyStatus 和 issueType 给出处理方案
result = await (agent as any).invoke(
  { messages: [new HumanMessage("我应该怎么处理？")] },

);
for (const msg of result.messages) console.log(msg.content);
```

## 多 Agent 子图

把每个阶段做成独立的 Agent 节点放在同一张图里。状态里通常有 `messages`（当前子图的对话历史）和 `active_agent`（当前处理的 Agent 名）两个字段。当某个 Agent 决定转交时，接力工具找到触发本次调用的 `AIMessage`，构造 `ToolMessage` 摘要，然后返回 `Command` 并把 `goto` 设成目标节点名。

不必把所有历史消息传给下一个节点，只需那条 `AIMessage` 加上一条简短 `ToolMessage` 摘要就够下一阶段用了，还能节省大量 token。

这种方式更适合"每个阶段本身也很复杂"的情况，比如账单处理需要查数据库、计算差额、走审批，拆成独立子图会更清晰。如果只是换一套提示词和工具，单 Agent 加中间件更省事。

## 设计前先想清楚几件事

动手写代码前，先把状态机画出来：

- 这条链路会经历哪些阶段，每个阶段的结束条件是什么
- 每个阶段需要哪些状态字段，下一阶段的决策依赖哪些信息
- 哪些节点之间可以跳转，有没有需要禁止的回跳
- 哪些节点需要插入人工审批或补充信息

阶段数量控制在三到五个比较合适。超过这个数量，状态图本身就难以理解，这时候应该考虑把部分阶段收进子图，或者用 `Subagents` 分担职责。
