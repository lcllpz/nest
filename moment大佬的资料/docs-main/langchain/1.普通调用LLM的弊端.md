在接触 LangChain 之前，先搞清楚一个问题，为什么要用编排框架？直接对着 `API` 发请求不行吗？

能跑，但跑不远。裸调 `API` 足以应对一次性 demo 或脚本工具，但当你尝试把它落地成真正的对话应用时，问题会接踵而至：上下文要靠手动维护，换个模型要改一堆代码，提示词散落各处根本没法统一管理，更别提什么工具调用、流式输出、错误重试了。

本章以 DeepSeek 官方文档中的入门示例为起点，逐条拆解这六类痛点，帮你建立对"编排框架到底解决了什么"的直觉。

## DeepSeek 官方调用示例

DeepSeek 提供与 OpenAI 兼容的 `HTTP API`，官方推荐使用 `openai` SDK 调用。在 [DeepSeek 开放平台](https://platform.deepseek.com) 申请 `API Key` 后，确认 `package.json` 已设置 `"type": "module"`，安装依赖后按如下方式发起一次单轮对话：

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.DEEPSEEK_API_KEY,
  baseURL: "https://api.deepseek.com",
});

async function main(): Promise<void> {
  const response = await client.chat.completions.create({
    model: "deepseek-chat",
    messages: [
      { role: "system", content: "你是一个乐于助人的助手。" },
      { role: "user", content: "你好！" },
    ],
    stream: false,
  });
  console.log(response.choices[0].message.content);
}

main();
```

运行后控制台会输出模型回复，效果类似下图：

![20260209214801](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260209214801.png)

这段代码干净清晰，跑通 demo 没有任何问题。难的不是跑通，而是当功能需求多起来之后，这套写法根本撑不住。

## 六大弊端详解

### 1. 上下文全靠手动维护

模型本身无状态，每次调用只知道本轮传入的 `messages`。想要多轮对话，就必须在应用层自己维护一个 `messages` 数组，每次请求都把完整的历史塞进去。

这件事听起来不复杂，做起来却很烦：

- 对话轮次越多，`messages` 越长，`token` 消耗飙升，成本直线上升
- 超出模型上下文窗口（以 DeepSeek 为例是 64K）时，请求直接失败
- 到底保留多少历史？超限时怎么截断？要不要做摘要压缩？这些全要你自己决定

![20260309095921](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260309095921.png)

上图按轮次依次展示了 `messages` 从 2 条增长到溢出的全过程：每一轮对话都要把完整历史传给模型，成本随轮次线性累积，超出窗口限制时请求直接报错，开发者不得不自己设计截断或压缩策略。

五轮对话的完整示例如下（`client` 由前文初始化）：

```typescript
import type { ChatCompletionMessageParam } from "openai/resources/chat";

const messages: ChatCompletionMessageParam[] = [
  { role: "system", content: "你是一个乐于助人的助手。" },
  { role: "user", content: "我叫小明，今年 25 岁。" },
  { role: "assistant", content: "你好小明，很高兴认识你！" },
  { role: "user", content: "我住在北京。" },
  { role: "assistant", content: "北京是个很棒的城市。" },
  { role: "user", content: "我叫什么？今年多大？" },
];

const response = await client.chat.completions.create({
  model: "deepseek-chat",
  messages,
  stream: false,
});

console.log(response.choices[0].message.content);
```

一旦对话超过几十轮，这套手动维护的方式就会失控。截断策略、摘要压缩、滑动窗口，这些记忆管理的职责被完全推给了开发者。

### 2. 换模型就要改代码

从 DeepSeek 切到 OpenAI、Claude 或智谱时，需要改 `baseURL`、`model` 名，有时还要换整个 SDK，甚至重写返回格式的解析逻辑。不同服务商的差异汇总如下：

| 参数 | DeepSeek | OpenAI | Claude |
| --- | --- | --- | --- |
| `baseURL` | `https://api.deepseek.com` | `https://api.openai.com` | `https://api.anthropic.com` |
| `model` 名 | `deepseek-chat` | `gpt-4.5` | `claude-3-7-sonnet-latest` |
| SDK 包名 | `openai` | `openai` | `@anthropic-ai/sdk` |
| 返回内容字段 | `choices[0].message.content` | `choices[0].message.content` | `content[0].text` |

实际项目里，调用通常散落在多个文件里，换一次模型就要全局搜索替换。如果同时想支持"主力模型 + 备用模型"的切换逻辑，代码会迅速臃肿，且极易遗漏某处。

### 3. 提示只能硬编码

`system` 和 `user` 的内容直接写死在代码里，没有任何管理机制：

```typescript
import type { ChatCompletionMessageParam } from "openai/resources/chat";

const messages: ChatCompletionMessageParam[] = [
  { role: "system", content: "你是一个乐于助人的助手。" },
  { role: "user", content: "你好！" },
];
```

现实中一个稍微复杂点的应用，可能需要客服、翻译、代码助手等多个角色，每套角色对应不同的系统提示。这些内容全部写死在代码里，改一处就要重新部署，历史版本也无从追溯。

更大的问题是无法做 A、B 测试。要对比两套提示的效果，要么复制一份代码手动切换，要么加一堆条件分支。上线后发现哪个版本效果更好，往往也说不清楚。

### 4. 无法调用外部工具

`API` 只接受文本、返回文本，不能查天气、不能搜数据库、不能执行计算。想做 `Agent` 或 `RAG`，全靠自己封装 `HTTP` 调用和解析逻辑。

典型场景：

- 用户问"北京今天天气怎么样？"，模型只能靠训练数据猜测，无法调实时天气 `API`
- 用户问"我们公司上季度销售额多少？"，模型看不到数据库，无法查询
- 用户问"帮我算一下 12345 × 67890"，复杂计算依赖外部工具时，光靠 `prompt` 引导并不可靠

要把 LLM 和外部工具打通，需要自己设计并实现完整的 `Agent` 流程：

![20260309100127](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260309100127.png)

图中每一个环节都需要开发者手写：判断是否需要工具、选择哪个工具、构造调用参数、发起 `HTTP` 请求、解析返回结果、拼回 `messages`、再次调用 LLM。这恰好是 `Agent` 的核心逻辑，裸调 `API` 完全不管。

### 5. 错误处理、重试、超时都要自己写

网络抖动、限流、模型过载、`token` 超限等异常，官方示例里通常不涉及，生产环境却必须考虑。

常见问题包括：

- 网络超时，需设置 `timeout`，实现有边界的重试逻辑
- 限流（429 错误），需退避重试，通常要做指数退避
- 模型过载（503 错误），需排队或切换备用模型
- `token` 超限，需截断输入或提示用户缩短问题
- `API Key` 失效，需友好提示和 `fallback` 机制

要在生产环境安全地跑起来，大概需要写这样的包装逻辑：

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.DEEPSEEK_API_KEY,
  baseURL: "https://api.deepseek.com",
  timeout: 30_000,
  maxRetries: 0,
});

async function callWithRetry(prompt: string, maxRetries = 3): Promise<string> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      const response = await client.chat.completions.create({
        model: "deepseek-chat",
        messages: [{ role: "user", content: prompt }],
        stream: false,
      });
      return response.choices[0].message.content ?? "";
    } catch (err: unknown) {
      const isRateLimit =
        err instanceof Error && err.message.includes("429");

      if (isRateLimit && attempt < maxRetries - 1) {
        await new Promise((resolve) =>
          setTimeout(resolve, 2 ** attempt * 1000),
        );
        continue;
      }

      throw err;
    }
  }

  throw new Error("超出最大重试次数");
}
```

这还只是最基础的限流重试，没有覆盖超时、备用模型、熔断等场景。完整的生产级错误处理远比这复杂，而这些全是业务逻辑之外的基础设施工作。

### 6. 流式输出要自己处理

若设 `stream: true`，返回的是 `SSE`（Server-Sent Events）流，需自己遍历异步迭代器、拼接字符串、处理断开重连：

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.DEEPSEEK_API_KEY,
  baseURL: "https://api.deepseek.com",
});

const stream = await client.chat.completions.create({
  model: "deepseek-chat",
  messages: [{ role: "user", content: "写一首诗" }],
  stream: true,
});

for await (const chunk of stream) {
  const content = chunk.choices[0]?.delta?.content;
  if (content) {
    process.stdout.write(content);
  }
}
```

在 `Node.js` 脚本里打印还好说，一旦需要把流式结果推给前端，就涉及 `WebSocket` 或 `SSE` 通道的管理、前端的渲染更新、中途断流的重连策略，以及如何在流式输出结束后触发后续逻辑（比如保存对话记录）。这些在官方示例里通常一笔带过。

## 总结

把以上六个问题汇总来看，裸调 `API` 把所有编排职责都推给了开发者，每一条单独来看都不是无法解决的问题，但六条加在一起，加上日后维护和扩展的成本，很快就会让代码变成一个难以维护的大泥球。LangChain 把这些能力封装成可复用组件，让开发者把精力放在业务逻辑上，而不是重复造基础设施。
