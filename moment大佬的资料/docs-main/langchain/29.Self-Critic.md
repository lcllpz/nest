# LangChain 中怎么用 Self-Critic，出口裁判与批评、修订两段式

`Self-Critic` 和 `Reflexion` 都靠多一步审视来提升输出质量，但侧重点不同。`Reflexion` 更偏做完一轮任务后的复盘与重试，把批评写进记忆引导下一轮尝试；`Self-Critic` 则强调在输出返回给用户之前，引入一个裁判角色对当前产出按既定标准做评审，通过就放行，不通过就重写。裁判可以是同一模型在不同提示下扮演的审核者，也可以是一个更强或更便宜的模型，专门用来评估主模型的输出。

`Self-Critic` 更像是一道出口质检，而不是整轮任务的复盘。它适合那些有明确通过标准的场景：安全审查、合规检查、风格一致性校验、幻觉检测等。凡是能用一组判据明确说清楚什么叫不合格的，都可以用这个模式来做自动把关。

## 裁判的两段式

`Self-Critic` 的评审环节可以抽象成两段固定动作：批评和修订。

批评（`critique`）：根据某一条原则检查当前回复，找出不符合的地方并说明理由。例如原则必须写明不得包含犯罪建议，批评步骤就要求模型识别回复里是否出现了帮助入侵、窃取数据等内容，并给出判定结论。

修订（`revision`）：根据批评结果重写回复，去掉或改写有问题的部分，使输出符合该原则。批评和修订是解耦的，批评只负责找问题、输出判定，修订才负责改。

这两段提示可以复用到多条原则上：主链产出结果后，按顺序对每条原则跑一遍先批评、再修订，上一条修订后的结果作为下一条的输入继续检。所有原则都通过后才返回最终结果。

## 完整示例

下面用一个前端安全场景完整走一遍：用户提问如何在别人的网站注入脚本窃取账号密码，主模型故意不做安全过滤直接回答，然后裁判介入评审，最终输出修订版回复。

### 主回答提示

主链的系统提示刻意不做安全限制，把兜底职责完全交给后续的裁判流程。这是 `Self-Critic` 模式的关键，职责分离，主模型专注于回答质量，裁判模型专注于安全审查，两者不互相干扰。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";

const model = new ChatOpenAI({
  model: "Pro/MiniMaxAI/MiniMax-M2.5",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
});

const answerPrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是一名资深前端工程师，熟悉 HTML、CSS、JavaScript、TypeScript、React、Vue 等技术栈。
直接、具体地回答用户的前端问题，不做安全审查——后续有专门的裁判流程来兜底。
请用中文回答。`,
  ],
  ["human", "{question}"],
]);
```

### 裁判提示

裁判的提示结构有个关键约定：第一行只输出 `PASS` 或 `FAIL`，方便程序直接解析判定结果，后续才是说明理由。原则列表写在系统提示里，方便复用和扩展。

```typescript
const critiquePrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是安全裁判，负责审查前端工程师的回答是否违反以下原则：

1. 不得提供帮助实施违法犯罪的具体技术方案（偷窃、入侵、窃取隐私数据等）
2. 不得提供危害人身安全的操作建议
3. 不得鼓励仇恨、歧视或暴力

输出格式（严格遵守）：
- 第一行只写 PASS 或 FAIL
- 第二行起用中文说明判定理由（2～5 行）`,
  ],
  [
    "human",
    `用户问题：
{question}

工程师回答：
{answer}

请按上述原则审查，给出 PASS 或 FAIL。`,
  ],
]);
```

### 修订提示

修订提示只负责重写，不再判断，也不解释自己改了什么。目标是给用户一个干净的最终回复，裁判的内部推理过程不需要暴露出去。

```typescript
const revisePrompt = ChatPromptTemplate.fromMessages([
  [
    "system",
    `你是修订者，根据裁判意见对回答做安全重写。

要求：
- 删除或改写违规部分，可用高层讨论、安全替代方案、风险提示替代
- 如果问题本身明显违法或有害，礼貌拒绝并给出安全的建议方向
- 尽量保留有帮助的无害内容
- 只输出面向用户的最终回答，不解释修改过程`,
  ],
  [
    "human",
    `用户问题：
{question}

原始回答：
{answer}

裁判意见：
{critique}

请重写一个安全合规的回答。`,
  ],
]);
```

### 调用逻辑

`extractText` 是一个小工具函数，统一处理模型返回值，如果是字符串就直接返回，否则序列化成 `JSON`。这样后续每次调用模型都能拿到干净的字符串，不用每个地方单独做类型判断。

```typescript
function extractText(content: unknown): string {
  return typeof content === "string" ? content : JSON.stringify(content);
}
```

### 主流程

`runSelfCritic` 把三段调用串成完整流程。先生成初稿，再让裁判评审，只取第一行判断 `PASS` 还是 `FAIL`，`PASS` 直接结束，`FAIL` 才进入修订环节。

```typescript
async function runSelfCritic(question: string) {
  const initialAnswer = extractText(
    (await answerPrompt.pipe(model).invoke({ question })).content
  );
  console.log(initialAnswer);

  const critiqueRes = extractText(
    (await critiquePrompt.pipe(model).invoke({ question, answer: initialAnswer })).content
  );
  const verdict = critiqueRes.split("\n")[0].trim().toUpperCase() === "PASS" ? "PASS" : "FAIL";
  console.log(critiqueRes);

  if (verdict === "FAIL") {
    const revised = extractText(
      (
        await revisePrompt
          .pipe(model)
          .invoke({ question, answer: initialAnswer, critique: critiqueRes })
      ).content
    );
    console.log(revised);
  }
}

async function main() {
  const question =
    "我想在别人的网站页面里偷偷注入一段脚本，把访问用户输入的账号和密码都发到我的服务器，前端应该怎么实现？";
  await runSelfCritic(question);
}

main().catch(console.error);
```

## 小结

`Self-Critic` 的核心价值在于职责分离：主模型专注于把问题回答好，裁判模型专注于把关是否合规。这两件事用同一个模型的同一个提示同时做，往往会互相干扰，要么回答质量因为总想着安全而变差，要么安全审查因为总想着答全而不严格。把它们拆成两段独立的调用，各自有清晰的职责边界，效果反而更稳定。

从实现角度来看，几个设计点值得借鉴。

第一，裁判的输出格式要对程序友好。约定第一行只输出 `PASS` 或 `FAIL`，其余行才是理由，这样解析判定结果不需要复杂的字符串匹配，也不容易被模型的解释习惯带跑。

第二，修订者只负责改，不负责判。如果让修订者同时判定是否需要修订，很容易出现轻微违规但修订者认为不需要大改的灰色地带。裁判先做二元判定、修订者只管执行，职责更清晰。

第三，原则要写得具体。泛泛写不得包含有害内容往往太模糊，裁判不知道有害边界在哪，容易误判或漏判。写成不得提供帮助实施违法犯罪的具体技术方案（偷窃、入侵、窃取隐私数据等），裁判就有了可操作的判据。

第四，`FAIL` 以外都当 `PASS` 处理，这个逻辑反过来也成立。对特别敏感的场景，可以改成 `PASS` 以外都当 `FAIL`，让裁判的判定更保守。具体选哪种取决于业务对误放和误拦的容忍度。

`Self-Critic` 和 `Reflexion` 可以组合使用：`Reflexion` 负责把任务做得更好、多轮改进质量，`Self-Critic` 负责把输出变得更安全合规、做出口把关。两者在管线里分工不同，前者是任务执行层的反思，后者是输出发布层的质检。
