前面已经讲过中间件能在哪些节点插逻辑，也讲过在运行时如何把当次调用的 `context` 和 `store` 带进链路里。这些能力都指向同一件事，每次调用模型时，让模型拿到的是"对"的信息。

Agent 在真实场景里容易翻车，多半不是模型不够强，而是该给的没给，不该给的又塞了一堆。上下文工程也就是 `context engineering`，讲的是在对的时机、用对的格式，把对的信息和工具交给模型。想把 Agent 做得稳定，先把这一块理清楚。这一章会把"能控制什么、数据从哪来、怎么在中间件里用"串起来讲，并给出可以直接套用的写法。

## Agent 为何容易失手

Agent 出错要么是模型能力不够，要么是传给模型的上下文不对，包括系统提示、历史消息、工具列表、响应格式，实践中后者更常见。上下文工程做的事就是在每次调用前，把该用的指令、消息、工具和格式准备好，且足够不冗余。`LangChain` 的中间件、Runtime、`state`、`store` 就是为这类控制设计的。概念关系见下图。

上图展示了上下文工程与 `LangChain` 各层能力之间的概念关系，`state`、`store`、runtime context 是数据来源，中间件是介入点，模型调用是最终目标。

## 你能控制什么、数据从哪来

Agent 循环就是两步往复：先调模型（带上 prompt 和工具，得到回复或 `tool_calls`），再执行工具（把结果塞回消息并再次调模型），直到模型不再要工具。要控制的不只是这两步内部，还有步骤之间发生的事，可以分成三块上下文：

- 模型上下文：每次请求里的系统提示、消息列表、工具集合、模型选择、响应格式。
- 工具上下文：工具执行时读到的用户信息、配置与记忆，以及工具写回的结果。
- 生命周期上下文：在调模型前后执行的摘要、护栏、日志等持久化逻辑。

![20260316115924](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316115924.png)

上图区分了三块上下文的职责边界，以及"只对当次生效（transient）"和"写回持久化（persistent）"的差别。

这三块用到的数据来自下面几个数据源：

- `runtime context`：当次调用的静态配置，会话级，例如用户 ID、API Key、数据库连接、权限。
- `state`：短期记忆，会话级，例如当前消息列表、上传文件、登录状态、工具返回结果。
- `store`：长期记忆，跨会话，例如用户偏好、历史抽取的洞察。

执行顺序可以概括为三步循环。每次请求进来后，先做第一步"调模型"，从 `state`、`store`、`runtime` 取数拼出当次的系统提示、消息、工具和模型，再调用模型。若模型返回 `tool_calls`，则进入第二步"执行工具"，工具可读写 `state` 与 `store`，并把结果塞回消息。随后跑第三步"步骤间逻辑"，做摘要或护栏，改完 `state` 后再回到第一步，如此往复，直到模型返回最终回复。

![20260316120256](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316120256.png)

上图按时间顺序画出三步循环与数据源的关系：第一步消费数据并拼出上下文，第二步工具读写数据，第三步持久化改 `state`，然后回到第一步。

实现上，这些控制几乎都通过中间件挂进去。用 `wrapModelCall` 可以改当次发给模型的内容（不改 `state`，属于 transient）。用 `beforeModel`、`afterModel` 等生命周期钩子则可以改 `state` 或做摘要、护栏，属于 life-cycle context 的持久部分。

## 模型上下文的四个控制点

模型上下文决定这一次模型看到什么。系统提示、消息列表、可用工具、用哪款模型以及最终回复的 `schema`，都可以根据 `state`、`store` 或 `runtime context` 动态决定。

### 动态系统提示

系统提示是这一轮模型的角色设定和行为边界，决定它扮演谁、该做什么、不该做什么。不同用户、不同会话阶段、不同部署环境，往往要给模型不同的身份说明与安全约束。

用 `dynamicSystemPromptMiddleware` 可以在调模型前，根据 `state`、`store` 与 `runtime` context 动态生成系统提示。下面这段代码根据 `runtime` 里的 `userRole` 与 `deploymentEnv` 自动拼出合适的指令。

```typescript
import * as z from "zod";
import { createAgent, dynamicSystemPromptMiddleware } from "langchain";
import { ChatOpenAI } from "@langchain/openai";

const contextSchema = z.object({
  userRole: z.enum(["admin", "editor", "viewer"]),
  deploymentEnv: z.enum(["production", "staging", "development"]),
});

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-your-deepseek-api-key",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const agent = createAgent({
  model,
  tools: [],
  contextSchema,
  middleware: [
    dynamicSystemPromptMiddleware((state, runtime) => {
      const { userRole, deploymentEnv } = runtime.context;
      let base = "你是贴心助手。";
      if (userRole === "admin") {
        base += "你拥有管理员权限，可执行全部操作。";
      } else if (userRole === "viewer") {
        base += "你仅有只读权限，请引导用户只做查询。";
      }
      if (deploymentEnv === "production") {
        base += "涉及数据修改时请格外谨慎。";
      }
      return base;
    }),
  ],
});
```

这样 Agent 在不同角色和环境下会自动带上对应的能力说明，不必在每次调用里手写固定 `prompt`。

### 消息注入

发给模型的消息列表不只有"用户对话历史"，还可以按需补充结构化的上下文信息。常见做法是从 `state` 里取本会话上传的文件或中间结果，从 `store` 里取用户偏好的语气或领域，从 `runtime` 里取当前生效的合规、风控要求，在 `wrapModelCall` 里把这些拼成额外说明，用 `request.override({ messages })` 塞进这一次请求，只对当前调用生效（transient，不写回 `state`）。

大多数对话模型更关注靠近末尾的消息，把补充说明放在最后一条消息里比塞进系统提示更容易生效。下面这段代码根据 `runtime` 里配置的合规框架，在每次请求末尾追加一条合规约束消息。

```typescript
import * as z from "zod";
import { createMiddleware } from "langchain";
import type { BaseMessage } from "@langchain/core/messages";
import { HumanMessage } from "@langchain/core/messages";

const contextSchema = z.object({
  userJurisdiction: z.string(),
  complianceFrameworks: z.array(z.string()),
});

const injectComplianceRules = createMiddleware({
  name: "InjectComplianceRules",
  contextSchema,
  wrapModelCall: (request, handler) => {
    const { userJurisdiction, complianceFrameworks } = request.runtime.context;
    const rules: string[] = [];
    if (complianceFrameworks?.includes("GDPR")) {
      rules.push("处理个人数据前须获得明确同意，用户有权要求删除数据。");
    }
    if (rules.length === 0) return handler(request);
    const complianceContext = `合规要求（${userJurisdiction}）：\n${rules.join("\n")}`;
    const messages: BaseMessage[] = [
      ...request.messages,
      new HumanMessage(complianceContext),
    ];
    return handler(request.override({ messages }));
  },
});
```

接入该中间件后，带 `userJurisdiction` 与 `complianceFrameworks` 的请求会在消息列表末尾自动多出一条合规说明，模型会据此约束回复内容。

### 工具过滤

工具太多会干扰模型选择、放大误用风险，太少又限制可完成的能力。更稳妥的做法是在中间件里先按当次场景筛一遍可见工具，再交给模型。筛的时候可以看 `state`（是否已登录、当前对话轮数）、`store`（用户已开启的功能）、`runtime`（角色权限、租户信息），最后用 `handler({ ...request, tools: filteredTools })` 把过滤后的列表传下去，让模型只在一小撮相关工具里做选择。

下面这段代码按 `runtime` 里的 `userRole` 做了一层权限划分：管理员保留全部工具，编辑角色去掉 `delete_data`，其他角色只保留以 `read_` 开头的只读工具。

```typescript
import * as z from "zod";
import { createMiddleware } from "langchain";

const contextBasedTools = createMiddleware({
  name: "ContextBasedTools",
  contextSchema: z.object({
    userRole: z.enum(["admin", "editor", "viewer"]),
  }),
  wrapModelCall: (request, handler) => {
    const { userRole } = request.runtime.context;
    let filteredTools = request.tools;
    if (userRole === "editor") {
      filteredTools = request.tools.filter((t) => t.name !== "delete_data");
    } else if (userRole !== "admin") {
      filteredTools = request.tools.filter((t) =>
        t.name.startsWith("read_")
      );
    }
    return handler({ ...request, tools: filteredTools });
  },
});
```

接入后，同一批工具在不同 `userRole` 下会变成不同的子集，模型拿到的始终是当前角色允许调用的那部分。更复杂的应用可以把动态工具过滤和运行时注册结合，按需注册和组合工具，让 Agent 每一轮只看到当前真正允许使用的工具集合。

### 动态切换模型与响应格式

不同模型在推理能力、调用成本、可用上下文长度上各有侧重。可以在 `wrapModelCall` 里根据会话长度（`state`）、用户偏好（`store`）、预算档位或运行环境（`runtime`）动态决定这一轮用哪个模型，再调用 `handler({ ...request, model: chosenModel })`。

常见策略是短对话和低风险问题用轻量模型，长上下文或复杂决策时切换到能力更强的模型。中间件可以根据 `state.messages` 长度、工具调用次数或业务标签在运行时选出合适模型。最终回复的结构也可以按场景调整，把结构化输出的 `schema` 当成模板，为不同角色或会话阶段选简版或详版，用 `handler({ ...request, responseFormat: chosenSchema })` 控制。

## 工具上下文的读写方式

工具除了接收模型传来的参数，往往还要访问当前用户信息、配置与记忆。工具实现里可以通过 `runtime` 参数读取 `state`、`store` 与 runtime context，也可以写入 `store`，或通过 `Command` 更新 `state`。

读取侧：

- 从 `runtime.state` 判断当前用户是否已登录。
- 从 `runtime.store.get(["preferences"], userId)` 取用户偏好。
- 从 `runtime.context` 取 API Key 和数据库连接等配置。

写入侧有两种方式：

- 若要把结果持久化，写入 `store`（例如 `runtime.store.put(...)`）。
- 若要更新当次会话的 `state`，返回 `new Command({ update: { authenticated: true } })`，后续轮次和中间件都能看到更新后的值。

需要区分的一点是，在 `wrapModelCall` 里改 `request.messages`、`request.tools`、`request.model` 等，默认只影响当次模型调用，不写回 `state`，属于 transient；在 `beforeModel`、`afterModel` 里改 `state`，或在工具里用 `Command`、`store` 写数据，则会持久化。

## 生命周期里的摘要与护栏

在"调模型"和"执行工具"之间，可以用中间件做摘要、护栏、日志等，这类逻辑会真正改 `state` 或 `store`，属于持久化的上下文工程。

对话太长时自动压缩历史是最常见的需求。`LangChain` 提供内置的 `summarizationMiddleware`，当消息超过一定 token 数时，用一个小模型把较早的消息摘要成一条，替换掉原文并写回 `state`，后续轮次看到的就是"摘要 + 最近若干条完整消息"，既省上下文又保留关键信息。

```typescript
import { createAgent, summarizationMiddleware } from "langchain";

const agent = createAgent({
  model: "deepseek-chat",
  tools: [],
  middleware: [
    summarizationMiddleware({
      model: "deepseek-chat",
      trigger: { tokens: 4000 },
      keep: { messages: 20 },
    }),
  ],
});
```

超过 `trigger.tokens` 时触发摘要，保留最近 `keep.messages` 条完整消息，其余压缩后永久写回 `state`。其他生命周期钩子（护栏、日志）的用法在第 10 章和第 11 章里已有说明，思路都是在"调模型前"或"调模型后"拿到 `state` 与 runtime，按需修改 `state`、打日志或做合规检查。

## 小结

Agent 不可靠往往是因为传给模型的上下文不对。上下文工程做的事是，在每次模型调用前，把合适的系统提示、消息、工具、模型和响应格式准备好，这些信息主要来自 `state`、`store` 和 runtime context，工具实现里按需读取这三类数据并写回，在工具执行前后用中间件做摘要、护栏、日志等持久化逻辑。

实践上，先用固定的系统提示和工具集把链路跑通，再逐步引入动态策略。每次只加一类上下文工程，方便观察效果和排查问题。动态拼接 prompt、切换模型、做摘要都会增加调用次数或 token，需要关注成本与延迟，并通过 `modelCallLimitMiddleware` 设置上限。

几条值得记住的原则：

- 尽量用好内置中间件，例如 `summarizationMiddleware`、`LLMToolSelectorMiddleware`，减少自写胶水代码。
- 分清 transient 和 persistent，仅当次生效的逻辑放在 `wrapModelCall` 里改 request，长期要保留的改 `state` 或在工具里写入 `store`。
- 与环境相关的信息通过 Runtime 传入，在调用时用 `invoke` 的 `context` 带上用户、租户与环境变量，在工具和中间件里通过 `runtime.context` 读取，这样 Agent 更接近无状态，更容易测试和多实例部署。
