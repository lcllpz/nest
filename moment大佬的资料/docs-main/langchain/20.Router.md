  Router 关心的是该把当前这条请求丢给哪个 Agent。当系统里已经有一批专门干不同事情的 Agent 时，总要有一个地方负责分流，把用户的问题送到最合适的那几位手里。

  Router 本身不直接面向用户，也不关心各个 Agent 内部如何工作，它只做一件事：根据当前请求，决定路由给哪些 Agent，并把原始问题改写成针对每个 Agent 的子查询。下面的代码示例用一个前端知识库问答场景走完整个流程，三个 Agent 分别负责 React 组件设计、CSS 视觉效果、前端性能优化，Router 决定需要查哪几个来源，再把结果流式综合成一条回复。如下图所示。

  ![20260320160528](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320160528.png)

  Router 分类后并行调用多个 Agent，最后由综合节点把各来源的结果整理成一条流式输出的回复。

  ## 什么时候用 Router

  Router 最适合这几类场景：有多条业务线或多个知识域，不同 Agent 负责完全不同的知识或工具集合，有些问题可能涉及多个领域，需要并行问多个 Agent 再综合。

  不太适合的场景：系统只有一两个 Agent、大多数请求都落在同一个领域里，或者各 Agent 的职责边界不够清晰，路由结果难以判断。如下图所示。

  ![20260320160813](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320160813.png)

  Router 适合多来源并行查询再综合的场景，不适合职责边界模糊或系统还很小的情况。

  ## 单路由与多路由扇出

  Router 有两种分流模式。

  单路由根据分类结果只选一个 Agent，适合问题归属明确、只需要一个来源就能回答的场景。在 `LangGraph` 里，单路由通常返回一个 `Command`，图执行器据此跳到对应的 Agent 节点。

  多路由扇出把请求同时发给多个 Agent，用 `Promise.all` 并行执行，收到所有结果后再交给综合节点整合。这种模式适合问题可能跨越多个知识域的情况。在 `LangGraph` 里，多路由对应 `Send` 实例，把同一轮请求扇出到多个节点并行执行。

  ## 如何做路由分类

  Router 决定路由目标的方式主要有三种：规则驱动根据关键词或路径前缀做简单判断，速度快、成本低；模型驱动用分类模型根据语义判断应该进哪个 Agent，适合描述模糊或跨域的问题；混合方式先用规则过滤掉明显不匹配的 Agent，再用模型在剩下的选。

  模型驱动的实现上，DeepSeek 当前版本不支持 `response_format: json_schema`，调用 `withStructuredOutput` 会报 `400 This response_format type is unavailable now`。可行的做法是在提示词里明确要求模型返回 JSON 数组，再用正则提取加 `JSON.parse` 手动解析，加一个回退策略兜底。

  子查询改写也非常重要。同一个原始问题在不同领域的措辞和侧重点完全不同，统一把原始问题转发给所有 Agent 会让每个 Agent 的检索效果大打折扣，效果远不如针对每个领域单独改写子查询。

  ## 如何实现 Router

  实现分三部分：`classifyQuery` 负责路由分类和子查询改写，`runSourceAgent` 用 `Promise.all` 并行调用各专家 Agent，`synthesizeAnswer` 流式综合结果。整个 Router 最后封装成工具挂给对话型 Agent。

  几个实现细节值得注意：

  - 子 Agent 执行时通过 `BaseCallbackHandler` 监听 `handleToolStart` 打印工具调用进度，用户不会误以为程序卡住
  - 综合阶段用 `baseModel.stream()` 替代 `baseModel.invoke()`，让用户在等待时就能看到答案逐步出现
  - 对话型 Agent 用 `.stream()` 而非 `.streamEvents()`，避免工具内部 LLM 调用事件冒泡到外层 Agent 的 callback 上下文，导致 Agent 反复触发工具
  - 每次对话通过闭包创建独立的 `alreadySearched` 标志，工具第二次被调用时直接返回已完成，从根本上截断重复调用

  ```typescript
    }

      { source: "react", query },
      { source: "css", query },
      { source: "performance", query },
    ];

  async function runSourceAgent(
    source: Source,
    query: string,

    const agentMap = {
      react: reactAgent,
      css: cssAgent,
      performance: performanceAgent,

    const result = await agentMap[source].invoke(
      { messages: [new HumanMessage(query)] },
      { callbacks: [new AgentProgressHandler(source)] },

    return { source, result: String(result.messages.at(-1)?.content ?? "") };

  async function synthesizeAnswer(
    originalQuery: string,
    results: AgentOutput[],
  ): Promise<string> {
    if (results.length === 0) {
      const fallback = await baseModel.invoke([
        {
          role: "system",
          content: "你是一名前端工程师助手，请直接回答用户的问题。",
        },
        { role: "user", content: originalQuery },
      ]);
      return String(fallback.content ?? "");
    }

    const formatted = results.map(
      (r) => `来自 ${r.source.toUpperCase()} 专家的分析：\n${r.result}`,
    );

    const stream = await baseModel.stream([
      {
        role: "system",
        content: `综合以下各领域专家的分析，回答用户的原始问题：${originalQuery}
  要求：合并多个来源的信息去除重复，突出最重要最可落地的建议，指出来源之间的差异，保持回复简洁有条理，包含具体代码片段。`,
      },
      { role: "user", content: formatted.join("\n\n") },
    ]);

    let fullContent = "";
    for await (const chunk of stream) {
      const text = String(chunk.content ?? "");
      if (text) {
        process.stdout.write(text);
        fullContent += text;
      }
    }
    return fullContent;
  }

  // ── Router 主函数 ──────────────────────────────────────────────────────────────

  async function runRouter(query: string): Promise<{ finalAnswer: string }> {
    const classifications = await classifyQuery(query);
    const sources = classifications.map((c) => c.source).join(" + ");
    console.log(
      `  路由 → ${sources}（${classifications.length} 位专家并行启动）`,
    );

    const startTime = Date.now();
    const results = await Promise.all(
      classifications.map((item) => runSourceAgent(item.source, item.query)),
    );
    console.log(
      `\n  所有专家完成（${Date.now() - startTime}ms），开始流式综合...\n`,
    );

    const finalAnswer = await synthesizeAnswer(query, results);
    return { finalAnswer };
  }

  // 外层 Agent 的 callback 上下文，导致 Agent 误把内部输出当作自己的 reasoning。

  async function chat(
    question: string,
    config: { configurable: { thread_id: string } },

        await runRouter(query);
        return `搜索完成，解答已直接输出给用户。`;
      },
      {
        name: "search_frontend_knowledge",
        description:
          "搜索前端知识库，覆盖 React 组件设计、CSS 视觉效果、前端性能优化三个领域。",
        schema: z.object({ query: z.string().describe("用户的前端技术问题") }),
      },
    );

    const assistant = createReactAgent({
      llm: baseModel,
      tools: [singleShotTool],
      prompt:
        "你是一名前端工程师助手。遇到技术问题时调用 search_frontend_knowledge 工具，" +
        "工具会直接输出完整答案，你只需用一句话确认即可，不要重复工具内容。",
    });

    type AgentChunk = {
      agent?: { messages: Array<{ content: unknown; tool_calls?: unknown[] }> };
    };

    const agentStream = await assistant.stream(
      { messages: [{ role: "user", content: question }] },
      config,
    );

    for await (const chunk of agentStream as AsyncIterable<AgentChunk>) {
      if (chunk.agent) {
        for (const msg of chunk.agent.messages) {
          const text = typeof msg.content === "string" ? msg.content.trim() : "";
          const isToolCall =
            Array.isArray(msg.tool_calls) && msg.tool_calls.length > 0;
          if (text && !isToolCall) {
            process.stdout.write(`\n[整合] ${text}\n`);
          }
        }
      }
    }
    console.log("\n");
  }

  // ── 主流程 ────────────────────────────────────────────────────────────────────

  const config = { configurable: { thread_id: "frontend-session-001" } };

  await chat("怎么实现一个好看的骨架屏加载动画？", config);
  await chat("页面首屏加载超过 3 秒，怎么排查和优化？", config);
  await chat("上面提到的懒加载，在 React 里具体怎么写？", config);
  ```

  ## Router 在多 Agent 架构中的位置

  在多 Agent 系统里，Router 通常处在比较靠前的位置，可以在入口处用一个 Router 把请求分到不同业务线的子图，在某条业务线内部再用 Router 选择更细分的 Subagent，形成粗分流到精分流再到专业执行的层级结构。

  Router 和 Subagents 的区别不在于是否调度 Agent，而在于谁决定路由。Subagents 模式里路由逻辑在主 Agent 的系统提示词里。Router 模式里路由决策是独立的一个节点，职责更单一，也更容易单独测试和替换路由策略。

  ## 小结

  Router 的核心价值是把选谁来做这件事从业务逻辑里拆出来，变成独立的一层，让下游各 Agent 只需要关注自己擅长的事。

  实现上三步走：提示词驱动的 JSON 分类（同时改写子查询）、`Promise.all` 并行扇出、`baseModel.stream()` 流式综合。写完之后把整个 Router 封装成工具挂给对话型 Agent，对上层完全透明。子查询改写比直接转发原始问题效果好得多，综合节点改用流式输出能显著改善等待体验，用 `.stream()` 而非 `.streamEvents()` 包裹对话型 Agent 可以避免工具内部 LLM 事件污染外层 callback 上下文。
