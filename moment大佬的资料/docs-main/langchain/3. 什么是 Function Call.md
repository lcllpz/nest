前一章介绍了 `LangChain` 的核心能力，其中工具调用是构建 `Agent` 的基础。要让 LLM 真正调用外部函数（查天气、查数据库、发邮件等），需要一套约定：模型如何表达想调用哪个函数、传什么参数，应用又如何执行并把结果交回模型。这套约定就是 Function Call（函数调用）。

读完本章，你会弄清 Function Call 的工作原理、请求与响应中关键字段的含义，并亲手用 TypeScript 跑通一次完整链路。最后会说明它和 `LangChain` 工具绑定、`Agent` 循环的关系，帮你建立后续章节所需的底层直觉。

## 为什么需要 Function Call

普通对话接口就是用户发文本、模型回文本。这意味着 LLM 无法直接回答"北京今天天气怎么样"这类依赖实时数据的问题，因为它看不到互联网、数据库或任何外部系统。

想解决这个问题，有两种思路。

第一种：让模型在回复里描述想调用什么，比如输出"请调用 `get_weather(location='北京')`"，再由应用用正则去解析。这种方式极其脆弱，格式稍有变化就解析失败，参数一复杂（嵌套对象、数组）就更难稳定处理。

第二种：使用 Function Call。在请求里声明可用工具的列表（名称、描述、参数格式），模型在需要时不输出自然语言，而是返回一个结构化的 `tool_calls`（工具名 + 参数 JSON），应用执行对应函数并把结果塞回对话，模型再基于结果生成最终回复。

第二种方式由 `OpenAI` 在 `Chat Completions API` 中标准化，现已被 `DeepSeek`、`Claude`、智谱等主流厂商兼容，成为模型与工具协作的事实标准。`LangChain` 的 `Tool` 和 `Agent` 都建立在这套机制之上。

## Function Call 是什么

一个直观的类比：把 LLM 想成一个博学但与世隔绝的顾问，他只能靠你告诉他的内容做判断。Function Call 就是你给他一份"服务目录"，列清楚有哪些外部服务可以调用、每个服务能做什么、需要提供哪些信息。他看完目录和你的问题后，告诉你"我需要调用第 3 项服务，参数是北京"——但他自己不会拨电话，真正打电话（执行函数）是你的工作，你拿到结果再告诉他，他才能给出最终回答。

具体到技术实现，整个约定分三个部分：

- 请求时：在 `tools` 字段里声明工具列表，每个工具包含名称、用途描述、参数格式（`JSON Schema`）
- 模型响应：如果模型判断需要调用工具，不返回文本，而是返回 `tool_calls`（包含工具名和填好的参数 JSON）
- 回传结果：应用执行函数后，把结果以 `role: "tool"` 的消息追加进 `messages`，再发一次请求，模型才生成最终回复

模型本身不执行任何函数，只负责"决定调哪个、填什么参数"。执行一定发生在应用侧。

## 典型流程

以"北京今天天气怎么样"为例，把整条链路拆成六步：

1. 用户发出消息"北京今天天气怎么样？"
2. 应用组装请求：`messages` 里带上用户消息，同时附带 `tools` 列表，声明 `get_weather` 工具（描述是"根据城市名查询当前天气"，参数格式用 `JSON Schema` 描述，`location` 为必填字符串）
3. 模型不直接答"北京晴天 25 度"，而是返回 `tool_calls`，指明 `name: "get_weather"`、`arguments: { "location": "北京" }`，同时 `finish_reason` 为 `"tool_calls"`
4. 应用从 `tool_calls` 里解析参数，执行本地或远程的 `get_weather("北京")`，得到"北京：晴，25°C"
5. 应用把结果写回对话：追加一条 `role: "tool"` 的消息，`content` 填天气结果，并带上与本次调用对应的 `tool_call_id`
6. 应用用更新后的 `messages` 再请求一次，模型看到工具结果后，生成自然语言回复"北京今天晴天，气温 25°C"

记住一点，只要触发了工具调用，交互至少要两轮。无论模型一次返回多个 tool_calls，还是出现多轮 `“调用 → 回传 → 再调用”`，都按同样流程依次执行并追加消息，再次请求模型，直到不再返回 tool_calls、只输出普通文本为止。

## 关键字段解析

理解请求里发了什么、响应里收到什么，是读懂 Function Call 的核心。

### 请求侧的 tools 和 tool_choice

`tools` 字段声明本次对话可用的函数列表，每个工具包含以下信息：

| 字段 | 说明 |
| --- | --- |
| `type` | 固定为 `"function"` |
| `function.name` | 工具的唯一标识，模型在 `tool_calls` 里用这个名字 |
| `function.description` | 给模型看的用途说明，写得越清晰，模型选对工具的概率越高 |
| `function.parameters` | `JSON Schema` 格式的参数描述，模型据此填写 `arguments` |

`tool_choice` 控制模型是否调用工具：

- `"auto"`：模型自主决定要不要调用，这是最常用的设置
- `"none"`：禁止调用，只生成文字
- 指定工具名：强制模型调用某个特定工具，如 `{ type: "function", function: { name: "get_weather" } }`

### 响应侧的 tool_calls 和 finish_reason

当模型决定调用工具时，响应中 `choices[0].message` 的结构如下：

- `content`：通常为 `null`，因为本轮只做工具调用，不生成文字
- `tool_calls`：数组，每一项包含 `id`（唯一标识）、`type`（固定为 `"function"`）、`function.name`（工具名）、`function.arguments`（JSON 字符串，需应用自行 `JSON.parse`）
- `finish_reason`：为 `"tool_calls"` 时表示本轮因工具调用结束，需要应用回传结果再继续；为 `"stop"` 时表示模型已给出完整回复，流程结束

`tool_call_id` 是连接"模型请求"和"应用回传"的纽带。模型在 `tool_calls` 里生成一个 `id`，应用在追加 `role: "tool"` 消息时必须带上对应的 `tool_call_id`，模型才能知道这条结果对应哪次调用——在多工具并发的场景下这一点尤为关键。

## 用 TypeScript 跑通一次完整调用

接下来用 TypeScript 和 `openai` SDK（以 DeepSeek 为例）跑通一次完整的 Function Call：定义一个获取天气的模拟工具，看模型如何返回 `tool_calls`、应用如何执行并把结果回传、模型最终如何生成自然语言回复。

```typescript
import OpenAI from "openai";
import type {
  ChatCompletionTool,
  ChatCompletionMessageParam,
} from "openai/resources/chat";

const client = new OpenAI({
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  baseURL: "https://api.deepseek.com",
});

const tools: ChatCompletionTool[] = [
  {
    type: "function",
    function: {
      name: "get_weather",
      description: "根据城市名称查询当前天气",
      parameters: {
        type: "object",
        properties: {
          location: {
            type: "string",
            description: "城市名称，例如北京、上海",
          },
          unit: {
            type: "string",
            enum: ["c", "f"],
            description: "温度单位，c 为摄氏度，f 为华氏度",
          },
        },
        required: ["location"],
      },
    },
  },
];

interface WeatherArgs {
  location: string;
  unit?: "c" | "f";
}

function getWeather(args: WeatherArgs): string {
  return `${args.location}：晴，25°C`;
}

async function main(): Promise<void> {
  const messages: ChatCompletionMessageParam[] = [
    { role: "user", content: "北京今天天气怎么样？" },
  ];

  // 第一轮：发送用户消息 + 工具定义，强制调用 get_weather
  const firstResponse = await client.chat.completions.create({
    model: "deepseek-chat",
    messages,
    tools,
    tool_choice: { type: "function", function: { name: "get_weather" } },
  });

  const assistantMessage = firstResponse.choices[0].message;
  messages.push(assistantMessage);

  // 解析 tool_calls，执行本地函数
  const toolCall = assistantMessage.tool_calls![0];
  const args = JSON.parse(toolCall.function.arguments) as WeatherArgs;
  const result = getWeather(args);

  // 把执行结果以 role: "tool" 追加进对话
  messages.push({
    role: "tool",
    tool_call_id: toolCall.id,
    content: result,
  });

  // 第二轮：带上工具结果，让模型生成自然语言回复
  const secondResponse = await client.chat.completions.create({
    model: "deepseek-chat",
    messages,
    tools,
    tool_choice: "auto",
  });

  console.log(secondResponse.choices[0].message.content);
}

main();
```

运行后终端会打印模型根据工具结果生成的自然语言回复，例如"北京今天天气很好，是晴天，气温 25°C"。效果见下图：

![20260213091018](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260213091018.png)

代码里有几处值得注意：

- `assistantMessage` 需要完整推入 `messages`，不能只推 `tool_calls` 部分，否则模型在第二轮看不到自己的上一条消息，对话会乱
- `JSON.parse(toolCall.function.arguments)` 的结果要做类型断言（`as WeatherArgs`），生产环境中建议用 `zod` 做运行时校验，防止模型填错参数
- 实际项目中把 `getWeather` 换成真实的 `HTTP`、数据库或其他 SDK 调用即可

多轮工具调用的处理也很简单：每次收到响应后，检查 `message.tool_calls` 是否存在，若存在则依次执行、追加 `tool` 消息，再请求；重复这个循环，直到响应里不再有 `tool_calls` 为止。

## 与 LangChain 的关系

在 `LangChain` 里，你通常不会手写上面的 `tools` 数组和轮询 `tool_calls` 的循环，框架帮你把这些细节封装了起来：

- 用 `@langchain/core/tools` 定义工具（名称、描述、执行函数），框架会自动转成当前模型所需的 `tools` 格式
- 用带工具的 `Agent` 跑多轮：模型输出 `tool_calls` → `LangChain` 执行工具 → 把结果写回消息 → 再调模型，直到模型给出最终回复

理解了 Function Call 的底层机制，再看 `LangChain` 的 `bind_tools`、`Agent` 循环时会顺畅很多，因为它们本质上就是对这套流程的封装。

## 小结

Function Call 是一套 API 约定，核心分工如下：

- 模型：决定是否调用工具、调哪个、填什么参数，输出结构化的 `tool_calls`
- 应用：真正执行函数，把结果以 `role: "tool"` 消息回传给模型
- 只要发生工具调用，至少有两轮请求

实践中有几点需要注意：

- 工具的 `description` 要写清楚，参数的 `JSON Schema` 要标好类型和 `required`，这直接影响模型选对工具、填对参数的概率
- 多轮调用时，当轮的 `assistantMessage`（含 `tool_calls`）和对应的 `role: "tool"` 消息必须按顺序完整保留在 `messages` 里，不能打乱或漏掉
- 工具执行失败时，把错误信息填进 `role: "tool"` 的 `content` 回传，让模型在回复里说明情况或建议重试，不要直接抛出异常让流程中断
- 一次返回多个 `tool_calls` 时，每个结果单独一条 `tool` 消息并带上对应的 `tool_call_id`，不能合并成一条

理解这套机制后，在 `LangChain` 里用工具与 `Agent` 会更顺畅。下一章将在此基础上跑通第一个带工具的 `Agent` 应用。
