前面几章一直在围绕"一个 Agent 搭配一堆工具"来讲，包括安全护栏、MCP 外部工具、人机协同审批等。在很多中小场景里，这已经够用，但当业务越来越复杂时，你会发现一个 Agent 扛不住所有职责，它既要当客服、又要写代码、还要查账，还得记所有历史状态，这时就需要多 Agent 协作。

多 Agent 的核心思路是"把职责拆开、把路径画清楚"。不同 Agent 分别负责不同的子任务，彼此之间通过明确的协议传递状态，而不是把一切压在同一个 prompt 和工具列表里。

## 多 Agent 适合解决什么问题

可以先把单 Agent 理解成"万能选手"，多 Agent 理解成"分工协作的团队"。如果任务本身非常单一、工具不多、逻辑简单，就没必要引入，以免带来额外复杂度。

## 几种常见的多 Agent 架构

`LangChain` 和 `LangGraph` 在多 Agent 这一块给出了几种常用模式，可以看成是你搭系统时的几个"积木"。如下图所示。

![20260320094213](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320094213.png)

### Supervisor 与 Subagents

最常见的模式。一个 Supervisor Agent 负责和用户对话、理解意图和规划步骤，再把具体子任务分配给各个 Subagent。Subagent 执行完后，把结果返回给 Supervisor，由 Supervisor 汇总后回复用户。

Subagent 被封装成工具，Supervisor 通过工具调用机制触发它们。

```typescript
import { z } from "zod";
import { tool } from "@langchain/core/tools";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "Pro/zai-org/GLM-5",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
  temperature: 0,
});

// Subagent：代码编写
const codeAgent = createReactAgent({
  llm: model,
  tools: [], // 实际替换为 readFileTool, writeFileTool, runTestTool
  prompt: "你是代码编写专家，专注于生成高质量 TypeScript 代码。",
});

// Subagent：代码审查
const reviewAgent = createReactAgent({
  llm: model,
  tools: [], // 实际替换为 readFileTool, searchDocsTool
  prompt: "你是代码审查专家，专注于发现潜在问题和安全漏洞。",
});

// 把 Subagent 封装成工具供 Supervisor 调用
const codeSubagentTool = tool(
  async ({ task }: { task: string }) => {
    const result = await codeAgent.invoke({ messages: [new HumanMessage(task)] });
    return result.messages.at(-1)?.content ?? "";
  },
  {
    name: "code_agent",
    description: "负责编写代码的专家 Agent，传入具体的编码任务描述",
    schema: z.object({ task: z.string().describe("编码任务描述") }),
  },
);

const reviewSubagentTool = tool(
  async ({ task }: { task: string }) => {
    const result = await reviewAgent.invoke({ messages: [new HumanMessage(task)] });
    return result.messages.at(-1)?.content ?? "";
  },
  {
    name: "review_agent",
    description: "负责代码审查的专家 Agent，传入需要审查的代码或任务描述",
    schema: z.object({ task: z.string().describe("审查任务描述") }),
  },
);

// Supervisor：调度两个 Subagent
const supervisor = createReactAgent({
  llm: model,
  tools: [codeSubagentTool, reviewSubagentTool],
  prompt: "你是开发团队主管，理解用户需求并协调代码编写和代码审查两个专家完成任务。",
});

const result = await supervisor.invoke({
  messages: [new HumanMessage("写一个 TypeScript 函数计算斐波那契数列，并审查代码质量。")],
});
console.log(result.messages.at(-1)?.content);
```

Supervisor 模式的好处是，主管可以根据任务情况决定先调哪个 Subagent、要不要重试，整个过程对用户透明。缺点是 Subagent 的调用次数和 token 消耗都会增加，复杂任务下成本需要关注。

### Handoffs 接力

Handoffs 模式适合有明确执行顺序的流程，每个 Agent 负责一段，干完后把状态交给下一个。和 Supervisor 的区别在于，没有一个持续在线的"主管"，而是每一棒结束后直接"传棒"。

实现接力最直接的方式是顺序调用：每个 Agent 完成后把输出作为输入传给下一个。

```typescript
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "Pro/zai-org/GLM-5",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
  temperature: 0,
});

const analysisAgent = createReactAgent({
  llm: model,
  tools: [], // 实际替换为 fetchDataTool 等
  prompt: "你是数据分析专家，拉取数据并输出分析结论。",
});

const reportAgent = createReactAgent({
  llm: model,
  tools: [], // 实际替换为 generateChartTool, exportPdfTool 等
  prompt: "你是报告生成专家，把分析结论制作成图文报告。",
});

const reviewAgent2 = createReactAgent({
  llm: model,
  tools: [], // 实际替换为 sendEmailTool 等
  prompt: "你是最终审核专家，检查报告无误后发送给相关人员。",
});

// 接力执行：分析 → 报告 → 审核
async function runPipeline(userRequest: string): Promise<string> {
  const step1 = await analysisAgent.invoke({ messages: [new HumanMessage(userRequest)] });
  const analysis = step1.messages.at(-1)?.content ?? "";

  const step2 = await reportAgent.invoke({
    messages: [new HumanMessage(`基于以下分析结果生成报告：\n${analysis}`)],
  });
  const report = step2.messages.at(-1)?.content ?? "";

  const step3 = await reviewAgent2.invoke({
    messages: [new HumanMessage(`请审核并发送以下报告：\n${report}`)],
  });
  return step3.messages.at(-1)?.content ?? "";
}

console.log(await runPipeline("分析上个月的用户留存数据"));
```

接力模式天然契合有明确阶段划分的业务流程，每个 Agent 的职责边界清晰，出了问题也很容易定位到哪一棒。

### Router 路由

Router 模式适合统一入口服务多个业务线的场景。一个专门的路由 Agent（或者一段分类逻辑）先对请求做意图识别，再把请求分发给对应的 Agent。

```typescript
import { z } from "zod";
import { tool } from "@langchain/core/tools";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "Pro/zai-org/GLM-5",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
  temperature: 0,
});

const billingAgent = createReactAgent({
  llm: model,
  tools: [], // 实际替换为 queryBillingTool, refundTool 等
  prompt: "你是账单客服专家，处理账单查询、退款申请等问题。",
});

const logisticsAgent = createReactAgent({
  llm: model,
  tools: [], // 实际替换为 trackPackageTool, updateAddressTool 等
  prompt: "你是物流客服专家，处理包裹追踪、地址修改等问题。",
});

const billingTool = tool(
  async ({ question }: { question: string }) => {
    const result = await billingAgent.invoke({ messages: [new HumanMessage(question)] });
    return result.messages.at(-1)?.content ?? "";
  },
  { name: "billing_agent", description: "处理账单、支付、退款相关问题", schema: z.object({ question: z.string() }) },
);

const logisticsTool = tool(
  async ({ question }: { question: string }) => {
    const result = await logisticsAgent.invoke({ messages: [new HumanMessage(question)] });
    return result.messages.at(-1)?.content ?? "";
  },
  { name: "logistics_agent", description: "处理快递、物流、地址修改相关问题", schema: z.object({ question: z.string() }) },
);

const router = createReactAgent({
  llm: model,
  tools: [billingTool, logisticsTool],
  prompt: "你是客服入口，判断用户问题属于账单还是物流，转发给对应专家 Agent 处理，不要自己回答业务问题。",
});

const result = await router.invoke({
  messages: [new HumanMessage("我的订单 #12345 什么时候能到？")],
});
console.log(result.messages.at(-1)?.content);
```

Router 的 prompt 要写清楚"你只负责转发，不要自己答"，否则模型容易绕过路由直接回复，失去分发的意义。

## 什么时候该用多 Agent

引入前先问一句：单 Agent 加好 prompt、工具和中间件，是不是已经够了？够了就别引入，架构越简单越好排查。

![20260320094513](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320094513.png)

左侧四条正向信号（任务可拆分、工具策略不同、多团队维护、统一入口路由）说明该引入，右侧三条负向信号（工具少且同领域、只是 prompt 没写好、单 Agent 一轮能搞定）说明不必要。

## 小结

多 Agent 的价值在于"分工"和"可控"，不在于"更聪明"。选择合适的架构模式（Supervisor、Handoffs、Router、Custom Workflow）取决于你的业务是否有清晰的职责划分、是否需要不同团队独立维护、是否有统一入口分发的需求。

和前面几章讲过的安全护栏、MCP 工具、人机协同是叠加关系，不是替代关系。安全护栏可以只挂在高风险 Subagent 上，MCP 工具挂在哪个 Agent 身上由你决定，`interruptBefore` 只加在需要人审的那段接力流程上即可。多 Agent 的拆分反而让安全策略更精细，可以给不同 Agent 配置完全不同的护栏和审批规则。

引入之前值得先问一句，单 Agent 加好 prompt、工具和中间件，是不是已经够了？如果答案是"是"，就先别引入多 Agent。架构越简单，出了问题越好排查，等真正遇到单 Agent 扛不住的场景，再按需选择合适的多 Agent 模式来扩展。
