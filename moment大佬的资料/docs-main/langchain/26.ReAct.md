`ReAct` 全称是 Reasoning + Acting，和前端 UI 库 `React` 只是拼写相近，不是同一个东西。核心做法是让模型在给出最终答案之前，把推理和行动显式拆成几段来输出：先写当前在想什么（`Thought`），再决定调用哪个工具、传什么参数（`Action`），等工具执行完后把结果写回上下文（`Observation`），然后根据观察进入下一轮思考，如此循环直到信息足够、可以作答为止。

这种把推理过程摊在明面上的做法，就是 `ReAct` 区别于一次性猜答案的关键所在。

## 三段内容分别是什么

### Thought

`Thought` 是模型用自然语言写的一小段推理，用来说明当前这一步为什么要做这件事。比如用户问的是订单物流，我需要先查到订单号，再拿订单号去查物流，这一段不调用任何工具，也不返回给用户，只作为上下文的一部分喂给模型本身，用于驱动下一步的 `Action`，也方便在日志里看到 Agent 的推理路径。

在实现里，`Thought` 通常和 `Action` 放在同一次模型输出里。有的框架要求模型严格按 `Thought: ... Action: ...` 这类格式输出，再由解析器把两段拆开。有的框架则把推理折叠进内部字段或 `system` 指令里，对外只暴露 `tool_calls`，但逻辑上仍是一轮先想再动。

### Action

`Action` 表示模型决定调用哪一个工具以及传入的参数，比如工具名 `query_order`、参数 `{ "order_id": "12345" }`。在基于 function calling 的实现里，`Action` 对应模型返回的 `tool_calls` 里的某一项，由运行时解析后去调对应的函数或 `API`，再把返回值整理成 `Observation` 写回消息流。

工具集合和工具描述通过 `systemPrompt` 或 `tools` 参数告诉模型，模型在 `Thought` 里想清楚要查什么之后，就在 `Action` 里选出合适的工具并填好参数。这样工具调用的时机和理由都由推理过程显式驱动，而不是黑箱里一次性猜一堆调用。

### Observation

`Observation` 是工具执行完成后返回的结果，由外部系统（代码、`API`、数据库）写回到对话上下文中。模型在下一轮推理时会把这段结果当作当前已知的事实来读，再决定是继续调工具还是收尾作答。比如 `query_order` 返回订单已发货，快递单号 SF1234567890，模型就会在下一个 `Thought` 里基于这条信息决定是再查物流详情，还是直接总结回复用户。

`Observation` 严格由工具执行结果填充，不让模型自己编造观察内容，以避免幻觉。工具执行出错时，同样把错误信息作为 `Observation` 写回（例如 Error: 订单不存在），让模型有机会在下一轮里换策略或向用户说明。

## 消息在上下文中的形态

`ReAct` 跑起来后，对话上下文里会交替出现用户消息、带 `tool_calls` 的助手消息、工具消息（Observation），最后以一条不含 `tool_calls` 的助手消息收尾。如下图所示。

六条消息交替追加，循环到模型不再输出 `tool_calls` 为止。

在 `LangChain`、`LangGraph` 里，这些分别对应 `HumanMessage`、带 `tool_calls` 的 `AIMessage`，以及 `ToolMessage`（content 为工具返回值，`tool_call_id` 与请求对应）。循环逻辑很直接：每次模型返回里若还有 `tool_calls`，就执行工具、构造 `ToolMessage` 追回到 `messages` 末尾，再调一次模型。没有 `tool_calls` 时，就认为模型已给出最终回答，循环结束。

## 循环怎么跑、什么时候停

一轮典型的 `ReAct` 流程可以概括为：用户问题进入后，模型先输出 `Thought` + `Action`，运行时执行 `Action` 对应的工具并把结果写成 `Observation` 追到上下文末尾，再调一次模型；新一次模型看到的上下文里就多了一条 `Observation`，它再输出新的 `Thought` + `Action`，或直接输出最终回答。流程如下。

![20260320184210](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260320184210.png)

什么时候停由模型自己决定：当模型认为当前信息已经足够回答用户时，就不再输出 `Action`，而是输出面向用户的最终文本。实现上通常对应本轮模型返回里没有 `tool_calls`，或解析器识别到结束标记。工程上会设一个最大步数（比如 10 轮），防止任务卡死或成本失控，超过上限就强制结束并走降级逻辑。

## 两种常见实现方式

按 `Thought` 是否显式出现在消息里，可以粗分为两种。

显式 `Thought` 文本：在 `systemPrompt` 里要求模型严格按 `Thought: ... Action: ...` 格式输出，解析器从模型回复里截取 `Thought` 和 `Action`，再把 `Observation` 按固定格式追回。优点是日志里每一步的推理都能直接看到，便于调试和审计。缺点是对模型输出格式依赖大，需要做解析和容错，且 `Thought` 会占不少 `token`。

仅 function calling（逻辑上仍是 ReAct）：不要求模型输出 `Thought` 段落，只把工具列表和描述传给模型，由模型直接返回 `tool_calls`。循环仍然是模型产出 `tool_calls`、执行工具、把结果写成 `ToolMessage`、再调回模型。实现简单、和主流 API 兼容好，代价是工具调用没有显式的推理说明，可观测性弱一些，可以在需要时通过扩展字段或 `reasoning` 接口补一层理由。

实际项目里两种可以混用，对敏感或复杂步骤要求显式 `Thought`，其余用 function calling 降低成本。

## 错误与边界处理

工具执行失败时，应把错误信息作为 `Observation` 写回，不能吞掉或只打日志。返回形如 Error: 订单不存在 或 Error: 物流接口超时 的短文，模型在下一轮可以据此改口问用户是否记错了订单号，或提示稍后重试。若同一工具被短时间内重复调用且参数相同，可以做去重或缓存，避免无意义的重复请求。

步数上限和超时是必须有的，比如最大 10 轮、单轮模型调用超时 30 秒。超过步数上限时，可以返回当前已收集到的信息摘要并说明步骤已达上限，或引导用户缩小问题范围。对有权限或风控校验的敏感工具，若校验不通过，同样把无权限或拒绝原因作为 `Observation` 写回，让模型用自然语言向用户解释，而不是直接抛异常让整轮对话崩溃。

## 和直接猜答案相比有什么好处

不用 `ReAct` 时，往往是把问题和工具描述一次性丢给模型，让它在一次调用里决定调什么，推理过程藏在模型内部，出错时很难定位是哪一步想歪了，工具调用也容易扎堆或遗漏。`ReAct` 把推理过程显式化，主要带来以下几个好处。

- 调试与审计：每一步都有 `Thought` 和对应的 `Action`、`Observation`，日志里能清楚看到 Agent 为什么调了某个工具、调完之后又根据什么决定下一步，方便排查错误和做合规审计。
- 中途纠偏：模型在看到 `Observation` 之后可以立刻调整计划，比如发现订单不存在就改口问用户是否记错了订单号，而不是一条路走到黑。
- 工具调用可解释：工具调用由 `Thought` 段显式驱动，便于做成本控制、权限控制和体验优化。

代价是对话轮次变多，`Thought` 和 `Action` 都会占用 `token`，延迟和成本都会上升，日志里也会多出大量中间内容，需要做好脱敏和归档。工程实践中可以根据任务复杂度控制 `ReAct` 的粒度，简单任务只允许一两轮循环，复杂任务再放开步数上限。

## 适合与不适合的场景

`ReAct` 适合需要边查边想、多步依赖工具的任务，比如先查订单再查物流、先搜索再总结、先算一遍再根据结果决定是否继续算。这类任务里，下一步往往依赖上一步的观察结果，`ReAct` 的循环正好贴合这种观察 → 推理 → 再行动的节奏。

不太适合的场景有两类：一是任务步骤非常多、依赖关系复杂，单靠局部推理容易漏步或重复，这时往往要配合 `Plan-and-Execute` 先列出完整步骤再执行；二是任务本身不需要工具、只需一次生成，引入 `ReAct` 只会增加成本和延迟。若模型经常在 `Thought` 里啰嗦或跑题，可以从 `systemPrompt` 或解析规则上约束格式，必要时用结构化字段代替长段自然语言。

## 完整代码示例

下面用一个前端技术顾问场景把整个 `ReAct` 流程串起来：定义三个工具、创建 Agent、发起问题，观察 Agent 如何自主决定调用哪些工具、按什么顺序调、最终给出综合建议。

三个工具分别负责查询 npm 包信息、检查 CSS 特性的浏览器兼容性、估算引入某个包对打包体积的影响。用户只问了一个问题，Agent 会自己判断需要调用哪几个工具，把每次工具返回的结果当作 `Observation` 读进去，再决定下一步是继续查还是可以作答了。

```typescript
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";

  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
  temperature: 0,
});

const getNpmPackageInfo = tool(

// createAgent 内部跑的就是 ReAct 循环：
// 模型每次返回 tool_calls → 执行工具 → 把结果作为 Observation 追回 messages → 再调模型
// 直到模型返回里不再有 tool_calls，循环结束，输出最终回答
const agent = createAgent({
  model,

});

const result = await agent.invoke({
  messages: [
    new HumanMessage(
      "我在做一个 React 项目，想用 CSS Grid 做响应式布局，再引入 framer-motion 做页面过渡动画。帮我查一下 CSS Grid 的浏览器兼容性，以及 framer-motion 的最新版本和打包体积，看看值不值得引入。",
    ),
  ],
});

console.log(result.messages.at(-1)?.content);
```

这个问题需要同时查兼容性、版本和体积三类信息，模型会自己决定依次调用 `check_browser_support`、`get_npm_package_info`、`analyze_bundle_impact`，把每次返回的结果读进去，再判断信息是否足够、要不要继续查，最终汇总给出建议。整个过程不需要手动编排调用顺序，正是 `ReAct` 循环的典型体现。

工具里用的是 mock 数据，实际项目里可以把 `getNpmPackageInfo` 换成调 `registry.npmjs.org` API，把 `checkBrowserSupport` 换成读 caniuse 数据，逻辑结构完全不用改。

## 小结

`ReAct` 的核心价值在于把推理过程从模型的黑箱里拉出来，让每一步为什么这么做都变得可读、可调试。它本质上是一个循环：模型输出 `Thought` + `Action`，工具执行后把 `Observation` 追回上下文，再调模型，直到不再需要工具为止。实现上用 `createAgent` 可以开箱即用，框架替你维护整个循环；若需要更细粒度的控制，手写循环也只需几十行代码，加超时、日志或权限校验都很方便。`ReAct` 最适合边查边想的多步任务，任务步骤多时可以配合 `Plan-and-Execute` 先规划再执行，输出质量要求高时再叠一层 `Self-Critic` 做审核，不同思考模式组合起来，才能覆盖真实工程里的大多数场景。
