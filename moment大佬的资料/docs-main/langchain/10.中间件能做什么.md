前面的内容里，我们把模型、消息、工具串成了一条可运行的 `Agent` 循环。接下来关注的问题是：如何在这个循环的不同节点上插入可复用的逻辑，这就是中间件（`middleware`）的职责。

## 中间件解决什么问题

不挂中间件时，限流、重试、`PII` 脱敏、日志这类横切需求只能写死在循环主流程里。它们会和业务逻辑缠在一起，复用和测试都会变难。

中间件把某个步骤前后要做的事情抽出来，做成可插拔的组件。你可以在钩子里读写消息、替换工具、决定是否继续，也可以打日志、抛错。需要时挂上去即可，不必动核心循环的代码。

![20260316092328](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316092328.png)

左列为无中间件时"横切需求无处安放"的困境，右列为挂上中间件后横切关注点变成可插拔的独立模块。

## Agent 循环里有哪些可插的时机

四个核心钩子分别是：

- `beforeModelCall`：调模型前，可以修改消息列表（如摘要压缩、`PII` 脱敏）、替换工具列表、或中止执行
- `afterModelCall`：调模型后、判断是否有 `tool_calls` 前，可以检查输出、打日志、或修改 `AIMessage`
- `beforeToolCall`：执行某个工具前，可以做人工审批、参数校验、限流检查
- `afterToolCall`：工具执行完后，可以做重试、修改返回结果、或打调用日志

![20260316092833](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316092833.png)

四个钩子分布在 Agent 循环的各个位置，工具执行完毕后会回到 `beforeModelCall` 形成循环，直到模型不再返回 `tool_calls` 时跳出。

## 在 LangChain 里怎么挂上中间件

在 LangChain 的 TypeScript 里，用 `createAgent` 时通过选项 `middleware` 传入一个中间件数组即可。框架在执行 Agent 循环时会按数组顺序依次调用各中间件的钩子，你要做的就是把需要的中间件（内置或自己写的）放进这个数组。

下面是一个最小示例，用 DeepSeek 创建模型实例、定义两个真实工具，在创建 Agent 时挂上模型调用次数限制和工具重试：

```typescript
import {
  createAgent,
  modelCallLimitMiddleware,
  toolRetryMiddleware,
  tool,
} from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";
import { z } from "zod";

const getWeather = tool(
  async (input: { location: string }): Promise<string> => {
    // 实际实现见第 6 章，此处省略 API 调用细节
    return `${input.location} 今天晴，25°C`;
  },
  {
    name: "get_weather",
    description: "根据城市名称查询当前天气。",
    schema: z.object({
      location: z.string().describe("城市名称，如北京、上海"),
    }),
  },
);

const internetSearch = tool(
  async (input: { query: string }): Promise<string> => {
    // 实际实现见第 6 章，此处省略 Tavily 调用细节
    return `关于"${input.query}"的搜索结果摘要…`;
  },
  {
    name: "internet_search",
    description: "在互联网上搜索最新信息。",
    schema: z.object({
      query: z.string().describe("搜索关键词或问题"),
    }),
  },
);

const llm = new ChatDeepSeek({
  model: "deepseek-chat",
  temperature: 0,
  apiKey: "你的 DeepSeek 密钥",
});

const agent = createAgent({
  model: llm,
  tools: [getWeather, internetSearch],
  middleware: [
    modelCallLimitMiddleware({ runLimit: 10, exitBehavior: "end" }),
    toolRetryMiddleware({ maxRetries: 2 }),
  ],
});

await agent.invoke({
  messages: [{ role: "user", content: "深圳今天天气怎么样？" }],
});
```

后面几节介绍的内置中间件，用法都是往 `middleware` 数组里再追加一项或多项。如果你是自己手写的"模型 + 工具循环"（例如第 7 章里的 `runAgent`），没有这一层抽象，要用中间件就得迁到 `createAgent`，或在循环里手写类似的 before、after 逻辑。

## 中间件大致能做什么

从能力上可以粗分为五类，方便你判断这类需求该不该用中间件。下面每类单独成节，并配一张图说明其典型用法和对应中间件。

### 控制成本与执行边界

限制单次对话里模型或某工具被调用的次数，避免无限循环或费用失控。也可以在某一步直接终止并返回，或切到更便宜的模型做后续步骤。典型中间件有 `modelCallLimitMiddleware`、`toolCallLimitMiddleware`。

如下图所示。

![20260316093734](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316093734.png)

从防止无限调用、设次数上限到达到上限后的行为与可选降级。

### 提升稳定性

模型或工具调用失败时自动重试（带退避），主模型不可用时回退到备用模型，让 Agent 在偶发故障下仍能尽量完成请求。典型中间件有 `modelRetryMiddleware`、`toolRetryMiddleware`、`modelFallbackMiddleware`。

如下图所示。

![20260316093910](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316093910.png)

从偶发失败、模型与工具重试到主模型不可用时的回退备用。

### 合规与安全

在用户输入或模型输出里检测并处理 PII（脱敏、打码、拦截），或在高风险操作（如发邮件、写库）前插入"人工确认"步骤，满足审计或合规要求。典型中间件有 `piiMiddleware`、`humanInTheLoopMiddleware`。

如下图所示。

![20260316094256](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316094256.png)

从 PII 检测与处理方式到高风险操作前的人工审批。

### 改造输入与输出

在接近上下文长度上限时把旧消息摘要压缩，或在调主模型前用小模型先筛出"本轮可能用到的工具"再传给主模型，从而省 token、提升效果。典型中间件有 `summarizationMiddleware`、`llmToolSelectorMiddleware`。

如下图所示。

![20260316094440](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316094440.png)

从上下文过长时的摘要压缩到工具过多时用小模型筛选再传主模型。

### 可观测与调试

在关键步骤前后打日志、上报指标，或把运行轨迹发给 LangSmith 等平台，便于排查问题和做数据分析。典型中间件或集成有 `loggingMiddleware`、LangSmith。

如下图所示。

![20260316094658](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260316094658.png)

从在钩子里打日志、上报指标到把运行轨迹接到 LangSmith 做排查与分析。

五类能力对应的中间件都通过 `createAgent` 的 `middleware` 数组挂载，执行顺序与数组顺序一致。

## 常用的内置中间件

LangChain 提供一批开箱即用的中间件，覆盖成本控制、稳定性、合规安全、上下文压缩、工具选择等场景。

### 把成本和执行次数管住

`modelCallLimitMiddleware` 给调模型的次数设上限，可以按整条会话（`threadLimit`，需配 `checkpointer`）或单次 `invoke`（`runLimit`）分别限制，`exitBehavior` 决定超出后是优雅结束（`"end"`）还是抛错（`"error"`）。

`toolCallLimitMiddleware` 限制工具调用次数，可以全局限制也可以按工具名单独设，超出后支持结束、抛错或把错误写回消息让模型自行处理（`"continue"`，默认）。

下面同时配置了模型次数上限与工具调用限制：

```typescript
import {
  createAgent,
  modelCallLimitMiddleware,
  toolCallLimitMiddleware,
  tool,
} from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";
import { MemorySaver } from "@langchain/langgraph";
import { z } from "zod";

const internetSearch = tool(
  async (input: { query: string }): Promise<string> => {
    return `关于"${input.query}"的搜索结果…`;
  },
  {
    name: "internet_search",
    description: "在互联网上搜索最新信息。",
    schema: z.object({ query: z.string().describe("搜索关键词") }),
  },
);

const writeDb = tool(
  async (input: {
    table: string;
    data: Record<string, unknown>;
  }): Promise<string> => {
    return `已写入 ${input.table}：${JSON.stringify(input.data)}`;
  },
  {
    name: "write_db",
    description: "向指定表写入一条记录，执行前请确认数据无误。",
    schema: z.object({
      table: z.string().describe("表名"),
      data: z.record(z.unknown()).describe("要写入的数据对象"),
    }),
  },
);

const llm = new ChatDeepSeek({
  model: "deepseek-chat",
  temperature: 0,
  apiKey: "你的 DeepSeek 密钥",
});

const agent = createAgent({
  model: llm,
  checkpointer: new MemorySaver(),
  tools: [internetSearch, writeDb],
  middleware: [
    modelCallLimitMiddleware({
      threadLimit: 10,
      runLimit: 5,
      exitBehavior: "end",
    }),
    toolCallLimitMiddleware({ threadLimit: 20, runLimit: 10 }),
    toolCallLimitMiddleware({
      toolName: "internet_search",
      runLimit: 3,
    }),
    toolCallLimitMiddleware({
      toolName: "write_db",
      runLimit: 1,
      exitBehavior: "error",
    }),
  ],
});
```

`write_db` 限为单次 `invoke` 只能调一次、超出抛错，适合"敏感操作只允许执行一次"的合规要求。`threadLimit` 需配 `MemorySaver` 才能生效，只做单次限制则只需设 `runLimit`。

### 失败时重试、顶不住就换模型

- `modelRetryMiddleware`：单次模型请求失败时自动重试，支持指数退避（`initialDelayMs`、`backoffFactor`）和随机抖动（`jitter`），可以只对特定 HTTP 状态码重试（`retryOn`）。
- `toolRetryMiddleware`：工具执行失败时自动重试，可以只对指定工具（`tools` 数组）生效。
- `modelFallbackMiddleware`：主模型不可用时按顺序尝试备用模型，适合多厂商容灾或主备降级。

下面把模型重试和模型回退一起挂上，先对主模型做最多 3 次指数退避重试，仍失败再切换到备用模型：

```typescript
import {
  createAgent,
  modelRetryMiddleware,
  toolRetryMiddleware,
  modelFallbackMiddleware,
  tool,
} from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";
import { z } from "zod";

const getWeather = tool(
  async (input: { location: string }): Promise<string> => {
    return `${input.location} 未来三天天气预报…`;
  },
  {
    name: "get_weather",
    description: "根据城市名称查询未来三天天气预报。",
    schema: z.object({
      location: z.string().describe("城市名称"),
    }),
  },
);

const primaryLlm = new ChatDeepSeek({
  model: "deepseek-chat",
  temperature: 0,
  apiKey: "你的 DeepSeek 密钥",
});

const fallbackLlm = new ChatDeepSeek({
  model: "deepseek-chat",
  temperature: 0,
  apiKey: "你的备用 DeepSeek 密钥",
});

const agent = createAgent({
  model: primaryLlm,
  tools: [getWeather],
  middleware: [
    modelRetryMiddleware({
      maxRetries: 3,
      initialDelayMs: 1000,
      backoffFactor: 2,
      maxDelayMs: 10000,
      jitter: true,
      retryOn: [429, 503],
      onFailure: "error",
    }),
    toolRetryMiddleware({
      maxRetries: 2,
      initialDelayMs: 500,
      tools: ["get_weather"],
    }),
    modelFallbackMiddleware(fallbackLlm),
  ],
});
```

`jitter: true` 在每次重试等待时间里加随机偏移，避免并发请求同时重试。

### 人审与敏感信息别乱跑

`humanInTheLoopMiddleware` 在指定工具执行前先暂停，交给人工选择通过、修改参数或拒绝，需配 `checkpointer` 保存暂停状态。

`piiMiddleware` 在发往模型前或拿到回复后检测并处理敏感信息，策略有脱敏（`redact`）、打码（`mask`）、哈希（`hash`）或直接拦截（`block`）。内置类型有 `email`、`credit_card`、`ip` 等，也可以用 `detector` 传正则识别业务自定义敏感模式。

下面对邮箱脱敏、信用卡打码、`API Key` 直接拦截，并对 `write_db` 开启人审：

```typescript
import {
  createAgent,
  piiMiddleware,
  humanInTheLoopMiddleware,
  tool,
} from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";
import { MemorySaver } from "@langchain/langgraph";
import { z } from "zod";

const writeDb = tool(
  async (input: {
    table: string;
    data: Record<string, unknown>;
  }): Promise<string> => {
    return `已写入 ${input.table}`;
  },
  {
    name: "write_db",
    description: "向指定表写入一条记录。",
    schema: z.object({
      table: z.string().describe("表名"),
      data: z.record(z.unknown()).describe("要写入的数据"),
    }),
  },
);

const llm = new ChatDeepSeek({
  model: "deepseek-chat",
  temperature: 0,
  apiKey: "你的 DeepSeek 密钥",
});

const agent = createAgent({
  model: llm,
  checkpointer: new MemorySaver(),
  tools: [writeDb],
  middleware: [
    piiMiddleware("email", { strategy: "redact", applyToInput: true }),
    piiMiddleware("credit_card", { strategy: "mask", applyToInput: true }),
    piiMiddleware("api_key", {
      detector: /sk-[a-zA-Z0-9]{32,}/,
      strategy: "block",
      applyToInput: true,
    }),
    humanInTheLoopMiddleware({
      interruptOn: ["write_db"],
      allowedActions: ["approve", "edit", "reject"],
    }),
  ],
});
```

多个 `piiMiddleware` 可以叠用，各自管一类敏感信息和一种策略。`humanInTheLoopMiddleware` 需要配 `MemorySaver`，暂停时要把当前状态存下来，等人工审批后再恢复。

### 上下文太长时摘要或裁剪

`summarizationMiddleware` 在接近上下文上限时用一个小模型把较早的消息压成摘要，只保留最近若干条完整消息。可以按消息条数、`token` 数或百分比触发（`trigger`），并配置保留多少最近完整消息（`keep`）：

```typescript
import { createAgent, summarizationMiddleware } from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";

const llm = new ChatDeepSeek({
  model: "deepseek-chat",
  temperature: 0,
  apiKey: "你的 DeepSeek 密钥",
});

// 用一个轻量模型专门做摘要，节省主模型的 token
const summarizerLlm = new ChatDeepSeek({
  model: "deepseek-chat",
  temperature: 0,
  apiKey: "你的 DeepSeek 密钥",
});

const agent = createAgent({
  model: llm,
  tools: [],
  middleware: [
    summarizationMiddleware({
      model: summarizerLlm,
      trigger: { type: "tokenCount", threshold: 6000 },
      keep: 10,
    }),
  ],
});
```

和第 8 章里的手动滑动窗口截断相比，语义压缩能更好地保留早期对话的核心信息。

### 少传点工具、做规划、本地假跑

工具一多，主模型光看工具列表就占不少 `token` 还容易选错。`llmToolSelectorMiddleware` 在调主模型前先用一个小模型从全部工具里筛出本轮可能用到的少数几个，可以配置最多选几个（`maxTools`）以及一定带上的工具（`alwaysInclude`）：

```typescript
import { createAgent, llmToolSelectorMiddleware, tool } from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";
import { z } from "zod";

// 假设这里有 10 个工具，实际使用时不可能每轮都全部传给模型
const tools = [
  // getWeather, internetSearch, writeDb, sendEmail, readFile, ... 等 10 个工具
];

const llm = new ChatDeepSeek({ model: "deepseek-chat", apiKey: "你的密钥" });
const selectorLlm = new ChatDeepSeek({
  model: "deepseek-chat",
  apiKey: "你的密钥",
});

const agent = createAgent({
  model: llm,
  tools,
  middleware: [
    llmToolSelectorMiddleware({
      model: selectorLlm,
      maxTools: 3,
      alwaysInclude: ["get_weather"],
    }),
  ],
});
```

联调或自动化测试时没有真实接口，可以用 `toolEmulatorMiddleware`：不真正执行工具，而是用 LLM 根据工具名和参数生成假结果，把 Agent 逻辑跑通而不调外部服务：

```typescript
import { createAgent, toolEmulatorMiddleware } from "langchain";
import { ChatDeepSeek } from "@langchain/deepseek";

const llm = new ChatDeepSeek({ model: "deepseek-chat", apiKey: "你的密钥" });
const emulatorLlm = new ChatDeepSeek({
  model: "deepseek-chat",
  apiKey: "你的密钥",
});

const agent = createAgent({
  model: llm,
  tools: [getWeather, internetSearch],
  middleware: [
    toolEmulatorMiddleware({
      model: emulatorLlm,
      tools: ["internet_search"],
    }),
  ],
});
```

这里只模拟 `internet_search`，`get_weather` 仍然真实调用，方便在联调阶段隔离外部依赖。

## 编写自己的中间件

当内置中间件满足不了需求时，可以自己实现一个中间件挂到同一个 `middleware` 数组里。下面是一个完整的可运行示例：在调模型前把当前 `messages` 条数打到日志，在执行工具前检查工具名是否在白名单里，不在就抛错拒绝。LangChain 里用 `BaseCallbackHandler` 实现等效的钩子逻辑，通过 `invoke` 的 `callbacks` 选项挂入：

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { tool } from "@langchain/core/tools";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { BaseCallbackHandler } from "@langchain/core/callbacks/base";
import type { Serialized } from "@langchain/core/load/serializable";
import type { BaseMessage } from "@langchain/core/messages";
import { HumanMessage } from "@langchain/core/messages";
import { z } from "zod";

// ── 模型 ──────────────────────────────────────────────────────────────────────

const model = new ChatOpenAI({
  model: "Pro/zai-org/GLM-5",
  apiKey: "sk-cjqjqkehyqpspbqcvnlkvifbvfjrmdusostyrvdbdaatrzyw",
  configuration: { baseURL: "https://api.siliconflow.cn/v1" },
  temperature: 0,
});

// ── 白名单 ────────────────────────────────────────────────────────────────────

const ALLOWED_TOOLS = new Set(["get_weather", "internet_search"]);

// ── 自定义中间件（继承 BaseCallbackHandler） ───────────────────────────────────

class AuditMiddleware extends BaseCallbackHandler {
  name = "audit-middleware";

  // 对应 beforeModelCall：调模型前触发
  override async handleChatModelStart(
    _llm: Serialized,
    messages: BaseMessage[][],
  ): Promise<void> {
    const messageCount = messages[0]?.length ?? 0;
    console.log(`[audit] 调模型前，当前 messages 条数：${messageCount}`);
  }

  // 对应 beforeToolCall：执行工具前触发
  override async handleToolStart(
    _tool: Serialized,
    _input: string,
    _runId: string,
    _parentRunId?: string,
    _tags?: string[],
    _metadata?: Record<string, unknown>,
    name?: string,
  ): Promise<void> {
    if (name && !ALLOWED_TOOLS.has(name)) {
      console.warn(`[audit] 工具 "${name}" 不在白名单，拒绝执行`);
      throw new Error(`工具 "${name}" 未授权，已被拦截。`);
    }
    console.log(`[audit] 允许执行工具：${name}`);
  }
}

// ── 工具定义 ──────────────────────────────────────────────────────────────────

const getWeather = tool(
  async (input: { location: string }): Promise<string> => {
    return `${input.location} 今天晴，25°C`;
  },
  {
    name: "get_weather",
    description: "根据城市名称查询当前天气。",
    schema: z.object({
      location: z.string().describe("城市名称，如北京、上海"),
    }),
  },
);

const internetSearch = tool(
  async (input: { query: string }): Promise<string> => {
    return `关于"${input.query}"的搜索结果摘要…`;
  },
  {
    name: "internet_search",
    description: "在互联网上搜索最新信息。",
    schema: z.object({
      query: z.string().describe("搜索关键词或问题"),
    }),
  },
);

// ── Agent ─────────────────────────────────────────────────────────────────────

const agent = createReactAgent({
  llm: model,
  tools: [getWeather, internetSearch],
});

// ── 运行：中间件通过 callbacks 挂入 ───────────────────────────────────────────

const result = await agent.invoke(
  { messages: [new HumanMessage("深圳今天天气怎么样？")] },
  { callbacks: [new AuditMiddleware()] },
);

const lastMsg = result.messages[result.messages.length - 1];
console.log("\n最终回复：", lastMsg.content);
```

在每次 `agent.invoke` 时传入 `{ callbacks: [new AuditMiddleware()] }` 即可，若要全局生效也可以在创建模型时直接传入 `callbacks`。

自定义中间件建议单独放在一个文件里，这样限流、打点、审计等逻辑都与业务解耦，方便独立单测和复用。

## 小结

中间件把限流、重试、脱敏、人审、摘要等横切关注点从 Agent 循环里拆出来，做成可复用、可配置的模块。四个核心钩子 `beforeModelCall`、`afterModelCall`、`beforeToolCall`、`afterToolCall` 覆盖了所有可插入的时机，所有中间件通过 `createAgent` 的 `middleware` 数组挂载，执行顺序与数组顺序一致。
