前面几篇我们已经把 `LCEL`、`Runnable`、`Tool` 这些执行相关的话题讲透了，接下来要解决另一类问题，模型本身是无状态的，但产品里几乎所有场景都需要上下文。只要对话多跑几轮，就会遇到几个非常现实的问题，历史越长 token 成本越高、关键事实容易被截断、长期偏好在下一次会话里又要重新问。

这些问题共同的答案就是 `Memory`。这一篇不追求把 `LangChain` 里所有 `Memory` 类都罗列一遍，而是围绕两件事展开，第一，主流 `Memory` 到底解决了什么，各自的代价是什么；第二，真实项目里应该怎么组合它们，才能在成本、精度、长期记忆之间做出合理取舍。

## Memory 要解决的本质问题

在讨论具体类型之前，先把 `Memory` 的目标说清楚。`LLM` 一次调用只能看到传进去的上下文，模型自己不会记住上一轮说过什么。所谓 `Memory`，其实就是在每次调用模型之前，帮我们完成两件事：

- 决定把哪些历史信息放进这次请求的上下文里
- 决定这些历史信息以什么形态存在，是原文、摘要，还是检索片段

不同的 `Memory` 类型，本质差别就在于这两个动作的策略不同。这也直接决定了三个关键指标：

- `token` 成本，和每次请求注入的内容长度成正比
- 信息保真度，原文最高、摘要次之、检索取决于召回质量
- 长期记忆能力，指跨会话、跨天级别是否还能回忆起关键事实

理解了这三个指标，后面选型就不再是背 `API`，而是在三角形里做平衡。

## 主流 Memory 类型快览

`LangChain` 里常用的 `Memory` 可以按策略归为三类，下面先给一张速查表，再逐个展开。

| 类型                         | 策略                     | 主要收益           | 主要代价                   |
| ---------------------------- | ------------------------ | ------------------ | -------------------------- |
| `BufferMemory`               | 原样保存全部历史         | 信息零损失         | `token` 成本随对话线性增长 |
| `BufferWindowMemory`         | 只保留最近 `k` 轮        | 控本简单、实现直观 | 超出窗口的内容直接丢失     |
| `SummaryMemory`              | 把历史滚动压缩成摘要     | 长对话下成本稳定   | 细节信息会被摘要平滑掉     |
| `SummaryBufferMemory`        | 摘要加最近几轮原文       | 兼顾全局与近期细节 | 实现稍复杂、摘要更新有成本 |
| `VectorStoreRetrieverMemory` | 历史向量化后按需 `top-k` | 支持长期记忆与召回 | 依赖向量库、可能跑题       |

表格里的信息是方向性的，真实选择还要结合业务类型、对话平均长度、是否跨会话。接下来按顺序展开。

## Buffer 系列作为原样保存的底线方案

`BufferMemory` 是最朴素的一种，它不做任何压缩，把每一轮对话原文追加进上下文。写 `demo`、做原型验证、对话轮次很少的工具类产品里，这种方式最合适，因为调试时我们能看到完整的历史，没有任何信息损失。

下面是一个最小可运行示例，用来演示 `BufferMemory` 在链路里的位置。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import {
  ChatPromptTemplate,
  MessagesPlaceholder,
} from "@langchain/core/prompts";
import { RunnableWithMessageHistory } from "@langchain/core/runnables";
import { ChatMessageHistory } from "@langchain/community/stores/message/in_memory";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个耐心的客服助手，请结合历史对话回答用户。"],
  new MessagesPlaceholder("history"),
  ["human", "{input}"],
]);

const chain = prompt.pipe(model).pipe(new StringOutputParser());

const historyStore = new Map<string, ChatMessageHistory>();

const withHistory = new RunnableWithMessageHistory({
  runnable: chain,
  getMessageHistory: (sessionId: string) => {
    if (!historyStore.has(sessionId)) {
      historyStore.set(sessionId, new ChatMessageHistory());
    }
    return historyStore.get(sessionId)!;
  },
  inputMessagesKey: "input",
  historyMessagesKey: "history",
});

async function main(): Promise<void> {
  const config = { configurable: { sessionId: "user-001" } };
  await withHistory.invoke({ input: "我叫小白" }, config);
  const reply = await withHistory.invoke({ input: "你记得我叫什么吗" }, config);
  console.log(reply);
}

main().catch(console.error);
```

这段代码的重点不是 `API`，而是让我们看到，`BufferMemory` 的代价是每次调用都会把整个 `history` 塞进 `prompt`。对话到几十轮之后，`token` 费用和延迟都会被拖起来。

`BufferWindowMemory` 是对这个问题最直白的回应，只保留最近的 `k` 轮，超出的直接丢掉。它实现简单，容易预估成本，适合对话主题切换频繁、旧内容复用价值不高的场景，例如搜索式问答、代码补全助手。

在 `LangChain JS` 的新写法里，等价做法是在进入 `prompt` 前用 `trimMessages` 做裁剪。

```typescript
import { trimMessages } from "@langchain/core/messages";
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const trimmer = trimMessages({
  maxTokens: 1200,
  strategy: "last",
  tokenCounter: model,
  includeSystem: true,
  startOn: "human",
});
```

这里的 `maxTokens` 是硬约束，`strategy: "last"` 表示从最新一条往前保留，`includeSystem: true` 可以始终保住系统消息。这个 `trimmer` 可以放到 `prompt` 之前，作为 `BufferWindowMemory` 的现代替代。

`Buffer` 系列的共同短板很明显，它只负责裁剪或保留，不做任何语义理解。对于需要保留较长历史但又控不住成本的场景，就要走 `Summary` 路线。

## Summary 系列用摘要压住长对话成本

`SummaryMemory` 的思路是，每当历史增长到一定程度，就调用一次模型把旧内容压缩成一段摘要，后续上下文里只带摘要。好处是上下文长度可以稳定在一个可控区间，坏处是摘要天然会损失细节，尤其是数字、专有名词、状态标识这类信息。

如果直接用 `SummaryMemory`，最近一两轮的原文也会被压掉，导致模型在做下一步决策时不知道用户刚刚说了什么。生产里更常用的是 `SummaryBufferMemory`，它的规则很简单：

- 最近的一小段对话原文完整保留
- 更早的内容滚动进摘要
- 每当近期窗口超过阈值，就触发一次摘要更新

这样既保住了短期细节，又把长期信息压在一个可控体积里。下面这段示例展示摘要更新的核心逻辑，实际项目里可以把 `summarize` 替换为调用 `DeepSeek`。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import {
  BaseMessage,
  HumanMessage,
  AIMessage,
  SystemMessage,
} from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

interface SummaryState {
  summary: string;
  recent: BaseMessage[];
}

async function summarize(
  prev: string,
  dropped: BaseMessage[],
): Promise<string> {
  const transcript = dropped
    .map((m) => `${m._getType()}: ${m.content}`)
    .join("\n");
  const res = await model.invoke([
    new SystemMessage(
      "请把历史对话压缩成一段不超过 120 字的摘要，保留用户画像与关键事实。",
    ),
    new HumanMessage(`已有摘要：${prev}\n新增对话：\n${transcript}`),
  ]);
  return res.content as string;
}

async function appendAndCompress(
  state: SummaryState,
  message: BaseMessage,
  keepTurns: number,
): Promise<SummaryState> {
  const recent = [...state.recent, message];
  if (recent.length <= keepTurns * 2) {
    return { summary: state.summary, recent };
  }
  const dropped = recent.slice(0, recent.length - keepTurns * 2);
  const kept = recent.slice(recent.length - keepTurns * 2);
  const summary = await summarize(state.summary, dropped);
  return { summary, recent: kept };
}
```

这段代码抽出了 `SummaryBufferMemory` 的核心状态机：一份滚动摘要加一个固定长度的近期窗口。真实业务里还要考虑摘要更新的触发条件、摘要里保留哪些字段、是否把用户画像单独建模。

`Summary` 系列的短板在两个地方，一是摘要会把细节抹平，比如订单号、金额、时间戳这类强事实容易丢失；二是摘要质量强依赖模型和提示词，如果提示词写得太宽泛，压几轮之后摘要就会变成套话。对这两个短板，答案通常不是换更强的摘要模型，而是把强事实从对话流里拆出去，交给向量记忆。

## Vector Memory 负责长期事实与跨会话记忆

`VectorStoreRetrieverMemory` 的思路和前面两类完全不同。它不把历史塞进上下文，而是把每一条有价值的信息向量化之后写入向量库，当用户提新问题时，按语义做 `top-k` 检索，只把最相关的几条召回并注入上下文。这种方式的优势在三个场景非常明显：

- 长期记忆，跨会话也能记住用户偏好和历史事实
- 海量历史，不需要把全部对话装进每次请求
- 事实型召回，强事实经过向量化后不会像摘要那样被平滑掉

下面这段示例使用 `MemoryVectorStore` 做最小演示，实际项目里可以替换为 `PGVector`、`Pinecone`、`Milvus` 这类持久化向量库。

```typescript
import { ChatOpenAI, OpenAIEmbeddings } from "@langchain/openai";
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { Document } from "@langchain/core/documents";

const embeddings = new OpenAIEmbeddings({
  model: "text-embedding-3-large",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

const store = await MemoryVectorStore.fromDocuments([], embeddings);

async function remember(userId: string, fact: string): Promise<void> {
  await store.addDocuments([
    new Document({ pageContent: fact, metadata: { userId, ts: Date.now() } }),
  ]);
}

async function recall(userId: string, query: string, k = 3): Promise<string[]> {
  const docs = await store.similaritySearch(query, k, { userId });
  return docs.map((d) => d.pageContent);
}

await remember("u-123", "用户偏好深色主题");
await remember("u-123", "用户在上海，常用地铁上下班");
const hits = await recall("u-123", "给他推荐一个通勤场景下能用的功能");
console.log(hits);
```

这段代码有两个实践细节值得注意。第一，写入时不要把整段对话都塞进去，写入的应该是事实片段，例如偏好、画像、关键历史动作。第二，读取时一定要加 `metadata` 过滤，至少按 `userId` 隔离，否则很容易召回到别的用户的信息。

向量记忆的短板也很现实，它容易跑题。如果 `top-k` 拉得太大、相似度阈值放得太松，会把不相关的历史也塞进上下文，反而干扰模型判断。这也是它通常不单用，而要和短期 `Buffer` 配合的原因。

## 选型决策的直观路径

上面三类方案在大多数场景里并不是互斥的。为了避免被具体类名带偏，我们可以把选型简化成一条决策路径。

![20260422173421](https://raw.githubusercontent.com/xun082/md/main/blogs.images20260422173421.png)

这张图的使用方式是，先回答对话轮次预期，再看是不是跨会话。大多数客服、助手类产品的结论都会落在右下那条分支，也就是 `Vector` 加短期 `Buffer`。

## 三层组合

真正在线上跑的系统，很少只用单一 `Memory`。实战里比较常见的做法是分三层，把不同性质的信息交给最适合的存储。

- 短期层，保留最近 `N` 轮原文，解决模型刚刚说过什么的问题
- 摘要层，把中期历史滚动压缩，解决这次会话主线是什么
- 长期层，用向量库存储用户画像、偏好、关键事实，解决之前到底发生过什么

下面这段示意把这三层组织到一次请求里，便于看到它们在 `prompt` 构造阶段是怎么合流的。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import {
  ChatPromptTemplate,
  MessagesPlaceholder,
} from "@langchain/core/prompts";
import { HumanMessage, BaseMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({
  model: "deepseek-chat",
  apiKey: "sk-60816d9be57f4189b658f1eaee52382e",
  configuration: { baseURL: "https://api.deepseek.com" },
});

interface MemoryContext {
  summary: string;
  recent: BaseMessage[];
  recalled: string[];
}

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一位客服助手。请综合历史摘要、相关事实、近期对话给出回答。"],
  ["system", "历史摘要：\n{summary}"],
  ["system", "相关事实：\n{recalled}"],
  new MessagesPlaceholder("recent"),
  ["human", "{input}"],
]);

async function answer(ctx: MemoryContext, input: string): Promise<string> {
  const messages = await prompt.formatMessages({
    summary: ctx.summary || "无",
    recalled: ctx.recalled.join("\n") || "无",
    recent: ctx.recent,
    input,
  });
  const res = await model.invoke(messages);
  return res.content as string;
}
```

这段代码里有两点值得单独说一下。一是三层信息通过三条不同管道填进 `prompt`，`recent` 来自短期缓冲，`summary` 来自摘要状态，`recalled` 来自向量检索，互相不串线。二是这种结构对排查问题很友好，模型答错时可以单独回看是哪一层的内容出了偏差，而不是对着一大段历史盲猜。

这种组合方式和 `LangGraph` 走的是同一条思路，只是这里没画成图。实际项目里很多团队会直接用 `LangGraph` 的 `checkpointer` 把状态持久化，`Memory` 的角色就退化为状态里的一个字段。

## 成本与精度之间的旋钮

无论选哪种组合，上线后都会遇到调优需求，归纳下来有几个关键旋钮可以拧。

- 历史裁剪长度，决定每次请求注入多少原文，直接影响 `token` 成本
- 摘要触发阈值，决定多久压一次，压得太勤摘要噪声大，太稀则上下文不稳定
- 摘要提示词，决定摘要里保留什么信息，建议显式列出画像、订单、目标这几类强事实字段
- 向量检索 `top-k` 与相似度阈值，`k` 大容易跑题，阈值紧会漏召
- 长期库的写入策略，是每条对话都写，还是只写被提取出来的事实

这些旋钮没有通用最优值，常见做法是先用保守参数跑起来，再通过离线评估集做迭代。评估集里至少要覆盖三类样本，短对话、长对话、跨会话，才能看清楚每一层记忆的实际表现。

## 新项目应该怎么组织 Memory

顺着上面的话题再往下说一个实际问题。如果你现在起一个新项目，其实不太建议直接照搬 `BufferMemory`、`SummaryMemory` 这些类。`LangChain JS` 的新版本里它们大都被标记为 `deprecated`，官方建议改用下面这套组合。

- 会话状态交给 `LangGraph` 的 `checkpointer`，持久化不用自己写存储层
- 消息裁剪用 `trimMessages`，替代 `BufferWindowMemory` 的作用
- 摘要逻辑写成一个独立 `node`，什么时候压、压成什么样都能单独调
- 长期事实走向量库，作为一个独立的检索 `node` 挂在图里

这样做真正的好处是把状态从 `Memory` 类里搬到外面。状态是 `LangGraph` 的一部分，可读、可测、可回放，裁剪、摘要、检索三件事各自都是显式的 `node`，可以单独换、单独加日志。前面那些传统 `Memory` 类在本文里出现的意义，更多是帮我们理解每种策略在解决什么问题，不一定要直接拿来写线上代码。

## 总结

回到开头的两个问题。第一，主流 `Memory` 解决的是同一件事，决定历史以什么形态进入下一次请求，差别在于成本、保真度、长期记忆能力的取舍。第二，真实项目里很少单用某一种，更合理的做法是按信息性质分层，短期原文、滚动摘要、长期向量各就其位。

落到选型上，可以记住一条简单的经验：

- 短对话原型阶段，`Buffer` 或 `trimMessages` 够用
- 单次会话较长、需要完整主线，选 `SummaryBuffer` 或等价自实现
- 跨会话、跨天需要长期记忆，必须引入向量记忆，并配合短期缓冲

再往工程方向走一步，`Memory` 本身不是孤立模块，它是状态管理的一部分。把裁剪、摘要、检索显式化，和 `LangGraph`、可观测性、评估集一起构成一条完整的对话系统骨架，才是这套技术真正落地的方式。
